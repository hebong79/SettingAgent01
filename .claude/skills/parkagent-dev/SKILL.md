# ParkAgent 개발 오케스트레이터

ParkAgent 코드 개발을 **4인 에이전트 팀**으로 조율한다. CLAUDE.md 5대 규칙(설계→유닛테스트→동작확인→한글문서화→영향도분석)을 파이프라인으로 실행한다.

## 실행 모드 (오케스트레이터가 판단해 셋팅)

작업 성격에 따라 **두 실행 모드** 중 하나를 리더가 Phase 0에서 선택한다. 어느 모드든 아래 **4인 에이전트 팀**과 CLAUDE.md 5대 규칙을 쓴다. 차이는 "누가 흐름을 주도하고, 성공을 어떻게 확인하느냐"다.

| 모드 | 언제(판단 기준) | 흐름 | 성공 확인 |
|------|----------------|------|-----------|
| **A. 선형 파이프라인**(기본) | 유닛테스트+리뷰로 성공을 확정할 수 있는 명확한 기능·버그·리팩토링 | 설계→구현→검증→문서 1회 통과(구현↔검증만 루프) | vitest 통과 + 경계면 교차 |
| **B. goal/loop**(경험적 반복) | 성공기준이 **관찰 가능**(시각·좌표정합·동작·튜닝)해 유닛테스트만으론 부족하거나 · 사용자가 goal/loop 형식으로 요청하거나("돌면서 확인", "스샷 확인", "~까지 반복", "틀어지면 다시") · 1회 구현으로 맞을 확률이 낮아 반복 보정이 필요할 때 | goal 명문화 → (분석→설계→구현→**경험적 검증**→틀어지면 재분석) 반복 → 성공 시 검증·문서 마감 | 리더의 **직접 경험적 관찰**(스샷/실행/로그) + vitest |

**모드 미확정 시 A가 기본.** 애매하면 B의 판단 기준(관찰형 성공·반복 보정 필요)에 하나라도 해당하는지 보고, 해당하면 B로 셋팅한다. 세부는 아래 "goal/loop 실행 모드 (B) 상세".

### 4인 에이전트 팀 (양 모드 공통)

| 단계 | 에이전트 | 타입 | 산출물 |
|------|----------|------|--------|
| 1. 설계 | architect | Plan | `_workspace/01_architect_plan.md` |
| 2. 구현 | developer | general-purpose | 소스 + `_workspace/02_developer_changes.md` |
| 3. 검증 | qa-tester | general-purpose | 테스트 + `_workspace/03_qa_report.md` |
| 4. 문서·영향도 | documenter | general-purpose | `*/docs/yyyyMMdd_hhmmss_*.md` + `_workspace/04_doc_impact.md` |

**모든 Agent 호출에 `model: "opus"` 필수.** 구현↔검증은 통과까지 루프.

## Phase 0: 컨텍스트 확인

작업 시작 전 실행 모드를 판별한다:
- `_workspace/` 존재 + 부분 수정 요청 → **부분 재실행**(해당 에이전트만 재호출)
- `_workspace/` 존재 + 새 입력 → **새 실행**(기존 `_workspace/`를 `_workspace_prev/`로 이동)
- `_workspace/` 미존재 → **초기 실행**

작업 대상 에이전트(Setting/Action/DM)와 범위를 파악하고, 관련 설계서를 먼저 읽는다.

**그리고 실행 모드(A 선형 / B goal·loop)를 판단해 셋팅한다**(위 "실행 모드" 표 기준). 사용자가 goal/loop 형식(goal + loop 단계 나열)으로 요청했으면 그 goal·loop을 그대로 채택한다. B로 정하면 Phase 2 대신 "goal/loop 실행 모드 (B) 상세"를 따른다.

## Phase 1: 팀 구성 및 작업 할당

1. `TeamCreate`로 4인 팀 구성(architect, developer, qa-tester, documenter).
2. `TaskCreate`로 의존성 있는 작업 생성: 설계 → 구현 → 검증 → 문서. 구현·검증은 양방향 의존(루프).
3. 작업 디렉토리 하위 `_workspace/`에 중간 산출물 저장. 파일명: `{단계번호}_{에이전트}_{산출물}.md`.

## Phase 2: 파이프라인 실행

1. **설계자**가 계획 수립 → `01_architect_plan.md`.
2. **구현자**가 계획대로 구현 → 소스 + `02_developer_changes.md`. 계획 결함 발견 시 설계자와 합의.
3. **검증자**가 vitest 작성/실행 → `03_qa_report.md`. 실패 시 구현자에게 보고, **통과까지 구현↔검증 루프**.
4. **문서화**가 01~03 + 코드를 종합 → 한글 `*/docs/yyyyMMdd_hhmmss_*.md` + `04_doc_impact.md`.

팀원은 `SendMessage`로 실시간 조율, `TaskUpdate`로 진행 공유, 파일로 산출물 전달.

## Phase 3: 종합 및 보고

리더가 결과를 종합해 사용자에게 보고: 변경 요약, 테스트 결과(통과/실패 그대로), 생성 문서 경로, 영향도 요약. `_workspace/`는 사후 감사용으로 보존.

## goal/loop 실행 모드 (B) 상세

리더가 B 모드를 택하면(또는 사용자가 goal/loop을 제시하면) 리더가 **루프를 직접 주도**하며 이터레이션마다 4인 팀을 호출한다. Phase 1(팀 구성)은 동일하되, Phase 2의 1회 통과 대신 아래 반복을 돌린다.

### 0) 구조화 프롬프트 (Goal / Loop / Requirements) — B 모드 확정 트리거
프롬프트가 `Goal:` / `Loop:` / `Requirements:` 세 섹션을 담고 있으면 **모드 판단 없이 B로 확정**하고, 각 섹션을 다음으로 매핑한다:

| 섹션 | 의미 | 실행 |
|------|------|------|
| **Goal** | 달성할 성공 목표 | 그대로 goal로 채택(§1 명문화 대체). 모호하면 관찰 가능한 성공기준으로만 구체화. |
| **Loop** | 리더가 따를 **이터레이션 절차** | 나열된 단계를 **순서대로** 실행. "틀어지면 다시 분석·설계·수정"류 단계는 **재시도 조건**으로 삼아 성공까지 반복. "스샷 확인"·"모든 프리셋 순회"류는 §3 경험적 검증 도구로 수행. |
| **Requirements** | 반드시 충족·검증할 **수용 기준(체크리스트)** | 설계·구현에 제약으로 반영. 마감 시 qa·문서가 **항목별로 대조**해 충족 여부 보고. 미충족·미검증은 은닉 없이 명시. |

Loop 단계 ↔ 4인 팀 매핑: 분석·코드설계=architect, 코드수정=developer, 스샷/동작 검증=리더 직접(§3), 최종 검증·문서=qa/documenter. **Requirements는 goal과 별개로 루프 내내 유지되는 불변 제약**이다. (구조화 프롬프트가 아니어도 §1~§4로 동일하게 수행.)

복사용 프롬프트 형식·예시는 같은 폴더의 `GOAL_LOOP_TEMPLATE.md` 참조.

### 1) goal 명문화
- 성공을 **검증 가능한 관찰**로 적는다. 예: "각 카메라·프리셋의 주차면이 데이터대로 화면에 그려진다", "정지 버튼이 현재 라운드 마무리 후 종료되고 UI가 이를 반영한다".
- 모호하면 설계자가 해석안 제시 → 리더가 사용자에게 확인.

### 2) loop 이터레이션
반복 단위: **분석 → 코드 설계(architect) → 구현(developer) → 경험적 검증(리더 직접) → 판정**.
- **틀어짐**(정합 어긋남·수치 이탈·동작 불일치) → 원인 재분석 후 설계·구현 재수정하고 다시 검증. 얕은 재시도로 덮지 말고 **먼저 사실(로그·렌더·수치)로 원인을 확인**(진단 플레이북 원칙과 동일).
- **성공** → 루프 종료 후 검증자(qa vitest)·문서화(documenter)로 마감. vitest는 순수 로직·수치·구조에 집중(DOM/렌더는 리더 경험적 검증이 담당).
- **은닉 금지**: 검증 못 한 항목(외부 서비스·시뮬레이터 미가동 등)은 성공으로 위장하지 않고 **한계로 명시** 보고.

### 3) 경험적 검증 도구 (리더가 이터레이션 사이 직접 사용)
- **스샷 렌더**: 시각·좌표 정합은 `sharp`(SVG→PNG)로 데이터/정규화 출력을 캔버스 크기(예: 1920×1080)로 렌더 → `Read`로 육안 확인. **실제 구현 함수(core.js 등)를 통과시켜** 원본↔파이프라인 출력의 픽셀 parity까지 본다. `node_modules`는 ParkAgent 루트에 호이스팅(`d:/Work/Parking3D/AgentVLA/ParkAgent/node_modules`). 임시 파일은 scratchpad에.
- **라이브 라우트**: 서버(13020) 구동 중이면 `curl`로 신규 라우트·응답 shape 확인(nodemon가 `src/` 변경 자동 리로드 — `config/`·정적자산은 미감시).
- **로그·시뮬레이터**: 아래 진단 플레이북의 검증 커맨드(로그 grep, `/cameras`, `/req_move`) 활용.
- 시뮬레이터·외부 서비스 미가동 시 렌더·유닛으로 대체하고, 실 프레임 위 정합 등 못 본 부분은 한계로 남긴다.

### 4) 카메라·프리셋 순회 패턴
"모든 프리셋을 돌며 확인"류는 프리셋별로 렌더/검증하고, 프리셋마다 데이터 검수(면 개수·좌표 범위·필수 필드)를 함께 수행한다. 결과 판정은 프리셋 단위로 기록한다.

### 참고 사례(이 저장소에서 수행)
- 정지버튼 버그(A 모드 예): 로그로 사실 확인 → 설계→구현→vitest→문서.
- PtzCamRoi 바닥 ROI 표시(B 모드 예): JSON 분석 → 좌표계를 sharp 스샷으로 검증 → 설계→구현 → 실제 `normalizePtzCamRoi`로 프리셋별 재렌더 parity 확인 → vitest→문서. 시뮬레이터 미가동은 한계로 명시.

## 데이터 전달 프로토콜

- **태스크 기반**(조율): `TaskCreate`/`TaskUpdate` — 의존 관계·진행 추적.
- **파일 기반**(산출물): `_workspace/{phase}_{agent}_{artifact}.md` — 단계 간 전달, 감사 추적.
- **메시지 기반**(실시간): `SendMessage` — 구현↔검증 루프, 설계 합의.

## 에러 핸들링

- 외부 서비스 미가동으로 스모크 불가 → 유닛(모킹)만 수행, 스모크는 보고서에 **누락 명시**(통과 위장 금지).
- 에이전트 1회 재시도 후 재실패 → 해당 결과 없이 진행하되 누락을 보고서에 명시.
- 상충 데이터(설계 vs 기존 코드)는 삭제하지 않고 출처 병기.
- 요청이 모호하면 설계자가 해석안을 제시하고 리더가 사용자에게 확인.

## 테스트 시나리오

**정상 흐름**: "ActionAgent에 LPR 호출 재시도 로직 추가" → 설계자가 `centering.run` 도구 경계·백오프 정책 계획 → 구현자가 `fetchWithLog` 패턴으로 구현 → 검증자가 LPR 모킹으로 재시도 테스트 작성/통과 → 문서화가 한글 문서 + LPR 클라이언트 영향도 작성.

**에러 흐름**: 검증자가 1-based/0-based 인덱스 불일치를 경계면 교차 비교로 발견 → 구현자에게 보고 → 구현자 수정 → 재검증 통과 → 문서에 발견 버그·수정 내역 기록.

## 후속 작업

description의 후속 키워드("다시 실행", "재실행", "업데이트", "수정", "보완")로 트리거된다. Phase 0에서 부분/새/초기 실행을 판별하고, 각 에이전트는 이전 산출물이 있으면 읽고 개선점만 반영한다.

## 알려진 이슈·진단 플레이북 (오진 반복 방지)

코드를 고치기 전에 **먼저 로그로 사실을 확인**한다. 추측으로 "ParkAgent 버그"라 단정하고 엉뚱한 곳을 고치면 증상이 재발한다.

### "유니티 시뮬레이터가 안 움직인다 / 정지 상태" 신고 시
근본 원인은 대개 ParkAgent 코드가 아니라 **(a) 시뮬레이터 렌더 구조 (b) 정밀수집 cadence (c) 시뮬레이터 측 미이행** 중 하나다. 순서대로 확인:

1. **`/req_img` vs `/req_move` 구분(핵심)**: Unity `/req_img`는 요청 PTZ의 이미지를 **가상 렌더만** 반환하고 카메라를 물리 이동시키지 않는다. 물리 이동은 **`/req_move`만** 한다. 뷰어 라이브 스트리밍은 매 프레임 `/req_img`를 쓰므로 **웹 화면은 갱신돼도 Unity 창은 그대로**인 게 정상이다.
2. **로그로 이동 여부 검증(추측 금지)**: `SettingAgent/logs/setting_*.log`(최신)에서 `grep req_move` → status가 **200이면 ParkAgent는 물리 이동을 정상 전송한 것**이다. 200인데 Unity가 안 움직이면 **시뮬레이터 측 문제**(그 카메라 리그가 /req_move를 물리 반영 안 함 — 예: cam2 사례). ParkAgent를 고치지 말 것.
3. **정밀수집 cadence 착시**: CaptureJob은 `moveBeforeCapture=true`이면 타깃마다 `/req_move` 후 캡처한다. 라운드당 프리셋 수만큼 **몇 초 동안 빠르게 순회**한 뒤 **`intervalMs`(기본 30s, 실측 60~80s)만큼 완전 정지**했다가 다음 라운드에 다시 움직인다. → 대기 구간이 "동작 안 함"으로 보인다. 로그의 `req_move` **타임스탬프 간격**을 보면 버스트+idle 패턴이 드러난다. 해결: intervalMs를 줄이거나 대기 중임을 UI에 명시. **버그 아님**.
4. **물리 이동 조건**: `captureTarget`은 `cfg.moveBeforeCapture && t.ptz.{pan,tilt,zoom}` **3개 모두 정의**일 때만 `/req_move` 한다. 타깃 PTZ가 불완전하면 이동을 건너뛴다. 타깃 출처(`presetProvider` /cameras 또는 `camerapos.json`)의 PTZ 완전성을 확인.
5. **뷰어 드롭다운 이동**: 프리셋/카메라 드롭다운 선택 시 `gotoPreset()`→`/move`→`/req_move`로 물리 이동한다(방향/절대/이동 버튼도 동일). 초기 페이지 로드 시엔 레이스 방지로 물리 이동 안 함(의도).

### 검증 커맨드(진단 시 바로 사용)
- 시뮬레이터 연결: `GET http://localhost:13100/cameras` (200이면 정상)
- 물리 이동 단독 테스트: `POST http://localhost:13100/req_move {cam_idx,pan,tilt,zoom}` → `{success:true}` 확인
- 뷰어 체인: `POST http://localhost:13020/viewer/api/move {cam,pan,tilt,zoom}` → `{ok:true}`
- 로그 이동 검증: 최신 `logs/setting_*.log`에서 `req_move` status·타임스탬프 간격 분석

### LLM/외부 서비스 "연결 안 됨" 신고 시
"연결 안 됨" 경고(예: floor ROI 폴백)는 대개 **연결이 아니라 설정값 문제**다. `llm.config.json` 등을 열기 전에: `GET {baseUrl}/v1/models`로 **모델명이 서버에 실재하는지** 먼저 확인(모델 오타→404→"연결 안 됨"으로 오인). config 파일(`config/`)은 nodemon 감시(src/) 밖이라 **변경 후 서버 재시작 필요**.

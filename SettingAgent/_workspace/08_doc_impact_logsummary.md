## 1. 기존 로그 판독 관행(줄 수 세기)이 과소평가로 바뀜

이번 변경 전에는 `cat:'packet'` 필터로 나온 줄 수를 그대로 세면 실제 통신 횟수와 일치했다(선행 문서 `20260724_225404`의 3fps 근본원인 추적도 이 방식으로 이루어졌다 — "10초당 패킷 수"를 로그 줄 수로 셈). 이번 변경 이후에는 **성공 반복 건이 요약으로 뭉쳐지므로 줄 수 세기가 실제 트래픽을 그대로 반영하지 않는다.** 예: 5분간 76회 통신이 있어도 로그는 3줄(즉시 1 + 요약 1 + 즉시 1)뿐이다.

**대체 산식** (요약 줄의 필드로 환산):

- 창 평균 케이던스: `n / win * 1000` (건/초) — 창 전체 길이 기준, 침묵이 섞이면 과소평가
- **활성 케이던스(권장)**: `(n - 1) / span * 1000` — 실제 활동 구간만 반영, `n >= 2`에서만 유효
- 침묵량: `win - span`

**영향받는 소비처**:
- 사람이 직접 로그를 grep해 트래픽/장애를 진단하는 모든 운영 관행(과거 3fps 사건 진단 방식과 동일한 패턴을 다시 쓰려는 경우) — 반드시 `n`/`win`/`span` 환산을 거쳐야 함. 최종 문서 §4에 표로 명시.
- 자동 파서/스크립트: 저장소 전체 grep 결과 `cat:'packet'`을 파싱하는 외부 스크립트·툴은 **0건**(설계서 §5 확인 사항, 재확인 완료) — 즉시 깨지는 자동화는 없음.

## 2. `cat:'packet'` 이 3종으로 분화

| 종류 | 발생 위치 | level | 식별 |
|------|-----------|-------|------|
| 즉시기록 | `http.ts`(`logPacket`), `HucomsClient.fetchResponse` | 30(성공) / 40(실패·비-2xx) | `win` 필드 없음 |
| 요약 | 동일 발생지(집계기가 창 만료 시 생성) | 30 고정 | `win`/`span` 필드 존재, msg가 `… 요약`으로 끝남 |
| CaptureJob 이동실패 warn | `src/capture/CaptureJob.ts:366` | 40(warn) | **본 변경과 무관** — "캡처 전 이동 실패(흡수)" 1회성 경고이며 `packetAggregator`/`packetLog`를 전혀 거치지 않음 |

**영향**: `cat:'packet'`으로만 필터링해 "통신 로그"로 취급하던 도구/스크립트/사람의 눈이 있다면, 세 번째 종류(캡처 이동실패)가 섞여 나온다는 점을 인지해야 한다. 다만 이는 이번 변경으로 새로 생긴 것이 아니라 원래도 같은 `cat`을 공유했다(구조 변경 없음, 신규 분화가 아니라 기존 사실의 재확인).

## 3. 비-2xx 응답의 level 승격 (30 → 40) — 행위 변경

기존에는 비-2xx 응답도 `logger.info`(level 30)로 기록됐다. 이번 변경으로 `packetLog.ts`의 `emit`이 `failed`(예외 또는 비-2xx) 여부로 `logger.warn`(level 40) / `logger.info`(level 30)를 가른다(`packetLog.ts:23`). **메시지 문자열 자체는 유지**(`'통신 패킷'`/`'Hucoms 통신 패킷'`, 예외만 `'… 실패'` 접미)되고 레벨만 오른다.

**영향**:
- `LOG_LEVEL` 환경변수를 `warn` 이상으로 좁혀 운영하는 배포가 있다면, 비-2xx 응답이 이제 그 필터를 통과해 보이게 된다(가시성 향상 — 의도된 변경).
- 이 레벨 변경을 단언하는 기존 테스트는 없었다(구현자 확인, 스파이 테스트는 200 경로만 사용). QA가 `packetLogAudit.test.ts`의 A7/B3에서 신규로 단언을 추가해 회귀 가드를 마련했다.
- 로그 레벨 기반 알림/모니터링을 별도로 구성해 둔 경우가 있다면(현재 저장소 내 그런 자동화는 확인되지 않음) 비-2xx 발생 빈도로 인식하는 "warn 건수"가 늘어날 수 있다.

## 4. `fetchWithTimeout` 시그니처에 선택 인자 `op?` 추가 — 기존 호출자 무영향 근거

```ts
export async function fetchWithTimeout(
  url: string,
  init: RequestInit,
  timeoutMs: number,
  op?: string,      // 신규, 선택적
): Promise<Response>
```

4번째 인자가 **선택적**(옵셔널)이므로 호출부에서 인자를 넘기지 않으면 `undefined`가 되어 집계 키가 종전과 동일한 `method+경로`로 유지된다. TypeScript 컴파일 관점에서 기존 3-인자 호출은 전부 유효하다(`tsc --noEmit` 0 error로 실측 확인, 07 §1).

실제 호출자를 직접 grep해 확인한 결과, `fetchWithTimeout` 호출 지점은 전체 **15곳**이다.

| 파일 | 호출 위치(줄) | `op` 전달 여부 |
|------|--------------|---------------|
| `src/clients/CRpcClient.ts` | 51, 90 | **전달함**(`method`/`'catalog'`) — 이번 변경으로 수정된 2곳 |
| `src/clients/CameraClient.ts` | 76, 94, 145, 163, 182 | 미전달(5곳, 무변경) |
| `src/clients/VpdClient.ts` | 55, 98, 144 | 미전달(3곳, 무변경) |
| `src/clients/LpdClient.ts` | 48, 69 | 미전달(2곳, 무변경) |
| `src/brain/AgentRuntime.ts` | 159, 428 | 미전달(2곳, 무변경) |
| `src/setup/presetProvider.ts` | 55 | 미전달(1곳, 무변경) |

즉 **수정된 곳은 CRpcClient의 2곳뿐**이고, `op`를 전달하지 않는 나머지 호출자는 **13곳**이다. 설계서(05 §1.2, §5)·구현 노트(06 §1)가 "나머지 12개 호출자"로 서술한 것은 실제 grep 결과(13곳)와 1건 어긋난다 — 사소한 계산 착오로 보이며, `op`가 선택 인자라는 무영향 결론 자체에는 영향이 없다(개수가 12든 13이든 "선택 인자이므로 전부 무변경"이라는 근거는 동일하다). 문서화 단계에서 실측치로 정정해 기록한다.

이들 13개 호출자는 코드 변경이 없고, 각자의 집계 키가 `method+경로`(op 없음)로 계속 동작하므로 **기능·테스트 모두 무영향**(07 §3 케이스 19·20, tsc 0, vitest 전체 green으로 확인됨).

## 5. 로그 회전 정책(1d/20M) 무변경 확인

`src/util/logger.ts:71`을 직접 확인:

```ts
const fileStream = createStream(logFileName, { size: '20M', interval: '1d', path: logDir, encoding: 'utf-8' });
```

이번 두 변경 모두 `logger.ts`를 건드리지 않았다. 회전 정책(1일 단위 + 20MB 초과 시 회전, `VITEST`에서는 파일 스트림 미생성)은 **그대로**다. 다만 로그가 5분 요약으로 압축되어 **줄 수 자체는 줄어들지만 각 줄의 필드 수가 늘어(요약 줄에 `win/span/n/ok/err/msAvg/msMax` 7필드 추가)** 바이트 단위 영향은 상쇄될 여지가 있다 — 정량 측정은 이번 범위에서 수행되지 않았다(확인 필요 항목으로 남김).

## 8. 확인 필요 (단정하지 않음)

- `op`가 자유 문자열로 유입되는 두 경로(`POST /viewer/api/rpc`, MCP `unity_rpc`)의 **실사용 빈도**는 이번 범위에서 측정되지 않았다. sweep으로 무한누적은 아니라는 점은 코드로 확인했으나, 운영 중 실제 distinct op 수 추이는 미관측.
- 로그 파일 크기(바이트)에 대한 순증감(줄 수 감소 vs 필드 증가의 상쇄)은 정량 측정되지 않음.
- `flushPacketLog()`를 종료 훅에 연결할지 여부는 설계서·구현 노트 모두 후속 과제로 남겨둠 — 이번 변경 범위 밖.

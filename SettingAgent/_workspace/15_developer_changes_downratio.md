# 15 구현 변경 — 점유영역 번호판 아래쪽 길이 2배 (downRatio 0.30 → 0.60)

> 마스터 지시: "점유영역에서 번호판 아래쪽 길이가 2배가 되면 좋겠다."
> 오케스트레이터 결정: `DEFAULTS.downRatio` **0.30 → 0.60**(2배). `upRatio=0.90` 유지 → 위/아래 비 **3.00 → 1.50**(여전히 위가 김).

## 1. 변경 파일 (3개 — 상수 1개 + 그 표기/핀)

| 파일 | 변경 |
|------|------|
| `web/occupancyRegion.js` | `DEFAULTS.downRatio: 0.30` → **`0.60`**. **이 한 줄뿐** — 다른 파라미터·로직 무변경. |
| `web/occupancyRegion.d.ts` | `downRatio` JSDoc 기본값 표기 `기본 0.30.` → `기본 0.60.` |
| `test/occupancyRegion.test.ts` | **T3** 갱신(제목 `(0.90/0.30)`→`(0.90/0.60)`, 확정값 핀 `0.9/0.3`→`0.9/0.6`, 주석 사유). |

### 누락 확인 (grep)
`downRatio|0\.30|0\.3\b` 를 SettingAgent 전역 `*.{ts,js,d.ts}` 로 grep — `downRatio` 기본값 0.30 을 전제한 코드 지점은 위 3곳이 전부.
그 외 히트는 전부 무관(`app.js:587` 은 채움 `rgba(...,0.30)` 알파, 나머지는 타 테스트의 좌표/임계값 `0.3`).

**정정하지 않은 곳(의도)**: `occupancyRegion.js:88` 주석 `upRatio > downRatio → 위가 김` — 0.90 > 0.60 에서도 **여전히 참**이므로 손대지 않았다(규칙 3: 외과적 변경).
**범위 밖**: `docs/*.md` 의 0.30 서술은 이번 범위 외(별도 처리).

## 2. T3 판별력 보존 (구조 그대로 유지)

지시대로 **2단 구조를 유지**했다 — 완화하거나 한쪽을 지우지 않았다.

```js
// 불변식(R2): 기본값이 어떻게 바뀌든 위가 아래보다 길어야 한다 — 구현을 베끼지 않는 하드코딩 부등식.
expect(ratio).toBeGreaterThan(1);
// 확정 기본값 핀: upRatio 0.90 유지 + 마스터 지시로 downRatio 0.30→0.60(아래쪽 2배) → 비 3.00→1.50.
expect(ratio).toBeCloseTo(0.9 / 0.6, 9);
```

- `> 1` 부등식: 구현 상수를 참조하지 않는 하드코딩 불변식(위가 길다) — 그대로 존치.
- `toBeCloseTo(0.9/0.6, 9)`: 기본값 확정 핀. 리터럴을 유지해 `DEFAULTS` 를 import 로 베끼지 않는다(구현을 그대로 반사하면 판별력 상실).

## 4. 실측 (b) 하단 경계 클램프

**클램프 발동 0건** (17개 인스턴스 전부 — p1 7 + p2 6 + p3 4). 0.30 때와 동일하게 0건이며, **형상이 잘린 인스턴스는 없다 → 전부 평행사변형 유지**.

미클램프 `buildTrapezoid` 결과와 `clampToUnit` 결과의 면적비로 판정. 하단 여유(1 − max y):

| 프리셋 | 최소 여유 인스턴스 | max y | 여유 |
|--------|-------------------|-------|------|
| p1 | idx=1 | 0.8028 | 0.1972 |
| p2 | idx=13 | 0.8008 | 0.1992 |
| p3 | **idx=14** | **0.9239** | **0.0761** ← 전체 최소 |

**계기 판별력 검증(위장 방지)**: "0건" 이 검출기 결함에 의한 공허한 음성이 아님을 확인하기 위해 극단값 `downRatio=3.0` 을 주입 → 10건 발동(p3 idx=14 45.06% 절단, p1 idx=1 27.01%, 정점수 4→5 증가 사례 포함). **검출기는 정상 작동하며, 0.60 의 0건은 실제 음성이다.**

**주의(보고 의무)**: p3 idx=14 의 여유는 **0.0761** 로 가장 얇다. 현재 0.60 에서는 안전하며, downRatio 를 더 키울 경우 이 인스턴스가 첫 절단 후보다.

> **정정(16번 QA 보고서 §4 반증)** — 본 절의 초안은 첫 절단 임계를 "약 0.66"으로 적었으나 **오류였다**. maxY 가 downRatio 에 비례한다고 외삽했으나, `y = c.y + downRatio·bw·v.y ± (bw/2)·u.y` 에서 **`c.y` 항은 downRatio 와 무관**하다. 올바른 식은 `down* = 0.60 + 여유/(bw·v.y)` 이며 QA 의 스윕 실측과 일치한다: **첫 절단 임계는 라이브 입력 0.94 / 본 픽스처 0.96**(둘 다 p3 idx=14). 현재 0.60 의 안전 여유는 초안 주장보다 훨씬 크다. 코드는 무결하며 본 문서의 서술만 오류였다.

## 6. 영향도 요약

- `buildTrapezoid` 기본 호출부(뷰어 `app.js` 오버레이, `computeOccupancyRegions`)는 cfg 를 넘기지 않으므로 **자동으로 새 기본값 적용**.
- `cfg.downRatio` 를 명시 주입하는 호출부는 없음(grep 확인) → 기본값 변경이 유일한 행동 변화.
- 겹침 탐색 결과(globalScale/overlapPairs/regions) 불변 → **R1·R5·R5b 회귀 봉인 전부 유지**.

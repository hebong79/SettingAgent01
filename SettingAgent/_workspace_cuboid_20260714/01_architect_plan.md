# 개정분

## §A. 🔴 진단 — 왜 육면체가 뒤로 밀렸는가

### A-1. H 과소는 tFront 밀림의 **정량 프록시**다 (해석식)

핀홀 + 지면 근사에서, footprint 가 깊이축으로 `Δb` 만큼 **뒤로** 밀리면:

```
h_est  =  h_true − Δb · (d − h_true) / D            D = 나디르로부터 수평거리 ≈ d / tan(tilt)
```
유도: 밀린 base 는 이미지에서 `f·d·Δb/D²` 만큼 **위로** 올라가 (관측 고정인) 마스크 상단까지의 화소 거리를 줄이고, 동시에 미터당 수직 이동량(`∝ 1/D`)이 줄어든다. **전자가 지배한다.**

**리더 실측(H = 1.00~1.22m, 실차 전고 1.445m)을 역산:**

| preset | tilt | d | D | 계수 `(d−h)/D` | H=1.22 → **Δb** | H=1.00 → **Δb** |
|---|---|---|---|---|---|---|
| 1 | 6.86° | 4.97 | 41.3 m | 0.0855 | 2.6 m | **5.2 m** |
| 2 | 7.54° | 4.99 | 37.7 m | 0.0940 | 2.4 m | 4.7 m |
| 3 | 18.76° | 4.95 | 14.6 m | 0.240 | 0.94 m | **1.9 m** |

→ **footprint 가 1~5m 뒤로 밀려 있다.** 차량 길이 L=4.7m 와 같은 자릿수다. **리더 가설의 방향이 맞다.**

### A-2. 오염원은 **둘**이고, **둘 다 단방향(+b)** 이다 ← 처방의 근거

| # | 오염원 | 크기 | b 방향 |
|---|---|---|---|
| **① flank(측면) 접지선** | 마스크 하단 윤곽은 앞범퍼선(b 일정)뿐 아니라 **근접 측면 접지선**(b 가 L=4.7m 에 걸쳐 퍼짐)을 포함한다. 최빈 밴드는 시야각에 따라 **flank 로 끌린다** | 최대 ~L | **+** |
| **② z>0 현(chord)** | 하단 윤곽의 상당수는 **지면이 아니다** — 로커패널·범퍼 하단·언더바디 그림자(z ≈ 0.15~0.25m). **z>0 점을 지면에 역투영하면 카메라에서 멀어지는 쪽으로 밀린다**: `push = D·z/(d−z)` | p1: **1.3~2.2m** / p3: 0.5~0.8m | **+** |
| — | b 를 **작게** 만드는 물리적 오염원 | **없다** — 지면 아래에는 아무것도 없다 | — |
| ⚠️ 예외 | bridge 아티팩트 · 마스크 누출(S-2) | 임의 | **±** |

> **② 는 이 프로젝트가 이미 아는 함정이다** — HANDOFF §2-6: *"높이 z 의 점은 nadir 로부터 `H/(H−z)` 배 바깥으로 밀린다"*. `onPlaceFilter` 가 중심점 규칙을 버리고 접지밴드를 쓴 이유가 바로 이것이다. **접지선에서 같은 함정이 재발했다.**
> 리더가 지목한 **F-1 폭 +18% 과대**의 뿌리도 동일하다(현 위의 점은 폭 방향으로도 나디르에서 바깥으로 밀린다).

### A-3. ∴ 처방은 하나 — **최빈 밴드가 아니라 근접 모서리(near-edge)**

오염이 **전부 단방향(+b)** 이므로 참된 접지는 **b 의 최솟값 쪽 구조**에 있다. **최빈·중앙값은 오염을 평균해서 삼킨다.**
(bridge/누출만 양방향 → **최솟값이 아니라 로버스트 하위 분위수**를 써야 한다.)

**리더의 3개 후보는 전부 옳고 서로 배타적이지 않다 — 셋을 합친다**(§B).

### A-4. ★ 가설을 가르는 **순수 수치 판정** (스샷 없이 즉시 — 요구 #1)

두 오염원은 **프리셋 의존성이 정반대**다:

| 오염원 | H 오차 | 프리셋 의존 |
|---|---|---|
| **② z>0 현** | `Δb·(d−h)/D` 이고 `Δb = D·z/(d−z)` → **`z·(d−h)/(d−z)`** | **D 가 소거된다 → 거리·프리셋 무관 상수.** z=0.2m → **0.148m 고정** |
| **① flank** | `Δb_flank·(d−h)/D`, `Δb_flank ≈` L 의 일정 분수(상수 m) | **D 에 반비례 → preset3 가 preset1 보다 2.8배 나쁘다** |

> **판정법 — H 오차를 프리셋별로 표로 내라.**
> - preset3 의 H 오차가 preset1 보다 **뚜렷이 크다** → **① flank 지배 (리더 가설 확정)**
> - 전 프리셋 **≈0.15m 균일** → **② z-chord 지배**
> - 둘 다 → 둘 다.
>
> **설계자 예측**: 관측 오차 폭이 **0.22~0.44m** 인데 ② 단독은 **0.148m 상수**밖에 못 만든다 → **① 이 지배한다. 리더 가설이 주범이고 ② 는 잔여 바닥(floor)이다.**

**스샷 판정(요구 #1 원안)** — 한 대에 겹쳐 그린다: 유효 접지열(빨강) · 기각열(회색) · **b 히스토그램**(앞범퍼 봉우리 vs flank 꼬리) · **현 tFront(최빈) 재투영 앞선 = 마젠타** · **신규 tFront(near-edge) 재투영 앞선 = 초록**. 초록이 바퀴에 붙고 마젠타가 뒤에 있으면 확정.

## §B. `fitContactLine` v2 — 근접 앞범퍼선 (요구 #2)

리더 후보 3개를 **전부 흡수**한다: 하위 분위수(near-edge) + u-스팬 검증(앞범퍼는 u 로 길다) + 앞범퍼군/flank군 분리.

```ts
// src/ground/contact.ts — LineFit 교체
export interface LineFit {
  tFront: number;      // 앞선 깊이축 좌표(m)
  aCenter: number;     // 앞선 폭축 중심 — buildFootprint 가 그대로 쓴다
  frontSpanM: number;  // 앞선 폭축 스팬 = W 관측치
  frontCount: number;
  frontMadM: number;   // 앞선 밴드 내 b 의 MAD — 직선성(= bridge 게이트)
  totalCount: number;
}
```

```
1. bNear = quantile(b, Q_NEAR = 0.05)                    // 하위 5% — bridge/누출 방어(최솟값 금지)
2. frontBand = { p : p.b ≤ bNear + FRONT_BAND_M(0.5) }   // ★ 근접 모서리 밴드

3. ★ 앞범퍼 검증 — 앞범퍼는 u(폭축)로 길고 flank 는 w(깊이축)로 길다
   frontSpanM = robustSpan_a(frontBand)
   frontSpanM < MIN_FRONT_SPAN_M(1.2) → **육면체 미산출**
     issue: "앞범퍼 접지선 미검출(폭 스팬 {n}m) — flank 만 보임 / 원경 / 가림 과다"

4. frontMadM = MAD(frontBand.b)
   frontMadM > FRONT_MAD_MAX_M(0.20) → **육면체 미산출**
     issue: "앞선 잔차 {n}m — 마스크 파편화(bridge) 의심"

5. frontCount < MIN_FRONT_COLS(8) → 미산출

6. tFront = median(frontBand.b)      aCenter = median(frontBand.a)
```

### B-1. `inlierRatio` 를 bridge 게이트로 쓰면 **안 된다** (v1 의 숨은 오류)

v1 은 `inlierRatio < 0.6` 을 파편화 게이트로 썼다. **near-edge 로 바꾸면 이 비율이 시야각의 함수가 된다** — flank 가 많이 보이는 차량은 앞선 밴드 비율이 자연히 낮아진다. 그대로 두면 **정상 차량을 파편화로 오진**한다.
→ bridge 게이트를 **`frontMadM`(앞선 밴드 내부의 직선성)** 으로 옮긴다. **비율이 아니라 잔차로 잰다.**

### B-2. `buildFootprint` 변경 (최소)

`fit.inliers` median 대신 **`fit.aCenter` / `fit.frontSpanM`** 사용. **F-1 폭 클램프·`wSource` 강등은 유지**(현 위의 점은 폭도 부풀리므로 클램프는 여전히 필요).

### B-3. ⚠️ 잔여 편향 — 은닉하지 않는다

1. **near-edge 의 정체가 tilt 에 따라 바뀐다** — 앞바퀴 접지면(z=0)이 최소 b 인 경우(얕은 tilt)와, 앞범퍼 하단(z≈0.25m, 0.9m 앞으로 돌출)의 겉보기 지면점이 최소 b 인 경우(급 tilt)가 갈린다. 산술상 둘의 차는 **≤ 0.3m** — 안정적이다.
2. **z>0 잔여 바닥 ≈ 0.15m H 오차**(§A-4 ②).

→ **지금은 보정하지 않는다.** `CONTACT_Z_OFFSET_M` prior 는 **G2a 에서 계통 잔차가 실제로 남을 때만** 넣고, 넣으면 반드시 `CuboidSource` 에 노출한다(**숨은 fudge 금지**). → **Q8.**

## §C. 성공 기준 교체 — IoU 폐기 (요구 #3)

## §D. 주차면 필터(모드 A) — 정식 파이프라인 단계 (요구 #4)

**누락이었다.** 없으면 원경·통행 차량이 앵커 지표를 오염시킨다(리더 실측: 슬롯중앙거리 **16m** 차량 혼입). 필터를 거니 p3 기준값 **−0.009** 로 정상화.

```
[0.5] filterVehiclesOnPlace(vehicles, slotPolys)   ← src/capture/onPlaceFilter.ts **재사용**(신규 기하 0줄)
      · 판정: bbox 하단 25% 접지밴드 ∩ 슬롯 폴리곤 면적비 ≥ 0.15, 전 폴리곤 OR
      · segment() 가 rect + mask 를 함께 주므로 **기존 rect 필터가 그대로 걸린다**
      · 강등: 폴리곤 부재/0개 → 전량 통과 + degraded warn (기존 정책 동일 — 드롭 금지)
      · 라우트 쿼리 `onPlace` (기본 1). 0 이면 모드 B(전량)
      · summary.filteredOut / summary.onPlaceDegraded 노출
```

파이프라인: `[0] segment → [0.5] 주차면 필터 → [1] 하단윤곽 → [2] 가림배제 → [3] 역투영 → [4] 슬롯축 → [5] 앞선적합 → [6] footprint → [7] H → [8] 육면체`

> ⚠️ **구현 주의**: **[2] 가림 배제의 "다른 차량"은 필터 *전* 전체 검출을 쓴다.** 필터로 제외된 차량도 **가리기는 한다**(파이썬 원형 `contact_filtered.py` 의 `others` 가 `cand` 전체를 쓰는 것과 동일). 필터 후 집합으로 가림을 판정하면 **가림이 조용히 누락**된다.

## §E. 앵커 지표에 미치는 영향 — **설계 무변경, 기준값만 재측정**

**§6 은 한 줄도 안 건드린다.** 다만 정직하게 기록한다:

- `tFront` 정의가 앞으로 1~5m 이동하면 **`depthDevM` 의 기준값(baseline)이 그만큼 이동**한다.
- **Δ 반응(−2.500 / +1.250)은 불변** — 앵커는 **차분량**이고 tFront 의 계통 편향은 **상수 오프셋**으로만 들어간다.
- ∴ **임계(`DEPTH_DEV_M`)만 G6 에서 재측정.** 지표의 원리·구조·테스트는 그대로.
- v1 §6-4 주석이 이미 예고했다: *"정상 상태의 값은 0 이 아니라 차량이 슬롯 앞선에서 물러난 실제 거리(물리량)다"*.

**앵커 회귀 게이트**: Loop 2 이후에도 **깊이 +2.5m → Δ=−2.500 / 폭 +1.25m → Δ=+1.250** 이 유지되어야 한다. 깨지면 `contact.ts` 변경이 앵커를 손상시킨 것이다.

## 3. 관측 vs prior 경계 — 타입 강제 ✅ 유지

카메라는 **카메라 쪽 접지 윤곽**만 본다. 뒤·반대편 접지선은 **자기 차체**가 가린다.

| 자유도 | 출처 |
|---|---|
| 위치 X, Y | **관측** — 접지 앞선 역투영 |
| yaw | **슬롯 prior** |
| **L** | **prior (항상)** — 원리적으로 관측 불가 |
| W | 관측 / prior 강등(F-1 클램프) |
| H | 관측 / prior 강등 |

```ts
export interface CuboidSource {
  position: 'observed';
  yaw: 'slot-prior';
  L: 'prior';                  // ★ 유니온 아님 — 타입이 먼저 막는다
  W: 'observed' | 'prior';
  H: 'observed' | 'prior';
}
```

## 6. ★ 앵커 지표 3종 — **실증 완료, 변경 금지**

`depthDevM`(비주기 → 임의 밀림에 선형) · `phaseDevM`(주기 2.5m → 비정수배만) · `unmatchedRate`(advisory).

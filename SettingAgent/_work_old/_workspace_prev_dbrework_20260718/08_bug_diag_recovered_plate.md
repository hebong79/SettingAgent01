## 1. Q1 — `recovered:true` 는 정확히 어떤 절차로 만들어지는가?

**줌 재시도다. 이웃 추정이 아니다.** 유일한 생성 지점은 `src/capture/detectPipeline.ts:319`.

```
detectPipeline.ts:299  for (let i = 0; i < vehicles.length; i++)
             :302        const baseQuad = matched.get(i);        // base 매칭 성공 → recovered:false (:304)
             :309        for (let a = 0; a < cfg.zoomFactors.length; a++)   // 미귀속 차량만 진입. zoomFactors=[2,3,4,5] (:114)
             :310          vehicleCenterZoomPtz(v.rect, basePtz, { frontBias: 0.62, zoomFactor: ... })
             :312          view = await deps.camera.requestImage(cam, preset, {pan,tilt,zoom})
             :314          pick = pickNearestPlate(await deps.lpd.detect(view.jpg), { x:0.5, y:0.5, w:0, h:0 })
             :316          recQuad = inverseProjectQuad(pick.quad, viewPtz, basePtz, fovOpts)
             :318          clamped = clampQuadCenterToRect(recQuad, v.rect, 0.62)
             :319          plate = { quad: clamped, confidence: pick.confidence, recovered: true, attempts: a+1 }
```

절차: **미귀속 차량의 앞쪽(frontBias 0.62) 을 화면 중심으로 당겨 zoomFactor 배 확대 → 그 뷰에서 LPD 재검출 → 뷰 중심(0.5,0.5)에 가장 가까운 판 1개 채택 → base 좌표로 역투영 → 차량 rect 로 중심 클램프.**

`grep -rn "recovered" src/` 실행 결과 `recovered:true` 를 세우는 지점은 `:319` **단 한 곳**이다. 이웃 기반 추정(`estimatePlateQuadFromNeighbors`)은 `floorRoi.ts:71` 에 존재하나 사용처는 `Finalizer.ts:317`·`FloorRoiReviewer.ts:69` 뿐 — **`detectPipeline` 에서 import 조차 하지 않는다**(검출 경로 무관).

## 2. Q2 — 귀속 판정 기준은? 왜 veh0 의 판이 veh6 에 붙었는가?

### 2-1. 두 개의 서로 다른 귀속 기준이 있다

| 경로 | 기준 | 위치 |
|---|---|---|
| **base 매칭** | 번호판 bbox 중심 ∈ 차량 rect(`containsPoint`) → 후보 중 **교집합 면적 최대** | `plateMatch.ts:26-31` |
| **줌 재시도(recovered)** | **뷰 중심(0.5,0.5) 최근접** — rect 포함도 IoU 도 **보지 않는다** | `detectPipeline.ts:314` |

veh6 의 판은 **후자**로 결정됐다. 즉 "veh6 의 rect 안에 있어서" 붙은 게 아니라 **"veh6 을 향해 줌한 뷰의 중심에 제일 가까워서"** 붙었다.

### 2-2. 실좌표 재현 (p1, `matchPlatesToSlots` 원본 알고리즘 재실행)

> `summary.lpdCount=6` · `lpdFilteredOut=0` · `plates.length=6` → **`detect.plates` 6건 = `platesBase` 원본 그대로**. 따라서 아래는 근사가 아니라 **base 매칭의 정확한 재현**이다.

```
plates[0] c=(223,759)  cand=[veh1:1583.8]                 -> veh1
plates[1] c=(1782,649) cand=[veh4:883.7]                  -> veh4
plates[2] c=(525,742)  cand=[veh5:1284.7]                 -> veh5
plates[3] c=(820,724)  cand=[veh3:1003.9]                 -> veh3
plates[4] c=(1559,674) cand=[veh4:847.7, veh6:847.7] ★동률 -> veh4  → 이후 폐기
plates[5] c=(1348,678) cand=[veh0:1055.2, veh6:1055.2]★동률 -> veh0
matched slots: 0,1,3,4,5      UNMATCHED(재시도 진입) = 2, 6
```

**veh6 은 두 판의 후보였으나 둘 다 뺏겼다.** 그래서 미귀속 → 재시도 → veh0 의 판 회수. 전체 인과 사슬:

1. **`intersectionArea` 가 포화한다** (`plateMatch.ts:27`). 번호판 bbox 가 두 차량 rect 에 **완전히 포함**되면 교집합 = 번호판 면적 = **양쪽 동일**. plates[4] 는 veh4·veh6 모두에 대해 정확히 847.7px² — **완전 동률**. 즉 면적 기준은 이 상황에서 **판별력이 0** 이다.
   - veh4.rect(x 1457~1842, y 483~693) ∋ (1559,674) ✔ / veh6.rect(x 1315~1619, y 527~710) ∋ (1559,674) ✔
2. **동률을 인덱스 순서가 깬다** (`plateMatch.ts:28`, `overlap > bestOverlap` 의 **strict `>`**). 먼저 순회된 veh4(4 < 6) 가 승자. **기하적 근거 0 — 배열 순서가 결정한다.**
3. **차선 슬롯으로의 폴백이 없다** (`plateMatch.ts:35`). veh4 는 이미 plates[1](1782)을 883.7px² 로 보유 → `847.7 > 883.7` = **false** → plates[4] 는 `result` 에도 못 들어가고 **차선 후보 veh6 에게 넘어가지도 못한 채 통째로 소멸**한다.
   - **순서 무관하게 동일**: plates[4] 가 먼저 처리돼도 `result[4]=1559` 설정 후 plates[1] 이 883.7 > 847.7 로 **덮어써** plates[4] 는 역시 소멸. 구조적 결함이지 순서 의존이 아니다.
4. `detectPipeline.ts:302` `matched.get(6)` = undefined → veh6 이 재시도 루프(:309) 진입.
5. `pickNearestPlate` 가 veh0 의 판을 채택 → `:316` 역투영 → base (1367,676).
6. **`clampQuadCenterToRect`(:318) 가 발동하지 않는다** — (1367,676) 은 이미 veh6.rect(1315~1619, 527~710) **내부**라 무변경(`detectMath.ts:120` early return). **유일하게 존재하는 방어 장치가, 훔친 판이 피해 차량 rect 안에 있다는 이유로 침묵한다.**
7. `:319` → `recovered:true, attempts:1`, 중심 (1367,676) — veh0 의 판(1348,678)과 **18.8px**.

## 3. Q3 — 중복 귀속 방지 장치가 있는가?

**없다. 코드베이스 전체에 존재하지 않는다.**

- `matchPlatesToSlots` 는 base 매칭 **내부에서만** 두 불변식을 보장한다: 번호판당 슬롯 ≤1(판별 argmax) · 슬롯당 번호판 ≤1(`bestArea`, :35).
- 그러나 **줌 재시도 경로(`:309-323`)는 이 불변식 밖에 있다.** `matched` 를 **한 번도 참조하지 않고** 역투영으로 **새 quad 를 생성**한다. 따라서 "이 판은 이미 veh0 것"이라는 사실을 알 수단이 없다.
- 좌표 동등성으로도 못 잡는다: 회수 quad 는 역투영 오차로 원본과 **수치가 다르다**(1367 vs 1348) → `07` 문서의 `quadKey`(4점 수치 직렬화) 방식으로는 **중복이 탐지되지 않는다**. 거리 임계가 필요하다.
- 검색 근거: `grep -rniE "estimatePlateFromNeighbors|중복 귀속|dedup|alreadyUsed|usedPlate|takenPlate" src/ web/ test/` → 유효 히트 **0**(`dedupe*` 는 전부 LLM Stage2 슬롯 중복제거로 무관).

### 결함인가 의도인가 → **결함(미검토 가정)이다**

| 근거 | 내용 |
|---|---|
| 설계 주석 | `detectPipeline.ts:2` "미귀속 차량 zoom 재시도(최대 4회) → 역투영 복원" — **줌한 뷰의 중심 근처 판 = 그 차량의 판**이라는 가정이 **명시되지도 검토되지도 않았다**. |
| 테스트 | `test/plateMatch.test.ts` 4건 중 **번호판 중심이 두 차량 rect 에 동시에 드는 케이스는 0건**. 슬롯들이 전부 비겹침(`slot(1,0.1,0.1)`/`slot(2,0.6,0.1)`)이거나 단일 슬롯. **결함 영역이 통째로 미커버**. |
| 유일한 관련 봉인 | `test/plateMatch.test.ts:48` "한 슬롯에 번호판 여럿이면 겹침 큰 것 유지" — **단일 슬롯** 경합이라 폐기가 **정답**인 케이스(차 1대 = 판 1개). 이 테스트는 **다중 슬롯 경합의 폐기를 정당화하지 않는다.** |
| 폐기 동작의 인지 여부 | `test/lpdFilterRegression.test.ts:325,377` 이 "상류 `matchPlatesToSlots` : 검출 1건 단위 경쟁 → **진 번호판은 드롭된다**" 를 **명시적으로 서술**한다. → **드롭 자체는 인지된 성질**이나, 그 문맥은 *한 차량 rect 안의 오검출 노이즈 제거*(:384 "손실이 아니라 잡음 제거")다. **차선 차량이 존재하는 경우는 고려된 흔적이 없다.** |

즉 **"드롭"은 의도, "차선 슬롯 폴백 부재"는 미검토 결함**이다.

## 4. Q4 — veh6 의 진짜 판(1559,674)이 standalone 으로 남은 이유 ★핵심

### 4-1. 질문의 전제 교정

> "attach 로직이 rect 포함 기준이면 1559 는 veh6.rect 안이므로 붙었어야 한다"

**포함은 필요조건일 뿐 충분조건이 아니다.** 1559 는 veh6.rect 안에 **있고**, 동시에 **veh4.rect 안에도 있다**(x 1457~1842). `matchPlatesToSlots` 는 rect 포함으로 **후보를 모은 뒤**(`:26`) **단 하나의 승자만 남기므로**(`:28`), 포함 사실만으로는 귀속되지 않는다.

### 4-2. 실제 이유 (§2-2 의 3단계)

```
plates[4](1559,674) → 후보 {veh4:847.7, veh6:847.7}  … 완전 동률(포화)
                    → 인덱스 순서로 veh4 승리          … plateMatch.ts:28 strict >
                    → veh4 는 이미 plates[1](1782)@883.7 보유
                    → 847.7 > 883.7 = false            … plateMatch.ts:35
                    → **폐기. veh6 으로 넘어가지 않음.**
```

→ 그 결과 `matched` 에 없음 → `filterPlatesOnPlace` 의 (A)귀속 분기(`onPlaceFilter.ts:88`)에서 탈락 → **(B)주차면 분기(`:89-90`)가 살려서** `plates[]` 에 **standalone 으로 잔류**. (`lpdFilteredOut=0` 실측과 정합 — 6건 전부 잔류.)

**즉 "standalone 으로 남은 것"은 (B) 분기가 구제한 결과이고, 원인은 (A) 에서 폐기된 것이다.** 아이러니하게도 veh6 의 진짜 판은 `plates[]` 안에 **멀쩡히 살아 있는데도** 아무 차량에도 못 붙었다.

### 4-3. 줌 재시도는 왜 이 판을 못 찾았나 — **여기가 유일한 미확정 고리**

이상 기하 모델(fovBaseV=34.6348, aspect=16/9, basePtz={22, 6.8, 1.69341}, zoomFactor=2)로 veh6 재시도 뷰를 재현:

```
veh6 타깃(frontBias 0.62) = base (1467, 640)px
명령 PTZ: pan=31.572° tilt=8.741° zoom=3.3868   (뷰 fovH=18.590° fovV=10.520°)

  veh6 자기 판(1559,674) → view(0.593, 0.562)  중심거리 d=0.1119  ← 더 가깝다
  veh0 의   판(1348,678) → view(0.379, 0.569)  중심거리 d=0.1391
  관측된 회수점(1367,676) → view(0.398, 0.565)  중심거리 d=0.1207
  veh4 의   판(1782,649) → view(0.820, 0.516)  중심거리 d=0.3203
```

**이상 모델대로면 `pickNearestPlate` 는 veh6 의 자기 판(d=0.1119)을 골랐어야 한다.** fovBaseV 민감도 스윕(30·32·34.6348·36·38·40·42) 전 구간에서 **순위가 뒤집히지 않는다** → **기하 모델 오차로는 관측을 설명할 수 없다**(`fovBaseV` 폴백 강등 가설 **기각**).

관측된 회수점은 view(0.398, 0.565), d=0.1207 — **자기 판(0.1119)보다 멀다**. `pickNearestPlate` 는 최근접을 고르므로(`controlMath.ts:27` `d < bestD`), 자기 판이 후보에 있었다면 **반드시 그것이 선택된다**. 선택되지 않았다는 관측으로부터 역산하면:

> **줌 뷰의 LPD 검출 결과에 veh6 의 자기 판이 없었다.** (뷰 안 좌표 0.593,0.562 로 화면 중앙부에 멀쩡히 들어옴에도 불구하고.)

**이것은 관측으로부터의 연역이지 직접 실측이 아니다** — 확인하려면 라이브 LPD 로 해당 줌 뷰를 재현해 검출 목록을 찍어야 한다(동결 데이터에 줌 뷰 응답이 없어 오프라인 확인 불가). **못 찾은 범위를 명시한다: "줌 뷰에서 LPD 가 왜 그 판을 놓쳤는가"는 미확정(LPD 모델 거동 — 고배율에서 판이 과대해지는 스케일 이탈이 유력하나 근거 없음).**

**단, 이 미확정 고리는 근본 원인 판단에 영향이 없다.** veh6 이 재시도에 들어간 것 자체가 §4-2 의 폐기 때문이며, 폐기가 없었으면 재시도도 LPD 미검출도 애초에 발생하지 않는다.

## 5. Q5 — 발현 조건 / p2·p3 에서 안 터진 이유

### 5-1. rect 겹침은 발현 조건이 **아니다** (실측 반증)

```
p1 rect 겹침쌍: 0-2:14256  0-6:17385  1-5:753  2-3:4836  3-5:6699  4-6:26892 px²
p2 rect 겹침쌍: 0-3:29184  1-2:10535  1-4:41170 1-5:19885 2-5:41949 3-4:24723 px²
p3 rect 겹침쌍: 0-1:25704  0-2:94770  0-3:98560 1-3:88220 px²   ← 압도적으로 크지만 결함 0
```

**p3 의 겹침이 가장 큰데 결함이 없다.** 조건은 rect 겹침이 아니라 **번호판 *중심* 이 2개 이상 차량 rect 에 드는 것**이다(p3 는 4개 판 전부 후보 차량이 **정확히 1개** → 폐기 0 → 미귀속 0 → 재시도 0).

### 5-2. 발현에는 **두 조건이 동시에** 필요하다

| | 조건 | p1 veh6 | p2 veh5 | p3 |
|---|---|---|---|---|
| **D1** | 판 중심이 ≥2 rect 에 듦 → 동률/argmax 승자가 이미 더 큰 판 보유 → **폐기 → 피해 차량 미귀속** | ✔ 발현 | ✔ **발현(잠복)** | ✘ |
| **D2** | 재시도 줌 뷰에서 LPD 가 **자기 판을 놓치고** 이웃 판이 시야에 있음 | ✔ 발현 | ✘ | — |
| | **관측 증상** | **오귀속** | 없음(운으로 은폐) | 없음 |

**p2 에서도 D1 은 똑같이 터졌다** (실측):
```
p2 plates[5] c=(408,636) cand=[veh2:945.4, veh5:945.4] ★동률 → veh2(인덱스 순) 승
             veh2 는 이미 plates[2](183,597)@983.9 보유 → 945.4 > 983.9 = false → 폐기
             → veh5 미귀속 → 재시도 → recovered (393,630)
```
그런데 p2 의 재시도는 **자기 판을 되찾았다** — 회수점 (393,630) 은 veh5 의 진짜 판 (408,636) 과 **16px**. 즉 **p2 는 결함이 없는 게 아니라, D2 가 안 터져서 증상이 은폐된 것**이다. 재시도 뷰 마진도 p1 과 대동소이(p2 veh5 자기판 d=0.1166 vs 2위 0.1480, **마진 0.0314** / p1 veh6 자기판 d=0.1119 vs 2위 0.1391, **마진 0.0272**) → **기하 조건은 사실상 같고, 갈린 것은 LPD 의 줌 뷰 검출 여부뿐**이다.

> **⚠️ 이것이 이 진단의 가장 중요한 함의다.** p2 는 현재 **운으로 정답**이며, 같은 장면에서 LPD 가 한 번 흔들리면 **즉시 p1 과 동일한 오귀속으로 전환**된다. "p2/p3 는 정상"이라는 07 문서 §6 의 서술은 **증상 기준으로는 참, 결함 기준으로는 거짓**이다.

### 5-3. 대조군 — 재시도가 정상 작동한 사례 (p1 veh2)

p1 veh2 도 미귀속(`recovered:true, attempts:1`, 회수점 (1092,705))이지만 **성격이 완전히 다르다**: veh2 는 **어떤 판의 후보도 아니었다**(`detect.plates` 어디에도 veh2.rect(869~1150, 550~749) 안에 드는 중심이 없다). 즉 **base 프레임에서 LPD 가 그 판을 아예 못 봤고**, 줌 재시도가 **신규 검출**로 되살렸다(회수점 1092 는 `detect.plates` 6건 중 어느 것과도 무관 = 진짜 신규). → **재시도 기능 자체는 설계대로 작동하며 실제 가치가 있다. 제거 대상이 아니다.**

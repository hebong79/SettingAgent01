## 5. 근본 원인 단정

원인은 **1개가 아니라 1주 + 2종속**이다.

### ① 주 원인 — `SettingAgent/web/core.js:474-485` `computeOccupancy`, 특히 **480행**
```js
const hit = cands.find((c) => pointInQuad(c.center.x, c.center.y, f.quad));
```
plate↔슬롯 매칭 기준이 **"번호판 중심점이 슬롯 바닥 폴리곤 내부"** 라는 단 하나뿐이다.
번호판은 지면 위로 떠 있고 바닥 ROI 는 지면 발자국이므로, 원경으로 갈수록 커지는 시차 오프셋(실측 10~33px)이 얇은 원경 폴리곤(y두께 27~37px)을 넘긴다.
- 결과 A: 열 중간 plate 가 **이웃 슬롯으로 오귀속**(라벨 한 칸 밀림).
- 결과 B: **열 끝 plate 는 받아줄 다음 슬롯이 없어 매칭 실패 → 결과에서 조용히 소실**(479행 `floorPolygons.map` 은 슬롯 기준 순회라, 어떤 슬롯에도 안 든 plate 를 보고할 자리가 없다).
→ **이것이 "마지막 사다리꼴 1개 부족"의 근본 원인.**

### ② 직접 발현 지점 — `SettingAgent/web/app.js:371-372`
```js
const region = computeOccupancyRegions(
  rows.filter((o) => o.source === 'plate' && o.plateQuad).map((o) => ({ idx: o.idx, quad: o.plateQuad })),

```
사다리꼴 모집단이 **슬롯에 매칭된 plate 행**뿐이다. ①에서 소실된 plate 는 `rows` 에 행이 없으므로 **사다리꼴이 생성될 기회 자체가 없다**. (`computeOccupancyRegions` 자체는 무결 — 입력 100% 반환, 탈락 0, overlap 0.)

### ③ 구제 실패(폴백이 못 막는 이유) — `SettingAgent/web/occupancy.js:165` 와 `188`
```js
// 165행
if (c && floorPolygons.some((f) => pointInQuad(c.x, c.y, f.quad))) continue;
// 188행
if (rows[pos].occupied) continue;
```
- **165행**: plate 중심이 *다른* 슬롯 안에 들면 그 차량은 bbox 폴백 후보에서 통째 제외 → p1 veh0(실제 슬롯 5)은 plate 가 슬롯 6 에 들어 제외 → **슬롯 5 는 차가 있는데 plate 도 bbox 도 못 받아 `source=null, occupied=false`**(화면에 아무 표시 없음).

- p3 만 우연히 열 끝 슬롯(17)이 plate 로 점유되지 않아 bbox 폴백이 걸렸다(H5).

## 6. 수정 방향 제안 (구현 금지 — 방향과 부작용만)

> 공통 전제: **인덱스 체계는 건드릴 것이 없다(H1 기각).** 고칠 것은 **plate↔슬롯 기하 매칭**이다.

### 방향 A — 매칭 기준을 plate 중심 → **차량 bbox 접지중심**으로 (권장도 높음)
`computeOccupancy` 의 매칭 앵커를 번호판 중심 대신 차량 bbox 접지중심(`x+w/2`, `y+h`)으로 바꾸고, 사다리꼴 축 소스로만 plate quad 를 쓴다.
- 근거: §2 에서 **접지중심은 3개 프리셋 16대 중 13대가 자기 슬롯에 정확히 들었다**(빗나간 3건도 슬롯 경계 0~37px 근접이며 최근접 슬롯이 정답). 접지중심은 지면 위 점이라 바닥 ROI 와 **같은 평면** → 시차 오프셋이 원리적으로 0.
- 영향 범위: `core.js:computeOccupancy` 시그니처가 `plates[]` → `detect{vehicles,plates}` 로 바뀜 → 호출처 `occupancy.js:149`(judge 1단계), `core.js:608`(`buildFlatSlotRows`) 동시 수정 필요. `test/` 의 computeOccupancy 유닛테스트 전면 갱신.
- 부작용: standalone `plates[]`(차량에 귀속 안 된 번호판)은 접지점이 없어 매칭 불가 → 차량 미검출 + 번호판만 검출된 케이스의 점유를 잃는다(현 데이터엔 해당 없음 — 전 차량이 plate 보유). bbox 폴백(`occupancy.js` 2단계)과 판정 로직이 중복되므로 **폴백 설계 재검토 필요**(폴백이 사실상 1단계에 흡수됨).

### 방향 B — plate 중심 매칭 유지 + **최근접 슬롯 스냅**(폴리곤 밖이면 최근접 슬롯에 귀속, 임계 거리 이내)
`pointInQuad` 실패 시 최근접 슬롯 폴리곤까지 거리를 재고 임계(예: 25px 정규화 환산) 이내면 그 슬롯으로 귀속.
- 근거: 소실된 3장의 최근접 거리는 11/15/20px 로 슬롯 간격(230~285px) 대비 매우 작다.
- 영향 범위: `core.js` 에 점→폴리곤 거리 함수 신규(현재 없음 — `occupancy.js` 의 clip 계열로는 대체 불가). `occupancy.js:165` 의 "다른 슬롯에 들었나" 판정도 같은 기준으로 맞춰야 정합.
- 부작용: **오귀속(§2 의 ★어긋남 5건)은 전혀 못 고친다** — 그 plate 들은 이미 이웃 슬롯 *안*에 들어 있어 스냅이 개입하지 않는다. 즉 "마지막 1개 없음"은 없어져도 **라벨 한 칸 밀림은 남는다**. 임계값이 새 튜닝 파라미터로 추가됨(빈 슬롯에 잘못 스냅할 위험).

### 방향 C — 바닥 ROI 폴리곤을 **번호판 높이 평면으로 보정**(호모그래피/수직 오프셋)
슬롯 폴리곤을 지면 발자국 대신 "번호판 높이"로 올린 사본을 만들어 매칭에 쓴다.
- 영향 범위: 카메라 캘리브레이션/호모그래피 정보 필요 → `PtzCamRoi.json` 스키마 또는 신규 파라미터 추가. 프리셋별 PTZ 마다 다른 보정값 필요.
- 부작용: 가장 큰 변경. 프리셋마다 보정 튜닝이 필요하고 zoom 변화에 취약. **현 시점 과설계로 판단** — A 로 해결되면 불필요.

### 방향 D — ③ 폴백 구제 완화 (`occupancy.js:165`/`188`)
165행의 "다른 슬롯에 들면 후보 제외"를 "**자기 접지 슬롯이 아닌 슬롯에 들었으면 제외하지 않음**"으로, 188행을 "plate 점유라도 그 plate 가 *이 슬롯 차량의 것*일 때만 유지"로 좁힌다.
- 부작용: A/B 없이 D 만 하면 **증상이 주황 '번호미인식' 원으로 바뀔 뿐 사다리꼴은 여전히 안 생긴다**(사다리꼴은 plate 축이 필요 — `app.js:372`). 근본 해결 아님. **단독 채택 비권장**, A 채택 시 폴백 정합을 위해 동반 검토.

### 권장
**A(주) + D(폴백 정합)**. A 는 오귀속(★어긋남)과 소실(★★) 을 **한 원인에서 동시에** 없앤다 — 접지중심이 바닥 ROI 와 동일 평면이라는 기하적 근거가 실측으로 뒷받침된다(16대 중 13대 정확 일치, 나머지도 최근접 슬롯이 정답).
B 는 증상(마지막 1개)만 가리고 라벨 밀림을 남기므로 단독으로는 불충분하다.

## 7. 배제된 범위 (재조사 불요)

- 전역 인덱스 체계(`normalizeGlobalIdx` / `globalIdxParity` / 1-based 규약) — 무관. 파일 idx 1..17 무결.
- `selectFloorRoi` / `normalizePtzCamRoi` — 무손실 확인(7/6/4).
- `computeOccupancyRegions` / `plateAxes` / `buildTrapezoid` / `clampToUnit` — 무결. 입력 items 100% 반환, 탈락 0건, `overlapPairs=[]`, `globalScale=4`(최대 배율에서도 겹침 없음 → 축소·클램프로 인한 소실 아님).
- `onPlaceFilter` / `placeRoi` 가장자리 필터 — 뷰어 경로에 존재하지 않음.
- `currentFrameKey` / `detectByKey` 키 조회 — 프리셋별 검출 매칭 정상(라이브 p1/p2/p3 각각 올바른 프레임).

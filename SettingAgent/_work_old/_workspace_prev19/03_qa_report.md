## 1. 검증 방법

"코드가 존재함"이 아니라 "정상 동작함"을 증명하기 위해, **구현자 보고서를 믿지 않고 재실측**했다.

| 방법 | 목적 |
|---|---|
| **뮤테이션 검사(6종)** | 지표·임계·해소로직을 무력화했을 때 테스트가 **실제로 실패하는가**(우연한 통과 배제) |
| **독립 기하 구성** | 두 지표의 사각을 구현자 표에 의존하지 않고 **직접 구성**해 탐색(지면 왕복 변환) |
| **런타임 데이터 파괴 시뮬레이션** | "사용자가 앱을 써도 안 깨진다"를 **직접 실증** |
| **정적 결선 감사** | 신규 UI id ↔ 핸들러 대응, dangling 0건 |
| **전수 grep** | C3(실카메라 호환) 준수 여부, 런타임 값 단정 테스트 잔존 여부 |

## 2. ★ 최상위 결함 — 두 정합 지표는 **상보적이지 않다** (공통 사각 존재)

### 2-1. 검출력 자체는 진짜다 (뮤테이션으로 실증)

먼저 공정하게: 구현자가 추가한 두 지표는 **우연히 통과하는 테스트가 아니다.** 무력화하면 실제로 실패한다.

| # | 뮤테이션 | 결과 |
|---|---|---|
| M1 | `ROI_MISALIGN_ERR` 0.008 → 0.5 (가로 임계 무력화) | **FAIL** ✅ 검출 |
| M2 | `TILT_MISALIGN_DEG` 1.0 → 90 (세로 임계 무력화) | **FAIL** ✅ 검출 |
| M3 | `tiltErrDeg` 를 항상 `null` (세로 지표 제거) | **FAIL** ✅ 검출 |
| M4 | `metricErr` 를 항상 `0` (가로 지표 제거) | **FAIL** ✅ 검출(2파일) |
| M5 | correspondence flip 자동해소 무력화(항상 배정 A) | **FAIL** ✅ 검출(2파일) |
| **M6** | **f 공동추정 무력화 → 프리셋 단독 f** | **PASS** ❌ **미검출 → §3** |

→ `roiAlignmentSensitivity.test.ts` 는 **이미지 평행이동**에 대해 실질 검출력을 가진다. 이 부분은 인정한다.

### 2-2. 그러나 — 두 지표가 재는 것은 정합이 아니다

두 지표가 실제로 재는 것은 이것뿐이다:

> *"이 ROI 는 (PTZ 가 보고한 tilt 를 가진) 어떤 카메라로 본 **2.5×5.0m 직사각형 스트립의 상(像)**인가?"*

그런데 **지면 위의 닮음변환(평행이동 2 + 수직축 회전 1 + 균일 스케일 1 = 4 DOF)은 이 성질을 그대로 보존한다.**
즉 ROI 를 지면에서 통째로 밀거나·돌리거나·키워도 두 지표는 **원리적으로 반응할 수 없다.**

기준 모델 `(f, n, d)` 로 역투영 → 지면에서 변환 → 재투영해 **정확히 구성**하고 측정했다(preset 1):

| 변환 | 이미지 평균변위 | `metricErr` (임계 0.8%) | `tiltErrDeg` (임계 1.0°) | 검출 |
|---|---|---|---|---|
| 기준(무변환) | 0px | 0.19% | 0.04° | – |
| [이미지] 가로 +105px (실제 발생) | 105px | **1.53%** | 0.06° | ✅ |
| [이미지] 세로 +200px | 200px | 0.38% | **−3.86°** | ✅ |
| **[이미지] 회전 10°** | **77px** | 0.57% | 0.05° | ❌ **둘 다 놓침** |
| **[지면] 평행이동 e1 +2.5m (한 칸)** | 70px | 0.19% | 0.04° | ❌ **둘 다 놓침** |
| **[지면] 평행이동 +3m/+3m** | **360px** | **0.19% (불변)** | **0.04° (불변)** | ❌ **둘 다 놓침** |
| **[지면] 수직축 회전 30°** | 144px | 0.23% | 0.04° | ❌ **둘 다 놓침** |
| **[지면] 균일 스케일 ×2.0** | **473px** | 0.14% | 0.03° | ❌ **둘 다 놓침** |

**지면 평행이동 3m/3m 은 이미지상 360px 어긋났는데 두 지표가 기준값에서 소수점까지 그대로다.**

### 2-3. 실질 위험 2가지

1. **지면 평행이동 2.5m = 주차면 정확히 한 칸.** ROI 가 옆 칸을 덮어도 두 지표는 침묵한다 → **점유가 통째로 옆 칸에 찍힌다.** 자동화가 *조용히* 틀리는 것(설계 §12 최상위 위험)의 정확한 실례.
2. **지면 균일 스케일은 `d`(카메라고 = metric 스케일의 유일한 담지자)를 조용히 배수로 틀리게 만든다** — ×2 스케일에서 `d` 4.96m → **2.48m**, 경보 0건. 육면체 높이·`measured` 실측치가 그대로 틀어진다.

### 2-4. 부수 — 소스 주석의 주장이 성립하지 않는다

`groundModel.ts:48`:
```
임계 0.8% 근거(실측): … → 0.8% 는 정상을 여유롭게 통과시키고 ≳60px 어긋남을 잡는다.
```
**거짓.** 이미지 회전 10°는 평균 **77px** 어긋남인데 `metricErr` 0.57% 로 임계 미만이다. 이 주장은 **순수 가로 평행이동에만** 성립한다.

### 2-5. 조치 요청 (구현자에게 전달)

- **(필수)** 위 주석·02 보고서 §5-B 의 과장 정정. **두 지표는 "이미지 평행이동 검출기"이지 정합 보증이 아니다.**
- **(권고)** 4 DOF 중 **2 DOF 는 이미 가진 데이터로 닫힌다**(검출력 실증 첨부):
  - **균일 스케일 → 프리셋 간 `d` 일관성.** 카메라는 프리셋 사이에 움직이지 않으므로 `d` 는 **프리셋 불변량**이어야 한다. 실측: 정상 스프레드 **<5%**(4.96/4.99/4.94), preset1 만 지면 ×2 → **>30%**.
  - **수직축 회전 → 슬롯 방위의 프리셋 간 일관성**(PTZ `pan` 활용).
- **(한계로 확정)** 남는 **2 DOF(지면 평행이동)는 이미지 증거(노면 도색) 없이 원리적으로 검출 불가.**

### 2-6. ✅ 구현자 수정 결과 — 회전·스케일 2 DOF 폐쇄 (재검증 통과)

구현자가 `groundModel.ts` 에 **§GROUND-SIMILARITY** 블록을 추가했다. **프리셋 간 불변량 대조**(신규 입력 0):

| 신규 검사 | 원리 | 신규 필드 | 뮤테이션 검출력 |
|---|---|---|---|
| **`dDevRel`** (균일스케일) | 카메라고 `d` 는 **프리셋 불변량** — 카메라는 프리셋 사이에 움직이지 않는다 | `GroundModel.dDevRel` | **N1**: `D_DEV_REL` 0.1 → 10 무력화 시 **FAIL** ✅ |
| **`bearingDevDeg`** (수직축 회전) | 슬롯 방위(PTZ pan 보정, mod 90)도 **프리셋 불변량** | `GroundModel.slotBearingDeg` / `bearingDevDeg` | **N2**: `BEARING_DEV_DEG` 8 → 90 무력화 시 **FAIL** ✅ |

- **검증 결과**: 지면 ×2.0 스케일 → `dDevRel` > 30% + `균일스케일 오류 의심` advisory 발화. 지면 30° 회전 → `bearingDevDeg` > 5° + `수직축 회전 의심` 발화. **정합된 ROI(Unity 원형)에서는 오탐 0.**
- **PTZ pan 사용법이 검증자 제안보다 낫다**: 검증자는 "PTZ pan 과 슬롯 방위를 직접 대조"를 제안했으나, 이는 **슬롯의 실제 세계 방위를 모르면 성립하지 않는다**(tilt 는 '수평 지면'이라는 세계 기준이 있어서 가능했지만, 방위각에는 대응하는 기준이 없다). 구현자는 pan 을 **공통 좌표계로 정규화하는 용도**로 써서 프리셋 간 대조를 성립시켰다 — **올바른 판단이며 검증자 제안을 정정한 것이다.**

### 2-7. ⚠️ 확정된 잔여 한계 (덮지 말 것)

| 잔여 구멍 | 왜 못 잡는가 | 해법 |
|---|---|---|
| **지면 평행이동 2 DOF** | `d` 도 슬롯 방위도 바꾸지 않는다 → **프리셋 간 불변량 검사조차 눈이 멀었다.** 실증: +3m/+3m(이미지 **360px**) → 전 지표·전 advisory **침묵** | **이미지 증거(노면 도색) 필요 → 2단계 median 배경.** 원리적으로 그 외 방법 없음 |
| **전 프리셋 동시 오류** | 프리셋 간 대조는 **상대** 불일치만 잡는다. 전역 균일스케일이면 불변량이 여전히 일치 | 절대 기준(주차면 폭 2.5m 가정)에 의존 — `d` 는 **그 가정만큼만 정확**하다 |
| **프리셋 1개** | 대조 불가 | advisory 발화(구현자가 처리) |

> **가장 위험한 시나리오는 여전히 열려 있다**: 지면 평행이동 2.5m = 주차면 **정확히 한 칸** → ROI 가 옆 칸을 덮어 **점유가 통째로 옆 칸에 찍힌다.** 전 지표가 침묵한다. 이것은 **1단계에서 해결 불가**이며, 반드시 문서·배지에 남겨야 한다.

**봉인**: `test/roiAlignmentBlindSpot.test.ts` (10 테스트) — ① 두 지표의 원리적 사각(수정 후에도 불변) ② 신규 검사 2종의 검출력 + 오탐 0 ③ **잔여 구멍(지면 평행이동)이 여전히 전 지표를 통과함**을 고정.

## 3. ★ 테스트 결함 — f 공동추정이 통합 경로에서 무방비 (M6)

**설계 §4-4 의 핵심 결정**("f 를 프리셋별로 독립 추정하면 안 된다", R1: 얕은 tilt 에서 f 가 조용히 20~35% 틀림)이 **테스트로 보호되고 있지 않았다.**

`estimateGroundModels` 의
```ts
f = focalFromZoom(preset.zoom, pooled.fovBaseV, imgW, imgH);   // 공동추정
```
을 **`f = fSolo`(프리셋 단독 추정)로 바꿔도 전 테스트가 그대로 통과했다.**

### 원인 (2중)
1. `groundModelRoundTrip.test.ts` 의 공동추정 테스트(`:171-200`)는 `poolFovBaseV`/`focalFromVPs`/`focalFromZoom` 을 **직접** 호출한다 → *함수가 동작함*은 증명하나 *프로덕션 진입점이 그것을 쓴다*는 증명하지 않는다.
2. `estimateGroundModels` 를 호출하는 유일한 테스트(`:202`)는 **σ=0(무노이즈)**. 합성데이터는 단일 `fovBaseV` 에서 생성되므로 **단독 f 와 공동추정 f 가 1e-6 이내로 완전 일치**한다(내가 확인) → 두 경로가 구별되지 않는다.

### 공동추정의 실효는 실재한다 (통합 경로로 재현)

| σ | preset 1 (얕은 tilt, 조건수 최악) | preset 2 | preset 3 |
|---|---|---|---|
| 0.5px | 단독 3.8% → **공동추정 0.9%** | 2.0% → 0.9% | 0.8% → 0.9% |
| 1px | 단독 7.9% → **공동추정 1.9%** | 4.1% → 1.9% | 1.5% → 1.9% |
| **2px** | **단독 20.2% → 공동추정 3.8%** (약 5배 개선) | 8.3% → 3.8% | 3.1% → 3.8% |

- **결론은 설계서와 동일**(단독 추정은 얕은 tilt 프리셋에서 무너지고, 공동추정이 구제한다).
- **단, 설계서의 "35.4% → 1.7%" 수치 그대로는 재현되지 않는다**(합성 기하가 실데이터와 달라서). 정직하게 명시한다.
- 조건수 최선 프리셋(3)은 공동추정으로 **약간 나빠진다**(3.1% → 3.8%) — 좋은 프리셋이 나쁜 프리셋을 구제하는 대가. 타당한 트레이드오프.

### 조치 (검증자가 직접 수행)
**신규 `test/groundModelPoolingIntegration.test.ts`** (3 테스트):
- **구조적 봉인(실데이터 픽스처)** — 모든 프리셋 f 가 **단일 `fovBaseV`** 에서 `fovV(zoom)` 로 유도된 값과 일치하는가(`< 1e-9`). + 단독 f 와 공동추정 f 가 실제로 다름을 먼저 확인(**공허한 단언 방지**).
- **통계적 봉인** — σ=2px·100시행에서 preset 1 f 오차 RMS **< 8%** 이며 단독 대비 **절반 이하**.

**M6 재적용 시 3개 전부 실패 확인** → 공백 폐쇄. **프로덕션 코드 수정 불요(로직은 옳다).**

## 4. C3(실카메라 호환) 준수 — 지면모델 경로는 준수, 단 프로젝트 전체 전제는 거짓

if (cm && Number(cm.fov) > 0) fovBaseV = Number(cm.fov);

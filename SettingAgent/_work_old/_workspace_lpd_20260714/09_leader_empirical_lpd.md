**일시**: 2026-07-14 (시뮬레이터·서버 가동 상태)
**대상**: 모드 A 에서 LPD 번호판도 필터 — `keepPlate = (유지된 VPD 차량에 귀속) OR (번호판 중심 ∈ 주차면 폴리곤)`
**목적**: 구현자가 "성공 판정의 본체"로 지목한 라이브 재현 — ① 마스터 스크린샷의 뒷줄 노란 번호판이 사라지는가 ② **점유 판정(`computeOccupancy`)이 뒤집히지 않는가**(OR 규칙의 존재 이유)

## 2. ★ 점유 무회귀 — 통제된 대조로 확인 (가장 중요)

설계서 §7-1 은 "필터가 지우는 번호판은 **점유를 참으로 만들 수 없는 것뿐** → `computeOccupancy` 결과 불변"을 **증명**했다고 주장했다. 이를 **실데이터로 반증 시도**했다.

**방법(통제)**: 모드 B 응답(원본 전량)을 받아 **같은 데이터**에 필터를 적용/미적용하고, **뷰어가 실제로 쓰는 `web/core.js:computeOccupancy`** 를 `app.js:335` 와 **동일한 소비 형태**(`floorPolys = {idx, quad}`, `plates = detect.plates ∪ vehicles[].plate`)로 호출해 점유 면 집합을 비교했다.

## 5. 검증하지 못한 것 (은닉 금지)

1. **브라우저 실제 클릭 미검증**(DOM 자동화 없음) — 라우트·서빙 HTML·app.js 배선은 확인했으나 사람이 체크박스를 누르는 경로는 여전히 코드 대조뿐. `#cap-vpd-onplace` rename 시 조용히 죽는 위험 유효.
2. **정밀수집 잡(DB 경로)의 LPD 필터는 라이브 재현 안 함** — 1차 때 VPD 는 run69 로 DB 끝단까지 확인했으나, 이번 LPD 필터의 DB 저장분은 **유닛테스트로만** 확인했다(추가 run 을 돌리면 DB 데이터가 또 늘어나므로 보류). 필요 시 1라운드 추가 실행으로 확인 가능.
3. **(A)·(B) 동시 실패 케이스**: VPD 미검출 + 번호판 중심이 폴리곤 밖 → 번호판 drop, 경고 없음. 실데이터에서 발생하지 않았으나 원리적으로 가능.
4. **중심 정의 ε**: 서버는 `quadBoundingRect+center`, `web/core.js:quadCentroid` 는 4점 평균. 직사각형 OBB 에선 일치하나 기울어진 quad 에서 미세 차이 → **후속 P-1**.

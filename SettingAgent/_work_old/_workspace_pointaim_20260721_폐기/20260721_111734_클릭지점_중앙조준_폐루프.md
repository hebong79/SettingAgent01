## 1. 문제와 실패 원인 (개루프 1스텝)

기존 `aimPointToCenter`는 배치 센터라이징의 `preAimPtz`(coarse 사전조준) 공식을 **1회만** 적용하는 개루프였다. 즉 클릭 지점의 오차를 한 번 측정해 그 오차에 고정 게인을 곱한 만큼 pan/tilt를 이동시키고 끝냈다.

**리더 라이브 경험적 검증(실카메라 cam1/preset1, 1920×1080, z1.69341)으로 확인된 실패 원인 2가지:**

| 항목 | 실측값 |
|---|---|
| pan +1° → 화면 dx | −0.0266 (화면 중앙 부근) |
| tilt +1° → 화면 dy | −0.0472 (화면 중앙 부근) |
| pan +1° → 화면 dx (변두리) | −0.0177 |

- 게인 크기 자체는 config(`fallbackGainPanDeg:-62`, `fallbackGainTiltDeg:-35.5` @zoomRef=1)를 zoom 스케일한 값과 일치 — 게인이 틀린 게 아니었다.
- 실패 원인은 ① **1스텝 개루프라 잔차(오차)가 전혀 재보정되지 않음**, ② **원근 비균일** — 화면 중앙(−0.0266)과 변두리(−0.0177)의 게인 차이가 약 30%에 달해, 한 지점에서 측정한 게인으로 다른 지점을 1스텝에 정확히 맞추는 것이 원리적으로 불가능.
- pan↔dx, tilt↔dy는 순수 분리·대칭(교차항 없음)임도 실측으로 확인 — 이는 유지하되, **매 스텝 재측정으로 잔차와 원근 비균일을 흡수하는 폐루프**로 재구현하면 해결된다는 것이 이번 재구현의 근거다.

## 2. 해법: 클릭 패치 NCC 추적 폐루프

추적 대상을 "번호판"이 아니라 **클릭된 지점의 이미지 패치 자체**로 삼는다. 시작점(클릭 위치)에서 항상 화면 안에 있고, 목표(화면 중앙)도 화면 안이므로 패치가 프레임을 벗어날 일이 없다. 매 pan/tilt 명령 후 새 프레임에서 그 패치가 어디로 이동했는지를 **정규화상관(NCC) 최대점**으로 재측정 → 잔차와 원근 비균일이 스텝마다 흡수되어 수렴한다.

신규 의존성 0(기존 `sharp`만 사용). LPD·번호판 인식 불필요.

### 2.1 `src/calibrate/patchTrack.ts` — 순수 함수 4종 (유닛테스트 대상)

| 함수 | 계약 |
|---|---|
| `toGray(jpeg, workW): Promise<Gray>` | JPEG → 그레이스케일 + `workW` 폭 다운스케일 단일채널 래스터 `{data,w,h}`. 정규화좌표는 해상도 무관이라 다운스케일해도 정확도 손실 없이 속도만 얻음(1920→960). |
| `patchTexture(g, cxN, cyN, halfPx): number` | 패치 로컬 표준편차(모집단 std). 저텍스처(특징 부족) 판정용. |
| `extractTemplate(g, cxN, cyN, halfPx): Template` | 클릭 패치를 NCC 템플릿으로 추출 — `{tpl(원시 그레이, 행우선), half, mean, std(평균차 L2노름)}`. |
| `nccSearch(g, template, predCxN, predCyN, radiusPx, step=1)` | 예측점 ±radius를 `step` 간격으로 훑어 정규화상관 `score = Σ(tpl-tmean)(win-wmean) / (tstd·√Σ(win-wmean)²)` 최대점 반환(`{cxN,cyN,score}`, score∈[-1,1]). full 패치가 경계를 벗어나는 후보는 스킵, 유효 후보 0이면 예측점·score 0.

좌표는 전 구간 정규화(0~1) 입출력 — 픽셀 인덱싱은 함수 내부에 한정된다. 중심은 `clampCenter`로 `[half, dim-1-half]`에 정렬해 모서리 클릭에도 항상 full 패치를 확보한다.

### 2.2 `src/calibrate/PointAimer.ts` — 폐루프 조준기 `aim()`

`centerOnPlate`(번호판 center 폐루프) 구조를 그대로 미러링하되 측정 계층만 패치 추적으로 교체했다. 무상태 클래스, `Repository`/`writer`/`DB` 의존 0 — **저장 호출이 코드 상 전혀 없다**(생성자 주입도 `camera`/`cfg`/`sleep`/`onFrame` 뿐).

흐름(`aim(camIdx, presetIdx, click, opts?)`):

1. `startPtz = resolvePresetPtz(...)` (실패 시 `{pan:0,tilt:0,zoom:1}` 폴백 + warn 로그).
2. **cap0 캡처** → `onFrame`(→`lastFrame` 갱신, `/calibrate/frame` 폴링에 표시) → `settle` → `toGray`.
3. **저텍스처 게이트(단계 확대 재시도)**: 설정 half(기본 32px)로 `patchTexture < TEX_MIN(8)`이면 즉시 포기하지 않고, `TEXTURE_RETRY_HALVES=[32,64,96]` 순으로 반경을 확대하며 재평가한다. 어느 단계가 `TEX_MIN` 이상이면 그 확대 half(`effHalf`)로 `extractTemplate`. 이미지 경계는 `maxHalf=floor((min(w,h)-1)/2)`로 클램프. 모든 확대에도 미달이면 `{ok:false, reason:'low_texture'}` 정직 반환(iterations=0).
4. 게인 = `scaleGainForZoom({gainPan:cfg.fallbackGainPanDeg, gainTilt:cfg.fallbackGainTiltDeg, zoomRef:1}, startPtz.zoom)` — **루프 내내 고정**(온라인 보정 없음, §2.3 참조).
5. 반복(`maxIter=20`):
   - `err = pNow - 0.5` → `isCentered(err, centerTol)`면 `{ok:true, ptz, finalErr, iterations}` 반환.
   - `panTiltCorrection(err, gain, pan, tilt, maxStepDeg=2.5)`로 `cmd`(zoom 불변) 산출 → `requestImage(cmd)` → `onFrame` → `settle` → `toGray`.
   - 예측점 `pred = predictPlateCenter(pNow, {dPan:cmd.pan-ptz.pan, dTilt:cmd.tilt-ptz.tilt}, gain)`(명령 변위를 게인으로 화면변위로 역산) 근처 ±`searchRadiusPx=28`을 `nccSearch`.
   - `score < minScore(0.5)`면 광역 재탐색 1회(반경 ×`WIDE_SEARCH_MULT=2.5`) — 그래도 미달이면 `{ok:false, reason:'patch_lost'}` 반환.
   - `pNow=측정점, ptz=cmd`로 갱신, 다음 반복.
6. `maxIter` 소진 시 `{ok:false, reason:'max_iterations'}`.

부호 규약(pan+ → dx−, tilt+ → dy−)은 `controlMath.panTiltCorrection`(Δ=−gain·err)과 `predictPlateCenter`(역산)가 이미 소유하고 있어 `PointAimer`는 그대로 재사용만 한다(신규 부호 코드 없음).

### 2.3 온라인 게인보정(refineGain) — 의도적 미포함

설계에서는 온라인 게인보정을 선택지로 검토했으나, 구현에서는 **채택하지 않았다**. 실측상 config fallback 게인이 참값과 일치하고, 매 스텝 재측정만으로 원근 비균일(~30%)을 흡수하므로 불필요하며, 저텍스처/오매칭 프레임의 잡음이 게인에 되먹임되면 과보정 발산 위험이 있기 때문이다(CLAUDE.md 단순함 원칙과도 정합). `PointAimer` 상단 주석에 EMA·발산가드 TODO로 명시해 두었다.

## 3. 저장 없음 · zoom 불변 · 상호배타

- `PointAimer` 생성자 의존은 `camera`/`cfg`/`sleep`/`onFrame` 뿐 — writer/DB/스냅샷/upsert를 import도, 호출도 하지 않는다(소스 정적 확인 + qa 저장 스파이 0회 테스트로 검증).
- 매 명령(`cmd`)의 `zoom`은 `startPtz.zoom`으로 고정 — 루프 내내 줌 변경 없음.
- `PtzCalibrator.aimPointToCenter`는 배치(`state==='running'`)·개별 진행(`pointBusy`, `centerOnPoint`와 공유) 양쪽에 대해 상호배타를 유지한다(가드·`try/finally` 무변경).

## 4. 라우트·프론트 계약

`POST /calibrate/point { cam, preset, point:{x,y}, mode:'point' }` → `calibrateRoutes.ts` 55~58행:
```
if (p.data.mode === 'point') {
  const r = await deps.calibrator.aimPointToCenter(p.data.cam, p.data.preset, p.data.point);
  return { ok: r.ok, ptz: r.ptz, ...(r.reason ? { reason: r.reason } : {}), ...(r.iterations != null ? { iterations: r.iterations } : {}) };
}
```
성공 시 `reason`/`iterations`가 스프레드로 빠져 응답은 기존과 동일한 `{ok,ptz}` 형태를 유지한다(조건부 스프레드 — 계약 회귀 0). 실패 시에만 `reason`(`low_texture`/`patch_lost`/`max_iterations`)과 `iterations`가 가산된다. `mode:'plate'`/`'plate-zoom'`(번호판 center) 분기는 완전히 무변경.

프론트(`web/app.js` `calPointCenter`, 2373~2380행): `mode==='point'`이고 실패(`!data.ok`)이며 `data.reason`이 있을 때만 한글 메시지로 매핑한다.

| reason | 한글 안내 |
|---|---|
| `low_texture` | "클릭 지점 텍스처가 부족합니다 — 차량 모서리/번호판 등 무늬가 뚜렷한 곳을 클릭하세요" |
| `patch_lost` | "추적 실패 — 다시 시도하세요" |
| `max_iterations` | "수렴 실패(반복상한) — 다시 시도" |

성공 메시지·`plate` 경로 메시지·409(busy/running) 처리는 무변경.

## 7. 한계 (정직)

- **저텍스처/야간/반복무늬**: 확대 재시도(32→64→96)에도 std<8이면 `low_texture`, 추적 중 패치를 잃으면 `patch_lost`를 정직 반환한다 — 기하 폴백이나 위장 성공 없음.
- **실측 게인의 카메라 의존성**: `fallbackGainPanDeg:-62`/`fallbackGainTiltDeg:-35.5`(@zoomRef=1)는 cam1 시뮬 실측 유래다. 타 카메라는 초기 게인이 다를 수 있으나, 폐루프가 매 스텝 재측정으로 잔차를 흡수하므로 원리적으로는 수렴한다 — 다만 초기 예측창이 커져 반복 수가 늘거나 `WIDE_SEARCH_MULT` 재탐색에 더 의존할 가능성은 있다(비-cam1 실측은 미수행, 확인 필요).
- **PTZ 지연**: `settleMs`(cfg 300ms)로 흡수 — 실카메라 이동 지연이 이보다 크면 조기 캡처로 오매칭 가능성(관측되지 않았으나 튜닝 대상으로 남김).
- **온라인 게인보정 미포함**: §2.3 참조 — 필요 시 EMA·발산가드로 추후 도입 여지를 코드 주석에 남겼다(TODO, 미구현).

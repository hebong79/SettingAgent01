# 개별 center — 클릭 지점을 화면중앙으로(번호판 무관)

- 작업 루트: `SettingAgent`
- 관련 산출물: `_workspace/02_developer_changes.md`(구현 노트) · `_workspace/03_qa_report.md`(검증) · `_workspace/04_doc_impact.md`(영향도, 이번 변경분 섹션 추가)
- 신규 테스트: `test/ptzCalibrator.point.test.ts`(+6) · `test/calibrateRoutes.point.test.ts`(+9)
- 직전 기능 문서(맥락): `docs/20260720_232555_클릭기반_개별센터라이징.md` — "개별 센터라이징" 기능 최초 도입분. 이번 변경은 그 문서의 **2번째 옵션(개별 center)의 동작을 재정의**한다. 3번째 옵션(개별 center+zoom)은 그 문서가 기술한 대로 그대로 유지된다.

## 1. 변경 개요 — "개별 center"가 가리키는 대상이 바뀜

기존(직전 문서 기준)에는 콤보 `#cal-click-mode`의 두 옵션(`center`/`center-zoom`) 모두 **클릭 지점 최근접 번호판**을 향해 정렬하는 동작이었다(차이는 줌 마감 여부뿐). 이번 변경으로 `center`(개별 center)만 **클릭한 지점 자체**(번호판과 무관한 순수 좌표)를 화면 중심(0.5, 0.5)으로 pan/tilt 조준하도록 재정의됐다. `center-zoom`(개별 center+zoom)은 번호판 기반 동작을 그대로 유지한다.

| | 개별 center (`mode:'point'`) — **이번 변경** | 개별 center+zoom (`mode:'plate-zoom'`) — **무변경** |
|---|---|---|
| 대상 | 클릭 **지점 자체**(x, y) | 클릭 지점 **최근접 번호판**(`pickNearestPlate`) |
| 검출 사용 | **미사용** — LPD/plate 경로 진입 없음 | 사용(`centerOnPlate` → 번호판 quad 검출) |
| 제어 방식 | **개루프 1스텝**(fallback 게인 기반 단발 명령) | **폐루프**(`centerOnPlate` 반복 수렴) + `zoomToPlateWidth` |
| 줌 | **불변**(현재 zoom 유지) | 변경(`targetPlateWidth`로 마감) |
| 백엔드 메서드 | `PtzCalibrator.aimPointToCenter()`(신규) | `PtzCalibrator.centerOnPoint()`(기존, 무변경) |
| 응답 shape | `{ ok, ptz }` | `{ ok, ptz, plateWidth, reason? }` |
| 저장 | 없음 | 없음(무변경 — 직전 문서에서 이미 확정된 정책) |

## 2. 신규 백엔드: `PtzCalibrator.aimPointToCenter()`

파일: `src/calibrate/PtzCalibrator.ts` (189~224행, `centerOnPoint` 바로 아래)

```ts
async aimPointToCenter(
  camIdx: number,
  presetIdx: number,
  point: NormalizedPoint,
): Promise<{ ok: boolean; ptz: Ptz }>
```

### 2.1 동작 흐름

1. **상호배타 가드**(206~207행): `state === 'running'`(배치 잡 진행 중)이면 `throw new Error('calibrate already running')`. `pointBusy`가 `true`면 `throw new Error('point centering busy')` — `centerOnPoint`와 동일한 락 필드를 공유하므로 배치·개별 center·개별 center+zoom 세 경로가 서로 배타적이다. `try/finally`(208~223행)로 정상/예외 종료 모두 `pointBusy`가 해제된다.
2. `startPtz = await this.startPtzFor({ camIdx, presetIdx } as PlateTarget)`(210행) — 기존 배치·`centerOnPoint`가 공유하는 `ptzByKey` 캐시를 그대로 재사용해 프리셋 base PTZ(pan/tilt/zoom)를 얻는다.
3. **개루프 1스텝 기하 조준**(211~217행) — `preAimPtz`(492~503행 부근)와 동일한 공식을 클릭점 기준으로 인라인 적용:
   - `g = scaleGainForZoom({ gainPan: cfg.fallbackGainPanDeg, gainTilt: cfg.fallbackGainTiltDeg, zoomRef: 1 }, startPtz.zoom)` — 현재 zoom에 맞게 게인을 스케일.
   - `err = { errX: point.x - 0.5, errY: point.y - 0.5 }` — 클릭 지점의 화면중심(0.5, 0.5) 대비 오차.
   - `pt = panTiltCorrection(err, g, startPtz.pan, startPtz.tilt, PREAIM_MAX_STEP)` — `PREAIM_MAX_STEP = 90`(25행)으로 1스텝 클램프.
   - `cmd: Ptz = { pan: pt.pan, tilt: pt.tilt, zoom: startPtz.zoom }` — **zoom은 `startPtz.zoom` 그대로 불변**(center only, 줌 접촉 없음).
4. `cap = await this.camera.requestImage(camIdx, presetIdx, cmd)` → `this.lastFrame = { jpeg: cap.jpg, camIdx, presetIdx }`(218~219행) — `/calibrate/frame` 폴링이 진행 프레임을 보여주는 기존 메커니즘을 그대로 이용(별도 `onFrame` 콜백 없이 직접 갱신).
5. `return { ok: true, ptz: cmd }`. **저장 호출 없음** — 메서드 본문에 `writer`/`saveCenteringSlots`/`saveSetupSnapshot`/`upsertSlotCentering` 어느 것도 없다(코드 직접 확인).

### 2.2 개루프 1스텝의 근거와 한계(정직 기록)

- **개루프인 이유**: 임의로 클릭한 지점은 프레임 안에서 추적할 특징(번호판 등)이 없다. 번호판이 있으면 매 스텝 재검출→재계산하는 폐루프 수렴(`centerOnPlate`)이 가능하지만, "지점 자체"는 그런 피드백 신호가 없으므로 게인 기반 단발 명령으로 조준한다.
- **정확도의 전제**: 조준 정확도는 fallback 게인(`fallbackGainPanDeg`/`fallbackGainTiltDeg`, 기본값 약 −62/−35.5 @ `zoomRef=1`)의 정확도에 전적으로 의존한다. 이 값은 cam1 시뮬 실측으로 유도된 값이라, **비-cam1 실카메라에서는 게인 편차만큼 조준 잔차가 남을 수 있다.** 재측정(probe)·재보정 루프가 없으므로 잔차는 그대로 남는다.
- 이는 결함이 아니라 요구("단순 지점 조준")에 부합하는 설계된 트레이드오프다. 정밀 수렴이 필요한 경우 번호판 기반 "개별 center+zoom"(3옵션)을 쓰도록 안내된다.

## 3. 라우트 계약: `POST /calibrate/point`의 `mode` 분기

파일: `src/api/calibrateRoutes.ts`

### 3.1 요청 스키마(`PointBodySchema`, 9~17행)

```ts
{
  cam: number (정수, ≥0),        // 1-based 카메라 인덱스
  preset: number (정수, ≥0),    // 1-based 프리셋 인덱스
  point: { x: number, y: number }, // 정규화 좌표(0~1)
  mode?: 'point' | 'plate' | 'plate-zoom', // 신규
  zoom?: boolean,                 // legacy — mode 미전달 시에만 사용
}
```

### 3.2 핸들러 분기(48~69행)

- `mode === 'point'` → `calibrator.aimPointToCenter(cam, preset, point)` 호출 → **200** `{ ok, ptz }`(`plateWidth`/`reason` 키 없음).
- 그 외(`'plate'` / `'plate-zoom'` / mode 미전달) → `const zoom = mode ? mode === 'plate-zoom' : body.zoom;`으로 zoom 여부를 결정한 뒤 기존 `centerOnPoint(cam, preset, point, { zoom })` 호출 → **200** `{ ok, ptz, plateWidth, reason? }`.
  - `mode === 'plate'` → `zoom = false`.
  - `mode === 'plate-zoom'` → `zoom = true`.
  - `mode` 미전달(legacy) → `zoom = body.zoom`(기존 하위호환 경로, 완전 보존).
- **에러 매핑**(64~68행, 무변경): `catch`에서 예외 메시지가 `'running'` 또는 `'busy'`를 포함하면 **409**, 그 외는 **400** `{ error: msg }`. `aimPointToCenter`의 두 throw(`'calibrate already running'`, `'point centering busy'`)도 동일 문자열 패턴이라 기존 매핑 로직에 자연히 포함된다(라우트 코드 자체는 손대지 않음).
- 스키마 검증 실패 시 **400** `{ error: 'invalid body', detail: <zod flatten> }`(무변경).
- 기존 4개 라우트(`/calibrate/ptz`, `/calibrate/status`, `/calibrate/frame`, `/calibrate/result`)는 코드상 미변경.

## 4. 프론트 매핑: `web/app.js`

### 4.1 오버레이 클릭 게이트(3273~3280행)

```js
const clickMode = $('cal-click-mode')?.value;
if (clickMode && clickMode !== 'off' && !e.ctrlKey) {
  const { nx, ny } = eventToNorm(e);
  e.preventDefault();
  const mode = clickMode === 'center-zoom' ? 'plate-zoom' : 'point';
  void calPointCenter(nx, ny, mode);
  return;
}
```

- `clickMode`는 콤보값(`off`/`center`/`center-zoom`) — `off` 필터를 통과한 뒤에는 `center` 아니면 `center-zoom` 둘뿐이다.
- 매핑: `center` → `'point'`(신규, 지점 자체 조준), `center-zoom` → `'plate-zoom'`(무변경, 번호판 center+zoom). `'plate'`(center-only, zoom 없음) 값은 이 UI 경로에서 프론트가 생성하지 않는다 — 콤보에 대응 옵션이 없기 때문(스키마·라우트는 지원하나 UI 미노출).

### 4.2 `calPointCenter(nx, ny, mode)`(2348~2379행)

- 3번째 인자가 기존 `zoom` 불리언에서 `mode` 문자열로 변경됐다.
- body: `{ cam, preset, point: { x: nx, y: ny }, mode }`(2362행) — legacy `zoom` 필드는 더 이상 프론트에서 보내지 않는다(라우트는 하위호환으로 여전히 받아들임).
- 진행 메시지 분기(2353행): `mode === 'plate-zoom'` → `'번호판 센터+줌 중…'`, 그 외(`'point'`) → `'클릭 지점을 화면 중심으로 이동 중…'`.
- 나머지(진행 프레임 폴링 `startCalFramePolling`/`stopCalFramePolling`, 완료 후 `startLive()` 복귀, 409/결과 메시지 처리)는 무변경.

### 4.3 `web/index.html`(무변경)

콤보 옵션(`off`/`center`/`center-zoom`) 자체는 직전 기능에서 이미 만들어진 그대로다 — 이번 변경은 `center` 옵션이 **가리키는 백엔드 동작**만 바꿨을 뿐, HTML 마크업은 손대지 않았다.

## 6. 한계 (그대로 승계 — 완료로 위장하지 않음)

1. **개루프 1스텝 조준의 실측 정확도는 vitest로 검증 불가**: `aimPointToCenter`는 fallback 게인(cam1 시뮬 실측 유도값) 기반 단발 명령이라, 비-cam1 실카메라에서는 게인 편차만큼 조준 잔차가 남을 수 있다. 클릭 지점이 실제로 화면중앙에 얼마나 근접하는지는 라이브/sim 실측으로만 확인 가능하며, 이번 검증은 결정형 유닛(공식 대조)에 한정된다.
2. **프론트 DOM E2E 미검**: `overlay.mousedown` → `eventToNorm` → `calPointCenter` → `fetch` 로 이어지는 실제 브라우저 클릭 배선은 vitest 범위 밖이다. body 조립 shape과 mode 매핑은 소스 코드 정적 대조로 확인했으나, 실제 DOM 이벤트 발화 자체는 미검(직전 기능 문서에서 이미 명시된 한계와 동일 성격, 이번 변경으로도 해소되지 않음).

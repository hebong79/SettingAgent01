## 1. 배경 / 목표

### 스트림 1 — Unity RPC 연동(방식 B 프록시)
- **현황(As-Is)**: SettingViewer 웹(`/viewer/*`, 13020)은 `CameraSource` 추상화를 통해 Unity 13100(REST 카메라 전용)에만 연결돼 있었다. Unity 13110의 JSON-RPC(현재 76개 method: `preset.*`/`car.*`/`cam.*`/`map.*`/`measure.*`/`scene.*`/`system.*`)는 뷰어와 전혀 연결되어 있지 않았다.
- **목표**: SettingViewer 웹이 브라우저에서 Unity 13110 RPC를 호출할 수 있게 한다. 리더 확정에 따라 **방식 B(13020 서버 경유 프록시)만** 도입하고, 보안은 **controlToken 게이트만**(화이트리스트 없음, 설계서 §4.2·§8) 적용한다.
- **핵심 결정**: 저장소에 이미 있는 `CRpcClient`(범용 JSON-RPC 클라이언트, 기존엔 `mcp/server.ts`만 사용)와 `/viewer/api/move`의 controlToken 인증 패턴을 **그대로 재사용** — 신규 클라이언트·인증 체계를 만들지 않는다.

### 스트림 2 — LLM 멀티 프로바이더(config 프로필 + 런타임 전환)
- **현황(As-Is)**: `llm.config.json`의 단일 `llm` 블록 하나만 존재. 프로바이더 변경 시 파일을 편집하고 서버를 재시작해야 했다.
- **목표**: qwen 외에 claude/gemini/codex 등 여러 프로필을 config에 등록해두고, **재시작 없이 런타임 API/UI로 활성 프로필을 전환**한다.
- **핵심 결정**: 모든 프로필을 **OpenAI 호환 엔드포인트로 통일** 호출(Claude/Gemini도 각자 OpenAI 호환 `/v1`) → 기존 `AgentRuntime`의 `chat.completions` 경로를 재사용. 활성 상태는 **AgentRuntime 인스턴스 내부가 소유**하고, 인스턴스 교체 없이 **활성 프로필만 스왑**한다(소비처 무변경).

## 2. 스트림 1 상세 — RPC 프록시

### 2.1 신규 REST(13020 가산)

| 라우트 | 메서드 | 게이트 | 동작 |
|---|---|---|---|
| `/viewer/api/rpc` | POST | controlToken 선택 게이트(변이) | body `{method, params?}` → `rpc.callRpc(method, params)` → `{ok:true, result}` / 실패 502 `{ok:false, error}` |
| `/viewer/api/rpc/catalog` | GET | 무게이트(읽기) | `rpc.getCatalog()` → `{methods:[...]}` / 실패 502 `{error}` |

- 두 라우트는 `ViewerDeps.rpc`가 **주입된 경우에만** 등록된다(`if (rpc) { ... }`, `src/viewer/routes.ts:268`). 미주입 시 라우트 미등록 → 404. 즉 **가산·graceful**(기존 호출부 무회귀).
- 정적 SPA(`@fastify/static` 와일드카드) register **이전**에 등록 — 정확 경로가 와일드카드보다 먼저 매칭되도록 하는 기존 순서 규약 유지.
- **입력 검증**: `RpcBody = z.object({ method: z.string().min(1), params: z.record(z.unknown()).optional() })`. `method` 누락/빈 문자열 → 400 `invalid body`. `params`는 객체(Record)만 — `CRpcClient.callRpc(method, params?)` 계약과 정합(배열/위치 인자 method는 범위 밖).
- **502 graceful**: `rpc.callRpc`/`getCatalog`가 throw(예: Unity 13110 미가동, `RpcClientError`) → 502로 흡수하고 에러 메시지를 그대로 반환. 서버는 죽지 않는다.
- **controlToken 게이트**: `POST /rpc`는 `viewer.controlToken`이 설정돼 있으면 `x-viewer-token` 헤더 일치를 요구(불일치 403). 기존 `/viewer/api/move` 라인과 동일한 **선택 게이트**(controlToken 미설정 시 통과). `GET /rpc/catalog`는 무게이트 — 토큰 없이 method 드롭다운을 로딩하는 현 UI 흐름을 유지하기 위함.

### 2.2 재사용(무변경) 자산
- `CRpcClient.callRpc(method, params?, timeoutMs?): Promise<unknown>` — 시그니처 그대로 사용, **클라이언트 코드 무변경**.
- `CRpcClient.getCatalog(): Promise<{ methods: string[] }>` — 그대로.
- `toolsConfig.unityRpc`(baseUrl `http://localhost:13110`, timeoutMs) — 무변경.

### 2.3 배선(ViewerDeps → server → index)
- `src/viewer/routes.ts`: `ViewerDeps`에 `rpc?: CRpcClient` 옵셔널 필드 추가(`routes.ts:15`), `import type { CRpcClient }`.
- `src/api/server.ts`: `ApiDeps`에 `rpc?: CRpcClient` 추가(`server.ts:92`), 뷰어 등록 블록에서 `registerViewerRoutes(instance, { sources, viewer, rpc: deps.rpc, llm: deps.brain })`로 전달(`server.ts:279`).
- `src/index.ts`: `const rpc = new CRpcClient(tools.unityRpc);`(MCP 서버와 동일 생성 방식, `index.ts:71`) → `buildServer({ ..., rpc })`(`index.ts:79`).

### 2.4 프론트 RPC 콘솔(옵션 탭)
- `web/index.html`: 옵션 탭에 RPC 콘솔 `panel-section` 추가 — `#viewer-token`(controlToken 입력), `#rpc-method`(catalog 드롭다운), `#rpc-params`(JSON textarea), `#rpc-reload`/`#rpc-call` 버튼, `#rpc-result`(결과 pre). 신규 상단 탭은 만들지 않음(레이아웃 침습 최소).
- `web/app.js`:
  - `tokenHeaders(base)` — `#viewer-token` 값이 있으면 `x-viewer-token` 헤더 부착(`app.js:1623`).
  - `callRpc(method, params)` — 같은 오리진 `POST /viewer/api/rpc`(`app.js:1629`).
  - `loadRpcCatalog()` — `GET /viewer/api/rpc/catalog` → `#rpc-method` 옵션 채움. catalog가 비면 "Unity 미가동?" 안내(`app.js:1641`).
  - `runRpcCall()` — params 텍스트를 `JSON.parse`(빈 값이면 undefined) 후 `callRpc` → `HTTP <status>` + 결과 JSON 표시(`app.js:1660`).
  - 옵션 탭 진입(`setTab('options')`) 시 `loadRpcCatalog()` 1회 호출(`app.js:1815`), `#rpc-reload`/`#rpc-call` 버튼 결선(`app.js:1983`).
- `web/app.css`: `.rpc-result`(JSON pre 스타일) 추가.

## 3. 스트림 2 상세 — LLM 멀티 프로바이더

### 3.1 config 스키마 확장(`src/config/llmConfig.ts`)
- `LlmSchema.provider` enum에 **`'gemini'` 추가**(기존 `qwen3|gemma|claude|codex|openai-compatible` → `+gemini`). `provider`는 warmup 게이트(claude/codex 스킵)에만 영향.
- `LlmModelProfileSchema = LlmSchema.extend({ id: z.string().min(1), name: z.string().min(1) })` — 프로필 = 기존 llm 필드 + 식별자(id/name).
- `LlmConfigSchema`에 `models?: LlmModelProfile[]`, `activeModel?: string` 추가(둘 다 옵셔널 → 하위호환).
- `resolveProfiles(cfg)`(순수 헬퍼, 로더·AgentRuntime 공용):
  - `models` 없음/빈 배열 → `profiles = [{ ...cfg.llm, id:'default', name: cfg.llm.model }]`, `activeId = 'default'`.
  - `models` 있음 → `active = profiles.find(p => p.id === cfg.activeModel) ?? profiles[0]`, `activeId = active.id`(불일치 시 첫 프로필 폴백).
- `DEFAULT_LLM` 상수를 추출해 `DEFAULT_LLM_CONFIG`에 `models:[default]`·`activeModel:'default'` 포함(신 필드도 기본값 만족 → `config.test.ts`의 `toEqual(DEFAULT_LLM_CONFIG)` 유지).

### 3.2 하위호환 alias(무회귀 핵심)
- `normalizeActiveProfile(cfg)`: `resolveProfiles`로 활성 프로필을 구한 뒤 **`cfg.llm`을 항상 활성 프로필의 alias로 정규화**한다(id/name을 제거한 순수 `LlmSchema` 형태). 동시에 `models`·`activeModel`도 채운다.
- `loadLlmConfig`: 기존 섹션 머지는 유지하되 `models`/`activeModel`은 배열/문자열이라 섹션 머지 대상이 아님(raw 값 그대로 통과). parse 후 `normalizeActiveProfile`로 정규화하여 반환.
- 효과: 레거시 리더(`index.ts`의 `llm.llm.enabled` 로그), 기존 테스트, brain 소비처의 `cfg.llm.*` 접근이 **모두 무회귀**로 활성 프로필을 본다.

### 3.3 AgentRuntime — 인스턴스 교체 없는 활성 프로필 스왑(`src/brain/AgentRuntime.ts`)
- `implements SetupBrain, LlmModelSelector`.
- 생성자: `resolveProfiles(cfg)`로 `this.profiles`/`this.activeId` 확정 + `ensureClient()`. **self-normalize** — `models` 없는 레거시 cfg(e2eSmoke·다수 테스트)도 default 프로필 1개로 정상 동작(`AgentRuntime.ts:53`).
- `private get active(): LlmModelProfile` — 활성 프로필(id/name + 모든 llm 필드) 조회(`AgentRuntime.ts:62`).
- `private ensureClient()` — 활성 프로필이 `enabled`이고 `activeId !== lastBuiltId`이면 `new OpenAI({ baseURL, apiKey, timeout })`로 (재)빌드, `lastBuiltId` 갱신. 비활성이면 `client`를 해제(`undefined`)(`AgentRuntime.ts:67`). `apiKeyEnv`는 전환 시점에 env에서 재읽기, 미설정이면 `'not-needed'` 폴백.
- 본문 `this.cfg.llm.*`(model/baseUrl/apiKeyEnv/temperature/maxTokens/enabled/provider/timeoutMs/api/think/imageMaxEdge, 약 18곳) → **`this.active.*`로 치환**. 전역 블록(`cfg.warmup`/`cfg.setupPrompts`/`cfg.floorRoi`/`cfg.occupancy`/`cfg.centering`)은 프로필 단위가 아니므로 **무변경**.
- `get enabled()` → `this.active.enabled`.
- `LlmModelSelector` 구현:
  - `listModels()` → `{id,name,provider,model,active}[]`(활성만 `active:true`)(`AgentRuntime.ts:86`).
  - `selectModel(id)` → 존재 검증(없으면 `{ok:false}`·활성 불변), 있으면 `activeId` 갱신 + `ensureClient()`로 클라 재빌드/해제 → `{ok:true, activeModel:id}`(`AgentRuntime.ts:96`).

### 3.4 `LlmModelSelector` 인터페이스(신규 `src/brain/llmRegistry.ts`)
- 뷰어 라우트가 `AgentRuntime` 구상 타입에 결합되지 않도록 최소 계약만 노출:
  ```ts
  export interface LlmModelSelector {
    listModels(): { id: string; name: string; provider: string; model: string; active: boolean }[];
    selectModel(id: string): { ok: boolean; activeModel?: string };
  }
  ```

## 4. 데이터 흐름

### 4.1 RPC 프록시
```
브라우저(web/app.js callRpc)
  └─ POST /viewer/api/rpc  { method, params? }  (+ x-viewer-token, 같은 오리진 13020)
       └─ SettingAgent Fastify (routes.ts) — controlToken 게이트 → zod 검증
            └─ CRpcClient.callRpc(method, params)   (MCP 서버와 동일 클래스)
                 └─ Unity CRpcServer(13110) /rpc
       ← { ok:true, result } / 502 { ok:false, error }(Unity 미가동 등)
catalog: GET /viewer/api/rpc/catalog → CRpcClient.getCatalog() → 13110 → { methods:[...] }
```

### 4.2 LLM 프로필
```
config/llm.config.json (models[] + activeModel)
  └─ loadLlmConfig → normalizeActiveProfile(resolveProfiles)
       └─ cfg.llm = 활성 프로필 alias, cfg.models/activeModel 채움
            └─ new AgentRuntime(cfg) — profiles/activeId 소유, ensureClient()
                 └─ 소비처(SetupBrain·FloorRoiReviewer·OccupancyReviewer·PtzCalibrator 등)는 active 프로필로 호출

전환: 브라우저 → POST /viewer/api/llm/select {id}
  └─ AgentRuntime.selectModel(id) → activeId 스왑 + ensureClient()(클라 재빌드/해제)
       (메모리 전환 — 파일 미기록. 재시작 시 config activeModel 로 복귀)
```

## 6. 한계(은닉 없이)

- **실 원격 LLM 미검증**: claude/gemini/codex는 `enabled:false` 예시이며 gemini/codex 모델ID는 자리표시자(`gemini-2.5-flash`/`gpt-4o`). 실 키·계정 없이 라이브 호출은 미검증 — 유닛(로컬 목 서버)으로 클라 재빌드 경로(baseUrl 스왑)만 검증.
- **모델 전환 비영속**: 메모리 스왑만 — 파일 미기록. 서버 재시작 시 config `activeModel`로 복귀.
- **config 핫리로드 없음**: `config/`는 nodemon 감시 대상이 아니므로 프로필 추가/수정 시 서버 재시작 필요.
- **RPC 무인증 개방 가능성**: `viewer.controlToken`이 빈 문자열(기본값)이면 게이트가 스킵되어 RPC 프록시가 무인증 개방된다(설계서 §8-1). 리더 확정 "controlToken 게이트만"에 따라 강제화는 하지 않되, 운영 시 controlToken 설정을 권고한다.
- **실 Unity 13110 RPC 실행 성공**: 유닛은 fake `CRpcClient`로 계약(인자/응답/502)만 검증. 실 method 실행 성공은 리더 라이브 실증(catalog·system.ping=200)으로 대체. Unity 미가동 시 502 graceful은 라우트 로직상 보장.
- **비전 미지원 프로필**: 텍스트 전용 프로필로 전환 시 floor ROI 등 비전 작업은 기존 graceful null 폴백으로 흡수(사용자 책임).
- **자연어 채팅(LLM이 RPC를 선택 호출)은 범위 밖**: 이번은 결정형 콘솔·전환까지. tool-calling 오케스트레이션은 후속(§7 my_think 반영).
- **params 형태 제한**: RPC params는 객체(Record)만 지원(현 `CRpcClient`/Unity 계약). 배열/위치 인자 method는 별도 논의 필요.

## 7. 사용법 메모

- **새 프로바이더 활성화**(재시작 경로): `config/llm.config.json`의 대상 프로필 `enabled:true` + 해당 `apiKeyEnv` 환경변수 설정(예: `ANTHROPIC_API_KEY`) → 서버 재시작.
- **런타임 전환**(재시작 없음): 옵션 탭의 모델 드롭다운 선택, 또는 `POST /viewer/api/llm/select {id}`. 단 비영속(재시작 시 activeModel 복귀).
- **RPC 콘솔**: 옵션 탭 진입 시 catalog 자동 로드. controlToken을 운영한다면 `#viewer-token`에 토큰 입력 후 호출.
- **옵션 페이지의 llm 편집 vs 모델 드롭다운**: 기존 옵션 폼은 `llm` 단일 블록을 파일에 저장(서버 재시작 필요), 모델 드롭다운은 런타임 메모리 전환(재시작 불필요) — 역할이 다르다.

## 1. 배경 / 목표

다른 팀이 상용 모델(Opus)로 "차량이 실제 점유한 바닥 사각형(4점 OBB)"을 육안 정합 수준으로 국소화하는 데 성공했다. 이번 작업의 목표는 **상용 전환이 아니라**, 로컬 vLLM `Qwen/Qwen2.5-VL-32B-Instruct`로 **동일한 국소화 정확도**를 얻는 것이다. 셋팅(캘리브레이션) 용도이므로 **지연은 무관**하고 **정확도가 최우선**이다.

산출물 우선순위:
- **floor ROI**(차량 바닥 접지 슬롯 4점) — 최우선.
- **occupancy**(주차면별 폴리곤 + 점유 판정) — 부수 개선.
- **centering**(PTZ 자문) — 좌표를 생성하지 않으므로 소폭만.

## 2. 근본 원인 (3중 결함)

| # | 결함 | 내용 |
|---|---|---|
| 1 | **입력 해상도 부족** | 긴변 960 다운스케일(1920×1080→960×540, ≈0.52M px). Qwen 처리 상한(~1.0M) 대비 절반. 원거리·기울어진 접지선을 픽셀 단위로 찍기엔 해상도가 낮아 국소화 정밀도 급감. |

교정 방향: **해상도 상향(A) + 픽셀 좌표 출력→코드가 정규화 변환(B) + 그라운딩 기법 프롬프트(C)**. 변환은 **AgentRuntime 경계에서만** 흡수 → 다운스트림(floorRoi.ts / reviewer / geometry / 반환타입) 무변경.

## 3. 해결책 상세

### 3-A. 해상도 상향 — smart-resize 28정렬 리사이저

**신규 `smartResizeJpegBase64(b64, maxLongEdge)` (`src/util/image.ts`)** — 반환 `{ base64, width, height }`.
- 상수 `FACTOR=28`, `MAX_PIXELS = 1280*28*28`(≈1,003,520). 헬퍼 `roundToFactor`(가까운 배수 반올림)·`floorToFactor`(상한 엄수용 내림).
- 결정형 알고리즘:
  1. sharp metadata로 원본 (W0,H0). 크기 못 읽으면 throw.
  2. `scale = min(1, maxLongEdge / max(W0,H0))` — **업스케일 없음**.
  3. 각 변을 28의 배수로 반올림(`roundToFactor`).
  4. `W*H > MAX_PIXELS`이면 `√(MAX/(W*H))` 비율로 재축소 후 **28의 배수로 내림**(`floorToFactor` — 상한 엄수, 서버 재축소 원천 차단).
  5. `sharp.resize(W,H, fit:'fill').jpeg(q80)` → **실제 전송 (W,H) 반환**(추정 아님, 재인코딩 실측치). 이 값이 좌표 정규화 기준이다(원본 아님).
- **왜 28정렬인가**: 두 변이 28의 배수이고 픽셀 수 ≤ max_pixels이면, vLLM 내부 smart-resize가 **항등**(크기 불변)이 된다. ⇒ "모델이 반환하는 픽셀 좌표계 == 우리가 보낸 이미지(W×H)". "리사이즈된 vs 원본" 좌표계 애매성이 원천 소거된다.
- 기존 `downscaleJpegBase64`는 **그대로 유지**(stage1/centering 등 비-그라운딩 경로가 계속 사용, 28정렬 불필요).

**`config/llm.config.json`**: `imageMaxEdge` 960→**1288**(=46×28, 16:9 시 1288×728 ≈ 0.94M < max_pixels 1.0M). 스키마 default(960)는 불변 — 실제 config 값만 상향.

### 3-B. 좌표 형식·변환 (핵심) — 픽셀 그라운딩 → 경계에서 정규화

**`src/brain/SetupBrain.ts` — RAW(절대픽셀) 파싱 스키마 추가**(반환 스키마·타입 불변):
- `FloorRoiRawSchema = { points_2d?: 4×[x,y], bbox_2d?: [x1,y1,x2,y2], confidence }` + `.refine(points_2d || bbox_2d)` — 둘 다 없으면 parse 실패 → chatJson 재시도 → null → 결정형 폴백.
- `OccupancyRawSpaceSchema`(`id, occupied, points_2d?, bbox_2d?`) + `OccupancyRawSchema`(`spaces default([]), confidence`). occupancy는 refine 없음 → 폴리곤 미보유 면도 graceful 통과(optional 계약 계승).
- **불변**: `FloorRoiResultSchema`(정규화 {x,y}×4)·`OccupancyResultSchema`·반환타입 `FloorRoiResult`/`OccupancyJudgment`. RAW는 파싱 전용, 정규화 스키마는 반환 shape 기준.

**`src/brain/AgentRuntime.ts` — pre-resize + 픽셀 파싱 + 정규화 변환**:
- import 조정: `smartResizeJpegBase64`(image.js), `normalizeBox/normalizeQuad/rectToQuad`(geometry.js) 추가. 고아화된 `FloorRoiResultSchema`/`OccupancyResultSchema` import 제거(RAW 스키마로 대체 — 내 변경이 만든 고아만 정리).
- **신설 헬퍼 `prepareGroundingImage(b64)`**: `smartResizeJpegBase64(cfg.llm.imageMaxEdge ?? 1288)` 호출, 실패 시 `logger.warn` + **null**(그레이스풀 스킵 — reviewer가 이미 null/폴백 처리).
- **`recognizeFloorRoi`**: `prepareGroundingImage` → `(base64, W, H)` → 템플릿에 `imgW=W, imgH=H` + **vehicle를 전송이미지 픽셀 bbox** `vehiclePx=[round(x·W),round(y·H),round((x+w)·W),round((y+h)·H)]` 주입 → `chatJson(FloorRoiRawSchema.parse, base64, jsonMode=true, floorRoi.timeoutMs, prepared=true)` → RAW→변환: `points_2d ? normalizeQuad(pts,W,H) : rectToQuad(normalizeBox(bbox_2d,W,H))` → `{ polygon, confidence }`(=FloorRoiResult).
- **`judgeOccupancy`**: 동일하게 pre-resize + `imgW/imgH/expected` → `chatJson(OccupancyRawSchema.parse, base64, jsonMode=false, occupancy.timeoutMs, prepared=true)` → 각 space의 픽셀 points/bbox를 정규화(둘 다 없으면 polygon 생략) → `{id,occupied,polygon?}` 매핑. **occupiedCount/total/rate 산술은 결정형 그대로**.
- **`adviseCentering`**: 좌표 생성이 아니므로 형식 전환 없음. 기본 chat() 경로로 다운스케일되며 imageMaxEdge↑ 혜택만 공유.
- **`chat`/`chatJson` `prepared` 플래그(기본 false) 추가**: `chat` 내부 `if (image && !prepared && cfg.llm.imageMaxEdge)` — **prepared=true면 재다운스케일 건너뜀**(좌표계 불일치 방지). 그 외 경로(stage1/centering)는 기존 downscale 동작 유지.

### 3-C. 프롬프트 — 절대픽셀 그라운딩 재작성

- **`config/prompts/floor_roi.yaml`**: 정규화 서술형 → **절대픽셀 `points_2d` 그라운딩** 전체 재작성. `You are a visual grounding expert...` 영어 지시 + 도메인 규약(앞=타이어 접지선/뒤=슬롯 먼끝, 지붕·차체높이 제외) 한국어 유지. 출력 `{"points_2d":[[x,y]×4],"confidence"}`([front-left,front-right,rear-right,rear-left]). 치환키 `{{vehicle}}`→`{{vehiclePx}}`, `{{imgW}}`/`{{imgH}}` 신규.
- **`config/prompts/occupancy.yaml`**: 정규화 4점 → **절대픽셀 `points_2d`**. 전 주차면 나열 + occupied + recall 우선 규약 유지. `{{imgW}}/{{imgH}}` 신규. jsonMode **off 유지**(다면 디코딩 가속·extractJson 회수, 기존 선례).
- **`config/prompts/ptz_centering.yaml`**: system 첫 문단만 그라운딩 어투로 소폭 명료화. **스키마·치환키·의미 불변**(좌표 출력 없음 → 픽셀 전환 없음).
- jsonMode 정책: floor=json_object 유지(단일 소형 top-level object → vLLM guided decode 호환), occupancy=off 유지.

### 3-D. config 스키마 — floorRoi.timeoutMs 신설

- **`src/config/llmConfig.ts`**: `FloorRoiSchema.timeoutMs: z.number().int().positive().default(120000)` 추가(occupancy 선례). 고해상도 32B floor 그라운딩이 `llm.timeoutMs`(30s)를 초과하는 문제 대응. `DEFAULT_LLM_CONFIG.floorRoi.timeoutMs=120000` 반영.
- **`config/llm.config.json`**: `floorRoi.timeoutMs = 300000`(5분, occupancy와 동일 여유).
- **하위호환**: `.default(120000)`라 기존 config(timeoutMs 생략)도 파싱 성공. `loadLlmConfig` spread 병합 그대로 커버.

## 4. 좌표 변환 파이프라인 (데이터 흐름)

```
원본 JPEG(예 1920×1080)
 → smartResizeJpegBase64(1288)         [image.ts]        → base64 + (W=1288, H=728)  ← 전송크기 실측 확보
   (28의 배수 & ≤ MAX_PIXELS 1,003,520)
 → 프롬프트: imgW/imgH + vehiclePx(픽셀 bbox)   [AgentRuntime]
 → chatJson(prepared=true) → vLLM Qwen2.5-VL-32B   → RAW 픽셀 {points_2d[[x,y]×4] | bbox_2d[x1,y1,x2,y2], confidence}
 → normalizeQuad(pts,W,H)  또는  rectToQuad(normalizeBox(bbox,W,H))   [AgentRuntime 경계 = 픽셀→정규화 0~1 흡수점]
 → FloorRoiResult.polygon / OccupancyJudgment.spaces[].polygon (정규화, 반환타입 불변)
 → resolveFloorPolygon → normalizePolygon(convexHull+캐노니컬 정렬+마진)   ※ floorRoi.ts / reviewer / geometry / store / viewer 전부 무변경
```

핵심 규약:
- **정규화 기준 = "보낸 이미지 (W,H)"**(원본 1920 아님). 다운스케일을 코드가 하므로 정확히 앎 — LpdClient가 캡처 해상도로 normalizeQuad 하던 것과 동일 패턴.
- **점 순서 강건**: Qwen 코너 순서가 규약과 달라도 다운스트림 `orderConvexCanonical`이 [앞왼,앞오,뒤오,뒤왼]로 재정렬. 값 정합만 보장하면 됨.
- **하위호환 폴백**: RAW 파서가 `points_2d` 부재 시 `bbox_2d` 수용, 둘 다 없으면 스키마 실패 → 재시도 → null → 결정형 폴백. 회귀 경로 보존.

## 5. Qwen2.5-VL 그라운딩 근거 (출처 명시)

- **좌표계**: Qwen2.5-VL은 **절대 픽셀**로 그라운딩하도록 학습. bbox는 `[x1,y1,x2,y2]`(코너). 반환 좌표는 모델이 실제 처리한(smart-resize된) 해상도 기준. → 본 설계는 전송 이미지를 28정렬 고정점으로 만들어 이 애매성을 **항등화**로 회피.
- **smart-resize**: 각 변을 28의 배수(패치 stride 14)로 리사이즈, 처리 픽셀 수를 `[min_pixels, max_pixels]`로 클램프. 검출 기본값 `max_pixels = 1280*28*28 ≈ 1.0M`. 16:9 기준 ~1.0M(예 1288×728)이 상한 근방.
- **출력 형식**: 네이티브는 JSON `{"bbox_2d":[...]}` 배열 + point 그라운딩. **회전 사변형/OBB/폴리곤 네이티브 출력은 문서화되지 않음** → points_2d 4코너는 상대적으로 OOD(리스크 §8-2).

출처:
- https://huggingface.co/Qwen/Qwen2.5-VL-7B-Instruct/discussions/13 — 좌표는 리사이즈 이미지 기준, 28의 배수, bbox_2d=[x1,y1,x2,y2].
- https://pyimagesearch.com/2025/06/09/object-detection-and-visual-grounding-with-qwen-2-5/ — min/max_pixels=256/1280*28*28, processor가 리사이즈 담당.
- https://deepwiki.com/jzh15/Qwen2.5-VL/8.2-object-detection-and-grounding, GitHub issue #866, arXiv 2502.13923(기술리포트) — box + point 그라운딩 능력.

## 8. 미확정 / 검증되지 않은 항목 (통과 위장 금지)

- **실 Qwen2.5-VL 국소화 품질(육안)**: 자동 미검증. **리더 라이브 스모크 진행 중 — 결과 별도**. floorRoi.enabled=true 실제 프레임 1~3장 → 오버레이 폴리곤이 차량 접지면과 정합하는지, occupancy 면별 폴리곤 육안, 좌표 sanity(0~1·4점·앞=y큰쪽) 확인 필요.
- **vLLM 서버 max_pixels ≥ 1.0M 가정**: 전송(≈0.94M)이 서버 max_pixels보다 크면 서버가 재축소 → 좌표 불일치. 어긋나면 `imageMaxEdge` 하향 필요. **서버 기동 옵션의 min/max_pixels 값 확인 필요**.
- **points_2d(OBB) 신뢰도**: Qwen 네이티브는 축정렬 bbox/point가 더 안정적, 임의 사변형 4코너는 상대 OOD. 실측 부정확 시 후속 대안 = bbox_2d 축정렬 우선 + 번호판 각도(`buildPlateAnchoredQuad` 자산)로 회전 재구성 하이브리드. RAW 파서가 이미 bbox_2d를 수용하므로 프롬프트 전환만으로 가능. **스모크 육안 후 결정.**

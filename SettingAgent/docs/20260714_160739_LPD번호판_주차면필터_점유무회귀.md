## 1. 규칙과 그 근거

```
keepPlate(p)  =  (A) 유지된 VPD 차량 중 하나에 귀속        OR   (B) 번호판 중심(quadCentroid) ∈ 주차면 폴리곤
                  ── matchPlatesToSlots 재사용 ──               ── pointInPolygon 재사용 ──
```

### 1.1 (A) 귀속 항 — `matchPlatesToSlots` 재사용 (신규 매칭 0줄)

`src/setup/plateMatch.ts:14 matchPlatesToSlots(slots, plates)` 가 이미 정확히 이 규칙이다: *번호판 중심이 차량 rect 안 → 후보, 그중 **겹침 최대** 차량에 귀속, **차량당 번호판 1개***.

**차량당 1개 규칙이 결정적이다.** VPD 는 가끔 `conf 0.39` / `(77,0)-(1380,716)`(프레임 절반) 짜리 **거대 병합 박스**를 뱉는다(후속 과제 V-1). 귀속을 *"번호판 중심 ∈ 유지된 차량 rect"* 로 **직접 구현했다면 그 박스가 배경 번호판을 전부 빨아들인다.** `matchPlatesToSlots` 는 차량당 1개로 제한하므로 **피해가 정확히 1건으로 봉쇄**된다.

> 리더 라이브가 이 근거를 **실증**했다(`09_leader_empirical_lpd.md` §3): 모드 A 렌더에도 거대 청록 박스는 남아 있으나 **번호판은 4장만 유지**됐다 — 그 박스가 배경 번호판을 빨아들이지 않았다.
> 유닛테스트 P4 가 이 성질을 봉인한다(거대 박스 + 배경 번호판 5개 → **정확히 1개만** 유지). **자체 귀속 로직으로 "단순화"하면 즉시 깨진다.**

### 1.2 (B) 주차면 보정 항 — **없으면 점유가 뒤집힌다** (OR 의 존재 이유)

점유 판정은 **번호판 중심 기반**이다 — `web/core.js:454 computeOccupancy(폴리곤, plates)` 는 *번호판 중심이 그 폴리곤 안이면 `occupied=true`*. 소비처 2곳(`web/app.js:338 updateLogicOccupancy`, `web/core.js:581 buildFlatSlotRows`/`buildGlobalPlaces`)이 **같은 union**(`[...detect.plates, ...detect.vehicles.map(v=>v.plate)]`)을 넣는다.

> **엄격 귀속(A)만 썼다면**: VPD 가 주차차를 놓친 프레임 → 그 차의 번호판이 어떤 kept 차량에도 귀속되지 않음 → **삭제** → `computeOccupancy` 가 그 주차면을 **`occupied:false` 로 뒤집는다**(있는 차가 없다고 나온다).

(B)가 이 회귀를 막는다. 이것이 1차 문서 §6 이 LPD 필터를 기각했던 바로 그 위험이며, **OR 규칙이 그 위험을 해소했기 때문에 결정을 번복할 수 있었다.**

## 2. 신규 코드

### 2.1 `src/domain/geometry.ts` — `quadCentroid` 신규 (D-1 수정으로 추가)

```ts
/** quad 4점의 **산술평균** 중심. `web/core.js:quadCentroid` 와 **동일 정의**(파리티: test/quadCentroidParity.test.ts). */
export function quadCentroid(q: NormalizedQuad): NormalizedPoint;
```

**존재 이유는 미용이 아니다.** 점유 판정(`computeOccupancy`)이 번호판 중심을 이 정의로 잡으므로, **점유에 관여하는 서버 로직은 반드시 이 함수를 써야 한다.** 정의가 갈리면 점유가 조용히 뒤집힌다 — 그게 §5 의 D-1 이다.
`polygon.ts:polygonCentroid`(면적가중)·`center(quadBoundingRect(q))`(bbox 중심)와 **값이 다르다.** 세 정의는 중심대칭 quad(직사각형·회전 OBB)에서만 일치한다.

### 2.2 `src/capture/onPlaceFilter.ts` — `filterPlatesOnPlace` 신규 (~38줄)

```ts
export function filterPlatesOnPlace(
  plates: PlateBox[],
  keptVehicles: readonly { rect: NormalizedRect; confidence: number }[],
  polys: readonly (readonly NormalizedPoint[])[] | null | undefined,
): { kept: PlateBox[]; filteredOut: number; degraded: boolean };
```

- **(A)**: `matchPlatesToSlots(keptVehicles→BuiltSlot 어댑트, plates)` 의 반환 Map 값 집합(=`plate.quad` **참조**)을 `Set` 으로 만들어 `attached.has(p.quad)` 로 판정.
- **(B)**: `quadCentroid(p.quad)` → `pointInPolygon`. **소비처와 동일 중심 정의**(D-1 수정 후).
- **강등**: `polys` 부재/빈 배열 → **전량 통과 + `degraded=true`**(드롭 금지 — 기준이 없다는 이유로 데이터를 조용히 지우지 않는다. `filterVehiclesOnPlace` 와 동일 정책, HANDOFF §2-3).
- **BuiltSlot 어댑터를 함수 안에 둔다** → 두 호출측(`detectPipeline`/`CaptureJob`)에 어댑터가 복제되지 않는다. `plateMatch.ts` **무변경**.
- **(A) 는 여전히 bbox 중심**(`matchPlatesToSlots` 내부 규칙). 의도적이다 — 그건 *차량 귀속*이지 점유가 아니다. 통일 대상 아님. 다만 "서버 안에 중심 정의가 2개"라는 사실은 주석에 명시했다.

**의도적으로 수용한 중복**: `runDetect` 는 `matchPlatesToSlots` 를 **2회** 호출한다(자기 매칭 1회 + 필터 내부 1회, 동일 입력·동일 결과). 순수 함수 O(n·m), n,m ≤ ~20. `matched` 맵을 인자로 넘기는 대안은 `CaptureJob` 에 어댑터를 복제해야 해서 더 나쁘다. **그대로 뒀다.**

## 3. 배선

```
[라이브검출]  POST /capture/detect  (captureRoutes.ts — 무변경, 이미 polys·onlyOnPlace 전달 중)
   └─ detectPipeline.runDetect
        vehicles = filterVehiclesOnPlace(raw, polys)          ← 1차 (기존)
        matched  = matchPlatesToSlots(vehicles, platesBase)   ← 무변경 (이미 필터된 vehicles 로 호출 중)
        if (onPlaceOnly)                                      ← 게이트 = **실제 적용된 모드**(요청값 아님)
          plates = filterPlatesOnPlace(platesBase, vehicles, polys).kept   ← 신규
        return { plates, summary: { lpdCount: platesBase.length, lpdFilteredOut, ... } }

[정밀수집]   CaptureJob.captureTarget()
        rawPlates = lpd.detect(jpg)
        plates    = vpdOnParkingOnly ? applyPlateFilter(rawPlates, vehicles, t) : rawPlates
        → insertDetections(kind:'plate')   ← **DB 저장분이 여기서 줄어든다**
```

- 게이트는 `onPlaceOnly`(= 필터가 실제 적용된 모드)를 쓴다 — `onPlace.onlyOnPlace`(요청)가 아니다. **강등 시 차량과 같은 이유로 번호판도 필터하지 않는다.**
- 강등 사유·warn 은 **차량 필터가 이미 낸다** → 번호판용 별도 강등 필드·로그 **없음**(중복 금지). `vpdOnPlaceDegraded` 하나로 공유(폴리곤 소스가 같으니 사유도 같다).
- `CaptureJob` 은 폴리곤 조회를 `presetPlace(t)` 로 추출해 **두 필터가 공유**한다(`placePromise` 캐시 — 파일 I/O 재발생 없음).
- **`vehicles[].plate` 무변경**: 귀속분은 `matched`(이미 필터된 vehicles 로 산출)라 정의상 kept 차량 것뿐이고, 복원분(`recovered`)은 zoom 뷰에서 새로 얻은 quad 라 `platesBase` 에 원래 없다. 회귀 0.

## 5. ★ D-1 — 설계서 «점유 불변» 증명은 **거짓이었다** (반증 → 수정 → 파리티 봉인)

### 5.1 설계서의 주장

> 「(B)는 중심이 폴리곤 안인 번호판을 전부 유지한다 → **필터가 제거하는 번호판은 점유를 참으로 만들 수 없는 것뿐** → `computeOccupancy` 결과는 필터 전후 **동일하다(불변)**」 — `06_architect_plan_lpd.md` §1

### 5.2 검증자의 반증 — **중심 정의가 서버와 뷰어에서 갈려 있었다**

| | 정의 | 위치 |
|---|---|---|
| 서버 필터 (B)항 | `center(quadBoundingRect(quad))` = **bbox 중심** | 구 `onPlaceFilter.ts` |
| **소비처** 점유 판정 | `quadCentroid` = **4점 산술평균** | `web/core.js:434` |

→ **4점평균 ∈ 폴리곤 · bbox중심 ∉ 폴리곤** 인 quad 는 서버가 **드롭**하는데, 뷰어 기준으론 **점유를 참으로 만들 수 있었다** → 점유가 **`true → false` 로 조용히 뒤집힌다**(경고 없음). 설계서의 "증명"은 **두 술어가 같다고 암묵 가정**한 데서 무너졌다.

검증자 반례(`08_qa_report_lpd.md` §3.2):
```
quad [(0.40,0.31), (0.44,0.31), (0.42,0.31), (0.42,0.80)]
  4점평균 y = 0.4325  → BACK_ROW(y 0.30~0.45) 안  → 모드B 점유 true
  bbox중심 y = 0.555  → BACK_ROW 밖              → 모드A 드롭 → 점유 false
```

### 5.3 이것은 **이론적 구멍이 아니었다**

구현자가 수정 후 "새 테스트가 구 구현을 실제로 잡는가"를 확인하려고 (B)를 bbox 중심으로 되돌리고 돌렸다 — **경계 근방 무작위 quad 300개 중 "드롭됐지만 점유를 참으로 만들었을" 번호판이 10건** 검출됐다(`10_developer_fix_d1.md` §6). 손으로 만든 quad 1개짜리 구멍이 아니라 **재현성 있게 도달 가능한 결함**이었다.

### 5.4 수정 — (B)항을 소비처와 같은 정의로

`src/domain/geometry.ts` 에 `quadCentroid`(4점 평균) 신규 → (B)항이 이를 사용. **`web/core.js` 무변경**(서버를 소비처에 맞췄다). **(A) 항 무변경**(차량 귀속이지 점유가 아니다).

```ts
// onPlaceFilter.ts:87-91
const kept = plates.filter((p) => {
  if (attached.has(p.quad)) return true;   // (A) 무변경
  const c = quadCentroid(p.quad);          // (B) ← 소비처와 **동일** 중심 정의
  return polys.some((poly) => pointInPolygon(poly, c));
});
```

### 5.5 해소 근거 — 주장이 **정의상 참**이 됐다

- (B)가 드롭 ⟺ `quadCentroid(quad) ∉ 모든 주차면 폴리곤`
- `computeOccupancy` 가 `occupied=true` 로 만들려면 ⟺ `quadCentroid(quad) ∈ 어떤 폴리곤`
- 두 조건은 **동일 술어의 부정**이다 → 드롭된 번호판이 점유를 참으로 만들 **수 없다**. **ε 예외 소멸.**

역방향(bbox중심 안 · 4점평균 밖 → 이제 드롭)은 **무해**하다: 그 번호판은 소비처 기준으로도 폴리곤 밖이라 애초에 점유를 참으로 만들지 못한다(테스트 P7 이 이 무해성을 직접 단언).

### 5.7 P7 이 구 버그를 "사양"으로 얼려 놓고 있었다

`test/onPlaceFilter.test.ts` P7 은 원래 `'비대칭 quad 는 quadBoundingRect+center 로 판정(4점 평균이 아니다)'` 를 단언하고 있었다 — **D-1 그 자체를 사양으로 못 박은 테스트**다. 삭제하지 않고 **방향을 뒤집어** 갱신했다: 같은 quad·같은 전제를 유지한 채 결론만 `"(B)는 quadCentroid 로 판정 → 이 skew quad 는 drop 되고, 그 drop 은 무해하다"` 로. **구 사양이 무엇이었고 왜 틀렸는지가 테스트에 남는다.**

## 6. ★ D-2 — 설계서 «aggregate 산출 불변» 도 **거짓** (코드 무수정, 문서 주장만 정정)

### 6.1 설계서의 주장

> 「상류 `matchPlatesToSlots`(프레임 단위)와 `Aggregator`(클러스터 대표 단위)는 **같은 규칙**이다. 상류가 버리는 것은 하류도 버렸을 것들이다 → **`aggregate()` 산출은 불변**」 — `06_architect_plan_lpd.md` §2

### 6.2 검증자의 반증 — **입도(granularity) 불일치**

| | 규칙 | 진 번호판의 운명 |
|---|---|---|
| 상류 `matchPlatesToSlots` | **검출 1건 단위** 경쟁(차량당 1개) | **드롭** |
| 하류 `Aggregator` | **클러스터 대표 단위** 매칭 | 드롭되지 않고 **승자 클러스터에 병합** → robust median 을 끌어당김 |

하류는 패자를 **버리지 않고 섞는다.** 한 kept 차량 rect 안에 번호판 2개가 있고 서로 `clusterDist`(0.06) 안이면, 상류가 진 쪽을 지우는 순간 **하류 대표 좌표가 이동한다.**

**실측**(검증자 봉인 테스트 — 진짜 번호판 y중심 0.335 + 근접 오검출 y중심 0.29, 거리 0.045 < clusterDist):

```
모드B(미필터)  plateY = 0.3005   ← 오검출에 오염된 중앙값
모드A(필터)    plateY = 0.323    ← 진짜 번호판 실좌표와 일치
차이 0.0225 정규화 ≈ 1080p 기준 24px  →  AggregatedSlot → Finalizer → parking_slots.lpdJson 에 그대로 전파
```

### 6.3 그래도 `Aggregator.ts`/`Finalizer.ts` **무변경이 옳다**

- **"손실 0"은 참**: 슬롯 소실 0, 차량 대표 rect·support·occupancyRate·status **전부 동일**.
- **방향은 개선**: 모드 A 값이 **정답**(진짜 번호판 좌표), 모드 B 값이 오염된 값이다.
- 따라서 **코드 수정 대상이 아니다. 틀린 것은 설계서 문장뿐이다.**

### 6.4 ⚠️ 그러나 **값은 바뀐다**

> **모드 A run 과 모드 B run 의 `parking_slots.lpdJson` 이 같다고 가정하는 하류 코드·비교는 틀린다.**

현재 그런 가정을 하는 코드는 발견되지 않았으나(→ `_workspace/11_doc_impact_lpd.md` §3), **run 간 `lpdJson` 대조를 하려는 사람은 모드가 같은지 먼저 확인해야 한다.**

## 9. 한계 · 미검증 (은닉 금지)

1. **브라우저 실제 클릭 미검증.** DOM 자동화가 없다. 서빙된 HTML 에 라벨 `주차면 위 차량만 검출(번호판 포함)` 과 `id="cap-vpd-onplace"` 존재는 확인했으나, 사람이 체크박스를 누르는 경로는 여전히 코드 대조뿐이다. **`#cap-vpd-onplace` 를 rename 하면 어떤 테스트도 깨지지 않고 프론트만 조용히 죽는다.**
2. **정밀수집 잡(DB 경로)의 LPD 필터는 라이브 재현하지 않았다.** 유닛테스트(C1~C3)로만 확인했다. 추가 run 은 DB 데이터를 늘리므로 보류. (1차 때 VPD 는 run69 로 DB 끝단까지 확인했다.) 필요 시 1라운드 추가 실행으로 확인 가능.
3. **(A)·(B) 동시 실패 조합**: VPD 미검출 + 번호판 중심이 폴리곤 밖(차량 앞으로 튀어나온 번호판) → **드롭. 경고 없음**(관측 불가). 두 항이 모두 실패하는 유일한 조합. 실데이터에서 발생하지 않았으나 원리적으로 가능하다.
4. **라이브 검증은 cam1 · 이 장면 · preset1~3 기준.** 다른 프리셋·시간대·차량 배치에서의 점유 무회귀는 미확인.
5. **`web/app.js` DOM 렌더 문자열은 봉인하지 않았다.** `cap-msg`·배지 문자열은 DOM 의존이라 테스트 하네스가 없다(기존에도 없음). **데이터 shape 만** 봉인했다(C4·R2).
6. **차량당 1개 규칙의 이면**: 한 kept 차량 rect 안에 번호판이 2개면 겹침 작은 쪽이 (A)에서 탈락한다. 그것이 실제 주차차 번호판이고 폴리곤 밖이면 뷰어 오버레이 표시에서 사라진다(집계 결과는 §6 참조).
7. **거대 병합 박스(V-1) 잔존**: 봉쇄는 "차량당 1개"까지다. 배경 번호판 **1개**는 여전히 통과한다. 근본 처방은 VPD conf 하한이며 이 필터의 일이 아니다.
8. **사후 모드 전환 불가**(1차 §8-4 상속): 검출 시점 필터라 원 번호판이 DB 에 남지 않는다. 모드를 바꾸려면 재수집해야 한다.
9. **#0 ROI 정합 사각지대 상속**: 주차면 ROI 자체가 틀어져 있으면 (B) 항도 **똑같이 틀린다**(경고 없음).

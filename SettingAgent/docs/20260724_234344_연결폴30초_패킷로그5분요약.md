## 1. 배경 — 두 변경의 연결고리

선행 문서(`20260724_225404`)에서 뷰어의 **3fps 스냅샷 폴링 폴백**(스트림 실패 시 편도 고착)을 제거하고 스트림 자체의 지수 백오프 재시도로 대체했다. 그런데 뷰어에는 그와 별개로 카메라 목록·연결 상태를 주기적으로 재수신하는 **상태체크 폴**(`connectionTick`, `web/app.js:4247`)이 남아 있었고, 이 폴 간격이 4초였다. 3fps 폭주는 사라졌지만 4초 간격 반복 패킷은 여전히 로그에 쌓여, 통신 로그로 실제 장애/변화를 추적하기가 번거로웠다.

이번 두 변경은 그 잔여 문제를 해소한다.

1. **`CONN_POLL_MS` 4000 → 30000**(`web/app.js:216`) — 폴 자체의 빈도를 낮춘다.
2. **통신 패킷 로그 5분 요약** — 30초로 줄여도 남는 반복 기록을, 순수 집계기로 "첫 발생 즉시 + 반복은 5분 창 요약"으로 압축한다.

두 변경은 독립적으로도 유효하지만(30초화만 해도 로그 줄 수가 1/7.5로 줄어든다), 함께 적용하면 정상 가동 중 통신 로그가 "즉시 1줄 → 무음 → 요약 1줄 → 즉시 1줄…" 패턴으로 5분당 2줄 수준까지 압축된다.

## 2. 왜 30초인가

`web/app.js:215`의 코드 주석(리더 결정 근거, 그대로 인용):

> `4s → 30s: 라이브 중이면 스트림 onerror/onload 가 연결 상태를 실시간으로 알리므로 폴은 배경 동기화 역할만 한다.`

즉 라이브 스트림이 떠 있는 동안은 연결 단절/복구를 스트림의 `onerror`/`onload`가 즉시 통지하므로(선행 문서의 자동 재시도 로직), `connectionTick` 폴은 카메라 목록·프리셋 변경 같은 "스트림과 무관한 변화"를 뒤늦게라도 반영하는 배경 동기화 역할로 격하된다. 이 역할에는 초 단위 정밀도가 필요 없다.

**30초를 상한으로 잡은 근거**: 그 이상으로 늘리면 Unity가 새로 기동되어 카메라 목록이 바뀌어도 뷰어가 이를 반영하기까지의 지연이 사용자 체감 상 길어져, 사용자가 수동 새로고침(F5)을 하게 만드는 지점에 이른다. 30초는 "패킷 감소"와 "목록 자동 반영 지연"의 균형점으로 정해졌다.

## 3. 통신 패킷 로그 5분 요약 — 정책과 집계 키

### 3.1 요약 로거 정책 4가지

| # | 정책 | 내용 |
|---|------|------|
| 1 | 첫 발생 즉시 | 같은 키의 트래픽이 창을 새로 열 때(또는 재무장 시) 첫 건은 **즉시** 기존 스키마로 기록한다 |
| 2 | 반복만 집계 | 같은 창 안에서 반복되는 **성공** 건은 방출하지 않고 개수·소요시간만 누적한다 |
| 3 | 실패·비-2xx 항상 즉시 | 예외(`err`)이거나 상태코드가 2xx가 아니면 창이 열려 있어도 **항상** 즉시 기록한다(집계로 숨기지 않는다) — 진단에 필요한 신호를 보존하기 위함 |
| 4 | `n<=1` 미요약 | 창이 만료돼도 그 창에 반복이 없었으면(시도 1건뿐) 요약을 방출하지 않는다 — `win`을 빼면 즉시 줄과 정보가 같아, 저빈도 엔드포인트의 로그가 오히려 2배가 되는 것을 막는다(QA 지적 L1 반영) |

### 3.2 집계 키와 그 이유

```
key = `${method} ${url.split('?')[0]}${op ? '#' + op : ''}`
```

- **쿼리 제거가 필수**: Hucoms 클라이언트는 좌표·`action`을 쿼리 문자열로 실어 보낸다(`HucomsClient.ts:buildUrl`). 쿼리를 키에 포함하면 호출마다 키가 달라져 집계가 영원히 발동하지 않고 `Map`이 무한증식한다. 부수 효과로 쿼리에 실리는 `id`/`passwd` 같은 자격증명이 키에 담기지 않는다.
- **`op` 도입 이유**: `/rpc`는 HTTP 메서드가 항상 `POST`이고 실제 RPC 메서드명(`cam.list`/`cam.setPTZ`/`cam.captureJPG` 등)은 요청 바디 안에 있다. 키를 `method+url`로만 두면 이 세 RPC가 하나의 키로 뭉쳐 "첫 발생 즉시기록" 정책이 그중 하나에만 적용되고, 나머지 RPC들의 최초 발생 시각이 로그에서 사라진다. 선행 문서의 3fps 근본원인 추적(RPC별 최초 발생·빈도로 원인을 특정)과 같은 진단을 다음에도 하려면 이 구분이 필요하다.

호출측에서 `op`를 채우는 곳은 `CRpcClient.callRpc`/`getCatalog`(RPC 메서드명/`'catalog'`)와 `HucomsClient.fetchResponse`(`params.action`) 뿐이며, 나머지 12개 `fetchWithTimeout` 호출자는 `op`를 지정하지 않아 키가 `method+경로`로 유지된다.

## 4. 요약 줄 읽는 법 (운영자용)

요약 줄 예시(QA가 실제 로그 파일에서 확보한 실측 1줄):

```jsonc
{"level":30,"cat":"packet","win":300000,"n":10,"ok":10,"err":0,
 "method":"GET","url":"http://127.0.0.1:13110/rpc","op":"cam.list",
 "msAvg":45,"msMax":49,"msg":"통신 패킷 요약"}
```

| 필드 | 의미 | 산식 |
|------|------|------|
| `win` | 창의 총 길이(ms). `flushAt - windowStart`. sweep이 지연되면(트래픽이 멈췄다 다른 키 트래픽으로 뒤늦게 방출되면) 침묵 구간이 섞여 **부풀 수 있다** | — |
| `span` | 이 창 안에서 실제로 활동이 있었던 구간(ms). `lastAt - windowStart` | — |
| `n` | 창 안의 총 시도 건수(즉시기록된 첫 줄 포함) | — |
| `ok` / `err` | 2xx 성공 / 실패(예외 또는 비-2xx) 건수 | — |
| `msAvg` / `msMax` | 응답 소요시간 평균(정수 반올림)/최대 | — |
| 창 평균 rate | 창 전체를 기준으로 한 평균 초당 건수. **침묵이 섞이면 과소평가된다** | `(n) / win * 1000` |
| **활성 rate** | 실제 트래픽이 있던 구간만의 초당 건수(권장 지표) | `(n - 1) / span * 1000` |
| 침묵량 | 요약이 얼마나 늦게(트래픽 정지 후) 방출됐는지 | `win - span` |

**예제 해석**: 위 줄은 `win=300000`(정확히 5분), `span`이 로그에 없다면(구 스키마 예시) 창 평균 `10/300000*1000 = 0.0333/s` = 정확히 **30초 간격 폴**과 일치한다. `win`과 `span`이 함께 찍히는 최신 스키마에서는 `(n-1)/span*1000`으로 **활성 구간만의** 케이던스를 읽어야 하며, 트래픽이 끊겼다 늦게 방출된 창일수록 `win-span`(침묵량)이 커진다.

## 5. 리더 라이브 관측 실측 (그대로 인용)

리더가 실제 뷰어 탭을 열어둔 상태에서 관측한 로그 타임라인:

```
23:22:12  즉시 1줄 (cam.list)
          — 5분 무음 —
23:27:16  win=303972 n=76 ok=76 err=0 msAvg=10 msMax=32   (요약 1줄)
23:27:16  즉시 1줄
```

**해석**: 5분 구간에 로그는 단 3줄(종전이라면 76줄). 그런데 이 요약에서 역산한 rate는 `76/303972*1000 ≈ 0.25/s` = **4초 간격**이다. 즉 이 관측 시점에는 브라우저가 아직 하드리로드 전이라 `CONN_POLL_MS=30000`이 미적용 상태(구 코드가 계속 4초 폴을 돌리고 있었음)였다는 사실이, 요약 로그 한 줄의 rate 역산만으로 드러난 사례다. `web/`은 nodemon 감시 밖이라(선행 문서 §6) 정적 자산 변경은 브라우저 하드리로드 전까지 반영되지 않는데, 이 사례는 그 사실을 요약 로그가 방증한 것이다.

## 7. 한계·주의 (위장 없이 명시)

- **L4 — `cam_idx` 단위 가시성 상실**: `CameraClient.getPtz`는 `${baseUrl}/ptz?cam_idx=N`으로 호출하는데, 쿼리 제거 집계 키 특성상 **카메라별로 키가 갈라지지 않는다**. 창 안에서는 최초 1대의 URL만 상세로 남고 나머지 카메라 호출은 무음 처리되며, `msAvg`/`msMax`도 여러 카메라가 섞인 값이 된다. "어느 카메라가 느린가"는 이 로그만으로는 볼 수 없다. 설계 의도(쿼리 제거로 무한증식 방지)의 자연스러운 대가이며, 필요해지면 호출부에서 `op`로 카메라를 구분하면 해소 가능(현재는 미조치).
- **지연 sweep 특성**: 창 만료는 별도 타이머가 아니라 "다음 패킷 도착 시 전 키 순회(sweep)"로 처리된다. 따라서 **트래픽이 완전히 정지하면 마지막 창의 요약은 다음 패킷이 올 때까지 방출되지 않는다**(그 창의 첫 상세 줄은 이미 즉시기록으로 남아 있으므로 "무슨 통신이 있었는지" 자체가 유실되지는 않음). 이 특성 때문에 `win`이 실제보다 부풀 수 있어 §4의 `span`/활성 rate를 함께 봐야 한다.
- **`flushPacketLog()`는 export만 되어 있고 어디에도 등록돼 있지 않다**. 프로세스 종료 훅(SIGINT 등)이 현재 0건이라, 이번 범위에서는 붙이지 않았다. 필요해지면 `src/index.ts`에 종료 훅 1개를 추가해 흡수할 수 있다(후속 작업 후보).
- **nodemon 재시작마다 창·로그 파일이 리셋된다**. 집계기는 싱글턴 인메모리 상태(`packetLog.ts`)라 서버가 재시작되면 진행 중이던 모든 창의 누적치가 소실되고(요약되지 않은 채 사라짐), 로거도 `logger.ts`의 파일명 규칙(`setting_<기동시각>.log`)에 따라 새 로그 파일을 연다. 재시작 직전 창의 반복 횟수는 어디에도 남지 않는다.
- **VITEST 환경에서는 집계가 기본 OFF**(`windowMs=0`)이다. `process.env.VITEST` 존재 여부로 결정되며, 유닛테스트에서는 매 호출이 항상 즉시기록되어 기존 테스트(`hucomsClient.test.ts` 등)의 스파이 기반 단언이 그대로 유지된다.

### L3 정정 (설계서 §1.2 서술 오류)

설계서는 "Map 크기는 (엔드포인트 × op) 유한집합으로 자연 상한"이라고 서술했으나, 이는 부정확하다. 집계 키의 `op`는 자유 문자열이 유입될 수 있는 두 경로가 있다.

- `POST /viewer/api/rpc` 프록시 라우트 — `RpcBody.method`가 `z.string().min(1)`로만 검증되어 임의 문자열 허용.
- MCP 도구 `unity_rpc` — 마찬가지로 `method: z.string().min(1)`이며 LLM이 값을 채운다.

즉 이론적으로는 임의 문자열마다 새 키가 하나씩 생길 수 있어 "엔드포인트×op의 고정 유한집합"이라는 전제는 성립하지 않는다. 다만 **영구 누수는 아니다** — sweep이 만료된 창을 지우므로, 실제 상한은 "**최근 5분 창 안에 관측된 distinct op 수**"이며 이는 시간이 지나도 무한히 누적되지 않는다(sweep 비용은 record당 O(활성 키 수)로, op가 많아지면 패킷마다 비용이 늘 뿐 메모리가 무한 증가하지는 않는다). 실사용 위험은 낮다(로컬 뷰어·자체 LLM 환경).

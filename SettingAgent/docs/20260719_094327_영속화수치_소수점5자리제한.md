- 작성일시: 2026-07-19 09:43:27

- 파이프라인 산출물: `_workspace/01_architect_plan.md`(설계) · `_workspace/02_developer_changes.md`(구현) · `_workspace/03_qa_report.md`(검증)

## 1. 무엇을 바꿨나

SettingAgent가 **영속화**(DB에 쓰거나 JSON 파일로 저장)하는 모든 부동소수 수치를, "소수점 이하 최대 5자리"로 반올림해 저장하도록 통일했다. 예: `0.11182877131922099 → 0.11183`, `0.5 → 0.5`(불필요한 뒤 0 없음).

범위는 **영속화 경계로 한정**했다. LLM 프롬프트, MCP 도구 응답, 카메라/RPC 네트워크 명령, 설정 파일(`llm.json`/`tools.config.json`)처럼 전송되거나 휘발되는 값, 혹은 제어 정밀도가 필요한 값은 그대로 두었다.

## 2. 신규 클래스/함수 — `src/util/round.ts`

전체 로직의 단일 출처로 신규 파일 하나만 추가했다(14줄, 의존성 없음).

```ts
/** 유한수만 소수점 최대 5자리로 반올림(round-half-up). 정수/비유한/비수치는 그대로. */
export function round5(n: number): number {
  return Number.isFinite(n) ? Math.round(n * 1e5) / 1e5 : n;
}

/** JSON.stringify 에 숫자 replacer(round5) 적용. 숫자 값만 반올림, 그 외 passthrough. */
export function stringify5(value: unknown, indent?: number): string {
  return JSON.stringify(value, (_k, v) => (typeof v === 'number' ? round5(v) : v), indent);
}
```

- `round5`: DB REAL 컬럼처럼 숫자 하나를 바로 바인딩하는 지점에서 사용. `Number.isFinite`로 정수/`NaN`/`Infinity`/`null`을 그대로 통과시켜 부작용이 없다.
- `stringify5`: `JSON.stringify`의 replacer 인자를 이용해 중첩 객체·배열을 재귀적으로 순회하며 숫자 값만 반올림한다. `Date`처럼 `toJSON`을 가진 값은 replacer에 문자열로 이미 변환된 뒤 전달되므로 영향받지 않는다.
- 반올림 방식은 `Math.round` 기반 **round-half-up**(0.5는 +∞ 방향)이다.

## 3. 적용 지점 — 무엇을, 왜 고쳤나

### 3-1. JSON 파일/TEXT 생산지에 `stringify5` 적용 (7개 파일)

### 3-2. DB REAL 바인딩에 `round5` 적용 — `capture/SqliteStore.ts`

## 4. 규약 — 향후 유지보수 시 반드시 지킬 것

**`slot_setup` JSON TEXT 컬럼의 5자리 보장은 DB 레벨이 아니라 "생산지 규약"으로 성립한다.**

`replaceSlotSetup`(SqliteStore.ts:211)은 인자로 넘어온 TEXT 문자열을 재파싱·재반올림하지 않고 그대로 바인딩한다. 즉 slot_setup의 `slot_roi`/`vpd_bbox`/`lpd_obb`/`occupy_range`/`slot3d_front_center` 컬럼이 5자리 이하로 유지되는 것은, 이 TEXT를 만드는 생산지(현재는 `Finalizer.ts` 6곳 + `migrateToSettingDb.ts` 1곳)가 전부 `stringify5`를 쓰기 때문이다.

> **향후 slot_setup TEXT를 새로 쓰는 코드(신규 writer, 신규 API 등)는 반드시 `stringify5`를 사용해야 한다.** `JSON.stringify`를 직접 쓰면 이 규약이 깨지고 6자리 이상 값이 슬쩍 섞여 들어갈 수 있다 — 코드 리뷰 시 체크포인트로 남긴다.

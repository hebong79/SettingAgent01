# 뷰어 시뮬레이터 연결 감지 · 카메라 정보 자동 갱신

작성일: 2026-07-12 17:27:18 / 작성: 문서화·영향도 분석가(documenter)
대상: SettingAgent 뷰어(`web/`) — `core.js`, `core.d.ts`, `app.js`
백엔드·config: **무변경**

---

## 1. 배경 / 문제

기존 뷰어는 카메라·프리셋 목록을 **페이지 로드 시(`init`)와 소스 변경 시(`sel-source` change)에만** `loadCameras()`로 수신했다. 그 결과:

- Unity(시뮬레이터, 13110) 씬에서 **주차면 프리셋을 추가·삭제**해도 뷰어의 프리셋 갯수가 갱신되지 않아 실제 씬과 어긋났다.
- Unity 연결이 **(재)수립되거나 세션 도중 변경**돼도 뷰어가 이를 반영하지 못했다.
- 연결 상태 뱃지(`badge-camera`)가 실제 Unity 연결이 아니라 13020 health 결과에 묶여 있어, Unity 종료/재기동을 나타내지 못했다.

목표: 뷰어가 세션 중에도 **연결 상태와 카메라/프리셋 구성 변화를 자동으로 따라가되**, 사용자의 현재 선택(cam/preset)과 라이브 스트림을 흔들지 않는다.

---

## 2. 해결 개요

리더 확정 정책에 따라 설계서(01)의 `system.ping` + 전이표(`shouldRefreshOnConnect`) 접근 대신 **4초 주기 폴 → 변경 감지 시에만 재렌더 → 선택 유지** 방식으로 구현했다. `loadCameras()` 성공 자체를 Unity 연결 신호로 사용하므로 별도 ping/RPC 호출과 `controlToken` 게이트 이슈를 회피한다.

핵심 파이프라인:

```
setInterval(connectionTick, 4000)
      │  (connInflight 가드 · 정밀수집 중 skip)
      ▼
loadCameras()  ── GET /viewer/api/cameras (no-store)
      │  성공(boolean) → simConnected · badge-camera 토글
      ▼
camerasChanged(state.cameras, next)   // camIdx + presetIdx 집합만 비교
      │  변경됨일 때만
      ▼
renderCamSelect() → renderPresetSelect()
      │  각 단계에서
      ▼
pickSelected(prevId, list, key)       // 현재 cam/preset 선택 유지
```

## 3. 신규 순수 헬퍼 (`web/core.js`, `core.d.ts`)

DOM/타이머/fetch를 참조하지 않는 순수 함수 2개를 신규 export 했다(테스트 가능화).

### 3.1 `pickSelected(prevId, list, key = 'camIdx')`

목록 갱신 후 이전 선택을 유지한다.

- `list`에 `item[key] === prevId` 항목이 있으면 → `prevId` 그대로 유지
- 없으면 → `list[0][key]`(첫 항목 폴백)
- 빈 목록·`null`/`undefined` → `null`
- 비교는 strict `===`. 소비부가 항상 숫자(`Number(e.target.value)`)를 저장하고 `/cameras`의 `camIdx`/`presetIdx`도 숫자라 매칭이 성립한다(문자열/숫자 혼용 없음).
- 입력 목록을 변형하지 않음(순수).

타입 선언(`core.d.ts`): `pickSelected<K extends string>(prevId, list, key?) : number | string | null`.

### 3.2 `camerasChanged(prev, next)`

카메라/프리셋 **집합**이 실제로 바뀌었는지 판정한다. 각 카메라를 `camIdx:presetIdx목록(정렬)` 시그니처로 만들어 정렬·결합 후 문자열 비교한다.

- camIdx 추가/삭제, presetIdx 추가/삭제 → `true`
- **라벨·PTZ만 변경 → `false`**(idx 집합 동일): 재렌더를 skip해 선택 튐·깜빡임을 막되, `state.cameras`는 무조건 갱신하므로 내부 상태(PTZ 등)는 최신을 유지한다.
- 카메라·프리셋 순서만 다른 동일 집합 → `false`(`.sort()` 정규화)
- `null`/`undefined`/`[]` 는 모두 "빈 시그니처"로 동등 처리

타입 선언: `camerasChanged(prev, next) : boolean`.

## 4. `app.js` 변경 로직

### 4.1 `loadCameras()` — boolean 반환 + 변경 감지 (105~122행)

- `try/catch` + `res.ok`로 성공 여부를 boolean 반환. 실패(HTTP !ok/네트워크 예외) 시 **목록 미변경 + `false`**.
- `cache: 'no-store'`로 시뮬레이터 구성 변화를 항상 최신 수신.
- 성공 시 `camerasChanged(state.cameras, next)`로 비교 → `state.cameras = next`는 **무조건** 갱신, `renderCamSelect()`는 **변경됐을 때만** 호출.

### 4.2 선택 유지 렌더 (170~199행)

- `renderCamSelect`: `state.cam = pickSelected(state.cam, state.cameras, 'camIdx')` (기존 "첫 항목 강제" 대체).
- `renderPresetSelect`: `state.preset = pickSelected(state.preset, presets, 'presetIdx')`.

### 4.3 주기 폴 (143~167행)

- 모듈 스코프 상수/상태: `CONN_POLL_MS = 4000`, `simConnected`(null|true|false), `connInflight`(중복 폴 가드).
- `captureActive()` = `capFrameTimer !== null`. 정밀수집(캡처 프레임 폴) 진행 중이면 `connectionTick`을 skip → 카메라/RPC 경합 방지(뱃지는 마지막 상태 유지).
- `connectionTick()`: `connInflight`/`captureActive` 가드 → `loadCameras()` 호출 → 반환값으로 `simConnected` 갱신 + `badge-camera` 토글.

### 4.4 뱃지 의미 분리 (133~141행)

- `loadHealth()`에서 `badge-camera` 토글을 **제거**. 이제 `badge-backend`만 13020 health를 반영.
- `badge-camera`는 `connectionTick`(=`loadCameras` 성공)이 관장 = **Unity(13110) 연결** 신호.

### 4.5 수동 카메라 변경 무회귀 가드 (2055~2059행)

- `sel-cam` change 핸들러가 `renderPresetSelect()` **직전에 `state.preset = null`** 로 설정 → `pickSelected`가 첫 프리셋으로 폴백. 수동 카메라 전환 시 "새 카메라 첫 프리셋으로" UX를 그대로 보존한다. 자동 갱신 경로만 선택을 유지한다.

### 4.6 `init` 연동 (2130~2142행)

- `const [camOk] = await Promise.all([loadCameras(), loadHealth()])` → `simConnected = camOk` 초기 확정 + `badge-camera` 초기 토글.
- 마지막에 `setInterval(connectionTick, CONN_POLL_MS)` 시작(기존 1s 경과틱·500ms 캡프레임틱과 독립).

## 5. 동작 정리

- **초기 로드**: `state.cameras=[]` → 첫 수신 시 `camerasChanged`=true → 정상 렌더. `init`이 연결/뱃지 초기화.
- **매 폴(4초)**: 최신 목록 재수신. 연결 전이(끊김→연결)와 프리셋 갯수 변경 모두 자동 반영.
- **라벨/PTZ만 변경**: 재렌더 skip(선택 튐 방지)하되 `state`는 갱신.
- **정밀수집 중**: 폴 억제(뱃지 마지막 상태 유지).

---

## 6. 입출력 · REST 계약

- 신규/변경 엔드포인트 **없음**. 기존 `GET /viewer/api/cameras`(no-store, `?source=` 옵션)와 `GET /viewer/api/health`만 사용.
- `GET /viewer/api/cameras` 응답 계약: `{ cameras: [{ camIdx, presets: [{ presetIdx, pan?, tilt?, zoom?, label? }] }] }`. `camerasChanged`는 `camIdx`+`presetIdx`만 소비.

### 한계 / 미검증(유닛 대상 아님)

`connectionTick` / `setInterval(4000)` / `connInflight` / `captureActive` / `badge-camera` 토글 / `loadHealth`는 DOM·타이머·fetch 부작용 계층이라 유닛테스트 대상이 아니다. 아래는 **브라우저 수동 스모크 확인 영역**(통과 위장하지 않음):

1. Unity 종료 → 4초 내 `badge-camera` off
2. 재기동 → on + 카메라/프리셋 자동 재수신
3. Unity에서 프리셋 추가 → 자동 반영(갯수 증가)
4. 자동 갱신 중 현재 cam/preset 선택 유지 · 라이브 스트림 무중단
5. 정밀수집 중 폴 억제(뱃지 마지막 상태 유지)

> 참고: 설계서(01)는 `system.ping` + `shouldRefreshOnConnect` 전이표 방식이었으나, 구현(02)은 리더 확정으로 **4초 폴 loadCameras + camerasChanged** 로 선회했다. 따라서 `shouldRefreshOnConnect`는 코드에 없고 테스트 대상이 아니다. 실제 구현 표면인 `pickSelected`/`camerasChanged`만 커버한다.

## 8. 무회귀

- **수동 카메라 변경**(`sel-cam`)은 `state.preset=null` 가드로 기존대로 첫 프리셋으로 리셋(UX 불변).
- **라이브 MJPEG 스트림 / `gotoPreset` / `reconnectLiveIfActive` / `drawRoiOverlay` / 정밀수집**: 자동 갱신은 `frame.src` 미변경 + 물리 카메라 이동 미유발. 변경 없을 땐 재렌더 자체를 skip → 무중단.
- 초기 로드 · 소스 변경 · 라이브 스트림 · 정밀수집 거동 불변.

---

## 1. 목적과 역할

주차장을 **운영 가능한 상태로 셋업**하는 에이전트. 카메라 프리셋을 순회하며 주차된 차량을
검지하여 **주차면(ROI)** 을 확보하고, 프리셋별 슬롯 인덱스를 부여하며, 최종적으로
**전 카메라·전 프리셋의 주차면을 정렬하여 전역 슬롯 인덱스로 매핑**한다.
산출물은 공유 저장소(Repository)에 기록되어 Action/DM 런타임의 기준 데이터가 된다.

> SettingAgent 는 **셋업/재셋업 시에만 동작**하는 배치성 에이전트다(상시 가동 X).

### 1-1. 두뇌(LLM)/도구 경계 (아키텍처 §8)
- **LLM 두뇌(MCP로 도구 호출)**: 셋업 전략, 재스캔 필요 판단, ROI 클러스터 이상(중복/누락) 검토 등 판단.
- **결정형 MCP 도구**: `camera.req_img`, `vpd.detect`, ROI 산출, 전역 인덱스 정렬 등 수치·반복 처리.
- 모델-불문(Claude/Codex/Gemma/Qwen3) — 능력은 모두 MCP 도구로 노출.

## 2. 할일 매핑

| # | 할일 | 설계 반영 |
|---|------|-----------|
| 1 | 주차된 차량 검지 → 주차영역(ROI) 확보 | VPD(`da_vpd_api /det/imgupload`) bbox 확보 후 ROI 산출 |
| 2 | 주차면이 있으면 인식하여 영역 확보 | **시뮬레이터는 강체 차량 배치 → 차량 검출 기반 ROI로 충분**(빈 주차면 라인 검지 미사용). 현장 적용은 2차 |
| 3 | 카메라 프리셋 셋팅(초기 PTZ) | 프리셋별 초기 pan/tilt/zoom 보관·적용 |
| 4 | 번호판 기준 중심점을 주차면 영역으로 설정 | (Action 센터라이징과 연계) ROI 중심 정의 |
| 5 | 프리셋별 주차면 인덱스 부여 | 프리셋 이미지 내 위치 정렬로 1-based 인덱스 |
| 6 | 프리셋 여러 개면 전부 순회하여 ROI 확보 | 프리셋 순회 스캔 |
| 7 | 전 카메라·전 프리셋 주차면 정렬 → 전역 매핑 | **전역 슬롯 인덱스 생성(핵심 산출물)** |
| 8 | 초기 위치 설정 후 순회하며 차량 존재 기준 주차면 탐색·매핑 | 순회 스캔 + 차량 검지 기반 매핑 |

## 3. 처리 흐름 (결정형 오케스트레이션)

```
[셋업 시작]
  └ 1. 카메라/프리셋 목록 로드 (config)
  └ 2. 각 카메라 × 각 프리셋 순회:
        a. req_img(cam, preset, presetPTZ) → 캡처 이미지
        b. VPD /vpd/api/v2/det/imgupload → 차량 bbox 검지
        c. 검지 결과로 프리셋 내 슬롯 ROI 산출 + 위치 정렬 → 프리셋별 슬롯 인덱스
        d. 프리셋.coveredSlotIds, slot.roiByPreset 갱신
  └ 3. 전 프리셋 슬롯 수집 → 정렬 규칙 적용 → 전역 슬롯 인덱스 매핑 (§7 정렬규칙)
  └ 4. Repository 에 Preset/ParkingSlot/GlobalSlotIndex 영속화
[셋업 완료] → 검증 리포트 산출
```

상태: `IDLE → SCANNING → MAPPING → PERSISTED → DONE`(오류 시 `FAILED`, 부분 재시도 가능).

## 8-1. 점유 판정에 ROI가 필요한가? (사용자 요청 검토)

**결론: 슬롯(주차면) 단위 점유 판정에는 ROI가 필요하다. → SettingAgent가 ROI를 산출·보관한다.**

근거:
1. 채택한 VPD(`da_vpd_api /det`)는 **이미지 전체에서 차량 bbox 목록**만 돌려준다.
   "몇 대 있는가"는 알지만 **"어느 주차면에 있는가"** 는 모른다.
   → bbox 를 슬롯에 귀속시키려면 각 슬롯이 이미지에서 차지하는 영역(=ROI)이 필요.
2. ROI 없이 점유를 알 수 있는 경우는 두 가지뿐인데 본 시스템엔 부적합:
   - (a) 한 프리셋이 **정확히 한 면만** 비춤 → bbox 존재 = 그 면 점유. (다면 프리셋이면 불가)
   - (b) **VLM(`occupy_slot`) 방식**으로 모델이 위치 매핑까지 수행 → 위치 좌표 손실로 센터라이징 곤란.
3. 시뮬레이터는 강체로 차량을 세우므로(빈 면 라인검지 불필요), **셋업 시 검출된 차량 bbox 자체를
   그 슬롯의 ROI 기준**으로 삼으면 된다(별도 라인검지 없이 ROI 확보 가능).

**판정 규칙(ActionAgent와 공유):**
```
슬롯_점유 = ∃ 차량bbox 중, (bbox 중심점 ∈ 슬롯ROI)  [1차 규칙]
           또는 IoU(bbox, 슬롯ROI) ≥ τ                [보조]
다중 후보(차 1대가 여러 ROI 걸침) → 겹침 최대 ROI에 귀속(1대=1면 보장)
```

**대안(ROI 최소화):** 시뮬레이터는 슬롯의 Unity 월드 좌표를 알므로, 이를 카메라 투영으로 이미지
좌표 ROI를 **자동 생성**할 수도 있다(수작업 ROI 불필요). 단, 투영 정확도 검증이 필요하므로
1차는 **검출 bbox 기반 ROI**, 2차 최적화로 **월드좌표 투영 ROI**를 검토 권장.

### 8-1-1. 실 PTZ 카메라 ROI 자동 생성 (수동 작성 불필요)

실 카메라에는 Unity 월드좌표가 없지만, **검출 기반 방식은 동일하게 동작**하므로 수동 ROI 없이 자동화 가능.

| 환경 | 1차 (수동 0) | 2차 보완 |
|------|--------------|----------|
| 시뮬레이터 | 검출 bbox ROI (강체 1프레임 배치) | 월드좌표 투영 ROI |
| **실 PTZ** | **검출 누적 + 클러스터링** | 주차선 검지 / 측량맵 호모그래피 투영 |

**실 PTZ 1차 방식 — 검출 누적 + 클러스터링:**
```
셋업 기간 동안 프리셋별 주기 캡처 → VPD 차량 검출(bbox)
  └ bbox 중심/영역을 시간축으로 누적
  └ 클러스터링(예: 그리드/DBSCAN) → 안정 클러스터 = 주차면
  └ 클러스터 대표영역(중앙값 박스) = 슬롯 ROI
```
- 시뮬은 강체로 즉시 1프레임 확보, 실 현장은 **자연 관측 누적으로 슬롯 위치 발견**(시간 필요).
- 보조: VPD `/seg` 마스크로 ROI 경계 정교화 가능.

**본질적 한계:** 검출 누적은 **한 번이라도 점유된 면만 발견**한다. **상시 빈 면**과 **빈 면 경계**는
차량 검출만으로 못 잡으므로 → 주차선 검지 모델 또는 사전 측량맵+호모그래피 투영으로 2차 보완.
(이는 §8의 "현장 빈 주차면" 미해결 항목과 동일 지점.)

## 9. 영향도 분석

- **신규 프로젝트**이므로 기존 코드 직접 수정 없음. ParkSimMgr의 클라이언트/도메인 타입을
  **이식 또는 공통 패키지(`@parkagent/types`)로 추출**할 경우 ParkSimMgr와의 타입 정합성 유지 필요.
- 산출물 스키마(Preset/ParkingSlot/GlobalSlotIndex)는 **Action/DM이 읽는 계약**이므로
  변경 시 3개 에이전트 동시 영향 → 공통 패키지로 버전 관리.
- 동일 Unity/VLM 서버를 Action과 공유 → 셋업 중 런타임 스캔과의 동시 접근 충돌 가능
  (셋업은 점검 시간대에 단독 수행 권장).

## 10. 테스트 계획

- 유닛: `RoiBuilder`(검지→ROI/정렬), `GlobalIndexer`(정렬규칙), `SetupOrchestrator`(상태전이) — 외부 모킹.
- 통합(스모크): 실제 Unity+da_vpd_api 기동 후 1카메라·N프리셋 셋업 → 매핑 산출 검증.
- 성공 기준: 동일 입력에 대해 결정적(deterministic) 전역 인덱스 산출 + 재스캔 멱등성.

# 장기 관측·반복 수집 → SQLite 누적 → LLM 정밀 주차면 확보 설계서

- 작성일: 2026-06-25
- 대상: SettingAgent(수집·DB·LLM 소유) + SettingViewer(시작/모니터/정지 UI)
- 한 줄 요약: **시간에 걸쳐 반복 캡처해 원시 검출을 SQLite에 누적 → 결정형 시공간 집계로 통계적 ROI → 중간 체크포인트 LLM 보정 + 최종 LLM 판정**으로 단발 1프레임보다 정확한 주차면(ROI·전역 인덱스)을 확보한다.

> 본 문서는 **설계(계획)** 다. CLAUDE.md 5대 규칙(설계→유닛테스트→동작확인→한글문서화→영향도분석)에 따라 승인 후 `parkagent-dev`로 구현한다.

---

## 0. 배경·목표

### 0.1 문제
현재 SettingAgent 셋업은 프리셋당 **단발(시뮬 `accumFrames=1`)** 캡처 → VPD 검출 → ROI. 한 시점 1프레임은:
- 그때 **비어 있는 면을 놓친다**(VPD는 "주차된 차량"을 검출 → 빈 면은 차가 들어와야 면으로 드러남).
- 검출 bbox가 흔들리거나(지터), 지나가는 차/반사로 **노이즈** 면이 생긴다.

### 0.2 목표
시간에 걸쳐 점유 상태가 바뀌며 반복 수집하면 **모든 면이 결국 한 번씩 점유되어 드러나고**, 같은 면에 검출이 누적되어 통계적으로 안정된 ROI가 나온다. 이를 DB에 영속화하고 LLM이 근거를 보고 판정하여 **정밀 주차면**을 확보한다.

### 0.3 성공 기준
| # | 목표 | 검증 |
|---|------|------|
| G1 | 반복 횟수 N(기본) 지정 + **언제든 수동 정지**로 캡처 잡 구동 | 잡 상태머신·정지 테스트 |
| G2 | 매 캡처의 원시 검출이 SQLite에 적재(관측·검출 행) | 적재/조회 단위테스트 |
| G3 | 누적 검출의 시공간 클러스터링으로 면별 지지수·통계 ROI 산출(결정형) | 집계 함수 단위테스트(합성 데이터) |
| G4 | 체크포인트 LLM 보정(병합/라벨/노이즈/커버리지) + 최종 LLM 판정 → `setup_artifact.json` | 브레인 모킹 테스트·산출물 shape |
| G5 | Viewer에서 시작/진행/정지 UI, SettingAgent `/capture/*` 프록시 | 프록시·UI 순수로직 테스트 |

## 1. 핵심 원리 (왜 시간 루프가 정확도를 높이나)
1. **점유 다양성 → 전면 커버리지**: 시간대마다 다른 면이 점유 → 누적하면 빈 면도 언젠가 점유되어 포착. 단발이 놓치는 면을 회수.
2. **반복 누적 → 통계적 안정**: 진짜 면엔 검출이 반복 누적(지지수↑), 일시 노이즈는 지지수↓ → 임계로 구분. bbox는 다수 관측의 중앙값/대표값으로 지터 제거.
3. **점유율(occupancy) 부산물**: 면별 "차가 있었던 비율"이 자연히 수집 → Action/DM 의 점유 추정 prior로도 유용(이번엔 저장만).

## 2. LLM 관여 방식 — 사용자 질문에 대한 답 + 채택

세 가지가 가능하다. 좌표는 **항상 검출+집계가 보장**(프로젝트 철학 "좌표는 검출, LLM은 판정")하며, 차이는 **LLM이 언제·무엇에 관여하나**이다.

| 방식 | LLM 호출 | 장점 | 단점 |
|------|----------|------|------|
| **A. 누적→집계→끝에 LLM 1회** | 1회(요약) | 가장 저렴·단순 | 수집 중 잘못된 방향(특정 프리셋 과소수집·명백 노이즈)을 끝나서야 인지 |
| **B. 매 프레임 LLM** | 매 캡처 | 즉시 반응 | 토큰 폭증·중복·느림. 비추천 |
| **C. 중간 체크포인트 LLM(권장)** | 주기적(K캡처마다) + 끝 1회 | 저렴하면서 **수집 중 보정·수렴 판단**. 단발 A의 "늦은 인지" 해소 | A보다 약간 복잡·호출 수 증가(여전히 소수) |

### 2.1 채택: **C(중간 체크포인트 하이브리드)**
사용자가 물은 "중간중간 LLM 보정"이 바로 C다. **A보다 낫다** — 이유:
- **온라인 보정**: K캡처마다 LLM이 "지금까지 집계"를 보고 **증분 정리**(중복 클러스터 병합 제안, 존 라벨 부여, 명백 노이즈 다운웨이트)와 **커버리지 진단**(기대치 faceCount 대비 누락 면, 프리셋별 수집 충분도)을 낸다.
- **수렴 신호(자문)**: "최근 K회 새 면 없음 → 수렴, 조기 종료 가능"을 사용자에게 **표시**(자동 정지 아님 — 사용자는 횟수 기반 + 언제든 수동 정지). 즉 LLM은 **추천만**, 결정은 사용자/횟수.
- **안전장치(좌표 불변)**: LLM은 **샘플링 전략·그룹화·라벨·수용/거부 판정**만. **bbox 좌표를 만들거나 바꾸지 않는다.** 모든 좌표는 검출+클러스터링 산출이며 LLM은 "이 클러스터를 면으로 인정/병합/제외" 수준의 판정·메타만 건드린다.
- **끝 1회 최종화**: 마지막에 전체 집계로 LLM이 최종 판정 → `setup_artifact.json`(+리포트). 체크포인트 결과는 최종화의 컨텍스트로 활용.

> 즉 **결정형 누적이 정확도의 뼈대, LLM은 전략·판정의 보조**. 비용은 A보다 크지만 호출은 (N/K)+1 수준으로 작고, 품질·수렴 이점이 명확하다. LLM 비활성(`llm.enabled=false`) 시 C는 자동으로 **A의 결정형 경로**로 강등(체크포인트 생략, 끝에 결정형 집계만) — 무LLM에서도 동작.

```
SettingViewer (:13030)  ──시작/진행/정지──►  SettingAgent (:13020)
   UI: 반복횟수 입력·시작·진행바·정지·요약          ├ POST /capture/start {count, intervalMs, targets?}
   /viewer/api/capture/* ──프록시──►              ├ GET  /capture/status   (진행·수렴 자문)
                                                ├ POST /capture/stop     (수동 정지)
                                                ├ POST /capture/finalize (집계+LLM→artifact)
                                                └ GET  /capture/runs     (이력)
                                                     │
       [CaptureJob 러너] ─매 주기→ CameraClient/req_img → VpdClient/LpdClient 검출
                          └─ SQLite 적재(관측·검출) ─ K마다→ 집계 + (LLM 체크포인트)
       [Finalizer] 집계 전체 → AgentRuntime(브레인) 최종 판정 → setup_artifact.json (+ DB 기록)
```
- **SettingAgent 소유**(결정 1): 카메라·VPD/LPD·두뇌(AgentRuntime)·집계(RoiAccumulator/클러스터)·산출물 계약을 이미 보유 → 자연 결합.
- **SettingViewer**: 얇은 제어·모니터 UI + `/capture/*` 프록시(기존 `/mapping` 프록시 패턴 동일). 무거운 로직 없음(분리 유지).
- **재사용**: `CameraClient`, `VpdClient`, `LpdClient`, `RoiAccumulator`, 클러스터 파라미터(`clusterDist`/`clusterMinSupport`), `AgentRuntime`, `SetupArtifact` 계약, `Repository.saveArtifact`.

## 4. 파이프라인 (3단계 + 체크포인트)

### 4.1 수집(반복) — CaptureJob
- 입력: `count`(기본 정지조건), `intervalMs`(주기, 점유 변화를 담기 위해 수십 초~분), `targets`(미지정 시 camerapos/discovery 로 프리셋 목록).
- 매 라운드: 프리셋 순회 → `/req_img` 캡처 → VPD(+LPD) 검출 → **원시 검출을 SQLite 적재**(LLM 호출 없음).
- 상태머신: `idle → running → (stopping) → finalizing → done | stopped | error`. **수동 정지**는 `running` 중 `stop` → 현재 라운드까지만 마치고 `stopped`.
- 동시성: 단일 잡(중복 시작 거부). 잡은 인메모리 + DB `capture_run` 행으로 추적(재기동 복구는 이번 범위 밖, 상태만 기록).

### 4.2 집계(결정형) — Aggregator
- 면별 시공간 클러스터링: 누적 검출(정규화 bbox)을 프리셋별로 `clusterDist`로 묶고 `clusterMinSupport` 임계로 노이즈 제거. 대표 ROI = 클러스터 중앙값/대표 bbox, `support`=관측 수, `occupancyRate`=점유 관측 비율.
- LPD 번호판 검출도 같은 방식으로 `plateRoi` 누적(센터라이징 prior).
- 출력: `aggregated_slot` 집합(프리셋·좌표·지지·점유). 결정형이라 LLM 없이도 산출.

### 4.3 체크포인트 LLM(주기, 선택) — CheckpointReviewer
- K라운드마다(설정) 현재 집계 요약(+선택 프리셋 썸네일)을 AgentRuntime에 전달 → 반환:
  - `merges`(같은 면으로 볼 클러스터 쌍), `labels`(존 라벨), `rejects`(노이즈 클러스터 id), `coverage`(기대 faceCount 대비 부족 프리셋), `convergence`(수렴 여부·조기종료 추천 자문).
- 적용 범위: 병합/라벨/거부 **메타 판정**만 반영(좌표 불변). `coverage`/`convergence`는 `/capture/status`로 사용자에게 **표시**. 결과를 `checkpoint` 행에 저장.

### 4.4 최종화 — Finalizer
- 전체 집계 + 체크포인트 누적 컨텍스트 → LLM 최종 판정(프리셋 간 중복 제거·라벨·노이즈 제외·누락 보완) → `GlobalIndexer`로 전역 인덱스 → `SetupArtifact` 생성 → `Repository.saveArtifact`(기존 `setup_artifact.json`) + DB `artifact_snapshot` 기록.
- `/mapping`은 그대로 최종 산출물 반환 → SettingViewer ROI 검수 화면이 즉시 정밀 결과 표시.

## 5. 데이터 모델 (SQLite)

라이브러리: **`better-sqlite3`**(동기·고속·프리빌트 바이너리 → Windows 네이티브 빌드 회피). 파일: `data/observations.sqlite`(`store.dataDir`).

```sql
-- 캡처 잡 1회
capture_run(
  id INTEGER PK, started_at TEXT, ended_at TEXT,
  planned_count INTEGER, done_count INTEGER, interval_ms INTEGER,
  status TEXT,            -- running|stopped|done|error
  stop_reason TEXT        -- count|manual|error
);

-- 라운드/프리셋 단위 관측(프레임 1장)
observation(
  id INTEGER PK, run_id INTEGER FK, round_idx INTEGER,
  cam_idx INTEGER, preset_idx INTEGER, captured_at TEXT,
  pan REAL, tilt REAL, zoom REAL, img_name TEXT
);

-- 검출 박스(정규화 0~1)
detection(
  id INTEGER PK, observation_id INTEGER FK,
  cam_idx INTEGER, preset_idx INTEGER,
  kind TEXT,             -- vehicle|plate
  x REAL, y REAL, w REAL, h REAL, conf REAL
);

-- 집계 결과(집계 실행 시 갱신)
aggregated_slot(
  run_id INTEGER, preset_key TEXT, cluster_id INTEGER,
  cam_idx INTEGER, preset_idx INTEGER,
  x REAL, y REAL, w REAL, h REAL,
  support INTEGER, occupancy_rate REAL,
  plate_x REAL, plate_y REAL, plate_w REAL, plate_h REAL,
  status TEXT            -- candidate|accepted|rejected|merged
);

-- LLM 체크포인트 기록
checkpoint(
  id INTEGER PK, run_id INTEGER, at_round INTEGER, created_at TEXT,
  summary_json TEXT      -- merges/labels/rejects/coverage/convergence
);

-- 최종 산출물 스냅샷(감사용)
artifact_snapshot(
  id INTEGER PK, run_id INTEGER, created_at TEXT, artifact_json TEXT
);
```
- 인덱스: `detection(observation_id)`, `observation(run_id, preset_idx)`.
- 보존/정리: 런 단위 보존. (대량 누적 시 오래된 런 정리 옵션은 후속.)

## 6. API · UI

### 6.1 SettingAgent REST (신규, `/capture/*`)
| 메서드 | 경로 | 설명 |
|--------|------|------|
| POST | `/capture/start` | `{count, intervalMs?, checkpointEvery?, targets?}` → 잡 시작(중복 거부) |
| GET | `/capture/status` | 진행(round/done/planned)·상태·최근 체크포인트(커버리지·수렴 자문) |
| POST | `/capture/stop` | 수동 정지(현재 라운드까지 마치고 stopped) |
| POST | `/capture/finalize` | 집계+LLM 최종화 → `setup_artifact.json` 갱신. (정지/완료 후 호출) |
| GET | `/capture/runs` | 런 이력(메타) |
| GET | `/capture/runs/:id/aggregate` | 특정 런 집계 결과(검수용) |

> 기존 `/setup/*`(단발 셋업)은 **유지**. `/capture/*`는 별도 능력으로 가산(외과적). 단발이 필요하면 그대로, 정밀이 필요하면 `/capture`.

### 6.2 SettingViewer 프록시 + UI
- 프록시: `/viewer/api/capture/*` → SettingAgent `/capture/*`(기존 `/mapping` 프록시와 동일 서버측 중계).
- UI(설정 탭 또는 신규 "정밀 수집" 탭):
  - 입력: **반복 횟수**(기본 정지조건), 주기(초), 체크포인트 간격.
  - 버튼: 시작 / **정지(언제든)** / 최종화.
  - 진행: 진행바(done/planned), 라운드·검출 수, **체크포인트 자문**(수렴됨/부족 프리셋), 경고.
  - 완료 후: ROI 검수 탭이 최종 `/mapping` 표시(정밀 결과).
  - 순수 로직(상태 폴링·진행률 계산·자문 표시 매핑)은 `core.js` 테스트 가능 모듈로.

## 7. 설정 추가 (`tools.config.json` → `capture` 섹션)
```jsonc
"capture": {
  "defaultCount": 50,        // 기본 반복 횟수(정지조건)
  "intervalMs": 30000,       // 라운드 주기(점유 변화 포착)
  "checkpointEvery": 10,     // K라운드마다 LLM 체크포인트(llm.enabled 시)
  "dbFile": "data/observations.sqlite",
  "clusterDist": 0.06,       // 집계 클러스터 거리(setup 값 재사용/독립)
  "clusterMinSupport": 3     // 면 인정 최소 지지수(단발보다 높게)
}
```
- `llm.enabled=false`면 체크포인트·최종 LLM 생략 → 결정형 집계만으로 산출(A 강등).
- SettingViewer 는 시작 시 UI 값으로 `count`/`intervalMs`를 덮어쓴다(설정은 기본값).

## 8. 정확도 보장 · LLM 경계 (불변식)
- **좌표는 검출+집계만 생성/수정.** LLM은 좌표를 만들지 않는다 — 클러스터 병합/라벨/수용·거부 **판정**과 샘플링·수렴 **자문**만.
- 집계 임계(`clusterMinSupport`)로 노이즈 1차 제거 → LLM은 경계 사례만 판정.
- 무LLM에서도 결정형으로 동작(품질은 다소↓, 안전).
- 기존 `SetupArtifact` 계약 **불변**(Action/DM 무영향) — 정밀화는 "더 좋은 같은 형식"을 만들 뿐.

## 10. 구현 단계(개요) + 검증
```
1. SQLite store(better-sqlite3) + 스키마·DAO       → 적재/조회 단위테스트(G2)
2. Aggregator(클러스터·지지·점유, 결정형)           → 합성 검출 집계 테스트(G3)
3. CaptureJob 러너(상태머신·count·수동정지·주기)     → 상태전이·정지 테스트(G1)  [주기는 fake timers]
4. /capture/* REST + zod                            → fastify.inject 라우트 테스트
5. CheckpointReviewer + Finalizer(브레인 모킹)       → 병합/라벨/거부 반영·artifact shape(G4)  [LLM off 강등 경로 포함]
6. SettingViewer /viewer/api/capture/* 프록시 + UI탭 → 프록시·core 순수로직 테스트(G5)
7. 설정(capture 섹션)·문서·영향도                    → config 테스트, 한글 문서
8. 동작확인: 시뮬레이터로 소수 반복 수집→집계→(LLM)→/mapping 확인
```
- 테스트: vitest, 외부 서버·LLM 모킹. SQLite는 임시 파일/`:memory:`.
- 동작확인: 시뮬레이터(:13100)로 N=소수·짧은 주기 수집 → 집계·산출물 확인. 실 PTZ·장시간은 별도.

## 11. 결정/미해결
1. **반복 종료 자동(수렴) vs 수동** — 결정: **기본 횟수 + 언제든 수동 정지**. 수렴은 LLM **자문 표시**만(자동 정지 아님). (사용자 확정)
2. **DB 라이브러리**: `better-sqlite3`(프리빌트, 권장) — 설치 시 네이티브 빌드 불필요한지 구현 단계 실측. 대안 `node:sqlite`(Node22+, 실험적).
3. **집계 주기/임계 기본값**(`intervalMs`, `clusterMinSupport`): 시뮬 기준 잠정값 → 실데이터로 튜닝.
4. **체크포인트 썸네일 전달 여부**(비전 LLM 비용) — 1차는 텍스트 요약만, 비전은 옵션.
5. **재기동 복구**(running 중 프로세스 종료) — 1차 범위 밖(상태만 DB 기록).

## 12. 요약
- **시간 반복 수집 → SQLite 원시 검출 누적 → 결정형 시공간 집계 → 중간 체크포인트 LLM 보정 + 최종 LLM 판정**으로 단발보다 정밀한 주차면 확보.
- 사용자 질문(중간 LLM 관여)의 답 = **방식 C(체크포인트 하이브리드)** 채택: A(끝 1회)보다 온라인 보정·수렴 자문에서 우수, B(매 프레임)보다 저렴. **좌표는 검출이 보장, LLM은 판정·전략만.**
- **SettingAgent 소유**(수집·DB·LLM), **SettingViewer는 시작/모니터/정지 UI + 프록시**. 정지조건 = **횟수 기본 + 수동 정지**.
- `SetupArtifact` 계약·`/setup/*`·기존 테스트 **불변**, 모두 가산적. `llm.enabled=false`면 결정형 강등.

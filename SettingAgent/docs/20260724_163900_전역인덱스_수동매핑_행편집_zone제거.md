# 전역 인덱스 수동 매핑 — 행 편집(카메라·프리셋·프리셋내 위치) + slotId=전역ID 명문화 + zone 열 제거

작성: 2026-07-24 / 브랜치: `feat/manual-index-editable-rows`

## 1. 요구와 결론

| 마스터 요구 | 결론 |
|---|---|
| 각 슬롯 리스트아이템을 모두 수정 가능하게(카메라, 프리셋, 프리셋내 위치) | 표 4개 열 전부 입력 박스로 전환. 저장 시 DB `slot_setup` 에 실제 반영(신규 `POST /mapping/placement`) |
| slotId 는 전역ID와 동일(전역ID를 따라간다) | 이미 그렇다(DB `slot_id` = 전역번호). 표에서 slotId 는 **읽기 전용 '현재값'** 으로 남기고, 저장하면 입력한 전역 ID 로 재번호된다는 것을 헤더·툴팁·안내문에 명문화 |
| zone 의 의미를 알려주고 의미 없으면 삭제 | **의미 없음 → 표에서 삭제**(아래 §2) |

## 2. zone 의 정체와 삭제 근거

- 값의 출처는 두 갈래뿐이다.
  1. LLM 게이트2가 돌려주는 `zoneLabels[slotId]` (`SetupBrain.ts:93`, `SetupOrchestrator.ts:296-297`) — "A구역/B구역" 같은 사람이 붙이는 라벨.
  2. 폴백 `zone = cam{camIdx}` (`Finalizer.ts:387`, `artifactFromSlotSetup.ts:46`).
- **DB `slot_setup` 에 zone 컬럼이 없다.** 그래서 `setup_result.json`(소비측 정본)에도 없다. zone 은 `setup_artifact.json` 안에서만 왕복한다.
- 현재 정책상 LLM 은 최소 보조로만 쓰므로 `zoneLabels` 는 실질적으로 비어 있고, `GET /mapping` 이 DB 폴백으로 조립할 때는 **항상** `cam{N}` 이 들어간다 = 바로 옆 '카메라' 열의 중복 표기.
- 따라서 매핑 표에서 zone 은 정보량 0 → 열 삭제 + `buildMappingRows` 의 `zone` 필드 제거(고아 필드 정리).
- **삭제 범위는 이 표까지다.** `ParkingSlot.zone` 필드 자체(스키마 `artifactSchema.ts:36`, Finalizer, LLM 경로, 분석 탭 '존' 요약)는 그대로 뒀다 — 요청 범위 밖이고 파급이 크다. 필드 자체를 없애려면 별건으로 지시 바란다.

## 3. 변경 파일

### 백엔드

| 파일 | 변경 |
|---|---|
| `src/capture/types.ts` | `SlotPlacementRow`(slotId/camId/presetId/presetSlotIdx) 신설 |
| `src/capture/SqliteStore.ts` | `updateSlotPlacement(rows, updatedAt?)` 신설 — 배치 3컬럼 + `updated_at` 만 부분 UPDATE |
| `src/setup/placementMapping.ts` (신규) | `validateSlotPlacement(current, placements, presetKeys)` — 순수 게이트 |
| `src/api/server.ts` | `PlacementBodySchema`, `placementHandler`, `POST /mapping/placement`, `POST /viewer/api/mapping/placement` |

**`updateSlotPlacement` 의 2단계 트랜잭션**: `slot_setup` 에는 `UNIQUE(cam_id, preset_id, preset_slotidx)` 가 걸려 있어서 위치 교환(A↔B)을 행 단위로 UPDATE 하면 중간상태에서 제약이 깨진다. 그래서 ① 대상 행의 `preset_slotidx` 를 전부 NULL 로 비우고 ② 최종값을 넣는다(SQLite 는 NULL 을 서로 다른 값으로 취급하므로 충돌하지 않는다). 제출 밖 행과의 충돌은 아래 검증이 트랜잭션 **전에** 차단한다.

**`validateSlotPlacement` 게이트**(실패는 전부 400 + DB 무변경):
1. 빈 제출 금지 / slotId 중복 제출 금지 / 전부 현재 DB 에 존재
2. `(camId,presetId)` 가 `preset_info` 에 등록돼 있을 것 — FK 위반을 500 대신 400 으로 잡는다
3. 최종 상태의 `(cam,preset,위치)` 삼중키 고유 — **미제출 행의 현재 배치도 시드로 포함**(부분 갱신이 남의 자리를 침범하는 것을 막는다)
4. `preset_slotidx` 1..M 연속성은 서버가 강제하지 않는다(DB 제약이 아니고, 부분 갱신 API 에서 전체 연속성을 요구하면 정상적인 단건 이동이 막힌다). 연속성은 UI 게이트가 담당한다.

**처리 순서(원자성)** — `renumberHandler` 규약을 그대로 따랐다: 검증(실패→400·DB무변경) → DB UPDATE(트랜잭션) → `setup_result.json`(이력본+고정본) → `setup_artifact.json` 재빌드. DB 커밋이 진실의 기준이고 파일 2종은 각자 best-effort.

| 파일 | 변경 |

| `web/core.js` | `buildMappingRows` 에서 `zone` 제거 / `applyManualPlacement(artifact, placementBySlot)` 신설 |
| `web/core.d.ts` | `MappingRow.zone` 제거, `PlacementSubmit`·`applyManualPlacement` 타입 |

| `web/app.css` | `.an-manual-place`(배치 입력 — 좁고 옅게) |

**표 구조**: `전역 ID | 카메라 | 프리셋 | 프리셋내 위치 | slotId (현재)` — 앞의 4개가 입력, 마지막은 읽기 전용.

**클라이언트 게이트(`applyManualPlacement`)**: 세 값이 1 이상 정수 / 삼중키 중복 금지 / 같은 (cam,preset) 안에서 위치 1..M 연속 / slotId 가 전역 정수일 것(구 형식 `c1p1s1` 산출물은 배치 변경 불가 — 위장하지 않고 정직하게 실패). 반환에 `changed` 를 포함해 배치가 그대로면 저장 요청을 건너뛴다(불필요한 `updated_at` 갱신 방지).

**저장 순서(중요)**: `POST /mapping/placement` → `POST /mapping/renumber`. 배치는 **현재** `slot_id` 를 키로 쓰므로 재번호보다 먼저 반영해야 한다. 배치 저장이 실패하면 재번호는 아예 보내지 않는다(DB 무변경 유지).

## 4. 의도적으로 하지 않은 것 (마스터 확인 필요)

- **ROI 좌표 변환 없음**: `slot_roi`/`lpd_obb`/`occupy_range` 는 원래 프리셋 화면 기준 정규화 좌표다. 슬롯을 다른 카메라·프리셋으로 옮겨도 좌표는 그대로 남는다 → 옮긴 슬롯은 **재수집이 필요**하다.
- **센터라이징 PTZ 를 지우지 않음**: `pan/tilt/zoom/centered` 는 옮긴 뒤엔 사실상 무효지만 자동으로 지우지 않았다(데이터 파괴 금지 원칙). 대신 안내문에 경고를 넣었고, 옮긴 슬롯은 **재센터라이징**이 필요하다.
- `slot_ptz.json` 은 배치 변경 시 갱신하지 않는다(재번호 경로만 remap). 위 재센터라이징으로 다시 쓰인다.
- 위 3건 중 "옮긴 슬롯의 센터링을 자동으로 비워라"는 정책은 지시가 있으면 한 줄로 추가할 수 있다.

## 6. 영향도 분석

| 대상 | 영향 |
|---|---|
| `slot_setup` 스키마 | **불변**(DDL 무변경). 새 UPDATE 경로만 추가 |
| `replaceSlotSetup`(finalize) | 무영향 — 별도 메서드, 전량 교체 경로는 손대지 않음 |
| `renumberSlotIds` / `POST /mapping/renumber` | 무영향(코드 무변경). 저장 흐름에서 **뒤에** 호출될 뿐 |
| `setup_result.json` / `setup_artifact.json` | 배치 저장 시 재생성됨(재번호와 동일 규약). 스키마 변화 없음 |
| `slot_ptz.json` | 무접촉(§4) |
| 뷰어 오버레이(`presetKey` 기준) | 배치를 바꾸면 해당 슬롯이 **다른 프리셋 화면**에 그려진다(의도된 동작). ROI 좌표는 그대로이므로 위치는 재수집 전까지 부정확 |
| 분석 탭 '존' 요약 카드 / 주차면 목록의 zone 표기 | **불변** — 이번 변경은 매핑 표 한정 |
| `MappingRow` 타입 소비처 | `slotMapModel` 뿐(zone 미사용). 타입 변경 파급은 테스트 1건에서만 발생했고 수정 완료 |
| ActionAgent / DMAgent | 무영향 — `setup_result.json` 스키마·의미 불변 |
| 헤드리스(뷰어 비활성) 구성 | `POST /mapping/placement` 는 `deps.sqlite` 없으면 501 반환(기존 renumber 규약과 동일). 라우트 등록은 무조건이지만 동작은 주입 의존 |

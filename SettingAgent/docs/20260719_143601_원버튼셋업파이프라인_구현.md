## 1. 배경·목표

기존 정밀수집 흐름은 **수집 → 최종화(finalize) → 센터라이징(calibrate)** 3개 버튼을 사용자가 순서대로 눌러야 완결됐다(설계서 F1: `CaptureJob.finishRun()`은 상태 전이만 하고 finalize를 자동 호출하지 않음). 주차장관리 에이전트가 MCP/헤드리스로 셋업을 원버튼으로 굴리려면 브라우저 개입 없는 백엔드 체인이 필요했다.

이번 변경은 정밀수집 페이지에 **`autoChain` 옵트인**을 추가해, 체크박스를 켜고 `[수집 시작]`을 한 번 누르면 수집 완료 → 자동 최종화 → 자동 센터라이징 → `slot_ptz.json`/DB 저장까지 백엔드가 이어서 수행하도록 한다. 옵트인이므로 **체크박스 미체크(기본값) 시 기존 수동 3버튼 흐름은 완전히 그대로 보존**된다.

---

## 2. 사용법

1. 정밀수집 패널에서 `완료 후 자동 최종화+센터라이징` 체크박스(`#cap-autochain`, 기본 unchecked, `web/index.html:176`)를 켠다.
2. `[수집 시작]`을 누른다 — `web/app.js`의 `capStart()`가 `POST /capture/start` 바디에 `autoChain: $('cap-autochain').checked`를 실어 보낸다(`web/app.js:2057`).
3. N라운드 수집이 완료(done)되면 백엔드가 자동으로 finalize → (LPD 대상이 있으면) 센터라이징을 이어서 수행한다. 화면은 `cap-msg`에 `자동 최종화 중…` → `자동 센터라이징 중…` → `자동 셋업 완료 — 센터링 대상 N / 전체 M · 미대상 M−N`을 순서대로 표시한다.
4. 진행 중 상태는 신규 라우트 `GET /capture/pipeline`으로 언제든 조회 가능(REST 클라이언트/MCP 도구용).
5. 실패 시 체인은 그 단계에서 멈추고 사유가 `cap-msg`/`failure` 필드로 그대로 드러난다(위장 성공 없음). 수동 3버튼으로 이어갈 수 있다.

## 3. 신규 클래스 `SetupPipeline` (`src/pipeline/SetupPipeline.ts`, 169줄)

이 3단계 체인 **전용** 인메모리 상태머신이다(범용 워크플로 엔진이 아니며 단계가 하드코딩되어 있다). 기존 `CaptureJob`/`Finalizer`/`PtzCalibrator`/`SqliteStore`는 `Pick<...>`으로 표면만 좁혀 재사용하고, 그 내부 로직은 한 줄도 바뀌지 않았다.

### 3.1 상태전이

```
idle ──[onCaptureStart(true)]──▶ capturing
capturing ──[onCaptureFinished('done') + dets>0]──▶ finalizing
capturing ──[onCaptureFinished('stopped')]──▶ failed{capture, 'stopped(수동 정지)'}
capturing ──[onCaptureFinished('error')]──▶ failed{capture, 'capture error'}
capturing ──[onCaptureFinished('done') + dets==0]──▶ failed{finalize, '검출 0건 — finalize 미실행(DB 보호)'}
finalizing ──[finalize 정상 + targets>0]──▶ calibrating (calibrator.start() 발화)
finalizing ──[finalize throw]──▶ failed{finalize, err.message}
finalizing ──[targets==0]──▶ done{note:'센터라이징 스킵 — LPD 보유 슬롯 0'}
calibrating ──[onCalibrateFinished('done')]──▶ done
calibrating ──[onCalibrateFinished('error') / start() throw]──▶ failed{calibrate}
(종단 상태에서) ──[onCaptureStart(true|false)]──▶ 리셋 후 capturing|idle
```

소스 근거: `onCaptureStart`(`SetupPipeline.ts:61-75`), `onCaptureFinished`(`:78-96`), `runFinalizeThenCalibrate`(`:127-157`), `onCalibrateFinished`(`:99-103`).

### 3.2 가드 3종 (데이터 파괴·위장 성공 방지 — 이번 변경의 핵심)

| 가드 | 위치 | 조건 | 동작 | 방지 대상 |
|------|------|------|------|-----------|
| F10 | `onCaptureFinished`(`:90-93`) | `snapshot.dets.length === 0` | `finalizer.finalize` **자체를 호출하지 않고** `failed{finalize,'검출 0건…'}` | `Finalizer.replaceSlotSetup`의 DELETE+INSERT가 검출 없는 finalize에서 `slot_setup` 테이블을 파괴하는 것을 차단(메모리: finalize-slotsetup-wipe-fragility) |
| 가드2 | `runFinalizeThenCalibrate`(`:132-135`) | `finalizer.finalize()` throw | `calibrator.start()`를 **호출하지 않고** `failed{finalize, err.message}` | 최종화 실패 상태에서 센터라이징으로 넘어가 잘못된 slot_setup 기준으로 진행되는 것을 차단 |
| F6 | `runFinalizeThenCalibrate`(`:143-148`) | `expandPlateTargetsFromSlotSetup(views).length === 0`(전 슬롯 lpd 미검출) | `calibrator.start()`를 **호출하지 않고** `done + note` | `PtzCalibrator.run([])`이 무조건 writer를 호출해 **빈 `slot_ptz.json`으로 기존 파일을 덮어쓰는 것**을 차단 |

세 가드 모두 QA가 스파이 호출 횟수 0회로 직접 증명했다(`_workspace/03_qa_report.md` §4).

### 3.3 커버리지 정직 리포트 (`runFinalizeThenCalibrate:138-141`)

finalize 성공 직후(=slot_setup이 방금 새로 쓰인 시점) `store.getSlotSetup()` → 기존 export `expandPlateTargetsFromSlotSetup`(신규 로직 없음, `slotPtzWriter.ts` 재사용)로 LPD 보유 슬롯만 추린다. `coverage = {targets, totalSlots, uncovered}`로 **센터링 대상에서 빠진 슬롯 수를 숨기지 않고 상태에 담는다**. 이는 LPD 미검출 슬롯이 `slot_ptz` 상 홀로 남는 기존 한계(F5)를 정직하게 드러내는 것이며, 근본 해결(A2, 앞면중심 prior 전슬롯 커버)은 이번 범위 밖 후속이다.

### 3.4 공개 API·응답 shape

```ts
onCaptureStart(armed: boolean): void
onCaptureFinished(status: 'done' | 'stopped' | 'error'): void
onCalibrateFinished(state: 'done' | 'error'): void
isBusy(): boolean   // stage ∈ {finalizing, calibrating}
getStatus(): PipelineStatus
```

`PipelineStatus`(`SetupPipeline.ts:23-33`):
```ts
{
  armed: boolean;
  stage: 'idle'|'capturing'|'finalizing'|'calibrating'|'done'|'failed';
  startedAt?: string; endedAt?: string;
  failure?: { stage: 'capture'|'finalize'|'calibrate'; reason: string };
  finalize?: { slots: number; globalCount: number };
  coverage?: { targets: number; totalSlots: number; uncovered: number };
  note?: string;
}
```
조건부 스프레드로 undefined 필드는 생략된다(`getStatus:110-121`). 영속화 없음(인메모리 전용), 좌표 수치 없음 — round5 정밀도 규약 비대상.

## 4. 완료 콜백 훅 (기존 클래스 옵셔널 가산)

두 기존 잡 클래스에 **옵셔널** 완료 콜백을 추가했다. 미주입 시 완전히 no-op이라 회귀가 구조적으로 0이다.

- `CaptureJob`: `CaptureJobDeps.onFinished?: (status: 'done'|'stopped'|'error') => void`(`CaptureJob.ts:51`). `finishRun()` 말미에서 `try { this.deps.onFinished?.(status); } catch (e) { logger.warn(...) }`(`:277-283`) — 콜백이 throw해도 잡은 절대 죽지 않는다(경고 로그로 흡수).
- `PtzCalibrator`: `PtzCalibratorDeps.onFinished?: (state: 'done'|'error') => void`(`PtzCalibrator.ts:38`). `run()`의 done(`:141-142`)·error(`:146-147`) 경로 직후 private `notifyFinished()`(`:152-158`)가 동일하게 try/catch로 흡수한다.

`src/index.ts`(`:61-98`)는 생성 순서 역전(잡들이 pipeline을 필요로 하지만 pipeline은 잡들을 필요로 함)을 **클로저 전방참조**로 해소했다: `let pipeline: SetupPipeline | undefined`를 먼저 선언하고, `captureJob`/`calibrator` 생성 시 `onFinished: (s) => pipeline?.onCaptureFinished(s)` 형태로 배선한 뒤, 두 잡과 finalizer/sqlite가 모두 준비된 후 `pipeline = new SetupPipeline({...})`를 할당한다. 콜백은 런타임(잡 종단)에만 발화하므로 그 시점엔 이미 `pipeline`이 할당돼 있어 안전하다.

## 6. 프론트(web/app.js) 연쇄 처리 — F7/F8 보완 포함

설계서는 기존 폴링 상호배타 구조(F7: capPoll↔calFramePoll, F8: precise 탭 상시 폴링)로 "프론트 추종이 대부분 공짜"라 예상했으나, 구현 중 **실측으로 반증**되었다(구현 노트 §6 발견 이슈 1): `pollPlan(state)`는 capture가 `done`이면, `calPoll`도 `idle`이면 각자 자기 폴링을 멈춘다. 즉 capture done 후 백엔드가 계속 finalize→calibrate를 진행해도 프론트 폴이 이미 죽어 추종할 수 없다. 이를 최소 보정했다(임의 리팩토링 아님 — 설계 의도 실행에 필수):

- **capPoll 체인-폴 유지**(`app.js:1982-1990`): capPoll 말미에서 `pollPipeline()`을 병행 조회 후 `chainBusy = pl.armed && stage∈{capturing,finalizing,calibrating}`이면 capture가 done이어도 `plan.poll || chainBusy` 조건으로 폴을 재예약. 체인이 종단(done/failed)에 이르면 자연히 폴이 멎는다.
- **calibrate 폴 재기동**(`app.js:2019`): `pollPipeline`이 `prevPipelineStage !== 'calibrating' && stage === 'calibrating'` 전이를 감지하면 `calPoll()`을 1회 직접 호출 — idle에서 멈춰 있던 calPoll이 running을 잡아 이후 프레임 폴·`renderCalResult`는 기존 경로를 그대로 재사용한다.
- **오버레이 자동 반영**(`app.js:2012-2017`): `prevPipelineStage==='finalizing' && stage∈{calibrating,done}` 전이(=finalize 완료, slot_setup 방금 갱신) 시 `loadParkingSlots()+loadMapping()+drawRoiOverlay()+renderSlotList()` — 기존 수동 `capFinalize` 성공 분기와 동일한 갱신을 호출.
- **cap-msg 소유권**(`app.js:2006, 2020-2033`): `!pl.armed || stage==='idle'`이면 조기 반환하여 cap-msg에 전혀 관여하지 않는다(수동 흐름 무간섭). 무장 중에만 finalizing/calibrating/failed/done 각 단계 메시지를 렌더.
- `capStart()`: 바디에 `autoChain` 추가 + `prevPipelineStage = 'idle'` 리셋(`app.js:2057, 2059`) — 새 런 시작 시 이전 런의 전이 추적 잔재 제거.
- `web/index.html:176`: 체크박스 `#cap-autochain`(기본 unchecked, 명시적 옵트인).

구현 노트는 이 항목을 "발견 이슈"로 별도 명시했으며(§6-1), 설계 문서가 틀렸던 부분(F7/F8이 자동 추종을 보장한다는 가정)을 구현 단계에서 정정한 사례다. 신규 폴링 루프를 발명하지 않고 기존 `capPoll`/`calPoll`을 재사용한 점에서 설계 원칙(추측성 기능 금지) 안에서의 보정이다.

## 8. 한계·후속

- **LPD 홀(F5, 근본 미해결)**: 센터라이징 대상은 lpd 보유 슬롯만이며, 원거리 프리셋에서 LPD 미검출된 슬롯은 `slot_ptz`에 홀로 남는다. 이번 파이프라인은 이를 `coverage`로 정직하게 리포트만 할 뿐 해결하지 않는다. 근본 해결(앞면중심 prior로 전 슬롯 커버, A2)은 `docs/20260719_123634_셋업PTZ_목표재정의_점유판정논의_필요기능정리.md` §3 A2 참조 — 이번 범위 밖 후속. A2 도입 시 `SetupPipeline`은 무수정(타깃 펼침 함수 `expandPlateTargetsFromSlotSetup`만 교체되는 구조).
- **수동 경합 시 정직 실패만 제공, 잠금 확대 없음**: armed 체인 진행 중 사용자가 수동으로 최종화/센터라이징을 누르면 `calibrator.start()`가 'already running' 등으로 throw → `failed{calibrate}`로 정직하게 중단·보고된다. 수동 버튼 비활성화 같은 잠금 확대는 하지 않음(옵트인 사용자의 의도적 동시 조작에 방어 코드를 쌓지 않는 결정 — 설계서 §5.3).
- **수동 경로 stale slot_setup 가드는 범위 밖**: 수집만 다시 하고 최종화 없이 (수동)센터라이징을 누르는 stale 시나리오에 대한 가드는 이번 자동 체인과 무관하게 기존부터 미해결이며 후속 과제.
- **실 VPD 미가동으로 인한 라이브 미관찰**: 위 §7 한계 참조.

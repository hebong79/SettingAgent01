## 1. 목표 및 성공기준(관찰 가능)

**목표**: 정밀수집 페이지의 `시작` 1버튼이 **LPD 탐색 → 점유영역 → 센터라이징 → setup_result.json** 을 규정된 대기시간과 함께 순차 수행하고, `최종화` 는 그 결과를 화면에 그려 보여준다.

성공기준(전부 시뮬레이터 실동작으로 관찰):

| SC | 관찰 기준 |
|----|-----------|
| SC1 | `시작` 클릭 → `GET /capture/pipeline` 의 stage 가 `discovering → calibrating → done` 순으로 관측된다(capturing 단계 없음) |
| SC2 | 탐색 진행 중 슬롯 전환 간격이 실측 ≥0.5s (로그 타임스탬프 diff) |
| SC3 | 탐색 done 시점에 `GET /capture/slots` 의 `lpd` 보유 슬롯 = discovery `found` 수, 같은 슬롯의 `occupyRange` 도 비어있지 않다 |
| SC4 | 탐색 done → 센터라이징 start 사이 간격 ≥1.0s |
| SC5 | 센터라이징 슬롯 간 간격 ≥1.0s, 각 슬롯 `slot_ptz.json` 항목의 `plateWidth` 가 0.18~0.25 이거나, 그 밖이면 `reason: zoom_saturated|zoom_resolution_limit` + `ptz.zoom` 이 clampZoom 상한(36) 근처 |
| SC6 | 완료 시 `save/setup_result.json` + `save/Setup_YYYYMMDD_HHMMSS.json` 2벌이 갱신되고 `slots[].centering` 이 non-null 인 슬롯이 센터링 성공 수와 일치 |
| SC7 | 완료 메시지가 `cap-msg` 에 "정밀수집 완료 — 탐색 n/N · 센터링 m/M · setup_result.json 저장" 형태로 출력 |
| SC8 | `최종화` 클릭 → 현재 프리셋 화면에 LPD quad(노랑)·점유영역(빨강 반투명)·센터라이징 위치(작은 노란 원)가 동시에 보인다 |
| SC9 | UI 에 `바닥 LLM 사용`·`완료후 자동 최종화` DOM 이 존재하지 않는다 |
| SC10 | 리얼 소스 선택 상태에서 `시작` 은 **조용히 실패하지 않는다** — 미지원 사유가 화면에 뜬다(§7) |

## 2. 현재 구조(As-Is)

### 2.1 지금 `시작` 이 하는 일
`web/app.js:3596` → `capStart()`(`web/app.js:2229`) → `POST /capture/start`(`src/api/captureRoutes.ts:176`) → `CaptureJob.start()`.
바디: `count/intervalMs/checkpoint*/floorRoiUseLlm/vpdOnParkingOnly/vpdEnabled(false 고정)/autoChain`(`web/app.js:2240~2250`).

즉 **현재 시작 = 반복 관측 수집 잡(CaptureJob)** 이다. 요구 1~8 의 어디에도 이 반복 수집이 등장하지 않는다.

### 2.2 autoChain 과의 관계
`autoChain:true` 일 때만 `SetupPipeline` 이 무장(`src/api/captureRoutes.ts:235`)되고, 수집 done 후
`finalize → discovering → calibrating` 이 자동 연쇄한다(`src/pipeline/SetupPipeline.ts:83~135, 167~186`).

**요구 1~8 은 사실상 이 autoChain 체인의 "finalize 를 뺀" 형태**다. 즉 이번 작업의 본질은
"autoChain 을 옵션이 아니라 시작 버튼의 기본 동작으로 승격하고, 앞단(CaptureJob·finalize)을 떼어내는 것"이다.

### 2.3 현재 체인 각 단계 소유자
| 단계 | 소유 | 산출 |
|------|------|------|
| capturing | `CaptureJob`(`src/capture/CaptureJob.ts`) | 인메모리 dets 스냅샷 |
| finalizing | `Finalizer.finalize`(`src/capture/Finalizer.ts:197~252`) | setup_artifact.json + **slot_setup 전량 교체**(`replaceSlotSetup`) + `slot3d_front_center` 부트스트랩 |
| discovering | `PlateDiscoveryJob`(`src/calibrate/PlateDiscoveryJob.ts:117`) | plate_discovery.json + `slot_setup.lpd`·`occupy_range` 부분 UPDATE |
| calibrating | `PtzCalibrator`(`src/calibrate/PtzCalibrator.ts:363`) | slot_ptz.json + `slot_setup` 센터링 UPDATE + **setup_result 2벌** |

### 2.4 핵심 제약: 앵커(앞면중심)의 출처 — **자동화 확정**
`PlateDiscoveryJob` 대상 펼침은 `slot3dFrontCenter != null` 슬롯만이다(`src/calibrate/plateDiscoveryWriter.ts:45~59`).
그 값을 쓰는 경로는 **둘뿐**:
- `Finalizer`(`src/capture/Finalizer.ts:246`) — 파괴적(`replaceSlotSetup`)
- `POST /capture/slots/cuboid`(`src/api/captureRoutes.ts:468~533`) — **비파괴**(`upsertSlotFrontCenter`), UI "3D육면체 ROI생성" 버튼(`web/index.html:191`)

**"보이는 것"과 "저장된 것"의 괴리(이번 결정의 근거)**
ROI 파일을 로딩하면 화면에는 육면체와 앞면 중심이 곧바로 나타나지만, 그것은 **DB 값이 아니다**:
- 화면의 육면체·앞면중심은 클라이언트가 `projectCuboid` 와 `#cuboid-h` 슬라이더 높이로 **매 프레임 재계산하는 라이브 오버레이**이며, `web/app.js` 에는 `slot3dFrontCenter` 를 **읽는 코드가 없다**(전 참조가 버튼 관련 주석뿐).
- `POST /capture/slots/load-roi`(`src/api/captureRoutes.ts:349~392`)는 `slot3d_front_center` 를 건드리지 않는다(주석: "검출·점유·센터링은 초기값").

즉 사용자는 "이미 있다"고 보지만 DB 컬럼은 비어 있고, 탐색 대상 조건은 그 컬럼이다(`plateDiscoveryWriter.ts:48`) → 대상 0.

**확정 결정**: `load-roi` 라우트가 성공(`result.ok`)하면 **같은 요청 안에서 앞면 중심을 자동 산출·저장한다**(§5 W6).
→ 선행 순서는 **"ROI 파일 로딩 → 시작" 2단계**로 확정. `3D육면체 ROI생성` 버튼은 **존치**하며(높이 변경 후 재산출 용도) 자동 경로와 **동일 함수 W6 를 공유**한다.

## 3. To-Be 상태기계 / 시퀀스

### 3.1 상태기계(기존 `SetupPipeline` 재배선 — 새 엔진 금지)

선행(별도 요청, 사용자 1클릭): `POST /capture/slots/load-roi` → slot_setup 재구성 → camerapos.json 갱신 → **W6 앞면중심 자동 산출·저장**.

```
idle
 └─(POST /capture/start-precise)→ preflight
       ├ 앵커 보유 슬롯 0 → failed{stage:'discover', reason:'앞면중심 0 — ROI 파일 로딩 먼저(지면모델 확인)'}
       └ ok
 → discovering        : PlateDiscoveryJob.start({})           [슬롯당 LPD 1건 → 0.5s]
                        (프리셋 그룹 완료 시 점유영역 생성 → 0.3s)
 → (sleep 1.0s)                                               [요구3]
 → calibrating        : PtzCalibrator.start()                  [슬롯당 center→zoom → 1.0s]
 → done               : setup_result 2벌은 PtzCalibrator done 경로가 이미 기록
```

`capturing`/`finalizing` stage 는 **코드에서 지우지 않는다**(수동 `/capture/start` + autoChain 경로·기존 테스트 `test/setupPipeline.test.ts` 회귀 0). 새 진입 메서드만 추가한다(§5 W1).

### 3.2 대기시간의 구현 위치와 근거

| 대기 | 위치 | 근거 |
|------|------|------|
| 0.5s (슬롯당, 요구1) | **백엔드** `PlateDiscoveryJob.run` 슬롯 루프 말미(`src/calibrate/PlateDiscoveryJob.ts:180` 부근), 기존 `sleep` dep 사용 | 프론트는 잡 진행에 개입 불가(잡은 fire-and-forget 백그라운드). 슬롯 경계는 백엔드만 안다 |
| 0.3s (점유영역 생성 후, 요구2) | **백엔드** `PlateDiscoveryJob.saveSlotLpd` 프리셋 그룹 루프(`:242~249`) | 점유영역 생성이 프리셋 단위 집합연산이라 그 단위가 유일한 실제 경계(§13 Q3) |
| 1.0s (탐색→센터라이징, 요구3) | **백엔드** `SetupPipeline.onDiscoverFinished` 진입부(`src/pipeline/SetupPipeline.ts:110`) | 단계 전이는 파이프라인 소유 |
| 1.0s (슬롯당 센터라이징 후, 요구6) | **백엔드** `PtzCalibrator.run` 슬롯 루프 말미(`src/calibrate/PtzCalibrator.ts:402` 부근) | 위와 동일 |

전 구간 백엔드인 이유: **정본이 백엔드에 있고**, 프론트가 sleep 하면 브라우저 탭 백그라운드화·폴링 지터로 실제 카메라 동작 간격이 보장되지 않는다. 프론트는 폴링·표시만 한다(기존 `capPoll`/`discPoll`/`calPoll` 그대로).

### 3.3 시퀀스(성공 경로)

```
[UI] 시작 클릭
  → POST /capture/start-precise {source?}      (신규 얇은 라우트 = W2)
```

## 7. 요건13 — 리얼/시뮬 분리 설계

### 7.1 현재 결선
- 파이프라인 카메라는 **부팅 시 1개로 고정**: `cameraRuntime.selectedCameraId`(현재 `simulator-1`) → `sourceRegistry` 첫 항목 → `CameraSourceClient` (`src/index.ts:36~44`).

### 7.2 "기준은 시뮬레이터" 의 구체화
1. **동작 정본 = 시뮬레이터 경로**. 프리셋 순회(전 슬롯 loop), `resolvePresetPtz` 로 프리셋 PTZ 해석, 기하 게인 pre-aim, `pointZoomLadder:'auto'`(=시뮬은 사다리 미진입, `PtzCalibrator.ts:310~315`) — 이 조합이 정본이며 QA 성공기준(§1)은 전부 시뮬 기준으로 판정한다.
2. **리얼은 어댑터**. `RealPtzSource` 는 `CameraSource` 계약을 만족하므로 같은 `CameraSourceClient` 로 붙는다. 단 아래 구조적 차이가 있다:
   - **프리셋 테이블이 없다** — `listCameras()` 가 `presets:[{presetIdx:1,label:'현재 위치'}]` 하나만 반환(`src/viewer/RealPtzSource.ts:210~215`). → `resolvePresetPtz` 미해결 → discovery 는 `echo 폴백` warn(`PlateDiscoveryJob.ts:212`), calibrator 는 `0/0/1 폴백` warn(`PtzCalibrator.ts:660`) 로 **시야가 엉뚱해진다**.
   - 네이티브 센터링 지원(`RealPtzSource.ts:372`) → 사다리 경로 활성('auto').
   - zoom 단위 매핑 [0,16384] ↔ [1,36](`RealPtzSource.ts:397,406`).
3. **분기 지점(단 2곳)**
   - **분기 A(진입)**: `POST /capture/start-precise` 의 `source` → `sources.get(id)` → `CameraSourceClient` 생성 → W1 이 잡에 주입(W2·W4). 프론트는 이미 `state.source` 를 갖고 있고 `/calibrate/point` 에서 같은 필드를 보낸다(`web/app.js:2523` 부근) — 그 관용구를 그대로 복제.
   - **분기 B(사전 판정)**: `startPrecise` preflight 에서 대상 프리셋 집합과 소스의 `listCameras()` 프리셋 집합을 대조. 대상 프리셋이 소스에 없으면 **정직 실패**:
     `failed{stage:'discover', reason:'리얼 소스는 프리셋 순회 미지원(프리셋 1개) — 시뮬레이터 소스로 실행하거나 슬롯을 현재 위치로 한정하세요'}`
4. **미지원 항목의 실패 처리 원칙**: 조용한 강등 금지. 리얼에서 프리셋 미해결 시 `echo/0-0-1 폴백`으로 **카메라를 엉뚱한 데로 보내며 계속 도는 것**이 현재 거동이므로, 분기 B 로 **시작 전에 끊는다**.

→ 요구 순서(센터라이징 완료 → 최종화 → 표시)대로 누르면 **방금 만든 센터라이징이 즉시 파괴된다.**

**설계 결론**: 요구9 의 최종화는 "표시(render)" 동작이다. 따라서
- `최종화` 버튼 핸들러를 `POST /capture/finalize` 호출에서 **표시 전용**으로 전환:
  `#roi-db` 체크 ON → `loadParkingSlots()` → `roiHidden=false` → `drawRoiOverlay()` → `renderSlotList()` (모두 기존 함수, `capFinalize` 후반부 `:2305~2311` 그대로 재사용)
- 라우트 `/capture/finalize` 자체는 **삭제하지 않는다**(수집 경로·테스트 보존). 다만 정밀수집 UI 에서는 도달 불가가 된다.
- 버튼 라벨을 `최종화(결과 표시)` 로 바꿀지는 Q5.

### 9.2 렌더 경로
`drawRoiOverlay`(`web/app.js:399`)는 이미 `#roi-db` 체크 시 DB 소스로 전환한다:
- LPD: `web/app.js:993` — `row.lpd` → `drawPlateQuad`(노랑, `:946`)
- 점유영역: `web/app.js:1026` `drawDbOccupancy` — `row.occupyRange`(빨강 반투명), 호출부 `:518~521`
- **센터라이징 위치: 없음 → W7**

### 9.3 W7 `drawDbCentering(ctx, rows)` 사양
```
for row of rows:
  if (!row.centered || !row.lpd) continue           // centered=1 인 슬롯만
  c = row.lpd quad 의 bounding rect 중심             // 화면 좌표계 기준점
  ctx.arc(c.px, c.py, 5, 0, 2π); strokeStyle '#ffd60a'; lineWidth 2; stroke()
```
호출 위치: `drawRoiOverlay` 의 DB 분기(`:993` LPD 렌더 직후) 1줄.

**왜 lpd 중심인가**: `slot_setup` 에 저장된 센터링 산출물은 **PTZ 값(pan/tilt/zoom)** 이지 프리셋 프레임 상의 점이 아니다. 프리셋 프레임에서 "그 슬롯이 센터라이징된 지점"의 유일한 기존 좌표는 센터링의 표적이었던 `lpd` OBB 중심이다. PTZ 를 프리셋 프레임 좌표로 역투영하려면 게인 역함수를 새로 써야 하므로 **요건10 위반**이다. → Q6 로 확인 요청.

### 9.4 오버레이 유지 정책 정합
"정밀수집 종료 후 VPD/LPD/점유 오버레이 유지·삭제금지, DB 소스는 `#roi-db` 체크 시 표시" 규약과 정합한다 — 본 설계는 `#roi-db` 를 **최종화 버튼이 자동으로 켜는 것**뿐이며, 삭제 경로는 만들지 않는다.

## 13. 미해결 질문·가정 (확인 못 한 것은 그렇게 적음)

**Q1 (앵커 부트스트랩) — ✅ 결정됨 (2026-07-22, 마스터 승인).**
> **ROI 파일 로딩이 끝나면 `slot3d_front_center` 를 자동 산출·저장한다. `3D육면체 ROI생성` 버튼을 따로 누를 필요가 없다.**

- **근거**: 화면에 보이는 육면체·앞면중심은 클라이언트가 `projectCuboid` + `#cuboid-h` 로 매 프레임 재계산하는 **라이브 오버레이**이고 `web/app.js` 는 `slot3dFrontCenter` 를 읽지 않는다. 반면 `load-roi`(`captureRoutes.ts:349~392`)는 그 컬럼을 채우지 않는다 → **보이는 것과 저장된 것이 달라** 사용자가 "이미 있다"고 믿는 상태에서 탐색 대상이 0 이 된다(`plateDiscoveryWriter.ts:48`).
- **구현**: W6 확정 승격(§5.1). `load-roi` 말미(`:388` 부근), `result.ok` 일 때만.
- **강등**: ground 미설정/산출 실패는 **로딩을 실패시키지 않고** `issues[]` 강등(§5.1 표). 슬롯 단위 실패는 기존 `skipped[]` 유지(위장 금지).
- **버튼 존치**: `3D육면체 ROI생성` 은 높이 변경 후 재산출 용도로 남기고, 자동 경로와 **동일 함수 W6 공유**.
- **파생 리스크**: R8(높이 불일치), R9(응답 지연) → §12. 파생 질문 Q11.

**Q2 (대기시간 적용 범위 — 결정 필요).** `index.ts` 에서 잡 인스턴스에 `betweenSlotMs` 를 주입하면 **수동 `/discover/ptz`·`/calibrate/ptz` 버튼도 같이 느려진다**(같은 싱글턴 인스턴스). 허용인가, 아니면 `startPrecise` 호출 시에만 적용되도록 `start()` 인자로 넘길 것인가? (후자가 회귀 0 이지만 시그니처가 하나 더 늘어난다.)

## 1. 배경 · 목적

### 1-1. 기존 구조 (결정형 빌더가 권위)

`FloorRoiReviewer.review()` → `resolveFloorPolygon(llmPoly, vehicle, plateQuad, plateRect)` 경로에서:

- LLM 이 4점 폴리곤을 반환해도, 그 폴리곤은 **깊이 힌트(`depthFloor` = 앞→뒤 축 span)로만** 소비됐다.
- 최종 좌표는 `buildPlateAnchoredQuad(vehicle, plateQuad, depthFloor)` 가 **번호판(VPD) 각도·좌우중앙·세로배치**를 강제해 재구성했다.
- 그 위에 `expandPolygonToContainRect(base, plate)` 로 **번호판 포함을 강제**했다.
- 즉 **번호판 앵커링(결정형)이 권위**였고, LLM 은 보조 신호였다. 프롬프트(`floor_roi.yaml`)도 "번호판 각도·좌우중앙에 맞춘 회전 사변형"을 지시했다.

### 1-2. 변경 목적 (LLM 폴리곤이 메인)

**LLM 유효 폴리곤을 최종 출력의 메인으로 격상**하고, 결정형 빌더(`buildPlateAnchoredQuad`)를 **LLM 실패 시 폴백·안전망으로 격하**한다. 좌표 **생성**(원근 접지면 사각형 추정)은 맥락·시각 판단이 필요한 LLM 두뇌의 몫이라는 MCP 경계 원칙을 강화하는 방향이다. 좌표 **검증·강등·안전망**(클램프/볼록껍질/정렬/마진/비겹침/폴백)은 결정형 도구로 유지한다.

### 1-3. 사용자 확정 결정 2건 (변경 불가 전제)

1. **per-vehicle 파이프라인 유지** — 차량(슬롯)마다 LLM 1회, 4점 폴리곤 1개. `spaces[]` 다중주차면 구조로 바꾸지 않는다.
2. **차량 bbox 는 대상 표시용으로만** LLM 에 전달. 번호판 각도·기하 앵커링(VPD 앵커링)은 **LLM 입력·프롬프트에서 완전 제거**한다.

리더 확정 3세부 결정: ① 번호판 포함강제(`expandPolygonToContainRect`)를 LLM 메인 경로에서 제거하고 폴백 경로에서만 유지, ② 고아가 된 `depthFloor` 파라미터 제거, ③ `FloorRoiInput` 의 `plate?`/`plateQuad?` 필드 제거.

## 2. 파일별 변경 상세

### 2-A. `config/prompts/floor_roi.yaml` — 프롬프트 철학 전환

- 상단 주석·`system` 블록에서 **번호판 기준·번호판 각도 앵커링 규칙(기존 1~7)을 전면 삭제**하고, "차가 깔고 앉은 **바닥(지면) 슬롯 사각형**을 매핑한다"는 철학으로 교체.
- 대상 지정을 재정의: 차량 bbox 는 "이 위치의 차량이 점유한 바닥 슬롯을 그려라"는 **대상 표시용**이며, "bbox 에 억지로 맞추지 말라"고 명시.
- 그리는 대상을 접지 기하로 명시: 앞(하단)변=타이어 접지선, 뒤(상단)변=슬롯 먼끝(휠스톱/후방 주차선), 좌우변=옆 슬롯 경계. 지붕·차체 높이·창문·그림자·범퍼 돌출 **제외**, 세로로 긴 실루엣 금지.
- `user` 템플릿에서 `{{plate}}`/`{{plateQuad}}` 두 줄 **제거**. `{{camIdx}}`/`{{presetIdx}}`/`{{vehicle}}` 유지.
- **출력 스키마·좌표 규약은 불변**: 정규화 0~1, 4점 순서 `[앞왼, 앞오, 뒤오, 뒤왼]`(앞=이미지 하단/카메라 근접, 시계방향), `{"polygon":[4점], "confidence":0.0~1.0}`.

### 2-B. `src/capture/floorRoi.ts` — `resolveFloorPolygon` 권위 역전 (핵심)

역전 후 최종 로직(`floorRoi.ts:236~249`):

```ts
export function resolveFloorPolygon(llmPoly, vehicle, plateQuad?, plateRect?): NormalizedPolygon {
  // LLM 메인: 유효 정규화 폴리곤이면 각도 재앵커·번호판 포함강제 없이 그대로 채택.
  const llmNorm = normalizePolygon(llmPoly);
  if (llmNorm) return llmNorm;
  // 폴백: LLM 무효/부재 → 번호판 앵커 빌더 + 번호판 포함강제(기존 불변식 유지).
  const base = buildPlateAnchoredQuad(vehicle, plateQuad);
  const containTarget = plateRect ?? (plateQuad ? quadBoundingRect(plateQuad) : predictPlateRect(vehicle));
  return expandPolygonToContainRect(base, containTarget);
}
```

- **시그니처 불변** `(llmPoly, vehicle, plateQuad?, plateRect?)`. `plateQuad`/`plateRect` 는 이제 **폴백 경로에서만** 소비 → 호출부(`FloorRoiReviewer.ts:93`) 무수정 호환.
- 유효 LLM → `normalizePolygon` 안전망(클램프·볼록껍질·캐노니컬 정렬·R1 마진)만 통과시켜 그대로 반환. 빌더·포함강제 코드에 **도달하지 않음**.
- `expandPolygonToContainRect`(번호판 포함강제)는 **LLM 메인 경로 미적용, 폴백 경로에서만 유지**.

**고아 제거(`buildPlateAnchoredQuad`):** 3번째 파라미터 `depthFloor` 와 `D` 계산의 `depthFloor ?? 0` 항 제거. 역전 후 전 호출처가 2인자 이하로만 호출한다.

```ts
// after (floorRoi.ts:167~186)
export function buildPlateAnchoredQuad(vehicle, plateQuad?): NormalizedQuad {
  ...
  const D = Math.max(vehicle.h * FLOOR_DEPTH_RATIO, hp * (1 + PLATE_CONTAIN_MARGIN));
```

`normalizePolygon`/`expandPolygonToContainRect`/`deconflictPolygons`/`estimatePlateQuadFromNeighbors`/`predictPlateRect` 본체 로직은 **무수정**. 오직 `resolveFloorPolygon` 조립 순서 역전 + `buildPlateAnchoredQuad` 의 `D` 항 정리만 수행.

### 2-C. `src/capture/FloorRoiReviewer.ts` — LLM 입력에서 번호판 차단

`recognizeFloorRoi` 호출 인자에서 `plate`/`plateQuad` 스프레드 제거(`vehicle` 만 전달, `FloorRoiReviewer.ts:78~84`):

- L92 인접 주석 1줄을 "LLM 메인, effQuad/plate 는 폴백 전용"으로 갱신(변경 라인 직결 정합).

`plate?: NormalizedRect`/`plateQuad?: NormalizedQuad` 두 필드 제거(`SetupBrain.ts:101~110`). 남은 필드: `camIdx`/`presetIdx`/`imageBase64`/`vehicle`/`slotHint?`. `vehicle` 주석을 "대상 표시용"으로 갱신. 필드 제거로 파일 내 유일 사용처가 사라진 `NormalizedQuad` **import 고아 제거**(`NormalizedRect` 만 유지).

## 3. 데이터 흐름 (before / after)

### Before (결정형 빌더 권위)

```
LLM 폴리곤 ──(깊이 힌트 depthFloor로만 소비)──┐
                                              ▼
buildPlateAnchoredQuad(vehicle, plateQuad, depthFloor)  ← 번호판 각도·좌우중앙 강제 (권위)
                                              ▼
expandPolygonToContainRect(base, plateRect)   ← 번호판 포함 강제(항상)
                                              ▼
                          저장 → deconflictPolygons → SqliteStore
```

### After (LLM 폴리곤 메인)

```
LLM 폴리곤
   │
   ▼
normalizePolygon (클램프 → 볼록껍질 → 캐노니컬 정렬 → R1 마진)
   │
   ├─ 유효(4~10점, 껍질 ≥4정점) ──────────────► 그대로 채택 (메인, 재앵커·포함강제 없음)
   │
   └─ 무효/null (점수 밖·NaN·공선붕괴)
          │
          ▼
       buildPlateAnchoredQuad(vehicle, plateQuad)   ← 폴백 전용
          │
          ▼
       expandPolygonToContainRect(base, containTarget)  ← 폴백 경로에서만 번호판 포함강제
          │
          ▼
                          저장 → deconflictPolygons → SqliteStore (안전망 유지)
```

핵심 차이: 메인 경로에서 **번호판 각도 재앵커와 포함강제가 사라지고**, LLM 좌표가 안전망(클램프/볼록껍질/정렬/마진)만 통과해 그대로 저장된다. 폴백 경로(무효/null)의 불변식은 종전 그대로.

## 4. 좌표 규약 불변 확인

정규화 4점 순서 `[앞왼, 앞오, 뒤오, 뒤왼]`(앞=이미지 하단 y 큰 쪽/카메라 근접, 시계방향)은 **변경 전후 불변**이다. 근거:

- **생성측(LLM/REST)**: `floor_roi.yaml` 좌표 규약 + `FloorRoiResultSchema`(`polygon: {x,y}[].length(4)`, `confidence`) 모두 무수정.
- **소비측(결정형)**: `normalizePolygon` → `orderConvexCanonical`(감김=음부호, 시작=max y·동률 min x)이 [0]=앞왼(하단 y 최대)·시계방향을 강제. 이 정렬은 무수정.
- QA 확인: `floorRoi.test.ts:129` "4점 캐노니컬"이 q0/q1(앞)이 q3/q2(뒤)보다 y 큼 + 앞왼<앞오·뒤왼<뒤오를 단언(그린).

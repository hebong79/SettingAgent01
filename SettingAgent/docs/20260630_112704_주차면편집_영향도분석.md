## 2. 의존성 그래프 추적 (파급 분석)

### 2-1. @parkagent/types / SetupArtifact shape — 불변 → 하류 무영향
- 본 작업은 **새 필드·새 타입을 추가하지 않는다.** PUT 의 zod `SetupArtifactSchema` 는 계약(`packages/types/src/index.ts` + `src/domain/types.ts`)과 **동형**(NormalizedRect/Point/Quad·Preset·ParkingSlot·GlobalSlotIndex)일 뿐 계약을 바꾸지 않는다.
- 편집은 **같은 형식의 산출물(`setup_artifact.json`)을 갱신**할 뿐이다.
- **전파 결과**: `setup_artifact.json` 을 읽는 **ActionAgent·DMAgent 는 무영향**. 이들은 형식이 동일한 파일을 그대로 읽으며, 값(슬롯 삭제·ROI 변경·전역번호 재정렬)이 사용자 편집으로 달라질 뿐 계약 위반은 없다.

### 2-2. REST 계약 변경 — 쓰기 1개 가산, 읽기·기타 라우트 불변
- 신규: `PUT /mapping`, `PUT /viewer/api/mapping`(동일 `saveMappingHandler`).
- 불변: `GET /mapping`·`GET /viewer/api/mapping`(읽기 직접 반환, 404 보존), `/setup/*`, `/capture/*`, `/brain/*`, `/health`, 뷰어 카메라·스냅샷·이동 라우트.
- **클라이언트 영향**: 뷰어 `app.js` 만 새 PUT 을 호출(`saveMapping`). 기존 REST 클라이언트(MCP 도구·Action/DM)는 GET 계약만 쓰므로 무영향.
- **테스트 영향**: 신규 PUT 경계는 `mappingPut.test.ts`(+검증자 inject)로 커버. 기존 281 테스트는 라우트 가산이라 회귀 없음.

### 2-3. 공유 도메인 타입(SlotState/ParkingEvent 등) — 본 작업 비관여
- 본 작업은 SettingAgent 산출물(설치 시 1회) 편집이며 런타임 도메인 이벤트 타입(SlotState/ParkingEvent 등)은 건드리지 않는다. 해당 타입 파급 없음.

### 2-4. 산출물 일관성 게이트 — validateCoverage 재사용
- PUT 은 저장 전 `validateCoverage(globalIndex, slots)` 로 globalIndex↔slots 집합 일치를 강제한다. 불일치(missing/extra) 시 **400 + 미저장**으로 파일을 보호한다.
- 클라이언트 재구성(`removeSlot`/`rebuildGlobalIndex`/`reorderGlobalIndex`)에 버그가 있어도 서버 게이트가 1차 차단 → `setup_artifact.json` 이 정합 깨진 상태로 기록되지 않는다. 이는 하류 Action/DM 의 산출물 신뢰를 유지한다.

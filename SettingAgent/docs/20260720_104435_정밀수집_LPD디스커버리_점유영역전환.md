## 1. 배경 및 목표

정밀수집 "시작"(`#cap-start`)의 자동 연쇄(autoChain)가 이미 `capture → finalize → centering` 순으로 배선되어 있었으나,
LPD(번호판) 채우기가 **CaptureJob 의 전체프레임 검출**에 의존하고 있었다. 마스터 요청은 이 LPD 채우기를
**3D 육면체 앞면중앙점을 기준으로 한 크롭-줌 탐색 loop(discovery)** 로 바꾸고, 그 결과로 점유영역까지 만드는 것이었다.

목표(마스터 나열 순서): ① LPD 찾기(앵커 loop) → ② 점유 영역 만들기 → ③ 최종화 → ④ 저장(DB/파일) → ⑤ 센터라이징 → ⑥ 저장.

## 2. 설계 결정의 요지 (01_architect_plan.md §0·§A·§B·§C)

### 2-1. 앵커 의존성으로 인한 실행순서 재해석 (핵심 편차)

마스터가 나열한 문자 그대로의 순서는 `LPD(1) → 점유(2) → 최종화(3)` 이다. 그러나 코드 정밀 매핑 결과:

- discovery(`PlateDiscoveryJob`)의 대상 산출(`expandDiscoveryTargets`)은 `slot_setup.slot3d_front_center` 값이
  있는 슬롯만 대상으로 삼는다.
- `slot3d_front_center`는 **오직 `Finalizer.finalize`만 부트스트랩**한다(`slotFrontCenter(sp.points, groundModel, H=1.5)`,
  검출과 무관한 기하 소스, 매 finalize 재계산).

따라서 discovery는 **finalize 이후에만** 유효한 대상을 얻는다. 콜드스타트(DB에 front_center가 없는 최초 셋업)에서
"LPD를 finalize보다 먼저" 돌리면 discovery 대상이 0이 되어 마스터의 실질 의도(최종 lpd = 앵커 loop 산출)를 달성할 수 없다.
→ 설계자는 마스터 의도를 **"방법"(앵커 loop 산출을 최종 lpd로 쓴다)**으로 해석하고, **"문자 그대로의 순서"**는
앵커 의존성 때문에 성립할 수 없다고 판단, 후보 C로 진행했다(Q1, 자율진행 원칙에 따라 착수·회신 대기 없이 진행,
설계서 §G Q1).

### 2-2. 후보 비교

| 후보 | 요지 | 판정 |
|---|---|---|
| (a) CaptureJob 내부 LPD를 discovery loop로 치환 | 라운드마다 전 슬롯 discovery 재실행(N배 낭비), front_center capture 시점 부재 | 기각 — 구조 부적합 |
| (A-literal) capture→discovery→finalize→centering (마스터 문자 순서) | 콜드스타트 시 앵커 부재로 discovery 대상 0, capture LPD 제거 시 점유영역 붕괴 | 기각 — warm DB 전제에서만 성립 |
| **(C) autoChain에 discovering 단계 삽입(채택)** | finalize가 앵커를 먼저 깔고, discovery가 그 위에서 lpd를 부분 UPDATE로 덮어씀 | **채택** |

최종 실행 순서(후보 C):

```
capturing(VPD off)
  → finalizing(집계·기하 front_center 부트스트랩·artifact/파일 저장)
  → discovering(앞면중심 하향앵커 loop → slot_setup.lpd 부분 UPDATE)   ← 신규 삽입 단계
  → calibrating(discovery가 채운 lpd로 센터라이징 → slot_ptz.json + centering_slot)
  → done
```

### 2-3. 앵커 정의: 하향앵커 유지(변경 없음)

"앞면중앙점"의 실제 조준점은 `slot3d_front_center`(육면체 앞면 4모서리 평균, 지상 약 0.75m)가 아니라
`lowerFrontAnchor(roi, front_center, plateH=0.4)`(앞 edge 중점과 front_center의 선형보간 하향값, 판 높이
0.3~0.5m에 맞춤)이다. 이는 기존 `expandDiscoveryTargets`가 이미 산출하던 값으로, **신규 코드 없이 그대로 재사용**했다
(메모리 `lpd-discovery-anchor-grid-exclusivity` 정책과 일치).

## 3. 변경/신규 클래스·함수 설명

### 3-1. `src/calibrate/PlateDiscoveryJob.ts`

- **`PlateDiscoveryJobDeps.onFinished?: (state: 'done' | 'error') => void`**(옵셔널) 추가 — `PtzCalibrator`의
  완료 콜백 패턴을 그대로 미러.
- **`notifyFinished(state)`**(신규 private) — `run()` 종단부(`state='done'`/`state='error'`)에서 호출.
  `try/catch`로 콜백의 throw를 흡수하여 콜백 오류가 잡 상태머신을 죽이지 않도록 함.
- **`saveSlotLpd(items)`**(이터레이션 2 확장) — found 슬롯에 한해
  `occupyRange = stringify5(buildPlateAnchoredQuad(quadBoundingRect(it.lpdOrig), it.lpdOrig))`를 계산하여
  `SlotLpdRow`에 동봉. `buildPlateAnchoredQuad`/`quadBoundingRect`는 `Finalizer.ts:254`의 판-only 점유영역
  경로와 **동일 함수를 재사용**(이중구현 없음). 계산 실패 시 `occupyRange`를 생략(undefined)하여 lpd만 저장하고
  occupy_range는 기존 값을 보존한다(위장 저장 금지).
- 시그니처·`start`/`getStatus`/`getLastFrame`은 불변. `onFinished` 미주입 시(수동 `/discover/ptz` 경로) no-op.

### 3-2. `src/pipeline/SetupPipeline.ts`

- `PipelineStage`에 `'discovering'` 추가, `failure.stage`에 `'discover'` 추가.
- `SetupPipelineDeps.discovery: Pick<PlateDiscoveryJob, 'start' | 'getStatus'>` 신규 필수 dep.
- `runFinalizeThenCalibrate`: finalize 성공 후 `stage='discovering'` → `discovery.start({})`(전 프리셋) 발화.
  `start()` throw(경합) 시 `fail('discover', ...)`.
- **`onDiscoverFinished(state)`**(신규 public) — `error`면 `fail('discover', 'discover error')`(F6, 위장 성공 금지).
  `done`이면 `expandPlateTargetsFromSlotSetup(store.getSlotSetup())`으로 **커버리지를 discovery 완료 이후 시점에서
  재계산**한 뒤, 대상 0이면 `note='센터라이징 스킵 — LPD 보유 슬롯 0'` + `finish('done')`, 아니면 `stage='calibrating'`
  → `calibrator.start()`.
- `isBusy()`에 `'discovering'` 포함 → discovery 진행 중 `/capture/start` 409.
- **핵심 이동**: 커버리지 산출 위치가 (기존) finalize 직후 → (변경 후) discovery 완료 직후로 이동. 최종
  `slot_setup.lpd`를 이제 discovery가 채우므로, 커버리지도 discovery 반영본 위에서 계산해야 정확하다.

### 3-3. `src/capture/SqliteStore.ts` — `upsertSlotLpd`

- 두 개의 prepared statement로 조건 분기:
  - `r.occupyRange === undefined` → `UPDATE slot_setup SET lpd_obb=?, updated_at=? WHERE slot_id=?`(occupy_range 무접촉)
  - 그 외(string 또는 명시적 null) → `UPDATE slot_setup SET lpd_obb=?, occupy_range=?, updated_at=? WHERE slot_id=?`
- **설계서 §I는 단일 statement + `r.occupyRange ?? null`을 지시했으나, 구현자가 조건부(2-statement)로 조정**했다.
  근거: `upsertSlotLpd`에는 `captureRoutes.ts:341` 수동 `/capture/slots/lpd`("현재화면 LPD DB추가" 버튼)라는
  **두 번째 프로덕션 호출자**가 있고, 이 경로는 `occupyRange`를 넘기지 않는다. 리터럴 SQL(`?? null`)을 그대로
  적용하면 이 수동 경로가 discovery/finalize가 채운 occupy_range를 매번 `null`로 wipe하는 회귀가 발생하고,
  기존 wipe-safety 봉인 테스트(`sqliteStore.test.ts:301`)도 깨진다. 설계서 §237의 명시 의도("미검출은 스킵 —
  기존 값 보존, wipe 없음")를 지키기 위해 조건부로 조정했으며, 검증자가 이 편차를 정당하다고 확인했다
  (03_qa_report.md §4).

### 3-4. `src/capture/types.ts` — `SlotLpdRow`

## 4. 수정 로직의 의도와 동작 (이터레이션 1 → 2)

### 이터레이션 1 (설계 §D)

목표는 "최종 persist되는 `slot_setup.lpd`가 전체프레임 검출이 아니라 앵커 loop 산출"이 되게 하는 것이었다.
capture 단계의 전체프레임 LPD 블록은 **점유영역 앵커링용**으로 남기고, discovery가 lpd 컬럼만 부분 UPDATE로
덮어쓰는 구조로 구현되었다.

### 이터레이션 2 (리더 라이브 재분석, 설계서 §I)

리더의 라이브 실증(count=1 auto-chain) 결과, 전이·discovery(`found:17/17`)·최종 lpd 좌표 일치는 정상이었으나
로그에 `"occupancy","accepted":0,"slots":0"` → **`occupy_range = null`**(마스터 순서 2번 "점유 영역 만들기"가
미산출)이 관측되었다.

- **근본원인**: `Finalizer`의 occupy_range는 **capture 전체프레임 LPD의 accepted 클러스터(hit)**에서만 생성된다
  (`Finalizer.ts:254`). VPD off 상태에서는 전체프레임 LPD 검출이 희소하여 hit이 0이 되고, occupy_range가 채워지지
  않았다.
- **재분석 결론**: 마스터 순서상 "점유 영역"은 "LPD 찾기" 직후에 오므로, **발견된 LPD로부터 점유영역을 만드는 것이
  본래 의도**다. discovery가 이제 슬롯별 판을 신뢰성 있게 채우므로, 점유영역도 그 판 quad로 결정형 생성하는 것이 옳다.
- **결정**: `PlateDiscoveryJob.saveSlotLpd`가 found 슬롯의 `lpdOrig`(판 quad)로
  `occupy_range = buildPlateAnchoredQuad(quadBoundingRect(lpdOrig), lpdOrig)`를 계산해 행에 동봉하고,
  `upsertSlotLpd`가 이를 조건부 부분 UPDATE로 반영한다. Finalizer의 occupy_range 경로는 **무수정**(capture hit이
  있으면 그대로 동작, 회귀 0) — discovery가 finalize 뒤에 실행되므로 판이 발견된 슬롯은 최종적으로 discovery
  산출값으로 갱신된다.

## 6. 동작확인 — 리더 라이브 실증 (goal/loop, count=1 auto-chain, 2026-07-20)

- 전이 관측: `capturing → finalizing → discovering → calibrating → done` 순으로 진행 확인.
- discovery: `found:17/17`, 17개 판 좌표 전부 고유·단조정렬·중복 0(배타성 유지, 위장 아님).
- `slot_setup.lpd` = `data/plate_discovery.json`의 `lpdOrig`와 좌표 일치(전체프레임 산출이 아님을 확인).
  preset2 slot8, slot9(이전엔 null)도 격자줌 재탐색으로 회수됨.
- `occupy_range` 17/17 생성, 각 occupy 사변형 중심이 판 중심과 4점 기준 ±0.003 이내(이터레이션 1에서는
  `occupy_range=null`이었던 부분이 이터레이션 2에서 해결됨).
- 컬럼 무결성: 부분 UPDATE가 `roi`/`front_center`/`pan`/`tilt`/`zoom`/`centered`를 훼손하지 않음. 수동
  `/capture/slots/lpd` 경로도 occupy_range를 보존(wipe 없음)함을 실증.
- 센터라이징: 전 프리셋 수렴, `slot_ptz.json` 갱신, `centered:true`.

이 실증은 **리더가 실제 서비스(카메라/LPD/DB)로 수행한 라이브 검증**이며, 아래 QA 유닛 커버리지와는 별개다.

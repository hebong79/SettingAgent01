## 1. 배경 · 요구사항

### 1-1. 문제
Analysis 탭 "전역 인덱스 수동 매핑" 패널에서 전역 ID(=주차면 번호)를 바꿔 저장하면, 기존 흐름은
`saveManualIndex()`(web/app.js) → `applyManualGlobalIds` → `PUT /mapping` → `saveMappingHandler` → `repo.saveArtifact()` 로 이어져 **`setup_artifact.json` 파일 한 곳만 갱신**되고 있었다. DB(`slot_setup.slot_id`)와 `setup_result.json`·`slot_ptz.json`은 전혀 반영되지 않아, "매핑을 바꿨는데 DB/센터링 결과와 번호가 어긋나는" 단절이 있었다.

### 1-2. 마스터 확정 A안
"매핑을 다시 하는 이유 = json 파일에 다시 저장하려는 것." 이라는 전제하에, **전역번호==slot_id 를 하나의 신원으로 결합**한다. 전역 ID 변경·저장 시:
1. DB `slot_setup.slot_id` 가 그 값으로 재번호(저장)되고,
2. `setup_result.json` 이 재생성되고,
3. `slot_ptz.json` / `setup_artifact.json` 도 함께 갱신된다.

## 2. 해결 연쇄 (설계 요지)

```
전역ID 변경(웹 UI, 표) → [저장] 클릭
  └─ saveManualIndex() : applyManualGlobalIds 로 클라 측 순열 1차 검증(빠른 피드백)
      └─ POST /mapping/renumber  { mapping: [{oldSlotId, newSlotId}, ...] }
          ├─ 1) validateRenumberMapping(currentIds, mapping)   ← 서버측 재검증(정본 게이트)
          │      실패 시 400, DB·파일 전부 무접촉(원자성)
          ├─ 2) SqliteStore.renumberSlotIds(idMap)             ← DB 트랜잭션 재번호(전 컬럼 보존)
          │      실패 시 500, 트랜잭션 자동 롤백
          ├─ 3) renumberSlotPtzFile(outFile, idMap)            ← slot_ptz.json 리맵(best-effort)
          ├─ 4) writeSetupResultFiles(getSlotSetup(), saveStore) ← setup_result.json 재생성(best-effort)
          └─ 5) repo.saveArtifact(buildArtifactFromSlotSetup(getSlotSetup())) ← setup_artifact.json DB 재빌드(best-effort)
      └─ 응답 { ok, renumbered, slotPtz, setupResult, artifactSaved }
  └─ 성공 시 loadMapping() → renderAnalysis() → renderSlotList() 재렌더
```

**원자성 규약:** 1)의 검증 실패 시 DB·파일 어느 것도 건드리지 않는다. 2)의 DB 재번호는 단일 트랜잭션(all-or-nothing) — 이것이 커밋되면 "진실"이다. 3)~5) 파일 3종은 순차 best-effort — 개별 실패는 로그·응답 필드로만 노출되고 앞 단계를 되돌리지 않는다(DB가 이미 정본이므로 롤백 불요).

### REST 계약

**`POST /mapping/renumber`** (+ 뷰어: `POST /viewer/api/mapping/renumber`, 동일 동작)

요청:
```json
{ "mapping": [ { "oldSlotId": 1, "newSlotId": 3 }, { "oldSlotId": 2, "newSlotId": 1 }, { "oldSlotId": 3, "newSlotId": 2 } ] }
```

응답(200):
```json
{
  "ok": true,
  "renumbered": 3,
  "slotPtz": "written",
  "setupResult": { "archive": "<path|null>", "fixed": "<path|null>" },
  "artifactSaved": true
}
```

오류:
- 501 `{ error: 'sqlite not configured' }` — sqlite deps 미주입.
- 400 zod 실패 `{ error: 'invalid body', detail }` — 스키마 위반(정수·양수·min(1) 등).
- 400 순열 위반 `{ error: <한글 사유> }` — 개수 불일치/old 누락·중복/new 범위밖·중복. **DB·파일 전부 무변경.**
- 500 `{ error: 'renumber failed', detail }` — DB 트랜잭션 예외(FK 참조행 존재 등). 트랜잭션 롤백으로 DB 무변경.

## 4. 데이터 파괴 방지 설계

1. **검증 선행 + 원자성**: `validateRenumberMapping`이 DB 접촉 이전에 순열을 게이트한다. 실패하면 라우트는 즉시 400을 반환하고 DB `renumberSlotIds` 호출 자체를 하지 않는다 — "검증 전 부작용 없음"이 설계·구현·QA 3단계 모두에서 일관.
2. **전 컬럼 바이트 보존**: `renumberSlotIds`는 SQL 원시 SELECT 결과를 가공 없이 그대로 재삽입한다. `round5`를 다시 적용하지 않고, `updated_at`도 재작성하지 않는다 — "라벨(slot_id)만 이동, 값은 불변"을 코드 레벨에서 보장.
3. **FK 방어**: `parking_evnt`/`parking_slot`이 `slot_setup(slot_id)`를 참조하는데(현재 writer 미작성으로 항상 비어 있음), 혹시 행이 있으면 전량 DELETE 전에 throw하여 FK 위반·고아 참조를 사전 차단.
4. **slot_ptz DB 재생성 금지**: `plateWidth`/`converged`는 DB에 없는 값이라 DB로부터 slot_ptz.json을 재생성할 수 없다. 따라서 반드시 기존 파일을 읽어 `slotId`/`globalIdx`만 리맵 후 rewrite하는 방식을 채택했다(DB 재생성 시도는 즉시 값 소실이므로 명시적으로 배제).

## 6. 한계 / 설계상 트레이드오프 (은닉 없음)

1. **setup_artifact.json ROI 표현 = bbox 고정.** 재번호 후 `setup_artifact.json`은 `buildArtifactFromSlotSetup`으로 DB 기준 재빌드되는데, 이 함수의 `roiByPreset`은 축정렬 bbox(`NormalizedRect`)로 산출된다. `setup_artifact.json`에만 있던 수동 ROI 폴리곤 편집(있었다면)은 이 재빌드로 사라진다. 다만 이는 이번 변경이 만든 신규 손실이 아니라 — 타입 자체가 사각형이고 GET `/mapping`의 기존 DB 폴백도 동일 표현이었다(기존 `docs/20260723_174313_분석페이지_DB즉석생성.md` 참조). 폴리곤 정본은 DB `slot_roi` 와 `setup_result.json`의 `floor_roi`에 보존되며, 재번호가 이 둘의 바이트를 보존함은 QA가 실증했다. 이 트레이드오프는 설계서(§5 결정 C)에서 명시했고 QA가 재확인했으며 마스터 A안 채택 시 기승인된 것으로 기록한다.
2. **FK 가드는 "비어 있음" 가정에 의존.** `parking_evnt`/`parking_slot`은 현재 writer가 미작성이라 항상 비어 있다는 전제(00_leader_context 가정 A) 위에서, 재번호는 이 두 테이블에 행이 있으면 무조건 차단(throw)한다. 향후 이 테이블에 writer가 도입되면 재번호는 cascade 갱신 전략을 다시 설계해야 한다 — 현재 범위 밖.
3. **스모크(라이브 REST) 미수행.** 이 기능은 외부 서비스(VPD/LPD/Unity RPC) 연동이 없는 순수 DB 트랜잭션 + 파일 IO라, QA는 fastify `inject` 라우트 통합 테스트로 전 경로를 실증했고 별도 스모크는 수행하지 않았다. 실제 브라우저 조작을 통한 육안 확인은 이번 문서 작성 시점에 **수행되지 않았다**(확인 필요 항목으로 남긴다).

# 3D육면체 ROI생성 버튼 — 지면모델 기반 앞면중심 산출·저장·표시

정밀수집 화면 상단 액션바 "ROI 파일 로딩" 바로 옆에 **"3D육면체 ROI생성"** 버튼을 추가했다.
클릭 시 지면모델(GroundModel)로 각 슬롯의 3D 육면체를 산출해 **앞면 중심점**(`slot_setup.slot3d_front_center`)을
DB에 **부분 UPDATE**로 저장하고, 화면에 육면체+앞면 중앙점 오버레이를 자동으로 켠다.

근거 문서: `SettingAgent/_workspace/05_architect_plan_cuboid.md`(설계) ·
`06_developer_changes_cuboid.md`(구현) · `07_qa_report_cuboid.md`(검증).

## 1. 배경 — 왜 이 작업이 필요했나

- 육면체 렌더(뷰어 `drawCuboidOverlay`)와 지면모델 산출 로직(`estimateGroundModels`), 앞면중심 산출식
  (`Finalizer` 내부의 `slotFrontCenter`)은 이미 각각 존재했지만, **"버튼 눌러 DB에 앞면중심을 저장"하는
  경로가 없었다**. 앞면중심은 지금까지 `Finalizer`(최종화) 시점에만 채워졌다.
- 조사 과정에서 더 근본적인 문제가 드러났다 — `buildGroundInputs`가 PTZ 값을 **camerapos.json 뷰**에서만
  가져오고 있었는데, 실측 결과 이 값이 **뒤처진 사본**이었다(cam1 preset3 zoom: ROI 파일 자체 값 = 1,
  camerapos.json 값 = 1.46583. ROI 값 쪽이 `fov 34.635`와 자기정합해 신뢰도가 높음). 이 때문에 이번 작업
  범위에서 **PTZ 소스 우선순위를 ROI 파일 자체 프리셋 PTZ > camerapos 뷰**로 바꿨다(필드 단위 폴백 — ROI에
  없는 필드만 camerapos로 보충).

## 2. 변경/신규 클래스·함수

### 2-1. `src/ground/slotFrontCenter.ts` (신규 파일)

`Finalizer.ts`에 module-private로 있던 `slotFrontCenter()`와 상수 `H_CONST`를 **이동(승격)**해 export했다.
본문·주석은 문자 단위로 그대로 옮겼으며(복사가 아니라 이동), import 경로만 신규 파일 위치에 맞게
`./project.js`, `../domain/types.js` 등으로 조정했다. `Finalizer.ts`는 이 모듈을 import하는 쪽으로 전환됐고,
이동으로 고아가 된 import(`backprojectToGround`/`projectCuboidPixels`/`frontFaceCenterPx`, `Vec3`,
`NormalizedPoint`)는 정리됐다. finalize 로직·출력 자체는 한 줄도 바뀌지 않았다.

```ts
export const H_CONST = 1.5;  // slot_setup 앞면중심 저장용 캐노니컬 높이(m). 뷰어 슬라이더 기본값과 동일.

export function slotFrontCenter(
  points: NormalizedPoint[], g: GroundModel, h: number
): { x: number; y: number } | null;
```

동작: 정규화 슬롯 quad(4점) → 픽셀 좌표 → `backprojectToGround`로 지면 3D 역투영 → `projectCuboidPixels(h)`로
높이 h의 육면체 픽셀 코너 산출 → `frontFaceCenterPx`로 앞면 중심 픽셀 → `/imgW, /imgH`로 0~1 정규화.
4점이 아니거나 지면모델이 퇴화(지평선 위 등)면 예외 없이 `null`을 반환한다(강등 철학 — throw 금지).

### 2-2. `src/ground/groundInputs.ts` — `buildGroundInputs()` PTZ 우선순위 변경

각 프리셋의 pan/tilt/zoom을 고를 때, 기존에는 camerapos 뷰(`views: CameraView[]`)만 사용했다. 이번 변경으로
ROI 파일 자체의 `preset.ptz.{pan,tilt,zoom}`(또는 평면형 `preset.{pan,tilt,zoom}`)을 **1순위**로,
camerapos 뷰를 **필드 단위 폴백**으로 사용하도록 `pick(a, b)` 헬퍼를 도입했다.

```ts
const pick = (a: unknown, b: unknown): number | null =>
  typeof a === 'number' && Number.isFinite(a) ? a
  : typeof b === 'number' && Number.isFinite(b) ? b
  : null;
```

비수치 값(`'x'`, `null`, `NaN`)은 무시하고 camerapos로 폴백하며, 둘 다 없으면 `null`(undefined 아님)이다.
`buildGroundInputs`의 시그니처·다른 파싱 로직(quad 추출 등)은 변경하지 않았다.

### 2-3. `src/capture/SqliteStore.ts` — `upsertSlotFrontCenter()` (신규 메서드)

```ts
/** slot3d_front_center 만 slot_id 키로 부분 UPDATE(updated_at 동반). 전량 delete 없음. */
upsertSlotFrontCenter(rows: Array<{ slotId: number; slot3dFrontCenter: string | null; updatedAt: string }>): number
```

`UPDATE slot_setup SET slot3d_front_center = ?, updated_at = ? WHERE slot_id = ?`를 단일 트랜잭션으로 실행한다.
기존 `upsertSlotLpd`(283행대)와 같은 형태의 **키 단위 부분 UPDATE**이며, `replaceSlotSetup` 같은 전량
DELETE+INSERT를 쓰지 않는다(메모리: finalize-slotsetup-wipe-fragility 재발 방지 — 검출없는 저장이 데이터를
파괴하는 사고 패턴을 피함). 존재하지 않는 `slot_id`는 조용히 무시하고, 반환값은 `stmt.run().changes`의
누적 합(실제 갱신 행수)이다.

### 2-4. `src/api/captureRoutes.ts` — `POST /capture/slots/cuboid` (신규 라우트)

`/capture/slots/occupy` 라우트 바로 뒤, `saveStore` 블록 앞에 추가됐다. 상세 계약은 §3 참조.
지면모델 조합 코드(`buildGroundInputs` + `estimateGroundModels`)는 기존 `GET /capture/ground-model`
라우트와 **동일**하게 라우트 내부에서 직접 조합한다 — `Finalizer.buildGroundModelMap`이 `private`라 재사용할
수 없었기 때문(설계에서 지시한 "새 조합·새 수학 금지" 원칙은 준수 — 조합에 쓰이는 두 함수 자체는 기존
모듈 소유 그대로).

### 2-5. `web/index.html` / `web/app.js` — UI

- `index.html`: `#cap-load-roi`(ROI 파일 로딩) 바로 뒤에 `<button id="cap-build-cuboid">3D육면체 ROI생성</button>` 추가.
- `app.js`: `buildSlotCuboids()` 신설(`loadRoiToDb()` 바로 아래) + `$('cap-build-cuboid')` click 등록.
- **부수 수정**: `loadRoiToDb()`에 있던 것과 같은 종류의 "1회 로드 가드" 버그를 `loadGroundModel()`에도
  발견해 함께 고쳤다. `state.groundLoaded`는 최초 1회만 지면모델을 로드하는 가드인데, ROI 정본이 바뀌어도
  재로딩되지 않는 문제가 있었다. `loadRoiToDb()`와 신규 `buildSlotCuboids()` 양쪽 모두 성공 후
  `state.groundLoaded = false` → `await loadGroundModel()` 순서로 가드를 해제하고 재로딩하도록 했다.

## 3. REST 계약 — `POST /capture/slots/cuboid`

**요청**
```
POST /capture/slots/cuboid
Content-Type: application/json
Body: { "heightM"?: number }   // 0.5~3.0. 생략 시 H_CONST(1.5). 빈 바디 {} 허용.
```

**200 성공**
```ts
{
  ok: true,
  updated: number,                                   // 실제 DB 갱신 행수
  skipped: Array<{ slotId: number; reason: string }>, // 모델 없음 / 육면체 퇴화 등
  models: Array<{ key: string; conf: number; issues: string[] }>, // "camId:presetId" 별 지면모델 요약
  issues: string[],                                  // estimateGroundModels 전역 이슈
  heightM: number,                                   // 실제 적용된 높이(에코)
}
```
- `updated + skipped.length === 전체 슬롯 수`.
- 모델이 없는 프리셋(`skipped: '지면모델 없음(cam:preset)'`)이나 육면체가 퇴화하는 슬롯
  (`skipped: '육면체 퇴화(지평선 위/quad 이상)'`)은 **저장하지 않는다** — 기존 `slot3d_front_center` 값을
  `null`로 덮어쓰지 않고 그대로 둔다(비파괴 원칙).

**오류**
| 코드 | 조건 | 응답 |
|---|---|---|
| 400 | `heightM`이 0.5~3.0 밖 또는 비수치 | `{ ok:false, error:'invalid body (heightM 0.5~3.0)' }` |
| 404 | `deps.placeRoiFile` 미설정 또는 `deps.ground.enabled !== true` | `{ ok:false, error:'ground/place-roi 미설정' }` |
| 404 | `PtzCamRoi.json` 파일 자체가 없음(ENOENT) | `{ ok:false, error:'PtzCamRoi.json 없음', detail }` |
| 409 | `slot_setup`이 0건(ROI 미적재) | `{ ok:false, error:'slot_setup 비어있음 — ROI 파일 로딩 먼저' }` |
| 500 | 산출 중 그 외 예외 | `{ ok:false, error:'ground-model 산출 실패', detail }` |

`camerapos.json`이 없거나 파싱에 실패해도 예외를 던지지 않고 **200으로 강등**한다(zoom 미상 프리셋은
`skipped`로 드러남 — `GET /capture/ground-model`과 동일 처리 방식).

## 4. UI 동작

1. 사용자가 "3D육면체 ROI생성" 클릭 → `POST /capture/slots/cuboid`에 `{ heightM: cuboidHeight() }`
   (현재 화면 `#cuboid-h` 슬라이더 값)를 전송.
2. 실패 시 `#cap-msg`에 오류 메시지 표시 후 조기 종료(오버레이 상태 무접촉).
3. 성공 시: `state.groundLoaded = false` → `loadGroundModel()` 재호출(지면모델 재산출) →
   `#roi-cuboid` 체크박스를 강제로 켜서(`checked = true`) 육면체 오버레이 자동 표시 →
   `state.roiHidden = false` → `loadParkingSlots()`(DB 갱신분 반영) → `drawRoiOverlay()` / `renderSlotList()`.
4. `#cap-msg`에 `산출 N건 / 스킵 M건(h=X m)` + 스킵 슬롯별 사유 + 모델별 conf + 이슈를 전부 노출한다
   (숨기지 않음).
5. 이 버튼은 **비파괴**(검출 VPD/LPD, 점유영역, 센터라이징 PTZ 무접촉 — `slot3d_front_center` 컬럼만
   부분 UPDATE)이므로, "ROI 파일 로딩"과 달리 `confirm()` 확인창이 없다.

## 5. 지면모델 품질 개선 (리더 실측)

PTZ 소스 우선순위 변경(§2-2) 전후로 지면모델 신뢰도(conf)와 오차가 실측 기준으로 달라졌다.

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| conf (cam1:preset1) | 0.106 | 0.791 |
| conf (cam1:preset2) | (미기록) | 1.000 |
| conf (cam1:preset3) | (미기록) | 1.000 |
| conf (cam2:preset1) | (미기록) | 1.000 |
| conf (cam2:preset2) | (미기록) | 1.000 |
| tilt 오차 | 추정 10.92° vs PTZ 6.80° = 4.12° 불일치 | 전부 ≈0.00° |
| issues | metric 잔차 8.7%, f 후보 불일치 20.7%, cam2 공동추정 표본 없음 등 | **0건** |
| 앞면중심 산출 | 21/23건(cam1:preset3 2면 모델 없음으로 스킵) | **23/23건 전량 산출**, 스킵 0, 전부 0~1 화면 안 |

즉 PTZ 소스를 camerapos.json(뒤처진 사본)에서 ROI 파일 자체 값으로 바꾼 것이 지면모델 신뢰도를
근본적으로 끌어올렸다. 이 개선은 `slot3d_front_center` 산출만이 아니라 `buildGroundInputs`를 쓰는
**모든 경로**에 동일하게 적용된다(§6 영향도 참조).

## 6. 육안·수치 검증

- **육안**: `sharp`로 5개 프리셋을 렌더링해 확인한 결과, 뷰어 측 `web/core.js frontFaceCenter`가 그리는
  점과 서버 측 `slotFrontCenter` 산출점이 **동심 일치**했다(표시==저장 정합). 단, 이는 좌표 렌더 비교이며
  실제 브라우저에서 버튼을 클릭해 화면 변화를 관찰하는 라이브 스모크는 수행되지 않았다(§8 한계 참조).
- **수치**: qa가 별도로 작성한 "표시==저장 종단 파리티" 테스트에서, 라우트로 h=1.5 저장 후 뷰어와 같은
  소스(`GET /capture/ground-model` 모델 + `projectCuboid → frontFaceCenter`)로 재계산한 좌표가 DB
  저장값과 1e-5 이내로 일치함을 확인했다(픽스처 전 슬롯 대상).

## 9. 알려진 특성·주의 (결함 아님 — qa 지적)

1. **화면 표시는 DB 저장값을 읽어서 그리는 것이 아니다.** `drawCuboidOverlay`(`web/app.js:613`)는
   `state.placeRoi` 폴리곤 + 클라이언트 `projectCuboid` + **라이브 슬라이더 높이**로 앞면 중심을 매번
   재계산한다. `slot3d_front_center`를 읽는 코드는 없다. 그 결과:
   - 버튼을 누른 뒤 `#cuboid-h` 슬라이더를 움직이면 화면만 바뀌고 DB 저장값은 누른 시점 높이 그대로다.
   - 오버레이는 ROI 파일(`state.placeRoi`) 기준, 저장은 DB `slot_setup.slot_roi` 기준이라 두 소스가
     갈라지면(수동 DB 편집 등) 화면과 저장이 다른 도형을 가리킬 수 있다.
2. `updated`는 DB 실제 변경 행수이므로, 산출은 됐는데 그 사이 행이 사라진 극단적 경합 상황에서는
   `updated + skipped.length < 전체 슬롯`이 될 수 있다(정상 경로에서는 발생하지 않음).
3. `heightM` 스키마 범위(0.5~3.0)와 슬라이더 `#cuboid-h`의 범위(min 0.5 / max 3.0 / step 0.05)가 정확히
   일치해, 정상적인 UI 조작만으로는 400이 발생할 수 없다(정합 확인됨 — 문제 없음).
4. 라우트는 `/capture/ground-model`과 동일 조합을 쓰지만 `fovBaseV`는 응답에 싣지 않는다. `app.js`가
   쓰지 않으므로 무해하다.

## 10. 권장 운영 순서

`ROI 파일 로딩` → `3D육면체 ROI생성` → `시작`(정밀수집) → `최종화`.

ROI 파일 로딩은 파괴적(검출/점유/센터링 전량 재구성)이므로 먼저 실행해 `slot_setup`을 채운 뒤,
3D육면체 ROI생성으로 앞면중심을 채우는 순서가 안전하다. `slot_setup`이 비어 있으면 3D육면체 생성은
409로 거절된다.

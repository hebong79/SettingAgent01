# LPD 검지 패널 "DB에 추가" 버튼 — 라이브 LPD 박스 공간배정 저장

- 작성: 2026-07-19 / documenter
- 범위: SettingAgent 웹뷰어 LPD 검지 패널에 **"DB에 추가"(`#lpd-db-add`) 버튼**을 추가해, 현재 프리셋의 **라이브 LPD 검출 박스**를 각 슬롯에 공간배정 후 `slot_setup.lpd_obb` 에 저장한다.

---

## 1. 배경

LPD 검지 패널(`#lpd-mode` 콤보 + `#lpd-run`)은 **순수 LPD / 현재화면 LPD / VPD→LPD** 세 모드로 검출을 실행하지만, 결과는 프론트 메모리(`state.detectByKey[presetKey(cam,preset)].plates`)에만 남고 **DB에는 저장되지 않았다**. 유일한 자동저장 경로는 discovery(앞면중심 LOOP, `plateDiscoveryWriter`)뿐이었다. 이번 변경은 discovery 이외 세 라이브 모드가 채운 검출 박스를 **수동으로 DB에 반영**할 수단을 제공한다.

## 2. 변경 파일 (4개, 신규 파일 0)

| 파일 | 변경 | 요지 |
|---|---|---|
| `src/setup/plateMatch.ts` | 가산 | 신규 export `assignPlatesToSlotViews(slots, plates)` + 모듈 상수 `MATCH_RADIUS=0.15`. 기존 `matchPlatesToSlots` 본체·export **불변**. |

## 3. 동작 흐름

1. 사용자가 `#lpd-mode`(순수 LPD / 현재화면 LPD / VPD→LPD)로 검출 실행 → 결과가 `state.detectByKey[presetKey(cam,preset)].plates` 에 축적.
2. `#lpd-db-add` 클릭 → `saveLpdToDb()`:
   - `cam/preset` 판별(`state.capFrameKey2?.cam ?? state.cam`, `runLiveDetect` 와 동일 규칙).
   - `plates` 가 비어 있으면 `#disc-msg` 에 안내 후 중단(`현재 프리셋에 저장할 LPD 검출이 없습니다`).
   - `POST /capture/slots/lpd { cam, preset, plates }` 호출.
   - 성공 시 `#disc-msg` = `DB 추가: {updated} 슬롯 (미배정 {unassigned})`, 이어서 `loadParkingSlots() → drawRoiOverlay() → renderSlotList()` 로 `#roi-db` 오버레이를 DB 정본과 재정합(오버레이 강제 삭제 없음 — retain 정책 준수).
3. 서버 `POST /capture/slots/lpd`:
   - body 검증 후 `store.getSlotSetup()` 에서 `camId===cam && presetId===preset && roi.length>=3` 필터.
   - `plates → PlateBox[]`(`confidence ?? 0`) 변환.
   - `assignPlatesToSlotViews(views, plateBoxes)` 로 슬롯별 quad 배정.
   - `rows = [...map].map(([slotId, quad]) => ({ slotId, lpdObb: stringify5(quad), updatedAt: now }))`.
   - `store.upsertSlotLpd(rows)`(부분 UPDATE, wipe 안전 — `slot_setup` 의 roi/vpd/다른 슬롯 불변).
   - 반환: `{ ok, updated, assigned: [{slotId, confidence}], unassigned }`.

## 4. 배정 로직 — 설계 반복(v1 → v2) 서사

### v1(초안): 기존 `matchPlatesToSlots` 재사용 시도
초안은 VPD 차량-슬롯 매칭에 쓰이는 `matchPlatesToSlots`(번호판 중심이 슬롯 ROI bbox 안이면 후보 → 겹침 내림차순 그리디)를 그대로 재사용하려 했다.

### 라이브 검증에서 드러난 실패 (05_live_finding.md)
cam1:preset2 에서 순수 LPD 검출 4개(중심 x = 0.325 / 0.463 / 0.635 / 0.783)를 DB에 추가한 결과:

| plate x | v1 배정 | 정답(nearest front_center) |
|---|---|---|
| 0.325 | slot9 ❌ | slot10 (front 0.323) |
| 0.463 | slot10 ❌ | slot11 (front 0.457) |
| 0.635 | slot12 ✓ | slot12 |
| 0.783 | slot13 ✓ | slot13 |

**근본 원인**: 원근 왜곡된 주차면 폴리곤의 bounding rect 가 인접 슬롯과 최대 ~60% 겹친다(예: slot9 x[0.145~0.443], slot10 x[0.261~0.561]). plate 중심이 3개 bbox 에 동시 포함되면 `matchPlatesToSlots` 의 "겹침 우선" 그리디가 **왼쪽 슬롯을 선택**해 한 칸 밀림 오배정이 난다. 이 함수는 애초 **타이트한 차량 ROI(rect)** 용으로 설계된 것으로, 넓은 원근 폴리곤에는 판별력이 없었다.

### v2(수정): nearest `lowerFrontAnchor` 전역 1:1 그리디로 교체
- **앵커**: `lowerFrontAnchor(view.roi, view.slot3dFrontCenter)` — discovery(앞면중심 LOOP, `src/calibrate/plateDiscoveryWriter.ts`)가 이미 쓰는 **동일 함수**를 그대로 import 재사용.
- **왜 raw `slot3dFrontCenter`(차량 전면중심)가 아니라 하향앵커인가**: raw front_center 는 실제 판 위치보다 y 방향으로 ~0.05–0.08 위에 있어, 오답 이웃 슬롯까지의 거리가 `hypot(0.13, 0.07)≈0.147` 로 게이트(0.15) 마진을 거의 잠식한다. `lowerFrontAnchor` 는 앵커를 판 높이만큼 하향시켜 정답 거리(~0.01–0.03)와 오답 이웃 거리(~0.13)를 깨끗이 분리한다 — discovery 가 애초 raw front_center 대신 이 앵커를 쓰는 이유와 동일하다.
- **알고리즘**: 모든 (plate, slot) 쌍의 `hypot(plate중심 − 앵커)` 를 계산 → `MATCH_RADIUS=0.15`(discovery `matchRadiusNorm` 과 동일값) 초과 쌍은 후보에서 제외 → 거리 오름차순 정렬(tie-break: plate 인덱스 → slot 인덱스) → 양쪽(plate·slot) 모두 미배정일 때만 확정(전역 1:1). `slot3dFrontCenter == null` 슬롯은 배정 대상에서 완전히 제외(폴백 없음 — 위장 배정 방지).
- 반환 quad 는 입력 `plate.quad` **참조 그대로** 보존 — 라우트가 참조 동등성으로 원 `confidence` 를 역조회하는 기존 계약(`matchPlatesToSlots:61` 과 동일 규약)을 그대로 따른다.
- `matchPlatesToSlots` 본체는 **손대지 않았다** — VPD 차량-슬롯 매칭(finalize/detectPipeline 소비)은 별도 함수로 계속 서비스된다.

### 라이브 재검증(v2 적용 후)
동일 시나리오(preset2, lpd 사전 null 정리 후) 재실측:

| plate x | v2 배정 | v1(오답) | 정합 |
|---|---|---|---|
| 0.326 | slot10 | slot9 | ✓ (앵커 0.324) |
| 0.463 | slot11 | slot10 | ✓ (앵커 0.457) |
| 0.620 | slot12 | slot12 | ✓ |
| 0.765 | slot13 | slot13 | ✓ |

4개 전부 discovery 앵커 기준과 일치, 한 칸 밀림 제거, `unassigned=0`. slot8/9 는 해당 판이 검출되지 않아(전체 프레임 LPD 희소) 정직하게 null로 남았다(위장 배정 없음). 검출 → `POST /capture/slots/lpd` → `slot_setup.lpd_obb` 반영까지 종단 확인.

## 5. REST 계약

### `POST /capture/slots/lpd` (신규)

**요청**
```json
{
  "cam": 1,
  "preset": 2,
  "plates": [
    { "quad": [{"x":0,"y":0},{"x":0,"y":0},{"x":0,"y":0},{"x":0,"y":0}], "confidence": 0.9 }
  ]
}
```
- `cam`/`preset`: 양의 정수(필수).
- `plates`: 배열(빈 배열 허용 → 0건 저장). `quad` 는 4점 정규화 좌표(필수), `confidence` 옵셔널(응답용, 배정 로직에는 미사용).
- 검증 실패(quad 누락/길이≠4, cam 누락/비양수 등) → `400 { ok:false, error:'invalid body' }`.

**응답**
```json
{ "ok": true, "updated": 2, "assigned": [{"slotId":10,"confidence":0.9}], "unassigned": 1 }
```
- `updated`: 저장된 slot 수(`assigned.length`와 동일).
- `unassigned`: `plates.length - updated`(게이트 초과 또는 경합 탈락으로 미배정된 plate 수, 정직 카운트).
- `assigned[].confidence`: quad 참조 역조회 성공 시 부착, 실패 시 `slotId` 만.

기존 `/capture/detect`·`/capture/slots`(GET)·`/capture/slots/reset` 은 **무변경**.

## 7. 한계 (은닉 없이 명기)

**현재화면 순수 LPD(수동 줌 PTZ) 모드**는 검출 프레임이 프리셋 base 좌표계와 일치하지 않을 수 있어, 이 버튼으로 저장 시 배정이 부정확할 수 있다. `POST /capture/slots/lpd` 는 **프리셋 정합 검출(순수 LPD/VPD→LPD)에서만 유효**하다. 수동 PTZ 프레임의 base 프레임 역투영은 이번 범위 밖이며 2차 과제로 남는다.

또한 `slot3d_front_center` 가 없는 프리셋(ground 미설정/강등 finalize)에서는 배정 대상이 0이 되어 `updated:0`·전량 미배정이 된다 — "지면모델(front_center) 선행 필요"가 이 버튼의 전제조건이다.

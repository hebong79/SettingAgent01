# ROI 파일 로딩 버튼 — `PtzCamRoi.json` → DB `slot_setup` 전량 재구성

작성일: 2026-07-22 15:24:21
대상: SettingAgent 정밀수집(capture) 페이지

## 1. 배경 및 설계 근거

정밀수집 페이지에는 이미 `POST /capture/slots/reset`(버튼 `#cap-reset-db`, "검출·센터링 초기화")이 있어
검출(VPD/LPD)·점유영역·센터라이징(PTZ)만 비우고 `slot_roi`(바닥 ROI)는 보존한다.

이번 기능은 그와 별개로, **바닥 ROI 정본 파일 자체를 다시 읽어 `slot_setup`을 통째로 재구성**하는
"ROI 파일 로딩" 버튼이다. 시뮬레이터가 `SettingAgent/data/Place01/PtzCamRoi.json`을 재생성했을 때
(주차면 배치·개수 변경 등) DB를 그 파일 기준으로 다시 맞추는 용도다.

핵심 설계 원칙(01_architect_plan.md Phase 0 조사 사실 기반):

- **이중구현 금지**: 동일 매핑 로직(`buildCameras`/`buildPresets`/`buildSlots`)이 이미
  `src/tools/migrateToSettingDb.ts`(CLI)에 존재했다. 이를 공용 모듈로 승격해 CLI와 신규 라우트가
  **동일 함수**를 호출하도록 했다(구현 시 CLI에서 `roiDbLoad.ts`로 *이동* — 복사 아님).
- **wipe 금지 가드**: `replaceSlotSetup`(`SqliteStore.ts`)은 `DELETE FROM slot_setup` 후 전량 INSERT를
  단일 트랜잭션으로 수행한다. 예외 시에는 자동 롤백되지만, "행 0건"인 정상 입력(빈 파일·파싱만 성공)에는
  롤백이 걸리지 않아 테이블이 통째로 비워지는 취약성이 기존에 알려져 있었다(memory:
  finalize-slotsetup-wipe-fragility). 이번 구현은 이 취약성에 대응해 **호출부(`loadRoiIntoDb`)에서
  사전 가드**를 두고, 실패 조건에서는 `replaceSlotSetup` 자체를 호출하지 않는다.

## 2. 신규/수정 클래스·함수

그대로 이동했다.

### 2.3 `src/tools/migrateToSettingDb.ts` (수정)

`build*` 3함수를 삭제하고 `roiDbLoad.js`에서 import하도록 전환. `buildCentering`·`main`·콘솔 출력은
전부 기존 그대로 유지(마이그레이션 CLI 전용 로직이므로 이관하지 않음).

### 2.4 `src/api/captureRoutes.ts` — `POST /capture/slots/load-roi` (신규 라우트)

```ts
app.post('/capture/slots/load-roi', async (_req, reply) => {
  if (!deps.placeRoiFile) { reply.code(404); return { ok:false, error:'placeRoiFile 미설정' }; }
  const result = loadRoiIntoDb(deps.store, {
    placeRoiFile: deps.placeRoiFile,
    cameraposFile: deps.mapFiles?.cameraposFile,
    now: new Date().toISOString(),
  });
  if (!result.ok) reply.code(409);
  return result;
});
```

`deps.job` 등 인메모리 캐시는 건드리지 않는다 — 슬롯 소스는 항상 DB 조회(`GET /capture/slots`)다.

- `index.html`: `div.cap-actions.toolbar` 맨 앞에 `#cap-load-roi` 버튼 추가. `title` 속성에 파괴성 경고 문구

  (`resetSlotSetupDb`가 성공 후 호출하는 것과 동일한 경로를 재사용).

## 3. 수정 로직의 의도와 동작

`loadRoiIntoDb(store, opts)` 처리 순서:

1. `opts.placeRoiFile` 존재 확인 → 없으면 `{ok:false, error:'ROI 파일 없음: … — DB 무변경'}` 즉시 반환.
2. JSON 파싱 시도 → 실패 시 `{ok:false, error:'ROI 파일 파싱 실패: … — DB 무변경'}` 반환.
3. `normalizePtzCamRoi(ptzRaw)`(정본 유틸, 로직 무변경) 호출 → `report.issues`를 `cam{N}:preset{M} {msg}`
   형태로 평탄화해 `issues[]`에 누적. `byPreset`이 비어있으면 `{ok:false, error:'ROI 파일에 유효한 주차면 없음 — DB 무변경'}`.
4. `buildSlots()` 결과가 0건이면 동일하게 실패 반환(DB 무변경).
5. `buildCameras()`로 카메라 목록 구성. `opts.cameraposFile`이 지정·존재하면 `buildPresets()`로 프리셋
   목록도 구성하고, 그렇지 않으면 preset upsert를 생략하며 `issues`에 사유를 남긴다
   ("camerapos.json 없음 — preset upsert 생략(기존 preset_pos로 FK 판정)").
6. 여기부터 `try/catch`로 감싼 DB 쓰기 구간: `upsertPlaceInfo → upsertCameraInfo → (upsertPresetPos)` 순으로
   FK 부모를 먼저 채운다.
7. `store.getPresetKeys()`로 현재 `preset_pos`의 `(camId,presetId)` 키 집합을 얻어, `buildSlots()` 결과 중
   부모가 없는 슬롯을 **INSERT 대상에서 제외**하고 `skipped[]`에 `{camId, presetId, count, reason:'preset_pos
   부모 없음(FK)'}`로 기록한다(전량 FK 실패 방지).
8. 제외 후 남은 행이 0건이면 `{ok:false, error:'FK 부모(preset_pos) 있는 슬롯 0건 — slot_setup 무변경'}`
   반환(이 시점에서도 `replaceSlotSetup`은 아직 호출되지 않았으므로 DB는 무변경).
9. 남은 행이 있으면 `store.replaceSlotSetup(keep)` 호출 → 성공 시
   `{ok:true, slots, cameras, presets, skipped, issues}` 반환.
10. 6~9 구간에서 예외 발생 시 better-sqlite3 트랜잭션이 자동 롤백되고
    `{ok:false, error:'DB 쓰기 실패: … — slot_setup 무변경'}` 반환.

**wipe 금지 가드는 4개 실패 경로(①파일없음 ②파싱실패 ③유효슬롯0건 ④FK필터후0건) 모두에서
`replaceSlotSetup`을 호출하지 않는 것으로 보장된다** — DELETE가 실행되지 않으므로 기존 데이터가
그대로 남는다.

## 4. 입출력·REST 계약

### `POST /capture/slots/load-roi`

요청 본문 없음(빈 객체 `{}` 전송해도 무시).

| 상태 | 조건 | 응답 shape |
|---|---|---|
| 200 | 로딩 성공(슬롯 1건 이상 INSERT) | `{ ok:true, slots:number, cameras:number, presets:number, skipped:[], issues:[] }` |
| 404 | `deps.placeRoiFile` 미설정(서버 구성 문제) | `{ ok:false, error:'placeRoiFile 미설정' }` |
| 409 | 파일 없음/파싱 실패/유효 슬롯 0건/FK 필터 후 0건/DB 쓰기 예외 — **DB 무변경** | `{ ok:false, error:string, slots:0, cameras:0, presets:0, skipped:[], issues:[] }` |

`skipped[]` 항목: `{ camId:number, presetId:number, count:number, reason:string }`.
`issues[]`: `normalizePtzCamRoi` 보고 + camerapos 관련 경고 문자열 배열.

## 7. 안전 규약 요약 (별도 강조)

**"빈 결과·실패 시 DB 무변경"이 이 기능의 핵심 안전 계약이다.** 아래 4가지 조건 중 하나라도 해당하면
`replaceSlotSetup`(DELETE+INSERT)이 **호출되지 않고** 기존 `slot_setup` 테이블은 손상 없이 그대로 남는다:

1. `placeRoiFile` 경로에 파일이 없음
2. 파일은 있으나 JSON 파싱 실패
3. 파싱은 되었으나 `normalizePtzCamRoi().byPreset`이 비어있거나 `buildSlots()` 결과가 0건
4. FK 부모(`preset_pos`)가 없는 `(cam,preset)` 슬롯을 제외한 뒤 남은 행이 0건

이 가드는 `SqliteStore.replaceSlotSetup`이 "행 0건 정상 입력에는 롤백이 걸리지 않아 테이블이 통째로
비워지는" 기존 알려진 취약성(memory: finalize-slotsetup-wipe-fragility)에 대한 **호출부 방어**다.
`replaceSlotSetup` 자체의 시그니처·동작은 변경하지 않았다 — 가드는 오직 `loadRoiIntoDb`에만 있다.

단, 이 가드는 "구조적으로 텅 빈 입력"만 막는다. **정상적으로 파싱·매핑된 데이터로 버튼을 실행하면
기존 VPD/LPD 검출·점유영역·센터라이징(PTZ)은 의도된 대로 전량 소실된다** — 이는 결함이 아니라
설계된 동작이며, UI `confirm()`과 버튼 `title` 양쪽에서 사용자에게 명시적으로 경고한다.

## 8. 후속 보완 (같은 날, 마스터 지시 — "DB 적재 후 웹페이지 갱신까지")

### 8.1 웹 갱신 누락 수정 (실버그)

`loadPlaceRoi()`는 `state.placeRoiLoaded` **1회 로드 가드**를 갖는다(재시도 폭주 방지, 세션당 1회).
그래서 최초 버전의 `loadRoiToDb()`는 DB(`/capture/slots`)만 다시 읽고 **파일 소스 `state.placeRoi`는
갱신하지 않았다**. `state.placeRoi`는 주차면 목록(`renderSlotList` → `buildFlatSlotRows`)·바닥 ROI
오버레이·검출 필터(`selectFloorRoi`)의 기준이므로, ROI 파일 정본이 바뀐 직후에도 **화면은 옛 주차면을
계속 표시**했다. 페이지를 새로고침해야만 반영되는 상태였다.

수정(`web/app.js:loadRoiToDb`) — DB 재조회 앞에 파일 소스 강제 재로딩을 추가:

```js
state.roiHidden = false;      // 로딩 결과를 즉시 표시(이전 수집·초기화로 숨김 상태였을 수 있음)
state.placeRoiLoaded = false; // 1회 로드 가드 해제 — 파일 정본이 바뀌었으므로 반드시 재로딩
state.placeRoiDirty = false;  // 이전 세션의 미저장 편집 버퍼 폐기(파일이 새 정본)
await loadPlaceRoi();         // 파일 소스 재조회
await loadParkingSlots();     // DB 재조회
drawRoiOverlay(); renderSlotList();
```

`state.placeRoiDirty = false`는 의도적이다 — 파일이 새 정본이 된 시점에서 이전의 미저장 편집 버퍼를
유지하면 화면과 파일이 어긋난다. 다만 **미저장 편집이 있었다면 그 편집은 폐기된다**(confirm 경고 대상).

### 8.2 프리셋 라이브 선갱신 (cam2 FK 스킵 완화)

`slot_setup`은 `preset_pos(cam_id,preset_id)`를 FK 부모로 요구한다. `camerapos.json`이 옛 프리셋만
담고 있으면 신규 카메라의 주차면이 통째로 `skipped` 된다(실측: cam2 10면).
`/capture/start`가 이미 쓰는 패턴(`presetProvider.listViews()` → `writeCamerapos`)을 라우트 앞단에 동일하게
적용해, 공급자가 있으면 **최신 프리셋을 먼저 받아 camerapos.json을 갱신**한 뒤 적재한다.
공급자 호출 실패는 **강등**(기존 camerapos.json으로 계속 진행)하고 사유를 `issues[]`에 남긴다.

**⚠️ 현재 설정에서는 이 경로가 동작하지 않는다.** `config/tools.config.json`의
`presetProvider.type`이 `"camerapos"`이고, `createPresetProvider`는 이 타입에 **null을 반환**한다
(`src/setup/presetProvider.ts:93`). 따라서 실제 운영에서 cam2는 여전히 `skipped` 된다.
활성화하려면 `presetProvider.type`을 `"unity-api"`(+`unityUrl`) 또는 `"discovery"`로 바꾸거나,
`config/camerapos.json`에 cam2 프리셋 PTZ를 직접 추가해야 한다.
(저장소·가동 중인 서비스 어디에도 cam2의 프리셋 PTZ 실측값이 없어 임의 생성하지 않았다.)

### 8.3 "시작 버튼은 DB로 동작한다"는 전제에 대한 정정

마스터의 전제와 달리, **`시작`(`POST /capture/start`)은 `slot_setup`을 읽지 않는다**:

| 구성요소 | 실제 소스 |
|---|---|
| 순회 대상(`targets`) | `presetProvider` 라이브 → 없으면 `camerapos.json` (`captureRoutes.ts:174-198`) |
| 주차면 폴리곤(검출 필터) | **`PtzCamRoi.json` 파일** (`CaptureJob.ts:243` `loadNormalizedPlaceRoi`) |
| `slot_setup` | 수집 경로에서 **미사용**. `finalize` 단계에서 **기록 대상**일 뿐 |

즉 DB `slot_setup`은 수집의 **입력이 아니라 산출물**이다. 따라서 "ROI 로딩 → 시작"의 순차 동작은
성립하지만, 그 이유는 DB 적재가 아니라 **두 경로가 같은 `PtzCamRoi.json` 정본을 공유**하기 때문이다.
DB를 수집의 입력으로 삼으려면 별도 설계 변경이 필요하다(본 작업 범위 밖).

## 9. 전 주차면 적재 + camerapos.json 의존 제거 (마스터 지적: "23면이어야 하는데 13면")

### 9.1 원인

`slot_setup` 의 FK 부모 `preset_pos` 를 **camerapos.json 만으로** 채웠는데, 그 파일은 cam1 프리셋 1~3만
담고 있었다. 재생성된 ROI 정본은 cam1 3프리셋 13면 + **cam2 2프리셋 10면 = 23면**이므로 cam2 10면이
FK 부모 부재로 `skipped` 되어 13면만 적재됐다.

### 9.2 핵심 발견 — ROI 파일이 이미 프리셋 PTZ 를 담고 있다

재생성된 `data/Place01/PtzCamRoi.json` 의 각 프리셋에는 `pan`/`tilt`/`zoom` 이 **이미 들어있다**:

```json
{ "preset_idx": 1, "pan": 19.8, "tilt": 8.7, "zoom": 1.6934098,
  "eulerAngles": [8.7, 19.8, 3.2e-7], "fov": 20.8654556, "parking_spaces": [ ... ] }
```

즉 프리셋 PTZ의 정본은 이미 ROI 파일이며, `camerapos.json` 은 **뒤처진 사본**이었다
(cam1 preset1 이 camerapos 는 pan 22/tilt 6.8, ROI 는 pan 19.8/tilt 8.7 — 값도 불일치).

### 9.3 변경

**(a) `buildPresetsFromRoi(ptzRaw, now)` 신설** — ROI 파일의 프리셋 PTZ → `preset_pos`.
두 형태를 모두 허용한다(하위호환):

```
{ "preset_idx":1, "ptz": { "pan":…, "tilt":…, "zoom":… }, "sname":"…" }   // 중첩형
{ "preset_idx":1, "pan":…, "tilt":…, "zoom":…  }                          // 평면형(현행 시뮬레이터)
```
pan/tilt/zoom 3개가 모두 유한수일 때만 채택한다(부분 입력은 무시).

**(b) `loadRoiIntoDb` 의 `preset_pos` 확보 우선순위** (뒤가 앞을 덮어씀):
1. `camerapos.json` (있으면)
2. **ROI 파일의 프리셋 PTZ** ← 정본. issues 에 `ROI 파일의 프리셋 PTZ N건 채택` 기록
3. 그래도 없는 `(cam,preset)` → **자리표시자**(`pan=0,tilt=0,zoom=1`, `sname='C{cam}-P{preset} (PTZ 미상)'`)
   생성 후 issues 에 명시. **camerapos/ROI 실측이 있으면 절대 덮어쓰지 않는다**(없는 키에만 생성).

자리표시자의 안전 근거: **`preset_pos.pan/tilt/zoom` 을 읽는 코드는 없다.** 카메라 이동은
`camerapos.json`/`presetProvider`/ROI targets 가 담당하며 `preset_pos` 는 쓰기 전용 + FK 키 확인용이다
(`grep preset_pos src/` 로 확인). 따라서 자리표시자가 카메라를 잘못 움직일 수 없다.

**(c) 계약 변경 — FK 스킵 폐지.** 이제 ROI 파일이 정의한 주차면은 **하나도 버려지지 않는다.**
`skipped[]` 는 방어적 잔존 경로로만 남는다.

**(d) `presets` 필드 의미 변경** — camerapos 행 수가 아니라 **ROI 가 정의한 `(cam,preset)` 그룹 수**
(= 적재된 프리셋 수). UI 의 "프리셋 N" 표시가 파일 실제와 일치한다.

**(e) `loadSetupTargetsFromRoi(placeRoiFile)` 신설 + `/capture/start` 폴백 체인 삽입.**
순회 대상 결정 순서를 `presetProvider` → **ROI 파일(프리셋 PTZ 보유 시)** → `camerapos.json` 으로 바꿨다.
"ROI 로딩 → 시작" 이 **같은 파일 하나**를 공유하므로 프리셋 집합이 어긋나지 않는다.
PTZ 미보유 구형 ROI 파일이면 빈 배열을 반환해 기존 camerapos 폴백이 그대로 동작한다(하위호환).

### 9.4 `config/camerapos.json` 은 이제 필요한가 — 아니오(조건부)

| 용도 | 이전 | 현재 |
|---|---|---|
| `preset_pos` FK 부모 | camerapos 필수 | **ROI 파일로 충족** |
| `/capture/start` 순회 대상 | camerapos 필수 | **ROI 파일로 충족**(PTZ 보유 시) |
| 그 외 | — | 없음 |

즉 **ROI 파일이 프리셋 PTZ 를 담는 한 camerapos.json 없이 동작한다.** 다만 파일을 삭제하지는 않았다 —
PTZ 미보유 ROI 파일에 대한 폴백이자 `migrateToSettingDb` CLI 의 입력으로 남아 있다.

## 0. 한 줄 요약 — 반전

전편은 "줌 사다리가 옆차를 확대한다"는 증상을 **사다리 알고리즘의 결함**으로 다뤘다. 마스터가 실카에서 직접 클릭한 뒤, 로그 분석으로 드러난 진짜 원인은 전혀 달랐다: **`RealPtzSource`의 zoom raw 범위 설정(`HUCOMS_DEFAULT_ZOOM_RANGE = [0, 65535]`)이 이 장비의 실제 상한(`16384`)보다 4배 넓게 잡혀 있어서, 뷰어 zoom 9.75 이상의 모든 명령이 "범위 밖 명령"이 되어 카메라가 엉뚱한 화각으로 튀었다.** 사다리·게이트·배율·예산은 전부 설계대로 동작하고 있었다. 이 반전이 이터레이션 2 전체의 뼈대다 — 이번 수정 7~16 중 사다리 알고리즘 자체(게이트 스케일링·배율·rung 예산·상한 판정 로직)를 건드린 것은 하나도 없고, 전부 **그 주변(zoom 범위·UI 동기화·정착 대기·상한 도달 시 성공 판정)**에 대한 대응이다.

## 1. 서사 — 증상에서 진짜 원인까지

### 1.1 마스터 실측
- 먼 차량(좌측 끝): 2회 모두 **옆차 확대**.
- 우측 끝 차량: 1회차 성공 · 2회차 실패.
- 가까운 차: 성공.
- 빈 바닥 클릭: 정직한 실패 — **반경 게이트는 의도대로 작동**하고 있었다(이 사실이 나중에 "사다리 자체는 무결하다"는 결론의 근거가 된다).

### 1.2 마스터의 결정적 관찰
> "UI 카메라 컨트롤로 이동시키면 변수에 보관이 안 된다. 그전에 움직였던 위치로 가서 다시 움직인다."

이 관찰은 결과적으로 **두 개의 독립된 결함 중 하나(수정 7의 대상)를 정확히 짚었지만, "옆차 확대"의 주된 원인은 아니었다** — §1.4 참조.

### 1.3 로그 분석으로 드러난 진짜 원인
`logs/setting_20260721_164233.log`를 구현자가 직접 열어 확인했다. rung 5·6·7의 `errX/errY/plateWidth`가 12초에 걸쳐 **소수 3자리까지 동일한 값**이었다 — 같은 이미지를 세 번 분석했다는 뜻이다. 원인을 산술로 검산했다:

- `real-camera-1` config에는 당시 `ptz` 키가 없어 `zoomRange`는 코드 기본값 `[0, 65535]`.
- `mapRange(zoom, [1,36], [0,65535])`는 뷰어 zoom 14.310/18.212/23.183을 raw **24921/32229/41537**로 각각 다른 정수로 매핑한다 — **반올림 뭉개짐(rounding) 가설은 이 산술만으로 기각됐다.**
- 그런데 장비 로그의 정착 상한 초과 warn 5건이 결정적 증거였다:

| 명령 zoom(raw) | 실제 도달(raw) | 경과 |
|---|---|---|
| 19127 | 16384 | 5022ms |
| 24922 | 16384 | 5084ms |
| 32227 | 16384 | 5074ms |
| 41536 | 16384 | 5140ms |
| 65535 | **6350** | 5137ms |

pan/tilt는 매번 목표에 정확히 도달했는데 zoom만 **16384(=2^14)에서 멈춘다.** 이 장비의 zoom raw 실제 상한은 16384이고, 마스터가 OSD `x36` 스크린샷으로 이 수치를 확증했다.

### 1.4 인과 사슬

```
config zoomRange [0,65535]  (사양서 최대 폭, 이 장비의 실제 상한 아님)
  └─ 뷰어 9.75 → raw 16384 = 이미 장비 x36 최대
      └─ 뷰어 12.5 / 18.2 / 23.2 → raw 21567 / 31435 / 41537 = 전부 범위 밖
          ├─ 장비는 16384 에 머무름 → 같은 물리 위치 = 같은 프레임
          │    └─ rung 5·6·7 의 errX/errY/plateWidth 가 소수 3자리까지 동일
          │    └─ rung 간 6.2초 = waitUntilSettled 5초 타임아웃 소진
          └─ 뷰어 36 → raw 65535 = 범위 한참 밖
               └─ 장비가 zoom 6350 으로 튐 (로그 last zoom 6350)
                    └─ 카메라가 엉뚱한 화각으로 이동 → 그 화면 중앙의 옆차가 latch
                         = 마스터가 본 "옆차가 확대됨"의 직접 원인
```

즉 **"옆차 확대"는 사다리 결함이 아니라 범위 밖 zoom 명령이 카메라를 날린 결과였다.** 게이트·배율·예산은 정상 동작했다는 것이 빈 바닥에서의 정직한 실패(§1.1)로 이미 증거로 남아 있었다.

마스터가 짚은 "UI 이동 후 변수 미보관"(수정 7 대상)은 이 사슬과 **독립적인 2차 요인**이다: 조준 직후 `getPtz`가 슬루 중 값을 반환하면 다음 rung 명령이 그 중간 지점으로 카메라를 되돌리는 별도 결함(수정 8 대상)이 실재했고, 코드로도 확증됐다(`nativeAimSettleMs` 근거 재확인, §2.2). 두 결함은 서로 다른 결함이며 둘 다 고쳐야 했다.

### 1.5 파장
이 천장(zoomRange 오설정)은 사다리뿐 아니라 **배치 센터라이징 폭 수렴(`zoomToPlateWidth`)·UI 수동 줌·acquire 사다리** 등 zoom을 쓰는 모든 경로에 동일하게 걸려 있었다. **실카는 지금까지 명령 범위의 25%만 실제로 썼다** — 뷰어 zoom 9.75 이상이 전부 같은 물리 위치였다. 과거 배치 센터라이징의 일부 폭 수렴 실패가 이 천장 때문이었을 가능성이 있으나 **재측정되지 않았다**(§6 한계).

**정직하게 밝힐 것**: "36 = 광학 36배"라는 보장은 없다. `VIEWER_ZOOM_RANGE [1,36]`은 뷰어 공통 좌표계의 스케일 라벨일 뿐이고, 장비의 실제 광학 배율과 선형 대응한다는 근거는 코드·문서 어디에도 없다. 마스터 OSD가 `x36`을 보고한 것은 **장비 자신의 표기**이며 뷰어 상한(36)과 우연히 일치한 것일 수 있다. 폭 기반 목표(`targetPlateWidth`)를 쓰는 한 실무상 문제는 되지 않는다.

## 2. 수정 7~16 상세 (변경/신규 함수·의도·동작)

### 2.1 수정 7 — `web/app.js` `state.ptz` 부패 수정
방향 버튼(`stepPtz`)·절대이동(`resolveAbsPtz`)이 **`state.ptz` 기준으로 절대 목표를 계산**하는데, 서버 잡이 카메라를 움직여도 이 값이 갱신되지 않아 다음 조작이 낡은 위치로 되돌아갔다가 한 스텝만 움직였다 — 마스터가 짚은 증상 그대로다.

- 신규 헬퍼 `syncPtzAfterJob(responsePtz)`(`app.js:332` 부근, `refreshCurrentPtz` 바로 뒤): 실카는 응답 명령값을 믿지 않고 `refreshCurrentPtz({quiet:true})`로 **장비 실측**을 읽는다(명령값은 슬루 중간에 잘리거나 광학 한계에서 클램프될 수 있다 — §1.3이 그 실례). 시뮬은 응답 `ptz`로 즉시 반영, 없으면 서버 조회 폴백. 양쪽 모두 `updatePtzDisplay()`로 화면 표시를 동기화한다.
- 최초 배선 4곳: `calPointCenter`(개별 센터라이징 point/plate/plate-zoom 전부) · `calPoll`(배치 센터라이징) · `discPoll`(discovery) · 수집 폴(캡처 프리셋 순회).
- **운영 주의**: `web/`은 nodemon 감시 밖이다 — 마스터가 브라우저를 강력 새로고침(Ctrl+F5)해야 반영된다.

### 2.2 수정 8 — 네이티브 조준 정착 대기 (`RealPtzSource.ts`)
`centerOnPoint`가 `currentPtz()`(슬루 중 값일 수 있음)를 그대로 반환하고, 사다리가 그 값을 다음 rung의 `requestImage` 명령으로 쓰면 `move`의 `waitUntilSettled`가 카메라를 그 중간 지점까지 실제로 되돌리는 경로가 코드로 확증됐다. 마스터 로그의 rung0(`k=1`, 조준 직후) `distAtAim`이 0.205/0.382/0.214로 게이트 0.10의 2~4배였던 것이 직접 증거다.

- 신규 `waitUntilStopped()`(`RealPtzSource.ts`): 목표를 모르는 이동의 "정지까지" 대기. **움직임을 한 번 본 뒤의 정지만 정착으로 인정**(명령 직후 미출발 구간을 정지로 오판하는 함정 회피). `SETTLE_START_GRACE_POLLS = 7`(150ms×7≈1050ms — 사다리가 기존에 무조건 물던 고정 대기 1000ms와 같은 크기).
- `centerOnPoint`가 `Ptz & {settled}` 반환. 타임아웃은 warn 로그 + `settled:false`(삼키지 않음).
- `platePtz.ts`의 `recenterTo`: `settled === false` → warn + `ok:false` → 사다리가 `aim_failed`로 종료(미정착 PTZ로 다음 rung을 명령하지 않는다). `settled === true`면 고정 sleep 생략, `undefined`(정착 판정 미제공 소스)는 기존 `nativeAimSettleMs` 폴백 유지.
- **`mode:'point'`(마스터가 "잘 된다"고 확인한 개별 center) 영향**: 동작은 개선(응답 PTZ가 슬루 중 값이 아니라 정지 위치라 더 정확해짐), 퇴행 없음. 다만 응답이 슬루 완료까지 기다리므로(최대 `settleTimeoutMs`) **체감 지연은 늘어난다** — 이는 원래 카메라가 움직이던 시간을 이제 숨기지 않고 보여주는 것이다.

### 2.3 수정 9 — rung 진단 계측 (`platePtz.ts`)
기존 `phase:'ladder'` 로그 구조·필드명은 유지하고 가산만 했다: `zoomCmd`(명령 zoom, 기존 `zoom`과 짝) · `zoomAct`/`panAct`/`tiltAct`(장비 실측 PTZ) · `bytes`/`sha`(프레임 지문, sha1 앞 8자). `zoomCmd`는 오르는데 `zoomAct`가 안 오르면 §1.3의 원인이 즉시 드러난다. 계측은 검출 rung·기각 로그·최대줌 warn·포화 warn 4곳 전부에 실었다. 구현자 스스로 "이 계측만 있었다면 이번 원인을 첫 클릭에 알 수 있었다"고 기록했다.

### 2.4 수정 10 — zoomRange 정정 (`config/tools.config.json`, `tools.config.example.json`)
`real-camera-1`·`real-camera-2`에 `"ptz": { "zoomRange": [0, 16384] }` 추가(외과적 텍스트 삽입, 나머지 포맷 보존). `RealPtzSource.ts`의 `HUCOMS_DEFAULT_ZOOM_RANGE` 값(`[0,65535]`)은 **그대로 유지**했다 — 여기서 낮추면 실제로 65535를 쓰는 다른 모델이 가용 배율의 3/4를 조용히 잃는다. 신규 장비는 config로 실측 상한을 지정하는 방식으로 대응한다.

검산: `mapRange(36, [1,36], [0,16384])` = **16384**, 장비 최대와 정확히 일치.

### 2.5 수정 11 — zoom 실측 정체 판정 (`platePtz.ts`)
`move()`는 `waitUntilSettled` 타임아웃에도 `true`를 반환하므로 사다리는 줌 명령이 성공한 줄 안다. `clampZoom`은 뷰어 범위만 알아 장비 물리 상한을 모른다 → **실측(`zoomAct`)으로 판정**한다.

```ts
if (actLive && dCmd > EPS && dAct <= EPS) stall += 1; else stall = 0;
if (stall >= LADDER_ZOOM_STALL_LIMIT) → zoom_saturated
```

| 상수 | 값 | 근거 |
|---|---|---|
| `LADDER_ZOOM_STALL_EPS` | 0.05(뷰어 배율, ≈raw 50@[0,16384]) | "조금이라도 움직였으면 정상"으로 보되, 로그에 실재하는 정상 저속 케이스(5초 안에 미도달이나 실측은 분명히 변함)를 오판하지 않도록 |
| `LADDER_ZOOM_STALL_LIMIT` | 2 | 1회 미상승은 폴링 타이밍·인코더 양자화로도 생김. 2회 연속이면 물리 상한으로 단정할 근거 |
| `actLive` 가드 | — | 실측이 명령을 따라 움직이는 것을 **한 번이라도 확인하기 전에는 판정하지 않는다** — echo를 신뢰할 수 없는 소스(Unity 시뮬 0/0/1 고정)에서 "항상 정체"로 오판해 시뮬 전체를 죽이는 것을 구조적으로 차단 |

### 2.6 수정 12 — 정착 타임아웃 5000 → 15000ms
실측(`logs/setting_20260721_163311.log`): 줌아웃 raw 16384→2478(Δ13906) 명령에서 5.03초 시점 실측 10095(Δ6289만 이동, ≈1250 raw/s). 전 구간(16384) 이동에 약 13초 필요 → 15초는 약 15% 여유. **트레이드오프**: 장비가 응답 불능이거나 도달 불가 목표를 받으면 이동 1회가 최대 15초(기존 5초) UI를 멈춘다. 미정착 반환이 곧 오조준(센터링 부분 취소)이므로 "빨리 틀리는 것"보다 "늦게 맞는 것"을 택했다. 수정 11(도달 불가 구간 조기 종료)과 짝으로 설계됐다.

### 2.7 수정 13 — 장비 상한 도달 시 성공/실패 경계 (`platePtz.ts` `saturatedOutcome`)
**이 이터레이션에서 성공/실패 경계를 실제로 옮긴 유일한 변경**이다. 상한(clampZoom 포화 또는 수정 11의 zoomAct 정체)에 도달했을 때, 폭 미달이라도 **대상 신원(latch)과 화면중앙 정렬이 모두 검증되면** `ok:true` + `widthShortfall:true`로 성공 처리한다.

```ts
if (!latched || !plate || !err) return false;  // 대상 미확보 → 실패 유지
return isCentered(err, centerTol);              // 중앙 tol 밖 → 실패 유지
```

**왜 위장 성공 금지(Goal 1순위)와 충돌하지 않는가**: 위장 성공은 이 작업 내내 "클릭한 대상이 아닌 것을 잡고 완료라 하는 것"으로 일관되게 정의됐다. 수정 13이 성공으로 바꾸는 상태는 **신원과 정렬이 모두 실측으로 검증된** 상태이고, 미달한 것은 장비의 물리 한계뿐이다. 그 사실을 `widthShortfall`/`reason`으로 결과에 남기므로 정보 은닉이 아니다. 반대로 이를 실패로 두면 마스터는 화면상 성공인 결과를 보면서 행동 불가능한 실패 메시지("더 확대하라"는 물리적으로 불가능)를 읽게 되어 오히려 정직하지 않다.

**금지선이 코드 어디에 고정됐는가**: `saturatedOutcome`은 latch·중앙정렬을 확인하는 게이트 두 겹이고, 호출 지점은 오직 두 곳 — ① `clampZoom` 포화 ② `zoomAct` 연속 정체(수정 11) — 즉 **"상한 도달"이 사실로 확인된 자리에서만** 호출된다(추정 판정 없음). 정렬 판정은 그 rung의 실측 오차로만 하며, 재중심 명령만 내리고 측정하지 못한 상태는 여전히 `ok:false`다. QA가 별도로 반례 7건(마스터 원래 시나리오 2가지 출구·신원 치환 공격·치환 경계·centerTol 경계·미측정 재중심·DB 오염 여부)을 설계해 돌렸고 **전건 차단을 확인**했다(`03_qa_report.md` §S2).

**정직하게 밝힐 한계(QA S2-①)**: 폭 미달 정도에 하한이 없다 — 목표의 9%(0.02/0.215)여도 `ok:true`다. 논리적으로는 옳지만(장비가 할 수 있는 일을 전부 함) 마스터 화면에는 판이 여전히 점으로 보이는데 "완료"가 뜬다. UI가 `장비 최대 배율(zoom_saturated)`을 병기하므로 정보 은닉은 아니나, QA는 이를 결함이 아니라 **UX 경계**로 분류했다.

- UI: `web/app.js` 완료 메시지에 `개별 센터라이징 완료 — 장비 최대 배율(zoom_saturated)` 문구 추가.

### 2.8 리더 파이프라인 밖 핫픽스 — `toolsConfig.ts` `ptz` 축별 optional
수정 10(config에 `zoomRange`만 추가) 적용 직후, **마스터 서버가 ZodError로 기동 실패**했다. 원인은 `CameraSourceConfigSchema.ptz`가 `panRange`/`tiltRange`/`zoomRange` 세 필드를 함께 요구하는 구조였는데(`.tuple` 개별은 이미 optional이 아니었을 가능성) config에 `zoomRange`만 지정하니 나머지 두 필드가 "Required"로 걸린 것이다. 실제 코드(`RealPtzSource.ts:132~134`)는 이미 축별 `?? HUCOMS_DEFAULT_*` 폴백이라 개별 축만 있어도 동작에 지장이 없었는데, **스키마만 코드가 요구하지 않는 것을 요구하고 있었다.**

리더가 파이프라인 밖에서 급히 `panRange`·`tiltRange`·`zoomRange` **세 필드 각각을 `.optional()`**로 바꿔 넣었다(`toolsConfig.ts:243~245`). **당시 유닛테스트 없이 `tsc` 통과만 확인하고 반영**했다 — 이 경위를 숨기지 않는다.

이후 QA가 최우선으로 검증했다(`03_qa_report.md` §S1): 기존 config(3축 전부 지정) 유지·`zoomRange`만 지정(현재 실카 구성)·`ptz` 자체 미지정(시뮬)·실제 `config/tools.config.json` 로드 시 throw 없음·축별 optional이 `RealPtzSource` 폴백과 raw 목표값 수준까지 정합함을 확인했고, 느슨해진 것이 "개별 필드의 필수 여부"뿐이며 3-튜플이 아닌 값(1-튜플·문자열·숫자)은 여전히 거부됨을 고정했다. `test/toolsConfigPtzOptional.test.ts`(11케이스, 신규 영구 테스트)로 이 크래시를 회귀 가드로 봉인했다 — 재발하면 이 테스트가 먼저 실패한다.

### 2.9 수정 14 — 카메라 이동 경로 전수 조사와 봉인
수정 7은 처음에 "잡 4곳"으로 열거했다. **QA가 코드를 라우트→서버까지 직접 추적해 2곳을 더 찾았다**(`03_qa_report.md` §S6): `/capture/detect`(VPD 검출 테스트 버튼, `detectPipeline.ts:329-332`가 미귀속 차량마다 카메라를 확대·이동시키고 복귀 코드가 없음)와 자동체인의 `discovering` 단계(원버튼 정밀수집 시작 시 프리셋 전체를 앵커 loop LPD로 도는 구간인데, `pollPipeline`이 `calibrating` 전이에만 `calPoll`을 재기동하고 `discPoll` 재기동이 없어 동기화가 발화하지 않음).

**열거가 왜 실패했는가**: "카메라를 움직이는 잡이 어디인가"를 사람이 나열하는 방식은 시야에 든 것만 잡는다 — 구현자도 QA도 처음엔 각각 4곳/누락 2곳까지만 봤다. 이를 대체한 것은 **소스 전수 grep + 라우트별 서버 코드 추적 + 미분류 라우트 0 검사**다:

1. `move(`·`requestImage(ptz)`·`centerOnPoint`를 소스 전수 grep한 뒤 app.js가 부르는 모든 라우트를 서버 코드까지 추적해 이동 여부를 판정(전체 40개 라우트, 이동 6종·비이동 34종으로 분류).
2. 신규 배선 2건: `runLiveDetect`에 `syncPtzAfterJob(null)` 추가(**실패 경로에도** — 서버가 몇 대를 이미 이동시킨 뒤 실패했을 수 있어서), `pollPipeline`에서 `discovering` 진입 시 `discPoll()` 재기동 + 체인 종단(`done`/`failed`)에서 한 번 더 동기화(discovery 실패·LPD 타깃 0 종결은 `calibrating`을 거치지 않으므로).
3. `test/viewerPtzSyncCoverage.test.ts`(신규 11케이스, app.js 소스 정적 검사)로 **미분류 라우트 0**을 봉인 — 새 라우트가 추가되면 이 테스트가 "이 라우트가 카메라를 움직이는지 판정하라"고 요구하며 실패한다. 세 번째 누락이 구조적으로 불가능해진다.

**한계(은닉 금지)**: 이 테스트는 app.js를 실행하지 않는다(브라우저 전역 의존). "호출이 존재한다"는 보장하지만 "런타임에 실제로 발화한다"는 보장하지 않는다. S6-①/②의 실제 동작은 마스터 브라우저에서만 확인 가능하다.

**손대지 않은 것**: S6-③(배치 잡이 `source`를 동봉하지 않는데 동기화는 `state.source`를 조회하는 구조적 불일치)은 이번 범위 밖으로 남겼다(§4 한계 참조).

### 2.10 수정 15 — 정지·목표미달 조기 반환 (`RealPtzSource.ts`)
`isNearTarget` 허용오차부터 검토했으나 **원인이 아니었다**: 라이브 타임아웃 41건 전수 분석 결과 40건이 pan/tilt는 이미 EPS(10) 이내로 도달했고 zoom만 미달이었다(zoom 차이 중앙값 4865raw) — "허용오차가 빡빡해서"가 아니라 "장비가 도달할 수 없는 목표를 받아서"다. EPS는 건드리지 않았다.

`waitUntilSettled`가 수정 8의 `waitUntilStopped` 관용구를 재사용해 두 조기 종료를 추가:
- `stopped_short`: 움직였다가 `SETTLE_STALL_SAMPLES(3)` 연속 동일 → 목표 미달이어도 반환. 근거: 150ms×3=450ms 무변화, 실측 줌 속도(≈1250raw/s)로는 정상 이동 중 연속 3회 동일이 물리적으로 불가능.
- `no_motion`: `SETTLE_START_GRACE_POLLS(7)` 동안 전혀 안 움직임 → 도달 불가 목표로 판단.

수정 11과의 관계(충돌 없음, 같은 방향): 수정 15(전송 계층)는 "더 기다릴지"만 결정하고 보고 PTZ를 바꾸지 않는다. 수정 11(제어 계층)은 그 실측이 정체면 종료한다. 둘이 합쳐 정체 rung이 15초 → 약 1초로 줄어 마스터 실패 케이스가 30초 → 2~3초로 단축된다. `move()`의 반환 계약(`true`)은 바꾸지 않았다 — `stopped_short`는 통신 실패가 아니라 장비의 물리 한계이고, false를 돌리면 수동 이동 UI가 "실패"로 보이거나 기하 폴백이 `aim_failed`로 죽기 때문이다.

**남는 한계**: QA S4가 지적한 "느리지만 계속 움직이는" 장비 최악 케이스는 그대로 남는다 — 실제 이동 중이므로 기다리는 것이 옳고, 조기 종료하면 미정착 오조준이 재발한다. 이 경우만은 rung당 최대 15초가 정당한 비용이다.

### 2.11 수정 16 — 영속 zoom 17건 출처 조사 (데이터 미변경)

### 4.2 운영 주의사항 (필독)
- **`web/` 변경분은 nodemon 감시 밖이다** — 서버 자동 재기동으로는 반영되지 않는다. 마스터가 브라우저를 강력 새로고침(Ctrl+F5)해야 수정 7·13·14의 UI 동작이 적용된다.
- **`config/` 변경(zoomRange 정정 포함)은 서버 재기동이 필요하다** — `loadToolsConfig()`는 기동 시 1회 로드이며 런타임 핫리로드가 아니다. 마스터가 `config/tools.config.json` 갱신 후 서버를 재기동하지 않으면 여전히 구 zoomRange로 동작한다.

### 4.3 REST 계약·config 하위 호환
이터레이션 1과 동일하게 응답 shape·엔드포인트는 무변경이며, 신규 필드는 전부 옵셔널이라 기존 클라이언트가 죽지 않는다. `CameraSourceConfigSchema.ptz` 축별 optional화는 기존 config(3축 전부 지정)를 그대로 수용하면서 부분 지정도 허용하도록 **완화만** 했다 — 3-튜플이 아닌 값(1-튜플·문자열 등)은 여전히 거부되므로 스키마의 형태 검증력은 유지된다.

## 6. 미해결·한계 (은닉 금지)

1. **마스터 실카 재검증이 아직 수행되지 않았다.** 이 문서의 인과 사슬(§1.4)은 로그 사후분석과 산술 검산으로 확정됐지만, zoomRange 정정 이후 실제로 먼 차량 클릭이 성공하는지는 **아직 실측되지 않았다.**
2. **QA S4** — "느리지만 계속 움직이는" 장비는 여전히 rung마다 최대 15초를 대기한다(§2.10 한계). 자동 예산이 크면(최대 19칸) 이론상 최악 약 5분이 될 수 있다. 라이브에서 이 패턴이 나오면 체감이 크게 나빠진다.
3. **`real-camera-2`(192.168.0.154)가 미실측이다.** `real-camera-1`과 같은 기종이라 같은 `zoomRange:[0,16384]`를 넣었으나 확인되지 않았다. 다른 값이면 같은 유형의 실패가 재발한다.
4. **S6-③(배치 잡 `source` 미동봉)은 이번 범위 밖으로 남겼다.** `/calibrate/ptz`·`/discover/ptz`·`/capture/start`는 `source`를 동봉하지 않아 고정 파이프라인 카메라를 명령하는 반면, `syncPtzAfterJob`은 `state.source`를 조회한다 — 단일 소스 운용에서는 드러나지 않지만 구조적 불일치이며 리더 판단이 필요한 별건으로 남아 있다.
5. **QA가 코드 정독으로만 판정한 항목(S6-①/②)은 브라우저 실행으로 재현되지 않았다.** `/capture/detect` VPD 테스트 버튼과 자동체인 `discovering` 단계의 실제 동작은 마스터 브라우저에서만 확인 가능하다.
6. **뷰어 UI 실측 미확인.** `ok:true` + `장비 최대 배율(zoom_saturated)` 문구가 마스터 화면에 실제로 어떻게 보이는지 아무도 육안으로 보지 않았다.
7. **배치 센터라이징 폭 수렴률 재측정 미수행.** zoomRange 정정으로 천장이 3.7배 넓어진 효과는 실장비에서만 관측 가능하다(§1.5, §4.1).

### 라이브 재검증 시 반드시 채록해야 할 것
1. `phase:'ladder'` 로그의 `zoomCmd`/`zoomAct`/`sha` — zoomRange 정정 후 명령과 실측이 일치하는지.
2. 클릭 → 완료까지 체감 시간(수정 8·11·12·15가 겹쳐 작용하는 실제 효과).
3. VPD 검출(테스트) 버튼은 **재검증 중 사용하지 말 것**을 권고한다(QA S10) — 그 버튼이 §2.9의 S6-① 경로를 그대로 타면 마스터 원래 증상이 재발할 수 있다.

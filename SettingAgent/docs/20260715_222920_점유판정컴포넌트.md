## 1. 개요 · 목적

`OccupancyJudge` 는 슬롯(바닥 ROI)별 차량 점유 여부를 판정하는 **순수 컴포넌트**(IO·DOM·전역 state 무의존)다. 파일 `web/occupancy.js` 에 단독 클래스로 구현되었다.

### 해결한 문제

기존 점유 판정(`web/core.js:computeOccupancy`)은 **번호판(LPD)** 중심이 바닥 폴리곤 안에 들면 occupied 로 본다. 이 방식은 번호판이 검출되지 않은 프레임에서 실제로 차량이 있어도 점유가 `false` 로 뒤집히는 약점이 있었다. `OccupancyJudge` 는 여기에 **차량 bbox(VPD 검출) 폴백**을 얹어, 번호판이 인식되지 않은 주차 차량도 "점유·번호미인식"으로 유지한다.

핵심 설계 원칙:
- **번호판 우선**: 번호판이 보이면 기존과 100% 동일하게(위임 호출) 판정 → 회귀 0.
- **bbox 폴백**: 번호판으로 점유되지 않은 슬롯에만, 차량 접지 밴드 겹침으로 보강.
- **기하 알고리즘 신규 발명 금지**: 접지 기준·겹침 판정은 이미 검증된 `onPlaceFilter`(모드A 필터) 정의를 그대로 재사용.

## 2. 클래스 API (`web/occupancy.js` / `web/occupancy.d.ts`)

### 2-1. 생성자

```ts
new OccupancyJudge(cfg?: {
  groundBandRatio?: number;  // 접지 근사 밴드 비율(bbox 하단 스트립). 기본 0.25 (= src GROUND_BAND_RATIO)
  minBandOverlap?: number;   // 밴드 면적 대비 슬롯 겹침 하한. 기본 0.15 (= src ON_PLACE_MIN_OVERLAP)
})
```

- **상태 없음**: 생성자 config(두 임계값)만 보유. 인스턴스를 매 프레임 재사용해도 안전(뷰어는 모듈 싱글턴 `const occupancyJudge = new OccupancyJudge()` 로 사용).
- 임계값은 **생성자 한 곳**에서만 관리 → goal/loop 튜닝 지점이 단일.

### 2-2. `judge` 시그니처

```ts
judge(
  floorPolygons: Array<{ idx: number; quad: NormalizedPoint[] }> | null | undefined,
  detect: {
    plates?:   Array<{ quad: NormalizedPoint[] }> | null;
    vehicles?: Array<{ rect: NormalizedRect; plate?: { quad: NormalizedPoint[] } | null }> | null;
  } | null | undefined,
): OccupancyJudgement[]
```

- 소비처는 검출 결과(`detect`)를 **그대로** 넘긴다. 번호판 union(standalone plates ∪ 차량 귀속 plate) 조립은 `judge` 내부에서 수행 → 소비처의 union 누락 실수 여지 제거.
- graceful: `floorPolygons` 가 배열이 아니면 `[]`. `detect`/`plates`/`vehicles`/`rect` 누락, 퇴화 rect(면적 0)는 해당 항목 skip. **throw 없음**(기존 `computeOccupancy` 강등 철학 동일).

### 2-3. 결과 shape

```ts
interface OccupancyJudgement {
  idx: number;                      // 바닥 폴리곤 전역 인덱스(입력 그대로)
  occupied: boolean;
  source: 'plate' | 'bbox' | null;  // 판정 근거. null = 빈 면
  center?: NormalizedPoint;         // source==='plate' 일 때만: 번호판 중심(기존 computeOccupancy center 그대로)
  vehicleRect?: NormalizedRect;     // source==='bbox' 일 때만: 판정 근거 차량 bbox(정규화)
}
```

세 상태가 구분 표현된다:

| 상태 | occupied | source | 부가 필드 |
|---|---|---|---|
| 번호판 점유 | `true` | `'plate'` | `center`(번호판 중심) |
| 점유·번호미인식 | `true` | `'bbox'` | `vehicleRect`(근거 차량 bbox) |
| 빈 면 | `false` | `null` | — |

> `center` 필드명을 유지한(설계 예시의 `plateCenter` 대신) 이유: 기존 소비처(`occComputeByKey.spaces[].center`, `drawOccupancyOverlay`)가 `center` 를 그대로 읽으므로 마이그레이션 diff 가 최소화되고 plate 경로 무변경이 자명해진다.

## 3. 2단계 판정 알고리즘

`judge(floorPolygons, detect)` 는 슬롯 단위로 **plate > bbox** 우선순위를 적용한다.

### 1단계 — 번호판 (기존 경로 위임)

```
plateCandidates = [...detect.plates, ...detect.vehicles.map(v => v.plate).filter(Boolean)]
base = computeOccupancy(floorPolygons, plateCandidates)   // web/core.js 함수를 그대로 호출(재구현 없음)
rows = base 행별로 occupied 면 {source:'plate', center} 부여, 아니면 {occupied:false, source:null}
```

- 번호판 판정 로직 자체는 **한 줄도 재구현하지 않고** `computeOccupancy` 를 호출한다. 번호판 중심(`quadCentroid`, 4점 산술평균)이 바닥 폴리곤 내부(`pointInQuad`)면 occupied — 기존 정의 그대로.

### 2단계 — bbox 폴백 (1단계 비점유 슬롯만)

```
for v in detect.vehicles:
  if v.plate 있고 quadCentroid(v.plate.quad) 가 어떤 폴리곤 안:  continue  // 1단계가 이미 소비 → 이중 마킹 차단
  band     = groundBand(v.rect, groundBandRatio)   // bbox 하단 25% 스트립(접지 근사)
  bandArea = area(band);  if bandArea <= 0: continue // 퇴화 rect skip
  각 슬롯 j 에 대해 ratio_j = convexIntersectionArea(rectCorners(band), floor[j].quad) / bandArea
  j* = argmax_j ratio_j                            // ★ 최대 겹침 슬롯 단 1개
  if ratio_{j*} >= minBandOverlap:  슬롯 j* 후보로 등록(최대 ratio 차량만 유지)

for 후보 있는 슬롯 pos:
  if rows[pos].occupied:  continue                 // plate 우선 — 1단계 점유 슬롯은 절대 덮지 않음
  rows[pos] = {occupied:true, source:'bbox', vehicleRect: 최대겹침 차량.rect}
```

### bbox → 슬롯 배정 규칙 (핵심 결정)

| 규칙 | 내용 | 근거 |
|---|---|---|
| 접지 근사 | bbox 하단 25% 밴드(`groundBand`). bbox **중심점** 규칙은 원근 이탈로 FP 확정(onPlaceFilter 주석) → 사용 안 함 | 필터(밴드 기준)와 판정(밴드 기준)의 일관성 |
| 배정 = **argmax 1슬롯** | 밴드 ∩ 슬롯 면적비 최대 슬롯 **하나만** 점유. 2슬롯에 걸친 차량이 두 면을 점유시키지 않음 | 이중 점유(FP) 방지 |
| 임계 `minBandOverlap = 0.15` | 최대 겹침비가 0.15 미만이면 어느 슬롯에도 배정 안 함 | `onPlaceFilter.ON_PLACE_MIN_OVERLAP=0.15` 재사용(모드A 필터에서 실측 검증) |
| tie-break | 겹침 ratio 비트 동일 시 배열 앞 슬롯(strict `>` 순회로 결정적) | 결정성 보장(실측 동률 확률 ≈ 0) |
| 한 슬롯 여러 후보 | occupied 1회, `vehicleRect` = 겹침 최대 차량 | 표시 근거 1개면 충분 |
| plate 보유 차량의 plate 중심이 폴리곤 밖 | 그 차량은 2단계 후보에 **포함**(bbox 로 점유 구제) | 점유 놓침(FN) 방지 우선 — §7 트레이드오프 참조 |
| det 신뢰도 하한 | 컴포넌트에서 **미적용** | detectPipeline 이 이미 신뢰도 컷·모드A 필터 수행 → 이중 필터는 추측성 기능 |

## 4. 기하 파리티 포트 — 왜 `web/` 에 포팅했나

bbox 폴백에 필요한 기하 함수(`groundBand`·`rectCorners`·`convexIntersectionArea`·`polygonArea`·`polygonCentroid`·`clipByHalfPlane`·`area`)와 상수(`GROUND_BAND_RATIO`·`ON_PLACE_MIN_OVERLAP`)는 **src(TypeScript)에만** 존재한다(`src/domain/polygon.ts`·`geometry.ts`·`src/capture/onPlaceFilter.ts`).

문제: 뷰어(`web/app.js`)는 **빌드 없이 브라우저가 ESM 을 직접 로드**한다. 브라우저 ESM 은 `src/*.ts` 를 물리적으로 import 할 수 없다(web → src 역방향 불가).

해결(코드베이스 확립 관례 — `quadCentroidParity`·`globalIdxParity` 전례):
- src 원본을 **자구 그대로(동일 식) `web/occupancy.js` 에 포팅**하고 export.
- `test/occupancyGeometryParity.test.ts` 가 src 원본과 **비트 동일** 출력을 봉인(고정 케이스 + 무작위 200조합).
- `core.js`/`core.d.ts` 에는 넣지 않음(무변경 보장 표면 유지).

이는 "신규 기하 알고리즘 발명 금지"의 취지(새 알고리즘 금지)를 지키는 유일한 실행 가능 경로다. 기각한 대안: ① web 빌드 도입(과설계), ② `src/domain/polygon.ts` 를 web 재수출로 리팩토링(검증된 육면체 파이프라인 의존 모듈 침범 — 외과적 변경 원칙 위반), ③ bbox 하단중심 점 규칙(원근 이탈 FP 재도입 + 필터↔판정 비일관).

## 5. 기존 `computeOccupancy` 와의 관계 — 래핑(위임), 회귀 0의 구조적 근거

`OccupancyJudge.judge` 의 1단계는 `computeOccupancy` 를 **그대로 호출**한다(대체·복사·재구현 아님).

**회귀 0 은 테스트로 확인하는 성질이 아니라 코드 구조가 보장하는 성질이다**:
- `vehicles` 가 비었거나 bbox 후보가 없으면 2단계는 no-op.
- 그 경우 결과는 `computeOccupancy(floorPolygons, plateCandidates)` 의 각 행에 `source` 필드만 얹은 것과 **항등**.
- plate 경로는 별도 구현이 아니라 기존 함수 그 자체이므로 "번호판만 있을 때 기존과 동일"은 구조적으로 참.
- 1단계 union 식(`plates ∪ vehicles[].plate`)도 기존 `app.js` 소비처 및 `lpdFilterRegression.test.ts` 가 규정한 식과 동일.

이 성질은 그래도 테스트 T1(§6)로 이중 고정된다.

## 7. 소비처 렌더 (`web/app.js`)

### `updateLogicOccupancy` (수정)

union 수동 조립을 제거하고 `occupancyJudge.judge(floorPolys, detect)` 를 호출. 결과를 `spaces` 로 매핑하며 `source`·`vehicleRect` 를 통과시킨다(기존 `id`·`occupied`·`center` 유지):

```js
state.occComputeByKey[key] = {
  spaces: occupancyJudge.judge(floorPolys, detect)
    .map((o) => ({ id: o.idx, occupied: o.occupied, source: o.source, center: o.center, vehicleRect: o.vehicleRect })),
};
```

### `drawOccupancyOverlay` (수정 — source 별 렌더)

- **source='plate'**: 번호판 중심(`center`)에 **빨강(#ff4d4d) 원 + id** — 기존과 완전 동일(픽셀 단위 회귀 0).
- **source='bbox'**: 차량 bbox 하단 중심(`(x+w/2, y+h)`, 접지 근사)에 **주황(#ff9f1a) 원 + "{id} 번호미인식" 배지**.
- 기존 `occupied && center` 가드와 호환 — bbox 표시는 순수 추가분이므로, 오버레이를 고치지 않았더라도 plate 행은 깨지지 않는다.

## 8. 한계 · 후속 과제

1. **bbox 배정은 접지 밴드 근사**: 하단 25% 스트립은 원근에서 근사이며, 급경사·밀착 차량에서 옆 칸 오배정 가능성. seg 마스크/3D 육면체는 실측상 밀착 차량 불안정으로 배제됨.
2. **det 신뢰도**: 컴포넌트에서 미적용(파이프라인이 이미 컷). 라이브에서 저신뢰 거대 병합 박스가 bbox FP 를 만들면 `minConfidence` 옵션 추가를 재논의(지금 넣는 것은 추측성 기능).
3. **`minBandOverlap` 튜닝점**: 기본 0.15 는 모드A 필터 검증값에서 출발. 점유 판정에서 최적값이 다를 수 있어 생성자 config 로 goal/loop 튜닝 대상.
4. **plate 중심 폴리곤 밖 → source='bbox' 표시**: 실제로는 번호판이 인식된 차량도 "번호미인식"으로 표시될 수 있는 표시 의미론 트레이드오프(점유 FN 방지를 우선). 엄격 해석을 원하면 candidates 규칙 한 줄 변경으로 대응 가능.
5. **후속 관찰(확인 필요)**: 실 라이브 브라우저 육안으로 source 별 렌더(주황 "번호미인식") 및 번호판 실제 미검출 케이스의 폴백 정확도를 계속 관찰. `lpdFilterRegression.test.ts` 주석의 union 소비처 참조 라인(`web/app.js:335`)은 union 이 `occupancy.js` 로 이동해 낡음 — 주석 갱신 대상.

## 1. 배경 · 문제(마스터 보고 원문 기준)

정밀수집(capture) 화면에서 두 가지 증상이 보고되었다.

1. **정밀수집 종료 리셋 미비**: 정밀수집이 끝나면(`done`/`stopped`/`error`) 화면에 검출 박스·차량 육면체·번호판 오버레이가 **마지막 프레임 위에 얼어붙은 채** 남는다. "시작 전 상태"(라이브뷰 + 오버레이 없음)로 돌아가야 한다.
2. **센터라이징 화면 미갱신**: 센터라이징(PTZ 정렬) 진행 중에는 화면이 아예 갱신되지 않는다.

### 마스터 질문에 대한 답: "매번 패킷을 시뮬레이터로 보내 화면을 갱신해야 하는가?"

**아니오.** 정밀수집의 화면 갱신은 이미 다음 패턴으로 동작한다.

- 잡(`CaptureJob`)이 자신의 작업 주기로 카메라에 `requestImage`를 보내 프레임을 찍는다(콘솔에 보이는 **3초 로그는 이 캡처 주기**이지, 웹의 폴링이 아니다).
- 잡은 방금 찍은 JPEG을 메모리 버퍼(`CaptureJob.lastFrame`, `CaptureJob.ts:349`)에 보관한다.
- 웹은 500ms 간격으로 `GET /capture/frame`(`captureRoutes.ts:190`)을 폴링해 **그 버퍼를 그대로 받아** 화면에 그린다.

즉 **웹 폴링은 카메라를 재촬영시키지 않는다** — 잡이 이미 찍은 프레임을 읽기만 한다. 이번 변경은 센터라이징에도 **동일한 버퍼-폴 패턴**을 그대로 복제해 추가 카메라 패킷을 0으로 유지하면서 화면을 실시간으로 갱신한다. 정밀수집 종료 리셋도 카메라 재명령이 아니라 **웹 로컬 상태 정리 + 기존 라이브 스트림 재연결**만으로 해결한다.

## 2. 변경 상세

### 2.1 서버 — 센터라이징 최신 프레임 버퍼 + 라우트

#### `src/calibrate/platePtz.ts`
- `PlatePtzDeps`에 `onFrame?: (jpeg: Buffer, camIdx: number, presetIdx: number) => void` 추가(옵셔널·가산).
- `captureAndDetect`(내부 캡처 헬퍼, `camera.requestImage` 직후)에 `this.onFrame?.(cap.jpg, camIdx, presetIdx);` 1줄 추가. **새 `requestImage` 호출은 없다** — 이미 찍은 `cap.jpg`를 그대로 훅에 흘려보낸다. 이 헬퍼가 `PlatePtz`의 유일한 캡처 지점이므로 center/probe/zoom 등 모든 캡처가 이 한 곳으로 커버된다.

```ts
const cap = await this.camera.requestImage(camIdx, presetIdx, ptz);
this.onFrame?.(cap.jpg, camIdx, presetIdx);   // 신규 1줄
await this.sleep(this.opts.settleMs);
```

#### `src/calibrate/PtzCalibrator.ts`
- private 필드 `lastFrame?: { jpeg: Buffer; camIdx: number; presetIdx: number }` 추가(`CaptureJob.lastFrame`과 동일 패턴).
- public `getLastFrame()` 추가 — 잡 종료(`done`/`error`) 후에도 버퍼를 **유지**한다(웹이 마지막 프레임을 읽은 뒤 라이브뷰로 전환할 수 있도록).
- 생성자에서 `onFrame` 클로저를 정의해 **기본 팩토리**(`deps.makePlatePtz` 미주입 시 사용하는 기본 `PlatePtz` 생성 경로)에 주입한다. `calibrateSlot`이 호출하는 `makePlatePtz({...})` 시그니처(옵션만 받음)는 변경하지 않았다 — `onFrame`은 팩토리 클로저 안에 이미 baked-in.

```ts
const onFrame = (jpeg: Buffer, camIdx: number, presetIdx: number): void => {
  this.lastFrame = { jpeg, camIdx, presetIdx };
};
this.makePlatePtz = deps.makePlatePtz ?? ((opts) => new PlatePtz({ camera: deps.camera, lpd: deps.lpd, sleep, onFrame }, opts));
```

#### `src/api/calibrateRoutes.ts` — `GET /calibrate/frame` 신규 라우트
`deps.calibrator.getLastFrame()`을 그대로 반환한다. `/capture/frame`과 헤더 규약을 미러링했다.

```ts
app.get('/calibrate/frame', async (_req, reply) => {
  const latest = deps.calibrator.getLastFrame();
  if (!latest) {
    reply.code(404);
    return { error: 'no frame' };
  }
  reply
    .header('Content-Type', 'image/jpeg')
    .header('Cache-Control', 'no-store')
    .header('X-Cal-Cam', String(latest.camIdx))
    .header('X-Cal-Preset', String(latest.presetIdx));
  return reply.send(latest.jpeg);
});
```

`CalibrateRouteDeps`는 이미 `calibrator`를 보유하고 있어 deps 인터페이스 확장은 없었다. 카메라 재명령 없이 버퍼 반환만 한다(인증/토큰 불요 — `/capture/frame`과 동일하게 관찰용 GET).

### 2.2 웹(`web/app.js`) — 정밀수집 종료 리셋 + 센터라이징 프레임 폴

#### A. 정밀수집 종료 리셋 (`capPoll`, 약 1909행)
활성 → 종료(`wasActive && (done|stopped|error)`) 전환 시, 기존 `showCaptureResult(status)` 직후에 3단계를 추가했다.

```js
state.finalizeOccSnapshot = state.occComputeByKey; // 리셋으로 지워지기 전 얕은 참조 스냅샷.
resetOverlayDisplay();                             // 검출·육면체·점유·마스크 삭제 + 재렌더.
startLive();                                       // 라이브 MJPEG 재연결(얼어붙은 마지막 프레임 대체).
```

- `resetOverlayDisplay()`는 기존 "표시 초기화" 버튼(`#roi-clear`)이 쓰던 기존 함수를 그대로 재사용한다(`detectByKey`/`occComputeByKey`/`occByKey`/`vcuboidByKey` 삭제 + 선택 해제 + 재렌더). 바닥 ROI(`placeRoi`)와 매핑은 건드리지 않아 이후 "최종화" 결과가 정상적으로 렌더된다.
- `capStart`에서 `state.finalizeOccSnapshot = {};`로 초기화(기존 `occByKey`/`occComputeByKey` 리셋 옆에 가산).

**finalize 힌트 보존 문제**: `finalize`는 클라이언트 `occupancy`를 optional 힌트로 받아 서버의 "공간배정 최대매칭"(직전 커밋 `66d9042`)에 참여한다. 종료 리셋이 `occComputeByKey`를 비우면 이 힌트가 사라져 최근 튜닝이 회귀할 수 있으므로, 리셋 직전 스냅샷을 저장해두고 `buildFinalizeOccupancy()`에 폴백을 추가했다.

```js
const src = Object.keys(state.occComputeByKey).length ? state.occComputeByKey : (state.finalizeOccSnapshot || {});
```

#### B. 센터라이징 프레임 폴 (신규 `calFrameTick`/`startCalFramePolling`/`stopCalFramePolling`)
정밀수집의 `capFrameTick`/`capFrameTimer` 패턴을 별도 이름(`calFrameTimer`/`calFrameUrl`)으로 복제했다.

```js
async function calFrameTick() {
  const res = await fetch('/calibrate/frame', { cache: 'no-store' });
  if (!res.ok) return; // 404(버퍼 없음) → 갱신 스킵
  const url = URL.createObjectURL(await res.blob());
  frame.src = url;
  ... // 이전 objectURL revoke, X-Cal-Cam/X-Cal-Preset 로 cal-msg 진행 표기
}
```

- 500ms `setInterval`로 폴링. 정밀수집과 달리 "라운드" 개념이 없어 직전 키 비교 없이 매 폴 갱신한다.
- 오버레이는 쌓지 않는다(순수 프레임 표시만) — 센터라이징은 검출/육면체를 그리지 않으므로 종료 시 `resetOverlayDisplay()`가 불필요하다.
- `calPoll`(진행 폴): `st === 'running'`이면 `startCalFramePolling()`, 아니면 `stopCalFramePolling()`. 활성→완료 전환 시 `startLive()`로 라이브뷰 복귀.

#### C. 상호배타(정밀수집 폴 ↔ 센터라이징 폴)
정밀수집과 센터라이징은 둘 다 카메라를 독점하므로 동시 실행될 수 없다는 전제 하에, 서로의 폴 타이머를 방어적으로 정리한다.

```js
function startCapFramePolling() {
  if (capFrameTimer) return;
  stopCalFramePolling(); // 신규
  stopLive();
  ...
}
function startCalFramePolling() {
  if (calFrameTimer) return;
  stopCapFramePolling(); // 신규
  stopLive();
  ...
}
```

`captureActive()`도 `calFrameTimer`를 포함하도록 확장해, 센터라이징 중에도 연결 폴(`connectionTick`)이 카메라와 경합하지 않게 했다.

```js
function captureActive() {
  return capFrameTimer !== null || calFrameTimer !== null;
}
```

## 3. 핵심 불변식: 추가 카메라 패킷 0

이번 변경 전체를 관통하는 제약이며, 3계층에서 각각 코드로 보장·검증되었다.

| 계층 | 보장 방법 |
|---|---|
| `PlatePtz.captureAndDetect` | `onFrame`은 기존 `requestImage` 결과(`cap.jpg`)를 그대로 넘길 뿐, 새 캡처를 만들지 않는다. |
| `PtzCalibrator` | `lastFrame` 버퍼는 `onFrame` 콜백이 단순 대입(`this.lastFrame = {...}`)할 뿐, 카메라를 호출하지 않는다. |
| `GET /calibrate/frame` | `deps.calibrator.getLastFrame()` 버퍼 반환만 — 라우트 핸들러는 `camera.requestImage`를 호출하지 않는다. |

3초 콘솔로그의 정체: `PlatePtz`가 센터링 반복(pan/tilt 조정 → 재캡처)마다 남기는 **잡 자체의 캡처 주기 로그**다. 웹 폴링(500ms `GET /calibrate/frame`)은 이 로그 빈도에 전혀 영향을 주지 않는다 — 폴링은 버퍼를 읽을 뿐이고, 잡의 캡처 빈도는 `PlatePtz`의 반복 알고리즘(수렴 판정·재시도 간격)이 독립적으로 결정한다.

## 4. 입출력·REST 계약

### 신규: `GET /calibrate/frame`
- **요청**: 파라미터 없음.
- **응답(버퍼 있음, 200)**: `Content-Type: image/jpeg`, `Cache-Control: no-store`, `X-Cal-Cam: <camIdx>`, `X-Cal-Preset: <presetIdx>`, body = 최신 캡처 JPEG 그대로.
- **응답(버퍼 없음, 404)**: `{ "error": "no frame" }`.
- **부작용**: 없음(카메라 재명령 없음). `/capture/frame`(기존)과 대칭 설계.

### `PtzCalibrator.getLastFrame()`
```ts
getLastFrame(): { jpeg: Buffer; camIdx: number; presetIdx: number } | undefined
```
잡 시작 전 `undefined`, 첫 캡처 이후 갱신, 잡 종료 후에도 마지막 값 유지.

### `PlatePtzDeps.onFrame`
```ts
onFrame?: (jpeg: Buffer, camIdx: number, presetIdx: number) => void
```
옵셔널 훅. 미지정 시 기존 동작과 완전히 동일(부작용 없음) — 하위 호환.

기존 `/capture/finalize`, `/calibrate/status`, `/calibrate/result` 등 다른 REST 계약은 이번 변경으로 전혀 바뀌지 않았다.

## 5. 영향도 분석

### 5.1 영향 모듈·의존성 방향
```
web/app.js (calFrameTick, capPoll 리셋)
  └─ GET /calibrate/frame (src/api/calibrateRoutes.ts)
       └─ PtzCalibrator.getLastFrame() (src/calibrate/PtzCalibrator.ts)
            └─ onFrame 클로저 ← PlatePtz.onFrame 훅 (src/calibrate/platePtz.ts)
                 └─ ICameraClient.requestImage() (기존, 미변경)
```
의존 방향은 라우트 → calibrator → PlatePtz 한 방향이며, 역방향 결합(PlatePtz가 라우트/웹을 아는 것)은 없다.

### 5.2 기존 기능 영향 — 회귀 0으로 설계·검증됨
- **정밀수집(capFrame 경로)**: `capFrameTick`/`/capture/frame`/서버측 `finalize` 로직은 **미변경**. 웹 리셋 로직은 종료 분기에 **가산**만 되었고, `buildFinalizeOccupancy()`는 기존 `occComputeByKey`가 비어있을 때만 스냅샷으로 **폴백**한다(정상 흐름에서는 기존 경로 그대로).
- **PtzCalibrator 기존 계약**: 개별 슬롯 실패 흡수, JSON 정본 저장, `centering_slot` DB 저장 등은 `onFrame`이 순수 가산이라 불변.
- **`calibrateSlot`이 호출하는 `makePlatePtz({...})` 시그니처**: 변경 없음(`onFrame`은 기본 팩토리 클로저 내부에만 존재).

### 5.3 잠재 리스크 · 주의점
- **정밀수집 ↔ 센터라이징 동시 실행 전제**: 두 기능이 카메라를 동시에 쓰지 않는다는 가정 위에 상호배타 폴 타이머(`stopCalFramePolling()`/`stopCapFramePolling()` 상호 호출)로 방어했다. 이 전제가 향후 깨지면(예: 두 기능을 병렬 실행하는 요구가 생기면) 폴 타이머가 서로를 강제 종료시키는 부작용이 생긴다 — **확인 필요 항목**(현재 UI/운영상 동시 실행 경로 자체가 없음을 설계·구현 문서가 전제로 명시).
- **`onFrame` 콜백 예외**: 현재 콜백은 단순 필드 대입(`this.lastFrame = {...}`)이라 `throw`할 코드 경로가 없다. 향후 `onFrame`에 더 복잡한 로직(예: 파일 쓰기)이 추가되면 `captureAndDetect` 호출 스택 안에서 예외가 전파되어 센터라이징 잡 자체를 실패시킬 수 있으므로 주의가 필요하다.
- **`/calibrate/frame` 인증 없음**: `/capture/frame`과 동일하게 토큰 없는 관찰용 GET이다. 기존 라우트와 보안 수준이 동일하므로 신규 리스크는 아니지만, 향후 인증 체계가 도입되면 두 라우트를 함께 검토해야 한다.
- **`finalizeOccSnapshot`은 얕은 참조 스냅샷**: `state.occComputeByKey`를 얕게 복사(사실은 참조 대입)한 것이라, 스냅샷 저장 이후 원본 객체가 in-place로 변형되면 스냅샷도 함께 바뀔 수 있다. 현재 코드 경로상 `resetOverlayDisplay()`가 `occComputeByKey`를 **새 객체로 재할당**(`state.occComputeByKey = {}`)하므로 문제가 없으나, 향후 in-place mutate 코드가 추가되면 재검토가 필요하다.

### 5.4 config·어셈블리 정의 파일 변경 여부
없음 — 이번 변경은 `src/`·`web/`의 순수 코드 가산이며, `config/*.json`이나 어셈블리 정의 파일은 건드리지 않았다.

### 5.5 영향받는 구체 파일 목록

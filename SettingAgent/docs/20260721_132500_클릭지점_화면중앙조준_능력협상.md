# 클릭지점 화면중앙 조준 — 능력 협상(방안 C) 재도입

작성: 2026-07-21 / 브랜치: `feat/click-center-pantilt` / 실행모드: B(goal/loop)
관련 산출물: `_workspace/00_goal.md`(목표) · `01_architect_plan.md`(설계) · `02_developer_changes.md`(구현) · `03_qa_report.md`(검증) · 본 문서(문서화) · `04_doc_impact.md`(영향도)

## 1. 배경 — 이전 시도와 무엇이 다른가

같은 날(2026-07-21) 오전, "개별 center" 콤보를 **클릭한 지점 자체**를 화면중앙으로 보내는 기능으로
재구현했다가(`PointAimer` + `patchTrack.ts` sharp NCC 폐루프) 마스터 실사용에서 전혀 동작하지 않아
전량 폐기·되돌렸다(`docs/20260721_120000_클릭지점중앙조준_폐기_되돌리기.md`). 그 결과 "개별 center"는
원래 설계(클릭 **최근접 번호판** center, `mode:'plate'`)로 복귀했었다.

이번 작업은 마스터 지시로 **"개별 center"의 정의를 다시 클릭 지점 기준으로 재정의**한 것이며,
폐기된 접근과는 구조가 근본적으로 다르다.

| 항목 | 오전 폐기안 (`PointAimer`) | 이번 안 (`aimPointToCenter`) |
|---|---|---|
| 제어 방식 | **폐루프**(patch-NCC로 이동 후 재관측→재보정 반복) | **오픈루프 1샷**(기하 계산 → move 1회, 끝) |
| 검출 의존 | patch 이미지 매칭(sharp NCC) 반복 | **없음**(LPD·NCC·patch 어느 것도 호출 0회) |
| 신규 수학 | `patchTrack.ts` 독자 알고리즘 | **없음** — 기존 `PtzCalibrator.preAimPtz()`가 이미 하던
"오차→게인→P제어 1스텝" 기하를 함수로 추출해 재사용(`aimPtzForPoint`) |
| 실카메라 경로 | 미고려 | 휴컴스 네이티브 `ptz_centering setcenter type=point` 능력 협상으로 우선 사용 |
| 결과 | 실사용 미동작 → 전량 삭제 | 라이브 실측 3케이스 이론값과 소수 3자리 일치(§5) |

핵심 차이는 "무엇을 반복해서 맞출 것인가"가 아니라 **"애초에 반복이 없다"**는 것이다.
오전 안은 관측(patch 매칭)이 실패하면 전체가 멈췄다. 이번 안은 관측을 아예 경로에 넣지 않고,
이미 검증된 게인 기반 결정형 계산 한 번으로 끝낸다 — pre-aim(선조준)이 배치 센터라이징에서
이미 매 슬롯 이렇게 동작해왔던 것과 동일한 신뢰 기반이다.

## 2. 설계 방안 3가지와 선택 근거

설계자가 검토한 세 방안(`01_architect_plan.md` 원문 기준 요약):

- **A. 장비 네이티브 단독** — 휴컴스 `ptz_centering.cgi/setcenter type=point`만 사용.
  장비가 스스로 정밀 계산하므로 정확도는 최선이나, **현재 선택 런타임이 시뮬**
  (`cameraRuntime.selectedCameraId = "simulator-1"`, Unity RPC)이라 **오늘 라이브 동작확인이 불가능**하다.
  실카메라(`real-camera-1`)가 런타임에 선택되어야만 검증할 수 있는데 오늘은 선택되어 있지 않다.
- **B. 기하 오픈루프 단독** — `aimPtzForPoint`(게인 기반 1스텝) 계산만 사용.
  시뮬에서 오늘 바로 검증 가능하지만, 실카메라 전용 네이티브 정밀 센터링 능력(요구사항 2)을 버리게 된다.
- **C. 하이브리드(능력 협상)** — 소스가 `centerOnPoint`를 지원하면 그것에 위임(`mode:'native'`),
  아니면 기하로 폴백(`mode:'geometric'`). **채택.**

선택 근거는 A 단독의 결정적 제약(시뮬 런타임에서 A는 오늘 검증 불가)과 B 단독의 요구사항 미달
(휴컴스 네이티브 함수를 실카메라 경로에서 실제로 쓴다는 Requirements 2)을 동시에 만족하는 유일한
방안이 C이기 때문이다. C는 시뮬에서 즉시 라이브 검증 가능(B 경로)하면서, 실카메라가 선택되면
코드 변경 없이 자동으로 네이티브 경로(A 경로)로 전환된다.

## 3. 기하 모델

`src/calibrate/controlMath.ts`의 신규 순수함수 `aimPtzForPoint(point, base, gain, maxStepDeg)`:

```
err = (point.x - 0.5, point.y - 0.5)          // 화면중앙 기준 오차, x>0=우측 y>0=하단
g   = scaleGainForZoom(gain, base.zoom)        // 게인 ∝ 1/zoom (zoomRef=1 기준 스케일)
(pan, tilt) = panTiltCorrection(err, g, base.pan, base.tilt, maxStepDeg)  // P제어 1스텝, ±maxStepDeg 클램프
반환 = { pan, tilt, zoom: base.zoom }          // zoom 은 입력 base.zoom 그대로 — 불변
```

- 게인 실측치: `fallbackGainPanDeg = -62`, `fallbackGainTiltDeg = -35.5` (@`zoomRef=1`, cam1 실측).
  게인은 zoom에 반비례(`scaleGainForZoom`)하므로 배율이 커질수록(줌인) 같은 오차에 대한 회전각 보정이 작아진다.
- `PREAIM_MAX_STEP`으로 1스텝 이동량을 클램프한다(대각선 이동 폭주 방지). 이 값은 기존 pre-aim(배치 센터라이징
  선조준)이 쓰던 상수를 그대로 재사용한다.
- 오픈루프이므로 이 계산은 **1회만** 실행되고 결과를 바로 `camera.move()`에 전달한다 — 이동 후 재관측·재보정 없음.

## 4. 변경 클래스·함수 상세

### 4.1 `src/calibrate/controlMath.ts` — `aimPtzForPoint()` (신규, 순수함수)
`point: {x,y}`(정규화 0~1), `base: Ptz`, `gain: {gainPan,gainTilt,zoomRef}`, `maxStepDeg: number`를 받아
`Ptz`를 반환한다. 내부는 §3 기하 그대로. `PtzCalibrator.preAimPtz()`의 인라인 계산(`scaleGainForZoom` +
`panTiltCorrection` 직접 호출)을 이 함수 호출로 치환했다 — "번호판 중심이든 클릭점이든 화면상 오차를
0으로 만드는 수학은 같다"는 판단에 따른 중복 제거이며, 동작은 바뀌지 않는다(기존 pre-aim 회귀 테스트가
그대로 통과하는 것으로 확인).

### 4.2 `src/calibrate/PtzCalibrator.ts` — `aimPointToCenter()` (신규, 가산)
```ts
async aimPointToCenter(camIdx, presetIdx, point):
  Promise<{ ok, ptz, plateWidth: null, mode: 'native'|'geometric', reason?: string }>
```
동작 순서(구현 시점 최종본 — QA 발견 §4-D 반영 후):
1. 상호배타 가드: `state === 'running'`(배치 진행 중) → `throw 'calibrate already running'`.
   `pointBusy` → `throw 'point centering busy'`. 기존 `centerOnPoint`(번호판 경로)와 **동일한 락**을 재사용한다.
2. **네이티브 분기를 먼저 판정한다**: `this.camera.centerOnPoint`가 존재(truthy)하면 즉시 위임하고
   `mode:'native'`로 반환한다 — 이 경우 현재 PTZ 조회를 하지 않는다(장비가 스스로 계산하므로 불필요).
3. 네이티브가 없으면 그때 `currentPtzFor()`로 기준 PTZ를 얻는다. **기준은 프리셋 base가 아니라 "지금
   장비의 실제 PTZ"**다 — 클릭은 "지금 보이는 화면" 기준이므로 프리셋 저장값을 쓰면 어긋난다.
   `camera.getPtz()` 실패 시 `startPtzFor()`(프리셋 PTZ) 폴백 + `logger.warn`(조용한 강등 금지).
4. `aimPtzForPoint(point, cur, gainRef(), PREAIM_MAX_STEP)`로 절대 pan/tilt를 계산하고 `camera.move()` 1회
   호출 → `mode:'geometric'`. `move()`가 `false`를 반환하면 `reason:'move_failed'`를 함께 반환한다(§6-B).
5. 저장 호출 0(`writer`/`upsertSlotCentering`/`saveSnapshot` 미호출), 검출(LPD) 호출 0, `plateWidth: null`.
6. `finally { pointBusy = false }`.

보조 메서드:
- `currentPtzFor(camIdx, presetIdx)`: `getPtz` 실패 시 `startPtzFor` 폴백 + warn. 네이티브 분기에서는
  호출되지 않는다(위 2번 순서 조정).
- `gainRef()`: `preAimPtz`와 `aimPointToCenter`가 공유하는 폴백 게인 리터럴 헬퍼(설계서에는 없었으나
  구현자가 중복 제거 목적으로 추출 — 값·동작 동일).

### 4.3 `ICameraClient`/`CameraSource` 옵셔널 `centerOnPoint` — 능력 협상 배선
- `src/clients/CameraClient.ts` `ICameraClient` 인터페이스에 `centerOnPoint?(camIdx, point): Promise<Ptz>` 추가.
- `src/viewer/CameraSource.ts` `CameraSource` 인터페이스에 동일 시그니처 추가.
- 둘 다 **옵셔널**(`?`)이라 TypeScript 구조적 타이핑상 기존 구현체는 이 메서드를 정의하지 않아도
  인터페이스를 계속 만족한다 — 컴파일 영향 없음(`04_doc_impact.md` §1에서 5개 구현체 확인).

### 4.4 `CameraSourceClient` — 조건부 할당
`src/clients/CameraSourceClient.ts` 생성자에서:
```ts
if (source.centerOnPoint) this.centerOnPoint = (camIdx, point) => source.centerOnPoint!(camIdx, point);
```
클래스 필드로 `centerOnPoint?: (...) => Promise<Ptz>;`를 선언해두고, 생성자가 **소스가 지원할 때만**
실제 함수를 채운다. 무조건 메서드를 정의하면 시뮬 소스도 "지원"으로 보여 능력 판정이 무너지므로
설계서(§1-c)가 명시적으로 요구한 패턴이다.

**QA가 발견한 지뢰(§6-A)**: `tsconfig.json`의 `target: "ES2022"`는 `useDefineForClassFields`를 기본 `true`로
켠다. 그 결과 필드 선언 자체가 생성자 본문 실행 전에 프로퍼티를 `undefined`로 *정의*해버려서,
`'centerOnPoint' in client`는 미지원(시뮬) 소스에서도 `true`가 된다. 실제 판정 로직
(`PtzCalibrator`의 `const native = this.camera.centerOnPoint; if (native)`)은 **값의 truthy 여부**로
판정하므로 오작동하지 않지만, 향후 `in`/`Object.keys`/스프레드로 능력을 판정하는 코드가 추가되면
시뮬을 네이티브로 오판할 수 있다. 이 계약은 `test/cameraSourceClient.test.ts`에 주석과 함께 고정했다.

### 4.5 `src/viewer/RealPtzSource.ts` — 휴컴스 네이티브 위임
```ts
async centerOnPoint(_camera, point: {x,y}): Promise<Ptz> {
  await this.client.centerPtz({
    type: 'point',
    pointX: Math.round(clamp01(point.x) * 1920),
    pointY: Math.round(clamp01(point.y) * 1080),
    speed: 50,
  });
  return this.currentPtz();   // setcenter 응답에는 PTZ echo가 없어 장비 재조회로 확정
}
```
좌표 기준은 휴컴스 HTTP API(V1.22) 사양 고정값 `1920×1080`이며, 실제 스트림 해상도와 다를 수 있다
(§7 한계). `clamp01`로 0~1 범위를 강제한 뒤 픽셀로 변환해 `HucomsValidationError`(장비측 범위 검증)를
사전 차단한다.

### 4.6 `src/api/calibrateRoutes.ts` — `mode:'point'` 라우팅
- zod enum이 `['plate', 'plate-zoom']` → `['point', 'plate', 'plate-zoom']`로 확장.
- `mode === 'point'`이면 `calibrator.aimPointToCenter(cam, preset, point)`로 위임하고
  `{ ok, ptz, plateWidth: null, mode, ...(reason ? {reason} : {}) }`를 200으로 반환.
- 그 외(`plate`/`plate-zoom`/미지정)는 **기존 `centerOnPoint`(번호판) 경로 그대로** — 코드 한 줄도
  건드리지 않았다.
- 409(`running`/`busy`)·400(zod 검증 실패) 매핑은 기존 로직 재사용, 변경 없음.

### 4.7 `web/app.js` / `web/index.html`
- 오버레이 클릭 게이트: `clickMode === 'center'` → `mode = 'point'`(기존 `'plate'`에서 변경).
  `clickMode === 'center-zoom'` → `mode = 'plate-zoom'`(변경 없음).
- 진행 메시지를 "클릭 지점을 화면 중앙으로 이동 중…"으로 교체.
- `point` 모드는 `/calibrate/frame`이 캡처하는 진행 프레임이 갱신되지 않으므로(검출을 안 하니 캡처도 없음)
  `startCalFramePolling()`을 **생략**하고 라이브 스트림을 그대로 유지한다. 대칭으로, 종료 시 호출하던
  `startLive()`도 `point`에서는 생략한다(설계서에 없던 구현자 판단 — `stopLive()`를 부른 적이 없는데
  `startLive()`를 부르면 멀쩡한 스트림을 불필요하게 재연결하기 때문).
- `index.html`의 콤보 라벨/`title` 문구를 새 시맨틱으로 갱신("클릭 지점을 화면 중앙으로(pan/tilt만·zoom
  불변·검출 없음)").
- `mode` 필드(native/geometric)는 200 응답에 실려 오지만 UI는 이를 읽지 않는다 — 미반영(§6-C, 범위 밖).

## 5. 라이브 검증 기록 (리더 실측)

시뮬 카메라 `cam1 preset1`(런타임 `simulator-1`)에서 3케이스 확인. 3케이스 모두 `mode:'geometric'`이며
`zoom = 1.6934098` 불변.

| # | 조건 | 클릭(정규화) | 응답/관측 | 비고 |
|---|---|---|---|---|
| 1 | 기준 PTZ = 진입 시점 현재값 | (0.117, 0.690) | 관측 (0.492, 0.483) | 중앙 근접, zoom 불변 |
| 2 | 기준 PTZ = 진입 시점 현재값 | (0.943, 0.479) | 관측 (0.506, 0.479) | 좌우 반대극단, 중앙 근접, zoom 불변 |
| 3 | 기준 PTZ = (24.5538, 9.881) 고정 | (0.20, 0.30) | 응답 ptz (13.570, 5.688) | **이론값과 소수 3자리까지 일치** |

케이스 3 검산: `err = (0.20-0.5, 0.30-0.5) = (-0.30, -0.20)`, `g = scaleGainForZoom({-62,-35.5,zoomRef:1}, 1.6934098)`
→ `pan = 24.5538 + err.x*g.gainPan(클램프 내)`, `tilt = 9.881 + err.y*g.gainTilt(클램프 내)`가 응답 `(13.570, 5.688)`과
소수 3자리까지 일치했다(`aimPtzForPoint`가 `panTiltCorrection`을 통해 계산하는 값과 부합). 이는 §3 기하
모델이 문서상 서술이 아니라 실측으로 재확인됐음을 뜻한다.

회귀 확인: `mode:'plate'`(번호판 center) 발화 시 200 정상 응답, 잘못된 `mode` 값(zod enum 밖) 발화 시 400.

## 8. 검증 한계 (은닉 금지)

1. **실카메라 네이티브 경로 라이브 미검증.** 장비(192.168.0.153)가 런타임에 선택되어 있지 않다
   (`cameraRuntime.selectedCameraId = "simulator-1"`). `RealPtzSource.centerOnPoint`는 모킹된 유닛 테스트와
   실 HTTP 서버를 상대로 한 **와이어 형식**(`GET .../ptz_centering.cgi?action=setcenter&type=point&...`)까지만
   확인했다. 실제 장비가 `setcenter`로 화면을 중앙에 맞추는지, 응답 지연, 이동 완료 전 PTZ 조회 시 과거값이
   반환될 가능성은 전혀 확인하지 못했다.
2. **1920×1080 픽셀 기준은 휴컴스 사양 고정값이며 실측 대조 없음.** 실제 스트림 해상도가 다를 때의 좌표
   오차는 장비 없이는 확인 불가.
3. **게인(`-62`/`-35.5` @zoomRef=1)은 시뮬 cam1 실측치이며 타 카메라 일반화 미검증.** 다른 카메라·프리셋·
   큰 zoom에서의 정확도는 확인되지 않았다.
4. **브라우저 실제 마우스 클릭 경로는 REST 직접 호출(§5)로만 확인했다.** `overlay.mousedown` →
   `calPointCenter` → `fetch`로 이어지는 DOM 이벤트 체인 자체는 vitest 대상이 아니며, 라이브 실측도
   REST 발화 기준이다.

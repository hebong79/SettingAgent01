## 1. 개요·목적

마스터 요구 3가지:

1. 카메라를 LPD 번호판 OBB **중심점**으로 pan/tilt 이동시킬 것.
2. **zoom** 으로 OBB 가로가 **화면 가로의 약 20%** 가 되게 할 것.
3. 이 둘을 **하나의 모듈/클래스**로 관리하되, 두 작업은 **완전히 분리**되어 각각 단독으로 호출 가능할 것.

기존 `PtzCalibrator.calibrateSlot()` 에 센터링(A단계)·줌(B단계) 폐루프가 이미 구현되어 있었으나, 잡 상태머신·`Repository`(setup_artifact 저장)·`slot_ptz.json` writer·LLM 자문에 결박되어 단독 호출이 불가능했다. 이번 작업의 본질은 **이 결박을 풀어 두 폐루프를 독립 공개 메서드로 재조립**하는 것이며, 제어 수식은 `controlMath.ts` 를 그대로 재사용하고 신규 수학은 최소 3개 순수 함수(§3)만 추가했다.

## 2. 공개 API

### 2.1 클래스 — `src/calibrate/platePtz.ts` `export class PlatePtz`

```ts
constructor(deps: PlatePtzDeps, opts?: PlatePtzOpts)

interface PlatePtzDeps {
  camera: ICameraClient;   // REST/RPC/모킹 교체 가능(인터페이스 주입)
  lpd: LpdClient;          // detect(Buffer) → PlateBox[]
  sleep?: (ms: number) => Promise<void>;  // 테스트 0ms 주입
}
```

무상태 — 호출마다 독립. `Repository`·writer·`SetupBrain`(LLM 자문) 의존이 전혀 없다.

### 2.2 `centerOnPlate` — 함수 1(센터링, pan/tilt 만 변경)

```ts
async centerOnPlate(camIdx: number, presetIdx: number, startPtz: Ptz): Promise<PlatePtzResult>
```

- `startPtz` 는 필수 인자다. 시뮬 응답의 pan/tilt/zoom echo(항상 0/0/1)는 신뢰할 수 없어 "현재 PTZ" 를 조회할 수단이 없기 때문에, 명령 PTZ 추적의 시작점을 호출측이 넘긴다. 프리셋 기본값이 필요하면 호출측이 `detectPipeline.resolvePresetPtz` 로 얻는다(이 클래스는 `listCameras` 등에 의존하지 않는다).
- 매 반복 `zoom: startPtz.zoom` 으로 PTZ 를 재구성하므로 **구조적으로 zoom 을 바꿀 수 없다**.

### 2.3 `zoomToPlateWidth` — 함수 2(줌, zoom 주도 + 드리프트 가드)

```ts
async zoomToPlateWidth(camIdx: number, presetIdx: number, startPtz: Ptz): Promise<PlatePtzResult>
```

- 매 반복 중심 오차가 `centerTol` 밖이면 그 반복은 **줌을 보류하고** pan/tilt 1스텝만 재중심한다(§3의 "가드 선행"). 이 재중심은 `centerOnPlate` 를 호출하는 것이 아니다 — 두 메서드는 서로 호출하지 않는다(코드 확인: 각 메서드 본문에 상대 메서드명 부재, 유닛 10·17이 단독 green 확인).

### 2.4 `PlatePtzOpts` — 전 옵션(전부 옵셔널)

| 필드 | 기본값 | 의미 |
|---|---|---|
| `centerTol` | 0.03 | 중심 수렴 판정 허용오차(정규화) |
| `targetPlateWidth` | 0.20 | 목표 폭(화면 가로 대비, §4 정의) |
| `widthTol` | 0.02 | 폭 수렴 허용오차 |
| `maxIterations` | 15 | 반복 상한(두 메서드 공통) |
| `probeStepDeg` | 1.0 | `centerOnPlate` 의 게인 실측용 probe 1스텝(°). r2 확정값 — §5 참조 |
| `maxStepDeg` | 5.0 | 반복당 pan/tilt 최대 이동량(°) |
| `fallbackGainPanDeg` | −62 | zoomRef=1 기준 fallback 게인(pan). probe 실패 시 및 `zoomToPlateWidth` 단독 호출 시 사용 |
| `fallbackGainTiltDeg` | −35.5 | zoomRef=1 기준 fallback 게인(tilt) |
| `settleMs` | 300 | 캡처 전 대기(ms) |
| `plateRoi` | `{x:0.5,y:0.5,w:0,h:0}` | **초기 대상 선정**에만 쓰이는 prior(화면 중앙 최근접). 이후 스텝은 예측 prior(§3)로 전환 |
| `gain` | 없음 | `PtzGain`(zoomRef 포함) — `zoomToPlateWidth` 의 드리프트 가드 게인을 `centerOnPlate` 결과로 체이닝할 때 사용 |
| `matchRadiusNorm` | 0.08 | 예측 prior 로부터 이 거리 초과 매칭은 기각(대상 소실 취급 — 이웃 갈아타기 차단) |
| `maxZoomStepRatio` | 1.5 | 1스텝 zoom 증배 상한(대칭 `[z/1.5, z·1.5]`) |

`PtzGain = { gainPan: number; gainTilt: number; zoomRef: number }` — 게인은 항상 측정 시점 zoom(`zoomRef`)과 함께 다니며, 사용 시점에 `gain(z) = gain(zRef)·zRef/z` 로 스케일한다(§3).

### 2.5 `PlatePtzResult`

| 필드 | 의미 |
|---|---|
| `ok` | 성공 여부 |
| `ptz` | 최종 **명령** PTZ(★ 응답 echo 아님 — 실패 시 복구 재료로도 사용) |
| `plate` | 마지막 검출 OBB(`PlateBox \| null`) |
| `err` | 마지막 중심 오차 `{errX, errY}` |
| `plateWidth` | 마지막 `quadBoundingRect(quad).w`(정규화) |
| `gain` | 실측/사용 게인 + 측정 시점 `zoomRef` — `zoomToPlateWidth` 에 체이닝용 |
| `iterations` | 실행된 반복 횟수 |
| `reason` | `ok=false` 일 때만: `no_plate`(시작부터 미검출) / `plate_lost`(도중 소실·매칭 기각) / `max_iterations`(반복 소진) / `zoom_saturated`(`zoomToPlateWidth` 전용 — zoom 클램프 상한인데 폭 미달) |

전송 계층 오류(`CameraApiError`/`LpdApiError`)는 `reason` 으로 삼키지 않고 **그대로 전파**한다(검출 실패=결과, 인프라 장애=예외라는 기존 철학 유지).

### 2.6 사용 예제

```ts
import { PlatePtz } from './src/calibrate/platePtz.js';

const deps = { camera, lpd }; // ICameraClient, LpdClient 구현체 주입
const ptz = new PlatePtz(deps);

// (a) 센터링만 단독 사용
const centerResult = await ptz.centerOnPlate(1, 1, { pan: 22, tilt: 6.8, zoom: 1.69341 });
if (centerResult.ok) {
  console.log('중심 정렬 완료', centerResult.err, centerResult.ptz);
}

// (b) 줌만 단독 사용(센터링 없이 base 에서 바로 확대)
const zoomOnly = await ptz.zoomToPlateWidth(1, 1, { pan: 22, tilt: 6.8, zoom: 1.69341 });

// (c) 센터링 → 줌 게인 체이닝(권장 조합 — 줌 단독의 fallback 무측정 의존을 회피)
const centered = await ptz.centerOnPlate(1, 1, { pan: 22, tilt: 6.8, zoom: 1.69341 });
if (centered.ok) {
  const ptzWithGain = new PlatePtz(deps, { gain: centered.gain });
  const zoomed = await ptzWithGain.zoomToPlateWidth(1, 1, centered.ptz);
}
```

(a)·(b) 는 각각 단독으로 완결되며 서로를 호출하지 않는다(요구 ③). (c) 는 `centerOnPlate` 가 probe 로 실측한 게인을 `zoomToPlateWidth` 의 드리프트 가드에 넘겨 fallback 무측정 의존(§8 D-2)을 피하는 권장 사용법이다.

## 3. 동작 원리

### 3.1 폐루프(probe 게인 실측 → P 제어) — 오픈루프를 쓰지 않은 이유

`detectMath.vehicleCenterZoomPtz` 같은 오픈루프(FOV 모델) 1샷 방식은 `fovBaseV` 취득에 의존한다. 이 값의 취득 경로(`detectPipeline.loadDetectCfg` 의 지면모델 공동추정)가 실패하면 폴백 상수로 강등되는데, **1샷이라 틀려도 보정 기회가 없다**(과거 `camera.fov` 혼동으로 재중심 30% 미달 사례가 `detectPipeline.ts` 주석에 기록되어 있다). 반면 폐루프는 매 반복 재검출로 수렴을 관측 자체로 확인하며, probe 1회(캡처+검출 1회 추가 비용)로 **부호를 포함한 게인을 실측**하므로 FOV 지식이 불필요하다. 두 메서드 모두 폐루프를 기본으로 채택했다.

### 3.2 예측 prior 대상 추적 — "지금 화면에 있는 것" 이 아니라 "내가 추적하던 대상이 예측상 있을 위치"

화면에 번호판이 여러 개 있으면(실측 6개, 간격 ≈0.15) prior 를 화면 중앙 고정으로 두면 카메라가 움직일 때마다 "지금 중심에 가장 가까운" 번호판이 다른 차로 갈아탄다. PlatePtz 는 **최초 1회만** `plateRoi` 로 대상을 선정하고, 이후 매 스텝은 "직전 관측 중심 + 명령 delta 로 예측한 위치"를 prior 로 갱신한다(`predictPlateCenter`/`predictCenterAfterZoom`, §3.4). 관측 앵커는 "직전 명령"이 아니라 "마지막으로 대상을 실제로 본 시점의 중심+PTZ" 다 — probe 위치는 `ptz` 상태를 전진시키지 않으므로, 앵커를 명령 상태로 잡으면 첫 반복의 예측이 probe 변위만큼 어긋나기 때문이다.

### 3.3 매칭 반경(`matchRadiusNorm`, 기본 0.08)

예측 prior 로부터 이 거리를 초과하는 검출은 매칭 기각(=그 스텝 검출 실패로 취급, `plate_lost` 경로)한다. 실측 번호판 간격 0.15의 절반 근사로, 대상이 한 프레임 검출 누락됐을 때 이웃(0.15 거리)을 조용히 갈아타는 것을 차단한다.

### 3.4 damp 상한(3회)

개선이 정체(improvement < 1e-3)될 때 `dampGain` 으로 게인을 감쇠시키는 기존 로직에 **누적 상한 3회**를 추가했다. 상한이 없으면 정체가 이어지는 한 매 반복 감쇠되어 `0.5^15 ≈ 3e-5` 로 게인이 소멸 → PTZ 정지 → improvement=0 → 영구 damp(회복 불가)에 빠진다. 이것이 r1 라이브 실패 A의 근본 원인이었다(§5).

### 3.5 게인의 zoom 종속 — `gain(z) = gain(zRef)·zRef/z`

게인[°/정규화] ∝ FOV ∝ 1/zoom 이다. 게인은 항상 측정 시점 zoom(`zoomRef`)을 달고 다니며, `zoomToPlateWidth` 는 매 반복 `scaleGainForZoom` 으로 현재 zoom 에 재스케일해서 쓴다. `centerOnPlate` 는 zoom 이 루프 내내 불변이므로 시작 시점에 한 번만 스케일한다.

### 3.6 줌 가드 선행 + 스텝비 클램프

zoom-in 은 FOV 를 좁혀, 번호판이 중심에서 벗어나 있으면 확대할수록 화면 가장자리로 밀려나 소실된다. `zoomToPlateWidth` 는 매 반복에서 **중심 오차가 tol 밖이면 줌을 보류하고** 1스텝 재중심부터 한다(가드 선행). 줌 스텝 자체도 인접 zoom 비를 `maxZoomStepRatio`(기본 1.5)로 대칭 클램프해 1스텝의 급격한 확대가 오차를 같은 배율로 키우는 것을 막는다.

## 4. 핵심 개념 정의 — "화면 가로의 20%"

`targetPlateWidth` 는 `quadBoundingRect(plate.quad).w`(정규화, 이미지 가로 대비 **축정렬 boundingRect 폭**)로 정의한다. 번호판 OBB(회전 사각형)의 **장변 길이가 아니다**.

근거:
- 마스터 문구 "화면 가로의 20%" 는 이미지 가로축 점유율을 뜻하며, 회전 OBB 의 장변은 화면 가로가 아니라 번호판 고유 길이라는 다른 개념이다.
- 기존 시스템 전체(`SlotPtzItem.plateWidth`, `zoomCorrection` 입력, config 기본값 `targetPlateWidth=0.2`)가 이미 이 정의로 통일되어 있다.
- 폐루프 관측값(LPD 재검출 → `quadBoundingRect`)과 동일 정의라 수렴 판정이 자기일관적이다.
- 주차장 번호판 기울기는 대부분 소각(<15°)이라 두 정의의 차는 cosθ 수준(≤3.5%)으로 `widthTol`(0.02=목표의 10%) 안이다.

검증자가 이 정의를 **기울어진 OBB** 로 구분 검증했다(유닛 21) — 기존 픽스처는 전부 축정렬이라 두 정의가 수치상 갈리지 않았고, 이 정의 결정이 모듈을 통해 실제로 구현되어 있는지 이번 케이스 전까지 검증된 적이 없었다. 12° 기울어진 OBB 에서 보고 폭이 `quadBoundingRect(quad).w` 와 12자리 일치하고 OBB 장변과는 실제로 다르며 그 차가 3.5% 이내임이 green 으로 확인되었다.

## 5. 개발 서사 — r0 → r1 → r2 (3라운드)

**r0(최초 구현)**: 유닛 12케이스(14 `it`) green, `npm test` 1655/1655 green. 그러나 **라이브 재검증 전패**: A(centerOnPlate) `max_iterations`, B(체이닝 줌) 발산, C(줌 단독) `plate_lost`.

**r1(1차 원인 규명 — 불완전)**: 라이브 실패 3건에 대응해 ① 예측 prior 추적 ② 게인 zoom 스케일링(`scaleGainForZoom`/`predictPlateCenter`/`predictCenterAfterZoom` 신설) ③ damp 상한 3회 ④ 가드 선행+줌 스텝비 클램프를 추가했다. probe 를 1°→3° 로 키우고 fallback 게인을 `+75/−35` 로 "정합"시켰다고 판단했으나, **이 실측 자체가 오염되어 있었다**. 라이브 재검증에서 B가 다시 `plate_lost` 로 실패했다.

**r2(근본 원인 규명 — 부호 오류)**: 리더가 `diagSweep`(추적 휴리스틱을 배제한 전체 검출 목록의 공통 변위 측정)로 참값을 재확정한 결과, 진짜 게인은 `gainPan≈−36.6~−37.0, gainTilt≈−21.0~−21.1`(@zoom 1.69341, 1°/2°/3° 완전 선형)이었다. r1의 `+75`(pan)는 **부호가 반대**였다.

**핵심 교훈**: 유닛이 green 이어도 모킹 모델의 물리(게인 부호)가 틀리면 라이브를 예측하지 못한다. r1의 모킹 모델(pan 게인 +50)은 fallback(+75)과 부호가 일치해 유닛 전체가 통과했지만, 실물리(−62)와는 정반대였다 — **유닛의 green 이 검증 능력을 갖는 것은 모킹 모델이 실측 물리를 재현할 때뿐**이라는 사실이 이번 3라운드가 남긴 가장 중요한 교훈이다.

**aliasing 자기확증 메커니즘(수치)**: r1 라이브 실패는 다음 경로로 재현되었다.
- 번호판 간격 ≈0.15(절반 0.075).
- probe 3° 의 실제 변위 0.082 는 간격 절반(0.075)을 **초과**(사정권).
- pan 이동으로 base 공백(0.427~0.702)에 신규 검출(미끼)이 등장(pan23 실측 0.540, pan25 실측 0.488).
- 틀린 fallback(+75)이 만든 예측 prior(0.495)가 참 위치보다 미끼(0.488, 거리 0.007)에 더 가까워 오매칭 성공.
- 오매칭으로 산출된 허상 게인 **+49.4**(구현 재현치 +49.77과 0.4% 일치 — 라이브 실패 보고 수치와 정합).
- 이 허상 게인이 가드를 반대 방향으로 밀어 오차가 확대되며 `plate_lost` 로 귀결.

r2는 probe 를 3°→1° 로 되돌렸다(1° 변위 0.027 = 간격 절반의 36%로 안전, 3° 변위 0.082는 사정권이라는 판정 근거는 §2.7). aliasing에 대한 **추가 방어 장치는 도입하지 않았다** — probe 축 분리, 매칭 반경 축소, 게인 부호 위생검사 3가지 후보를 모두 수치로 기각하고(설계 §2.7), fallback을 실측 참값으로 정정하는 것과 probe 1°만으로 충분하다고 판단했다. 이 판단은 실패를 미화한 결정이 아니라 실측 수치(여유 ~90배, ~30배)로 논증된 것이며, 라이브 재검증에서 실제로 확인되었다(§7).

## 9. 한계·미검증 (은닉 없음)

QA 보고(`03_qa_report.md`) R-1~R-4를 그대로 인용한다.

- **R-1(중간 — 문서 작성 후 일부 해소)**: QA 보고 시점의 실증 범위는 cam1 preset1 단일 프리셋·단일 대상뿐이었다. 이후 수동 실행기(`_workspace/plate-ptz/_live/try.mts`)로 **cam1 의 preset2·preset3 을 추가 실증**했다(§9.4). 현재 실증 범위는 **cam1 의 프리셋 3개 전부**다. **여전히 미검증**: cam2 이상, 실카메라(Hucoms), 야간·우천·원거리. 일반화의 근거는 여전히 cam1 시뮬 1대에 한정된다.
- **R-2(중간)**: fallback 게인(−62/−35.5)은 **cam1 시뮬 카메라 1대의 실측에서 유도된 상수**다. 다른 카메라에서 타당하다는 근거가 없다. 코드 확인 결과 뷰어 계층 `RealPtzSource.toNativePtz` 의 기본 range 설정(단조 증가)은 부호를 보존하지만, `panRange` 를 역순으로 설정하면 부호가 반전될 여지가 있고, `HUCOMS_PTZ_PARAMS` 에는 "실측 보정 필요" 주석이 붙어 있어 실장비 매핑 자체가 미검증임을 코드가 자인한다(단, `PlatePtz` 는 `RealPtzSource` 가 아니라 `RpcCameraClient` 경로만 라이브 검증되었다 — 실카메라를 `PlatePtz` 에 물리는 경로는 아직 존재하지 않는다).
- **R-3(중간, 후속 과제)**: `config/tools.config.json` 의 `fallbackGainPanDeg:+20`/`fallbackGainTiltDeg:+15` 는 실측 부호(−36.6/−21.0)와 반대다. 이번 범위에서 `PtzCalibrator`·config 스키마는 무접촉이며 후속 과제로 남겨졌다(§10 문서화 영향도 참조).
- **R-4(중간)**: 번호판 간격이 0.15보다 좁은 장면에서는 여유가 선형으로 준다. probe 1°는 tilt 축 변위 0.048로 간격 ≈0.095에서 안전 여유(간격 절반 대비)가 소진되며 — **tilt probe 가 pan 보다 먼저 한계에 도달**한다. fallback 구동 시(=`zoomToPlateWidth` 단독)의 안전 여유는 31%로, 간격이 0.08 미만이면 `matchRadiusNorm` 이라는 안전망 자체가 이웃을 배제하지 못한다. 장면 밀도를 자동 감지해 조정하는 장치는 설계상 없다.
- **줌 단독 호출의 비대칭 열화(D-2, 별도 명시)**: `zoomToPlateWidth` 는 probe 를 전혀 하지 않는다. `opts.gain` 없이 단독 호출하면 드리프트 가드 게인은 fallback 에 **100% 무측정 의존**한다. `centerOnPlate` 는 probe 로 자가 교정되어 멀쩡한데, 줌 단독만 조용히 틀어지는 **비대칭 열화**이므로 진단이 어렵다. 라이브 C(줌 단독) PASS 는 "fallback 이 이 카메라의 참값이었기 때문"이며, 단독 경로가 게인을 스스로 알아낸 결과가 아니다. JSDoc(`zoomToPlateWidth`, `PlatePtzOpts.gain`/`fallbackGain*`)에 이 의존이 명시되어 있다(§2.4/§2.5, QA D-2 정정 반영 완료).

**"완벽 통과" 로 요약하지 않는다** — 검증된 범위는 정확히 위와 같으며, 그 밖은 확인되지 않은 상태로 남아 있다.

# SettingAgent 영향도 분석 (1차 구현)

- 작성일: 2026-06-24
- 대상: SettingAgent 1차 구현이 프로젝트 다른 부분에 미치는 영향

---

## 1. 신규/수정 파일

- **전부 신규**(`SettingAgent/` 하위). **기존 코드 수정 0건** → 기존 동작에 직접 영향 없음.
- 생성 위치: `SettingAgent/{config,src,test,docs}`, `package.json`, `tsconfig.json`, `vitest.config.ts`.

## 2. 외부 서비스 의존 (REST, 읽기/호출만)

| 서비스 | 호출 | 영향 |
|--------|------|------|
| Unity 카메라 `:13100` | `/health`, `/req_img`, `/req_move` | 셋업 중 카메라를 **이동·점유**. ActionAgent 런타임 스캔과 동시 구동 시 **PTZ 자원 충돌** 가능 → 셋업은 단독 시간대 권장 |
| da_vpd_api | `/vpd/api/v2/det/imgupload`, `/health` | 추론 부하만 추가. 읽기성 |
| 로컬 LLM `:8000` | (두뇌 활성 시) chat completions | 기본 비활성이라 평시 영향 없음 |

> 모든 호출은 타임아웃·재시도 적용, 실패 시 예외로 격리(셋업 FAILED 전이). 외부 서버를 변경하지 않음.

## 3. 데이터/계약 영향 (다른 에이전트와의 인터페이스)

- **`setup_artifact.json`(Preset/ParkingSlot/GlobalSlotIndex)** 은 **Action/DM 이 읽을 공유 계약**.
  - 스키마 변경 시 → ActionAgent(점유 귀속·센터라이징 대상), DMAgent(슬롯 상태/전역인덱스) **동시 영향**.
  - 대응: `domain/types.ts` 를 추후 `@parkagent/types` 공통 패키지로 추출하여 3개 에이전트가 동일 버전 참조.
- 좌표/인덱스 규약(1-based, 정규화 ROI, 전역 정렬 `cam→preset→position`)은 아키텍처 §7 확정값과 일치 → Action/DM 구현 시 그대로 전제.

## 4. ParkSimMgr 와의 관계

- ParkSimMgr 코드를 **수정하지 않음**. 다만 다음을 **패턴 재사용/이식**:
  - `UnityCameraClient`(req_img/req_move) → `CameraClient` 로 단순 이식.
  - `domain/types.ts`, `config` zod 패턴 → 동형으로 재작성.
- 향후 공통 패키지화 시 ParkSimMgr 와 타입 정합성 유지 필요(현재는 독립 복제이므로 충돌 없음).

## 5. 어셈블리/빌드 영향

- 독립 npm 패키지(`parkagent-setting`). 다른 디렉터리 빌드에 영향 없음.
- `node_modules` 신규 설치(206 패키지). 디스크만 사용.
- esbuild postinstall 스크립트는 수동 실행 필요할 수 있음(tsx/vitest 의존) — 본 환경에서 실행 완료.

## 6. 리스크 / 주의

| 리스크 | 영향 | 완화 |
|--------|------|------|
| 셋업 중 카메라 점유 ↔ Action 스캔 충돌 | PTZ 경합 | 운영 시간대 분리 또는 카메라 락(후속 DM 조율) |
| `setup_artifact.json` 스키마 드리프트 | Action/DM 파손 | 공통 타입 패키지 + 버전관리 |
| VPD 포트(18080)가 실제와 다를 수 있음 | 연동 실패 | `tools.config.json` 으로 교정(코드 변경 불필요) |
| 프리셋 정의 수동 입력(targets) | 운영 번거로움 | mapConfig 자동로딩 후속 구현 |

## 7. 결론

- 기존 코드 **무수정**, 신규 패키지로 격리되어 **현재 시스템 동작에 영향 없음**.
- 유일한 *미래* 영향은 `setup_artifact.json` 공유 계약 → Action/DM 구현 시 이 스키마를 입력으로 설계하면 됨.
- 다음 차수(ActionAgent) 착수 시 `domain/types.ts` 공통화를 먼저 진행하면 계약 안정성 확보.

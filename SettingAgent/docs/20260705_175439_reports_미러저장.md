## 1. 배경 / 요구

정밀수집(capture) 결과를 최종화(finalize)하거나 수동 저장할 때, 기존에는 결과 artifact 를 `save/result_YYYYMMDD_HHMMSS.json`(정본 스냅샷) **한 곳에만** 기록했다. 요구는 여기에 더해 **같은 서버의 로컬 디렉터리 `reports/`** 에도 **동일한 SetupArtifact JSON 을 복사 저장**하는 것.

사용자 확정 사항:
- `reports` = 같은 서버의 **로컬 디렉터리**(원격·네트워크 저장소 아님).
- 저장 내용 = **결과 artifact JSON 복사** — `report_ko` 텍스트가 아니라 `save/` 와 **완전 동일한 SetupArtifact JSON 파일**.
- `save/` 는 권위(정본), `reports/` 는 보조(미러). CLAUDE.md 원칙(단순함·외과적 변경)에 따라 과설계 없이 구현.

## 2. 구현 — 방식 A(SaveStore.save 단일 지점 미러)

"정밀수집 후 결과저장"의 1차 트리거는 `Finalizer.finalize()` 자동 저장이지만, 수동 라우트 `POST /capture/save` 도 동일한 "결과 저장" 행위이며 **두 경로 모두 결국 `SaveStore.save()` 를 호출**한다. 따라서 미러 로직을 `SaveStore.save()` **단일 지점**에 두어 모든 저장 경로를 자동 커버한다(대안 B: Finalizer 만 복사 — 수동 저장 누락·로직 분산으로 기각).

### 2.1 `src/store/SaveStore.ts` — reportsDir 주입 + save() 미러 (핵심)

- **생성자**: `constructor(private saveDir: string)` → `constructor(private saveDir: string, private reportsDir?: string)`. 두 번째 인자 **optional** — 미주입 시 미러 생략(하위호환).
- **save()**: `JSON.stringify(artifact, null, 2)` 를 **1회** 계산해 `json` 으로 공유한 뒤,
  1. `save/{safe}.json` 을 **먼저** 기록(정본·권위, 실패 시 그대로 throw — 기존 계약 불변).
  2. `reportsDir` 가 있으면 `try { mkdirSync(recursive); writeFileSync(reports/{safe}.json, json) } catch { logger.warn(...) }` 로 **best-effort** 미러.
- `safe`(안전화 이름)·`json`(직렬화 문자열) 을 두 경로에 **공통 사용** → 경로 traversal 방어·`.json` 확장자 처리·바이트 동일성이 별도 로직 없이 재사용됨.
- **import 추가**: `import { logger } from '../util/logger.js';`(기존 미사용 → 미러 실패 관측용).
- **`list()` / `load()` 는 불변** — 조회는 `save/` 권위 유지. `reports/` 는 **쓰기 전용 미러**로, 조회 대상이 아니다.

실제 구현(SaveStore.ts:34-49, 검증 완료):
```ts
save(name: string, artifact: SetupArtifact): string {
  const safe = this.sanitizeName(name);
  if (!safe) throw new Error(`invalid save name: ${name}`);
  const json = JSON.stringify(artifact, null, 2);
  mkdirSync(this.saveDir, { recursive: true });
  writeFileSync(join(this.saveDir, `${safe}.json`), json, 'utf-8'); // 정본(권위) — 실패 시 throw.
  if (this.reportsDir) {
    try { // 보조 미러 — 실패는 격리(정본 save/ 는 이미 성공).
      mkdirSync(this.reportsDir, { recursive: true });
      writeFileSync(join(this.reportsDir, `${safe}.json`), json, 'utf-8');
    } catch (err) {
      logger.warn({ err, name: safe }, 'reports 미러 저장 실패(정본 save/ 는 정상)');
    }
  }
  return safe;
}
```

## 3. 입출력 / 계약

- **입력**: `save(name: string, artifact: SetupArtifact)` — 시그니처 불변.
- **출력(파일)**: `save/{safe}.json`(정본, 항상) + `reports/{safe}.json`(미러, reportsDir 주입 시). 두 파일은 **동일 이름·동일 바이트**.
- **REST 계약 불변**: `POST /capture/save`, `GET /capture/saves*`(list/load) 응답 shape 변화 없음. `reports/` 는 쓰기 전용이라 조회 계약에 등장하지 않는다.

### 저장 흐름도
```
finalize (또는 POST /capture/save)
  └─ repo.saveArtifact(...)              // 기존 Repository 저장(불변)
  └─ saveStore.save(name, artifact)
       ├─ save/{name}.json      (정본·권위, 실패 시 throw)
       └─ reports/{name}.json   (미러·best-effort, 실패 시 logger.warn 만)
```

- **무변경으로 커버되는 경로**: Finalizer 자동저장·수동 `POST /capture/save` 모두 동일 saveStore 인스턴스를 재사용하므로 코드 수정 없이 미러가 자동 적용됨. server.ts 도 무변경.
- **save/ 권위·계약 불변**: 정본이 미러보다 먼저·확실히 기록되고, 조회(list/load, GET)는 계속 save/ 기준. save/ 의 어떤 동작도 바뀌지 않음.
- **하위호환**: `reportsDir` 스키마 default → 기존 config 파일 무수정에도 `reports/` 생성. SaveStore 생성자 optional → reportsDir 미주입 경로(테스트 등)도 기존 동작 그대로.
- **런타임 산출물**: 서버 실행 디렉터리 하위에 `reports/` 폴더 신규 생성. 결과 JSON 은 소용량이나 save/ 와 이중 저장으로 **디스크 사용 약 2배**(결과 스냅샷 소용량이라 실무 영향 무시 가능).
- **리스크**: reports 쓰기 실패는 best-effort(try/catch + logger.warn)로 격리 — save/·finalize·라우트 응답을 절대 깨뜨리지 않음. save/ 성공·reports/ 실패 시 두 폴더 내용이 갈릴 수 있으나 설계상 허용(reports 는 보조). 정상 경로에서는 동일 `safe`·동일 `json` 공유로 불일치 없음.
- **성격**: 순수 파일 IO(결정형). 좌표 불변식·1-based 인덱스·MCP 두뇌/도구 경계와 무관.

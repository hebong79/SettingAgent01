## 0. 한 줄 요약

이터레이션 2가 "옆차 확대"의 근본 원인(zoomRange 오설정)을 고쳤고, 마스터 2차 실카 검증에서 **양쪽 끝 먼 차량 조준이 성공**했다. 이터레이션 3은 그 검증에서 드러난 나머지 3건 — ① 36배줌까지 가도 폭이 목표에 못 미칠 때의 처리, ② UI 카메라 컨트롤이 낡은 위치로 되돌아가는 현상, ③ 로그에서 발견된 사다리의 "더 나쁜 상태로 종료" — 을 처리한 것이다. 그 과정에서 ③의 진짜 정체가 **1회성 사고가 아니라 3주기 반복 진동**이며, 원인이 **뷰어 zoom이 광학 배율과 선형이 아니라는 것**임이 밝혀졌고, 이를 해소하는 **모델 무가정 이분 탐색**이 이터레이션 3의 핵심 결과물이다. 마지막으로 구현자가 스스로 신고한 "폭 수렴 성공 출구가 정렬을 재검증하지 않는" 계약의 틈을 무회귀 설계로 닫았다.

**사다리 제어 방정식(게이트 스케일링·기존 배율 사상)은 이번에도 원칙적으로 손대지 않았다** — 유일한 예외는 수정 20으로, 이는 "다음 zoom을 고르는 방법"을 선형 외삽에서 이분 탐색으로 바꾼 것이며, latch·게이트·성공 판정 로직 자체는 무변경이다.

## 1. 서사

### 1.1 마스터 2차 실카 검증
전편(이터레이션 2)의 zoomRange 정정이 주효해 양쪽 끝 먼 차량 조준이 성공했다. 남은 것은 3건이었다.

### 1.2 마스터 직접 요청 — "36배줌 해도 안 되면 거기서 최종위치로"
> "36배줌 해도 번호판이 20% 안되면 거기서 그부분이 최종위치가 되도록."

로그 실패: rung 3, zoom 36, `errX 0.053`, `plateWidth 0.185` → `ok:false`. 원인은 이터레이션 2의 수정 13이 도입한 `saturatedOutcome`이 **정렬을 전제로 요구**했기 때문이다 — `errX 0.053`이 `centerTol 0.03` 밖이라 실패로 떨어졌다. 마스터의 요청은 "정렬이 이미 되어 있으면 인정하라"가 아니라 **"정렬이 안 되어 있으면 만들어서 인정하라"**였다. → 수정 17.

### 1.3 마스터 재보고 — UI 카메라 컨트롤 위치 미보관
로그 확증: `pan=3512/tilt=1184`(센터링 종료 시점) 직후 UI 조작이 `pan=4721/tilt=1116`(센터링 **이전** 값)을 명령으로 내보냈다. 서버는 최신 값을 갖고 있었으므로(계측 필드로 확인) 원인은 브라우저 캐시가 유력하다. 리더가 이 증상을 **장비 명령 로그**로 확정했다는 점이 중요하다 — 코드 정독이 아니라 실제 발화한 명령값의 시계열로 원인을 좁혔다. 이와 별개로 리더는 "UI가 서버 응답에 의존해 자기 상태를 갱신하는" 현재 설계 자체가 취약하다고 판단했다 → 수정 19(이동 직전 장비 실측 기준화. 조회 실패 시 낡은 값으로 조용히 이동하지 않고 차단).

### 1.4 리더가 로그에서 추가 발견 — 사다리가 더 나쁜 상태로 종료
`rung8 z36 폭0.238` → `rung9 z32.5 폭0.102`로 종료. 사다리가 한때 목표에 가까웠던 상태를 지나쳐 더 나쁜 상태에서 멈춘 것이다 → 수정 18(best 지점 복귀).

### 1.5 ★이번 이터레이션 최대 발견 — 3주기 반복 진동, 원인은 zoom의 비선형성

`logs/setting_20260721_183256.log`의 rung 4~9를 다시 보면 1회성이 아니다:

| rung | zoom | plateWidth | sha |
|---|---|---|---|
| 4 | 36 | 0.238 | 290f2462 |
| 5 | 32.481 | **0.102** | b28a3d4f |
| 6 | 36 | 0.240 | 0a90429f |
| 7 | 32.202 | **0.100** | 0ab64c77 |
| 8 | 36 | 0.238 | 6bacdcd2 |
| 9 | 32.503 | **0.102** | bd757505 |

`sha`가 매번 다르므로 "낡은 프레임"이 아니고, `zoomAct`가 명령을 정확히 따라가므로 "장비 미작동"도 아니다. **같은 판이 실제로 그만큼 작아진 것**이다.

같은 로그의 실측 `w/z`(폭/줌)를 계산하면 16배(0.00131) → 36배(0.00661)로 **5배 변화**한다. 특히 zoom 35.153→36(배율 ×1.024)에서 폭이 0.157→0.238(×1.52)로 뛴다. **뷰어 zoom [1,36]은 raw 엔코더 단위의 선형 사상이지, 광학 배율의 선형 사상이 아니다.** 실제 렌즈는 raw 상단 구간에서 배율이 급격히 증가한다.

이 비선형성이 사다리가 쓰던 `zWant = zoom × target/width`(선형 외삽)를 상단에서 완전한 극한 순환으로 만든다:

```
zoom 36,   w 0.238 → zWant = 36 × (0.215/0.238) = 32.5   (선형 예측)
zoom 32.5, w 0.102 (실측)                                 ← 예측을 절반 이하로 배신
        → zWant = 32.5 × (0.215/0.102) = 68.5 → clampZoom 36 → w 0.238 → 반복
```

로그의 3주기와 이 산술이 정확히 일치한다. → 수정 20(**모델 무가정 이분 탐색**).

### 1.6 구현자가 스스로 신고한 계약의 틈과 그 해소
수정 20 구현 중 발견: 사다리의 주 성공 출구(`isWidthConverged`)는 폭 수렴만 보고 **그 순간 중심 정렬을 재검증하지 않는다**. 테스트로 재현한 결과 `errX 0.054`(centerTol 밖)로도 `ok:true`가 나올 수 있었다 — 실무상 rung마다 재중심이 있어 드물지만, 이론적으로 존재하는 구멍이었다. → 수정 21. **무회귀 설계가 핵심**이다: 정렬이 tol 안이면 추가 왕복 0회로 즉시 반환, tol 밖이면 best-effort 재중심 1회를 시도하되 **그 재중심이 실패해도 `ok:true`는 유지**한다("오늘 되던 것이 이 변경으로 내일 안 되면 안 된다"). 잔차는 `centerShortfall` 필드로 정직하게 남긴다.

## 2. 수정 17~21 상세

### 2.1 수정 17 — 장비 줌 상한 지점을 최종 위치로 (`finalizeAtDeviceLimit`)

마스터 요청을 코드로 옮긴 신규 private 메서드 `finalizeAtDeviceLimit()`(`platePtz.ts:930`)이, 상한 도달 두 지점(`clampZoom` 포화 · 수정 11의 `zoomAct` 정체) **모두**에서 아래 순서로 동작한다:

1. latch 실패 → **카메라를 건드리지 않고 실패**(정렬을 만들 대상 자체가 없다 — 금지선).
2. **현재 위치에서 먼저 실측**. 이미 tol 안이면 그 자리가 최종(`attempts:0`).
3. tol 밖이면 **방금 잰 판 중심**으로 마지막 재중심 1회(줌 상한에서도 `setcenter`는 pan/tilt만 움직이므로 동작한다).
4. **재확인 캡처**로 다시 실측 → tol 안이면 성공, 아니면 **실패 유지**("재중심을 시도했으니 됐다"는 인정하지 않는다).

결과·로그에 `recenterAttempts`와 최종 `errX/errY`를 남긴다. `widthShortfall`은 성공하더라도 보존된다(폭 미달 사실을 지우지 않는다).

**구현 중 자체 발견한 결함(중요)**: 최초 구현은 호출 시점에 이미 갖고 있던 `plate`/`err`로 재중심했는데, 이 값은 그 rung 내에서 재중심이 이미 한 차례 나간 뒤라면 **낡은 값**이었다. 낡은 중심으로 다시 밀면 같은 오프셋을 **두 번** 적용해 반대편으로 넘어간다 — 실측으로 `errX 0.06`이던 것이 재중심 후 `−0.06`이 되는 것을 관측했다. 2단계("먼저 실측")를 넣어 해결했다. "최종 위치"라고 보고하려면 그 자리를 직접 재야 한다는 원칙과도 부합한다.

### 2.2 수정 18 — 최선 상태 복귀 (`restoreBest`)

- rung마다 `|plateWidth − targetPlateWidth|`가 최소인 지점을 `best`로 기억한다.
- 종료 시 현재 상태가 최선보다 **`widthTol`보다 더 나쁘면** 최선 지점으로 복귀한다(새 임계값 없이 기존 `widthTol` 재사용).
- 복귀도 **실측으로 확인**한다. 재검출이 실패하면 위치만 복귀하고 폭 값을 지어내지 않는다(warn + `plateWidth:null`).
- 적용 지점: 예산 소진(`max_iterations`)과 `plate_lost`. 결과에 `restoredToBest`를 남긴다.
- **포화 경로(`zoom_saturated`)에는 적용하지 않는다** — 포화는 폭이 목표 미달인 채 도달 가능한 최선인 상태이므로, 복귀해도 아무것도 바뀌지 않는다(근거 있는 생략).

관측 검산: best = rung 4/6/8(0.238, Δ0.023), 종료 = rung 9(0.102, Δ0.113) → 차이 0.090 > `widthTol`(0.015) → 복귀 조건 성립.

### 2.3 수정 19 — UI 이동 기준을 장비 실측으로 (`web/app.js` `moveBasePtz()`)

§1.3의 증상을 신규 헬퍼 `moveBasePtz()`(`app.js:361`)로 처리한다:
- **실카**: 이동 직전 `refreshCurrentPtz({quiet:true})`로 장비 실측을 읽어 그것을 이동 기준으로 삼는다. 캐시된 값이 낡을 구조적 경로 자체가 사라진다(어떤 캐시 원인이든 무관해진다).
- **조회 실패 시 이동하지 않는다** — `null` 반환 + 상태 문구 표시. "낡은 값으로 조용히 점프"가 이번 증상의 본질이므로, 대신 명시적으로 차단한다.
- **시뮬은 기존 경로 유지**: 명령이 곧 상태이고(응답 즉시 반영) 라이브뷰도 `state.ptz` override로 렌더되므로 캐시가 낡을 경로가 없다. 왕복 1회를 추가할 근거가 없다.
- 적용 지점: 방향 버튼(`stepPtz`) · 절대 이동(`resolveAbsPtz`, `app.js:3695`/`3704`).

**지연 트레이드오프**: 실카 버튼 1회당 `GET /viewer/api/ptz` 왕복이 1회 추가된다. 조회 자체는 장비 `getptzfpos` 1회(이동·정착 없음)라 수십~수백ms 수준으로, 뒤이어 나가는 `goptzfpos`+정착 대기(수 초)에 비하면 무시할 수준이지만, 버튼 반응이 즉시가 아니게 되므로 체감될 수 있다. "가끔 엉뚱한 곳으로 점프"가 사라지는 대가로 정확성을 택했다.

### 2.4 ★수정 20 — 선형 외삽 → 괄호(bracket) 이분 탐색

latch 이후 폭 수렴 구간에서만, 실측쌍으로 **괄호**를 유지하고 그 안에서 이분한다(`platePtz.ts:287` `LADDER_BRACKET_MIN_SPAN`, `:748` 부근 로직).

```ts
// 매 검출 rung에서 갱신 — 검출된 rung에서만 채워지므로 latch 전 탐색에는 구조적으로 무영향
if (plateWidth < target) zLo = max(zLo, ptz.zoom);   // 목표를 아래에서 스침
else                     zHi = min(zHi, ptz.zoom);   // 목표를 위에서 스침

if (zLo && zHi && zHi - zLo > EPS) zNext = clampZoom((zLo + zHi) / 2);   // 괄호 안 이분
else                               zNext = /* 기존 외삽 + 대칭 클램프 그대로 */;
```

- **괄호 미형성**(전 구간 목표 미달) → 기존 상승 탐색 그대로. 실카에서 이미 성공한 경로를 건드리지 않았다.
- **괄호 형성** → 이분. 외삽으로 괄호 밖을 벗어나는 일이 없으므로 진동이 **원리적으로 불가능**하다.
- 종료: `widthTol` 수렴(기존 성공 출구) 또는 **괄호 폭이 `LADDER_BRACKET_MIN_SPAN`까지 좁혀짐**(`zoom_resolution_limit`) → 수정 18의 `restoreBest`로 최선 복귀 후 수정 17의 `finalizeAtDeviceLimit`으로 최종 확정.

**왜 이분이 옳은가(모델 가정 제거)**: 선형 외삽은 `width ∝ zoom`을 가정하는데 실측(`w/z`가 5배 변화)이 이를 깬다. 이분 탐색이 요구하는 것은 **"zoom↑ ⇒ width↑"라는 단조성 하나뿐**이며, 이는 물리적으로 항상 보장된다(줌인이 폭을 줄이는 렌즈는 없다). 따라서 광학 곡선의 실제 모양(볼록·오목·3차·구간별 꺾임 등)에 관계없이 수렴이 보장된다.

**`maxZoomStepRatio` 클램프를 괄호 안에서 면제한 근거**: 이 클램프의 존재 이유는 "측정하지 않은 zoom으로 크게 튀어 중심오차를 배율만큼 확대해 대상을 날리는 것"을 막는 것이다(`platePtz.ts:85`). 그런데 괄호의 두 끝은 **이미 측정했고 그 자리에서 대상을 검출한** zoom이므로, 중점은 측정된 구간 내부이고 클램프가 막으려는 위험 자체가 존재하지 않는다. 반대로 걸었을 경우의 해악은 실재한다 — 넓은 괄호(예: 16↔36)의 중점 26이 `1/1.3` 클램프 밖이라 막혀 수렴이 지연되거나 괄호가 좁혀지지 않는다. 괄호 미형성 구간(외삽)에는 클램프를 그대로 유지했다 — 그쪽은 정확히 "측정하지 않은 zoom으로 가는" 경우이기 때문이다.

**`zoom_resolution_limit`을 `zoom_saturated`와 분리한 이유**: 둘은 원인이 다르다. `zoom_saturated`는 장비의 **배율 상한**(clampZoom 포화 또는 zoomAct 정체)에 도달한 경우다. `zoom_resolution_limit`은 목표를 **괄호 안에 두고도**(즉 상단이 아니라 중간 어딘가) 괄호 폭이 해상도(`0.01` 뷰어 단위 ≈ raw 5)까지 좁혀졌지만 tol 안에 못 들어온 경우다. 사유를 뭉개면 마스터가 "더 못 확대함"과 "이미 목표 근처인데 해상도가 부족함"을 구분할 수 없다.

`LADDER_BRACKET_MIN_SPAN = 0.01`(뷰어 배율)의 근거: 뷰어 [1,36] ↔ raw [0,16384]이므로 1 raw ≈ 0.0021 뷰어 단위. 0.01 ≈ 5 raw는 양자화 노이즈보다 확실히 크면서, 실측 최급구간(35.153→36에서 폭 0.157→0.238)에서도 0.01 구간의 폭 변화는 ≈0.001로 `widthTol`(0.015)의 1/15에 불과해 더 좁혀도 판정이 바뀌지 않는다.

### 2.5 수정 21 — 폭 수렴 출구의 정렬 확인 (`finalizeConverged`, 무회귀 설계)

수정 20이 보고에 남겼던 계약의 틈("주 성공 출구는 정렬을 재검증하지 않는다")을 해소하는 신규 private 메서드 `finalizeConverged()`(`platePtz.ts:881`)다. 이번 지시의 핵심 제약은 **"성공을 취소하지 않는 best-effort 확인"**이었다:

| 상황 | 동작 | 결과 |
|---|---|---|
| 이미 tol 안(대부분) | **추가 카메라 왕복 0회**로 즉시 반환 | `ok:true`, 필드 추가 없음 = 기존과 완전히 동일 |
| tol 밖 → 재중심 성공 | 재중심 1회 + 재확인 캡처로 실측 | `ok:true`, `recenterAttempts:1` |
| tol 밖 → 재중심 거절 | warn | **`ok:true` 유지**, `centerShortfall:true`, 잔차는 `err`에 |
| tol 밖 → 재확인에서 대상 소실 | warn | **`ok:true` 유지**, `centerShortfall:true`, 마지막 실측값 유지(지어내지 않음) |

**이 변경으로 실패로 바뀌는 케이스는 하나도 없다.** `ok`는 이 경로에서 항상 `true`다 — 분기는 오직 "정보를 더 싣느냐"뿐이다. 신규 필드 `centerShortfall?: boolean`(수정 13의 `widthShortfall`과 동형의 정직성 관용구)이 "`ok:true`인데 정렬이 `centerTol` 밖으로 남았다"를 표시한다. 로그에는 `errBefore`(보정 전)까지 남겨 개선 여부가 보이게 했다.

수정 18(best 복귀)과는 코드 경로가 **상호 배타**다 — `restoreBest`는 `max_iterations`·`plate_lost` 경로 전용이고, 이 정렬 확인은 폭 수렴 성공 출구 전용이라 동시 실행될 수 없다.

## 6. 미해결·한계 (은닉 금지)

1. **마스터 3차 실카 검증이 아직 수행되지 않았다.** 이 문서의 수정 17~20 인과(§1.5의 진동 산술 포함)는 로그 사후분석과 유닛 테스트(`opticalMock` 실측점 보간 포함)로 확정됐지만, 이분 탐색이 실제 장비에서 진동을 없애는지, `recenterAttempts`/`centerShortfall`이 실카에서 어떻게 발동하는지는 **아직 실측되지 않았다.**
2. **`centerOnPlate`·`zoomToPlateWidth`(배치 경로)는 여전히 실카 미검증**이다. 실카로 배치 센터라이징을 돌리기 전에 별도 검증이 필요하다 — 이번 세 이터레이션 전체가 개별(클릭) 경로에 집중됐다.
3. **`real-camera-2`가 여전히 미실측**이다(이터레이션 2에서 지적된 한계, 이번에도 해소되지 않음).
4. **느리지만 계속 움직이는 장비의 rung 대기**(이터레이션 2 QA S4가 지적) — rung당 최대 15초 대기가 여전히 남아 있으며, 이번 이터레이션은 이 항목을 다루지 않았다.
5. **QA의 독립 재실행이 이 범위(수정 17~21)에 없다**(§5.4) — 반례 재점검(§4.2)·무회귀 고정(§5.2)은 구현자 자체 확인이며, 별도 검증자가 독립적으로 반박을 시도한 기록은 없다.
6. `web/app.js`의 UI 실측(완료 메시지·`moveBasePtz` 체감 지연) — 아무도 육안으로 마스터 화면에서 직접 확인하지 않았다.

### 라이브 재검증 시 반드시 채록해야 할 것 (구현자 인계 그대로)
1. `phase:'ladder'` 로그의 `zLo`/`zHi`/`span`(이분 탐색 동작) — 실카에서 괄호가 실제로 형성·수렴되는지.
2. `recenterAttempts`·`centerShortfall`·`restoredToBest`·최종 `errX/errY`.
3. 클릭 → 완료까지 체감 시간(이분 전환으로 rung 수가 줄어들 것으로 예상되나 실측 미확인).

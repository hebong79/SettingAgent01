## 1. 배경 및 요구

정밀수집(CaptureJob)은 한 라운드 안에서 여러 프리셋(타깃)을 순회하며 각 타깃에서 `camera.move → 캡처 → 검출`을 수행한다. 기존 구현은 타깃 사이에 별도 간격이 없어, 한 사이클이 짧으면(약 0.6초 수준) 연속 프리셋 이동이 **버스트(burst)** 형태로 몰려 발생했다.

요구사항(사용자 확정):

- 라운드 내 **연속 프리셋 이동 사이에 최소 1초 간격**을 둔다(버스트 완화).
- 단, 한 타깃 사이클(이동+캡처+검출)이 **이미 1초 이상 걸렸으면 추가 대기 없이 즉시** 다음 타깃으로 넘어간다.
- 즉 1초는 **최소 간격(floor)**이며 초과분에 패딩을 더하지 않는다: `sleep(max(0, moveIntervalMs - elapsed))`.
- 이동↔캡처 사이의 settle 지연이 아니라 **타깃 간 이동 간격**이다. 이동 후에는 기존과 동일하게 즉시 캡처한다.

## 2. 신규 설정값 — `moveIntervalMs`

### 2.1 정의 (`src/config/toolsConfig.ts` CaptureSchema)

```ts
/**
 * 라운드 내 프리셋 이동 최소 간격(ms, floor). 한 타깃 사이클(이동+캡처+검출)이
 * 이 값보다 짧으면 남은 시간만큼 대기해 연속 이동이 버스트되지 않게 한다.
 * intervalMs(라운드 간)와 구분되는 타깃 간 간격. 0 이면 페이싱 없음(기존 동작).
 */
moveIntervalMs: z.number().int().nonnegative().default(1000),
```

- 타입: 음이 아닌 정수(ms). 기본값 **1000**.
- **`0`**이면 페이싱 자체를 끈다(기존 무간격 동작으로 복귀).
- 반영 위치:
  - `DEFAULT_TOOLS_CONFIG.capture` 기본값 `moveIntervalMs: 1000` (toolsConfig.ts L226)
  - `config/tools.config.json` capture 섹션 `"moveIntervalMs": 1000` (L53) — 운영 가시성 목적. `loadToolsConfig`가 섹션 단위 `{...DEFAULT, ...raw}` 병합으로 기본값을 주입하므로 JSON에 없어도 런타임 기본값은 1000이지만, 운영자가 값을 눈으로 확인·조정할 수 있도록 명기.

### 2.2 `intervalMs`와의 구분 (혼동 주의)

두 설정은 이름이 비슷하지만 **적용 범위가 다르다.**

| 설정 | 범위 | 의미 | 기본값 |
|------|------|------|--------|
| `intervalMs` | 라운드 **간** | 한 라운드(전 프리셋 순회) 종료 후 다음 라운드 시작까지 대기(점유 변화 포착 주기) | 30000 (30초) |
| `moveIntervalMs` | 라운드 **내부**, 타깃 **간** | 라운드 안에서 프리셋 이동 사이 최소 간격(floor) | 1000 (1초) |

`moveIntervalMs`는 라운드 내부에만 작용하며, 라운드 주기(`intervalMs`)와는 독립적이다.

## 3. 페이싱 로직 (`src/capture/CaptureJob.ts` runRound)

타깃 순회를 `for...of`에서 **인덱스 순회**로 교체하여 "마지막 타깃" 판정을 가능하게 했다. 실제 구현(L183~203):

```ts
const targets = this.params.targets;
const moveIntervalMs = this.deps.cfg.moveIntervalMs;
for (let i = 0; i < targets.length; i++) {
  const t = targets[i];
  if (this.currentState() === 'stopping') break;      // (A) 다음 타깃 캡처 전 정지 탈출
  const t0 = this.monotonic();                        // 이동 시작 시점(captureTarget 진입 직전 ≈ move 직전)
  try {
    await this.captureTarget(this.runId, roundIdx, t);
  } catch (e) {
    logger.warn({ err: e, cam: t.camIdx, preset: t.presetIdx }, '캡처 라운드 프리셋 실패(흡수)');
  }
  const isLast = i === targets.length - 1;
  if (moveIntervalMs > 0 && !isLast && this.deps.cfg.moveBeforeCapture && this.currentState() !== 'stopping') {
    const rest = moveIntervalMs - (this.monotonic() - t0);  // floor 잔여
    if (rest > 0) await this.sleep(rest);
  }
}
```

### 3.1 floor 수식

- `elapsed = monotonic()(캡처 후) - t0`. **캡처+검출(VPD/LPD)을 포함한 전체 사이클** 소요.
- `rest = moveIntervalMs - elapsed`. `rest > 0`일 때만 `sleep(rest)`.
- `elapsed >= moveIntervalMs`이면 `rest <= 0` → **sleep 미호출**(초과분 패딩 없음, 즉시 다음 타깃).
- `t0`은 `captureTarget` 진입 직전(= `camera.move` 직전)으로 고정. "이동 간격" 정의에 맞춰 이동 시작 기준으로 측정한다.

### 3.2 게이트 4조건 (모두 만족해야 페이싱)

`if (moveIntervalMs > 0 && !isLast && this.deps.cfg.moveBeforeCapture && this.currentState() !== 'stopping')`

1. **`moveIntervalMs > 0`** — 0이면 페이싱 자체 off.
2. **`!isLast`** — 마지막 타깃 뒤에는 대기하지 않는다. 라운드 종료 후 어차피 `intervalMs`로 대기하므로 중복 지연 방지.
3. **`this.deps.cfg.moveBeforeCapture`** — **실제 이동이 일어날 때만** 페이싱한다. 이동이 없으면(move=off) 적용하지 않는다. (설계서 §3의 기본안은 "move=off에도 적용"이었으나, **리더 확정 A**로 게이트에 `moveBeforeCapture`를 포함하는 것으로 결정. 검증도 이 계약에 맞춰 수행됨 — §5 TC5b.)
4. **`this.currentState() !== 'stopping'`** — 정지 요청 중이면 sleep을 생략하여 정지 즉시반응성을 유지한다.

### 3.3 catch 흡수 타깃도 동일 처리

개별 프리셋 캡처 실패는 경고로 흡수(잡 중단 아님)한다. 흡수된 타깃도 동일한 페이싱 분기를 거치지만, 실패로 elapsed가 이미 크면 `rest <= 0`이 되어 sleep이 생략되므로 무해하다. 별도 분기를 두지 않아 로직을 단순하게 유지했다.

### 3.4 monotonic 시계 주입 (테스트 결정성)

- 기존 `now: () => string`(ISO 문자열)은 간격 산술에 부적합하여, **숫자 단조 시계**를 별도로 주입한다.
- `CaptureJobDeps`에 `monotonic?: () => number;` 추가(옵셔널).
- 생성자: `this.monotonic = deps.monotonic ?? (() => Date.now());`
- **운영 시 무변경**: 주입하지 않으면 기본 `Date.now`를 사용한다.
- **테스트 시 결정적 제어**: 가변 카운터를 주입해 사이클별 elapsed를 결정적으로 지정할 수 있다. `sleep`은 테스트에서 즉시 resolve(mock)되므로 monotonic은 sleep 지속과 무관하게 호출 시점 값만 반환하면 된다.

## 4. 동작 예시

**전제**: `moveIntervalMs = 1000`, `moveBeforeCapture = true`.

### 예 1) 프리셋 5개, 각 사이클이 짧음(0.6초)

| 타깃 | elapsed | isLast | 게이트 | 페이싱 sleep |
|------|---------|--------|--------|--------------|
| 1 | 600ms | X | 통과 | `sleep(400)` |
| 2 | 600ms | X | 통과 | `sleep(400)` |
| 3 | 600ms | X | 통과 | `sleep(400)` |
| 4 | 600ms | X | 통과 | `sleep(400)` |
| 5 | 600ms | O(마지막) | isLast 차단 | — |

→ 타깃 간 이동 간격이 각 1초(0.6s 사이클 + 0.4s 대기)로 맞춰진다. 마지막 타깃 뒤에는 대기 없음.

### 예 2) 사이클이 이미 1초 초과(1.2초)

| 타깃 | elapsed | rest | 페이싱 sleep |
|------|---------|------|--------------|
| 1 | 1200ms | -200 | 미호출(패딩 없음) |

→ 이미 1초를 넘겼으므로 즉시 다음 타깃으로. 추가 대기 0.

### 예 3) `moveIntervalMs = 0` 또는 `moveBeforeCapture = false`

→ 게이트에서 차단되어 어떤 elapsed에서도 페이싱 sleep 미호출(기존 무간격 동작).

**페이싱 sleep 격리 근거**(검증 신뢰성): CaptureJob 내 `sleep()`은 페이싱 전용이다. 라운드 간 대기는 `setTimer(intervalMs)`, checkpoint 경로는 `sleep` 미사용. 따라서 `sleep` spy 호출 수 = 페이싱 호출 수로 그대로 계수 가능. 테스트는 라운드 1개(count=1) + checkpointEvery=99로 라운드 간 대기·checkpoint를 배제해 페이싱만 격리.

### 5.2 수동 동작확인 (미실행 — 누락 명시)

자동 유닛은 `sleep`/`monotonic`을 모킹하므로 **실제 시뮬레이터의 이동 간격 1초는 검증 범위 밖**이다. 실제 SettingAgent + Unity 시뮬레이터 가동 시 아래를 수동 확인해야 한다:

- **로그 `req_move` 타임스탬프 간격 ≈ 1s**: 정밀수집 라운드 내 연속 프리셋 이동(`camera.move → /req_move`) 발화 시각 간격이 ≈1000ms 이상(기존 ~0.6초 버스트 완화)인지 로그로 확인.
- 한 타깃 사이클이 이미 1초↑이면 추가 대기 없이 즉시 다음 타깃으로 넘어가는지(패딩 없음) 로그 간격으로 확인.
- **현재 상태**: 외부 시뮬레이터 미가동으로 본 검증 세션에서 스모크 미수행. 유닛(모킹)만 완료. **이 항목은 "검증됨"이 아니라 "수동 확인 필요"로 남아 있다.**

## 6. 운영 주의사항

- **설정 반영 시점**: `moveIntervalMs`는 `loadToolsConfig`가 기동 시 로드하는 값이다. `config/tools.config.json` 변경 후 **SettingAgent 재시작**이 필요하다(런타임 핫리로드 아님).
- **페이싱 끄기**: `config/tools.config.json`의 `capture.moveIntervalMs`를 `0`으로 설정하면 페이싱이 비활성화되어 기존 무간격 동작으로 복귀한다.
- **move=off 조합**: `moveBeforeCapture: false`이면 게이트 조건 3에 의해 페이싱이 적용되지 않는다. 페이싱은 실제 이동이 일어나는 정밀수집에서만 유효하다.
- **라운드 소요 증가**: 페이싱 적용 시 라운드 총시간이 최대 `(프리셋수 - 1) × moveIntervalMs`만큼 늘어날 수 있다(사이클이 이미 1초↑면 증가 0). 상세는 `_workspace/04_doc_impact.md` 영향도 분석 참조.

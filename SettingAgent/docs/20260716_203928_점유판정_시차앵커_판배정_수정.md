## 2. 1층 — 시차 기하 결함 (plate 중심 → 차량 접지밴드 앵커 교체)

### 2-1. 원인 — 왜 원경일수록 심한가

구 `web/core.js:computeOccupancy`(수정 전)는 plate(번호판) bbox 중심점이 슬롯 바닥 폴리곤 내부(`pointInQuad`)인지만으로 슬롯을 귀속시켰다. 번호판은 차체 전면 지상 ~0.5m 높이에 떠 있고, 바닥 ROI 는 지면 발자국(footprint)이므로 카메라 시점에서 시차(parallax)가 생겨 **plate 중심이 항상 화면 위(원경 방향)로 10~33px 밀린다**(05 진단 §2 실측).

근경 슬롯 폴리곤은 y 두께가 100px 대라 이 오프셋을 흡수하지만, 원경으로 갈수록 폴리곤이 얇아져(27~37px) 오프셋이 슬롯 경계를 넘긴다. 그 결과:
- 열 중간 차량의 plate 는 **이웃(원경 쪽) 슬롯으로 오귀속**(예: p1 실제슬롯5→6, 6→7).
- 열 끝 차량의 plate 는 **받아줄 다음 슬롯이 없어 결과에서 조용히 소실**(예: p1 slot7, p2 slot8, p3 slot17).
- 폴백(`occupancy.js` 구 165행/188행)이 위 두 케이스를 구제하지 못하고 오히려 정상 슬롯의 표시까지 억제했다.

### 2-2. 수정 전/후 매칭 알고리즘

**전**: plate 중심 단일점 `pointInQuad` 매칭 1단계(주 매칭기) → bbox 폴백 2단계.

**후**(`web/occupancy.js` `OccupancyJudge.judge`, 06 설계 §2 그대로 구현):

1. **1단계(주 매칭기, 승격)** — 차량 bbox 하단 25%(`groundBand`, `GROUND_BAND_RATIO=0.25`)와 슬롯 폴리곤의 **겹침 면적비 argmax**(`convexIntersectionArea / bandArea ≥ ON_PLACE_MIN_OVERLAP=0.15`)로 슬롯을 귀속시킨다. 이 기하는 신규 발명이 아니라 기존 2단계 bbox 폴백이 쓰던 것을 1단계로 승격한 것(신규 기하 0줄). 접지밴드는 바닥 ROI 와 **같은 평면**이므로 시차가 원리적으로 0이다.
   - 귀속된 차량이 plate 를 보유하고 `quadCentroid`가 정상(퇴화 아님)이면 `source:'plate'` + `plateQuad`(사다리꼴 축 소스) + `vehicleRect`, plate 없거나 퇴화면 `source:'bbox'` + `vehicleRect`(강등, throw 없음).
2. **2단계(폴백, 비점유 슬롯 한정)** — 1단계로 채워지지 않은 슬롯에 한해, 차량에 귀속 안 된 standalone plate(좌표키로 attached plate 제외)와 임계 미달/경합 패배 차량의 plate 를 기존 `computeOccupancy`(plate 중심 매칭)에 위임한다.

`web/occupancy.js:152-234`(실물 코드) — 위 알고리즘 그대로 확인.

### 2-3. `source` 재정의

수정 전 `source`는 "판정 근거"(plate 매칭이면 `'plate'`, bbox 폴백이면 `'bbox'`)였다. 수정 후에는 **슬롯 귀속을 차량 접지가 전담**하므로, `source`는 **번호 인식 여부**만을 뜻한다:
- `'plate'` = 귀속 차량이 번호판 보유(또는 미귀속 차량 번호판 폴백)
- `'bbox'` = 차량 귀속됐으나 번호판 미인식
- `null` = 빈 슬롯

`web/occupancy.d.ts` 주석이 이 재정의를 반영해 갱신됐다(타입 형태 자체는 무변경). 이는 **파괴적 의미 변경**이며, 구 T7("plate 보유 차량, plate 중심이 폴리곤 밖 → `source:'bbox'`")을 신 T7("동일 입력 → `source:'plate'`+`plateQuad`+`vehicleRect`")로 재봉인했다(`test/occupancyJudge.test.ts:117`). 이것이 바로 열 끝 차량 소실 수정의 본체다.

`computeOccupancy(floorPolygons, plates)` 자체의 시그니처·동작은 **무변경**이며 2단계 폴백기로 재사용된다(06 설계 논점 3 — 소비처 4곳·테스트 3벌 파괴 회피). 단, plate 매칭 시 반환 행에 `plateQuad`가 추가됐다(사다리꼴 축 연결용, 직전 세션 `점유영역 사다리꼴 표시` 기능의 일부 — 본 세션이 새로 만든 것이 아니라 재사용, `web/core.js:471-486` git diff 로 확인).

## 3. 2층 — plate→vehicle 판배정 결함 (전역 그리디 + frontAnchor tie-break)

### 3-1. 잔존 증상 발견과 원인

1층 수정 직후 goal/loop 검증 중 p1 프리셋에서 `overlapPairs=[[5,6]]`(사다리꼴 5·6번이 겹침)이 관측됐다(p2/p3는 정상). 07 구현 §6이 이를 리더에게 즉시 보고했고, 08 진단이 별도로 착수했다.

원인은 `src/setup/plateMatch.ts:35`(+`:28`, **수정 전** 코드)의 판배정 로직이었다:
- 번호판 bbox 가 두 차량 rect 에 **완전 포함**되면 `intersectionArea`가 포화해 **완전 동률**(판별력 0)이 된다.
- 동률을 strict `>` 비교가 **인덱스 순서**로 깨뜨린다(기하 근거 0).
- 승자가 이미 더 큰 판을 보유하면, 패자 판은 **차선 차량으로 넘어가지 않고 통째로 폐기**된다.

p1 실측: plates[4](1559,674)가 veh4·veh6 양쪽에 847.7px²로 완전 동률 → 인덱스 순 veh4 승 → veh4 는 이미 더 큰 판(883.7px²) 보유 → **plates[4] 폐기**. veh6 이 미귀속으로 남아 줌 재시도(`detectPipeline.ts:314`, **rect·중복 검사가 전무한 뷰중심 최근접**)에 들어갔고, 재시도가 veh0 의 판(1348,678)을 18.8px 거리에서 회수(`recovered:true`)했다. 이 오귀속이 사다리꼴 5·6 겹침의 직접 원인이다. p2 도 동일 결함(plates[5] 폐기)이 발현했지만 재시도가 우연히 자기 판을 되찾아 **증상이 은폐**돼 있었다(잠복, 08 진단 §5-2).

### 3-2. 수정 — 전역 그리디 + frontAnchor tie-break

`src/setup/plateMatch.ts:matchPlatesToSlots`(시그니처·반환형 무변경, 내부 알고리즘 교체):

1. 후보쌍 = {(판 p, 슬롯 s) | p 중심 ∈ s.roi}(1차 조건 현행 유지).
2. 전체 후보쌍을 **(겹침면적 내림차순 → frontAnchor 거리 오름차순 → 판 인덱스 → 슬롯 인덱스)** 로 정렬(전순서 comparator, `plateMatch.ts:53`).
3. 정렬 순서대로 순회하며 **판·슬롯 양쪽이 모두 미배정일 때만** 확정(maximal matching = "차선 폴백"의 본체).

**frontAnchor**: 판 중심 ↔ `(rect.x+rect.w/2, rect.y+rect.h×0.62)`의 거리(`FRONT_BIAS=0.62`, 번호판은 차량 전면 하부에 맺힌다는 물리 사실). 겹침이 포화해 판별력을 잃는 지점에서만 개입하는 tie-break 다.

### 3-3. tie-break 후보 5종 실측 비교와 3종 기각 경위(설계 09 §1)

09 설계 단계에서 동결 픽스처(`test/fixtures/occupancyAnchor/detect_cam1_p{1,2,3}.json`)에 5가지 tie-break 후보를 직접 실행(`_qa_tiebreak_09.mjs`)해 비교했다:

| tie-break | p1/p2/p3 정답 | `lpdFilterRegression.test.ts:332` 합성 반례 | 판정 |
|---|---|---|---|
| 인덱스 순 | 전부 정답 | TRUE 통과 | 통과하나 기하 근거 0(자의적) |
| **frontAnchor(채택)** | 전부 정답 | **TRUE 통과** | **유일하게 전 케이스 기하 근거로 통과** |
| 판중심↔rect중심 거리 | 전부 정답 | **NOISE 채택(회귀)** | 기각 |
| rect 면적(작은 쪽) | p1 5/7 실패(veh6←옆차판) | 통과 | **동결 데이터가 직접 기각** |
| 접지 거리(rect하단↔판) | p1 동일 실패 | 통과 | **동결 데이터가 직접 기각** |

`lpdFilterRegression.test.ts:332`(PARKED rect, TRUE/NOISE 완전포함 동률)에서 rect중심 tie-break 는 rect 중심 대칭 가정 탓에 실제 노이즈(NOISE, y=0.29)를 진짜 판(TRUE, y=0.335)보다 가깝다고 오판(0.020<0.025)한다. frontAnchor(cy=0.3412) 는 TRUE 를 8배 차(0.0062 vs 0.0512)로 정확히 선택한다 — "판이 전면 하부에 맺힌다"는 물리 사실이 대칭 가정을 이긴 실측 사례다. 면적/접지거리 계열 2종은 p1 실좌표에서 옆차 판을 채택하는 오답을 실측으로 직접 냈다.

**헝가리안(전역 최적화)은 과설계로 기각**됐다(09 설계 §1) — 목적함수(겹침 합)가 포화 상황에서 퇴화해 tie-break 문제를 해결하지 못하고, 그리디+frontAnchor 가 이미 동결 데이터 전량 정답을 실측 달성했으며, 규모(P,S≤~30)에서 O(n³) 구현의 추가 이득이 없다.

### 3-4. 재시도 중복 가드 (`DUP_EPS=0.03`)

`src/capture/detectPipeline.ts` 줌 재시도 루프에 심층 방어를 병행했다: 역투영 직후(클램프 **전**) 좌표가 이미 배정된 판(base 매칭분 + 이 실행에서 확정된 recovered 분) 중심과 정규화 거리 ≤ `DUP_EPS=0.03` 이면 그 pick 을 기각하고 로그를 남긴 뒤 **다음 zoomFactor로 계속**한다(루프 상한 불변, 카메라 호출 수 증가 없음).

클램프 **전** 좌표로 판정하는 이유는, 클램프가 `v.rect` 로 중심을 당기기 때문에 훔친 판(veh6.rect 내부)과 자기 판이 뭉개져 클램프 후에는 구분이 불가능해지기 때문이다(08 진단 §2-2 6항이 지적한 지점).

ε=0.03 근거(전부 실측): 중복 회수점↔원본 판 거리 0.0100 / 정상 신규 회수점(p1 veh2)↔최근접 배정판 0.1357 / base 판간 최소 간격 0.1102 → 관측 역투영 오차의 3.0배 위, 정상 케이스 최소 거리의 1/4.5 아래. G3 테스트가 정상 신규 회수(과차단 없음)를 대조군으로 봉인한다.

`src/capture/Aggregator.ts:276` 은 동일 규칙을 **독립 복제**한 코드로 이번 수정과 인과가 없어 코드는 건드리지 않고 주석 1줄만 "규칙 이원화 — 별도 이슈"로 정정했다(09 설계 논점 4 — granularity 상이: Aggregator 는 클러스터 대표 단위·차량별 argmax 구조라 그리디 통합 불가).

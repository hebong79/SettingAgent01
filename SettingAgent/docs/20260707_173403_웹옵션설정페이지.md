## 1. 목적·범위

웹 UI에서 SettingAgent의 핵심 설정값을 **조회·수정·저장**할 수 있는 "옵션" 탭을 신설한다. 기존에는 설정 편집 UI·엔드포인트가 전무했고, 설정은 부팅 시 1회 로드(`loadToolsConfig`/`loadLlmConfig`)뿐이었다.

### 편집 대상 값 (화이트리스트)

| 섹션 | 파일 | 편집 가능 키 | 검증 규칙(zod 재사용) |
|------|------|-------------|----------------------|
| `llm` | `config/llm.config.json` | `provider`, `model`, `baseUrl` | provider ∈ `qwen3\|gemma\|claude\|codex\|openai-compatible`, model 비어있지 않음, baseUrl 은 URL |
| `vpd` | `config/tools.config.json` | `endpoint`, `detPath` | endpoint 는 URL, detPath 는 `/` 로 시작 |
| `lpd` | `config/tools.config.json` | `endpoint`, `detPath` | endpoint 는 URL, detPath 는 `/` 로 시작 |

### 범위 제외
- **LPR**: 마스터 확정(설계서 §8 A-2, 실서버 사양 보류)에 따라 config·클라이언트·폼 필드 **전부 미구현**. `SettingsPatchSchema.strict()` 가 `lpr` 섹션을 거부하여 보류 상태와 코드가 정합.
- **핫리로드 미도입**: 파일 쓰기 + "재시작 필요" 안내로 최소화(§7 핵심 주의).
- **API 키 값 편집·노출**: env(`apiKeyEnv`)로만 주입, 웹 편집 대상 아님(§6 보안).

## 2. 신규 파일

### 2-1. `src/config/settingsStore.ts` (결정형 순수 I/O)

두 config 파일에서 편집 대상 값만 추출/부분갱신하는 순수 I/O 모듈. `process.env` 를 절대 접근하지 않아 키 값 노출이 구조적으로 불가능하다.

**export 심볼**

| 심볼 | 종류 | 설명 |
|------|------|------|
| `SettingsPatchSchema` | zod 스키마 | patch 유효성 검사. 기존 스키마를 **pick·partial** 로 재사용(§2-3). 상위 `.strict()` 로 미허용 섹션 거부. |
| `SettingsPatch` | 타입 | `z.infer<typeof SettingsPatchSchema>`. |
| `EditableSettings` | 인터페이스 | `GET /settings` 반환 shape. `{ llm, vpd, lpd }` 각각 편집키 + `apiKeyEnv`(이름). |
| `SettingsPaths` | 인터페이스 | `{ toolsPath, llmPath }`. |
| `DEFAULT_SETTINGS_PATHS` | 상수 | 로더 기본 경로와 동일(`config/tools.config.json`, `config/llm.config.json`). |
| `readEditableSettings(paths)` | 함수 | 두 raw JSON → 편집 대상만 추출. `apiKeyEnv` 는 **env 변수 이름 문자열**만(키 값 아님). |
| `writeEditableSettings(patch, paths)` | 함수 | 허용 키만 in-place 병합 후 기록. **변경된 파일만** 재기록. |

**설계상 비자명한 결정**
- **부분 병합(in-place)**: raw JSON 객체를 읽어 화이트리스트 필드만 교체 후 `JSON.stringify(obj, null, 2)` 로 기록. 전체 스키마 덮어쓰기가 아니므로 `_comment`·`mcp`·`setup`·`capture`·`calibrate`·`camera` 등 **다른 모든 섹션·키·순서가 보존**된다.
- **변경 파일만 기록**: `patch.llm` 만 오면 `llm.config.json` 만 쓰고 `tools.config.json` 은 손대지 않는다(무변경 파일 재포맷 방지 — 바이트 무변경). `mergeSection` 이 실제 변경 여부(`changed`)를 반환해 write 여부를 게이트.
- **화이트리스트 최종 방어**: `mergeSection` 이 `keys` 인자(예: llm=`['provider','model','baseUrl']`)에 있는 키만 교체. 스키마를 우회해 `{llm:{apiKeyEnv:'HACKED'}}` 를 raw 로 밀어넣어도 `apiKeyEnv` 는 화이트리스트 밖이라 무시된다(2중 방어: 스키마 `.strict()` + write 화이트리스트).

### 2-2. `src/api/settingsRoutes.ts` (REST)

`registerCaptureRoutes` 패턴을 따른 가산 라우트. 기존 라우트 불변.

| 메서드 | 경로 | 동작 | 응답 |
|--------|------|------|------|
| GET | `/settings` | `readEditableSettings(paths)` | 편집 대상 값(키 값 미노출, `apiKeyEnv` 이름만) |
| PUT | `/settings` | `SettingsPatchSchema.safeParse` → `writeEditableSettings` | 성공 `{ ok:true, restartRequired:true }` / 검증실패 400 `{ error:'invalid body', detail }`(zod flatten) / 쓰기예외 500 |

- **`restartRequired:true`**: config 가 nodemon watch(src) 밖이라 저장해도 런타임 반영 안 됨을 응답으로 명시(§7).
- **빈 body**: `req.body ?? {}` → 검증 통과(no-op) → 200.

### 2-3. 스키마 재사용 (drift 방지)

```
SettingsPatchSchema = z.object({
  llm: LlmSchema.pick({provider,model,baseUrl}).partial().optional(),
  vpd: VpdSchema.pick({endpoint,detPath}).partial().optional(),
  lpd: LpdSchema.pick({endpoint,detPath}).partial().optional(),
}).strict()
```

기존 config 로더 스키마(`LlmSchema`/`VpdSchema`/`LpdSchema`)를 `pick`·`partial` 로 재사용한다. 따라서 provider enum·URL·detPath(`/` 시작) 규칙이 **config 로더와 자동으로 동일**해진다(별도 검증 규칙 중복·drift 없음). 이 재사용을 위해 세 스키마에 `export` 키워드만 추가(§3).

### 2-4. `web/core.js` `settingsFormErrors(form)` (순수함수)

저장 전 클라이언트 사전검증. 오류 메시지 문자열 배열 반환(정상 시 `[]`).

- 검사: `llm.model` 필수, `llm.baseUrl`/`vpd.endpoint`/`lpd.endpoint` 는 http/https URL, `vpd.detPath`/`lpd.detPath` 는 `/` 시작.
- WHATWG `URL`(브라우저·node 공통 전역)만 사용 — DOM/fetch 미참조로 vitest 직접 import 가능(core/app 분리 컨벤션 준수).
- **의도된 근사**: 프론트는 `http`/`https` 만 허용하나 백엔드 zod `.url()` 은 스킴 무관. 프론트가 더 엄격한 방향이라 오탐 방지용 근사 사전검증이며 **최종 검증은 백엔드**(QA 03-B §6).

## 4. 백엔드 동작 흐름

### GET /settings
1. `readEditableSettings(paths)` → 두 파일 raw JSON 읽기.
2. `llm`/`vpd`/`lpd` 섹션에서 편집키 + `apiKeyEnv`(이름) 추출.
3. 비편집 필드(`temperature`·`maxTokens`·`enabled`·`timeoutMs` 등)·타 섹션은 반환 제외.
4. 200 응답(키 값 부재).

### PUT /settings
1. `SettingsPatchSchema.safeParse(body)` — 실패 시 400 + `detail`(zod flatten).
2. `writeEditableSettings(patch, paths)`:
   - `patch.llm` 있으면 `llm.config.json` 만 부분병합·기록(변경 시).
   - `patch.vpd`/`patch.lpd` 있으면 `tools.config.json` 부분병합·기록(둘 중 하나라도 변경 시).
3. 성공 `{ ok:true, restartRequired:true }` / 쓰기예외 500.

## 5. 프론트 동작 흐름

- **옵션 탭 진입**(`setTab('options')`): analyze 처럼 전체폭(뷰포트·패널 숨김)으로 처리 후 `loadSettings()` 호출.
- **`loadSettings()`**: `GET /settings` → 폼 채움. `apiKeyEnv` 는 **이름 라벨만** 표시(`.opt-keyname`), 키 값 입력란 없음.
- **`saveSettings()`**: 폼 수집 → `settingsFormErrors` 사전검증(오류 시 `#opt-msg` 표기, 전송 중단) → `PUT /settings` → 성공 시 `#opt-restart-banner`(재시작 필요) 노출, 400 시 `detail` 표기.
- **재시작 배너**: 저장 성공 시에만 노출. "config 변경은 서버 재시작 후 반영" 안내.

## 6. API 키 보안

- **GET**: `apiKeyEnv`(env 변수 **이름**)만 반환. `settingsStore` 는 `process.env` 를 절대 접근하지 않아 키 값이 응답 직렬화 어디에도 나타나지 않는다(QA: `process.env.LLM_API_KEY='super-secret'` 주입 상태에서도 응답 바디에 비밀 값 부재 확인).
- **PUT**: 화이트리스트에 `apiKeyEnv` 없음 → 편집 불가. 스키마 우회 raw 주입도 write 화이트리스트가 무시.
- **전제**: 웹이 신뢰 네트워크 내라는 가정. **확인 필요** — 현 `server.ts` 에는 요청 단위 인증 미들웨어(preHandler·API 키 검사)가 **없다**(grep 결과 인증 훅 부재). `/settings` 는 다른 라우트와 동일하게 무인증으로 노출되므로, 신뢰 네트워크 밖 배포 시 별도 인증 계층이 필요하다(범위 밖, 명시).

## 7. ★핵심 주의: 저장 후 서버 재시작 필요 (런타임 반영 안 됨)

- `nodemon.json` 의 `watch:["src"]` 는 `config/` 를 **감시하지 않는다**. 따라서 `PUT /settings` 로 `config/*.json` 을 저장해도 **실행 중 프로세스에는 반영되지 않는다**.
- 설정은 부팅 시 1회 로드(`loadToolsConfig`/`loadLlmConfig`)뿐이므로, VPD/LPD endpoint·LLM provider/model/baseUrl 변경은 **서버 재시작 후에만** 정밀수집·캘리브레이션 런타임에 적용된다.
- 이 사실을 응답 `restartRequired:true` + 웹 재시작 배너로 명시. 핫리로드는 본 구현 범위 밖(설계서 §4-4·§8).

## 9. 영향도 요약

상세 영향도는 `_workspace/04_doc_impact.md`(04-B 섹션) 참조. 핵심:
- `export` 추가는 순수 가산 → 기존 소비자 무영향.
- `/settings` 라우트는 항상 등록되나 기존 라우트·인증과 충돌 없음(가산, 인증 계층 자체가 부재).
- `writeEditableSettings` 파일 쓰기는 nodemon 감시 밖 → **재시작 필요**(런타임 무영향, 의도된 설계).
- tools.config 최초 저장 시 탭→2스페이스 재포맷 가능(설계 허용 부수효과).
- 향후 ①(floor)/②(occupancy)/⑤(LPR) UI 토글·필드 확장 여지(옵션 페이지가 자연스러운 확장 지점).

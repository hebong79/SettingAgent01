# Touring Test — 정밀수집 결과 순회 점검 기능

작성일: 2026-07-23 14:20:42
작업 브랜치: `feature/touring-test` (main에서 분기)
대상: `SettingAgent/web/core.js`, `web/core.d.ts`, `web/app.js`, `web/index.html`

---

## 1. 기능 개요

정밀 수집(Capture) 탭에 **"Touring Test"** 버튼을 추가했다. 이미 저장된 `setup_result.json`(정밀수집 최종 산출물, 읽기 전용 고정본)을 카메라 → 프리셋 → 주차면(slotId) 순으로 순회하며, 각 프리셋의 home PTZ와 각 주차면의 센터링 PTZ로 카메라를 물리 이동시키고(각 위치에서 1초 대기), 순회가 끝나면 완료 팝업(모달)을 띄우는 **점검용 독립 기능**이다.

핵심 설계 목표는 "**완전 독립**" — 이 기능은 정밀수집 파이프라인의 상태(검출 결과, 오버레이, DB)를 일절 읽거나 쓰지 않는다. 순수하게 `setup_result.json`을 읽어 이미 저장된 좌표로 카메라를 다시 돌려보는 리허설/QA 도구다.

## 2. 요구사항 6단계 → 구현 매핑

| # | 요구사항 | 구현 |
|---|---|---|
| 1 | `setup_result.json` 로딩, DB 미저장(읽기 전용) | `runTouringTest()`가 `fetch('/capture/saves/setup_result', {cache:'no-store'})`로만 로딩. 서버 라우트 `GET /capture/saves/:name`(captureRoutes.ts:627)은 `saveStore.load(name)` 결과를 shape 검증 없이 그대로 반환하는 순수 조회 라우트 — DB 쓰기 없음. |
| 2 | 순서: 카메라 → 프리셋 순 | `buildTouringPlan()`이 `camId → presetId → presetSlotIdx → slotId` 순으로 방어적 재정렬 후 그룹(camId:presetId)별로 `preset` 스텝을 1개씩 발행. |
| 3 | 프리셋 home PTZ 이동 후 1초 대기 | preset 스텝에서 `findPresetPtz(state.cameras, camId, presetId)`로 home PTZ를 구해 `move(home)` 호출, PTZ 미제공 시 `gotoPreset()` snapshot 폴백. 이후 `setTimeout 1000ms` 대기. |
| 4 | 프리셋의 주차면(센터링 PTZ)을 slotId 순으로 이동, 각 1초 대기 | slot 스텝(`centering` 값 기반 `ptz`)을 `move(step.ptz)`로 이동, 이후 1초 대기. 정렬 tie-breaker에 slotId 포함해 순서 결정성 보장. |
| 5 | 전체 순회 완료 시 팝업 안내 | 순회 종료 후 `#touring-done-modal`을 표시(`hidden=false`), 프리셋/슬롯 방문 수·스킵 수를 본문에 기재. |
| 6(수용기준) | centering=null 슬롯은 스킵(위장 이동 금지) + 카운트 | `buildTouringPlan`이 pan/tilt/zoom 모두 non-null인 경우에만 slot 스텝 발행, 그 외는 `skipped++`로 카운트만 하고 이동 스텝을 만들지 않음. |

## 3. 신규 함수

### 3-1. `buildTouringPlan(setupResult)` — `web/core.js` (순수 함수)

```
export function buildTouringPlan(setupResult): { steps: TouringStep[], skipped: number }
```

- **역할**: `setup_result.json`의 raw `slots` 배열을 순회 스텝 배열로 변환한다. 부작용 없음(네트워크·DOM·state 접근 없음) — 그래서 vitest로 결정적 검증이 가능하다.
- **입력**: `{ slots?: TouringSetupSlot[] } | null | undefined`. `slots`가 배열이 아니면(`null`/`undefined`/`{}`) 빈 결과를 graceful하게 반환(throw 없음).
- **정렬**: `camId → presetId → (presetSlotIdx ?? 0) → slotId`로 방어적 재정렬. `setup_result.json`이 이미 이 순서로 저장되어 있어도(설계서 확인) 입력 순서에 의존하지 않도록 재정렬을 항상 수행 — 결정성 보장.
- **preset 스텝**: `camId:presetId` 그룹이 바뀔 때마다 `{kind:'preset', camId, presetId}` 1개 발행. `ptz`는 갖지 않는다(런타임에서 `findPresetPtz`로 채움). 그룹 내 모든 슬롯이 `centering=null`이어도 preset 스텝은 발행된다 — 요구 3(프리셋 이동)과 요구 4(슬롯 이동)를 독립 요건으로 취급했기 때문.
- **slot 스텝**: `centering.pan/tilt/zoom`이 모두 non-null인 경우에만 `{kind:'slot', camId, presetId, presetSlotIdx, slotId, ptz:{pan,tilt,zoom}}` 발행. 하나라도 null/undefined면 이동 스텝을 만들지 않고 `skipped`를 1 증가시킨다.
- **반환**: `{steps, skipped}`. `skipped`는 완료 모달 문구("센터링 없는 N개 슬롯 건너뜀")에 쓰인다.

타입 시그니처는 `web/core.d.ts`에 `TouringSetupSlot`(빌더가 실제로 읽는 필드만 선언, `floor_roi`/`occupy_roi`는 `[k:string]: unknown`으로 무시)과 `TouringStep`(preset|slot 유니온)으로 추가되었다.

### 3-2. `runTouringTest()` — `web/app.js` (런타임 오케스트레이션)

```
async function runTouringTest()
```

- **역할**: 로딩 → 플랜 산출 → 재진입 가드 → 순회 실행(물리 이동+대기) → 완료 모달 표시까지 전체 흐름을 조립한다. 신규 알고리즘 없음 — 기존에 검증된 `move()`/`gotoPreset()`/`findPresetPtz()`/렌더 함수들을 순서대로 호출하는 오케스트레이션 계층이다.
- **로딩**: 루트 경로 `fetch('/capture/saves/setup_result', {cache:'no-store'})`. viewer 라우트 전용 `api()` 헬퍼는 쓰지 않는다(`/capture/*`는 루트 경로 라우트이기 때문). 실패 시 `#cap-msg`에 오류 메시지를 표시하고 조기 종료 — DB/파이프라인 상태는 건드리지 않는다.
- **재진입 방지**: `state.touringActive` 플래그. 이미 순회 중이면 클릭을 무시(`return`)한다. 순회 시작 시 버튼을 `disabled` 처리하고 라벨을 `순회 중… (n/총)`으로 바꾸며, `finally` 블록에서 플래그 해제와 버튼 상태 원복을 보장한다(중간에 예외가 나도 버튼이 영구히 잠기지 않음).
- **순회 실행**: `buildTouringPlan()`이 만든 스텝 배열을 순서대로 `for...of`로 순회하며 각 스텝을 `await`로 처리한다.
  - `preset` 스텝: `syncTouringPreset(camId, presetId)`로 `state.cam`/`state.preset`을 전환하고 관련 UI를 동기화한 뒤, `findPresetPtz`로 home PTZ가 있으면 `move(home)`, 없으면 `gotoPreset()`(snapshot 폴백)을 호출.
  - `slot` 스텝: `move(step.ptz)`로 센터링 좌표로 직접 이동.
  - 매 스텝 후 `await new Promise(r => setTimeout(r, 1000))`로 1초 대기.
- **완료 처리**: 순회가 모두 끝나면 `#touring-done-body`에 "프리셋 N곳, 주차면 M곳 순회 완료" (+스킵 있으면 스킵 수) 텍스트를 넣고 `#touring-done-modal.hidden = false`로 표시.

### 3-3. `syncTouringPreset(camId, presetId)` — `web/app.js` (헬퍼)

기존 수동 카메라/프리셋 드롭다운 핸들러(app.js:3897/3909 부근)와 **동일한 절차**를 그대로 재현한다: `state.cam`/`state.preset`/선택 상태 초기화 → `renderDetectSelection()` → `renderCamSelect()`(내부에서 `renderPresetSelect`→`syncPtzFromPreset`+`renderSlotList` 연쇄 호출, 드롭다운 value도 동기화) → `drawRoiOverlay()` → `renderSelectionInfo()`. 물리 이동은 이 함수가 하지 않고 호출측(`runTouringTest`)이 `move`/`gotoPreset`으로 별도 수행한다.

## 4. 데이터 흐름

```
[클릭] #cap-touring
   │
   ▼
runTouringTest()
   │  GET /capture/saves/setup_result  (읽기 전용, saveStore.load — DB 미변경)
   ▼
data (raw setup_result.json: {slots:[{slotId,camId,presetId,presetSlotIdx,floor_roi,occupy_roi,centering}]})
   │
   ▼
buildTouringPlan(data)  ── 순수 함수 ──▶ { steps: TouringStep[], skipped: number }
   │
   ▼
for step of steps:
   preset → syncTouringPreset() + findPresetPtz()/move() 또는 gotoPreset()
   slot   → move(step.ptz)
   각 스텝 후 1초 대기
   │
   ▼
#touring-done-modal 표시 (프리셋 N / 슬롯 M / 스킵 K)
```

로딩 실패나 빈 결과(`steps.length===0`)일 때는 `#cap-msg`에 메시지만 표시하고 이동·모달 없이 종료한다.

## 5. UI

- **버튼**: `#cap-touring`("Touring Test"), 정밀 수집 실행 툴바(`div.cap-actions.toolbar.capture-actions`, `#cap-reset-db` 뒤)에 위치. 타이틀 툴팁에 "읽기 전용/DB·검출·오버레이 미변경" 명시.
- **완료 모달**: `#touring-done-modal` — 기존 `#cap-result-modal`(최종화 버튼과 결합된 수집완료 모달)과는 **별개의 신규 마크업**. `.modal`/`.modal-box`/`h3`/`#touring-done-body`(`.modal-body`)/`.modal-actions`의 `#touring-done-close`(확인 버튼 1개, 닫기만) 구조. 기존 `cap-result-modal` 로직과 id·이벤트 결선이 완전히 분리되어 있어 서로 간섭하지 않는다.
- 이벤트 결선은 `wire()`(app.js:4045~4048)에서: `#cap-touring` 클릭 → `runTouringTest`, `#touring-done-close` 클릭 → 모달 `hidden=true`.

## 6. 완전 독립성 근거

- **DB**: 접근 라우트는 `GET /capture/saves/setup_result` 하나뿐이며 이는 `saveStore.load()`만 호출하는 순수 조회 — 어떤 쓰기 라우트도 호출하지 않는다. `slot_setup`/`centering_slot`/`slot_ptz.json` 등 정본 데이터는 미접근.
- **검출/오버레이 상태**: `state.detectByKey`, `state.discoverByKey`, `state.parkingSlotsByKey`, `state.occByKey`, `state.occComputeByKey`와 관련 API(`discoverByKey`/`detectByKey`), `resetOverlayDisplay`는 일절 호출하지 않는다. 이는 MEMORY의 "정밀수집 종료 후 오버레이 유지 정책"과 충돌하지 않는다.
- **사용하는 것**: `fetch`(읽기), `move()`, `gotoPreset()`, `findPresetPtz()`, UI 렌더 함수(`renderCamSelect`/`renderPresetSelect`/`syncPtzFromPreset`/`renderSlotList`/`drawRoiOverlay`/`renderSelectionInfo`/`renderDetectSelection`), `state.cam`/`state.preset`/`state.selectedSlotId`/`state.selectedDetect`/`state.ptz`/`state.cameras`/`state.touringActive`.
- **서버 무변경**: 신규 라우트 없음. 기존 `/capture/saves/:name`, `/move`, `/cameras` 등 기존 라우트만 재사용.

## 8. 사용법

1. 정밀 수집 탭에서 "Touring Test" 버튼 클릭.
2. 자동으로 `setup_result.json`을 로딩하고, 카메라1의 프리셋1부터 순서대로 home PTZ → 소속 슬롯들의 센터링 PTZ로 순회(각 위치 1초 대기). 버튼 라벨이 `순회 중… (n/총)`으로 진행률을 표시하며 클릭 비활성화된다.
3. 순회가 끝나면 "Touring Test 완료" 모달이 뜨며 방문한 프리셋 수·슬롯 수·(있다면) 스킵된 슬롯 수가 표시된다. "확인"으로 닫는다.
4. `setup_result.json`이 없거나(정밀수집 미완료) 로딩에 실패하면 화면 하단 메시지 라인(`#cap-msg`)에 오류가 표시되고 이동은 시작되지 않는다.
5. 순회 중 버튼을 다시 눌러도 재진입되지 않는다(`state.touringActive` 가드).

## 9. 한계 — 정직 명시 (미검증 항목)

vitest는 **순수 로직(`buildTouringPlan`)만** 확정 검증했다. 아래는 시뮬레이터/브라우저 DOM 의존이라 이번 검증 범위 밖이며, **미검증**으로 명시한다(위장 없음):

- **카메라 실제 물리 이동**: `move(ptz)`가 실제로 PTZ를 구동하고 프레임을 재렌더하는지는 시뮬레이터 또는 실카메라 가동이 필요하다. 리더가 확인한 것은 `/capture/saves/setup_result` 로딩 경로(HTTP 200, 데이터 정합)뿐이며, **카메라가 실제로 그 좌표로 이동했는지 육안 검증은 수행되지 않았다.**
- **UI 동기화(`syncTouringPreset`) 실제 DOM 반영**: 정적 코드리뷰로 수동 핸들러와 동일 절차임을 확인했을 뿐, 브라우저에서의 실제 렌더 결과는 미검증.
- **완료 모달의 실제 표시/닫힘**: 마크업 존재와 이벤트 결선 코드는 확인했으나, 브라우저에서 실제로 뜨고 닫히는지는 미검증.
- **재진입 방지·버튼 라벨 진행률의 런타임 동작**: 코드 경로상 정상이나 실제 실행 중 관찰은 미검증.

이 항목들의 실동작 확정에는 시뮬레이터 가동 + 브라우저 수동/E2E 검증이 별도로 필요하다.

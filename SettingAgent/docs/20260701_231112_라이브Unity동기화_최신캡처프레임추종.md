## 1. 배경 — decoupled 표시로 인한 혼란

정밀수집(정밀 캡처)은 다음과 같이 동작한다.

- Unity는 **라운드 캡처 순간에만** 물리적으로 카메라를 이동해 각 타깃(cam·preset)을 촬영한다.
- 라운드와 라운드 사이(`intervalMs` 대기 구간)에는 Unity 카메라가 **정지**한다.

기존 뷰어(`capFrameTick`)는 이 실제 동작과 **분리(decoupled)** 되어 있었다.

- `capPresetKeys`(응답 헤더 `X-Cap-Presets` 파생)를 **700ms마다 순환**하며, 이번 run에서 캡처된 여러 프리셋 프레임을 번갈아 표시했다.
- 그 결과 **Unity는 대기(정지) 중인데도 웹 화면만 700ms 주기로 옛 프레임들을 계속 갈아치우며 반복 갱신**되어, "Unity는 멈춰 있는데 웹만 혼자 돌아간다"는 인상을 주었다.

### 사용자 요구 (확정)

- **(a) 수집 중 추종**: 캡처가 각 타깃을 찍을 때마다 그 **최신 프레임**을 따라간다(카메라 이동을 웹이 그대로 반영).
- **(b) 대기 중 고정**: 라운드 사이 대기 중에는 **순환하지 않고 마지막 프레임을 고정**한다.
- 즉 "라이브 Unity 동기화" = 웹이 캡처 진행(=Unity 이동)을 추종하고, 대기 중에는 정지.

## 2. 채택 방안 — A(서버 무변경) vs B(라이브 스트림 유지)

| 구분 | A. 최신 캡처 프레임 추종 (채택) | B. 라이브 스트림(loop) 수집 중 유지 (기각) |
|---|---|---|
| 방식 | `capFrameTick`이 `/capture/frame`을 **파라미터 없이** 호출 → 서버가 `getLastFrame()`(가장 최근 캡처)만 반환 | 수집 중에도 라이브 스트림 `loop`을 켜둠 |
| 동기 원리 | 잡이 타깃을 찍을 때마다 서버의 `lastFrame`이 갱신 → 웹이 캡처 진행을 그대로 추종. 대기 중엔 `lastFrame` 불변 → 같은 프레임 반환 → 사실상 정지 | loop은 `/snapshot`을 `state.ptz`(현재 PTZ) override로 요청하는데, 잡은 `/req_move`로 프리셋을 물리 이동한다. 둘은 무관 → 카메라를 두고 다투고 동기도 안 됨 |
| 서버 변경 | **불필요**(기존 라우트가 파라미터 없을 때 최신 프레임 반환을 이미 지원) | — |
| 판정 | **채택** | **기각**(카메라 경합·동기 불가) |

핵심 근거: 방안 B의 loop가 사용하는 `state.ptz`는 잡의 `/req_move` 프리셋과 **독립적**이라, 라이브를 켜둔 채 수집하면 두 주체가 같은 카메라를 서로 다른 곳으로 명령해 화면 동기가 불가능하다. 따라서 라이브를 끄고(A) 서버가 반환하는 "가장 최근 캡처 프레임"을 폴링으로 추종하는 방식을 택했다.

## 3. 변경 상세

### 3-1. `web/core.js` — 순수 헬퍼 `capFrameKey` 신규 (`core.js:109-117`)

```js
export function capFrameKey(cam, preset, round) {
  if (cam == null && preset == null && round == null) return null;
  return `${cam ?? ''}:${preset ?? ''}:${round ?? ''}`;
}
```

- **의도**: 최신 캡처 1건을 유일하게 식별하는 키를 만든다. 라운드 사이 대기 중에 반복 fetch되는 동일 프레임을 "직전과 같음"으로 판정해 재디코드/깜빡임을 막는 데 쓴다.
- **왜 cam·preset·round 모두 필요한가**: 같은 라운드 안에서도 프리셋(카메라 타깃)을 순회하므로, round만으로는 서로 다른 타깃 프레임을 구분할 수 없다. 세 값을 조합해야 캡처 1건이 유일해진다.
- **전부 null → null 규약**: 세 인자가 모두 null/undefined면 식별 불가로 보고 `null`을 반환한다. 소비 측은 `key != null` 가드로 이 경우 스킵하지 않고 **항상 갱신**한다(식별 불가한데 스킵하면 화면이 영영 안 바뀔 위험 회피).
- **타입 안정성**: 서버 헤더는 항상 문자열이므로 문자열 조합이 안정적이다. 숫자/문자 혼용(`1` vs `'1'`)도 템플릿 리터럴에서 동일 문자열로 수렴한다.
- 순수 함수(부작용·DOM 의존 없음)로 분리해 **vitest 단위 테스트 대상**으로 만들었다(CLAUDE.md 규칙2). `web/core.d.ts:65`에 타입 선언 추가.

### 3-2. `web/app.js` — `capFrameTick` 재작성 (`app.js:535-560`)

- import에 `capFrameKey`(from `./core.js`) 추가(`app.js:17`).
- 프리셋 순환(`capPresetKeys`/`capCycleIdx`)과 `?cam=&preset=` 쿼리를 **완전 제거**. `fetch('/capture/frame', { cache: 'no-store' })`로 **파라미터 없이 최신 프레임만** 요청.
- 응답 헤더 `X-Cap-Cam / X-Cap-Preset / X-Cap-Round`로 `capFrameKey(...)`를 계산.
- **깜빡임/재디코드 방지(스킵 판정)**: `if (key != null && key === lastCapFrameKey) return;` — 직전과 같은 캡처면 blob 생성·`URL.createObjectURL`·`frame.src` 교체·revoke·cap-msg 갱신을 **모두 스킵하고 조기 return**(`app.js:545`).
- 새 키면 `lastCapFrameKey` 갱신 후 blob 표시 + 직전 url `revokeObjectURL`(메모리 누수 방지). 스킵 시엔 새 blob을 만들지 않으므로 직전 url을 유지한다.
- **cap-msg 문구**: 순환 제거로 근거가 사라진 "대상 카메라 목록"(capPresetKeys 파생) 문구를 제거하고 `수집 중 — cam{c} 프리셋{p} (라운드 {r})`로 단순화.

### 3-3. `web/app.js` — `startCapFramePolling` (`app.js:562-568`)

- `loop.stop()` **유지**(라이브 스트림 중지 — 카메라를 캡처와 다투지 않게). 방안 A에서도 loop의 `/snapshot`은 프리셋과 무관하므로 켜두면 안 된다.
- `capPresetKeys=[]`/`capCycleIdx=0` 초기화 라인을 `lastCapFrameKey = null`로 교체(이전 run 잔여 키 제거).
- 폴링 주기 **700ms → 500ms**. 캡처 타깃 간 이동 간격이 약 0.6s이므로, 500ms면 각 타깃 프레임을 1~2회 폴 안에 따라잡아 추종성을 확보한다. 대기 중엔 동일 키 스킵으로 재디코드 비용이 없어 과부하가 없다.

### 3-4. 고아(orphan) 제거 — 내 변경으로 미사용화된 것만

- `capPresetKeys`, `capCycleIdx` 변수 선언 및 관련 주석 삭제.
- app.js 내 잔여 참조 0건 확인(`capPresetKeys`/`capCycleIdx`/`X-Cap-Presets` grep 무매치). 유지 대상은 `capFrameTimer`, `capFrameUrl`(revoke 대상), 신규 `lastCapFrameKey`.

### 3-5. 미사용화(데드 아님, 요청 범위 밖 → 무변경)

- 서버 `captureRoutes.ts`의 `X-Cap-Presets` 응답 헤더(`captureRoutes.ts:132-133`)와 `CaptureJob`의 `getFrameByPreset`/`getFramePresets`는 이제 뷰어가 **소비하지 않는다**. 그러나 이는 REST 라우트 계약의 일부이므로 삭제하지 않았다(요청 범위 밖). 데드 코드가 아니라 "미사용화" 상태로만 남겨둔다.

## 4. 입출력 / REST 계약 (무변경)

`GET /capture/frame` — 계약 자체는 **불변**. 뷰어의 **소비 패턴만** 바뀌었다.

- 요청: 파라미터 없이 호출(기존엔 `?cam=&preset=`으로 특정 프리셋을 지정했음).
- 서버 동작(`captureRoutes.ts:117-121`): 파라미터가 없으면 `latest`(가장 최근 캡처 = `getLastFrame`)의 jpeg·camIdx·presetIdx를 반환. 이 분기는 기존부터 존재.
- 응답 헤더(`captureRoutes.ts:129-133`): `X-Cap-Cam`=`String(camIdx)`, `X-Cap-Preset`=`String(presetIdx)`, `X-Cap-Round`=`String(latest?.roundIdx ?? 0)` — 항상 문자열(round 미상 시 `'0'`). `X-Cap-Presets`는 여전히 방출되나 뷰어 미소비.

경계면 shape 대조(검증자 확인): 서버 헤더명·타입(문자열)과 소비 측 `capFrameKey(cam→preset→round)` 인자 순서가 전 구간 정확히 일치. 불일치 없음.

## 5. 동작 흐름

1. **수집 시작** (`capPoll`에서 `active` → `startCapFramePolling`, `app.js:604-606`): 라이브 스트림 `loop.stop()`, `lastCapFrameKey=null`, 500ms 폴링 시작.
2. **수집 중 추종**: 잡이 타깃을 찍을 때마다 서버 `lastFrame` 갱신 → `capFrameTick`이 새 키를 받아 프레임을 교체 → 웹이 Unity 카메라 이동을 추종.
3. **대기 중 고정**(라운드 사이 `intervalMs`): `lastFrame` 불변 → 동일 키 반환 → `if (key === lastCapFrameKey) return`으로 스킵 → 마지막 프레임이 그대로 고정(순환 없음, cap-msg 라운드 번호 불변).
4. **수집 종료**(`done`/`stopped`/`error` → `stopCapFramePolling`, `app.js:608`): 타이머만 끈다. **자동 라이브 복귀 없음**(리더 확정 #1) → 마지막 캡처 프레임이 화면에 유지된다. 이후 사용자가 **수동 ▶(loop.start)** 를 눌러야 라이브 스트림이 재개된다(기존 경로, 무변경).

폴링 주기: **500ms**. 깜빡임 스킵 판정: `key != null && key === lastCapFrameKey`(전부 null인 식별 불가 프레임은 항상 갱신).

## 7. 수동 동작확인 체크리스트

- [ ] 수집 시작 → 라이브 스트림 꺼지고 캡처 프레임이 표시된다(`loop.stop()`).
- [ ] **수집 중 추종**: 라운드 내 타깃 순회 시 웹 프레임이 캡처(=Unity 카메라 이동)를 따라 갱신된다.
- [ ] **대기 중 고정**: 라운드 사이(intervalMs)에 프레임이 바뀌지 않고 고정된다(순환 없음, cap-msg 라운드 번호 불변).
- [ ] **깜빡임 없음**: 대기 중 동일 프레임 반복 fetch에도 이미지 재디코드/깜빡임이 없다(조기 return).
- [ ] **종료 후**: done/stopped 시 얼어붙지 않고 마지막 캡처 프레임이 유지된다. 결과 모달 1회 표시, floor 경고 정상.
- [ ] **수동 라이브 재개**: 종료 후 ▶(loop.start)를 누르면 라이브가 정상 재개된다.

## 8. 운영 주의

- **수집 종료 후 화면이 "멈춘 것처럼" 보이는 것은 정상**이다. 자동 라이브 복귀를 하지 않도록(자동 카메라 재명령 회피, 보수적 선택 — 리더 #1) 설계했기 때문이며, 마지막 캡처 프레임이 그대로 유지된다.
- 라이브 스트림을 다시 보려면 **수동으로 ▶(라이브) 버튼**을 눌러 `loop.start`로 재개해야 한다.
- 자동 복귀가 필요해지면 `capPoll`의 종료 분기(else)에 `loop.start(fps)` 한 줄 추가로 확장 가능하나, 현 기본안은 자동 이동 유발을 피하기 위해 복귀하지 않는다.

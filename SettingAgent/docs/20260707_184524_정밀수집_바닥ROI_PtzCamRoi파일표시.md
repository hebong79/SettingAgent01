## 2. 입력 JSON 구조 (`data/Place01/PtzCamRoi.json`)

```
{
  cameras: [
    {
      camera: {
        cam_id: 1,              // 1-based 카메라 번호
        name, position, eulerAngles, fov,   // (표시 로직 미사용 — 무시)
        imageWidth: 1920,       // 절대 픽셀 좌표의 기준 폭
        imageHeight: 1080       // 절대 픽셀 좌표의 기준 높이
      },
      presets: [
        {
          preset_idx: 1,        // 1-based 프리셋 번호
          parking_spaces: [
            { idx: 0, points: [[x,y],[x,y],[x,y],[x,y]] },  // 4점 절대 픽셀
            ...
          ]
        },
        ...
      ]
    }
  ]
}
```

- 실제 파일: cam_id=1, preset_idx 1/2/3, 주차면 수 **7 / 6 / 4**. 각 주차면 `points`는 4개의 `[x, y]` 절대 픽셀(예: preset1/idx0 첫 점 `[57.31739, 828.721436]`).
- `points`는 파일에서 `[x, y]` 배열 형태이나, 구현은 `{x, y}` 객체도 방어적으로 파싱한다(`Array.isArray(p) ? p[0] : p.x`).

## 3. 좌표계 — 절대 픽셀 → 정규화 → 픽셀 재투영

파일 ROI가 기존 오버레이(vehicle/plate/floor/occupancy)와 **동일한 좌표 파이프라인**을 타도록 3단계로 정합한다.

1. **절대 픽셀(1920×1080)**: 파일의 `points[i] = [x, y]`.
2. **정규화(0..1)**: 각 카메라의 `imageWidth(W)/imageHeight(H)`로 `{ x: x/W, y: y/H }`(core.js `normalizePtzCamRoi`). y축은 아래로 증가(그대로 유지 — 기존 `toPixelQuad`와 일치). 카메라마다 W/H가 다르면 각 카메라 값으로 개별 정규화(다중 카메라 확장 안전).
3. **픽셀 재투영**: 렌더 시 `toPixelQuad(points, overlay.width, overlay.height)`로 현재 캔버스 크기에 재투영(core.js:19 — `p.x * imgW, p.y * imgH`). 기존 floor/occupancy 폴리곤과 **완전히 동일한 함수**를 재사용하므로 정합이 보장된다.

> 검증(설계 기대): preset1/idx0 첫 점 `[57.31739, 828.721436]` → `x = 57.31739/1920 ≈ 0.0298528`, `y = 828.721436/1080 ≈ 0.7673347`. QA 실측 일치(±1e-4).

## 4. 백엔드 — `GET /capture/place-roi` (raw JSON 서빙)

### 4.1 라우트 (`src/api/captureRoutes.ts`)

- import 추가: `import { readFile } from 'node:fs/promises';`(1행).
- `CaptureRouteDeps`에 `placeRoiFile?: string` 필드 추가(51행) — 주차면 폴리곤 파일 경로.
- `registerCaptureRoutes` 말미(saveStore 블록 뒤, 264–277행)에 라우트 등록. **항상 등록 + 내부 미설정 가드**:
  ```ts
  app.get('/capture/place-roi', async (_req, reply) => {
    if (!deps.placeRoiFile) { reply.code(404); return { error: 'place-roi 미설정' }; }
    try {
      const raw = await readFile(deps.placeRoiFile, 'utf8');
      return JSON.parse(raw);          // raw JSON 그대로 — 정규화는 클라이언트 소유
    } catch (err) {
      const e = err as NodeJS.ErrnoException;
      reply.code(e.code === 'ENOENT' ? 404 : 500);
      return { error: e.code === 'ENOENT' ? 'PtzCamRoi.json 없음' : '읽기/파싱 실패', detail: e.message };
    }
  });
  ```
- **설계 의도**: 라우트는 얇은 결정형 파일 I/O만 담당하고 **정규화·검수는 하지 않는다**(클라이언트 `normalizePtzCamRoi` 소유). 에러 대칭: 미설정=404 / 파일 없음(ENOENT)=404 / 파싱·읽기 실패=500.

### 4.2 경로 배선 (index.ts → server.ts → captureRoutes)

- `src/index.ts`(1행 `import { join } from 'node:path'`, 71행): `buildServer({...})`에 `placeRoiFile: join(tools.store.dataDir, 'Place01', 'PtzCamRoi.json')` 추가 → 하드코딩 회피, config(`store.dataDir`) 기준 상수 도출.

## 5. 프론트 — 정규화·렌더·결선

### 5.1 순수 함수 `normalizePtzCamRoi(json)` (`web/core.js`, 364–409행)

DOM/fetch 미참조 순수 함수(vitest 직접 import). 반환:
```
{
  byPreset: { "<camId>:<presetIdx>": [ { idx, points:[{x,y}...] } ] },  // points 정규화(0..1)
  report:   [ { camId, presetIdx, spaceCount, issues:[문자열] } ]        // 프리셋별 검수
}
```

- 키 생성은 `presetKey(camId, presetIdx)`(= `` `${camId}:${presetIdx}` ``)로, 프론트 `currentFrameKey()`와 **동일 함수**를 써서 별도 매핑 로직 없이 직접 매칭된다.
- `byPreset` 기록 조건: `sizeOk && normSpaces.length`(크기 오류·빈 프리셋은 미기록). `report`는 **항상** 프리셋 항목을 산출(검수 목적).

**검수(report[].issues) 항목 목록**:

| 조건 | issue 문자열 | byPreset 기록 |
|------|--------------|:---:|
| `camera` 없음 또는 `imageWidth/Height` 비수치·≤0 | `이미지 크기 누락/오류` | 미기록(정규화 불가) |
| `parking_spaces` 비배열/빈 | `주차면 없음` | 미기록 |
| `cam_id` 없음 | `cam_id 누락` | (크기 OK면) 기록 |
| `preset_idx` 없음 | `preset_idx 누락` | 기록 |
| space `idx` 없음 | `idx 누락` | 해당 space skip |
| space `points` 비배열 | `idx N: points 누락` | 해당 space skip |
| `points` 개수 ≠ 4 | `idx N: 점 4개 아님(M개)` | 기록(면은 유지) |
| 좌표가 `[0,W]×[0,H]` 이탈 | `idx N: 좌표 범위 이탈` | 기록(면은 유지) |

- **throw 금지**: 어떤 malformed 입력에도 부분 결과 + issues 반환(백엔드 UNKNOWN 강등 철학과 정합). `json`이 null/비객체 → `{ byPreset:{}, report:[] }`.
- 타입 선언(`web/core.d.ts` 156–169행): `PlaceRoiSpace`, `PlaceRoiReport`, `normalizePtzCamRoi(json: unknown)` 시그니처.

## 6. 좌표계·매칭 계약 (1-based)

`cam_id ↔ camIdx`, `preset_idx ↔ presetIdx`가 **동일 1-based 넘버링**이라는 가정 위에 동작한다.
- `normalizePtzCamRoi`가 `presetKey(camId, presetIdx)`로 키 생성.
- `currentFrameKey()`(app.js:239–242)는 수집 중이면 `presetKey(state.capFrameKey2.cam, .preset)`, 아니면 `presetKey(state.cam, state.preset)` — 동일 `presetKey` 함수.
- 실제 파일 cam_id=1, preset_idx 1/2/3 → 키 `1:1`/`1:2`/`1:3`가 라이브 프리셋 키와 직접 매칭(별도 변환 로직 없음). QA 전용 테스트로 `byPreset`에 `presetKey(1, n)` 존재 확인.

## 8. 한계 / 확인 필요 (누락 명시 — 통과 위장 금지)

- **표시 전용**: 파일 ROI 편집·저장·핫리로드 없음. **세션 1회 로드**(`placeRoiLoaded` 가드 — 탭 재진입해도 재요청 안 함). 파일이 바뀌면 브라우저 새로고침 필요.
- **DOM/Canvas 결선(F2~F5)**: `loadPlaceRoi`/`drawPlaceRoiOverlay`/토글/setTab 훅은 브라우저 통합이라 vitest 자동화 비대상. 로직은 순수 함수(F1)로 이관되어 F1 유닛으로 커버되나, 실제 버튼·폴리곤 렌더는 **수동 스모크** 필요:
  1. precise 탭 진입 → 콘솔 검수 로그 없음(정상 파일) + 현재 프리셋 파랑 폴리곤 + idx 라벨.
  2. 프레임 순환으로 프리셋 전환 시 해당 프리셋 폴리곤만 갱신(다른 프리셋 것 미표시).
  3. `바닥ROI(파일)` 토글 해제 → 파일 ROI만 사라지고 vehicle/floor/occupancy 유지.
  4. 파일 부재(404) → 오버레이 없음, 다른 기능 정상(조용한 강등).
- **실 카메라 프레임 위 오버레이 미검증**: 시뮬레이터 미가동으로 라이브 프레임 위 실 정합은 자동/수동 어느 쪽으로도 미검증. 리더 sharp parity는 정지 이미지 재렌더 기준(브라우저 캔버스 실렌더와 등가 가정).
- **다중 카메라·다중 Place 미검증**: 현재 파일은 `cameras[]` 1대·`Place01` 1폴더. 정규화 로직은 카메라별 W/H 개별 처리로 다중 카메라 안전하나, 2대 이상/다중 Place 실데이터는 미검증(확장 여지는 §9 참조).

## 1. 배경·목표

각 주차면 슬롯에서 번호판 검지율 100%에 도전한다. 원본(와이드) 프레임에서 번호판이 미검출이어도, 슬롯의 **육면체 앞면중심**(`slot3d_front_center` — 지면모델 기반 기하 산출값, 검출 여부와 무관하게 항상 존재)을 조준점 삼아 **디지털 크롭 후 업스케일**하여 재탐지하고, 검출되면 **아핀 역계산**으로 원본 프레임 좌표계 LPD 박스를 복원해 `slot_setup.lpd`(원본 프레임 기준 정본 LPD 값)를 채운다. 카메라 이동이 없는 순수 이미지 연산이라 결정론적이며(LLM 미사용), 원거리·소형 번호판으로 인한 미검출 슬롯을 사후에 구제하는 것이 핵심 가치다.

광학(실제 PTZ 이동) tier는 진짜 저해상도(원본에 번호판 픽셀 자체가 소멸된 경우) 전용 2차 에스컬레이션으로 설계서에 정의돼 있으나, 이번 구현 범위에는 포함되지 않는다(§6 한계 참조).

## 2. 신규 모듈 상세

### 2-1. `src/calibrate/cropZoom.ts` — 순수 기하 + sharp IO

| 함수 | 역할 |
|---|---|
| `computeCropWindow(center, frac, aspect)` | 앞면중심 `center` 기준 크롭 창 산출. 폭 `w = min(1, frac)`, 높이 `h = min(1, frac·aspect)`(`aspect` = 원본 imgW/imgH → 픽셀 정사각 크롭). 좌상단은 창 **크기를 보존한 채** `[0, 1-size]` 로 클램프되어 모서리 anchor에서도 창이 프레임 밖으로 나가지 않는다(cropZoom.ts:23-30). |
| `toCropPoint(p, W)` | 원본 정규화 점을 크롭 창 `W` 기준 정규화 좌표로 순변환(`(p.x-W.x)/W.w`) — anchor를 크롭 좌표로 환산해 최근접 후보 선택에 사용(cropZoom.ts:33-35). |
| `backmapQuad(cropQuad, W)` | **아핀 역계산의 핵심**: `orig = W.xy + q·W.wh`를 quad 4점 각각에 독립 적용해 원본 좌표 복원(cropZoom.ts:38-40). |
| `cropAndUpscale(jpeg, W, outLongPx)` | sharp `extract`(픽셀 경계 반올림·클램프)+`resize`(장변 `outLongPx`, 종횡비 보존)+`jpeg` 인코딩. 유일한 부수효과 함수(cropZoom.ts:46-68). |

**역계산이 오차 0에 가까운 이유(설계서 §2-1 실증):** `LpdClient.detect`는 입력 이미지 자체의 해상도로 좌표를 정규화하므로, 크롭 JPEG를 넣으면 반환 quad는 이미 "크롭 정규화 좌표"다. 따라서 원본 복원식에는 **업스케일 배율이 등장하지 않는다** — 배율은 검출 성패(작은 번호판이 보이느냐)에만 영향을 주고, 좌표 역산 정확도와는 무관하다. QA가 vitest로 재현한 왕복 오차는 실측 <1e-12(요구 기준 <1e-9)였다.

### 2-2. `src/calibrate/plateDiscovery.ts` — 탐색 루프 `PlateDiscovery`

`discoverSlot(target, presetPtz)` 상태전이(plateDiscovery.ts:95-144):

1. **anchor 부재** → 즉시 `found:false, reason:'no_anchor'`(카메라 호출조차 안 함).
2. **원본 프레임 1회 캡처**(`camera.requestImage`, 이후 tier는 이 프레임을 재사용 — 카메라 무이동).
3. **Tier 0(full)**: 원본 전체 이미지에 LPD → `pickNearestPlate`(controlMath.ts 재사용)로 anchor 최근접 후보 선택 → 거리 ≤ `matchRadiusNorm`(기본 0.15) 이면 **역계산 불요**로 즉시 채택(plateDiscovery.ts:111-119). 반경 밖 후보는 이웃 슬롯 오선정 방지를 위해 기각.
4. **Tier 1(crop)**: `k=1..maxSteps`(기본 5) 반복. `frac_k = frac0·shrink^(k-1)`(0.40, 0.24, 0.144, 0.0864, 0.05184 — 기본값 기준)로 창을 축소(=실효 줌 상승)하며 `computeCropWindow`→`cropAndUpscale`→LPD 재탐지→`pickNearestPlate`(anchor의 크롭좌표 환산 기준). 검출되면 `backmapQuad`로 원본 좌표 복원 후 `found:true, tier:'crop', step:k, cropWindow:W` 반환(plateDiscovery.ts:122-140). `frac_k < minFrac`(0.05) 이면 루프 조기 종료.
5. **전부 실패** → `found:false, reason:'no_plate', tier:'crop', step:maxSteps`.

camera/lpd/crop(사용자 지정 시)는 생성자 주입(DI)이라 sharp 없이 순수 상태전이만 단위테스트 가능하다.

### 2-4. `src/calibrate/PlateDiscoveryJob.ts` — 잡 상태머신

`PtzCalibrator` 패턴을 그대로 미러한 단일 인메모리 상태머신(idle/running/done/error).

- `start(slotIds?)`: 실행 중 재시작은 `throw`(라우트가 409로 변환). `expandDiscoveryTargets`로 대상 산출 후 슬롯을 **순차 await** 처리(PlateDiscoveryJob.ts:111-127).
- 슬롯 개별 실패는 `try/catch`로 흡수해 `reason:'no_plate'`로 기록하고 잡 전체를 중단시키지 않는다(PlateDiscoveryJob.ts:135-155) — 정직 리포트 원칙.
- 완료 시 `writePlateDiscovery`(JSON 정본) → `saveSlotLpd(items)` 호출. `saveSlotLpd`는 `found && lpdOrig != null && globalIdx != null` 인 항목만 `SlotLpdRow`로 모아 `store.upsertSlotLpd(rows)`를 **best-effort**(실패해도 JSON 정본·잡 완료 상태는 유지)로 호출한다(PlateDiscoveryJob.ts:187-204).
- 프리셋 PTZ는 `resolvePresetPtz`(detectPipeline.ts 재사용)로 키(`cam:preset`)별 1회만 조회해 캐시.
- `getLastFrame()`(뷰어 `/discover/frame`용), `getStatus()`(state/done/total/found/current) 는 `PtzCalibrator`의 대응 메서드와 동일 패턴.

`calibrateRoutes.ts` 패턴을 그대로 미러한 얇은 라우트 등록 함수(discoverRoutes.ts:18-59).

## 3. DB 반영 — `slot_setup.lpd` 부분 UPDATE

`SqliteStore.upsertSlotLpd(rows: SlotLpdRow[])`(SqliteStore.ts:297-303):

```sql
UPDATE slot_setup SET lpd_obb = ?, updated_at = ? WHERE slot_id = ?
```

트랜잭션으로 일괄 실행. **`replaceSlotSetup`(DELETE 후 전량 INSERT)과 달리 slot_id 키 단위 부분 UPDATE만 사용** — 대상 외 슬롯, 대상 슬롯의 타 컬럼(`slot_roi`/`vpd_bbox`/`occupy_range`/`pan`/`tilt`/`zoom`/`centered`/`img1`/`slot3d_front_center`)은 완전히 불변이다. 미존재 `slot_id`는 조용히 무시(throw 없음). `lpdObb`가 `null`이면 해당 슬롯의 `lpd_obb`를 클리어한다(호출측은 `found` 항목만 넘기므로 실사용은 항상 채움 방향).

이 결정은 사용자 메모리 노트 "[[finalize-slotsetup-wipe-fragility]]"(검출 없는 finalize의 `replaceSlotSetup`이 데이터를 파괴할 수 있다는 취약점 기록)를 정확히 회피하도록 설계·구현된 것이며, 기존 `upsertSlotCentering`(SqliteStore.ts:274)과 동일한 부분 UPDATE 계약을 그대로 재사용한다.

DB 스키마 변경은 없다(기존 `slot_setup.lpd_obb` 컬럼 재사용). `capture/types.ts`에 신규 `SlotLpdRow`(`slotId`/`lpdObb`/`updatedAt`) 타입만 추가됐다.

## 4. 사용법(배치 잡 파이프라인)

```
1. (사전) finalize 완료 → slot_setup 에 slot3d_front_center 채워짐(camera pos + ground model 기반, Finalizer.ts:271)
2. POST /discover/ptz { slotIds?: [...] }  → 잡 시작(전체 또는 필터)
3. GET  /discover/status  (폴링)          → { state, done, total, found, current }
4. GET  /discover/frame                   → 진행 중 최근 캡처 원본 프레임 관찰(뷰어용)
5. (완료 후) GET /discover/result          → plate_discovery.json 내용 그대로
   +  slot_setup.lpd 가 발견된 슬롯만 부분 UPDATE 됨(원본 프레임 좌표계 LPD OBB)
```

배선: `index.ts`가 `PlateDiscoveryJob({camera, lpd, store: sqlite, outFile: 'data/plate_discovery.json'})`을 생성해 `buildServer`에 `plateDiscovery`/`discoverOutFile`로 전달하고, `server.ts`는 두 값이 모두 있을 때만 `/discover/*`를 조건부 등록한다(server.ts:271-272). **기존 캡처/파이널라이즈/캘리브레이션 자동 연쇄에는 포함되지 않는 수동 실행 상류 잡**이다(index.ts:88-91).

## 6. 한계·후속(정직 리포트)

1. **광학 PTZ tier(2차 에스컬레이션) 미구현**: 진짜 저해상도(원본에 번호판 픽셀이 소멸된 경우)는 디지털 업스케일로 구제 불가 — 설계서 §2-2가 남긴 인터페이스 심(`reason:'needs_optical'`)만 타입에 존재. `G_i`(줌 상태 지면모델) 유도 방식(가정 A: 회전+f스케일 합성 vs 가정 B: 재추정)은 미확정 — 별도 goal/loop 후속.
2. **후면주차(뒷번호판) 미대응**: 앞면중심은 "전면주차·앞번호판 카메라향" 전제. 후면주차 차량은 슬롯 뒤·다른 위치라 조준이 빗나간다. 앞/뒤 후보 2개 또는 주차방향 정보 도입은 범위 밖.
3. **번호판 세로 오프셋·차종별 높이 편차**: 앞면중심(육면체 중앙높이 0.75m 부근)과 실제 번호판(≈0.4m) 위치 차이는 넉넉한 `frac0=0.40` 창으로 흡수하는 근사일 뿐, 정밀 보정은 아니다. 개선안(finalize 시 번호판높이 앵커 별도 저장)은 이번 범위 밖.
4. **상수 눈대중 초기값**: `frac0`/`shrink`/`minFrac`/`maxSteps`/`matchRadiusNorm`는 실측 없는 초기값(설계서 §3-3, §9-4). QA가 경계 근접(step5 frac=0.05184가 minFrac=0.05 바로 위 — `maxSteps`와 `minFrac`이 상호 의존)을 관찰했으나 결함은 아니며, 라이브 튜닝 대상으로 남는다.
5. **`slot3d_front_center` 부재 슬롯**: 지면모델이 없거나 퇴화된 슬롯은 anchor가 `null` → `reason:'no_anchor'`로 정직 리포트(위장 성공 없음).

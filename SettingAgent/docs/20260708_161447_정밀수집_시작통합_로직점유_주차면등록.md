## 1. Goal / 요구사항(R1~R6)

정밀수집 "시작"을 하나의 통합 흐름으로 엮는다. 시작하면 **VPD(차량)/LPD(번호판)가 자동 표시**되고, **바닥 LLM 토글이 OFF면 캡처 전 과정에서 LLM을 쓰지 않으며**(파일 바닥ROI로 진행), **파일 바닥ROI의 주차면이 하단 목록에 등록**되고, **점유는 로직으로 판정하여 LPD 번호판 중심에 작은 원**으로 표시한다. 최종화 시에만 LLM으로 점유를 1회 검증한다.

| 수용기준 | 대응 | 구현 위치 |
|---|---|---|
| **R1** VPD·LPD 모두 화면 자동 표시 | 시작/프리셋전환마다 `runLiveDetect()` 자동 호출 → 기존 `drawDetectOverlay`(청록 VPD/노랑 LPD) | G2 (app.js) |
| **R2** 주차면 등록 = 파일 바닥ROI(PtzCamRoi.json) | `renderSlotList`를 파일 모드에서 `state.placeRoi[key]`(파일 idx) 소스로 분기, idx+점유 뱃지 | G5 (app.js) |

## 2. 통합 구조 — 앞선 조각(§02~§04)을 시작 흐름으로 엮음

본 작업은 **신규 기능을 크게 만들지 않는다.** §02~§04에서 이미 구현·동작 중인 조각을 정밀수집 시작(capStart) 흐름으로 결선한다.

- **§02**: `GET /capture/place-roi` + `normalizePtzCamRoi`(파일 바닥ROI 정규화, 프리셋별 `byPreset["cam:preset"]`).
- **§03**: `selectFloorRoi`(파일/LLM floor 소스 선택 순수함수) + `drawFileFloorRoi`(파일 바닥 초록 렌더) + `cap-floor-llm` 토글 + 백엔드 `floorReviewer` 게이트.
- **§04**: `runLiveDetect`/`drawDetectOverlay`(라이브 VPD/LPD 검출·표시).

§05는 여기에 (a) 기본값 1개(G1/R6), (b) 자동 검출 트리거(G2/R1), (c) 신규 로직 점유 순수함수(G3/R4·R5), (d) 점유 오버레이 원 교체(G4/R5), (e) 주차면 목록 소스 분기(G5/R2), (f) 최종화 LLM 1회 검증 + 캡처 중 occupancy LLM 게이트(G6/R3·R4)만 얹는다. 신규 클래스 없음 — **순수함수 2개(`computeOccupancy`/`quadCentroid`) + Finalizer 비교 블록 + 옵셔널 라우트 필드 1개**가 실질 신규 코드.

### G3 — 로직 점유 순수함수 `computeOccupancy` (R4·R5) — core.js

`web/core.js`, `selectFloorRoi` 바로 뒤 신규 export 2개:
- **`quadCentroid(quad)`**(core.js:434): 4점 산술평균 중심. 면적가중(백엔드 `polygonCentroid`)이 아니라 단순 평균 — 번호판 근사 중심으로 충분. 4점 아님/좌표 비수치 → `null`.
- **`computeOccupancy(floorPolygons, plates)`**(core.js:454): 각 floor polygon(`{idx, quad}`)에 대해 `plates`(`{quad}[]`)의 centroid 중 `pointInQuad`(core.js:492, **기존 함수 재사용·미러 없음**)로 내부 판정 → **첫 매칭** 번호판 중심을 점유 근거로 채택 → `[{idx, occupied, center?}]`(center는 occupied일 때만). `floorPolygons`/`plates` 누락·비배열 → `[]`(throw 금지, 강등 철학).

의도: 점유 판정은 번호판 centroid + ray-casting = 전부 결정형이고, 프레임틱마다 재계산되는 고빈도 수치 루프이므로 LLM이 아니라 프론트 순수함수가 정합. `core.d.ts`에 시그니처(`quadCentroid`/`computeOccupancy`/`OccupancySpace`)만 추가.

### G4 — 점유 오버레이 = 작은 원 (R5) + 로직 점유 계산 배선 — app.js

- **신규 `updateLogicOccupancy()`**(app.js:259): 현재 프리셋(`currentFrameKey()`)의 파일 바닥ROI(`selectFloorRoi({useLlm:false, placeRoi, key})`) × 현재 프리셋과 일치하는 `state.detect`(base LPD `plates` + `vehicles[].plate` 복원분 합집합)로 `computeOccupancy` 호출 → `state.occComputeByKey[key]` 갱신. 파일 바닥ROI 없으면 계산 skip(이전 값 보존).
- `drawRoiOverlay()`(app.js:194): `drawOccupancyOverlay(ctx)` 호출 직전에 `updateLogicOccupancy()` 추가 — 검출·파일ROI가 모두 준비된 시점의 단일 계산 지점("단순안").
- **`drawOccupancyOverlay(ctx)`**(app.js:281): 다각형 채움/stroke → **`ctx.arc` 반경 5px 빨강 소원**(`#ff4d4d`) 교체. 소스 `state.occByKey` → `state.occComputeByKey`. `occupied && center` 보유분만 그림(원 + idx 라벨). `#roi-occupancy` 토글 가드·iteration 구조 유지.
- `capStart()`(app.js:1009): `state.occComputeByKey = {}` 신규 런 리셋 추가(기존 `state.occByKey = {}` 옆).

## 5. `state.occComputeByKey`(로직 점유) vs `state.occByKey`(LLM 점유) 분리 근거

설계 §05 §6은 `state.occByKey` shape 재사용을 기본안으로 두되 "충돌 관찰되면 분리"를 조건부 승인했다. 구현 중 충돌이 실제로 확인되어 **분리 컨틴전시를 이행**했다(임의 설계 변경 아님):

- **`state.occByKey`**(app.js:63): 백엔드 LLM 판정 결과(`occupancyByKey`/`occupancyRows`/`occupancyAverage`가 `occupiedCount/total/rate`를 참조 → **최종 결과 모달·점유율 표** 소스).
- **`state.occComputeByKey`**(app.js:64): 프론트 로직 점유(`{spaces:[{id,occupied,center?}]}` → **오버레이 원·목록 뱃지** 소스, R4/R5).

`occByKey`를 로직 점유로 덮으면 `occupiedCount/total/rate`가 undefined가 되어 결과 모달(`showCaptureResult`→`occupancyRows`)이 깨진다. 두 소스는 성격(LLM 요약 통계 vs 프레임별 로직 점유)이 달라 분리가 정합.

## 8. 한계 · 정직 기록 (통과로 위장하지 않음)

- **DOM/Canvas 결선(G1 기본 OFF·G2 자동검출·G4 원 렌더·G5 목록 분기)은 vitest 밖** — 브라우저 통합이라 자동화 비대상. 리더 경험적 스모크(§6)로만 확인됨. 순수 로직(G3)만 vitest로 커버.
- **최종화 LLM 실호출·실 VPD/LPD·시뮬 연동 스모크는 모킹 대체** — 순수 유닛은 저장분 재사용 경로(설계 권고안)만 검증했다. **실 LLM 검증은 미구동**. 프레임을 새로 LLM에 던지는 fresh-call 경로는 설계 스코프 밖.
- **최종화 LLM 검증은 저장분 재사용 방식** — `getLatestOccupancy`(캡처 중 이미 저장된 LLM 점유)를 재사용해 로직 점유와 1회 비교한다. Finalizer가 프레임 버퍼를 보유하지 않으므로 프레임 새 배선을 회피한 가장 단순한 안. "1회 판단"의 해석은 fresh-call이 아니라 저장분 대조.
- **스테일 주석 1건**: `web/app.js:194` 주석이 `state.occByKey[현재프리셋]`로 기재되나 실제 코드는 `state.occComputeByKey[key]`에 기록(§5 필드 분리 반영). 동작 무영향 — 외과적 원칙(CLAUDE.md 3항)상 미수정, 문서로 인계. 후속 터치 시 정정 권장.
- **"주차면 등록" = 목록 표시로 한정** — 파일/artifact 영속화·slotId 매핑은 범위 밖(파일 ROI는 이미 `PtzCamRoi.json` 원본에 존재). 표시 idx만 사용, 기존 `slotId`(`c{cam}p{preset}s{pos}`) 체계와 매핑하지 않음.

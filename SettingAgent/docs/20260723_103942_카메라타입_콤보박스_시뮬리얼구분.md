# 카메라 타입 콤보박스 전환(시뮬레이터 / 리얼카메라)

- 일자: 2026-07-23 (최초 작성) / 2026-07-23 후속 갱신(protocol 자동 정합 추가)
- 대상: SettingAgent 웹 UI("카메라 실행 설정" 패널)
- 변경 파일: `SettingAgent/web/index.html`, `SettingAgent/web/app.js`, `SettingAgent/web/core.js`
- 신규 테스트: `SettingAgent/test/cameraKindSelect.test.ts`
- 백엔드: **무변경**(기존 스키마가 이미 편집을 지원)

## 1. 변경 배경·목표

"카메라 실행 설정"의 "카메라 타입"(`#opt-camera-kind`) 필드는 지금까지 `readonly` 텍스트 입력으로, 현재 등록된 카메라의 `kind` 값을 화면에 보여주기만 하고 사용자가 바꿀 수 없었다. 반면 백엔드(`settingsStore.ts`)는 이미 `kind` 필드 편집을 완전히 지원하고 있었다(§3 참조) — 즉 프론트엔드에만 편집 경로가 빠져 있는 상태였다.

목표: 이 필드를 **콤보박스(select)** 로 바꿔, 사용자가 카메라를 **시뮬레이터 / 리얼카메라** 2분류 중 하나로 직접 선택하고 저장할 수 있게 한다. 선택을 바꾸면 RTSP 입력 활성/비활성과 안내문(streaming note)이 즉시 화면에 반영되어야 한다.

## 2. 데이터 모델

`SettingAgent/src/config/toolsConfig.js`의 `CameraSourceConfigSchema`가 정의하는 카메라 소스의 `kind` 필드는 두 값만 허용한다.

| kind 값 | 콤보박스 라벨 | 의미 |
|---|---|---|
| `sim` | 시뮬레이터 | Unity 시뮬레이터 카메라(HTTP MJPEG 스트리밍) |
| `hucoms` | 리얼카메라 | 실제 IP 카메라(RTSP → FFmpeg → 브라우저 MJPEG) |

## 3. UI 변경 상세

### 3.1 `index.html` (~line 380)

기존 `readonly` 텍스트 `<input>`을 아래 `<select>`로 교체했다.

```html
<label class="settings-field">카메라 타입
  <select id="opt-camera-kind">
    <option value="sim">시뮬레이터</option>
    <option value="hucoms">리얼카메라</option>
  </select>
</label>
```

### 3.2 `app.js` — 렌더(로드 시 콤보 반영)

`renderCameraSource(id)` (~3280행)에서 과거에는 `kind`를 사람이 읽는 문자열("Hucoms 실카메라" 등)로 표시했으나, 이제는 콤보박스의 `value`를 `kind` 원본 값으로 직접 세팅한다.

```js
$('opt-camera-kind').value = source?.kind === 'hucoms' ? 'hucoms' : 'sim'; // 콤보: 시뮬레이터(sim)/리얼카메라(hucoms).
```

같은 함수 안에서 RTSP 입력 활성/비활성과 안내문(note)도 `kind`에 연동되어 갱신된다(이 로직 자체는 기존 것이며, 이번 변경으로 콤보 전환 시에도 계속 정상 동작함을 확인했다).

```js
$('opt-camera-rtsp').disabled = source?.kind !== 'hucoms';
const streaming = source?.kind === 'hucoms'
  ? '스트리밍: RTSP → FFmpeg → 브라우저 MJPEG'
  : '스트리밍: 시뮬레이터 URL → HTTP MJPEG';
```

### 3.3 `app.js` — 캡처(사용자 선택 확정)

`captureCameraSourceEdits()` (~3268행)에서 콤보 선택값을 편집 중인 `source` 객체의 `kind`로 확정한다.

```js
source.kind = $('opt-camera-kind').value === 'hucoms' ? 'hucoms' : 'sim'; // 콤보 선택을 데이터 모델(kind)로 확정.
```

### 3.4 `app.js` — change 이벤트 결선

`setupEventListeners` 안(~3852행)에 `#opt-camera-kind`의 `change` 리스너를 신규 추가했다. 사용자가 콤보를 바꾸면 즉시 편집을 확정(capture)하고 같은 소스를 다시 렌더링해 RTSP 활성/비활성·note를 동기화한다.

```js
// 카메라 타입(시뮬레이터/리얼카메라) 전환 → 편집 확정 후 재렌더(RTSP 활성·note 갱신).
$('opt-camera-kind').addEventListener('change', () => {
  captureCameraSourceEdits();
  renderCameraSource(renderedCameraSourceId);
});
```

### 3.5 `app.js`/`core.js` — protocol 자동 정합(후속 변경)

최초 구현에서는 `kind`만 콤보로 바꿀 수 있었고 `protocol`(`unity-rpc`/`unity-rest`/`hucoms-v1.22`) 필드는 손대지 않았다(§7 비목표에 명시했던 한계). 이후 이 한계가 실제 라우팅 버그로 이어질 수 있음이 확인되어 자동 정합 로직을 추가했다.

**근거**(`SettingAgent/src/viewer/sourceRegistry.ts:47-59`): 런타임 소스 선택이 `kind`뿐 아니라 `protocol`도 참조한다.

```ts
if (src.kind === 'sim') {
  if (src.protocol === 'unity-rpc') { /* RpcCameraSource(RPC 경로) */ }
  else { /* SimulatorSource(REST 경로) — protocol이 'unity-rest'거나 미정의여도 여기로 빠짐 */ }
} else {
  /* RealPtzSource — kind만으로 결정, protocol은 참조하지 않음 */
}
```

즉 `hucoms → sim`으로 전환한 뒤 `protocol`이 이전 값(`'hucoms-v1.22'`)으로 남아 있으면, sim임에도 `unity-rpc` 분기를 타지 못하고 REST 경로(`SimulatorSource`)로 잘못 빠진다.

**해결**: `web/core.js`에 순수함수 `alignProtocolToKind(kind, protocol)`을 신규 추가(약 222행).

```js
export function alignProtocolToKind(kind, protocol) {
  if (kind === 'hucoms') return 'hucoms-v1.22';
  return protocol === 'unity-rpc' || protocol === 'unity-rest' ? protocol : 'unity-rpc';
}
```

- `kind === 'hucoms'` → 항상 `'hucoms-v1.22'`(유일 옵션).
- `kind === 'sim'` → 기존 `protocol`이 이미 unity 계열(`'unity-rpc'`|`'unity-rest'`)이면 그대로 유지(사용자가 골라둔 RPC/REST 선택을 보존), 아니면 기본값 `'unity-rpc'`.
- **멱등**: `sim` + unity 계열 조합은 재적용해도 값이 바뀌지 않는다 — 실제로 `kind`가 전환될 때(hucoms↔sim)만 값이 바뀐다.

`app.js`에서는 `core.js` import 목록에 `alignProtocolToKind`를 추가하고(app.js:53), `captureCameraSourceEdits()` 안에서 `source.kind`를 확정한 직후 이 함수로 `source.protocol`을 재계산한다.

```js
source.kind = $('opt-camera-kind').value === 'hucoms' ? 'hucoms' : 'sim'; // 콤보 선택을 데이터 모델(kind)로 확정.
source.protocol = alignProtocolToKind(source.kind, source.protocol); // kind 계열에 맞춰 protocol 정합(hucoms↔sim 전환 시).
```

`cameraSettingsPatch()`는 이전부터 이미 `protocol: source.protocol`을 patch에 포함하고 있었으므로, 이 로직 변경만으로 정합된 protocol 값이 자연스럽게 `PUT /settings`를 통해 `tools.config.json`에 반영된다. 백엔드는 무변경(`protocol` enum·병합 키는 `toolsConfig.ts:245` 부근에 기존부터 정의돼 있었음).

## 4. 저장 흐름(콤보 → 서버 파일 반영)

1. 사용자가 콤보에서 "시뮬레이터"/"리얼카메라"를 선택 → `change` 리스너가 `captureCameraSourceEdits()`를 호출해 `editableCameraSources` 안의 해당 소스 객체 `source.kind`를 갱신하고, 이어서 `alignProtocolToKind()`로 `source.protocol`을 kind 계열에 맞춰 정합한다(§3.5).
2. 저장 버튼(`opt-save`) 클릭 → `saveSettings()` → `cameraSettingsPatch()`가 `captureCameraSourceEdits()`를 다시 호출(최신 kind·protocol 확정) 후 `{ id, label, kind, protocol, baseUrl, username, rtspUrl, password? }` 형태의 patch 객체를 구성. 이때 `protocol`은 이미 정합된 값이다.
3. `PUT /settings` 요청 body의 `camera.source.kind`·`camera.source.protocol`로 전송.
4. 서버(`settingsStore.ts` `writeEditableSettings`)가 `config/tools.config.json`의 `cameraSources[]` 배열에서 해당 `id`를 찾아 화이트리스트 필드(kind 포함)만 부분 병합·기록. 배열 전체를 덮어쓰지 않으므로 `ptz` 등 편집 대상이 아닌 필드는 보존된다.
5. 저장 성공 시 서버가 `{ ok: true, restartRequired: true }`를 반환하고, 프론트는 `opt-restart-banner`를 노출한다(`config/`는 nodemon watch 범위 밖이라 런타임에 자동 반영되지 않음 — 기존 배너 동작 그대로, 이번 변경으로 새로 생긴 동작 아님).

## 5. 프론트 사전검증 ↔ 백엔드 검증의 관계

두 검증은 검증 시점(저장 버튼 클릭 시 프론트에서 선차단 vs. 서버가 최종 방어)만 다를 뿐 **동일한 규칙**(`kind === 'hucoms'`일 때 RTSP URL 필수 + 프로토콜 형식)을 따른다. 이번 변경으로 이 두 검증 경로 자체는 건드리지 않았고, 콤보가 프론트 검증이 이미 기대하던 `kind` 편집 경로를 실제로 열어준 것뿐이다.

- **프론트**: `web/core.js`의 `settingsFormErrors()` (185~201행 부근)가 저장 전에 `form.camera.source.kind === 'hucoms'`이면 RTSP URL 필수·형식(`rtsp:`/`rtsps:`)을 검사해 에러 메시지를 만든다. 이 함수는 이번 변경 이전부터 이미 `kind` 분기를 처리하고 있었으나, 콤보가 없었던 예전에는 `kind`가 항상 로드된 값 그대로였으므로 이 분기가 실질적으로 발동할 일이 없었다. 콤보 도입으로 이제 사용자가 `hucoms`로 전환한 뒤 RTSP를 비워두면 이 검증이 실제로 저장을 막는다.
- **백엔드**: `src/config/settingsStore.ts`의 `SettingsPatchSchema.superRefine` (51~79행)이 동일 규칙을 서버 측에서 재검증한다. `kind === 'hucoms'`인데 `rtspUrl`이 없으면 `'실카메라 RTSP URL이 필요합니다'` 이슈를 던지고, 있으면 `new URL()`로 파싱해 `rtsp:`/`rtsps:` 프로토콜인지, URL에 계정 정보가 섞여 있지 않은지 검사한다. 이 경로는 이번 변경으로 손대지 않았다 — 애초에 `kind`가 `SettingsPatchSchema.camera.source` 화이트리스트에 `required`로 포함되어 있었고, `writeEditableSettings`의 병합 키에도 `kind`가 들어 있었기 때문에(설계 단계 근거, `settingsStore.ts:32-79`, `:212` 부근) 백엔드는 애초부터 `kind` 편집을 완전히 받아들일 준비가 되어 있었다.

## 6. 테스트 결과

### 6.1 신규: `test/cameraKindSelect.test.ts` (최초 6 → 후속 확장 11/11 통과)

DOM을 실제로 렌더링하지 않는 저장소 규약(`viewerDisplayReset.test.ts` 선례)에 따라, app.js/index.html/core.js 소스 텍스트·순수함수를 단언하는 방식으로 작성됨.

최초 6케이스:
1. `index.html`: `#opt-camera-kind`가 `<select>`이고 `readonly` `<input>`이 아님을 확인.
2. `index.html`: `sim`(시뮬레이터)·`hucoms`(리얼카메라) 두 `<option>`이 존재함을 확인.
3. `renderCameraSource`: `kind`로 콤보 `value`를 세팅하는 대입식이 존재하고, 과거의 표시 문자열("Hucoms 실카메라") 대입이 남아있지 않음(회귀 가드).
4. `captureCameraSourceEdits`: 콤보 선택값을 `source.kind`로 확정하는 대입식 존재.
5. `change` 리스너: `captureCameraSourceEdits()` 호출 후 `renderCameraSource(renderedCameraSourceId)` 재렌더 호출이 리스너 콜백 안에 존재.
6. `cameraSettingsPatch`: 저장 patch에 `kind: source.kind`가 포함됨(백엔드 편집 경로 연결 확인).

후속 추가 5케이스(`alignProtocolToKind` 순수 단위 + 결선):
7. `alignProtocolToKind('hucoms', *)` → 입력 protocol과 무관하게 항상 `'hucoms-v1.22'`.
8. `alignProtocolToKind('sim', 'unity-rpc'|'unity-rest')` → 기존 unity 계열 값을 그대로 유지.
9. `alignProtocolToKind('sim', 'hucoms-v1.22'|undefined)` → 기본값 `'unity-rpc'`로 대체.
10. 멱등성: `sim`+unity 계열, `hucoms`+임의값 모두 재적용 시 값 불변.
11. `captureCameraSourceEdits` 본문에 `source.protocol = alignProtocolToKind(source.kind, source.protocol)` 대입식이 `source.kind` 확정 직후에 존재함(결선 가드).

### 6.3 한계(은닉 없이 명시)

- 실제 브라우저 DOM 렌더링·클릭 상호작용 테스트는 이 저장소에 구성되어 있지 않다(jsdom 미구성). 위 테스트는 소스 텍스트/순수함수 정적 단언으로 결선(로드→콤보 반영, 콤보 change→capture+protocol 정합+재렌더, 저장 patch 포함)만 가드하며, 실 화면에서 콤보 클릭 시 픽셀 단위로 올바르게 그려지는지는 서버 구동 후 육안 확인이 별도로 필요하다.

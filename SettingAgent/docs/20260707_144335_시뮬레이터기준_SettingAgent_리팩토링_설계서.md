## ① 주차면 바닥면 그리기 (Floor ROI)

### 1-1. 원시 요구 요약
- 바닥면(ROI) 그리기에 **LLM 사용여부 토글**을 UI 에 추가.
- 미사용 시 **기존 시뮬레이터에서 바닥면(ROI) 정보를 수신**한다.
- 사용(체크) 시 기존대로 LLM 사용(리얼 카메라 촬영 시 필요).

### 1-2. 현황 (근거)
- LLM 비전 경로: `src/capture/FloorRoiReviewer.ts` → `brain.recognizeFloorRoi()`(`SetupBrain.ts` `FloorRoiInput/FloorRoiResult`). 좌표를 "생성"하는 유일한 LLM 단계.
- 결정형 폴백: `src/capture/floorRoi.ts` `resolveFloorPolygon()` — LLM 무효 시 `buildPlateAnchoredQuad()`(번호판 앵커 사변형) + `expandPolygonToContainRect()`(번호판 포함강제). **항상 폴리곤을 보유**한다(권한역전 후 LLM 메인, 폴백은 번호판앵커).
- 토글은 현재 **전역 config**(`llm.config.json` `floorRoi.enabled`)뿐 — **런타임/UI 토글 없음**.
- **시뮬레이터에서 ROI 를 직접 받는 경로는 존재하지 않는다.** `SimulatorSource.ts`(=`CameraClient`)는 `snapshot`/`move`/`listCameras` 만 제공. floor 관련 API 없음.

### 1-3. 목표 동작
- 웹 UI 에 "바닥면 LLM 사용" 체크박스. 정밀수집 시작(`/capture/start`) 시 토글값 전달.
- 토글 ON → 기존 LLM 경로(리얼 카메라).
- 토글 OFF → 시뮬레이터에서 프리셋별 바닥면 ROI(지면 접지 폴리곤)를 수신해 그대로 채택(LLM·앵커 폴백 미사용).

### 1-4. 설계 변경 — 추가/변경/제거
**추가**
- `CameraClient`(또는 신규 `SimulatorSource.getFloorRoi`)에 **시뮬레이터 floor ROI 수신 메서드** 신설.
  - 제안 계약(가정 — 시뮬레이터 측 미구현): `POST /req_floor_roi { cam_idx, preset_idx }` → `{ cam_idx, preset_idx, rois:[{ slot_hint?, points:[[x,y]×4..] }] }`. 좌표는 픽셀 또는 정규화(**확인 필요**, 1-7 참조).
- `StartBodySchema`(`captureRoutes.ts`)에 `floorRoiUseLlm?: boolean` 추가 → `CaptureJob` → `FloorRoiReviewer.review()` 로 전달.

**변경**
- `FloorRoiReviewer.review()`: `useLlm===false` 이면 LLM 호출을 생략하고 **시뮬 수신 ROI 를 `upsertFloorRoi` 로 저장**하는 분기 추가. 시뮬 ROI 부재 시에만 기존 번호판앵커 폴백(누락 방지).
- 웹: floor LLM 경고 모달(`floor-llm-warn-modal`)은 **LLM 모드일 때만** 노출(OFF 모드에선 폴백이 정상 동작이므로 경고 억제).

**제거**: 없음(기존 LLM·폴백 경로 보존 — 리얼 카메라용).

### 1-5. 영향 파일·경계
- src: `capture/FloorRoiReviewer.ts`, `capture/CaptureJob.ts`, `api/captureRoutes.ts`, `clients/CameraClient.ts`(or `viewer/SimulatorSource.ts`).
- web: `index.html`(체크박스), `app.js`(토글 상태→start 바디).
- config: 없음(런타임 토글). REST 계약: `/capture/start` 바디 확장(하위호환 optional), 신규 시뮬 `/req_floor_roi`(외부 계약).
- 경계: 시뮬 ROI 수신=**결정형**(REST fetch, LLM 무관). LLM 경로=두뇌(불변).

### 1-6. 리팩토링 단계
1. 시뮬 floor ROI 계약 확정(1-7 해소) → `CameraClient.getFloorRoi()` 스텁 + 정규화 변환. **검증**: 목 응답으로 정규화 폴리곤 반환 유닛테스트.
2. `FloorRoiReviewer` 비LLM 분기 + 폴백 우선순위(시뮬>앵커). **검증**: `useLlm=false` + 목 시뮬 ROI → `upsertFloorRoi` 호출 인자 검증, 시뮬 부재 시 앵커 폴백.
3. `/capture/start` 바디·`CaptureJob` 전달 배선. **검증**: start 바디에 토글 넣으면 reviewer 로 전파되는지 통합 테스트.
4. 웹 체크박스 + 경고 모달 조건화. **검증**: OFF 모드에서 경고 미노출(수동 확인).

### 1-7. 미해결/확인 필요
- **[必] 시뮬레이터 floor ROI 엔드포인트 명세 부재**: 경로·요청/응답 shape·좌표계(픽셀 vs 정규화)·1면당 점 수(4점 고정 vs 가변)가 미정. 위 `/req_floor_roi` 는 **가정 제안**. 시뮬레이터 팀 확인 필요.
- 시뮬 ROI 를 어떤 슬롯(`slotId`/`clusterId`)에 매칭할지: VPD 검출 슬롯과의 대응 규칙 미정(위치 최근접? slot_hint 제공?).

## ② 주차면 점유 판정 (Occupancy)

### 2-1. 원시 요구 요약
- 점유 판정에 **LLM 사용여부 토글**. **LLM 없이도 점유 판정 가능**하게.
- **예외(차량을 가리는 오브젝트=차량·사람 발생) 시에만 LLM 이 판단**.
- 파이프라인: **VPD 차량검출 → LPD 번호판검출 → (차량은 있는데 번호판 box 없으면 차량중심 zoom-in 후 LPD 재검출 → 검출되면 원위치로 zoom-out + 검출 번호판 위치를 역계산해 번호판 위치 추정, 기존 번호판 크기·위치와 비교) → 점유 판정(camid, presetid, slotid, is_occupy)**.
- 점유영역은 **작은 원으로 표시만**(판정이 중요).

### 2-2. 현황 (근거)
- **두 지표가 공존**(혼동 주의): (a) 결정형 `occupancyRate`(`Aggregator`, 슬롯이 검출된 라운드 비율, LLM 무관), (b) **LLM 점유율 판정**(`OccupancyReviewer`, 프리셋별 "M면 중 N면 점유"). 원시요구 ②는 (b)에 해당하나, 결정형 경로 신설은 (a)와도 연계 가능.
- `src/capture/OccupancyReviewer.ts`: **LLM 전용**(`brain.judgeOccupancy`). **폴백 없음** — LLM 불가/실패 시 graceful skip(저장 생략) + 경고 플래그. 산술(occupiedCount/total/rate)은 결정형이나 면별 점유 플래그는 전적으로 LLM. 프롬프트 `occupancy.yaml` 은 **절대픽셀 `points_2d`** 로 전 프리셋 면 국소화+occupied 판정.
- 저장 shape: `insertOccupancy(runId,{camIdx,presetIdx,atRound,occupiedCount,total,rate,spacesJson})`. 프리셋 단위 "M면 중 N면 점유". **slotId·is_occupy per-slot 개념 아님**. confidence 미영속.
- 점유영역 표시: 현재 **반투명 채움 박스**(`web/app.js drawOccupancyOverlay` — `strokeRect+fillRect` + id 라벨, 점유=빨강 `#ff4d4d`·공차=초록 `#39ff14`). `web/core.js` `occupancyByKey`/`occupancyRows`/`occupancyAverage` 로 표 렌더. 수집 중 라이브 미표시(최종화 후). → 원시요구의 "작은 원"과 **불일치**(교체 대상).
- zoom-in 재검출→역산 흐름: occupancy 에는 **없음**. 유사 패턴은 `PtzCalibrator.captureAndDetect`(명령 PTZ override 캡처→LPD→`pickNearestPlate`)에 존재 → **재사용 가능**.

### 2-3. 목표 동작
- 토글 OFF(기본): **결정형 점유 판정**. 프리셋 프레임에서 VPD→LPD 검출 → 검출 차량/번호판을 셋업 아티팩트의 슬롯 ROI 에 매핑 → 슬롯별 `is_occupy` 산출.
- 번호판 미검출 차량: 차량중심 **zoom-in** 재캡처→LPD 재검출 → 성공 시 **zoom-out 복귀 + 번호판 정규화 위치 역산**(zoom 배율 기반 역투영), 기존 번호판(setup 저장 prior)과 크기·위치 비교로 타당성 확인.
- **예외(가림)** 판정 시에만 LLM(`judgeOccupancy`) 호출.
- 저장·표시: 슬롯별 `{camId, presetId, slotId, isOccupy}` + 점유 표시는 **작은 원**(슬롯/번호판 중심)만.

### 2-4. 설계 변경 — 추가/변경/제거
**추가**
- `src/capture/occupancyLogic.ts`(신규, 결정형 순수 모듈): VPD/LPD 검출 + 슬롯 ROI → `is_occupy` 판정. `controlMath.ts`/`floorRoi.ts` 처럼 외부 의존 0, 유닛테스트 대상.
- zoom-in 재검출·역산 서브루틴: `PtzCalibrator.captureAndDetect`·`pickNearestPlate` 패턴 재사용. 역산=`plateInverseProject(observedPlateNorm, zoomIn, zoomBase)`(신규 결정형 함수).
- `is_occupy` 스키마 + 저장: `SqliteStore` 에 슬롯 점유 컬럼/테이블(또는 기존 occupancy 행 확장 `slotId`,`isOccupy`).
- `StartBodySchema` 에 `occupancyUseLlm?: boolean`.

**변경**
- `OccupancyReviewer.review()`: (a) 결정형 경로 우선, (b) 가림 예외 감지 시에만 LLM. `usable && useLlm` 게이트를 예외 트리거로 강등.
- 저장·조회 라우트 `/capture/runs/:id/occupancy`: 반환 shape 에 slot 단위 `isOccupy` 포함.

**제거/단순화**
- 점유영역 **채움 박스(strokeRect+fillRect) 표시 제거** → `web/app.js drawOccupancyOverlay` 를 **작은 원 마커**(슬롯/번호판 중심 1점 + 반지름 상수)로 교체. occupancy `spaces[].points_2d`(4점) 생성·저장은 표시 목적 한정으로 **축소**(판정=is_occupy 만 유지).
- (floor ROI 다각형 기능 자체는 ①영역 소관으로 유지 — occupancy **표시**만 원으로 단순화. 2-7 확인 필요.)

### 2-5. 영향 파일·경계
- src: `capture/OccupancyReviewer.ts`, 신규 `capture/occupancyLogic.ts`, `capture/SqliteStore.ts`, `capture/CaptureJob.ts`, `api/captureRoutes.ts`, (역산은 `calibrate/controlMath.ts` 인접 또는 신규).
- web: `core.js`(occupancy 표시 로직→원), `app.js`/`index.html`(occupancy 토글·원 렌더).
- 경계: VPD/LPD 검출·매핑·zoom 재검출·역산·is_occupy=**결정형 도구**. **가림 예외 판단만 LLM 두뇌**.

### 2-6. 리팩토링 단계
1. `occupancyLogic.ts` 결정형 판정(슬롯 ROI ∩ 차량/번호판 → isOccupy). **검증**: 합성 검출+슬롯으로 점유/비점유 케이스 유닛테스트.
2. zoom-in 재검출 + `plateInverseProject` 역산. **검증**: 목 카메라/LPD 로 번호판 미검출→zoom-in 성공→역산 좌표가 기대 범위 내 유닛테스트.
3. `SqliteStore` slot 점유 저장 + 조회 라우트 shape 확장. **검증**: insert/select 왕복 + 라우트 응답 shape.
4. `OccupancyReviewer` 결정형 우선 + 가림 예외 LLM. **검증**: 가림 없음→LLM 미호출, 가림 감지→`judgeOccupancy` 1회 호출 모킹.
5. 웹 원 마커 표시 + 토글. **검증**: 판정 결과가 원으로 표시(수동), 가변 다각형 미표시.

### 2-7. 미해결/확인 필요
- **[必] 역산(inverse projection) 좌표계 정의**: zoom-in 프레임의 번호판 정규화 좌표 → zoom-out(원 프리셋) 정규화 좌표 환산식. 시뮬 zoom 이 **중심확대(FOV 선형)** 가정이면 `p_base = 0.5 + (p_in − 0.5) × (zoomIn/zoomBase)⁻¹` 수준으로 근사 가능하나, 시뮬 zoom 모델(중심고정 여부·팬틸트 결합) 확인 필요.
- **[必] 가림 예외 감지 기준**: "차량/사람이 가림"을 결정형으로 어떻게 트리거? (VPD bbox 상호 겹침 임계? 번호판 zoom 재시도까지 실패? person 클래스 검출?) — 원시요구 모호.
- **[必] slotId 매핑 원천**: 점유 판정의 슬롯은 setup 아티팩트 `slots[]` ROI 기준인가, 프리셋별 신규 검출인가. (camid/presetid/slotid 출력 형식은 setup 슬롯 인덱스 재사용 가정.)
- 정밀수집 라운드 중 zoom-in 은 그 프리셋의 활성 화면을 흔든다 — 다른 리뷰어(floor/occupancy)가 쓰는 `lastFrameByPreset` 오염 우려. 별도 캡처(override PTZ, 상태 미갱신)로 격리 필요(PtzCalibrator 방식 차용).
- occupancy 표시를 "작은 원"으로 단순화할 때, 기존 floor ROI 폴리곤 표시와의 **UI 공존/전환** 방침(둘 다 표시? 원만?) 확인 필요.

## ③ 차량 번호판 중심 센터라이징 (Centering)

### 3-1. 원시 요구 요약
- 카메라를 **번호판 중심으로 pan/tilt 만** 이동 → 번호판이 이미지 중심.
- **LLM 만 가능하면 LLM, 로직 가능하면 로직 — 두 채널 모두 사용 가능**.
- zoom 을 점점 확대해 **번호판 가로크기가 이미지의 20%** 될 때까지.

### 3-2. 현황 (근거)
- `src/calibrate/PtzCalibrator.ts` + `controlMath.ts`: **이미 요구를 충족**한다.
  - (A) pan/tilt 중심정렬(`plateCenterError`→`panTiltCorrection`, 목표 (0.5,0.5)) → (B) zoom 폭 정렬(`zoomCorrection`, `targetPlateWidth`).
  - `tools.config.json calibrate.targetPlateWidth=0.2`(=20%), `centerTol=0.03`, `widthTol=0.02` **이미 존재**.
  - **이중채널**: 결정형 비례제어(주 엔진) + LLM 자문(`brain.adviseCentering`, `ptz_centering.yaml`). `calibrate.llmAdvise` 토글. LLM off/실패 시 순수 결정형 폴백.
  - 순서 엄수(pan/tilt 수렴 후 zoom)·probe 게인추정·명령 PTZ 추적(시뮬 응답 PTZ=0/0/1 echo 라 명령값만 신뢰) 모두 구현됨.
- 산출물: `data/slot_ptz.json`(`slotPtzWriter.ts`). 라우트 `POST /calibrate/ptz`·`GET /calibrate/status`·`GET /calibrate/result`(`calibrateRoutes.ts`).
- 참고: `FloorRoiInput` 은 최신 코드에서 plate/plateQuad 신호가 **제거**됨(LLM 은 이미지만, 번호판 앵커는 폴백 전용). ②/③에서 LLM 입력 계약 변경 시 유의.

### 3-3. 목표 동작
- **기존 PtzCalibrator/calibrate config 재사용**. 신규 센터라이징 모듈을 만들지 않는다(중복 금지, CLAUDE.md §3).
- UI 에서 LLM 자문 토글(`llmAdvise`) 노출·제어(선택). 필요 시 슬롯 단건 센터라이징 트리거 노출.

### 3-4. 설계 변경 — 추가/변경/제거
**추가**: (원칙적으로 없음) UI 에 centering LLM 토글 스위치(선택).
**변경**: 없음 또는 최소. `calibrate.llmAdvise` 를 웹 옵션(④)에서 편집 가능하게 노출.
**제거**: 신규 중복 센터라이징 로직 **금지**(이미 존재).

> **정합성 판단**: 원시요구 ③은 현행 PtzCalibrator 로 **이미 만족**. 이 영역은 "신설"이 아니라 "재사용 확인 + UI 노출"이 실질. 별도 코드 변경 최소화.

### 3-5. 영향 파일·경계
- src: (변경 최소) `calibrate/*` 그대로. 옵션 편집 시 `calibrate.llmAdvise`.
- web: 선택적 토글 노출.
- 경계: 결정형 제어(주) + LLM 자문(토글) — **이미 올바른 경계**.

### 3-6. 리팩토링 단계
1. 원시요구 ③ ↔ PtzCalibrator 커버리지 매핑 문서화(신규 코드 불요 확인). **검증**: targetPlateWidth=0.2·pan/tilt-only·이중채널 존재를 테스트/코드로 대조.
2. (선택) UI 토글 노출. **검증**: 토글이 `/calibrate` 자문 on/off 로 반영.

### 3-7. 미해결/확인 필요
- 센터라이징을 **뷰어에서 슬롯 단건 즉시 실행**(액션)으로 원하는지, 기존 배치 `/calibrate/*`(전체 슬롯) 유지인지 확인. 원시요구 문맥("카메라를 …이동시킨다")은 단건 액션 뉘앙스.
- ②의 zoom-in 재검출과 ③의 zoom 센터라이징은 목적이 다르나(판정용 vs 캘리브레이션용) 로직이 유사 → 공용 헬퍼(`captureAndDetect`/zoom 제어) 추출 여부(과설계 경계 유의, 중복이 2곳 이상일 때만).

## ⑤ LPR API 차량번호 검출 (Recognition, 확인용)

### 5-1. 원시 요구 요약

### 5-2. 현황 (근거)

### 5-3. 목표 동작

- 미인식/무효 시 zoom 을 목표(**번호판 가로 25%**)까지 확대하며 LPR 재시도.

### 5-4. 설계 변경 — 추가/변경/제거
**추가**

- 번호 형식 유효성 검사(결정형): 한국 번호판 정규식(예: `\d{2,3}[가-힣]\d{4}`) 또는 길이/문자 검증.

**변경**

**제거**: 없음.

### 5-5. 영향 파일·경계

### 5-6. 리팩토링 단계
1. `lpr` config 스키마·기본값. **검증**: 로드/파싱 유닛테스트.

4. zoom 25% 재시도 루프(목 카메라/LPR). **검증**: 미인식→zoom 확대 재시도→인식 성공/상한 종료.

### 5-7. 미해결/확인 필요
- **[必] LPR 엔드포인트/경로/응답 스키마 미정**: `recPath`, 응답 필드(plate/text/confidence). LPD(`/lpd/api/v1/imgupload`) 대응하는 LPR 실서버 사양 확인 필요.

13. **테스트 부채**: 신규 결정형 모듈(occupancyLogic·plateInverseProject·LprClient·settingsStore) 유닛테스트 필수(CLAUDE.md §2). 외부 REST(LPR/시뮬)는 모킹.

### D. 단순함 우선 권고 (과설계 방지)
- ③ 센터라이징은 **신규 코드 거의 불필요**(기존 충족) — "재구현" 유혹 배제.
- ④ 옵션 페이지는 **핫리로드 미도입**(파일쓰기+재시작 안내)로 최소화.
- ②의 zoom 제어·역산은 ③/⑤와 중복될 수 있으나, **실제로 2곳 이상에서 중복될 때만** 공용 헬퍼 추출(선제적 추상화 금지).

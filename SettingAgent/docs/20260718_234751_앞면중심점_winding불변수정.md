# 앞면 중심점(slot3d_front_center) 버그수정 — 감김순서(winding)-불변화

작성: 문서화 담당(documenter) · 작성일시: 2026-07-18 23:47
근거 산출물: `_workspace/02_developer_changes.md`("[후속] 앞면 기하 자동판정 수정" 섹션) · `_workspace/03_qa_report.md`("[후속 검증]" 섹션) · 실제 diff(`web/core.js`, `src/ground/project.ts`)

**상호참조**: 이 문서는 어제 작성된 원 기능 문서 `docs/20260718_224213_주차면3D육면체앞면중심점_slot3d_front_center.md`의 **후속 버그수정**이다. 원 문서 §1(Q1)·§2·§6에 기술된 `FRONT_FACE_IDX=[0,3,7,4]` 고정 상수 방식이 이번 수정으로 감김순서-불변 로직으로 교체되었으며, 원 문서의 나머지 내용(DB 스키마, REST 계약, Finalizer 배선, 마이그레이션)은 전부 그대로 유효하다.

## 1. 증상

프리셋 2 웹 오버레이에서 앞면 중심점(흰 점)이 슬롯 8번에 0개, 12번에 2개로 오귀속되어 표시되었다(리더 육안 보고).

## 2. 근본 원인

앞면 4-corner를 **고정 상수 `FRONT_FACE_IDX = [0,3,7,4]`** 로 골랐던 것이 원인이다. 이 상수는 원 기능 설계(Q1) 당시 프리셋 1의 바닥 corner 감김순서(winding) — `points = [근좌, 원좌, 원우, 근우]`, 근접(앞) edge = `p0–p3` — 를 암묵적으로 가정하고 있었다.

그런데 프리셋 2의 데이터는 감김순서가 한 칸 회전되어 있다 — `points = [근우, 근좌, 원좌, 원우]`, 근접(앞) edge = `p0–p1`. 이 경우 고정 상수 `[0,3,7,4]`는 앞(근접)면이 아니라 **오른쪽 측면**을 가리키게 된다. 결과적으로 모든 앞면 중심점이 우측으로 균일하게 밀려 나갔고, 그 경계 드리프트가 최좌측 슬롯(8번)은 점을 잃고(0개) 이웃 슬롯(12번)은 점을 두 개 받는(2개) 현상으로 나타났다.

이 계산은 웹 표시(`web/core.js`)뿐 아니라 DB 저장값(`slot3d_front_center` / `frontFaceCenterPx`)에도 동일하게 반영되므로, 저장 단계의 데이터도 함께 오염되어 있었다.

## 3. 수정 — 감김순서-불변 기하 판정

고정 인덱스 방식을 폐기하고, **바닥 4개 edge `[0,1], [1,2], [2,3], [3,0]` 중 두 끝점의 이미지 y좌표 평균이 최대인 edge**를 앞모서리로 선택하는 방식으로 교체했다. 하향 틸트 카메라에서는 화면 아래쪽(y가 큼)이 카메라에 가장 가까운(=앞) 위치이기 때문에, 이 판정은 corner가 어떤 순서로 나열되어 있어도(어떤 winding이어도) 항상 실제 근접면을 고른다.

선택된 두 바닥 corner를 `a, b`라 하면 앞면 4점은 `[a, b, a+4, b+4]`(바닥 2점 + 대응 상면 2점)이고, 중심은 이 4점의 산술평균이다. 판정은 바닥 corner의 y값만 사용한다(상면 corner는 카메라에서 멀어질수록 화면에서 작아지고 위로 올라가므로 판정에 쓰지 않는다).

### 변경 파일 (2곳, 완전히 동일한 로직 + 교차참조 주석)

| 파일 | 변경 내용 |
|---|---|
| `web/core.js` | 상수 `FRONT_FACE_IDX=[0,3,7,4]` 제거. `CUBOID_BOTTOM_EDGES` + 순수 헬퍼 `frontFaceCornerIdx(bottomY)` 추가. `frontFaceCenter(cuboid)`가 바닥 corner 0..3의 y값으로 근접-edge를 판정한 뒤 그 4개 corner(바닥 2 + 대응 상면 2)의 평균을 반환하도록 변경. |
| `src/ground/project.ts` | export 상수 `FRONT_FACE_IDX` 제거. `BOTTOM_EDGES` + `frontFaceCornerIdx(bottomY)` 추가. `frontFaceCenterPx(corners)`가 픽셀 y값으로 동일 판정 수행. |

**시그니처는 양쪽 모두 완전히 불변**이다 — `frontFaceCenter(cuboid)`, `frontFaceCenterPx(corners)`의 입출력 타입과 null 가드 규칙(corners<8, 비유한 좌표 → null)은 그대로다. 이 덕분에 호출부인 `web/app.js`, `src/capture/Finalizer.ts`는 수정할 필요가 없었다(외과적 변경).

부수적으로, `src/ground/project.ts`의 `FRONT_FACE_IDX` export 상수는 어디에서도 import되지 않던 **고아 코드**였음이 확인되어 이번에 함께 정리되었다.

## 4. 동작 확인

**리더 경험적 검증(브라우저 라이브)**: 수정된 `core.js`로 프리셋 2를 재렌더한 결과, 흰 점 6개가 각 슬롯 앞면 중심에 1개씩 정확히 표시됨을 확인했다(슬롯 8번 x≈177로 정상 표시, 12번의 중복 2개 문제 해소). 프리셋 1·3은 렌더 결과에 변화 없음(무회귀).

신규 테스트 5종 요지(동일 물리 사각형에 winding만 회전시켜 검증):
1. **winding-불변**: corner 나열 시작점을 0~3칸 회전시켜도 `frontFaceCenter` 결과가 4가지 회전 전부 `(0.500000, 0.657277)` — 1e-9 이내 완전 동일.
2. **앞면=근접면 + 옛 로직 반례**: 프리셋2형 회전(k=3)에서 새 로직 `x=0.500000`(근접면 중심, 근접 edge=[0,1] 정확 선택) vs 옛 고정 `[0,3,7,4]` 로직 `x=0.591584`(측면으로 밀림) — 버그가 실제로 재현됨을 수치로 확인.
3. **파리티(회전 포함)**: 4가지 회전 모두에서 `core.js`(정규화)와 `project.ts`(픽셀→정규화)가 1e-6 이내 일치 — 표시=저장 파리티가 winding에 무관하게 유지됨.
4. **프리셋1 무회귀**: 프리셋1 winding에서 새 로직과 옛 `[0,3,7,4]` 결과가 1e-12 이내 완전 동일 — 프리셋 1·3 결과가 이번 수정으로 바뀌지 않음을 봉인.
5. **실데이터 스모크**: `data/Place01/PtzCamRoi.json` 프리셋2(idx 8~13) 실제 슬롯 quad + 라이브 서버 지면모델(`GET /capture/ground-model`)로 검증. 6개 슬롯 전부 근접 edge=`[0,1]`(회전형 winding)로 판정되었고, 새 로직의 앞면중심 x가 6/6 모두 **자기 슬롯의 x범위 안**에 들어왔다. 옛 로직은 매 슬롯 x를 +0.148~+0.152만큼 우측(측면/이웃 방향)으로 밀어냈다 — 보고된 "8번 0개/12번 2개" 오귀속의 기하적 원인과 정합.

## 6. 영향도 분석

- **변경 파일은 2개뿐**: `web/core.js`, `src/ground/project.ts`. `app.js`, `Finalizer.ts`, `SqliteStore.ts`, DB 스키마, `types.ts`는 **전부 무변경**이다(양쪽 함수 시그니처가 그대로이기 때문).
- **DB 재계산 필요(중요)**: 라이브 DB에 이미 저장되어 있는 프리셋2 슬롯(idx 8~13)의 `slot3d_front_center` 값은 옛(측면) 로직으로 계산된 값이라 **여전히 오염된 상태**로 남아 있다. 다만 `slot3d_front_center`는 검출과 무관한 순수 기하 소스라 **매 finalize마다 새로 계산되어 덮어써진다** — 별도 마이그레이션 스크립트 없이 **다음 finalize 실행 시 자동으로 교정**된다. 프리셋 1·3의 기존 값은 이번 수정으로 값이 바뀌지 않으므로(위 무회귀 테스트로 확인) 영향 없다.
- 프리셋별로 winding이 다른 것은 코드 문제가 아니라 데이터(Unity export 순서) 자체의 특성이다 — 이번 수정으로 코드가 감김순서에 대해 불변이 되었으므로, 향후 어떤 순서의 winding이 들어오더라도 안전하다.
- 어제 작성된 원 기능 문서(`20260718_224213_...md`)의 §1(Q1)·§2·§6에 기술된 `FRONT_FACE_IDX` 고정 상수 서술은 이번 수정으로 대체되었다 — 그 문서를 참조할 때는 본 문서를 후속 수정으로 함께 참고할 것.

## 확인 필요(단정 보류)

- 실 서비스 DB(`data/setting.sqlite`)의 프리셋2 슬롯이 실제로 다음 finalize에서 교정되는지는 이번 세션에서 라이브 finalize 실행으로 재확인하지 않았다 — 로직상(순수 기하 재계산) 자동 교정되어야 하나, 운영 환경에서의 실제 재계산 시점은 별도 확인 대상이다.

## 1. 신규·변경 함수

### 1-1. `web/core.js` — 신규 순수함수 4종 (DOM/fetch 미참조, 불변·throw 금지)

| 함수 | 시그니처 | 역할 |
|---|---|---|
| `normalizeGlobalIdx` | `(placeRoi) → { placeRoi, changed, issues[] }` | **전역번호 정규화(마이그레이션 진입점)**. idx 집합이 정확히 `1..N` 의 순열이면 **손대지 않고 그대로 반환**(`changed:false`) — 사용자가 재지정한 번호 보존. 중복·0-based·누락·비정수면 `(cam asc → preset asc → 배열순)` 으로 **1..N 재부여**(`changed:true`) |
| `reindexPlaceSpace` | `(placeRoi, fromIdx, toIdx) → placeRoi` | **'수정'**. 전역 시퀀스에서 대상을 뽑아 목표 위치에 삽입 후 1..N 재부여(밀어내기). **프리셋 소속·좌표·배열 순서 불변 — idx 값만 갱신**. `toIdx` 는 1..N clamp, 없는 `from`/비수치 `to`/`from===to` → 원본 |
| `removePlaceSpace` | `(placeRoi, idx) → placeRoi` | **'삭제'**. 제거 후 상대순서 유지한 채 1..N 재압축. **프리셋이 비어도 키를 `[]` 로 유지**(저장 시 그 프리셋을 `spaces:[]` 로 PUT 해야 하므로 키 삭제 금지) |

## 2. 인덱스 체계 변경 (이번 작업의 본질)

### 2-1. 이전 → 이후

| | 이전 | 이후 |
|---|---|---|
| `parking_spaces[].idx` 의 의미 | **프리셋별 0-based 순번** (프리셋 간 중복) | **파일 전체 고유 전역번호 1..N** |
| 프리셋 내 순번 | idx 가 겸함 | **`parking_spaces` 배열 위치**로 분리 |
| JSON 스키마(키 집합) | `cameras / camera,presets / preset_idx,parking_spaces / idx,points` | **동일 — 변경 없음** |

실데이터(`data/Place01/PtzCamRoi.json`) 기준: 이전 `1:1=[0..6] · 1:2=[0..5] · 1:3=[0..3]`(중복) → 이후 `1:1=1..7 · 1:2=8..13 · 1:3=14..17` (총 17면).

### 2-2. 왜 스키마를 안 바꾸고 의미만 바꿨나

`PtzCamRoi.json` 은 **Unity(`CParkingSpace3DTo2D.cs`)가 생성하는 파일**이다. 스키마에 새 필드를 넣으면 Unity 측 동시 변경이 필요하므로, **기존 `idx` 필드의 해석만 전역번호로 승격**했다. 대신 §4-1 의 Unity 재생성 위험을 떠안는다.

### 2-3. 멱등성이 왜 중요한가

`normalizeGlobalIdx` 는 "**이미 1..N 순열이면 절대 손대지 않는다**". "항상 재부여"가 코드는 짧지만 **사용자가 '수정'으로 지정한 번호를 매 로드마다 파괴**한다. 정확성을 위해 멱등 분기를 명시적으로 둔 것이며, QA가 이 성질을 회귀 감시 테스트로 고정했다.

### 3-2. DB 태그 게이트 — QA가 잡은 결함과 수정 (High)

**결함**: `buildFlatSlotRows` 가 DB 행을 `slotIdx === globalIdx` 로 **행별 매칭**했다. 구 run 의 `slot_idx` 집합 `{0..6}` 과 신 전역번호 `{1..7}` 이 **1..6 에서 부분 겹침** → DB 의 `slot_idx=1`(실제로는 배열 위치 1 = 신 번호 **2**)이 전역번호 **1** 행에 붙어 **모든 태그가 한 칸 밀려 오귀속**되고, `slot_idx=0` 데이터는 소실됐다. 크래시가 없어 **사용자가 인지할 수 없는 정보 오염**이었다.

**수정**(core.js:588~591) — **프리셋 단위 all-or-nothing 게이트**:

```js
const fileIdx = new Set(spaces.map((sp) => sp.idx));
const usable = dbRows.length > 0 && dbRows.every((r) => fileIdx.has(r.slotIdx)) ? dbRows : [];
```

DB 행 집합이 파일 전역번호 체계에 **전부 포함될 때만** 태그를 채택하고, 부분 겹침이면 **그 프리셋의 DB 행을 통째로 기각** → 파일 계산 점유로 폴백(태그 미부착). "틀린 값을 진실처럼 표시"하는 대신 "태그가 없다"로 강등한다(graceful degradation).

**근본 해소**(§1-3): `Finalizer` 가 서버측에서 직접 정규화하므로 **신규 run 은 항상 전역번호로 기록**된다. 사용자가 뷰어에서 '저장'을 누르기 전(Unity 생성 0-based 파일)에 최종화해도 태그가 정합한다 — **운영상 사전 '저장' 강제 불필요**.

### 3-3. '저장'은 전 프리셋 **직렬** PUT (병렬 금지)

```js
for (const key of Object.keys(state.placeRoi)) {   // ★ 반드시 await 직렬
  await fetch('/capture/place-roi', { method:'PUT', body: JSON.stringify({ camId, presetIdx, spaces }) });
}
```

`PUT /capture/place-roi` 라우트는 매 요청 `readFile → applyPlaceRoiUpdate → writeFile` 을 수행한다. **병렬로 보내면 read-modify-write 경합으로 앞선 프리셋의 갱신이 유실**된다. 실패 시 즉시 중단 + `#place-msg` 에 실패 프리셋 명시(부분 저장 상태를 숨기지 않음), `placeRoiDirty` 유지. **자동 롤백은 없다.**

## 5. 영향도 분석

→ 상세는 `_workspace/04_doc_impact.md`. 아래는 요지.

### 5-1. Unity 연동 위험 — **최상위**

`Parking3D/Assets/Scripts/02_ParkSimulator/CParkingSpace3DTo2D.cs` 가 `PtzCamRoi.json` 을 **생성**한다(`parkingSpace.idx = space.index`, 파일이 있으면 **해당 cam_id·preset_idx 항목만 교체/추가**). Unity가 파일을 재생성하면 idx 가 **프리셋별 0-based 로 리셋**되어 **사용자가 지정한 전역번호가 유실**된다.

- **완화**: 로드/최종화 시 `normalizeGlobalIdx` 가 중복을 감지해 **graceful 재부여**(1..N) — 데이터 손상·크래시 없음.
- **완화 불가**: **사용자 커스텀 번호는 복원 불가.** 스키마 무변경 제약의 필연적 귀결.
- **주의**: Unity 는 **일부 프리셋만** 교체할 수 있다. 한 프리셋만 0-based 로 덮여도 파일 전체가 1..N 순열을 잃으므로 **전 프리셋이 재부여**된다(부분 손상이 아니라 전체 재번호).
- **운영 수칙**: **Unity 에서 PtzCamRoi 를 재생성한 뒤에는 전역번호를 재지정해야 한다.**

### 5-2. 기존 DB run 정합 — 마이그레이션 없음(의도)

구 run 의 `slot_idx`(프리셋별 0-based)는 **마이그레이션하지 않는다**. §3-2 게이트에 의해 그 run 의 태그는 **통째 기각 → 미부착**되고, 점유는 파일 계산으로 정상 표시된다. **오귀속은 발생하지 않는다.** 재최종화하면 정합된다.

### 5-3. 규칙 이중구현 — **향후 수정 시 반드시 양쪽 + 파리티 테스트**

`normalizeGlobalIdx` 규칙이 **뷰어(`web/core.js`)와 서버(`src/capture/placeRoi.ts`) 두 곳에 물리적으로 존재**한다(브라우저가 TS 서버 모듈을 import 할 수 없어 단일 소스 공유 불가).

- `test/globalIdxParity.test.ts` 가 **동일 raw JSON → 동일 전역번호**를 강제하므로 **한쪽만 고치면 CI 가 실패**한다.
- QA 뮤테이션 검사: 서버 정렬키를 `preset asc → desc` 로 **1글자** 바꾸자 파리티 테스트 **4/9 즉시 실패** → 판별력 실증.
- **규칙 변경 시 반드시 `core.js` + `placeRoi.ts` + `globalIdxParity.test.ts` 를 함께 갱신할 것.**

### 5-4. `idx` 소비자 — 영향 없음(근거)

`selectFloorRoi`(라벨 표시만) · `computeOccupancy`(pass-through) · `transformPlaceRoiPreset`/자동보정 `alignApply`(좌표만 변환, idx 보존 PUT) · `SqliteStore` PK(상위집합이라 안전) · `loadDetectCfg`(fov 만 사용). **깨지는 코드 0곳.**

### 5-5. "전역 인덱스" 동음이의 2체계 (혼동 주의)

| 체계 | 소스 | 사용처 |
|---|---|---|
| `artifact.globalIndex` | `GlobalIndexer`(setup_artifact 의 `slotId` 기준) | `slotPtzWriter` → **`slot_ptz.json`** · PTZ 캘리브레이션 |
| `PtzCamRoi.idx`(**신규**) | PtzCamRoi.json 파일 | 정밀수집 주차면 목록 · DB `parking_slots.slot_idx` |

**서로 무관한 별개 체계**다. `slot_ptz.json` 은 `PtzCamRoi.idx` 를 키로 쓰지 않으므로 **기존 PTZ 캘리브레이션 데이터는 깨지지 않는다**. 다만 분석 탭에 두 체계 UI 가 공존하므로 혼동에 주의.

## 7. 미해결 / 후속 과제

1. **[해소됨 — 마스터 지시로 수정]** `web/core.d.ts` 의 `buildFlatSlotRows` **입력** `parkingSlotsByKey[].vpd/lpd` 가 `boolean` 으로 선언돼 있었으나 실제 소스(`SqliteStore.getParkingSlots` → `ParkingSlotView`)는 **검출 객체 | null** 을 반환했다. → `vpd?: NormalizedRect | null; lpd?: NormalizedQuad | null` 로 정정(**출력** `FlatSlotRow.vpd/lpd: boolean` 은 `!!` 결과이므로 유지). 이 정정이 `test/finalizerParkingSlots.test.ts` 의 픽스처(`vpd: !!r.vpd` 불리언 강제 — 실제 뷰어 경로 `app.js:1560` 은 DB 행을 그대로 push)를 즉시 노출시켜 함께 실 계약대로 수정했다. tsc exit 0, 1328 tests 전량 통과.
2. **Unity 재생성 후 전역번호 재지정** — 운영 수칙(§5-1).
3. **Unity 가 `parking_spaces[]` 에 새 필드를 추가하면** `applyPlaceRoiUpdate` 가 `{idx, points}` 로만 재구성하므로 **그 필드가 저장 시 소실**된다(현재 실데이터엔 해당 없음). Unity 스키마 변경 시 `{...sp, idx, points}` 로 고칠 것.
4. **저장 부분 실패 시 자동 롤백 없음** — 앞 프리셋만 파일 반영. 실패 프리셋을 `#place-msg` 에 명시하고 dirty 유지로 고지(설계된 동작).
5. **최종화 후 목록 소스가 `state.placeRoi` 로 바뀜** — DB(`parkingSlotsByKey`)는 이제 **태그 소스로만** 쓰인다. PtzCamRoi 파일이 없으면 최종화 후에도 목록이 비어 `'표시할 주차면 없음 — PtzCamRoi.json 확인'` 이 뜬다(정상 경로에선 Finalizer 가 PtzCamRoi 를 소스로 하므로 발생하지 않음).

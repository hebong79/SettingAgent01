## 1. 구현 범위 (이번 차수)

사용자 지시: **"LLM·MCP 설정 config 와 MCP 도구·기타 config 를 각각 분리해서 동작하게 + SettingAgent부터 구현 착수"**.

| 영역 | 구현 | 상태 |
|------|------|------|
| **Config 분리** | `llm.config`(두뇌+MCP) / `tools.config`(도구+기타) 2파일 + 로더 분리 | ✅ |
| 결정형 핵심 | 위치정렬·ROI 산출·전역 인덱스 매핑 | ✅ |
| REST 클라이언트 | Camera(req_img/req_move), VPD(da_vpd_api /det) | ✅ |
| 셋업 오케스트레이터 | 프리셋 순회→ROI→전역인덱스→영속화 상태머신 | ✅ |
| MCP 도구 서버 | camera_req_img/req_move/vpd_detect 노출 | ✅(스캐폴드 동작) |
| LLM 두뇌 런타임 | model-agnostic(OpenAI호환), 셋업 검토 보조, 기본 비활성 | ✅(스캐폴드) |
| REST API | /health, /setup/run, /setup/status, /mapping | ✅ |
| 유닛테스트 | 30개 통과 | ✅ |

## 2. Config 분리 (핵심 요구사항)

두 파일이 **역할로 완전히 분리**되어 독립 로드된다.

### 2.1 `config/llm.config.json` — LLM 두뇌 + MCP 연결
```jsonc
{
  "llm": { "provider": "qwen3", "model": "qwen3-8b",
           "baseUrl": "http://localhost:8000/v1",  // OpenAI 호환(vLLM/Ollama/llama.cpp)
           "apiKeyEnv": "LLM_API_KEY", "temperature": 0.1, "maxTokens": 2048,
           "enabled": false },                       // 결정형만 쓰려면 false
  "mcp": { "enabled": true, "transport": "stdio",
           "servers": [ { "name": "...", "command": "node", "args": ["dist/mcp/server.js"] } ] }
}
```
- 로더: `src/config/llmConfig.ts` (`loadLlmConfig`). zod 검증. `provider` 는 qwen3/gemma/claude/codex/openai-compatible.
- **로컬/자체호스팅 우선(Qwen3/Gemma)** 결정 반영 → 기본 baseUrl 이 로컬 OpenAI 호환 엔드포인트.

### 2.2 `config/tools.config.json` — MCP 도구(능력) + 기타
```jsonc
{
  "camera": { "baseUrl": "http://localhost:13100", "zoomMin": 1.0, "zoomMax": 36.0, ... },
  "vpd":    { "endpoint": "http://127.0.0.1:18080", "detPath": "/vpd/api/v2/det/imgupload", ... },
  "setup":  { "presetSettleMs": 1000, "minConfidence": 0.5, "roiPadding": 0.05, "yBandTolerance": 0.1 },
  "server": { "port": 13020 },
  "store":  { "dataDir": "data", "captureDir": "data/captures" }
}
```
- 로더: `src/config/toolsConfig.ts` (`loadToolsConfig`). 섹션 단위 병합.
- `_comment` 등 부가 키는 무시(테스트로 검증).

> **분리 검증**: `test/config.test.ts` 에서 tools 에 `llm` 키가 없고 llm 에 `camera` 키가 없음을(교차 오염 없음) 확인.

## 4. 처리 흐름 (결정형, LLM 미개입)

```
POST /setup/run { targets:[{camIdx,presetIdx,ptz?},...] }
  └ for each target:
      presetSettle 대기 → camera.req_img → vpd.detect(차량 bbox 정규화)
      → buildSlots(위치정렬+패딩, minConfidence 필터) → slotId=c{cam}p{preset}s{pos}
      → preset.coveredSlotIds / slot.roiByPreset 갱신
  └ buildGlobalIndex(cam→preset→position) → validateCoverage
  └ Repository.saveArtifact(setup_artifact.json)
  └ (llm.enabled 시) AgentRuntime.reviewSetup() 보조 검토
```
- **LLM 두뇌**: 셋업은 결정형으로 충분하므로 두뇌는 *보조 검토*에만 개입(기본 비활성). 아키텍처 §8.2 경계 준수.
- **MCP**: 동일 능력(camera/vpd)을 `mcp/server.ts` 가 MCP 도구로도 노출 → 두뇌가 표준 도구로 호출 가능.

## 5. 동작 확인 (실측)

### 5.1 유닛테스트 — `npm test` → **30/30 통과**
- geometry(7), ordering(4), roiBuilder(4), globalIndexer(5), jpeg(3), config(4), setupOrchestrator(3)
- 핵심 검증: 전역 인덱스 결정성/멱등성, 위치정렬 규칙, ROI 패딩, config 분리·교차오염 없음, 오류 시 FAILED 전이.

### 5.2 타입체크 — `npm run typecheck` → **에러 0**

### 5.3 서버/Tool 스모크 (fastify inject + MCP 구성)
```
GET  /health         200 {"status":"ok","camera":true,"vpd":false,"brain":false}
POST /setup/run      200 {"ok":true,"status":{"state":"DONE",...},"slots":1,"globalCount":1}
GET  /mapping        200 {"presets":[...],"slots":[...],"globalIndex":[...]}
POST /setup/run(bad) 400 (zod 검증)

```
> 실제 Unity/VPD 서버 연동 스모크는 해당 서버 기동 후 별도 수행 필요(현재는 가짜 의존성으로 배선 검증).

## 6. 설계 대비 미구현/후속 (다음 차수)

- **프리셋 정의 출처**: 현재 `/setup/run` 이 targets 를 입력으로 받음. 확정된 "mapConfig + Unity CameraPos PTZ 추출" 자동 로딩은 후속(ParkSimMgr `setupFromMapConfig` 패턴 이식 예정).
- **실 PTZ ROI 자동생성**(검출 누적+클러스터링, 설계서 §8-1-1): 1차는 단일 프레임 bbox→ROI. 누적 클러스터링은 후속.
- **MCP 두뇌 실연동**: 로컬 LLM(Qwen3/Gemma) 서버 기동 후 `llm.enabled=true` 로 reviewSetup 실측 필요.
- **공통 타입 패키지화**: `domain/types.ts` → `@parkagent/types` 추출(Action/DM 공유 시).

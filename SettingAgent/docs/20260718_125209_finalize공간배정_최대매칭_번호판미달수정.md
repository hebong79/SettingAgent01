## 1. 문제 · 근본원인(리더 진단)

화면(라이브 정밀수집)에서 검출되는 번호판은 프리셋 전체 **약 16~17개**인데, finalize 이후 `slot_setup.lpd_obb`가 채워지는 슬롯은 **8개뿐**인 증상이 보고되었다.

진단 결과 **검출·집계(Aggregator) 단계의 손실은 0** — 17개 클러스터 전량이 번호판까지 붙어 `accepted` 상태로 집계됨을 확인했다. 병목은 **`Finalizer.finalize()`의 공간배정(space assignment) 단계**였다.

### 원인 코드 (수정 전, `Finalizer.ts` 약 209~239줄)

각 주차면 폴리곤(`PlaceRoiSpace`)마다 `clusters.find(...)`로 **선착순 1개**를 집는 구조:

```ts
spaces.forEach((sp, i) => {
  const hit = clusters.find((c) => {
    const pc = c.plateQuad ? quadCentroid(c.plateQuad) : null;
    if (pc && pointInPolygon(sp.points, pc)) return true;
    return pointInPolygon(sp.points, { x: c.x + c.w / 2, y: c.y + c.h / 2 });
  });
  ...
});
```

이 구조에는 세 가지 결함이 있었다.

- **ⓐ 몰림 충돌**: 한 폴리곤에 대표점이 2대분 들어오면 `find()`가 먼저 매치되는 1개만 취하고 나머지는 완전히 버려짐(다른 빈 슬롯으로 재배정되지 않음).
- **ⓑ 상호배타 아님**: `spaces.forEach`가 슬롯별로 독립 실행되므로, 동일 클러스터가 이론상 여러 슬롯에 중복 배정될 수 있는 구조(각 슬롯이 전체 `clusters`를 매번 처음부터 훑음).
- **ⓒ 프레임 밖 고아**: 번호판중심·차량중심 모두 어느 폴리곤에도 포함되지 않으면(원근 오프셋으로 셀 경계를 벗어난 경우) 무조건 미배정 — 실제로는 인접 슬롯의 정당한 차량인 경우가 다수였다.

**시각화(오버레이) 확인**: 차량 bbox 중심은 바닥 폴리곤 위로 뜨는(원근 왜곡) 경향, 번호판 중심은 바닥에 근접하지만 셀 경계로 흘러 이웃 칸과 충돌하는 경향이 실측으로 확인됨. 이 실측이 대표점 우선순위(번호판 quad 중심 우선)와 거리 게이트 설계의 근거가 되었다.

**실측 손실 규모**: 동일 라이브 씬에서 17클러스터 검출 대비 finalize 결과가 런마다 14(비교적 깨끗한 run) ~ 8(노이즈 심한 run)로 널뛰었다 — `find()` 선착순의 입력 순서 의존성 때문.

## 2. 해결 — A안: 전역 상호배타 1:1 배정 (2단계로 재설계됨)

### 2.1 신규 함수 `assignClustersToSpaces` (`src/capture/spaceAssign.ts`, 신설)

```ts
export function assignClustersToSpaces(
  spaces: readonly PlaceRoiSpace[],
  clusters: readonly AggregatedSlot[],
  opts: { centroidGate: number },
): Map<number, AggregatedSlot>   // key = spaces 배열 인덱스(0-based), 미배정 공간은 키 없음
```

DB·IO 비의존 순수 함수. 한 프리셋 범위의 배정만 담당하며 LLM은 개입하지 않는다(결정형 기하 연산).

**대표점(reprPoint) 우선순위**
1. 번호판 quad 4점 산술평균 중심(`plateQuad` 보유 시) — 최우선. 실측상 대부분 클러스터가 이 경로로 정확히 매칭됨.
2. 폴백: 차량 bbox 중심 `{x + w/2, y + h/2}` — 번호판이 없는 클러스터만.

**후보(유효)쌍 조건** — 한 (공간, 클러스터) 쌍이 배정 후보가 되려면:

> `pointInPolygon(space.points, repr)` **OR** `dist²(repr, polygonCentroid(space)) < centroidGate²`

`pointInPolygon` 단독으로는 원래 충돌 버그를 해결하지 못한다. 한 폴리곤에 두 클러스터의 대표점이 **둘 다 내부**로 들어온 경우, 폴리곤-포함 조건만으로는 두 번째 클러스터가 인접한 빈 폴리곤과 후보쌍을 만들 수 없어 결국 미배정(빈 슬롯 잔존)으로 끝난다. **centroid 거리 게이트가 있어야** 인접 슬롯 centroid 근처라는 이유로 후보쌍이 형성되고, 아래 매칭 단계가 이를 회수할 수 있다.

**비용**: 대표점↔폴리곤 centroid 제곱거리(`dx²+dy²`, sqrt 불필요).

### 2.2 알고리즘 재설계: 그리디 → Kuhn 최대 카디널리티 이분매칭

최초 구현은 "후보쌍을 비용 오름(→ clusterId 오름 → 공간idx 오름)으로 정렬 후 순회하며 양쪽 미배정일 때만 확정"하는 **전역 그리디**였다. 그러나 리더의 라이브 검증(B모드 loop)에서 이 방식이 **최대 카디널리티(maximum matching)가 아님**이 드러났다: 값싼 엣지를 먼저 소비한 클러스터가 다른 회수 가능한 클러스터의 유일한 슬롯을 선점해버리는 **cascade** 현상이 관측되었다.

- gate 0.12: greedy 15 vs 최대매칭 16 (폴리곤 내부·centroid 거리 0.072인 클러스터가 그리디에서는 다른 클러스터에 슬롯을 뺏겨 미배정).
- gate 0.18: greedy 15 vs 최대매칭 17 (원근 오프셋 큰 가장자리 클러스터 0.172도 최대매칭에서는 정당 회수).

→ 알고리즘을 **Kuhn 증가경로 탐색**(max-cardinality bipartite matching)으로 교체했다.

- 각 클러스터의 인접(유효) 슬롯 리스트를 비용 오름(동률 spaceIdx 오름)으로 정렬해 비용 선호를 유지.
- 클러스터를 **clusterId 오름차순**으로 방문하며 `tryKuhn(u, seen)` 증가경로 탐색, `matchOfSpace[]` 역포인터로 매칭 상태 관리 → 매 실행 동일 결과(결정성).
- 프리셋당 슬롯·클러스터 규모가 소규모(≤~10)라 O(V·E) 완전탐색으로 충분, Hungarian(가중 최소비용) 알고리즘은 과설계로 판단해 채택하지 않음.

반환 타입·유효 후보쌍 조건·대표점 우선순위·`plateMatch.ts` 무접촉 원칙은 전부 그대로 유지된 채 **알고리즘 본체만** 교체되었다.

### 2.3 `plateMatch.ts`를 재사용하지 않은 이유

기존 `src/setup/plateMatch.ts::matchPlatesToSlots`도 전역 그리디 상호배타 1:1 매칭이지만, 의미론·기하가 다르다.

| 항목 | plateMatch(기존) | finalize 공간배정(신규) |
|---|---|---|
| 매칭 대상 | 번호판 ↔ 차량 ROI | 집계 클러스터 ↔ 주차면 폴리곤 |
| 슬롯 형상 | 축정렬 `NormalizedRect` | 임의 4점 폴리곤(사선 주차면) |
| 게이트 | 점-in-rect | 폴리곤 포함 OR centroid 거리 |
| 비용 | 겹침 면적 내림 → 거리 | 대표점→centroid 거리 오름 |

결정적으로, `matchPlatesToSlots`는 `detectPipeline.ts:288`·`onPlaceFilter.ts:82`가 `plate.quad` **참조 동등성 계약**에 의존한다. 여기에 폴리곤·거리게이트 분기를 끼워 넣으면 두 공유 사용처가 회귀할 위험이 있어, **패턴만 차용하고 신규 함수로 분리**하는 것으로 결정했다. `Aggregator.ts`의 클러스터-내 번호판 귀속 로직도 별개 관심사로 무접촉.

## 4. config 승격 — `capture.slotAssignGate`

거리 게이트는 **관측형 튜닝 파라미터**(원근·슬롯 간격에 의존, 씬마다 재조정 가능)이므로 모듈 상수가 아닌 config 필드로 승격했다.

- `src/config/toolsConfig.ts` — `CaptureSchema.slotAssignGate: z.number().min(0).max(1).default(0.18)`, `DEFAULT_TOOLS_CONFIG.capture.slotAssignGate = 0.18`.
- `config/tools.config.json` — `capture.slotAssignGate: 0.18`.
- `FinalizerDeps.cfg`는 이미 `ToolsConfig['capture']`로 Finalizer에 주입되어 있어 신규 배선 코드는 0.

**초기값 변경 이력**: 설계 시점 0.12(정규화 슬롯 centroid 간격 ~0.10~0.15의 절반 미만) → 리더 라이브 검증 후 **0.18로 상향**. 최대매칭이 상호배타를 보장해 과배정을 스스로 억제하므로, 게이트를 더 크게 열어 원근 오프셋이 큰 프리셋의 정당 슬롯(실측 0.172)까지 포함하도록 조정.

**운영 주의**: `config/tools.config.json`은 서버(nodemon)가 `src`만 watch하고 `config`는 감시하지 않는다. 게이트 값을 바꿔도 **서버 재기동 전까지 반영되지 않는다.** 이번 라이브 검증에서 리더가 src 파일을 touch해 강제 리로드시켜 0.18 반영을 확인했다.

## 6. 입출력·REST 계약

- **입력**(`assignClustersToSpaces`): `PlaceRoiSpace[]`(정규화 폴리곤), `AggregatedSlot[]`(집계 클러스터), `{ centroidGate: number }`.
- **출력**: `Map<number, AggregatedSlot>` — key는 `spaces` 배열의 0-based 인덱스, 미배정 공간은 키가 존재하지 않는다.
- **Finalizer 소비측 인덱스 일치**: `spaces.forEach((sp, i) => { const hit = assigned.get(i) ?? null; ... })` — forEach 인덱스 `i`가 그대로 반환 Map의 key와 일치. 1-based/0-based 혼선 없음(qa 교차비교로 확인).
- **REST 계약**: `/capture/finalize`, `/capture/slots` 등 외부 인터페이스와 `slot_setup` row shape, `slotId`, `presetSlotIdx` 규칙은 **완전히 불변**. 이번 변경은 서버 내부의 배정 알고리즘만 교체하며, 어떤 API 응답 형태도 바뀌지 않는다.

### 7.4 리더 라이브 검증(B모드 loop, 관찰형 — vitest 범위 밖)

정밀수집으로 17클러스터 전량 accepted 검출 → finalize(gate 0.18, 최대매칭) → `slot_setup` **vpd=17 / lpd=17 / occ=17**(전 프리셋 7·6·4 완전배정, 손실 0) → 센터라이징 17/17 → 최종 `slot_setup` `lpd=17` · `ptz=17` · `centered=17`. 개선 전 그리디@0.12 대비(15~16)로 완전 해소를 확인. 이 관찰은 vitest 스위트 밖의 리더 실측이며, 유닛/통합 테스트의 cascade(+1) 및 gate 0.18/0.12 대조 케이스가 이를 결정형으로 고정 재현한다.

## 1. 배경 · 마스터 요청 (2026-07-23)

"LPD 관련 실행이 되면 (박스가) 남아있다. 표시 초기화 버튼을 누르면 화면의 모든 박스들을 지워줘. 그리고 DB를 체크하면 다시 DB에 있는 vpd, lpd, 점유영역 + 센터라이징 위치 등을 보여줘(물론 체크된 것들만)."

요구는 두 갈래다.

1. **전체 삭제**: `표시 초기화` 버튼을 누르면 화면에 남은 박스가 하나도 없어야 한다.
2. **DB 재표시**: 이후 `DB 보기`를 다시 체크하면, 각 항목 체크박스(차량/번호판/점유)가 켜진 것만 DB 데이터로 되살아나야 한다.

## 2. 원인 진단

### 2.1 LPD 탐색(discovery) 잔여 박스 — 요구1 미충족의 직접 원인

`resetOverlayDisplay()`(`web/app.js:148`)는 종전에 `detectByKey`/`occComputeByKey`/`occByKey`/`vcuboidByKey` 4개만 비웠다. 그런데 노란 번호판 박스 중 일부는 이 4개가 아니라 **`state.discoverByKey`**(cam:preset → discovery(앞면중심 LOOP) 결과 LPD OBB quad, `app.js:109` 정의, `app.js:2904`에서 매 discovery 완료 시 전체 대체)에서 나온다. `drawDetectOverlay`(`app.js:976~1018`)는 `#roi-plate` 체크 하나만 게이트로 보고 `state.discoverByKey[key]`를 그린다(`app.js:1014~1016`). `resetOverlayDisplay`가 이 상태를 비우지 않았으므로, LPD 탐색(discovery) 실행 이력이 있는 프리셋은 `표시 초기화`를 눌러도 discovery quad 가 화면에 그대로 남았다 — 이것이 마스터가 관찰한 "LPD 관련 실행이 되면 박스가 남아있다" 증상이다.

### 2.2 DB 보기 게이트 — 요구1(전체 삭제) vs 요구2(재체크 시 재표시)의 충돌 지점

`#roi-db`가 체크돼 있으면 `drawDbVpd`/DB LPD quad/`drawDbCentering`/`drawDbOccupancy`가 `state.parkingSlotsByKey`(DB `slot_setup` 조회 결과)를 계속 그린다(`app.js:982~1017`, `app.js:1061~`). 종전 `resetOverlayDisplay`는 `#roi-db`를 건드리지 않았으므로, DB 보기가 켜진 상태에서는 "모든 박스 삭제"가 성립하지 않았다.

단순히 `state.parkingSlotsByKey = null`로 지우는 방식은 쓸 수 없다 — `renderSlotList()`의 `finalized` 판정(`app.js:1127`: `!!(state.parkingSlotsByKey && Object.keys(state.parkingSlotsByKey).length)`)이 이 값을 최종화 여부의 근거로 삼기 때문에, 지우면 좌측 주차면 목록 패널이 회귀한다(기존 메모리 `finalize slot_setup wipe fragility` 참조). 따라서 **데이터는 보존하고 표시 게이트(`#roi-db` 체크박스)만 끄는** 방식을 택했다.

## 3. 변경 내용

전부 `SettingAgent/web/` 프런트엔드. 서버·DB·REST 계약·타 에이전트 코드는 무변경.

### 3.1 `resetOverlayDisplay()` — `discoverByKey` 삭제 + `#roi-db` 강제 해제

```js
function resetOverlayDisplay() {
  state.detectByKey = {};        // 검출 차량/번호판.
  state.discoverByKey = {};      // discovery(앞면중심 LOOP) LPD quad 삭제 — 없으면 LPD 실행 잔여 박스가 남는다.
  state.occComputeByKey = {};    // 로직 점유(원) 삭제.
  state.occByKey = {};           // 점유율 요약 삭제.
  state.vcuboidByKey = {};       // 차량 육면체 + seg 마스크 삭제.
  $('roi-db').checked = false;   // DB 소스 오버레이 게이트 off — DB 박스도 화면에서 사라진다(데이터는 DB에 보존).
  ...
}
```

`state.discoverByKey = {}`는 다른 4개 필드와 동일한 방식(데이터 삭제, 재수집 전까지 복원 안 됨)이다. `$('roi-db').checked = false`는 데이터가 아니라 **표시 게이트만** 끈다 — `state.parkingSlotsByKey`는 손대지 않으므로 `renderSlotList`의 `finalized` 판정과 DB 데이터 자체는 그대로 보존된다. 함수 상단 JSDoc에 이 예외 1건을 별도로 명시했다(`app.js:144~146`).

### 3.2 `#roi-db` change 핸들러 — "소스 미로드 시 1회 로드" → "켤 때마다 재조회"

```js
// 변경 전
if (e.target.checked && !state.parkingSlotsByKey) await loadParkingSlots();
// 변경 후
if (e.target.checked) await loadParkingSlots();
```

종전 규약(2026-07-21 커밋, `20260721_215536_DB보기_소스전환_수정.md`)은 `state.parkingSlotsByKey`가 이미 있으면 재조회를 생략했다. 그런데 `표시 초기화`가 `#roi-db`를 강제로 끄는 이번 변경과 조합하면, 사용자가 초기화 후 다시 체크했을 때 `parkingSlotsByKey`는 여전히 이전 값이 캐시돼 있어 "재조회 없이 옛 값 재표시"가 돼버린다. 이는 요구2("DB에 있는 것을 다시 보여줘")의 취지 — 최신 DB 상태 — 와 어긋날 수 있으므로, 캐시 유무와 무관하게 **체크할 때마다 `GET /capture/slots`를 재조회**하도록 단순화했다.

### 3.3 `web/index.html` 툴팁 갱신

- `#roi-db` 라벨: "센터라이징 위치"를 소스 전환 대상에 추가, "켤 때마다 DB를 재조회한다"를 명시, "표시는 각 항목 체크박스(차량/번호판/점유)를 따른다"를 추가해 마스터의 "체크된 것들만" 요건을 문구로도 확인시켰다.
- `#roi-clear`(표시 초기화) 버튼: "LPD탐색"을 삭제 대상 목록에 추가, "'DB 보기'는 체크가 해제되며(DB 데이터는 보존), 다시 체크하면 DB 내용이 되살아난다"를 명시.

## 4. 항목별 게이트 (현행 유지 — 이번 변경으로 손대지 않음)

DB 보기(`#roi-db`) ON 상태에서, 무엇이 그려지는지는 항목 체크박스가 최종 결정한다(마스터 "체크된 것들만" 요건).

| 항목 | 렌더 함수 | 게이트(항목 체크박스) | 데이터 소스 |
|------|-----------|----------------------|-------------|
| 차량(VPD) | `drawDbVpd` | `#roi-vehicle` | `row.vpd` (null이면 skip) |
| 번호판(LPD) | DB LPD quad 인라인 루프 | `#roi-plate` | `row.lpd` (null이면 skip) |
| 센터라이징 위치 | `drawDbCentering` | `#roi-plate`(LPD와 동일 게이트, LPD quad 직후 그림) | `row.centered && row.lpd` (둘 다 있어야, `lpd` OBB 중심을 표시점으로 사용) |
| 점유영역 | `drawDbOccupancy` | `#roi-occupancy` | `row.occupyRange` (null이면 skip) |

센터라이징 위치가 `#roi-plate` 게이트를 공유하는 이유(`app.js:1035~1039` 주석): 센터링 산출물은 PTZ(pan/tilt/zoom)라 프리셋 프레임 좌표계가 아니고, 프리셋 프레임에서 "그 슬롯이 센터라이징된 지점"을 가리키는 유일한 기존 좌표가 센터링 표적이었던 `lpd`이기 때문이다(PTZ 역투영 신규 구현 없이 기존 함수만 재사용하는 규약).

## 5. 입출력 · REST 계약

서버 API(`GET /capture/slots`)와 응답 shape(`SlotSetupRow[]`)는 전혀 변경하지 않았다. 이번 수정은 클라이언트 상태(`state.discoverByKey`)의 삭제 시점과 `#roi-db` 체크박스의 온/오프·재조회 조건만 바꾼 것이다. REST 계약·DB 스키마 변경 없음. 다만 `#roi-db`를 켤 때마다 `GET /capture/slots` 왕복이 매번 발생하므로(§7-3), 서버로의 요청 빈도가 늘어난다.

## 7. 영향도 분석

요약은 `_workspace/04_doc_impact.md` 참고. 핵심 결론만 기술한다.

1. `resetSlotSetupDb()`(DB 검출·점유·PTZ 초기화 버튼)는 내부에서 `resetOverlayDisplay()`를 먼저 호출한 뒤 `loadParkingSlots()`를 직접 호출한다(`app.js:2490~2491`). 이번 변경으로 `resetOverlayDisplay()`가 `#roi-db`를 강제로 끄므로, DB 초기화 직후에는 `#roi-db`가 항상 unchecked 상태가 된다 — 초기화 직후 값은 어차피 대부분 null이므로 표시상 손실은 미미하나, 사용자가 초기화 전 `#roi-db`를 켜 두었다면 초기화 후 화면에서 체크가 풀리는 부수효과가 새로 생겼다.
2. `loadRoiToDb()`(ROI 파일 → DB 전량 재구성 버튼)도 동일 패턴(`app.js:2503, 2513`) — `resetOverlayDisplay()` 호출로 `#roi-db`가 꺼진 뒤 `loadParkingSlots()`가 새 `slot_setup`을 반영하지만, 체크박스가 꺼져 있으므로 새로 로딩된 DB 값이 자동으로는 보이지 않는다(사용자가 `#roi-db`를 다시 켜야 함).
3. `capFinalize()`는 `resetOverlayDisplay()`를 호출하지 않고 `$('roi-db').checked = true`를 프로그래밍 방식으로 대입한 뒤 `loadParkingSlots()`를 별도로 호출한다(`app.js:2471, 2473`). 프로그래밍 방식 대입은 `change` 이벤트를 발화시키지 않으므로, `#roi-db`의 change 핸들러(§3.2)에 의한 추가 `loadParkingSlots()` 호출은 일어나지 않는다 — 이중 조회 없음.
4. `renderSlotList()`의 `finalized` 판정(`app.js:1127`)은 `state.parkingSlotsByKey`의 존재 여부에 의존한다. 이번 변경은 `resetOverlayDisplay()` 안에서 이 필드를 건드리지 않았으므로 `finalized` 판정과 좌측 주차면 목록 패널은 영향받지 않는다(의도적 보존).
5. `#roi-db`를 켤 때마다 `GET /capture/slots` 왕복이 1회 늘어난다(종전은 `parkingSlotsByKey`가 이미 있으면 생략). 데이터 규모(슬롯 수십 건)를 고려하면 비용은 경미하나, 반복 토글 시 서버 요청 빈도가 그만큼 증가한다.

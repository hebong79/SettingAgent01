## 1. 증상 · 근본 원인

### 1.1 증상
정밀 수집(capture) 화면에서 진행 중 **"정지" 버튼을 눌러도 반응이 없어 보인다.** 버튼을 다시 눌러도 아무 변화가 없고, 체크포인트 타이밍에 정지하면 수십 초간 멈춘 듯 보인다.

### 1.2 근본 원인 (3가지 — 리더 코드 분석으로 확정)

| # | 원인 | 위치 | 성격 |
|---|------|------|------|
| 1 | **UI 피드백 즉시 소실 (프론트)** — `capStop()`이 `cap-msg`에 '정지 요청됨'을 써도 500ms마다 도는 `capFrameTick()`이 '수집 중 — cam…'으로 덮어씀. 버튼에 상태별 disabled 처리가 없어 재클릭 시 백엔드가 `400 not running`을 반환 → "반응 없음"으로 체감. | `app.js` `capStop`/`capFrameTick` | 수정 대상 |
| 2 | **체크포인트 중 정지 지연 (백엔드)** — `checkpoint()`의 advisory reviewer 호출은 취소 콜백이 없어, 정지 클릭이 checkpoint 진행 중 들어오면 advisory LLM 응답(및 그 앞의 warmup 콜드로드, 수 분)까지 정지가 블록됨. floor/occupancy reviewer는 이미 `() => this.currentState() === 'stopping'` 취소 콜백으로 자체 단락됨. | `CaptureJob.ts` `checkpoint()` | 수정 대상 |
| 3 | **재시작 = 처음부터** — `start()`가 `round=0`으로 초기화. "이어하기"는 원래 없는 기능. | `CaptureJob.ts` `start()` | **스코프 밖** (별개 개선 항목, 이번 수정 미포함) |

### 1.3 목표
- 정지 클릭 시 **즉시** 시각적 피드백(정지 버튼 비활성 + '정지 중…' 안내)이 나타나고 라운드 마무리까지 유지.
- checkpoint 진행 중 정지 클릭 시 advisory LLM/warmup 대기로 정지가 지연되지 않음(floor/occupancy와 동일한 반응성).
- 기존 좌표/집계 불변식은 그대로 유지.

## 2. 변경 내역

### 2.1 백엔드 — `src/capture/CaptureJob.ts` (B1: advisory reviewer stopping 게이트)

**위치**: `checkpoint()` 내 advisory reviewer 블록 (line 331).

| | 코드 |
|---|------|
| Before | `if (this.deps.reviewer) {` |
| After | `if (this.deps.reviewer && this.currentState() !== 'stopping') {` |

**의도·동작**: warmup(line 325)이 이미 사용하는 결정형 게이트 패턴(`if (this.currentState() !== 'stopping') await ...warmup()`)과 동일하게, advisory reviewer 호출 **진입 전** state를 검사한다. `stop()`이 checkpoint 초반 await(warmup 콜드로드 등) 중 발화하면 advisory LLM 호출로 진입하지 않고 checkpoint를 조기 종료 → 정지 반응성 확보.

**불변식 보존** (핵심): 집계 라인(line 326–329: `getDetectionsForRun` → `aggregate` → `replaceAggregatedSlots`)은 게이트 **밖에 그대로 유지**된다. 따라서 정지 시에도 집계 결과/좌표는 변하지 않는다. advisory reviewer는 "자문(advice) 표시"만 담당하는 비결정형 호출이므로 건너뛰어도 산출 데이터에 영향이 없다.

**관련 기존 게이트** (무변경, 참고): 같은 방향의 결정형 게이트가 이미 존재한다 — 라운드 레벨 `if (this.shouldCheckpoint() && this.currentState() !== 'stopping')`(line 229), 타깃 이동 페이싱 `... && this.currentState() !== 'stopping'`(line 219), floor/occupancy reviewer의 취소 콜백(전달됨). B1은 이 패턴을 advisory 경로에 마저 적용한 것이다.

### 2.2 프론트 — `web/core.js` (F1: `captureUiState` 순수 함수 신규)

`pollPlan` 바로 아래에 named export 추가 (line 116–125). 브라우저 API 미참조 → vitest 직접 import 가능.

```js
export function captureUiState(state) {
  const active = state === 'running' || state === 'stopping' || state === 'finalizing';
  return {
    startDisabled: active,               // 활성 중 중복 start 방지 (409 대칭)
    stopDisabled: state !== 'running',   // stop 은 running 에서만 허용 (400 not running 대칭)
    finalizeDisabled: active,            // 활성 중 finalize 금지 (409 대칭)
    suppressFrameMsg: state === 'stopping', // stopping 중 프레임틱이 cap-msg 를 덮지 않게
    stoppingNote: state === 'stopping',  // '정지 중…' 안내 표시 여부
  };
}
```

**설계 의도**: disabled 규칙을 각 REST 라우트의 거부 조건과 **정확히 대칭**으로 두어, 재클릭 시 무반응/에러(400/409)를 UI 레벨에서 원천 차단한다. 로직을 순수 함수로 뽑아 DOM 결선(app.js)은 얇게 유지(기존 core/app 분리 컨벤션).

## 3. 입출력 · REST 계약

**REST 계약은 변경 없음.** 프론트가 기존 라우트의 거부 조건에 버튼 상태를 맞췄을 뿐이다. `captureUiState`의 각 필드는 다음 라우트 거부 조건과 대칭이다 (`src/api/captureRoutes.ts` 확인).

| UI 필드 | 대칭 라우트 거부 조건 | 위치 |
|---------|----------------------|------|
| `stopDisabled = state !== 'running'` | `POST /capture/stop` → `if (st.state !== 'running') 400 'not running'` | captureRoutes.ts:156–158 |
| `finalizeDisabled = active` | `POST /capture/finalize` → active 시 `409 'capture still running'` | captureRoutes.ts:171–173 |
| `startDisabled = active` | `POST /capture/start` → 이미 실행 중이면 `409` | captureRoutes.ts:102–103 |

`state`는 백엔드 `CaptureState = idle | running | stopping | finalizing | done | stopped | error` 중 하나로만 온다(기존 `pollPlan`·`captureProgress`가 이미 사용하는 가정과 일관).

## 6. 잔여 한계

- **in-flight advisory 1회 대기**: B1 게이트는 `stop()`이 advisory reviewer 호출 *진입 전* 발화한 경우만 건너뛴다. 이미 advisory LLM 요청이 in-flight인 순간 stop이 오면 그 **1회 응답까지는 대기**한다(advisory reviewer에 취소 콜백을 새로 추가하는 것은 범위 초과 — floor/occupancy와 대칭을 맞추는 별개 개선 항목). 다만 warmup(수 분 콜드로드) 대기가 게이트로 제거되어 체감 지연은 크게 감소.
- **이어하기 미지원**: 재시작 시 이전 라운드를 유지하는 "이어하기"는 원래 없던 기능으로, 리더 확정에 따라 이번 스코프 밖(별개 개선 항목).

## 1. 배경 · 근본원인 (마스터 신고 원문 기준)

> "최종화해도 db에 vpd/lpd/occupy 저장 안됨, 이전엔 됐음"

로그 확인 결과 finalize가 `accepted=0`(검출 클러스터 배정 0건)으로 완료되고 있었고, 그 결과 `slot_setup` 17행 전체의 `vpd_bbox`/`lpd_obb`/`occupy_range`가 `null`로 채워져 있었다.

원인 분석:

- `Finalizer.finalize`는 파일 바닥 ROI(`PtzCamRoi.json`)의 슬롯 전량에 대해 `SlotSetupRow`를 조립하고 `store.replaceSlotSetup(rows)`(DELETE 후 전량 INSERT, [SqliteStore.ts:189-207](../src/capture/SqliteStore.ts#L189))로 `slot_setup` 테이블을 통째로 교체한다.
- 각 행의 `vpdBbox`/`lpdObb`/`occupyRange`는 해당 슬롯에 검출 클러스터가 배정(`hit`)됐을 때만 채워지고, 없으면 무조건 `null`이었다.
- 따라서 검출 매칭이 0인 finalize(짧은 캡처, 차량 미검출, 주차면필터로 후보 전량 제거 등 — `accepted=0`)가 한 번이라도 실행되면, **직전에 정상 저장돼 있던 vpd/lpd/occupy(예: 커밋 `66d9042` 시점 라이브 17/17 검출)를 전부 null로 덮어써 파괴**했다.
- 이 조립·교체 파이프라인 자체는 이번 세션에서 변경된 적이 없다(회귀가 아니라 기존에 잠재해 있던 결함). 다만 직전 세션의 "DB 오버레이 상시 렌더" 작업이 화면에 DB 값을 그대로 노출하게 되면서, 비어버린 DB 상태가 눈에 띄는 형태로 드러났다.
- `accepted=0`의 상위 원인(왜 클러스터가 0이 되는지)은 `support < clusterMinSupport`(기본 3) 또는 주차면 필터/정합 조건 — 데이터·런타임 상황이지 코드 결함이 아니다. 이번 변경은 그 상위 원인을 고치는 것이 아니라, **그 상황이 재발해도 기존 정상 데이터를 파괴하지 않도록 하는 방어 가드**다.

## 2. 변경 상세 — `src/capture/Finalizer.ts`

변경은 `Finalizer.finalize`의 `slot_setup` 행 조립부([Finalizer.ts:186-237](../src/capture/Finalizer.ts#L186)) 안에서만 이루어졌다. `replaceSlotSetup`(전량 DELETE+INSERT)의 시그니처·시맨틱은 그대로 두고, **행을 조립하기 전에 기존 데이터를 병합 소스로 1회 읽어와** 검출이 없는 슬롯에 한해 기존 값을 재사용한다.

### 2.1 기존 slot_setup 사전 조회 (1회)

```ts
const existingBySlot = new Map<number, SlotSetupView>();
for (const v of this.deps.store.getSlotSetup()) existingBySlot.set(v.slotId, v);
```

`place`(바닥 ROI) 로드 성공 확인 직후, `byPresetAcc` 조립 전에 1회만 조회한다. 이 맵의 키는 `slotId`(전역 슬롯 번호).

### 2.2 슬롯별 hit 유무 3분기 병합 (`spaces.forEach` 내부)

```ts
const prev = existingBySlot.get(sp.idx); // hit 없을 때 기존 검출값 보존(빈 finalize 파괴 방지).
...
vpdBbox: hit ? JSON.stringify({ x: hit.x, y: hit.y, w: hit.w, h: hit.h }) : (prev?.vpd ? JSON.stringify(prev.vpd) : null),
lpdObb: hit?.plateQuad ? JSON.stringify(hit.plateQuad) : (prev?.lpd ? JSON.stringify(prev.lpd) : null),
occupyRange: occupyRange ? JSON.stringify(occupyRange) : (prev?.occupyRange ? JSON.stringify(prev.occupyRange) : null),
```

세 컬럼 각각 동일한 3단계 우선순위를 따른다.

1. **hit 있음** → 새 검출값으로 갱신 (기존 동작과 동일, 회귀 없음).
2. **hit 없음 + 기존행에 값 있음** → 기존 값을 재직렬화해서 보존 (이번 가드의 핵심).
3. **hit 없음 + 기존행도 없음(신규 슬롯)** → `null` (파괴가 아니라 정상적인 최초 상태).

`getSlotSetup()`이 반환하는 `SlotSetupView`는 파싱된 형태(`vpd={x,y,w,h}`, `lpd`=4점 quad, `occupyRange`=폴리곤)이므로, 보존 시 `JSON.stringify`로 재직렬화해 저장 shape과 왕복 일치시킨다.

### 2.3 키 정합

기존 행의 `slotId`와 신규 조립 중인 `sp.idx`는 둘 다 동일 파일(`PtzCamRoi.json`)에 `normalizeGlobalIdx`가 적용된 값이며 이 함수는 멱등이다. 따라서 재-finalize 시에도 두 값이 안정적으로 일치해 `existingBySlot.get(sp.idx)`가 올바른 기존 행을 찾는다. 파일이 바뀌어 슬롯 번호 체계가 달라지는 경우에는 매칭이 되지 않아 보존이 적용되지 않고 `null`로 강등되는데, 이는 파괴가 아니라 안전한 쪽으로의 강등이다.

### 2.4 스코프 밖 (의도적 미변경) — 후속 검토 필요

`pan`/`tilt`/`zoom`/`centered`/`img1`(센터라이징 컬럼)은 이번 가드 대상이 아니며, 기존 동작 그대로 finalize 시점에 `null`/`0`으로 채워진다(이후 `/calibrate`가 `upsertSlotCentering`으로 채우는 기존 흐름).

**주의: 검출 컬럼(vpd/lpd/occupy)과 동일한 파괴 취약점이 센터라이징 컬럼에도 구조적으로 존재한다.** 즉 finalize가 재실행되면 이미 완료된 센터라이징 결과(pan/tilt/zoom/centered/img1)도 매번 초기화된다는 뜻이다. 이번 작업 범위(설계서 §설계-4)에서는 명시적으로 스코프 밖으로 분리했으며, 별도 판단·작업이 필요한 항목으로 남긴다.

## 3. 입출력 계약

- REST 계약(엔드포인트 경로·요청/응답 스키마) 변경 없음. `finalize`가 내부적으로 조립하는 `slot_setup` 행의 컬럼 구성과 `SlotSetupRow`/`SlotSetupView` 타입도 변경 없음.
- 동작 변화는 오직 "hit 없는 슬롯의 `vpd_bbox`/`lpd_obb`/`occupy_range` 값"에 한정된다: 이전에는 무조건 `null`, 이후에는 "기존 행이 있으면 그 값을 유지, 없으면 `null`".

## 5. 한계 · 후속 조치

- 이번 가드는 "빈 finalize가 이후 기존 데이터를 파괴하지 않는다"는 것만 보장한다. **현재 이미 null로 파괴된 실 DB 값을 복구하지는 않는다** — 정상 검출이 되는 재캡처(clean finalize)로만 복구 가능.
- `accepted=0`의 상위 원인(clusterMinSupport 미달, 주차면 필터/정합)은 이번 스코프에서 다루지 않았다. 필요 시 별도 조사 대상.
- §2.4에서 언급한 센터라이징 컬럼(pan/tilt/zoom/centered/img1)의 동일 파괴 취약점은 후속 검토 항목으로 남긴다.

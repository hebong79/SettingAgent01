# 클릭 조준 정확도 — 휴컴스 "광각 보정율" 조사와 실측 기각

작성: 2026-07-21 · 브랜치 `feat/centering-wide-correction` · 결론: **코드 변경 없음(보정 미적용)**

## 1. 요청

> 휴컴스 api에는 광각 보정율이 존재한다고 들었다. 확인해보고 마우스 클릭시 정확한 위치로 이동시키기 위해 보정율을 적용해줘.

즉 장비에 넘길 "보정율" 인자는 존재하지 않는다. 보정을 원한다면 **우리 쪽 기하에서** 계산해야 한다.

## 3. 우리 쪽 광각 보정을 구현해 A/B 실측 → **기각**

### 3.1 가설(교과서 핀홀 모델)
화면 오프셋은 각도가 아니라 tan(각도)에 비례한다:
`offset = tan(θ)/(2·tan(HFOV/2))` → `θ = atan(offset · 2·tan(HFOV/2))`.
선형 게인 k(°/정규화)는 중앙 소각 기울기이므로 `2·tan(HFOV/2) = k·π/180`,
보정각 `= atan(err · k·π/180)`. 중앙에서 선형과 일치하고 **가장자리에서 과조준을 걷어낸다**는 예측.
(게인 −62/−35.5 @zoom1 → HFOV 56.8°/VFOV 34.6°, 비 1.75 ≈ 16:9 로 사양과 모순 없음.)

### 3.2 측정 방법(눈대중 아님)
`POST /capture/detect`(LPD)로 **같은 번호판의 중심 좌표를 조준 전·후 수치로** 측정.
같은 base PTZ(cam1 preset1, pan 22 / tilt 6.8 / zoom 1.6934098)에서 선형 목표 PTZ와 보정 목표 PTZ를
각각 `ptz` 오버라이드로 렌더해 **동일 조건 A/B**.

### 3.3 결과

| 표본(조준 대상) | 선형(현행) 잔차 | tan 보정 잔차 |
|---|---|---|
| 좌측 끝 번호판 `errX −0.3856` | errX **−0.0076**, errY −0.0151 (r 0.0169) | errX −0.0141, errY −0.0149 (r 0.0205) |
| 우측 끝 번호판 `errX +0.4208` | errX **+0.0041**, errY −0.0144 (r 0.0149) | errX +0.0135, errY −0.0149 (r 0.0201) |

**두 표본 모두 보정을 넣으면 나빠졌다.** 좌·우 양끝에서 일관되므로 우연이 아니다.
이 카메라의 화면↔각도 관계는 tan 곡선이 아니라 **거의 선형**이고, 보정은 조준각을 깎아 과소조준을 만든다.

→ 구현했던 `tangentAimDeg`/`aimPtzForClick` 과 호출부를 **전량 되돌렸다**(작업트리 revert).
   이론이 아니라 측정이 정본이다.

## 4. 실측이 새로 드러낸 것(별건, 미수정)

- **pan**: 잔차가 클릭 방향과 같은 부호(−0.0076 / +0.0041) → 게인 크기가 약 1~2% 부족한 성격.
  다만 2표본이라 검증된 상수(−62)를 건드리지 않았다. 화면의 0.4~0.8% 수준.
- **tilt**: 두 표본의 errY 잔차가 **−0.0144 / −0.0144 로 동일**(오차 크기와 무관한 **상수 오프셋**).
  게인 오차라면 오차 크기에 비례해야 하는데 그렇지 않다. 후보: LPD 번호판 quad 중심의 스케일 의존 편향,
  또는 렌더 주점(principal point)의 수직 오프셋. **원인 미규명 → 보정하지 않았다**(원인 모르는 상수 보정은
  번호판이 아닌 클릭 대상에서 오히려 편향이 된다).

## 5. 현재 정확도(있는 그대로)

- 시뮬 기하 경로: 화면 양끝 클릭에서 최종 중심 오차 **|r| ≈ 0.015~0.017**(화면의 1.5~1.7%), 주성분은 위 tilt 상수.
- 실카 네이티브 경로(`setcenter`): 장비가 계산하며 우리가 개입할 인자가 없다. 실측 잔차 x −0.031(1회).

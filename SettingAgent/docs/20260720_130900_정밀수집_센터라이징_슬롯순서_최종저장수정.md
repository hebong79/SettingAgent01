## 1. 배경·증상

정밀수집(discovery → calibrating) 과정에서 슬롯별 번호판(LPD Box) 중심으로 PTZ 카메라를 센터링하는
`PtzCalibrator` 잡이 다음 세 가지 증상을 보였다(리더가 `data/slot_ptz.json` 17항목을 실증 분석해 확정).

1. **한 주차면 중복 센터링**: 서로 다른 슬롯이 물리적으로 같은 번호판에 수렴한다.
   - 증거: slot1·slot2 가 동일 PTZ `7.80/10.72`(둘 다 `plate_lost`), slot3·slot4 가 동일 PTZ `24.52/9.97`(둘 다 `converged`).
2. **다른 주차면 건너뛰기**: 위 중복의 결과로 실제로는 처리되어야 할 슬롯이 미처리로 남는다.
3. **최종 결과 저장 안 됨**: 센터링이 끝나도 그 결과를 한 파일로 갈무리한 최종 산출물이 없고, DB 저장도
   zoom 수렴 실패 슬롯을 통째로 누락했다.

## 2. 근본원인 3가지 (R1~R3)

### R1 — 공유 시작 PTZ 로 인한 latch(중복/건너뛰기)

`PtzCalibrator.startPtzFor`가 슬롯마다 **프리셋 공유 PTZ**(예: preset1 base ≈ `pan29.52/tilt9.45/zoom1.69`)를
그대로 `centerOnPlate`의 시작점으로 반환하고 있었다. `PlatePtz.centerOnPlate`는 시작점에서 가장 가까운
번호판을 `pickNearestPlate`로 골라 폐루프 추적을 시작하므로, 슬롯 A와 슬롯 B가 같은 넓은-FOV 시작점을
공유하면 인접한 두 슬롯이 **같은 번호판에 latch**된다.

DB 실측(`slot_setup`)으로 preset2 슬롯 8~13의 LPD 중심 `cx`는 `0.096 / 0.212 / 0.325 / 0.464 / 0.62 / 0.766`로
서로 뚜렷이 구분되는데도(간격 0.11~0.15), `slot_ptz.json`에서는 slot11/12/13이 동일 PTZ `55.44`로 수렴했다.
prior(LPD 중심)는 구분되는데 결과가 겹친다는 것은, 원인이 prior 자체의 중복이 아니라 **공유 시작점에서 폐루프가
이웃으로 갈아탄다**는 것을 뜻한다. 이 실측이 pre-aim(선조준) 처방의 유효성 근거다 — 만약 prior 자체가 중복이면
pre-aim으로도 동일한 latch가 재현되었을 것이나, 이번 경우는 아니다.

### R2 — DB 저장 게이트 과잉 필터링

`saveCenteringSlots`가 `it.centered && it.converged` 인 행만 `upsertSlotCentering`에 넘겼다. 번호판 pan/tilt는
정확히 맞았지만 zoom 폭만 못 맞춘(`centered:true, converged:false`) 슬롯은 pan/tilt 정보 자체가 유효함에도
DB에서 통째로 누락됐다(증거: slot1/2).

### R3 — 최종 결과 파일 부재

센터링이 끝난 뒤 그 결과를 담은 `save/` 스냅샷이 없었다. `Finalizer`가 만드는 `result_*.json`은 센터링 **이전**
시점의 `SetupArtifact`라 PTZ 값이 반영되어 있지 않다.

## 3. 수정 내용

### 3.1 R1 처방 — `preAimPtz` 선조준 + 타깃 정렬

**`PtzCalibrator.ts` — `preAimPtz(t, base)` 신규(private)**

슬롯의 LPD 박스 중심을 화면중앙(`0.5, 0.5`)으로 끌어오는 결정형 1스텝 좌표계산을 프리셋 공유 base PTZ에
얹어, 슬롯마다 **다른** 시작점을 만든다.

```
g   = scaleGainForZoom({ gainPan: cfg.fallbackGainPanDeg, gainTilt: cfg.fallbackGainTiltDeg, zoomRef: 1 }, base.zoom)
c   = center(t.plateRoi)                       // plateRoi 는 이미 quadBoundingRect rect
err = { errX: c.cx - 0.5, errY: c.cy - 0.5 }
pt  = panTiltCorrection(err, g, base.pan, base.tilt, PREAIM_MAX_STEP)   // PREAIM_MAX_STEP = 90°
return { pan: pt.pan, tilt: pt.tilt, zoom: base.zoom }                 // zoom 불변
```

- `controlMath.scaleGainForZoom`/`panTiltCorrection`, `domain/geometry.center`를 그대로 **재사용**하고
  본문·시그니처는 건드리지 않았다(PlatePtz/controlMath 코어 무접촉 원칙 준수).
- `PREAIM_MAX_STEP = 90`(°)은 별도 상수로 새로 도입했다. 기존 `cfg.maxStepDeg`(=5, 폐루프 미세보정용)는
  coarse 선조준에는 너무 작아 재사용하지 않는다는 판단(정상 오프셋은 ~18° 이내라 클리핑되지 않고, 이상 게인 방어
  상한으로만 작동).
- `calibrateSlot(t)`는 이제 `baseStart = startPtzFor(t)`(프리셋 base, 캐시 유지) → `startPtz = preAimPtz(t, baseStart)`
  순으로 계산하고, `centerOnPlate` 호출에서 **`plateRoi`를 더 이상 전달하지 않는다**. `PlatePtz`의 기본값
  `{0.5,0.5,0,0}`(화면중앙 최근접)이 그대로 적용되며, 선조준으로 대상이 이미 화면중앙 근처에 있으므로 그 슬롯의
  번호판이 픽업된다. zoom 단계(`quadBoundingRect(c.plate.quad)`, `gain: c.gain`)는 손대지 않았다.
- 부호 검증: 박스가 우측(`cx>0.5`)이면 `errX>0`, `gainPan<0`(cam1 실측 −62/z) → `-errX·gainPan>0` → `pan` 증가(우향).
  `detectMath`의 기존 방향 규약과 일치.

**`slotPtzWriter.ts` — `expandPlateTargetsFromSlotSetup` 정렬 추가**

```
targets.sort((a, b) => a.camIdx - b.camIdx || a.presetIdx - b.presetIdx || (a.globalIdx! - b.globalIdx!));
```

카메라 asc → 프리셋 asc → 슬롯(`globalIdx`=정수 slot_id, 항상 존재) asc 순서를 결정형으로 보장한다. `getSlotSetup`의
`ORDER BY` 자체는 손대지 않았다 — 뷰어/dbOverlay 등 공유 소비자에게 영향이 번지는 것을 피하기 위해 정렬 책임을
이 펼침 함수 안으로 국한했다.

### 3.2 R2 처방 — 저장 게이트 완화

**`PtzCalibrator.ts` — `saveCenteringSlots(items)`**

```
- if (!it.centered || !it.converged) continue;
+ if (!it.centered) continue;
```

`centered`(pan/tilt 수렴)만을 게이트로 삼는다. zoom 미수렴(`converged:false`)이어도 pan/tilt는 유효한 정보이므로
저장한다. `centered:false`(번호판 자체를 못 찾은 경우)만 제외해 오염을 막는다. DB `slot_setup` 스키마에는 zoom-수렴
여부를 담는 컬럼이 없으므로, 그 뉘앙스(정확한 converged/reason)는 `slot_ptz.json`이 정본으로 계속 보존한다.

### 3.3 R3 처방 — `save/Setup_*.json` 최종 스냅샷

**`SaveStore.ts` — `saveSnapshot(name, data)` / `setupSaveName(date)` 신규**

- `saveSnapshot`은 기존 `save(name, SetupArtifact)`의 미러이지만 `SetupArtifact` 타입 제약이 없다. `sanitizeName`
  (경로 traversal 차단) → `stringify5(data, 2)` → `save/{safe}.json` 기록, `reportsDir`가 있으면 best-effort 미러
  (실패해도 정본 `save/` 쓰기는 이미 끝난 뒤이므로 격리·warn 로그만 남긴다). 안전화 자체가 실패하면 throw.
- `setupSaveName(date)`은 `Setup_YYYYMMDD_HHMMSS`(로컬 시각) 포맷 — 기존 `defaultSaveName`(`result_` prefix)의 미러.
- 기존 `save(name, SetupArtifact)`는 **무변경**.

**`PtzCalibrator.ts` — `saveSetupSnapshot(items)` 신규(private) + `run()` done 경로 배선**

```
this.writer(buildSlotPtzJson(items, this.now()), this.cfg.outFile);
this.saveCenteringSlots(items);   // DB UPDATE 먼저
this.saveSetupSnapshot(items);    // done 경로에서만, best-effort
```

```ts
private saveSetupSnapshot(items: SlotPtzItem[]): void {
  if (!this.saveStore) return;
  try {
    this.saveStore.saveSnapshot(setupSaveName(new Date()), {
      createdAt: this.now(),
      slots: this.store.getSlotSetup(),   // DB UPDATE 이후 재조회 → PTZ 반영된 최신 뷰
      centering: items,
    });
  } catch (e) {
    logger.warn({ err: e }, 'Setup 스냅샷 저장 실패(격리)');
  }
}
```

- **리더 확정(D-1=a2)**: 설계서 원안(A-1)은 스냅샷 payload를 `buildSlotPtzJson(items)`(a1, `slot_ptz.json`과 동형)로
  제안했으나, 리더가 "완전한 최종 셋업 1파일(기하+PTZ)"을 요구해 **a2**로 확정했다. `slots` 필드는
  `saveCenteringSlots`(DB UPDATE) **이후** `this.store.getSlotSetup()`을 재조회한 값이라 PTZ가 반영된 최신 뷰이고,
  `centering` 필드는 `items`(converged/reason 등 센터링 상세)를 그대로 담는다.
- `saveStore`는 생성자 옵셔널 의존성(`Pick<SaveStore, 'saveSnapshot'>`)으로 가산됐다. 미주입 시 스냅샷은
  no-op이라 기존 수동 흐름·테스트는 회귀 없이 그대로 동작한다.
- `run()`의 **error 경로에서는 스냅샷을 기록하지 않는다** — 부분·불신 상태를 최종 산출물로 남기지 않기 위함.

**`index.ts` — 배선 1줄**

```
- const calibrator = new PtzCalibrator({ camera, lpd, cfg: tools.calibrate, store: sqlite,
+ const calibrator = new PtzCalibrator({ camera, lpd, cfg: tools.calibrate, store: sqlite, saveStore,
    onFinished: (s) => pipeline?.onCalibrateFinished(s) });
```

이미 생성돼 있던 `saveStore` 인스턴스를 주입한 것으로, `PtzCalibrator` 인스턴스는 수동 `/calibrate/ptz`
라우트와 auto-chain(`SetupPipeline`) 양쪽이 공유하므로 이 한 줄로 **양 진입점 모두**에 스냅샷 기능이 반영된다.

## 5. 입출력·계약

- **REST 계약 변경 없음** — `/calibrate/ptz`, `/calibrate/status`, `/calibrate/frame` 등 기존 라우트의 요청/응답
  shape은 그대로다. 이번 수정은 `PtzCalibrator` 내부 로직과 그 부수효과(파일/DB 저장)에 한정된다.
- **신규 파일 산출물**: `save/Setup_YYYYMMDD_HHMMSS.json` — 센터링 잡이 `done`으로 끝날 때마다 1개 생성.
  기존 `/capture/save`(수동 저장, `SetupArtifact` 검증 경유)와는 **별도 경로**로, 이 스냅샷은 잡 내부에서
  `saveStore.saveSnapshot()`을 직접 호출해 만든다(REST 바디 검증 없음, `SetupArtifact` 스키마 아님). 상세 shape
  불일치는 §7 영향도 분석 참조.
- **DB 계약**: `slot_setup`의 pan/tilt/zoom/centered 컬럼에 대한 UPDATE 대상 행 집합이 넓어졌다(게이트 완화).
  컬럼 자체·UPDATE 방식(부분 UPDATE, DELETE+INSERT 아님)은 무변경.

이동)을 구계약이 반영하지 못한 stale 테스트였다. QA는 이를 산식으로 재계산해 일치를 확인한 뒤(예: base
`0/0/1` + LPD 중심 `(0.645,0.635)` → `errX=0.145, errY=0.135`, 게인 `{-62,-35.5}` → `dPan=8.99, dTilt=4.7925`,
실측과 완전 일치), 매직넘버를 구현 미러 함수 `preAimOf(base)`로 대체한 **동등 이상으로 엄격한 신 계약**으로
갱신했다(게인 상수가 바뀌면 기대값도 같이 움직여 회귀를 자동 감지).

**프로세스 지적(QA→향후 반영)**: 컴포넌트 단위 회귀 주장은 그 컴포넌트의 모든 관련 테스트 파일(사이드카 포함)을
포함한 전체 스위트 실행 기준으로 해야 한다. 부분 실행 결과만으로 "PASS"를 보고하면 이번처럼 사이드카 파일의
stale 실패를 놓칠 수 있다.

## 7. 라이브 검증 한계 (은닉 금지)

**시뮬레이터(13100) DOWN 상태로 이번 라운드에는 검증 불가능**했던 항목:

- 실 PTZ 물리 수렴 — pre-aim이 실카메라에서 정말로 정판을 화면중앙에 두는지.
- **비-cam1 게인 정확도** — fallback 게인(`-62/-35.5`)은 cam1 실측값이다. 다른 카메라에서 pre-aim은 coarse
  용도라 게인 오차 50%에도 이웃 간 disambiguation(간격 0.11~0.15의 절반)에는 충분하다는 것이 설계 가정이지만,
  **최종 확증은 라이브 필요**.
- `/calibrate/ptz` 실발화 후 `data/slot_ptz.json` 순서, DB `slot_setup.centered` 행, `save/Setup_*.json` 실파일
  존재를 실데이터로 확인하는 라우트/데이터 실측(설계 B-2) — **미수행**.

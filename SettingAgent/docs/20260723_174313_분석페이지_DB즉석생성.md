# 분석 탭 DB 즉석 생성 — GET /mapping DB 폴백 조립

## 1. 배경 / 문제

SettingAgent 웹 UI의 '분석(analyze)' 탭은 요약카드 · 프리셋별 요약 · 주차면 목록 · 전역 인덱스 매핑 · 경고 · LLM 리포트를 표시한다. 이 데이터는 `GET /mapping`(및 뷰어용 `GET /viewer/api/mapping`) → `repo.loadArtifact()` → `data/setup_artifact.json` **파일만**을 소스로 삼는다(파일 없으면 404).

`setup_artifact.json`을 쓰는 유일한 경로는 `Finalizer.finalize → repo.saveArtifact`인데, 실제 UI의 3가지 진입점 모두 이 경로에 도달하지 못했다.

| 진입점 | 실제 동작 | 결과 |
|---|---|---|
| 정밀수집 시작(`SetupPipeline.startPrecise`) | `discovering → calibrating → done`으로 종료, `finalizing` 스킵 | 파일 안 씀 |
| 수집 시작(`capCaptureStart`) | `autoChain` 기본 false → 체인 미진행 | `finalize` 미호출 |
| 최종화 버튼(`capFinalize`) | 의도적으로 표시전용(`Finalizer.replaceSlotSetup`이 `pan/tilt/zoom=null·centered=0`으로 센터라이징을 파괴하므로 미호출) | 파일 미갱신 |

결과적으로 디스크의 `setup_artifact.json`은 `presets:[], slots:[], globalIndex:[]`인 빈 상태로 남아 있었고, 분석 탭은 항상 0으로 표시됐다. 반면 실제 슬롯 기하 · LPD · 센터링 데이터는 `slot_setup`(SQLite DB)에 이미 채워져 있다.

## 2. 해결 방향 (마스터 결정, 2026-07-23)

**"DB에서 즉석 생성".** `GET /mapping`·`GET /viewer/api/mapping`이 파일이 없거나 `slots`가 비어 있을 때, `store.getSlotSetup()`(SQLite `slot_setup` 조회)에서 `SetupArtifact`를 즉석 조립해 반환한다.

- **파괴적 경로 미경유**: `Finalizer.replaceSlotSetup`/`finalize()`는 호출하지 않는다. 순수 읽기(`getSlotSetup`)만 사용하므로 센터라이징(`pan/tilt/zoom/centered`)이 보존된다.
- **파일 우선 규칙**: 파일에 `slots.length > 0`이면 DB 조회 자체를 하지 않고 파일 그대로 반환한다(수동 `PUT /mapping` 편집 보존).
- 파일도 없고 DB도 빈 경우에만 404.

## 3. 신규 함수 — `src/setup/artifactFromSlotSetup.ts`

순수·결정형 함수. `server.ts`/`captureRoutes.ts`를 import하지 않아(타입만 import) 순환참조가 발생하지 않는다.

```ts
export function buildArtifactFromSlotSetup(
  views: SlotSetupView[],
  now: () => string = () => new Date().toISOString(),
): SetupArtifact
```

### shape 변환 규칙

| artifact 필드 | 소스(SlotSetupView) | 변환 규칙 |
|---|---|---|
| `slot.slotId` | `v.slotId: number` | `String(v.slotId)` |
| `slot.zone` | (DB에 없음) | `` `cam${v.camId}` `` — Finalizer 강등값과 동일 |
| `roiByPreset[presetKey]` | `v.roi: NormalizedPoint[]`(폴리곤) | 축정렬 **bbox rect** `{x=min,y=min,w=maxX-minX,h=maxY-minY}` (전용 `bboxOf` 헬퍼, 빈 배열이면 0-rect 방어) |
| `plateRoiByPreset[presetKey]` | `v.lpd: quad \| null` | **lpd 있을 때만** 키 추가(quad 그대로) |
| `presetKey`/`preset.label` | `v.presetKey`(`` `${camId}:${presetId}` ``) | DB 파생값 그대로 재사용, 그룹당 preset 1개 |
| `preset.coveredSlotIds` | 그룹 내 각 행 | `presetSlotIdx` 오름차순(SQL 정렬 보존) |
| `globalIndex[]` | 각 행 | `{globalIdx: v.slotId, slotId: String(v.slotId), camIdx: v.camId, presetIdx: v.presetId}` — globalIdx는 DB slotId(전역번호)를 그대로 사용, 재부여하지 않음 |
| `createdAt` | — | `now()`(기본 현재 ISO, 테스트 결정성을 위해 주입 가능 — Finalizer와 동일 패턴) |
| `report`/`warnings`/`floorRoiByPreset` | — | **생략**(DB에 해당 데이터 없음) |

**coverage 보장**: `slots[].slotId` 집합과 `globalIndex[].slotId` 집합이 항상 동일한 `String(v.slotId)` 집합이므로 `validateCoverage(...).ok === true`가 항상 성립한다. 이 덕분에 DB 조립 artifact를 그대로 `PUT /mapping`으로 재저장해도 `validateArtifactBody`를 통과한다.

## 4. 라우트 변경 — `src/api/server.ts`

`buildServer(deps)` 본문에 폴백 결정 클로저를 추가했다(라우트 등록 전, `deps`·뷰어 클로저 모두 접근 가능):

```ts
function resolveMapping(): SetupArtifact | null {
  const file = deps.repo.loadArtifact();
  if (file && Array.isArray(file.slots) && file.slots.length > 0) return file; // 파일 우선
  const views = deps.sqlite ? deps.sqlite.getSlotSetup() : [];
  if (views.length > 0) return buildArtifactFromSlotSetup(views); // DB 폴백
  return null; // 404
}
```

`GET /mapping`(server.ts:249)과 `GET /viewer/api/mapping`(server.ts:314) 두 라우트 모두 기존 `deps.repo.loadArtifact()` 직접 호출을 `resolveMapping()`으로 교체했다. `PUT /mapping`·`PUT /viewer/api/mapping`(`saveMappingHandler`)은 무변경.

**404 판정 미세 변경**: 기존에는 파일이 `slots:[]`인 빈 상태로라도 존재하면 200(빈 배열)을 반환했으나, 이제는 그 경우에도 DB를 조회해 DB가 비어 있으면 404로 통일된다(과제 명세 "빈파일+빈DB→404"와 일치, QA 테스트 케이스 d로 커버).

## 5. 프론트 변경 — `web/app.js`

`pollPipeline()`의 정밀수집 done 분기(2391행대)에서, 전이 1회 가드 안에 분석 탭 자동 갱신을 추가했다.

```js
if (prevPipelineStage !== 'done') {
  showPreciseResult(pl, msg);
  // 정밀수집 종료 → DB 조립 매핑을 분석 탭이 즉시 반영(그 탭을 보고 있을 때만 1회).
  if (!$('analyze-view').hidden) await renderAnalysis();
}
```

- 활성 탭 판별은 `!$('analyze-view').hidden`(`setTab()`이 토글하는 hidden 플래그) 사용 — 별도 탭 상태 변수가 없어 이 신호가 가장 직접적이다.
- 분석 탭이 보이지 않으면 no-op, 이미 `done`으로 전이 처리됐으면 no-op(멱등).
- 기존 트리거(`#an-refresh` 클릭, `setTab()` 전환, 수동 저장 후 재렌더)는 전부 무변경으로 보존된다.

# 뷰어 오버레이 드래그 크기조정 미동작 — 근본 원인 수정

- 작성: 2026-07-01 16:06
- 대상: SettingAgent 웹 뷰어(`web/app.css`, 테스트 `test/viewerOverlayInteractive.test.ts`)
- 증상: 주차면 목록에서 아이템 선택 시 분홍(빨강 `#ff4d4d`)으로 표시는 되나, 캔버스 위에서 마우스 좌클릭·드래그해도 박스 크기(코너/변)가 전혀 변경되지 않음.

## 1. 근본 원인 (단일)

`web/app.css` 의 오버레이 캔버스 규칙:

```css
.viewport canvas {
  position: absolute; left: 0; top: 0;
  pointer-events: none;   /* ← 근본 원인 */
}
```

- HTML 구조상 `<canvas id="overlay">` 가 `<img id="frame">` 위에 절대배치로 정확히 겹쳐 있다(`web/index.html`).
- 캔버스에 `pointer-events: none` 이 걸려 있어 **모든 마우스 이벤트가 캔버스를 통과해 하위 `<img>` 로 빠진다**. `<img>` 에는 편집 핸들러가 없으므로 아무 일도 일어나지 않는다.
- 그 결과 `web/app.js` `wireOverlayEditing()` 의 `overlay.addEventListener('mousedown' | 'mousemove' | 'mouseup')` 가 **한 번도 발화하지 않는다** → 코너/변 드래그, 캔버스 클릭 선택 모두 무효.
- 목록 선택(분홍 표시)은 별도 리스트 클릭 핸들러(`renderSlotList`)라 정상 동작 → "선택은 되는데 드래그만 안 되는" 증상과 일치.

### 왜 이전 유닛테스트가 못 잡았나
- 크기조정 순수 로직(`core.js resizeRect` 등)은 vitest 로 검증됨(정상). 그러나 실제 미동작의 원인은 **CSS 계층(pointer-events)** 이라 순수 함수 테스트 범위 밖이었다.
- 도입 이력: `pointer-events: none` 은 2026-06-26 뷰어 초기 CSS(커밋 b9262f0f)에 ROI 편집 기능이 생기기 전 추가됨. 이후 편집 기능이 그 위에 얹혀 **실브라우저에서 한 번도 동작한 적이 없었다**(코너 리사이즈 포함).

## 2. 수정

`web/app.css` — 오버레이 캔버스가 마우스 이벤트를 수신하도록 변경(외과적 1줄):

```css
.viewport canvas {
  position: absolute; left: 0; top: 0;
  pointer-events: auto; /* 오버레이에서 슬롯 선택·크기조정(코너/변) 드래그 수신 — none 이면 img 로 통과되어 편집 불가 */
}
```

- `<img>` 는 클릭 동작이 없으므로 이벤트를 캔버스가 받아도 부작용 없음.
- ROI 숨김/수집 중에는 `mousedown` 핸들러가 `if (state.roiHidden || !state.mapping) return;` 로 조기 반환 → 무해.
- JS 결선·리사이즈 로직은 이미 정상이라 **수정 없음**. 이 CSS 한 줄이 유일한 차단 지점이었다.

## 3. 테스트

신규 회귀 가드 `test/viewerOverlayInteractive.test.ts` (CSS 계층 회귀는 순수함수 테스트로 못 잡으므로 소스 레벨로 가드):

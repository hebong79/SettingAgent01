## 0. 마스터가 확정한 전제 (재정의)

1. **셋팅 전용 루틴**이다. 실시간성 불필요 — **누적 허용, 느린 속도 허용**.
2. **결정론 우선 원칙**: 결정론적으로 해결하고, 안 되면 그때 LLM 사용.
3. **시뮬레이터는 전 주차면 만차에서 시작** → 엄밀히는 "차량 존재 파악" 자체가 불필요.
4. **리얼 카메라는 공차 가능성이 높음** → 누적(시간에 걸친 커버리지)이 필요.
5. **핵심 산출물 = 주차장관리 에이전트용 setup 파일: 각 주차면 슬롯의 번호판 인식을 위한 최적 PTZ 값**(slot_ptz).
6. **리얼 현장을 3D화**해 리얼과 시뮬레이터를 같은 조건(디지털 트윈)으로 만들 예정. 시뮬 방법을 리얼 PTZ 카메라에도 직·간접 연계.

## 1. 논의 경과 요약 (점유 판정 탐색의 기록)

| 단계 | 내용 | 결론 |
|------|------|------|
| VPD bbox 점유 | 현행: 접지밴드 argmax 귀속 | ~70% 검출·오검 → base로 취약 |
| seg 마스크 점유 | 마스크 ∩ 슬롯 면적비 | bbox 보다 강하나 **미검출 여전** (마스터 실측) |
| 앞면중심(기하) | 슬롯별 고정 프로브 지점 — 검출 무관 | "어디를 볼지(WHERE)"는 완전 해결. "있나(WHETHER)"는 별도 |
| VLM yes/no (Qwen VL 3B) | 육면체 마킹 크롭에 "이 구역 바닥에 차?" | 구조적으로 타당하나 **100% 증명 아님**, 3B는 악조건 취약 |
| 실카 전이성 | 기하+VLM은 전이됨, 시뮬 정답은 전이 안 됨 | 시뮬 정답은 **오프라인 오라클**(검증·튜닝)로만 사용이 정공 |
| 결정론 원칙 적용 | 누적·저속 허용 반영 | **에스컬레이션 사다리**가 최적 (아래 §2) |
| "프로브가 증명인가" | 조준·판정룰=결정론, 증거(검출)=통계 | **절차적 결정론이지 증명 아님**. 증명은 시뮬 오라클/사람만 |

### 에스컬레이션 사다리 (점유가 필요할 때의 확정안)
```
1단계  라운드 누적(Aggregator support) — 결정론 룰, 기존 인프라
2단계  앞면중심 조준 PTZ 근접 프로브 — 기하 조준(결정론) + 근접 검출(고신뢰 증거) + 다수결 룰(결정론)
3단계  잔여 애매 슬롯만 VLM yes/no(육면체 마킹 크롭) — "결정론 안 되면 LLM" 원칙
4단계  최후 잔여 사람 1회 확정(뷰어) — 증명 간극을 닫는 게이트
```
- 성격 명시: 1~2단계는 오류율을 결정론적 룰로 압축, 3단계는 꼬리 처리, 4단계가 확실성 담보.
- 비전에서 "증명"은 원리적으로 불가 — 세계 상태를 아는 오라클(시뮬 정답, 사람)만 증명 가능.

## 2. 목표 재정의가 바꾸는 것 (핵심 통찰)

### 2-1. 점유 판정은 base가 아니라 **게이트**로 강등된다
- 진짜 base = **슬롯 폴리곤(PtzCamRoi, 결정론 파일) + 슬롯별 센터링(PlatePtz 결정론 폐루프)**.
- 점유 판정의 역할은 "이 슬롯에 지금 센터링을 시도할 가치가 있나"의 **게이트**뿐.
- **시뮬(만차 시작)**: 게이트 자체가 불필요 → "전 슬롯 점유 가정" 모드로 충분.
- **리얼**: 게이트가 틀려도 해가 없다 — 공차 슬롯에 센터링을 시도하면 `no_plate` 로 실패할 뿐이고, **그 실패 자체가 "공차" 신호다(결정론, 공짜)**. 즉 별도의 정밀 점유판정 모듈 없이도: *시도 → no_plate → 나중에 재시도(누적)* 로 자연 해결.
- ∴ 이전 논의의 "점유 100% 필요" 부담이 대부분 소멸. VPD 불안정도 핵심 산출물(slot_ptz)에는 치명적이지 않게 된다.

### 2-2. 앞면중심(slot3d_front_center)의 진짜 자리 = **보편 조준 prior**
- 현행 센터링 대상 열거는 slot_setup 의 번호판 ROI 보유 슬롯만(`expandPlateTargetsFromSlotSetup`) → 원거리 LPD 미검출 슬롯은 **대상에서 누락**(검출 의존).
- `slot3d_front_center` 는 **검출 무관·전 슬롯 항상 존재**(기하 산출, 이미 DB 저장) → 이것을 조준 prior 로 쓰면 **대상 열거가 검출에서 독립**된다.
- 스테이지드 접근: 앞면중심으로 조준 → 슬롯 폭 기준 1차 줌 → 근접에서 LPD 검출 → PlatePtz 폐루프(센터링·줌 수렴). 원거리 검출 실패가 더 이상 슬롯을 누락시키지 못함.

## 3. 필요한 기능 / 해야 할 일 (객관 정리)

### A. 공통 코어 (시뮬·리얼 동일)
| # | 기능/과업 | 상태 | 비고 |
|---|-----------|------|------|
| A1 | 파이프라인 순서 가드: **최종화 없이 센터라이징 방지**(stale slot_setup 검출 — updatedAt/세대 비교) | 미구현(취약점 확인됨) | PtzCalibrator.start 가 낡은 slot_setup 을 그대로 사용 가능 |
| A2 | **앞면중심 조준 prior 센터링**: slot3d_front_center 기반 대상 열거·조준(§2-2). 원거리 LPD 의존 제거 | 미구현(핵심 제안) | PlatePtz 재사용 + 조준·1차 줌 스테이지 추가 |
| A3 | **PTZ 검증 패스**: 저장된 slot_ptz 로 슬롯 재방문 → LPD 재검출 → 중심·폭 허용치 확인 → 검증 상태 기록 | 미구현 | converged 플래그만으론 부족. 산출물 신뢰의 마감재 |
| A4 | **증거 이미지 저장(img1)**: 센터링 성공 시 근접 JPEG 저장 → 사람 검수·사후 감사 | 컬럼만 존재(null) | slot_setup.img1 채우기 |
| A5 | 점유 판정 **게이트로 강등/옵션화**: "전 슬롯 점유 가정" 모드(시뮬), no_plate=공차 신호 활용(리얼) | 정책 변경 | 기존 occupancy 로직은 표시·통계용으로 유지 |
| A6 | slot_ptz 산출물 계약 확정: slotId/globalIdx·pan/tilt/zoom·plateWidth·quality·calibratedAt·source(sim-prior/real-refined)·verified | 부분 존재 | 관리 에이전트(Action/DM)와의 소비 계약 명문화 |

### B. 시뮬 전용 (지금 바로)
| # | 기능/과업 | 비고 |
|---|-----------|------|
| B1 | 만차 전제 전 슬롯 센터링 일괄 실행 → slot_ptz 완성 (A2+A3 적용) | 현행 흐름의 완성형 |
| B2 | 시뮬 정답(있다면)은 **오프라인 오라클**로: 판정·센터링 결과 정확도 측정에만 사용, 런타임 로직에 미사용 | 실카 전이성 보존. 현재 시뮬 API에 정답 엔드포인트 없음(추가 요청 대상) |

### C. 리얼 전용 (누적 체계)
| # | 기능/과업 | 비고 |
|---|-----------|------|
| C1 | **슬롯별 누적 상태머신**: uncalibrated → (차량 존재 시) 센터링 시도 → calibrated(질·시각 기록) / no_plate → 재시도 큐. 전 슬롯 커버까지 반복(일 단위 누적 허용) | 핵심 신규. 재개 가능한 잡 + 스케줄링 |
| C2 | **카메라별 게인 캘리브레이션**: PlatePtz fallback 게인(−62/−35.5)은 cam1 시뮬 실측 상수 — 리얼 카메라마다 probe/diagSweep 자동 실측 루틴 필요 | 코드에 위험 명시돼 있음 |
| C3 | **드리프트 관리**: 실 PTZ 기계 오차 — 주기 재검증(A3 재실행)·기준프레임 자동보정(refframe/estimateAlign 연계)·재캘리브레이션 트리거 | 기존 자동보정 인프라 확장 |
| C4 | **줌 매핑 캘리브레이션**: 리얼 벤더 줌 스케일 ↔ 시뮬 줌 스케일 대응(지면모델 f 추정과 정합) | 트윈 전이의 전제 |
| C5 | 악조건 대응 정책: 야간/역광 시 재시도·시간대 스케줄, LPD 저신뢰 시 보류(위장 성공 금지) | 은닉 금지 원칙 |

### D. 디지털 트윈 (리얼 3D화)
| # | 기능/과업 | 비고 |
|---|-----------|------|
| D1 | **트윈↔리얼 정합 도구**: 주차선 코너 등 기준점(fiducial) 기반 등록, 카메라 위치·높이·FOV 매칭 검증 | estimateAlign 상호상관 재활용 가능 |
| D2 | **트윈에서 셋업 리허설**: 트윈에서 PtzCamRoi·지면모델·slot_ptz 후보를 선산출 → 리얼에서는 탐색이 아니라 **검증·미세보정만**(리얼 현장 시간 최소화) | sim-prior → real-refine 구조. source 필드(A6)로 이력 구분 |
| D3 | **파리티 검증**: 트윈 렌더 vs 리얼 프레임 오버레이 비교(슬롯 폴리곤·육면체·앞면중심 투영 오차 지표) | 지면모델은 이미 "이미지 점+zoom"만 사용(Unity 내부값 불사용)이라 구조적으로 전이 가능 — 수치로 확인 |
| D4 | 트윈 갱신 절차: 현장 변경(차선 재도색·카메라 이설) 시 재정합 워크플로 | 운영 단계 |

### E. 우선순위 로드맵 (권고)
```
P0 (지금·시뮬):   A1 순서가드 → A2 앞면중심 prior 센터링 → A3 검증패스 + A4 img1 → A5 게이트 강등 → B1 전슬롯 slot_ptz
P1 (리얼 준비):   C2 게인 캘리브레이션 → C1 누적 상태머신 → C3 드리프트 → C4 줌 매핑 → C5 악조건 정책
P2 (트윈):        D1 정합 → D2 리허설(선산출→미세보정) → D3 파리티 → D4 갱신 절차
```

## 4. 정직한 한계 (기록)

- 근접 센터링·검출도 이론상 100% 증명은 아니다(§1). 증명 간극은 **A3 검증패스 + 사람 확정 게이트**가 닫는다.
- 리얼 신뢰도는 시뮬 성적으로 보장되지 않는다(도메인 갭) — C 단계의 실데이터 검증이 필수.
- VLM(3단계)은 잔여 꼬리 전용으로 유지 — 전면 사용은 결정론 원칙과 비용상 비최적.

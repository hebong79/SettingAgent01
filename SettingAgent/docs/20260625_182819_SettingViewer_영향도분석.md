## 2. 기존 기능 보존 (핵심 — 의존성 그래프 추적)

### 2.1 SettingAgent `CameraClient` — 시그니처 불변, `listCameras` 는 SettingViewer 로 이동
- **불변 확인**: SettingAgent CameraClient 의 `requestImage`/`move`/`health`/`clampZoom` 4개 메서드 시그니처·본문 변경 0. `listCameras()` 는 뷰어 전용이라 SettingAgent CameraClient 에서 제거되고 **현재는 SettingViewer 의 CameraClient 에 위치**한다.
- **기존 소비자 무영향**: `SetupOrchestrator`(셋업 캡처)·`UnityPresetProvider`(프리셋 목록)·`discover.ts`(자동 탐색)는 모두 기존 메서드만 호출 → 영향 없음. 검증서에서 기존 81 테스트 그린으로 회귀 부재 확인.
- **`listCameras` vs `UnityPresetProvider` 파서 중복 아님**: SettingViewer 의 `listCameras`는 A타입 **전체 구조**(중첩 presets, enabled=false 보존) 반환, SettingAgent 의 `presetProvider`는 **flatten된 `CameraView`**(enabled=false 제외) 반환. 용도가 달라 의도적 분리. 통합은 이번 범위 밖.

### 2.2 셋업 파이프라인 · `SetupArtifact` 계약 — 무영향
- 뷰어는 `/mapping`(GET)으로 `SetupArtifact`를 **읽기 전용 소비**만 한다(`app.js`의 `loadMapping`). 산출물 생성 경로(`SetupOrchestrator`/`Repository`/`mapTargets`)는 한 줄도 건드리지 않음.
- `setup_artifact.json` 구조(presets/slots/globalIndex/roiByPreset/plateRoiByPreset) 계약 무변경 → ActionAgent/DMAgent의 산출물 소비에 영향 0.

### 2.3 `@parkagent/types` — 무영향
- 뷰어 모듈(`SettingViewer/src/viewer/*`)은 `@parkagent/types`를 **import하지 않음**(grep 확인). `CameraSource.ts`는 자체 타입(`Ptz`/`CameraList` 등)을 정의한다.
- 공유 도메인 타입(SlotState/ParkingEvent/Preset/ParkingSlot/NormalizedRect 등) **변경 0** → 이 타입을 소비하는 ActionAgent·DMAgent로의 파급 없음. 뷰어는 `NormalizedRect`(0~1 ROI)를 `/mapping` JSON으로 간접 소비할 뿐 타입 패키지에 의존하지 않는다.

### 2.4 ActionAgent / DMAgent — 무관
- 뷰어는 SettingViewer 단독 서비스(:13030). REST 계약(`/viewer/api/*`)은 신규 prefix이며 타 에이전트가 호출하지 않는다. SettingAgent 에는 ROI(/mapping) 프록시로만 의존. 공유 타입·산출물 계약 무변경이므로 타 에이전트 코드·테스트에 전파 없음.

## 3. 의존성 추가 (`@fastify/static@9.1.3`) 영향

- 설치본: `package-lock.json` 기준 **9.1.3**(fastify 5.x 호환). `package.json`은 `^9.0.0`.
- 사용처: `SettingViewer/src/viewer/routes.ts` 1곳(`app.register(fastifyStatic, { root: resolve(viewer.staticDir), prefix:'/viewer/', redirect:true })`). SettingAgent package.json 에서는 제거됨.
- 영향: 셋업/MCP 경로와 무관. 정적 자산 서빙 전용. `viewer.enabled=false`면 라우트 자체가 등록되지 않으므로 정적 서빙도 비활성. 번들·런타임 부담 미미.
- 검증: typecheck 통과, viewerRoutes 테스트가 임시 staticDir로 정적 서빙·`/viewer`→302·API 우선순위 확인. (단, 실제 `web/` 자산 브라우저 로딩은 수동확인 영역 — 미검증.)

## 4. 어셈블리/조립부 변경 영향

### 4.1 SettingViewer 부트스트랩
- `buildSourceRegistry(config)` 호출 → 서버에 `viewer`, `sources` 주입. ROI 는 `/viewer/api/mapping` 으로 SettingAgent :13020 의 /mapping 프록시.
- SettingAgent 부트스트랩(`index.ts`)의 기존 조립(camera/vpd/lpd/repo/brain/orchestrator/presetProvider)은 순서·인자 불변(뷰어 주입 제거됨).

### 4.2 SettingViewer 서버 진입점 (`ApiDeps`)
- `ApiDeps`에 `viewer?`·`sources?` → `deps.viewer?.enabled && deps.sources`일 때만 `void registerViewerRoutes(...)`.
- SettingAgent 서버(`server.ts`)는 뷰어 라우트 등록이 제거되어 셋업·/mapping 제공에 집중.
- **비동기 호출 주의(설계 결함 아님)**: `registerViewerRoutes`는 async(@fastify/static 등록 await)지만 `void`로 호출. Fastify가 플러그인/라우트를 큐잉하므로 `app.ready()`/`listen()` 시점에 순서대로 등록·완료된다. `buildServer`를 async로 바꿔 `index.ts`/테스트로 파급시키지 않은 외과적 선택. inject 스모크로 API→static 우선순위·정적 서빙 모두 확인됨.

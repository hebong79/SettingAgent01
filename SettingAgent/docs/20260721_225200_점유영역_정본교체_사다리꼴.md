## 1. 지적(마스터, 2026-07-21)

참조 이미지 `etc/주차면점유영역_02.jpg`의 녹색 박스가 점유영역 정본 형상이다 — 번호판 박스와 상하·좌우 평행이고, 위쪽이 약간 길고 아래쪽은 주차면 아래선까지 내려오는 **큰 사변형**(번호판 폭의 여러 배).

그런데 직전 작업("점유영역 생성" 버튼 신설, `docs/20260721_221700_점유영역생성_버튼_공통진입점.md`)이 만든 생성식은 **번호판 크기의 작은 박스**를 만들었다. 마스터 지시: "기존 방법이 존재하면 그 방법을 사용하라", "외부에서 독립적으로 사용 가능하게 함수화하라".

## 2. 원인

직전 구현은 다음과 같았다(`floorRoi.ts`, 이제는 제거됨):

```ts
export function buildOccupyRangeFromPlate(plateQuad: NormalizedQuad): NormalizedQuad {
  return buildPlateAnchoredQuad(quadBoundingRect(plateQuad), plateQuad);
}
```

`buildPlateAnchoredQuad(vehicle, plateQuad)`는 원래 **차량 VPD bbox**(`vehicle`)를 받아 그 위에 번호판 위치를 얹어 차량 발자국을 만드는 함수다. 그런데 이 경로엔 VPD가 없어 번호판 자신의 bounding rect(`quadBoundingRect(plateQuad)`)를 `vehicle` 자리에 대리로 넣었다 — 그 결과 산출 폭·깊이가 "번호판 bounding rect" 크기로 축소됐다(차량 크기가 아니라 번호판 크기). `PlateDiscoveryJob`(discovery)도 같은 식을 인라인으로 쓰고 있었으므로 DB에 쌓인 `occupy_range` 값 전부가 이 축소된 형상이었다.

## 3. 기존 정본(재발견)

뷰어 라이브 오버레이(`web/app.js`의 `updateLogicOccupancy`)가 이미 참조 이미지와 같은 형상을 그리고 있었다 — `web/occupancyRegion.js`의 `computeOccupancyRegions`. 이 파일은 이번에 새로 만든 게 아니라 `docs/20260703_102157_주차면점유영역_가변다각형_비겹침.md` / `docs/20260703_200500_주차면점유영역_번호판LPD기준_재정의.md`에서 이미 설계·구현돼 있던 정본이며, 저장 경로(discovery·버튼)만 이를 몰랐다.

알고리즘 요지(번호판 quad의 중심·단위 가로축 û/세로축 v̂ 기준):
- 아래변 폭 = 번호판 폭 × 배율(3.5~4.0, 슬롯 간 겹침 회피를 위해 전역 이진탐색 → 그래도 겹치면 겹치는 쌍만 개별 shrink)
- `topWidthRatio = 1.0` → 위/아래 변 폭 동일(평행사변형, 원근 수축 없음)
- `upRatio = 0.90`(중심에서 위로), `downRatio = 0.60`(중심에서 아래로) → **위가 더 김**(참조 이미지와 일치)
- 화면 경계(단위 정사각형)로 클립

## 4. 변경 내역

### 4.1 신규 `src/domain/occupancyRegion.ts`
위 알고리즘의 서버(TS) 판. DB/HTTP/DOM 의존 0, 난수·시각 0(순수·결정형). 기하 프리미티브(`clipByHalfPlane`/`convexIntersectionArea`/`polygonArea`)는 `src/domain/polygon.ts`를 재사용 — 신규 발명 없음.

**외부 독립 사용 진입점**(마스터 지시대로 함수화):
```ts
export function buildOccupyRegionsBySlot(
  plates: Array<{ slotId: number; quad: NormalizedQuad }>,
  cfg?: Partial<RegionConfig>,
): Map<number, NormalizedPoint[]>
```
같은 프리셋(같은 프레임)의 번호판 quad들을 한 번에 넘기면 슬롯ID→점유영역 폴리곤 Map을 돌려준다. 겹침 회피 배율 탐색이 **프레임 단위 집합 연산**(어떤 슬롯 쌍이 겹치는지는 같은 화면의 다른 슬롯들이 있어야 판단 가능)이라 단일 슬롯씩 독립 호출하면 안 되고, 반드시 같은 프리셋 단위로 묶어 호출해야 한다. 그 외 `plateAxes`/`buildTrapezoid`/`clampToUnit`/`computeOccupancyRegions`/`REGION_DEFAULTS`도 export해 필요시 더 세밀하게 재사용 가능하다.

**구현 2벌 존재의 이유와 안전장치**: 정적 배포 경계상 서버(Node)가 브라우저 순수 ESM(`web/occupancyRegion.js`)을 import할 수 없어 같은 알고리즘을 TS로 옮겨 적었다. 신규 `test/occupancyRegionParity.test.ts`(10케이스: 단독·회전·7면 실측유사·경계클립·밀집·퇴화 + 형상 상수 고정)가 두 구현의 출력 완전 동일성을 강제해 정의가 갈라지는 것을 막는다.

### 4.2 `src/api/captureRoutes.ts` — `POST /capture/slots/occupy`
기존 라우트(§API 계약 자체는 무변경 — 요청/응답 스키마 동일)의 내부 생성 로직만 `buildOccupyRegionsBySlot`으로 교체했다. 겹침 회피가 프레임 단위 연산이므로 대상 슬롯을 `${camId}:${presetId}` 키로 그룹핑한 뒤 그룹(프리셋)별로 한 번에 생성한다.

### 4.3 `src/calibrate/PlateDiscoveryJob.ts` — `saveSlotLpd`
같은 이유로 인라인 계산을 `camIdx:presetIdx` 그룹핑 + `buildOccupyRegionsBySlot` 호출로 교체했다. 더 이상 쓰지 않는 `buildPlateAnchoredQuad`/`quadBoundingRect` import를 제거했다(고아 import 정리).

### 4.4 `src/capture/floorRoi.ts` — 원복
직전에 추가했던 `buildOccupyRangeFromPlate`(판-only 폴백)를 제거했다. 차량 bbox를 받는 `buildPlateAnchoredQuad` 자체(Finalizer가 실제 VPD를 가졌을 때 쓰는 함수)는 시그니처·본문 전부 그대로 유지 — 이 파일은 결과적으로 diff가 없다(추가했던 걸 되돌려 HEAD 상태로 복귀).

## 5. 영향 없는 부분(확인됨)

- **`POST /capture/slots/occupy`의 요청/응답 계약**: 무변경(`{cam?, preset?} → {ok, updated, skipped, failed}`). 내부 생성식만 교체.
- **뷰어 라이브 오버레이**(`web/occupancyRegion.js`, `web/app.js`): 무변경 — 애초에 정본이던 그 알고리즘 그대로다.
- **Finalizer 차량 bbox 경로**(`buildPlateAnchoredQuad` 직접 호출): 무변경. VPD가 실제로 있는 경우엔 이번 교체와 무관하게 그대로 동작한다.
- **DB 스키마 / `@parkagent/types`**: 무변경. 기존 `occupy_range` 컬럼에 대한 값 갱신뿐.

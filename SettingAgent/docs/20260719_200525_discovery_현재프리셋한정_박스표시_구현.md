## 1. 배경

콤보박스 3모드(순수 LPD / 앞면중심 LOOP=discovery / VPD→LPD) 중 discovery만 두 지점에서 다른 모드와 어긋나 있었다.

1. **범위 불일치**: 다른 두 모드는 화면에 보이는 **현재 프리셋 1회**만 검사하는데, discovery는 `POST /discover/ptz` body `{}` 호출로 **카메라 전체·전 슬롯**을 순회했다. 사용자가 보는 화면과 실제 잡의 작업 범위가 어긋났다.
2. **박스 미표시**: 다른 두 모드는 검출 결과를 캔버스에 노란 박스(quad)로 그리는데, discovery는 진행률·"발견 N/전체" 텍스트만 갱신하고 검지된 번호판 박스를 화면에 그리지 않았다.

이번 변경은 이 두 가지만 고친다. 폴링·상호배타·`slot_setup.lpd` 부분 UPDATE 등 잡의 다른 동작은 그대로 둔다.

## 2. 변경 내용

### 2-1. 백엔드 — 현재 프리셋 한정 필터

**`src/api/discoverRoutes.ts:6-12`** — `StartBodySchema`에 `cam`/`preset` 옵셔널 필드 2개 추가(`z.number().int().positive().optional()`). `.default({})`는 유지해 빈 body 시 기존 동작(전체 배치)이 그대로 보존된다.

```ts
const StartBodySchema = z
  .object({
    slotIds: z.array(z.string()).optional(),
    cam: z.number().int().positive().optional(),
    preset: z.number().int().positive().optional(),
  })
  .default({});
```

라우트 핸들러(L32)는 `deps.discovery.start(parsed.data.slotIds)` 대신 `deps.discovery.start(parsed.data)`로 파싱된 객체 전체를 전달한다.

**`src/calibrate/PlateDiscoveryJob.ts:111-121`** — `start`의 시그니처를 `slotIds?: string[]`에서 필터 객체 `{ slotIds?, cam?, preset? }`로 바꾸고, `expandDiscoveryTargets` 직후·기존 `slotIds` 필터 **앞**에 cam/preset 필터를 삽입했다.

```ts
start(filter: { slotIds?: string[]; cam?: number; preset?: number } = {}): { total: number } {
  if (this.state === 'running') throw new Error('discover already running');
  let targets = expandDiscoveryTargets(this.store.getSlotSetup());
  if (filter.cam != null && filter.preset != null) {
    targets = targets.filter((t) => t.camIdx === filter.cam && t.presetIdx === filter.preset);
  }
  if (filter.slotIds && filter.slotIds.length > 0) {
    const set = new Set(filter.slotIds);
    targets = targets.filter((t) => set.has(t.slotId));
  }
  // ... state/total/run 발화 이하 불변
}
```

- 게이트는 `cam`과 `preset`이 **둘 다** 전달됐을 때만 발동한다(`!= null` — undefined/null 모두 커버). 하나만 주면 전체 배치가 보존된다.
- `expandDiscoveryTargets`(펼침 순수함수, `plateDiscoveryWriter.ts`)는 손대지 않았다. slotIds 필터와 cam/preset 필터는 함께 적용될 수 있으며(교집합), 순서는 cam/preset → slotIds.

### 2-2. 프론트 — 현재 프리셋 전달

`web/app.js`의 `discStart`(L2325~)가 body를 `{}`에서 현재 프리셋으로 바꿔 보낸다.

```js
const cam = state.capFrameKey2?.cam ?? state.cam;
const preset = state.capFrameKey2?.preset ?? state.preset;
body: JSON.stringify({ cam, preset })
```

이 해석은 `runLiveDetect`(app.js:928-929)의 "현재 프리셋" 정의를 그대로 미러한 것이다 — 정밀수집 폴링 중이면 표시 중인 캡처 프레임의 프리셋, 아니면 라이브 선택 프리셋. total=0 안내 문구도 "현재 프리셋에 앞면중심 보유 슬롯 없음"으로 원인을 정확히 표기하도록 수정했다.

### 2-3. 프론트 — 검지 LPD 박스 표시

세 지점을 가산했다.

1. **상태 필드**(app.js:100): `discoverByKey: {}` — `detectByKey`와 대칭인 신규 격리 필드. `cam:preset` 프리셋키 → 정규화 quad 배열.
2. **결과 파싱**(`renderDiscResult`, app.js:2381-2399): 상태 요약 뒤에 `GET /discover/result`를 조회해 `items[]` 중 `found && lpdOrig`인 항목만 `presetKey(camIdx, presetIdx)`별로 묶어 `state.discoverByKey`에 **대체 저장**(누적 아님 — 잡이 끝날 때마다 이전 프리셋 결과를 지우고 새 결과로 교체).
3. **렌더**(`drawDetectOverlay`, app.js:908-909): 기존 `if (showPlate)`(`#roi-plate` 토글) 블록 끝에 `state.discoverByKey?.[key]`(key = 기존 지역변수 `currentFrameKey()`)를 조회해 `drawPlateQuad(ctx, q, false)`로 그린다. 라이브/DB LPD와 **동일 헬퍼·동일 색(노랑 실선)**을 재사용했다.

렌더 트리거는 별도로 추가하지 않았다 — discPoll의 기존 완료 흐름(`await renderDiscResult()` → `startLive()` → 내부 `drawRoiOverlay()`)이 저장 직후 자연히 그려준다.

`lpdOrig`는 원본(프리셋) 프레임 정규화 4점 OBB로, 라이브 LPD가 쓰는 것과 **동일 좌표계**이므로 별도 변환 코드는 필요 없었다.

## 3. 입출력 계약

### `POST /discover/ptz`

| 필드 | 타입 | 필수 | 의미 |
|---|---|---|---|
| `slotIds` | `string[]` | 옵션 | 기존 — 지정 슬롯만 |
| `cam` | `number`(int, positive) | 옵션 | 신규 — `preset`과 동시 지정 시만 적용 |
| `preset` | `number`(int, positive) | 옵션 | 신규 — `cam`과 동시 지정 시만 적용 |

- 빈 body(`{}`)는 기존과 동일하게 전체 배치.
- `cam`/`preset` 중 하나만 주면 게이트 미충족으로 **전체 배치가 그대로 보존**된다(부분지정은 필터 미적용).
- `cam`/`preset`과 `slotIds`를 함께 주면 **교집합**(먼저 프리셋 한정 → 이후 slotIds 필터).
- 잘못된 타입(예: `cam: "x"`)은 기존과 동일하게 400.

### `GET /discover/result` → 프론트 소비

- 백엔드 item(`PlateDiscoveryItem`): `{ found, lpdOrig, camIdx, presetIdx, ... }` — 변경 없음.
- 프론트는 `found && lpdOrig` 항목만 소비해 `state.discoverByKey[presetKey(camIdx,presetIdx)]`에 저장. 저장키와 조회키(`currentFrameKey()`)는 동일 포맷(`presetKey`, `core.js:24`)이므로 프리셋 전환 시 자동으로 안 맞아 은닉된다.

### 4-2. 불변 확인(설계·구현·검증 공통 명시 + 직접 대조)

- **`expandDiscoveryTargets`**(`src/calibrate/plateDiscoveryWriter.ts`): 코드 직접 확인 — cam/preset 필터가 `PlateDiscoveryJob.start` 내부, 이 함수 **호출 이후**에 삽입되어 함수 자체는 미변경.
- **`PtzCalibrator.ts`**: `start(slotIds)` 시그니처·호출부는 discovery 필터와 무관, 이번 변경에서 미접촉(다른 모드·필터 불요 — 비대칭 허용은 설계 §3-1 명시).
- **`src/calibrate/types.ts` / `plateDiscovery.ts`**(`lpdOrig` 산출): 이미 선행 작업에서 산출되던 필드를 재사용만 함 — 신규 타입·로직 추가 없음.
- **`web/index.html` / `web/core.js` / `web/core.d.ts`**: 콤보박스·폴링·discoverView는 선행 완성분 — 이번 변경은 `app.js` 내부 로직만 건드림.
- **`slot_setup.lpd` 부분 UPDATE**(`upsertSlotLpd`/`saveSlotLpd`): 이번 변경은 필터·렌더에만 관여하고 저장 경로는 무접촉 — wipe 금지 정책(MEMORY: finalize slot_setup wipe fragility) 그대로 유지.

### 4-3. 파급 경로

- `PlateDiscoveryJob.start` 시그니처 변경(문자열배열 → 필터객체)의 호출부는 `discoverRoutes.ts` 1곳뿐(코드베이스 전체에서 grep 근거는 개발자 보고 기준, 문서화 단계에서 재확인은 안 함 — **확인 필요** 항목으로 남김. 다만 `PlateDiscoveryJob`을 import하는 다른 라우트/테스트가 `start()`를 무인자 또는 `{}` 형태로 호출한다면 신규 필터 필드가 옵셔널이라 회귀 없음).
- REST 계약 변경(`cam`/`preset` 필드 추가)은 **추가적(additive)**이라 기존 클라이언트(빈 body 호출자)에 영향 없음. 프론트(`web/app.js`)가 유일한 실제 호출자.
- `state.discoverByKey`는 `detectByKey`와 별도 격리 필드라 기존 라이브/DB LPD 렌더 경로에 간섭하지 않음.

### 4-4. 확인 필요(미검증)

- 박스가 실제 화면(canvas)에서 올바른 픽셀 위치에 그려지는지, `#roi-plate` 해제 시 실제로 사라지는지는 DOM 렌더링이 필요해 vitest 범위 밖 — **리더 라이브 관찰 필요**(qa-report §5 명시, 이번 세션에서 관찰 미수행).
- discovery의 실제 번호판 검지율은 실 LPD 서비스(da_lpd_api)·실 카메라가 필요해 이번 범위 밖(유닛은 `foundItem` 시임으로 대체).
- QA가 지적한 "정밀수집 순환 중(`capFrameTimer` 활성) 저장키 vs 조회키 비대칭" — discStart는 `capFrameKey2?.cam ?? state.cam` 무조건 사용, `currentFrameKey`는 `capFrameTimer && capFrameKey2` 조건부라 정밀수집 순환 뷰 상태에서 키 불일치 가능성. 단 이는 기존 `detectByKey`·`runLiveDetect`가 이미 갖고 있던 키 시맨틱을 그대로 상속한 것으로, 이번 변경이 만든 신규 결함은 아니다(QA 결론). 순환 뷰 상태에서의 표시 일치는 **리더 라이브 sanity 확인 권고** 항목으로 남는다.

## 6. 남은 가정/미해결 (설계서 §6 인용, 문서화 단계 재확인 없음)

- 현재 프리셋 정의 = `capFrameKey2?.cam/preset ?? state.cam/preset`(runLiveDetect 미러).
- 박스는 완료 후 일괄 표시(진행 중 실시간 표시 아님).
- discovery 박스는 라이브/DB LPD와 동색(노랑) — 시각 구분 없음.
- `discoverByKey`는 현재 프리셋 한정이라 실무상 단일 키이며, 매 완료 시 **대체**(누적 아님).

이 네 가지는 리더가 다른 정책을 원하면 변경 필요 항목으로 유지된다.

## 1. 증상 (사용자 인지)

정밀수집(CaptureJob)을 시작하면:

1. Unity 카메라가 프리셋 사이를 **무한 반복**하는 것처럼 보인다.
2. **정지 버튼이 안 먹힌다** — 눌러도 즉시 멈추지 않는다.
3. 잡(job)이 `state: 'stopping'` 상태에 **갇힌다**(stopped 로 전이하지 못함).

실제로는 "무한 반복"이 아니라 각 라운드가 수 분씩 걸려 진행이 사실상 정체된 것이며, 정지 신호는 접수되었으나 블로킹된 라운드/체크포인트가 끝날 때까지 반영되지 못한 상태였다.

## 2. 근본 원인 (로그로 확정 — 3가지)

이 버그는 단일 원인이 아니라 세 가지가 겹쳐 발생했다.

### 원인 ① LLM 클라이언트 타임아웃 없음
`AgentRuntime` 생성자가 `new OpenAI({ baseURL, apiKey })` 로 클라이언트를 만들면서 **타임아웃을 지정하지 않았다**. OpenAI SDK 기본 타임아웃은 **600초(10분)**, 기본 재시도(maxRetries)는 **2회**다. 즉 느리거나 무응답인 LLM 호출 1건이 최악 `600s × 3 = 30분`까지 매달릴 수 있었다.

### 원인 ② 모델이 비전 불가 (qwen3.6:27b)
당시 `llm.config.json` 모델은 **qwen3.6:27b** — 텍스트 모델이라 멀티모달(이미지) 입력을 처리하지 못한다. floor ROI 는 이미지 기반 4점 사변형 추론이므로:

- **로그 증거**: 한 체크포인트에서 floor ROI **12건 시도(attempted=12) / 0건 성공(succeeded=0)**. 즉 모든 비전 호출이 유효 quad 를 반환하지 못하고 폴백(결정형 bbox 유도 사변형)으로 떨어졌다.
- floor ROI 경고 로그(`floor ROI: LLM 비활성/불가 — 결정형 폴백 사용`)가 반복 출력.
- 비전 호출이 느리게 응답/무응답 → 원인 ①과 결합해 **라운드당 체크포인트가 약 3.7분(≈220초) 소요**. 로그의 `req_move` 타임스탬프와 floor ROI 경고 타임스탬프 사이 **약 220초 gap**이 이를 뒷받침한다.

### 원인 ③ stop()이 협조적(cooperative)이라 블로킹 중엔 무효
`CaptureJob.stop()` 은 상태를 `'stopping'` 으로 바꾸고 라운드 루프가 **스스로 확인해 탈출**하기를 기다리는 협조적 방식이다. 그런데 수정 전에는 라운드 루프가:

- 타깃 순회 중간에 stopping 을 확인하지 않았고,
- 체크포인트 진입 전에도 확인하지 않았다.

따라서 라운드/체크포인트가 원인 ①②로 수 분간 블로킹되면, 그 시간 동안 정지 신호는 **완전히 무시**되었다. 사용자에게는 "정지 무한 대기 / state:'stopping' 에 갇힘"으로 보였다.

> **결합 효과**: ①(600s 타임아웃)이 개별 호출을 길게 늘이고 → ②(비전 불가)가 매 체크포인트마다 12건의 무의미한 느린 호출을 유발하고 → ③(비확인 stop)이 그 블로킹 동안 정지를 원천 차단. 세 원인이 곱해져 "무한·정지불가"로 인지됨.

## 3. 진단 방법 (재현 시 참고)

로그 타임스탬프 대조로 근본 원인을 특정했다:

1. **`req_move` 로그 타임스탬프**: 프리셋 이동(카메라 물리 이동) 시각. 라운드/타깃 경계의 기준점.
2. **floor ROI 경고 로그 타임스탬프**: `{ llmUsable, attempted, succeeded }` 필드를 포함한 경고. `attempted=12, succeeded=0` 이 비전 불가의 결정적 증거.
3. 두 로그 사이 **약 220초(≈3.7분) gap** → 체크포인트가 라운드를 수 분 블로킹하는 것을 확인.
4. 정지 클릭 후에도 `state` 가 `'stopping'` 에 머무름 → stop 이 협조적인데 확인점이 없어 블로킹 중 반영 불가임을 확인.

## 4. 수정 3축 (상세)

상태머신·수집·검출·집계·폴백 시맨틱은 **무변경**. 신규 파라미터는 전부 옵셔널(하위호환).

### 축 1 — LLM 요청 타임아웃 + 재시도 축소

**`src/config/llmConfig.ts`**
- `LlmSchema` 에 필드 추가: `timeoutMs: z.number().int().positive().optional()`.
- `DEFAULT_LLM_CONFIG.llm` 에 `timeoutMs: 30000` 명시(기본 30초).
- 옵셔널 + DEFAULT 병합이라, config 에 키가 없어도 `loadLlmConfig` 의 `{ ...DEFAULT_LLM_CONFIG.llm, ...raw.llm }` 병합으로 항상 30000 이 보강된다.

**`src/brain/AgentRuntime.ts:43`** (생성자, 확인점 1곳)
```ts
// 변경 전
this.client = new OpenAI({ baseURL: cfg.llm.baseUrl, apiKey });
// 변경 후
this.client = new OpenAI({ baseURL: cfg.llm.baseUrl, apiKey, timeout: cfg.llm.timeoutMs ?? 30000, maxRetries: 0 });
```
- **생성자 1곳**에 적용하여 클라이언트 인스턴스 전역에 반영 → per-call 옵션 누락 위험 없음.
- `maxRetries: 0`: 재시도는 대기를 배수로 키운다(`timeout × (1+retries)`). 수집 체크포인트는 실패 시 항상 결정형 폴백이 있으므로 재시도 불필요. 최악 대기 = timeout 1회.
- **30000ms 근거**: 정상 floor ROI 비전 1건 ≈18초 → 약 1.6배 여유를 두면서 SDK 기본 600초 대비 20배 단축. 30초를 넘으면 정체로 간주해 폴백함이 합당.
- `chatJson` 내부의 애플리케이션 레벨 파싱 재시도(attempt<2)는 **유지**(네트워크 재시도와 목적이 다른, JSON 파싱 실패 회수용). 단 각 시도가 이제 timeout 으로 상한되므로 최악 = 2×30s = 60s.

### 축 2 — CaptureJob.runRound stop 확인점 2곳 (핵심)

**`src/capture/CaptureJob.ts` `runRound()`**

- **(a) 타깃 사이 확인** (179~181행): 타깃 for 루프 상단에서
  ```ts
  for (const t of this.params.targets) {
    if (this.currentState() === 'stopping') break; // 다음 타깃 캡처 전 탈출
  ```
  진행 중 stop 시 **다음 타깃 캡처 전** 루프를 빠져나온다. 이미 캡처된 타깃 적재는 유지 → 데이터 손실 없음.

- **(b) 체크포인트 진입 전 확인** (194행): 체크포인트 게이트 조건에 `&& this.currentState() !== 'stopping'` 추가
  ```ts
  if (this.done % this.params.checkpointEvery === 0 && this.currentState() !== 'stopping') {
    await this.checkpoint(roundIdx);
  }
  ```
  정지 중이면 수 분짜리 체크포인트 진입 자체를 스킵.

- **부분 라운드 계수**: 타깃 루프를 중도 탈출해도 `this.round = roundIdx; this.done = roundIdx; updateRunProgress(...)` 는 그대로 실행한다(부분 라운드도 1 라운드로 계수). 진행 표시 단조성 유지 + done 역행/off-by-one 회피 목적. 이후 기존 종료 로직(206행 `currentState()==='stopping'` → `finishRun('stopped','manual')`)이 그대로 stopped 로 전이시킨다.
- **`finishRun`/`stop`/상태머신 시그니처는 무변경.**

### 축 3 — FloorRoiReviewer.review 조기 탈출(shouldStop)

**`src/capture/CaptureJob.ts` `checkpoint()`**: floorReviewer 호출에 콜백 전달
```ts
this.deps.floorReviewer.review(this.runId, slots, this.lastFrameByPreset,
  () => this.currentState() === 'stopping');
```

**`src/capture/FloorRoiReviewer.ts` `review()`**:
- 시그니처에 옵셔널 4번째 파라미터 `shouldStop?: () => boolean` 추가.
- 후보 슬롯 for 루프 각 반복 시작(51행)에서 `if (shouldStop?.()) break;` (기존 `used >= maxPerCheckpoint` 가드 직전).
- **이미 시작된 체크포인트**도 다음 슬롯 전(≤1 LLM 호출) 조기 탈출. AbortController/in-flight 취소는 미사용 — 슬롯 간 boolean 체크만.

축 2(b)가 "새 체크포인트 진입 차단"을 담당하고, 축 3이 "이미 시작된 체크포인트의 슬롯 루프 조기 탈출"을 담당해 잔여 대기(최악 12슬롯)를 없앤다.

> **AbortController 미채택 근거**: Abort 전파는 `stop → checkpoint → reviewer.review → recognizeFloorRoi → chatJson → chat → openai.create({signal})` 로 6~7개 시그니처에 `signal` 추가가 필요(광범위·회귀 위험). 축 1(30s 상한) + 축 2·3(경계 확인)만으로 무한 대기가 제거되어, 추가 이득(≤30초)이 비용을 정당화하지 못함.

## 5. config 모델 교체 이유 (비전 필수)

**`config/llm.config.json`**: `model` 을 `qwen3.6:27b` → **`gemma4:12b`** 로 교체(+ `timeoutMs: 30000` 명시).

- floor ROI, judgePreset(stage1), adviseCentering 등은 **이미지 입력이 필수인 비전 작업**이다. 텍스트 전용 모델(qwen3.6:27b)로는 원인 ②처럼 attempted=12/succeeded=0 이 발생한다.
- gemma4:12b 는 RTX-3090 서버 Ollama(OpenAI 호환 `/v1`)의 **멀티모달(비전)** 모델. floor ROI quad 를 실제로 생성할 수 있다.
- config 파일 `_comment` 에 "floor ROI 등 비전 작업에 멀티모달 필수. 1회성 셋업에서 enabled=true" 로 의도 명시.

> **주의**: 코드는 model 문자열에 무관(model-agnostic, OpenAI 호환 호출). 모델 교체는 순수 config 변경이며 **서버 재시작으로만 반영**된다(§8 운영 주의).

## 6. 동작 흐름 (수정 후)

정지(stop) 클릭 시 반영 지연 상한:

| 시점 | 반영 경계 | 최악 지연 |
|------|-----------|-----------|
| 타깃 순회 중 stop | 다음 타깃 캡처 전(루프 상단 확인) | 진행 중인 1캡처+검출 ≈ 수 초 |
| 체크포인트 직전 stop | 체크포인트 진입 스킵 | 즉시(다음 라운드 미예약) |
| 체크포인트 진행 중 stop | 다음 슬롯 전(shouldStop) | ≤1 LLM 호출(≤30s 타임아웃 상한) |
| 개별 LLM 호출 매달림 | timeout 발동 | ≤30초(maxRetries 0) |

즉 정지는 "다음 타깃/슬롯 경계(수 초 ~ ≤1 LLM 호출)" 내에 반영되고, 어떤 경우에도 무한 대기가 없다.

## 8. 수동 동작확인 체크리스트 (실서버 — 유닛 범위 밖, 미수행)

아래는 외부 서비스(카메라 시뮬레이터·gemma 비전 LLM)가 필요해 유닛 모킹으로 대체 불가. **실 환경에서 담당자 확인 요망** (03 QA §4 그대로 반영):

- [ ] **재시작 후 정지 즉시 반응**: 서버 재배포+재시작 후 정밀수집 중 정지 클릭 시 "다음 타깃 경계(수 초)" 내 stopped 전이 확인.
- [ ] **gemma4:12b floor ROI 성공**: `floorRoi.enabled=true`·gemma 엔드포인트 실동작으로 체크포인트에서 floor ROI 가 폴백이 아닌 **LLM quad** 로 생성되는지(`llmFloorUnavailable` 미표식, 로그 `succeeded > 0`) 확인.
- [ ] **타임아웃 실동작 스모크**: 느린/무응답 LLM 엔드포인트에서 단건 호출이 **30s 내 폴백 전환**되는지(수 분 매달림 없음). 유닛은 생성자 인자만 검증, 실제 abort 타이밍은 SDK 소관.

## 9. 운영 주의사항

- **config 변경은 서버 재시작 필요**: `llm.config.json` 은 프로세스 기동 시 1회 로드된다. 모델(gemma4:12b)·timeoutMs 변경은 **프로세스 재시작으로만 반영**된다.
- **갇힌 잡은 재시작으로 해소**: 현재 `state:'stopping'` 에 갇힌 잡은 인메모리 단일 잡이라 코드 미반영 상태다. **재배포 + 프로세스 재시작(런타임 리셋)** 으로만 해소된다.
- **롤백**: 문제가 생기면 `llm.config.json` 의 model/timeoutMs 만 되돌려도 되나(순수 config), 코드 축 2·3(stop 확인점)은 회귀 없이 안전하므로 유지 권장. 상세는 `_workspace/04_doc_impact.md` §롤백 참조.

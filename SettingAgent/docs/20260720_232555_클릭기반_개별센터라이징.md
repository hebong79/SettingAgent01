## 1. 기능 개요

정밀수집 페이지의 센터라이징 UI에 **개별(클릭) 센터라이징**을 추가했다. 기존 `센터라이징` 버튼(전체 슬롯 순회, `/calibrate/ptz`)은 그대로 두고, 그 옆에 콤보박스 `#cal-click-mode`를 신설한다.

| 옵션 값 | 의미 |
|---|---|
| `off` (기본) | 개별 센터라이징 비활성. 오버레이 클릭은 기존 편집(검출/슬롯 선택) 그대로 동작 |
| `center` | 라이브뷰 클릭 → 클릭 지점 최근접 번호판으로 pan/tilt만 정렬 |
| `center-zoom` | 위 정렬 + 목표 번호판 폭으로 줌 마감(1회) |

**사용법**: 콤보에서 `개별 center` 또는 `개별 center+zoom`을 선택한 뒤 라이브뷰(오버레이) 위에서 차량을 클릭하면, 그 클릭 위치에서 가장 가까운 번호판을 향해 카메라가 pan/tilt(옵션 줌)로 정렬된다. 진행 중에는 센터라이징 진행 프레임이 실시간으로 표시되고, 완료 후 라이브 스트림으로 자동 복귀한다. **어떤 경우에도 결과를 저장하지 않는다** — `slot_ptz.json`, DB `centering_slot`/`slot_setup`, Setup 스냅샷 중 어디에도 기록되지 않는다. 카메라는 물리적으로 그 지점을 향한 채 남는데(정상 동작), "저장 안 함"은 영속화 파일/DB에 기록하지 않는다는 의미로 한정된다.

Ctrl 키를 누른 클릭은 기존 편집 제스처(정점/슬롯 조작)이므로 게이트에서 제외되어 개별 센터라이징으로 소비되지 않는다.

## 2. 신규 백엔드: `PtzCalibrator.centerOnPoint()`

파일: `src/calibrate/PtzCalibrator.ts` (140~187행 부근)

```ts
async centerOnPoint(
  camIdx: number,
  presetIdx: number,
  point: NormalizedPoint,
  opts?: { zoom?: boolean },
): Promise<{ ok: boolean; ptz: Ptz; plateWidth: number | null; reason?: string }>
```

### 2.1 동작 흐름

1. **상호배타 가드**: `state === 'running'`(배치 잡 진행 중)이면 `throw new Error('calibrate already running')`. 신규 필드 `pointBusy`(96~118행 부근, private)가 `true`면 `throw new Error('point centering busy')`. 두 예외 모두 라우트에서 409로 매핑된다.
2. `pointBusy = true` 로 잠그고 `try/finally`로 해제 — 정상 종료·예외 종료 모두 락이 풀린다.
3. `startPtz = await this.startPtzFor({ camIdx, presetIdx } as PlateTarget)` — 기존 배치 경로가 쓰는 `ptzByKey` 캐시를 그대로 재사용해 프리셋 base PTZ를 얻는다. `startPtzFor`는 `camIdx`/`presetIdx`만 참조하므로 최소 객체 캐스팅이 안전하다(구현 노트에서 확인).
4. `prior: NormalizedRect = { x: point.x, y: point.y, w: 0, h: 0 }` — 클릭 지점을 폭·높이 0인 prior ROI로 만든다.
5. `centered = await this.makePlatePtz({ ...this.baseOpts(), plateRoi: prior }).centerOnPlate(camIdx, presetIdx, startPtz)`. **`peerOffsets`를 주입하지 않는다** — 이는 기존 `platePtz.ts`의 소유권(peer exclusivity) 게이트를 우회하는 것이 아니라, 원래부터 `centerOnPlate`가 `peerOffsets` 미주입 시 "클릭 최근접 번호판"(`pickNearestPlate`)을 선택하도록 설계되어 있는 결정형 도구의 기존 동작을 그대로 쓰는 것이다.
6. `centered.ok && opts?.zoom !== false` 이면 `zoomToPlateWidth`를 1회 체이닝한다. 이때 두 번째 `plateRoi`는 `quadBoundingRect(centered.plate!.quad)`(센터링 직후 판의 사각형), `gain`은 `centered.gain`을 넘겨 폐루프 게인을 이어받는다. `z.ok`이면 `z`의 결과(ptz/plateWidth/reason)를 반환한다.
7. **줌 실패 시맨틱(중요)**: 6번에서 `z.ok === false`면 낙하하여 **`centered`의 결과를 반환**한다(`ok: centered.ok`는 `true`). 즉 "센터링은 성공했지만 줌은 실패"한 경우 응답의 `ok`는 `true`이고 `reason`은 줌 실패 이유가 아니라 비어 있다(centered에 reason이 없으므로). 이는 설계서 §3.1 step6에 명문화된 정본 동작이며, qa 리포트가 이 시맨틱을 명시적으로 테스트하고 확인했다(§5-A-4).
8. **저장 미호출**: 메서드 본문 어디에도 `this.writer(...)`, `saveCenteringSlots`/`store.upsertSlotCentering`, `saveSetupSnapshot`/`saveStore.saveSnapshot` 호출이 없다(코드 상 부재를 직접 확인). `PlatePtz`(`platePtz.ts`) 자체도 무상태 결정형 도구로 저장 로직을 갖지 않는다. `onFrame` 콜백은 `makePlatePtz` 생성자에 이미 배선되어 있어 `lastFrame`만 갱신하며, 이는 `/calibrate/frame` 폴링이 진행 중 프레임을 보여주기 위한 기존 메커니즘이지 저장이 아니다.

### 2.2 배치(전체순회) 센터라이징과의 차이

| 항목 | 배치 센터라이징 (`start()`→`run()`) | 개별(클릭) 센터라이징 (`centerOnPoint()`) |
|---|---|---|
| 트리거 | `센터라이징` 버튼 → 전체(또는 필터) 슬롯 순회 | 콤보 선택 상태에서 오버레이 클릭 1회 |
| 대상 선정 | `slot_setup` ROI 전개 + acquire 줌인 사다리(먼 작은 판 재포착) | 클릭 지점 최근접 번호판(`pickNearestPlate`, 조작자가 직접 가리킴) |
| 소유권 게이트 | 있음(peerOffsets로 인접 슬롯과 배타) | 없음(단일 클릭 대상이므로 불필요) |
| 줌 마감 | `zoomToPlateWidth`로 `targetPlateWidth` 마감 + 실패 시 줌아웃 복구 사다리 | `zoomToPlateWidth` 1회, 실패 시 복구 사다리 없이 center 결과로 낙하 |
| **저장** | **함(`slot_ptz.json` writer, DB `centering_slot`, Setup 스냅샷)** | **안 함(전부 미호출)** |
| 카메라 물리 위치 | 이동 후 유지 | 이동 후 유지(정상, 저장과는 무관) |
| 상호배타 | `state==='running'`으로 개별 진입 차단 | `pointBusy`로 재진입 차단, 배치 진행 중이면 진입 차단 |

## 3. 신규 라우트: `POST /calibrate/point`

파일: `src/api/calibrateRoutes.ts`

### 3.1 요청 스키마 (`PointBodySchema`, zod)

```ts
{
  cam: number (정수, ≥0),      // 1-based 카메라 인덱스
  preset: number (정수, ≥0),  // 1-based 프리셋 인덱스
  point: { x: number, y: number }, // 정규화 좌표(0~1)
  zoom?: boolean,              // 생략 시 undefined → centerOnPoint 에서 zoom!==false 로 처리(=true와 동치)
}
```

### 3.2 응답

- **200**: `{ ok: boolean, ptz: Ptz, plateWidth: number | null, reason?: string }` — `centerOnPoint`의 반환값을 그대로 매핑(reason은 값이 있을 때만 스프레드).
- **400**: 스키마 검증 실패 시 `{ error: 'invalid body', detail: <zod flatten> }`. `centerOnPoint`가 던진 예외 메시지가 `'running'`/`'busy'`를 포함하지 않는 경우도 400 `{ error: msg }`.
- **409**: `centerOnPoint`가 던진 예외 메시지에 `'running'` 또는 `'busy'`가 포함될 때(= 배치 진행 중 또는 개별 진행 중 재진입).

기존 4개 라우트(`/calibrate/ptz`, `/calibrate/status`, `/calibrate/frame`, `/calibrate/result`)는 코드상 변경되지 않았다.

## 4. 프론트 배선

### 4.1 `web/index.html`

`#cal-start` 버튼 뒤, `.centering-progress` 앞에 콤보박스 삽입(203~214행):

```html
<label class="field" title="선택 후 라이브뷰에서 차량을 클릭하면 그 번호판으로 센터라이징(저장 안 함)">개별 센터라이징
  <select id="cal-click-mode">
    <option value="off" selected>사용 안 함</option>
    <option value="center">개별 center</option>
    <option value="center-zoom">개별 center+zoom</option>
  </select>
</label>
```

설계서는 체크박스를 제안했으나 실제 구현은 3옵션 `<select>`로 구현되었다(off/center/center-zoom) — 요청에 명시된 것과 일치하는 형태이며 기능적으로 동등(off=미체크, center=체크+zoom:false, center-zoom=체크+zoom:true)하다.

### 4.2 `web/app.js` — 오버레이 클릭 게이트 (최우선)

`wireOverlayEditing()` 내 `overlay.addEventListener('mousedown', ...)` 핸들러 맨 앞(3270~3279행):

```js
const clickMode = $('cal-click-mode')?.value;
if (clickMode && clickMode !== 'off' && !e.ctrlKey) {
  const { nx, ny } = eventToNorm(e);
  e.preventDefault();
  void calPointCenter(nx, ny, clickMode === 'center-zoom');
  return;
}
```

`off`(기본값)이면 이 분기 자체를 건너뛰어 이후의 검출/슬롯 편집 분기로 그대로 진입한다 — 기존 클릭 동작이 물리적으로 100% 보존된다. `center`/`center-zoom` 선택 시에는 `return`으로 뒤의 편집 분기 진입을 차단해 클릭 소비가 배타적이다.

### 4.3 `calPointCenter(nx, ny, zoom)` (2344~2379행)

- `calPointBusy` 모듈 전역 락으로 중복 클릭(연타)에 의한 동시 발화를 차단(락 걸린 동안은 `return`).
- `cam = state.capFrameKey2?.cam ?? state.cam`, `preset = state.capFrameKey2?.preset ?? state.preset` — 기존 `runLiveDetect`가 쓰는 것과 동일한 "현재 라이브뷰 프레임 키" 규약을 재사용.
- `$('cal-msg')`에 진행 메시지 표시, `startCalFramePolling()`(기존 배치 폴링 함수 재사용)으로 진행 프레임을 실시간으로 보여준다.
- `POST /calibrate/point` 호출 → 완료 후 `stopCalFramePolling()` → 결과에 따라 메시지 분기(409: "전체 진행 중" 안내 / `ok`: "완료" / 그 외: `종료(reason|error|status)`) → `startLive()`로 라이브 스트림 복귀.

### 4.4 `web/app.css`

```css
#overlay.click-centering {

}
```

콤보 `change` 리스너(3427~3429행)가 `overlay.classList.toggle('click-centering', value !== 'off')`로 클래스를 토글해, 개별 센터라이징 모드일 때 오버레이 커서가 조준 십자선으로 바뀐다(시각적 피드백, 기능에는 영향 없음).

### 한계/미검증 (qa 리포트에서 승계, 은닉 없이 명시)

1. **프론트 DOM E2E 미검**: `overlay.mousedown` 게이트 → `calPointCenter` → `fetch`로 이어지는 실제 브라우저 클릭 동작은 vitest 대상이 아니다. body shape·zoom 값 매핑은 소스 코드 대조로만 확인했으며, 실제 브라우저 클릭 → 라우트 발화의 E2E는 라이브 스모크가 필요하다.
2. **줌 실패 reason 미전파**: "센터링 성공 + 줌 실패" 상황에서 응답은 `ok: true`이고 줌 실패 이유(`reason`)가 전파되지 않는다. 조작자는 `plateWidth` 값을 보고 간접적으로만 줌 미달을 판별할 수 있다. 설계 의도(정직=ok는 센터링 성공 여부만 반영)와 일치하는 정본 동작이나, UX상 줌 미달 신호가 필요하면 후속 논의 대상이다(결함 아님).
3. `centerOnPlate`/`zoomToPlateWidth` 내부 폐루프 자체는 `centerOnPoint` 테스트에서 팩토리 시임으로 목킹되어, 오케스트레이션(가드·prior·체이닝·저장 부재)만 검증되었다. 폐루프 로직 자체는 기존 `platePtz.test.ts`/`ptzCalibrator.test.ts`가 별도로 커버한다.

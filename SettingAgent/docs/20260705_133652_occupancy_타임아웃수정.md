# occupancy 판정 타임아웃 수정 — per-request 120초 격리 + 관측성 로깅

작성: 문서화·영향도 분석가(documenter) / 대상: SettingAgent
근거: `_workspace/01_architect_plan.md`(설계) · `02_developer_changes.md`(구현) · `03_qa_report.md`(검증) + 실제 변경 소스 대조
범위: **A(occupancy 전용 per-request 타임아웃) + C(관측성 로깅)**. B(비용 절감: box 제거·jsonMode off·imageMaxEdge 하향)는 이번 범위 제외(설계 후속).

## 1. 배경 — 무엇이 문제였나

라이브 vLLM 스모크 로그에서 occupancy 판정이 **전량 실패**했다.

```
occupancy: usable:true, attempted:5, succeeded:0
```

- 배선(브레인 주입·`enabled`·메서드 보유)은 정상 — `OccupancyReviewer.review()`가 5개 프리셋을 정상 순회·시도(`attempted:5`)했다.
- 그러나 `judgeOccupancy`가 전부 `null`/미완료로 저장 0건(`succeeded:0`).
- 실측 대비:
  - 8×8 **더미 이미지** 응답: 약 3초
  - 실제 캡처 프레임(1920×1080, 398KB): **60초+**
- 당시 LLM 클라이언트 타임아웃은 **30초 고정**(`config/llm.config.json` → `llm.timeoutMs:30000`).

즉 실프레임 판정이 30초 안에 끝나지 못해 전부 컷 되었다. 이 문서는 그 수정 내용을 기록한다.

## 2. 근본 원인

호출 경로: `OccupancyReviewer.review` → `brain.judgeOccupancy` → `chatJson` → `chat` → `chatOpenai`(`this.client.chat.completions.create`).

- 고해상도 프레임 + guided JSON 디코딩(vLLM `response_format:{type:'json_object'}`, "전 주차면 나열")의 왕복 시간이 클라이언트 전역 타임아웃 30초를 초과 → SDK가 `APIConnectionTimeoutError`(AbortError) throw → `chatJson`의 `await this.chat(...)`(try 밖)에서 예외 전파 → `judgeOccupancy` throw → `OccupancyReviewer` catch(`'occupancy 판정 실패(스킵)'`)로 흡수. 5개 전부 타임아웃 → `succeeded:0`.
- 클라이언트는 `maxRetries:0`이라 재시도 배수 지연은 없음(순수 30초 1회 컷).

**핵심 제약**: OpenAI 클라이언트의 `timeout`은 **생성자 단일값이 전역 공유**된다(`new OpenAI({ ..., timeout: cfg.llm.timeoutMs ?? 30000 })`, `AgentRuntime.ts:49`). 이 값을 올리면 occupancy뿐 아니라 floor ROI·centering·stage 등 **모든 LLM 호출이 함께 느려진다**(회귀 위험). 그래서 전역값 상향이 아니라 **per-request 타임아웃 override**로 occupancy만 국소 상향하는 것이 정답이다.

## 3. 수정 상세

### 3-1. `src/config/llmConfig.ts` — `OccupancySchema.timeoutMs`

`OccupancySchema`에 occupancy 전용 타임아웃 필드 추가(`llmConfig.ts:71-77`):

```ts
const OccupancySchema = z.object({
  enabled: z.boolean(),
  prompt: z.string().min(1),
  /** occupancy 판정 전용 요청 타임아웃(ms). 고해상도 프레임+guided JSON 디코딩이 llm.timeoutMs(30s)를 초과하는 문제 대응. */
  timeoutMs: z.number().int().positive().default(120000),
});
```

- `DEFAULT_LLM_CONFIG.occupancy`에도 `timeoutMs:120000` 명시.
- 병합 로직(`occupancy:{ ...DEFAULT.occupancy, ...raw.occupancy }`)은 무변경 — `.default(120000)`이라 json이 생략해도 채워짐(하위호환).
- `llm.timeoutMs`(전역)는 **무변경**(여전히 optional, 미지정 시 30000).

### 3-2. `src/brain/AgentRuntime.ts` — per-call 타임아웃 배선 (핵심)

OpenAI SDK v4는 `client.chat.completions.create(body, { timeout: ms })` **2번째 인자로 요청 단위 타임아웃 override**를 지원(전역 클라이언트 timeout보다 우선). 이를 내부 체인에 옵셔널 후행 파라미터로 흘려보낸다. 4개 private 메서드 전부 **순수 additive**(미전달 시 기존 동작).

- `chatJson(..., timeoutMs?)` (`AgentRuntime.ts:280,284`): `await this.chat(system, user, imageBase64, jsonMode, timeoutMs)` — 2회 재시도 루프 내부 동일 전달.
- `chat(..., timeoutMs?)` (`:299,310-311`): 분기 시 `chatNative(..., timeoutMs)` / `chatOpenai(..., timeoutMs)`로 그대로 전달.
- `chatNative(..., timeoutMs?)` (ollama 대칭, `:315,331`): `fetchWithTimeout(endpoint, {...}, timeoutMs ?? this.cfg.llm.timeoutMs ?? 30000)`.
- `chatOpenai(..., timeoutMs?)` (vLLM — 현재 실효 경로, `:341,360`):
  ```ts
  await this.client.chat.completions.create(
    { ...body },
    timeoutMs !== undefined ? { timeout: timeoutMs } : undefined,
  );
  ```
  `undefined` 전달 시 2번째 인자 자체가 `undefined` → SDK가 생성자 timeout(30초) 사용.
- `judgeOccupancy(input)` (`:219-226`): **occupancy만** 전용 타임아웃 전달:
  ```ts
  const parsed = await this.chatJson(
    system, user, (j) => OccupancyResultSchema.parse(j),
    input.imageBase64, true,
    this.cfg.occupancy?.timeoutMs,   // ← occupancy 전용(120s)
  );
  ```
- 전역 클라이언트 생성자 `new OpenAI({ ..., timeout: cfg.llm.timeoutMs ?? 30000 })`(`:49`)는 **무수정**.

### 3-3. `config/llm.config.json`

  "timeoutMs": 120000

### 3-4. `src/capture/OccupancyReviewer.ts` — 관측성 강화

120초로도 실패할 경우 원인(타임아웃 vs 파싱/스키마 실패)을 즉시 판별하기 위한 로깅. **제어 흐름·저장 경로·`llmUnavailable` 판정은 불변**, 로깅 필드만 가산.

- 루프 밖 카운터: `let nullCount = 0; let errorCount = 0;` (`OccupancyReviewer.ts:43-44`)
- null 반환을 **조용한 continue → 명시 warn**으로 승격(`:58-59`):
  ```ts
  nullCount += 1;
  logger.warn({ key }, 'occupancy 판정 null(파싱/스키마 실패 추정 — 스킵)');
  ```
- catch에 에러 `name` 포함(타임아웃 `APIConnectionTimeoutError`/`AbortError` 구분, `:74-77`):
  ```ts
  errorCount += 1;
  logger.warn(
    { err: e instanceof Error ? e.message : String(e), name: e instanceof Error ? e.name : undefined, key },
    'occupancy 판정 실패(스킵)',
  );
  ```
- 요약 warn에 `nullCount`/`errorCount` 분리 집계(`:84`).

### 3-5. `web/core.d.ts` 동기화 (부수 — occupancy 트랙 회귀 아님)

- 원인: 뷰어 트랙에서 occupancy 표시 함수 4개(`formatRatePct`/`occupancyByKey`/`occupancyRows`/`occupancyAverage`)가 `web/core.js`에 추가될 때, 수기 유지 선언 파일 `web/core.d.ts`가 **동기화되지 않아** tsc가 "no exported member"로 4건 보고. tsconfig `allowJs:false`라 tsc는 `import '../web/core.js'`를 `core.d.ts`로 해석하기 때문.
- 런타임은 **원래 정상**이었음 — `test/occupancy.test.ts` 27케이스는 esbuild로 실제 `core.js`를 로드해 계속 통과. tsc 선언 동기화 갭일 뿐, occupancy 타임아웃 트랙(AgentRuntime/llmConfig/llm.config.json/OccupancyReviewer)과 무관.
- 조치: QA가 `core.d.ts`에 4개 선언 + 타입(`OccupancyEntry`/`OccupancyTableRow` 등)을 실제 런타임 export와 1:1 동기화(`core.d.ts:137-140`). `@ts-ignore`·테스트 완화 없이 선언을 정직하게 맞춤 → tsc 0.

## 4. 격리 보장 — occupancy 120초 / 나머지 30초

occupancy만 120초를 쓰고 나머지 8개 호출부는 30초를 유지함을 코드로 보장하는 두 지점:

```
[occupancy 경로 — 120s]
judgeOccupancy(input)                         ← cfg.occupancy.timeoutMs(120000) 주입 (주입점, 단 한 곳)
  └ chatJson(..., timeoutMs=120000)
      └ chat(..., timeoutMs=120000)
          └ chatOpenai(..., timeoutMs=120000) → create(body, { timeout: 120000 })   (격리점)

[나머지 경로 — 30s 유지]
recognizeFloorRoi / judgePreset / dedupeAndLabel / finalReport /
reviewCheckpoint / finalizeCapture / adviseCentering / reviewSetup
  └ chatJson/chat(..., timeoutMs=undefined)
      └ chatOpenai(..., undefined)            → create(body, undefined) = 생성자 timeout(30s)
```

- **주입점**: `AgentRuntime.judgeOccupancy`의 `chatJson(..., this.cfg.occupancy?.timeoutMs)` 단 한 곳.
- **격리점**: `chatOpenai`의 `timeoutMs !== undefined ? { timeout } : undefined` — 미전달(다른 8개 호출부)은 `undefined`가 되어 SDK가 생성자 30초를 사용.

## 6. 미확정 사항

- **`timeoutMs=120000`은 초기 가정치**(실측 60초+에 여유). 라이브 스모크에서 `errorCount`(타임아웃, AbortError) vs `nullCount`(파싱/스키마)로 원인을 재판별한 뒤, 실측 p95×여유로 확정한다. §3-4 관측성 로깅이 판별 근거를 제공(T6로 로깅 자체는 검증됨). **확정은 config 1값 수정으로 충분 — 코드 변경 불요.**
  - `errorCount>0`(타임아웃)이면 값 재상향 또는 후속 B2(guided JSON off) 검토.
  - `nullCount>0`(파싱/스키마)이면 프롬프트/스키마 조정.

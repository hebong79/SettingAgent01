# SettingViewer 절대 zoom 입력 버그 수정 — 빈 칸이 현재값을 0/1로 리셋하던 문제

## 1. 증상 (마스터 보고)

> "zoom 조절 기능이 잘 안됨 (2가지만 줌만 됨 — 멀리, 가까이). 버튼은 되는데 절대입력이 안 됨."

- **+zoom / −zoom 버튼**: 정상 동작 → 사용자가 관찰한 "멀리·가까이 2가지"
- **절대 이동(zoom 숫자 입력 후 `이동`)**: 원하는 배율이 반영되지 않음

## 2. 원인 분석

전 경로(웹 UI → Node REST 클라이언트 → Unity REST/RPC 서버)는 zoom(1~36)을 **연속값**으로
탄젠트 광학 모델(`tan(hFOV/2)=tan(58°/2)/zoom`)에 그대로 적용한다. 즉 **양자화(2단계) 코드는 없다.**

문제는 웹 뷰어의 절대이동 핸들러 한 곳이었다. ([app.js] 기존)

```js
$('btn-abs').addEventListener('click', () => {
  move({
    pan:  Number($('abs-pan').value)  || 0,   // 빈칸 → Number('')=0 → 0||0 = 0
    tilt: Number($('abs-tilt').value) || 0,   // 빈칸 → 0
    zoom: clampZoom(Number($('abs-zoom').value) || 1), // 빈칸 → 0||1 = 1 (최광각으로 리셋!)
  });
});
```

- `Number('') === 0` 이므로 빈 칸은 `|| 0` / `|| 1` 폴백에 걸려 **현재 카메라 상태를 무시하고 리셋**된다.
- zoom 만 채우고 이동 → pan/tilt 가 0 으로 튀어 프레이밍이 깨진다.
- zoom 을 비우고 이동 → 배율이 **1(멀리)** 로 돌아간다.

반면 `+zoom/−zoom` 버튼은 `stepPtz(state.ptz, dir, step)` 로 **현재값(state.ptz) 기준 ±1** 을 계산하므로
빈 입력창의 영향을 받지 않아 항상 동작했다. → "버튼은 되는데 절대입력이 안 됨" 증상과 정확히 일치.

## 3. 수정 내용

### 3.1 순수 헬퍼 추가 — [SettingAgent/web/core.js]

프로젝트 컨벤션(순수 로직은 `core.js`, DOM/브라우저 비의존, vitest 직접 검증)에 맞춰
`resolveAbsPtz(cur, raw)` 를 추가했다.

```js
export function resolveAbsPtz(cur, raw) {
  const pick = (s, fallback) => {
    const t = (s ?? '').trim();
    if (t === '') return fallback;          // 빈 칸 → 현재값 유지
    const n = Number(t);
    return Number.isFinite(n) ? n : fallback; // 비수치 → 현재값 유지
  };
  return {
    pan:  pick(raw.pan,  cur.pan),
    tilt: pick(raw.tilt, cur.tilt),
    zoom: clampZoom(pick(raw.zoom, cur.zoom)),
  };
}
```

- **빈 칸/비수치**: 현재 PTZ(`cur`) 유지 → 리셋 금지.
- **입력 `0`**: 유효값으로 반영(빈 칸과 구분).
- **zoom**: 기존과 동일하게 `clampZoom`(1~36) 적용.
- 순수함수(원본 불변).

### 3.2 핸들러 교체 — [SettingAgent/web/app.js]

```js
$('btn-abs').addEventListener('click', () => {
  move(
    resolveAbsPtz(state.ptz, {
      pan:  $('abs-pan').value,
      tilt: $('abs-tilt').value,
      zoom: $('abs-zoom').value,
    }),
  );
});
```

- import 목록에 `resolveAbsPtz` 추가. `clampZoom` 은 다른 사용처(1327행)가 있어 유지.

### 3.3 타입 선언 — [SettingAgent/web/core.d.ts]

`resolveAbsPtz(cur: Ptz, raw: { pan?; tilt?; zoom? }): Ptz` 선언 추가.

## 5. 영향도 분석

- **변경 파일**: `web/core.js`(순수 추가), `web/app.js`(핸들러 1곳 교체·import 1줄), `web/core.d.ts`(선언), `test/viewerCore.test.ts`(테스트).
- **동작 변화 범위**: 뷰어 "절대 이동" 버튼 클릭 경로 **한 곳**. `+/−`·프리셋 이동·라이브 스트림·폴링 경로는 불변.

## 6. 후속 수정 — Unity 서버 zoom 역산 불일치 해결 (2026-07-15, 유니티 서버 재점검)

마스터의 "zoom 방식 재점검"(유니티 서버 수정 직후) 요청으로 서버 zoom 방식을 전면 재검증했다.

### 6.1 재검증 결과 — 계약은 탄젠트 광학 모델

`Assets/Tests/Editor/CObjCamera_ZoomFOV_Test.cs`(12 케이스)가 계약을 명시한다:
- **적용(forward)** `SetZoomByFOV`: `halfTanH = tan(58°/2)/zoom → hFov = 2·atan(halfTanH) → vFov`

## 7. 후속 수정 — +zoom/−zoom 이 step 값을 무시하던 문제 (2026-07-15)

### 7.1 증상 (마스터 보고, 라이브 화면)
> "step 값에 무관하게 zoom 이 된다. step 값에 따라 zoom 이 되게 수정."

step 입력을 0.01 로 두어도 `+zoom/−zoom` 버튼이 항상 배율 1씩 이동.

### 7.2 원인
`stepPtz`(core.js)에서 pan/tilt 는 `±step` 인데 **zoom 만 `±1` 하드코딩**되어 step 입력이 무시됐다.

```js
case 'zoomIn':  next.zoom = clampZoom(cur.zoom + 1); break;  // step 무시
case 'zoomOut': next.zoom = clampZoom(cur.zoom - 1); break;
```

### 7.3 수정 — zoom 도 ±step
```js
case 'zoomIn':  next.zoom = clampZoom(cur.zoom + step); break;
case 'zoomOut': next.zoom = clampZoom(cur.zoom - step); break;
```
- pan/tilt·zoom 모두 동일하게 step 입력값을 증분으로 사용(clampZoom 1~36 유지).
- **변경 파일**: `web/core.js`, 테스트 `test/viewerCore.test.ts`(zoom step 케이스 갱신).

### 7.4 검증
```
npx vitest run test/viewerCore.test.ts  → 72 passed
```
- `stepPtz(cur10, 'zoomIn', 5) → 15`, `'zoomOut', 3 → 7`, `'zoomIn', 0.01 → 10.01`(미세 step 반영), 상·하한 클램프 유지.

### 7.5 영향도
- 영향 범위: `+zoom/−zoom` 버튼 경로 한 곳. 절대이동·프리셋·방향패드는 종전 동작(방향패드는 이미 ±step).
- 참고: step 입력은 pan/tilt(도)와 zoom(배율) 공용이므로, 미세 zoom(0.01)을 쓰면 pan/tilt 도 0.01° 단위로 미세해진다(기존 공용 설계 유지, 별도 요청 시 분리 가능).

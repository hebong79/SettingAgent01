- 기준 산출물: 설계서 `20260625_224842_*.md`, `_workspace/01~03`(계획·구현·검증)
- 한 줄 요약: 시간에 걸쳐 반복 캡처한 원시 검출을 SQLite에 누적 → 결정형 시공간 집계로 통계적 ROI → 중간 체크포인트 LLM 보정 + 최종 LLM 판정으로 **단발 1프레임보다 정밀한 주차면(ROI·전역 인덱스)**을 확보한다.

## 1. 개요 — 무엇을·왜

### 1.1 문제(단발 1프레임의 한계)
기존 `/setup/*`는 프리셋당 단발 캡처(`accumFrames=1`) → VPD 검출 → ROI. 한 시점 1프레임은:
- **빈 면을 놓친다** — VPD는 "주차된 차량"을 검출하므로, 그 순간 비어 있는 면은 면으로 드러나지 않는다.
- **지터·노이즈** — 검출 bbox가 흔들리거나, 지나가는 차/반사로 가짜 면이 생긴다.

### 1.2 해법(시간 누적 정밀화)
시간에 걸쳐 점유 상태가 바뀌며 반복 수집하면 **모든 면이 결국 한 번씩 점유되어 드러나고**, 같은 면에 검출이 누적되어 통계적으로 안정된 ROI(다수 관측의 중앙값)가 나온다. 부산물로 면별 **점유율(occupancyRate)**도 자연히 수집된다. 이를 SQLite에 영속화하고 LLM이 근거를 보고 판정하여 정밀 주차면을 확보한다.

### 1.3 방식 C(체크포인트 하이브리드) 채택 근거
LLM 관여 시점에 따라 세 안이 있었다(설계서 §2).

| 방식 | LLM 호출 | 평가 |
|------|----------|------|
| A. 끝에 1회 | 1회 | 저렴하나 수집 중 잘못된 방향(특정 프리셋 과소수집·명백 노이즈)을 끝나서야 인지 |
| B. 매 프레임 | 매 캡처 | 토큰 폭증·중복·느림(비채택) |
| **C. 중간 체크포인트(채택)** | (N/K)+1 | 저렴하면서 **수집 중 온라인 보정·수렴 자문**. A의 "늦은 인지" 해소 |

**채택 = C.** 결정형 누적이 정확도의 뼈대이고 LLM은 전략·판정의 보조다. K라운드마다 LLM이 지금까지의 집계를 보고 증분 정리(병합·라벨·노이즈 거부)와 커버리지 진단·수렴 자문을 내며, 마지막에 1회 최종 판정한다. `llm.enabled=false`면 C는 자동으로 **A의 결정형 경로로 강등**(체크포인트·최종 LLM 생략, 결정형 집계만)되어 무LLM에서도 동작한다.

## 2. 파이프라인

```
[수집(반복)]  CaptureJob
   매 라운드: 프리셋 순회 → CameraClient.requestImage → VpdClient(+LpdClient) 검출
            → SqliteStore 적재(observation/detection)   ※ LLM 호출 없음
        │  (K라운드마다)
        ▼
[결정형 집계]  Aggregator.aggregate (순수함수)
   프리셋별 분리 → 그리디 클러스터(clusterDist) → 지지(support)·중앙값 대표 bbox
   → occupancyRate(점유 라운드/총 라운드) → plateRoi 귀속 → status(candidate|rejected)
        │
        ▼
[체크포인트 LLM(선택)]  CheckpointReviewer.review
   집계 텍스트 요약 → brain.reviewCheckpoint → merges/labels/rejects(status 메타 갱신, 좌표 불변)
   + coverage/convergence(checkpoint 행 저장 + status.latestAdvisory 표시)
        │  (정지/완료 후 /capture/finalize)
        ▼
[최종화]  Finalizer.finalize
   전체 집계(체크포인트 status 보존) → (LLM 활성)finalizeCapture 보조판정(중복/라벨/거부/리포트)
   → orderByPosition → buildGlobalIndex → validateCoverage → SetupArtifact
   → Repository.saveArtifact(setup_artifact.json) + artifact_snapshot 기록
        │
        ▼
   /mapping 이 최종 산출물 반환 → SettingViewer ROI 검수가 정밀 결과 표시
```

## 3. 신규 모듈별 설명 (`SettingAgent/src/capture/`)

### 3.1 `SqliteStore.ts` — SQLite DAO
- `better-sqlite3`(동기·프리빌트) 기반. 생성자에 **dbPath 주입**(`:memory:` 또는 파일경로 — 테스트 가능). 파일경로면 디렉터리를 `mkdirSync(recursive)`로 자동 생성하고 `journal_mode = WAL` 설정. 생성 시 6테이블 + 2인덱스를 `IF NOT EXISTS`로 보장(재생성 무해).
- **좌표·통계 read/write만 담당**, 비즈니스 로직(클러스터링/판정) 없음.
- 공개 메서드: `createRun`/`updateRunProgress`/`endRun`/`getRun`/`listRuns` · `insertObservation`/`insertDetections`(트랜잭션 일괄) · `getDetectionsForRun`(observation 조인으로 round_idx 부여) · `getPresetRounds`(occupancy 분모, `${cam}:${preset}` → distinct round 수) · `replaceAggregatedSlots`(run 기준 delete+insert 멱등, 트랜잭션 원자적)/`getAggregatedSlots`/`updateAggregatedStatus`(status 컬럼만 — 좌표 불변) · `insertCheckpoint`/`getLatestCheckpoint`/`getCheckpoints` · `insertArtifactSnapshot` · `close`.
- 계획 대비 가산 DAO 3개: `getPresetRounds`·`updateAggregatedStatus`·`getCheckpoints`(occupancy 분모·체크포인트 status 반영·Finalizer 컨텍스트 충족). snake_case 컬럼 ↔ camelCase 필드는 SELECT 별칭/`mapRun`으로 변환.

### 3.2 `Aggregator.ts` — 결정형 시공간 집계 (순수함수, DB·IO 비의존)
- `aggregate(dets: DetectionRow[], presetRounds: Map<string,number>, opts): AggregatedSlot[]`.
- 알고리즘:
  1. `conf < minConfidence` 제외, **프리셋별 분리**(`${camIdx}:${presetIdx}` 키 — 다른 프리셋 좌표계 혼합 금지).
  2. `vehicle`/`plate`를 각각 그리디 클러스터링(중심 유클리드 거리 `clusterDist` 이내면 같은 클러스터).
  3. 대표 bbox = 멤버 **중앙값**(`medianRect` — 지터에 평균보다 강건).
  4. `support`(멤버 검출 수) `< clusterMinSupport` → `status:'rejected'`, 이상 → `'candidate'`.
  5. `occupancyRate` = 클러스터가 등장한 distinct round 수 / 프리셋 총 라운드 수(0 division 방어).
  6. plate 귀속: 번호판 대표 중심이 vehicle 대표 ROI 내부 + 겹침(`intersectionArea`) 최대인 vehicle 클러스터에 `plate_*` 채움(기존 `matchPlatesToSlots` 규칙 차용).
- `domain/geometry`의 `center`/`containsPoint`/`intersectionArea`/`median` 재사용. positionIdx·zone은 부여하지 않음(Finalizer 책임 — 집계는 좌표·통계만).

### 3.3 `CaptureJob.ts` — 반복 수집 상태머신
- 상태: `idle → running → (stopping) → finalizing → done | stopped | error`. **단일 인메모리 잡**(중복 시작 시 `throw 'capture already running'`).
- `start(p)`: 런 생성 후 첫 라운드를 `setTimer(fn, 0)`로 즉시 예약(주기는 2번째 라운드부터 `intervalMs`).
- 라운드: 프리셋 순회 → `camera.requestImage` → `vpd.detect`(+`lpd.detect`, `lpdEnabled` 시) → `insertObservation`/`insertDetections`. **개별 프리셋 캡처 실패·LPD 실패는 경고로 흡수**(잡 미중단). 라운드 전체 예외 → `error`.
- `done % checkpointEvery === 0`이면 집계 후 체크포인트(reviewer 있을 때).
- `stop()`: `running → stopping`. 라운드 진행 중이 아니면 즉시 `stopped`(`manual`), 진행 중이면 현재 라운드 마치고 종료. `done >= planned` 도달 시 자동 `done`(`count`).
- `setTimer`/`clearTimer`/`sleep`/`now` 주입(fake timers 테스트). `getStatus()`/`getRunId()` 제공.

### 3.4 `CheckpointReviewer.ts` — 중간 체크포인트 보정
- `review(runId, atRound, plannedCount, slots, newFacesRecentK, expectedByPreset?)`.
- `brain` 미주입/`enabled=false`/`reviewCheckpoint` 미구현/예외 시 **null no-op**(체크포인트 실패가 잡을 막지 않음 — 장애 격리).
- 프리셋별 텍스트 요약(slotCount=rejected 제외 후보 수, avgOccupancy) → `brain.reviewCheckpoint` → 결과의 `rejects`→status `rejected`, `merges` 그룹 2번째부터→status `merged`로 **status 메타만 갱신(좌표 불변)**. 결과 JSON을 `checkpoint` 행에 저장.
- 순수 헬퍼 export: `clusterRef`(`presetKey#clusterId`), `advisoryLines`(coverage short/convergence → 표시 문자열 배열).

### 3.5 `Finalizer.ts` — 최종화
- `finalize(runId): Promise<{artifact, slots, globalCount}>`.
- 전체 재집계 후 **체크포인트에서 갱신한 status(merged/rejected) 보존 병합** → `replaceAggregatedSlots`.
- (LLM 활성 시) `finalizeCapture` 보조 판정(중복 그룹·zone 라벨·거부·한국어 리포트). 예외 시 결정형 강등.
- `status==='candidate'` & LLM rejects/duplicates 비대상 클러스터만 채택 → `orderByPosition`으로 positionIdx → slotId `c{cam}p{preset}s{pos}` → `pad(roiPadding)` → `ParkingSlot`/`Preset`/`IndexableSlot` 조립 → `buildGlobalIndex`/`validateCoverage`(불일치 시 throw) → `SetupArtifact` → `repo.saveArtifact` + `artifact_snapshot`.
- 좌표 불변식: `ParkingSlot.roi` = 집계 대표 bbox(+패딩) 그대로. LLM은 zone/report/채택여부 메타만.

### 3.6 `types.ts` — capture 내부 타입
- `CaptureRunRow`/`ObservationRow`/`DetectionRow`/`AggregatedSlot`/`CheckpointRow`/`CaptureState`/`CaptureStatus`. **`@parkagent/types`(공유 계약)와 분리** — 공유 계약 오염 금지. 좌표 정규화(0~1), cam/preset/round 1-based.

## 4. SQLite 스키마 · DB 파일 위치

DB 파일: `data/observations.sqlite`(`capture.dbFile`, `SqliteStore`가 디렉터리 자동 생성).

| 테이블 | 역할 | 핵심 컬럼 |
|--------|------|-----------|
| `capture_run` | 캡처 잡 1회(런) | planned_count, done_count, interval_ms, status(running\|stopped\|done\|error), stop_reason(count\|manual\|error) |
| `observation` | 라운드·프리셋 단위 관측(프레임 1장) | run_id, round_idx, cam_idx, preset_idx, captured_at, pan/tilt/zoom, img_name |
| `detection` | 검출 박스(정규화 0~1) | observation_id, cam_idx, preset_idx, kind(vehicle\|plate), x/y/w/h, conf |
| `aggregated_slot` | 집계 결과(집계 시 갱신) | run_id, preset_key, cluster_id, x/y/w/h, support, occupancy_rate, plate_x/y/w/h, status(candidate\|accepted\|rejected\|merged) |
| `checkpoint` | LLM 체크포인트 기록 | run_id, at_round, created_at, summary_json |
| `artifact_snapshot` | 최종 산출물 스냅샷(감사용) | run_id, created_at, artifact_json |

인덱스: `idx_det_obs ON detection(observation_id)`, `idx_obs_run_preset ON observation(run_id, preset_idx)`.

## 5. REST API (`/capture/*`, SettingAgent :13020)

기존 `/setup/*`·`/mapping` **불변**, 전부 가산. 의존성(`captureJob`/`finalizer`/`sqlite`/`capture`) 미주입 시 라우트 미등록.

| 메서드 | 경로 | 요청 | 응답 |
|--------|------|------|------|
| POST | `/capture/start` | `{count:int>0, intervalMs?, checkpointEvery?, targets?:Target[]}` | `{ok, runId}` / 400(invalid·target 미해결) / 409(already running) |
| GET | `/capture/status` | — | `{state, runId?, round, done, planned, latestAdvisory?}` |
| POST | `/capture/stop` | — | `{ok, state}` / 400(not running) |
| POST | `/capture/finalize` | `{runId?}` | `{ok, slots, globalCount}` / 409(running) / 404(no run) / 500 |
| GET | `/capture/runs` | — | `CaptureRunRow[]` |
| GET | `/capture/runs/:id/aggregate` | params id:int>0 | `AggregatedSlot[]` / 400 / 404 |

- `targets` 미지정 시 `loadSetupTargets(mapFiles)`(camerapos)로 해결. 미해결 시 400.
- `intervalMs`/`checkpointEvery` 미지정 시 `cfg.capture` 기본값.
- `finalize`의 기본 runId: `body.runId` → `job.getRunId()` → `listRuns(1)[0].id`(종료된 최근 런).

## 6. 설정 (`tools.config.json` → `capture` 섹션)

```jsonc
"capture": {
  "defaultCount": 50,       // 기본 반복 횟수(정지조건)
  "intervalMs": 30000,      // 라운드 주기 ms(점유 변화 포착)
  "checkpointEvery": 10,    // K라운드마다 LLM 체크포인트(llm.enabled 시)
  "dbFile": "data/observations.sqlite",
  "clusterDist": 0.06,      // 집계 클러스터 거리(정규화)
  "clusterMinSupport": 3,   // 면 인정 최소 지지수(단발보다 높게)
  "minConfidence": 0.5      // 집계 대상 최소 신뢰도(setup 과 독립값 허용)
}
```
- `CaptureSchema`(zod)로 검증. `loadToolsConfig`는 `DEFAULT_TOOLS_CONFIG` 키 루프 병합이라 `capture` 자동 포함 → 기존 config(capture 없음)도 기본값으로 채워져 파싱 성공(하위호환).
- **`llm.enabled=false` 결정형 강등**: 체크포인트·최종 LLM 생략 → 결정형 집계만으로 `SetupArtifact` 산출(zone=`cam{N}` 기본, 중복 제거·report 생략, rejected 제외).

## 7. SettingViewer 사용법 — 정밀 수집 탭

- **프록시**(`src/server.ts`): `/viewer/api/capture/{status,runs,runs/:id/aggregate}`(GET)·`{start,stop,finalize}`(POST) → SettingAgent `/capture/*` 중계. `registerViewerRoutes`(정적 와일드카드) **전**에 등록. 에러 처리는 `/mapping` 패턴 + 400/409 패스스루(중복 시작·정지 불가 본문 그대로), 5xx→502, 미가동→502 `unreachable`.
- **순수 로직**(`web/core.js`, DOM/fetch 미참조): `captureProgress(status)`(percent·label, 0 division 방어·100 클램프), `mapAdvisory(status)`(자문 배열 복사), `pollPlan(state)`(running/stopping/finalizing 만 폴링·간격).
- **UI**(`web/index.html` "정밀 수집"(`precise`) 탭 + `web/app.js`): 반복횟수·주기·체크포인트 입력 → 시작/정지/최종화 버튼 → 진행바(done/planned)·라운드·체크포인트 자문(수렴/부족 프리셋) 표시 → 최종화 후 `/viewer/api/mapping` 재로딩으로 ROI 검수.

### 실행 흐름 예
1. 정밀 수집 탭에서 반복횟수 입력 → **시작**(`POST /capture/start`).
2. N회 라운드 수집(프리셋 순회 캡처→검출→DB 적재). 진행바·라운드 갱신.
3. (K라운드마다) 체크포인트 자문 표시 — 수렴/부족 프리셋. **자동 정지 아님**(표시만).
4. 횟수 도달(자동 `done`) 또는 **정지**(`POST /capture/stop`, 언제든).
5. **최종화**(`POST /capture/finalize`) → 집계+LLM → `setup_artifact.json` 갱신.
6. 검수 탭에서 `/mapping` 확인 → 정밀 ROI 결과.

## 8. 좌표 불변식 (정확도 보장)

> 좌표는 검출+집계만 생성/수정한다. LLM은 좌표를 만들거나 바꾸지 않는다 — 판정·전략만.

- 유일한 bbox 산출처 = `Aggregator`의 `medianRect`(검출 멤버 중앙값). plate도 동일.
- `CheckpointReviewer`: `merges`/`rejects` → `aggregated_slot.status`만 변경(`updateAggregatedStatus`는 status 컬럼만).
- `Finalizer`: LLM `zoneLabels`→slot.zone, `report_ko`→artifact.report, `rejects`/`duplicates`→채택 제외. `roi`는 집계 대표 bbox(+패딩) 그대로.
- `coverage`/`convergence`는 `checkpoint.summary_json` 저장 + `status.latestAdvisory` 표시만(자동 정지 아님).
- 검증자가 `toEqual`로 입력=출력 bbox 동일을 명시 단언, 전부 통과(상세 §9).

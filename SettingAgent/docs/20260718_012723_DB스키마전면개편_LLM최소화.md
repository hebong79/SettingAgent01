## 1. 배경 · 목표

마스터 지시: "현재 사용중인 DB를 모두 지우고 `my_db_table.md` 기준으로 만들고, 테이블을 최적화하라." 부수 지시로 LLM 보조를 "점유 여부 판정" 하나만 남기고 최소화하라는 요구가 있었다.

확정된 결정(설계서 §0):
1. 기존 10테이블 폐기, `my_db_table.md` 기준 최종 **6테이블**로 신규 DB 파일(`data/setting.sqlite`)을 만든다. 구 DB(`data/observations.sqlite`)는 롤백 지점으로 무접촉 보존한다.
2. `run_id`를 DB에서 완전히 제거한다 — 결과 테이블은 "슬롯당 1행" 불변식을 갖는다.
3. DB를 슬롯 ROI/프리셋 PTZ의 정본으로 전환한다(카메라 외부 기하는 후속 과제로 부분 정본화, §4).
4. LLM 보조를 `occupancy.yaml`(점유 여부만)로 축소하고 나머지 11개 프롬프트를 아카이브하며, 죽은 코드 `adviseCentering`을 삭제한다.
5. `dbRoutes`의 `camera_info.password` 평문 노출을 차단한다.

## 2. 최종 6테이블 스키마

### 2.1 DDL 요지 (`src/capture/SqliteStore.ts:41-127`)

```sql
PRAGMA foreign_keys = ON;   -- 연결마다 명시(SqliteStore.ts:31). SQLite 기본은 OFF.

place_info(place_id PK, place_name)
camera_info(cam_id PK, cam_name, cam_uuid, url, user_id, password, rtsp_url,
            cam_type CHECK IN('ptz','static'), cam_company, place_id FK→place_info,
            img_w, img_h, updated_at)
preset_pos(cam_id, preset_id, sname, pan, tilt, zoom, updated_at, PK(cam_id,preset_id),
           FK cam_id→camera_info)
slot_setup(slot_id PK, cam_id, preset_id, preset_slotidx, slot_roi, vpd_bbox, lpd_obb,
           occupy_range, pan, tilt, zoom, centered CHECK IN(0,1), img1, updated_at,
           FK(cam_id,preset_id)→preset_pos, UNIQUE(cam_id,preset_id,preset_slotidx))
  INDEX idx_slot_setup_campreset(cam_id, preset_id)
parking_evnt(evnt_id PK AUTOINCREMENT, slot_id FK→slot_setup, is_occupy CHECK IN(0,1),
             update_time, plate_num, img1, img2)
  INDEX idx_evnt_slot_time(slot_id, update_time DESC)
parking_slot(slot_id PK FK→slot_setup, last_evnt_id FK→parking_evnt)
```

- **좌표계 규약**: `slot_roi/vpd_bbox/lpd_obb/occupy_range`는 정규화 0~1 JSON TEXT. PTZ는 `pan/tilt/zoom` REAL 3컬럼(JSON TEXT 아님). 이미지는 BLOB이 아니라 상대경로 TEXT.
- **PK/FK/인덱스/CHECK**는 위 DDL 그대로이며, 실제 소스(`SqliteStore.ts:41-127`)와 설계서(`01_architect_plan.md` §1) DDL이 일치함을 직접 대조 확인했다.
- **slot_roi JSON 포맷은 설계 주석과 다르게 확정**되었다: 설계 DDL 주석은 `[[x,y],...]`이나, 실제 구현은 `[{x,y}×4]`(`NormalizedPoint[]`)로 저장한다(`02a_developer_dbcore.md` §2, `SqliteStore.ts:242` `roi: parseJsonOrNull<NormalizedPoint[]>`). 근거: 기존 정본 경로(`Finalizer.roiJson = JSON.stringify(sp.points)`)와 뷰어 소비 형태가 이미 객체형이라, 배열형으로 바꾸면 뷰어/파서 전면 개편이 유발된다 — CLAUDE.md 규칙3(외과적 변경) 적용 결과다.

### 2.3 병합 근거 (floor_ROI + centering → slot_setup)

`my_db_table.md`의 `floor_ROI`(§1: slot_id/cam_id/preset_id/preset_slotidx/slot_roi/**pos**)와 `centering`(§5: slot_id/cam_id/preset_id/vpd_bbox/lpd_obb/occupy_range/**ptz**/img1)은 키·카디널리티가 동일("전체 슬롯 개수만큼")하고, `pos`(floor_ROI)와 `ptz`(centering)는 둘 다 번호판중심 센터라이징 PTZ로 **동일값**이다. 코드상으로도 `Finalizer.ts:242-253`이 한 슬롯의 roi/vpd/lpd/occupied/pan·tilt·zoom을 이미 한 행으로 조립하고 있어, 마스터 지정대로 `pan/tilt/zoom` 단일 3컬럼으로 병합해 중복 저장을 제거했다(설계서 §1.1).

### 2.4 slot_id 전역 체계

`slot_id`는 `normalizeGlobalIdx(placeRoi.ts)`가 산출하는 **전역 정수 1..N**이다. 문자열 `c{cam}p{preset}s{pos}`(구 `centering_slot.slot_id`)는 `(cam_id, preset_id, preset_slotidx)`에서 파생 가능하므로 slot_setup에는 저장하지 않는다(설계서 §1.4). QA가 `boundaryCrossCheck.test.ts`로 `getSlotSetup().slotId` 집합과 `setup_artifact.globalIndex.globalIdx` 집합의 일치를 직접 검증했다(§8 검증결과 참조).

## 3. run_id 제거와 확정본 보호 설계

- **제거 범위**: `createRun/updateRunProgress/endRun/getRun/listRuns` 등 run_id 기반 메서드 전량을 `SqliteStore`에서 삭제했다(`02a_developer_dbcore.md` §2.5). run_id는 이제 `CaptureJob`의 인메모리 카운터(`runSeq`, 로그·status 표시 전용)로만 남는다.
- **확정본 보호**: `finalize` 시 `slot_setup` 전량을 `DELETE` 후 `INSERT`하는 **단일 트랜잭션**(`SqliteStore.replaceSlotSetup`, `SqliteStore.ts:189-208`)으로 교체한다. 중간에 예외가 발생하면 better-sqlite3 트랜잭션이 자동 롤백되어 DELETE까지 포함해 전체가 되돌아가고, 이전 확정본이 보존된다.
- **원자성 검증**: QA가 `sqliteStore.test.ts`에서 정상 삽입 1건 저장 후, PK 충돌을 유발하는 두 번째 `replaceSlotSetup([dup1, dup2])`를 호출해 throw + 전체 롤백 + 이전 확정본(`[slotId 1]`) 보존을 확인했다(`03_qa_report.md` "신규 테스트" 표).
- **재시작 복구는 의도적으로 미지원**: 크래시 시 진행 중(미확정) 인메모리 누적은 소실되지만, 확정본(slot_setup)은 항상 온전하다는 것이 설계 전제다(설계서 §2.4). "부분 복구"가 필요하면 append-only 스냅샷이 추가로 필요하나 마스터 결정이 이를 배제하므로 채택하지 않았다.

## 4. 캡처 파이프라인 인메모리 누적 전환

구 구조는 관측·검출·집계·floor·occupancy·checkpoint를 전부 `run_id` 키로 DB에 기록했다. 신 구조는 `CaptureJob`이 이를 필드로 보유한다(`02c_developer_capture.md` §2-3):

```ts
export interface CaptureSnapshot {
  dets: DetectionRow[];
  presetRounds: Map<string, number>;
  aggregated: AggregatedSlot[];
  occByPreset: Map<string, OccupancyJudgment>;   // LLM off → 빈 맵
}
```

- `dets`는 push, `roundsByPreset`은 Set, `aggregated`는 체크포인트마다 `aggregate()` 재호출로 갱신, `occByPreset`은 OccupancyReviewer가 기록한다. `start()`에서 전부 clear한다.
- `Finalizer.finalize(runId)` → `Finalizer.finalize(snapshot, opts)`로 시그니처가 바뀌었다. `getDetectionsForRun/getPresetRounds/getAggregatedSlots/getCheckpoints/getLatestOccupancy` 등 DB 조회는 전부 snapshot 필드 참조로 대체됐다.
- floor 발자국(`occupy_range`)은 영속하지 않고 finalize 시 결정형(`buildPlateAnchoredQuad`)으로 항상 재계산한다(`Finalizer.ts:320,344`).
- **동시성 전제**: `CaptureJob`은 이미 단일 인메모리 잡이고 중복 시작을 거부하므로(`CaptureJob.ts:202-204`) 인메모리 단일 소유자 전제가 성립하며, 병렬 런은 없다.

## 6. LLM 최소화 내역

### 6.1 occupancy.yaml 축소 (`config/prompts/occupancy.yaml`)
좌표/`points_2d` 요구를 제거하고 "점유 여부만" 판정하도록 축소했다. 출력 스키마는 `{"spaces":[{"id":1,"occupied":true}],"confidence":0.0}`. 파싱 코드(`OccupancyRawSpaceSchema`, `AgentRuntime.judgeOccupancy`)는 애초에 좌표를 optional로 다루고 있어 **스키마·파싱 코드 변경은 0**이었다(설계서 §6.1).

### 6.4 死코드 삭제 — `adviseCentering`
`AgentRuntime.adviseCentering`은 구현만 있고 호출자가 없는 死코드였다(센터라이징은 `PlatePtz`의 순수 P제어이며 LLM을 참조하지 않는다). 삭제 범위: `AgentRuntime.ts`(메서드+고아 import 3종), `SetupBrain.ts`(`CenteringAdviceInput/CenteringAdviceSchema/CenteringAdvice`+인터페이스 선언), `calibrate/types.ts`(`CenteringAdvice`), `llmConfig.ts`(`CenteringSchema`+`LlmConfigSchema.centering`+기본값+머지 라인). `grep -rn "adviseCentering|CenteringAdvice|CenteringSchema" src/ test/` → **0건**(개발자·QA 양측 재확인).

### 6.5 점유 정본 원리
캡처 루프의 점유 판정 정본은 기하 `pointInPolygon`(`Finalizer.ts:237`)이며, LLM(occupancy.yaml)은 finalize의 `occupancyAgreement` 교차검증용 **보조 지표**로만 남는다(`Finalizer.ts:133` "best-effort 지표"). 센터라이징은 원래부터 LLM이 아니라 `platePtz.ts`의 순수 P제어 기하였다 — 이번 변경으로 새로 결정형화된 것이 아니라 死코드(adviseCentering)만 제거된 것이다.

### 6.6 캡처 루프 reviewer 배선 정리
`CheckpointReviewer`/`FloorRoiReviewer`는 LLM off 상태에서 캡처 배선(`CaptureJob`/`index.ts`)에서 분리됐다(호출 자체가 제거). 단, **클래스/메서드 코드는 삭제하지 않았다** — CLAUDE.md 규칙3(요청 없는 기존 데드코드 삭제 금지)에 따라 배선만 끊는 최소안을 채택했다(설계서 §6.5, 리스크 5).

## 7. 보안 — password 마스킹

`GET /db/table/camera_info`가 `SELECT *`로 평문 password를 그대로 반환하던 문제를, `registerDbRoutes`(`src/api/dbRoutes.ts:16-18`)의 `SENSITIVE = { camera_info: Set(['password']) }` 맵으로 해결했다.

- **응답 마스킹**(`dbRoutes.ts:126-132`): 조회된 rows를 후처리해 값이 있으면 `'****'`, `null`이면 `null` 유지.
- **검색 제외**(`dbRoutes.ts:107-108`): `SENSITIVE`에 든 컬럼은 LIKE 검색 대상에서 원천 제외한다 — 마스킹된 값이라도 "검색어가 password 값과 일치하면 행이 노출되는" 존재여부 유출을 차단하기 위함.
- write 라우트가 없는 read-only 뷰어 전용이라 마스킹은 응답 계층 1곳으로 충분하다.

## 8. 마이그레이션 절차 · 검증 (리더 경험적 검증 포함)

### 8.1 스크립트
`src/tools/migrateToSettingDb.ts`(신규, 1회성 CLI): PtzCamRoi.json(cam_id/preset/slot 기하) + camerapos.json(preset PTZ) + centering 산출물(slot_ptz.json 우선, 없으면 구 DB의 `centering_slot`)을 신 6테이블로 이관한다(설계서 §5).

### 8.2 리더 실행 결과 (직접 라이브 확인 — 문서에 그대로 반영)
```
npx tsx src/tools/migrateToSettingDb.ts data/setting.sqlite
→ place=1 / camera=1 / preset=3 / slot=17
```

## 9. 테이블 최적화 관점 정리

마스터의 "테이블 최적화 검토" 요청에 대한 정리:

1. **중복 제거**: floor_ROI의 `pos`와 centering의 `ptz`가 동일값이던 것을 `slot_setup.pan/tilt/zoom` 단일 3컬럼으로 병합(§2.3). 구 `parking_slots`/`centering_slot` 두 테이블의 슬롯 단위 중복 저장도 하나로 합쳐졌다.
2. **정규화**: PTZ를 JSON TEXT 한 덩어리가 아니라 `pan/tilt/zoom` REAL 3컬럼으로 분해해 인덱스·산술 연산이 가능한 형태로 만들었다(구 `centering_slot.pos`는 PTZ JSON 문자열이었다).
3. **타입/CHECK 제약 강화**: `cam_type IN ('ptz','static')`, `centered IN (0,1)`, `is_occupy IN (0,1)` CHECK로 잘못된 값 삽입을 DB 레벨에서 차단(구 스키마엔 없던 제약).
4. **run_id 소거로 카디널리티 단순화**: 구 `parking_slots`는 런마다 스냅샷을 쌓아 561행(33개 run_id, 런당 약 17행)이 누적됐지만, 신 `slot_setup`은 "슬롯당 1행" 불변식으로 17행 고정 — 결과 테이블의 무한 증식을 제거했다.
5. **인덱스**: `idx_slot_setup_campreset(cam_id,preset_id)`(프리셋 단위 조회/뷰어 오버레이용), `idx_evnt_slot_time(slot_id, update_time DESC)`(슬롯 최신 이벤트 조회용) 신설.
6. **FK 무결성 활성화**: 구 `SqliteStore`는 `journal_mode=WAL`만 설정하고 `foreign_keys`는 SQLite 기본값 OFF였다. 신 스키마는 연결마다 `PRAGMA foreign_keys = ON`을 명시해 `slot_setup→preset_pos`, `parking_evnt/parking_slot→slot_setup` 등 참조 무결성을 실제로 강제한다(§8.2 `foreign_key_check` 빈 결과로 실효 확인).
7. **중간 산물의 비영속화**: observation/detection/aggregated/floor_roi/occupancy/checkpoint 6종을 DB에 남기지 않고 인메모리로 전환해, "정본(확정본)"과 "진행 중 산출물"의 경계를 테이블 설계 차원에서 분리했다 — 이는 컬럼/인덱스 최적화라기보다 스토리지 정책 최적화다(§3, §4).

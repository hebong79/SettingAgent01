## 0. 한 줄 요약

프리셋1 주차면 1~7 중 일부가 개별(클릭) center+zoom에서 실패하던 원인은 **줌이 아니라 그 앞 센터링**(`plate_lost`)이었고, 근본 원인은 **LPD 검출기가 픽셀 단위 프레이밍에 민감해 같은 대상을 1px대 차이로 놓친다**는 것이었다(리더 실측: 동일 장면 tilt 1.7px 차이로 검출 4개↔7개, 동일 이미지 1/2/3px 시프트로 5/7/7개, 육안으로 선명한 판을 검출기가 미검). 해법은 **① 미세 디더 재포착 ② 디더 단위를 각도가 아니라 픽셀 등가 정규화 변위 + 에스컬레이팅 사다리로 재설계 ③ 줌 스텝 직후 지점만 줌 승법 디더**의 3단으로 수렴했다. 그 과정에서 **줌 실패를 삼켜 거짓 성공(`ok:true`)을 내던 기존 결함(P1)**이 표면화되어 함께 제거됐다. 최종 라이브 실측 **슬롯 1~7 × 3라운드 = 21/21 성공**, 유닛테스트 **198파일/2327건 전건 통과**. 실카 미검증·A·B 지점 zoom 데드존 미대응 등 한계는 §8에 은닉 없이 남긴다.

## 1. 문제·원인(리더 실측 그대로)

### 1.1 증상
프리셋1 슬롯 1·2·3만 개별 center+zoom이 되지 않았다(수정 전 실측: slot1·2·3 = `ok:false reason=plate_lost`, zoom은 클릭 시점 그대로 1.69에 머무름. slot4~7은 `ok:true`, zoom 13.1~16.4).

### 1.2 실패 지점 — 줌이 아니라 센터링
`platePtz.ts` `centerOnPlate` 추적 루프(수정 전 :424~427)가 1회 미검이면 즉시 `plate_lost`를 반환하고, `PtzCalibrator.centerOnPoint`의 `if (centered.ok && ...)` 가드가 그 실패를 그대로 통과시켜 줌 단계는 시작조차 못 했다. goal의 재현 로그:

```
slot3 click(0.4263,0.6666) first err=(-0.074,0.167) probe gain=(-37.4,-21.4) 정상
  it0 cmd(pan=19.243,tilt=10.367) predict=(0.5,0.5) picked=null dist=0.126 count=4 REJECTED → plate_lost
```
그 PTZ 프레임에서 판은 실제로 `(0.5, 0.496)`에 있었고(스냅샷 육안 확인, 번호판 `402조8795` 선명), LPD가 그 판을 내지 않아 최근접 후보까지의 거리 0.126이 `matchRadiusNorm`(0.08)을 초과해 기각됐다.

### 1.3 근본 원인 — LPD 검출기의 픽셀 단위 프레이밍 민감도
리더가 직접 LPD를 호출해 확인한 두 실측:

| 실측 | 결과 |
|---|---|
| 동일 장면, tilt 10.367 | 검출 4개 |
| 동일 장면, tilt 10.4(0.033° ≈ 1.7px 차이) | 검출 7개 |
| 동일 이미지, 세로 1px 시프트 | 검출 5개 |
| 동일 이미지, 세로 2px 시프트 | 검출 7개 |
| 동일 이미지, 세로 3px 시프트 | 검출 7개 |

같은 프레임 내에서는 **결정적**(3회 반복 동일)이라 같은 PTZ로 재캡처만 해서는 회복할 수 없다. 즉 "재시도"가 아니라 "재프레이밍"이 필요하다는 것이 이번 goal의 핵심 통찰이다.

## 2. 해법 — 3단계로 수렴한 설계

### 2.1 이터레이션 1 — 미세 tilt 디더 재포착(각도 고정)
설계(`01_architect_plan.md` §3)는 추적 캡처 1회 미검 시 `tilt ± 0.03°`로 재캡처하는 신규 private 헬퍼 `captureTrack()`(`platePtz.ts`)을 `centerOnPlate`(A)·`zoomToPlateWidth`의 가드 재중심(B)·줌 후 캡처(C) 3곳에 주입했다. 발동 조건은 **검출 0(`count=0`)과 반경 기각(`rejected`) 둘 다**(§3.1) — goal의 재현 사례가 후자였기 때문이다. 게이트(`matchRadiusNorm`)는 재시도에도 **그대로** 재사용해 거짓 성공 금지선을 건드리지 않았다(§4 증명).

**라이브 결과(리더 실측)**: slot1 ✗ · slot2 ✓(신규) · slot3 ✓(신규) · slot4 ✓ · slot5 ✗ · slot6 ✓ · slot7 ✓. 디더 기구는 **작동**했으나(로그 `recovered:true` 3건) 남은 실패는 **디더 크기 부족**이었다. slot1 실패 프레임에서 디더 감도를 직접 스윕한 결과:

| 디더 | 결과 |
|---|---|
| tilt ±0.03°(≈1.5px) | 미검 |
| tilt ±0.06°(≈3px) | 미검 |
| **tilt ±0.12°(≈6px)** | **검출 ✓** |
| pan ±0.06° | 미검 |
| zoom 1.693→1.75 | 미검 |

→ 결론: 디더는 **고정 각도**가 아니라 **화면 픽셀 기준 등가 변위**로 잡고 **에스컬레이팅 사다리**로 키워야 한다.

### 2.2 이터레이션 2 — 단위 전환(각도 → 정규화 변위) + 에스컬레이팅 사다리
옵션 `plateRecaptureDitherDeg`(각도)를 하위호환 없이 폐기하고 **`plateRecaptureDitherNorm`**(정규화 변위, 기본 **0.0014** ≈ 1080p 1.5px)으로 교체했다. `ditherSeq(n,d)`(등차)도 **`ditherMultipliers(n)`** = `[+1,−1,+2,−2,+4,−4,...]`(배수 사다리, `2^floor(i/2)`)로 교체했다. 실제 명령 각도는 매 시도:

```
degPerNorm = |fallbackGainTiltDeg| / zoom   // 무측정 fallback 게인 × 현재 zoom — 고정 물리모델
dNorm = mult × plateRecaptureDitherNorm      // 부호 포함 화면 변위(정규화)
dDeg  = dNorm × degPerNorm                   // 실제 tilt 명령
```
로 환산한다. 검산: zoom 1.6934(클릭 base)에서 ±1배수 = 0.0294°(이터1의 0.03°와 거의 동일), **±4배수 = 0.1174°**(리더가 회복을 확인한 실측 0.12°와 일치). zoom과 무관하게 픽셀 등가가 고정되므로(zoom 36에서도 ±4배수 = 0.0055°는 여전히 같은 6px), 이터1에서 zoom이 커질수록 게이트 여유가 38%까지 깎이던 문제도 함께 해소됐다.

★ 환산에는 루프의 살아있는 게인(damp로 감쇠)이 아니라 **무측정 fallback 게인**을 쓴다 — 감쇠는 제어 사정이지 검출기 사정이 아니라서, 흔들어야 할 때 디더가 함께 작아지는 자기무력화를 막기 위함이다. prior 예측에는 종전대로 루프 게인을 쓴다(두 용도가 코드에서 분리됨).

재시도 기본값도 2 → **6**(에스컬레이팅 사다리 전체를 쓴다는 뜻)으로 올렸다. 라이브 결과(2라운드): slot1~7 **14/14 센터링 성공**(slot1은 mult 4 = ±0.117°로 두 번 재포착해 수렴, 이터1에서 실패하던 slot1·5 포함). 그러나 이 라운드에서 **새로운 결함 2건**이 드러났다(§3).

### 2.3 이터레이션 3 — 줌 승법 디더(C 지점) + P1 위장 성공 제거
줌 스텝 직후 캡처(C 지점)에서 실측한 zoom 데드존을, tilt 디더 대신 **zoom 승법 디더**로 처리한다. 신규 private 헬퍼 `captureTrackZoom()`을 도입하고, 신규 옵션 `plateRecaptureZoomStep`(기본 **0.01**)을 같은 `ditherMultipliers` 배수와 곱해 `×[1.01, 0.99, 1.02, 0.98, 1.04, 0.96]` 사다리를 구성한다. A·B 지점(pan/tilt 이동 후 캡처)은 **tilt 디더 그대로** 둔다(실측으로 효과가 이미 확인된 축이라 지시대로 손대지 않았다).

**줌 데드존 실측표(같은 pan/tilt, zoom만 변경)**:

| zoom | 7.8 | 8.0 | **8.1738** | 8.25 | 8.4 | 8.6 | 9.0 | 6.0 | 10.0 |
|---|---|---|---|---|---|---|---|---|---|
| 검출 | ✗ | ✗ | **✗**(plates:0) | ✓ 0.137 | ✓ 0.142 | ✗ | ✓ 0.156 | ✓ 0.117 | ✓ 0.188 |

그 프레임에서 **tilt 디더 7시도 전패**, **zoom +1%(8.1738→8.25)로 회복**됐다. 좁고 산발적인 데드존이 줌 축에 존재하며(8.6도 재차 ✗), 줌 스텝 직후 프레임은 배율 자체가 바뀐 새 프레임이라 회복 축도 zoom이어야 한다는 결론이다.

## 3. P1 — 위장 성공 제거(거짓 성공 금지선 위반, 최우선)

### 3.1 결함
`PtzCalibrator.centerOnPoint`의 비-사다리 경로가 (수정 전) `if (z.ok) return z;`만 갖고 있어, **줌이 실패하면 그대로 흘러내려 센터링 결과를 `ok:true` + reason 없이** 반환했다. 이터2 라이브 R1 slot1이 정확히 이 경로였다:

```
ok:true / zoom 1.69(=클릭 시점 그대로) / plateWidth 0.032(=base 폭, 줌 미발생)
```
UI는 이걸 "개별 센터라이징 완료"로 표시했다 — 줌이 전혀 안 됐는데 완료로 보인 것이다. **이 결함은 이번 이터레이션들 이전부터 존재했고, 센터링이 §2.1~2.2로 고쳐져 줌 단계까지 진입하는 슬롯이 늘어나면서 표면화됐다.**

### 3.2 수정
```ts
// 구: if (z.ok) return { ... };            // 실패면 낙하 → center 결과를 ok:true 로 반환(위장)
// 신: return { ok: z.ok, ptz: z.ptz, plateWidth: z.plateWidth, ...(z.reason ? { reason: z.reason } : {}) };
```
줌을 시도했으면 그 결과가 정본이다. `ptz`·`plateWidth`는 줌 단계가 마지막으로 도달한 실측값(카메라의 실제 현재 위치)을 싣는다 — 센터링 시점 값을 싣는 것은 위치까지 거짓말하는 것이었다. 사다리 경로(`centerAndZoomByLadder`)는 무변경 — 그쪽의 "장비 한계라 폭 미달이지만 `ok:true`+reason"은 *장비가 할 수 있는 일을 다 한* 경우고, 이쪽 실패는 **대상 소실**이라 성격이 다르다.

### 3.3 기존 테스트가 결함을 고정하고 있었다는 사실
`test/ptzCalibrator.point.test.ts`의 `centerOnPoint — 줌 실패 시맨틱(§5-A-4, 구현이 정본)` 케이스는 "z.ok=false면 낙하해 center 결과 반환(`ok:true`, reason 미전파)"을 **명시적으로 단언**하고 있었다 — 즉 이번 결함을 스펙으로 받아적은 스냅샷 테스트였다. 구현자가 설계 문서를 대조한 결과 이 단언에는 요구사항 근거가 없었고, **의미를 뒤집어 다시 썼다**(테스트 수 2295→2296, 수정 1건·추가 1건·삭제 0건).

QA는 이 반전에 대해 **독립적으로 재검토**했고(구현자의 판단을 그대로 받아들이지 않고), 사후조건 위반·위치까지 거짓말한 점·저장 미접촉(UI 문구 영향뿐)을 근거로 "반전은 타당하며 되돌리면 안 된다"고 결론지었다(`03_qa_report.md` §4). 남는 비대칭(사다리 경로는 여전히 `ok:true+reason`을 낼 수 있어 "완료"라는 단어가 두 성격이 다른 경우에 다 붙는다는 점)은 이번 범위 밖으로 사실만 기록했다.

## 4. 거짓 성공 금지선 — 어떻게 지켜지는가

1. **게이트 값 불변**: `captureTrack`/`captureTrackZoom` 시그니처에 `radius` 인자가 1개뿐이라 "재시도용 완화 반경" 개념 자체가 존재하지 않는다. 모든 디더 시도가 **호출측이 준 같은 `radius` 변수**로 게이트를 통과해야 한다.
2. **최대 디더 변위는 zoom과 무관하게 고정**: `4 × 0.0014 = 0.0056`(정규화). `matchRadiusNorm`(0.08)의 7%, 이웃 판 최소 간격(0.15)의 3.7%다. prior는 매 시도 디더분을 포함해 정확히 재계산되므로(대상·이웃은 강체 평행이동이라 상대거리가 디더에 불변), 예측 모델이 정확한 한 게이트 확장은 이론상 0이다.
3. **QA의 경계 두드림**: 게이트 바로 밖(0.0805 / 0.083 / 0.085)에 유령 판을 놓고 6디더를 전부 소진해도 **채택 0건** — 실측 기각거리(0.126)나 이론적 최악 확장선(0.0856)까지 어느 지점도 통과하지 않는다.
4. **최초 대상 선정(첫 캡처)에는 디더를 적용하지 않는다**(설계 §3.5) — 그 게이트 기준점은 조작자의 클릭 좌표라, 카메라를 흔들면 클릭이 가리키던 세계점이 프레임에서 이동해 기준 자체가 흐려지고, 검증할 관측 앵커도 없다. goal의 실패는 전부 추적 중(`plate_lost`)이라 범위 밖이었다.
5. **디더된 PTZ는 원복하지 않고 그대로 채택**한다 — 원복은 미검이 났던 그 프레임으로 되돌아가는 것이라 회복을 원점으로 되돌리며, "상태=내가 명령한 값"이라는 이 파일의 제1 불변식과도 어긋난다. 물리적 대가는 `centerTol`(0.03) 대비 base zoom에서 1/21(0.0014) 수준이고, 다음 반복이 실측 오차로 흡수한다.
6. **zoom 승법 디더도 폭 판정을 왜곡하지 않는다**(QA §3-2): 수렴 판정은 항상 디더된 그 프레임에서 실측한 폭으로만 내리며, 디더 전 폭을 재사용하는 코드 경로는 없다. 대가는 "최종 zoom이 무디더 대비 최대 4% 다르다"뿐이고, 그 폭은 그 배율의 참값이다.

### 6.2 배치 무회귀가 "구조로" 보장되는 근거

1. **옵션 기본값이 전부 0/기존값**: `PlatePtzOpts`의 신규 3키는 각각 `plateRecaptureRetries` 기본 **0**(재시도 없음), 나머지는 값이 있어도 `retries=0`이면 `mults = [0, ...ditherMultipliers(0)] = [0]`이라 사용되지 않는다. `retries=0`이면 `captureTrack`/`captureTrackZoom`은 캡처 1회·prior 동일·반환 `ptz===cmd`로 **기존 코드와 바이트 단위로 동일**하게 동작한다.
2. **주입 경로가 물리적으로 분리**: `recaptureOpts()`는 `PtzCalibrator.centerOnPoint`의 `makePlatePtz` 2곳에만 스프레드되고, `baseOpts()`·`ladderOpts()`·배치가 쓰는 다른 호출부에는 넣지 않았다. 즉 "무해할 것"이라는 논증이 아니라 **배치 코드가 신규 키를 담은 opts 객체를 만들 방법 자체가 없다**는 구조로 보장된다.
3. **QA 실측 확인**: `start()` 종단 실행 중 생성된 모든 PlatePtz opts에 `plateRecapture*` 키가 **하나도 없음**을 단언(`03_qa_report.md` §2-5), 사다리(`centerAndZoomByLadder`)도 `retries 0`과 `6`의 캡처 궤적·결과가 **완전 동일**함을 확인(D 지점 미적용 고정).

### 6.3 REST 계약 변화의 UI 영향(관찰 가능)

`web/app.js` `:2474-2487`은 `data.ok`가 true면 완료 문구, false면 `종료(${reason})`을 표시한다(코드는 무변경). P1 이전에는 slot1류 위장 성공이 `ok:true`로 떨어져 "개별 센터라이징 완료"만 보였다. P1 이후 같은 상황은 `ok:false + reason:'plate_lost'`가 되어 **"종료(plate_lost)"로 표시가 바뀐다**. 문자열 `plate_lost` 자체는 기존에도 나가던 값이라 UI에 새 분기가 필요하지는 않지만, **운영자가 보는 화면상의 결과 판정이 "완료"에서 "실패"로 뒤집히는 케이스가 실재한다** — 이는 결함 수정의 의도된 결과다. 육안 확인은 하지 않았다(§8-1).

### 6.4 config/nodemon 운영 주의사항

`src/`는 nodemon 감시 대상이라 자동 리로드되지만, **`config/` 디렉터리는 감시 밖**이다(전편 문서와 동일 주의사항). 신규 config 키(`pointRecaptureDitherNorm`/`pointRecaptureRetries`/`pointRecaptureZoomStep`)를 `config/tools.config.json`에 추가해 튜닝하는 경우 **서버 재시작이 필요**하다. 이번 이터레이션은 그 파일 자체를 건드리지 않았으므로 즉시 재시작 요구는 없다.

### 7.4 검증 중 발견한 경미 결함·비대칭(전부 실패로 분류되지 않음, 은닉 금지)

1. **`zoom_saturated` 반환에 `recaptureDithers`가 실리지 않는다**(`platePtz.ts`의 zoom 포화 반환 경로). 다른 모든 출구(성공/`plate_lost`/`max_iterations`)는 싣는데 이 출구만 빠져, 그 호출에서 앞선 반복이 디더를 이미 썼어도 횟수가 조용히 버려진다. 정직성 관용구의 유일한 구멍이며, 동작 결함은 아니고 라이브 진단 정보의 손실이다. 수정은 1줄(스프레드 추가)이지만 이번 범위에서 손대지 않았다.
2. **`captureTrack`과 `captureTrackZoom`의 `dithers` 산정 방식이 다르다.** `captureTrack`은 성공 시 시도 인덱스, 실패 시 `mults.length−1`을 그대로 쓰고(스킵 개념 없음), `captureTrackZoom`은 **실제 캡처만** 증가시킨다(clamp 스킵 제외). 둘 다 "실제 추가 왕복 수"로 해석하면 일관되지만 코드만 봐서는 다르게 보인다. QA가 각각의 현 정의를 테스트로 고정했다.

## 8. 한계·미해결(은닉 금지)

1. **실카(real-camera) 미검증.** 이번 세 이터레이션 전부 시뮬(`cam1/preset1`)로만 실측됐다. 디더 각 환산(`degPerNorm = |fallbackGainTiltDeg| / zoom`)이 **`fallbackGainTiltDeg`(−35.5 @zoomRef=1, cam1 시뮬 실측)에 100% 무측정 의존**한다 — 게인이 다른 카메라에서는 같은 `plateRecaptureDitherNorm`이 다른 픽셀 폭이 된다. 실카에서 디더가 안 먹으면 디더 상향이 아니라 **`fallbackGainTiltDeg` 실측치 점검**이 먼저다.
2. **사다리(`centerAndZoomByLadder`, D 지점) 경로는 이번 재포착 기구가 미적용.** 사다리는 rung마다 zoom이 달라 다음 프레임이 이미 재프레이밍되므로 디더의 목적이 부분적으로 내장돼 있고, 실카 전용 경로라 이번 루프(시뮬)에서 검증할 수단도 없었다. 실카에서 latch 후 `plate_lost`가 관측되면 같은 헬퍼를 그 지점에 붙이는 것이 다음 루프 후보다.
3. **A·B 지점(pan/tilt 이동 후 캡처)의 zoom 데드존은 미대응.** C 지점만 zoom 축 디더로 바뀌었다. 원리적으로 A·B도 같은 데드존을 밟을 수 있으나, 그 지점은 tilt 디더가 실측으로 효과가 확인됐다는 이유로 그대로 두었다(두 축을 겹쳐 쓰면 왕복이 6×6으로 폭증해 근거 없이 하지 않았다). 이번 라이브 21/21 성공으로 당장 문제가 되지 않는다는 증거는 있으나, 원리적 위험은 남는다.
4. **QA가 지적한 `captureTrack`/`captureTrackZoom`의 `dithers` 산정 방식 차이**(§7.4-2)는 동작 결함이 아니지만 코드 일관성 관점에서 남아 있다.
5. **`zoom_saturated` 출구의 `recaptureDithers` 누락**(§7.4-1)도 정직성 관용구의 구멍으로 남아 있다.
6. **왕복 비용(체감 지연)**: 1왕복 ≈ 0.6s 기준, tilt 디더는 회복까지 최대 +3.0s(4배수), 전패 확정 +3.6s. zoom 승법 디더도 같은 예산(+0.6~+3.6s)이며 두 축은 서로 다른 지점이라 곱해지지 않는다. **리더의 실측 체감**: slot1 클릭이 이터1 이전 8s에서 이번 재포착 도입 후 14s로 늘어났다(단일 슬롯 실측, 통계적 대표값 아님). 유닛테스트는 sleep을 0으로 주입하므로 이 지연 자체는 유닛으로 관측되지 않는다.
7. **8배수 이상(재시도 7·8)은 실측 근거가 없다.** `ditherMultipliers`는 `±8`(12px 등가)까지 자연 확장되지만 검증된 것은 ±4(6px)까지다.
8. **UI 육안 확인 없음.** P1로 문구가 바뀌는 경로(§6.3)는 코드 경로까지만 확인했고 브라우저 화면에서 직접 보지 않았다.
9. **`plateWidth` 목표 기준의 이중성**(goal 문구 0.2±0.02 vs 라이브 config 정본 0.215±0.015, §7.5)은 유닛에서 판정하지 않았으며, 이 문서에 정직하게 병기한다.

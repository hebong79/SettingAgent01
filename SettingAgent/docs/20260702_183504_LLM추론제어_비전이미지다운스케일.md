## 1. 배경 / 근본 원인

### 증상
`floorRoi.enabled=true`, `enabled=true` 상태로 정밀수집을 돌려도 **LLM 비전 추론이 전부 실패→결정형 폴백**만 동작했다. 로그 실측상 성공한 비전 추론은 **0건**이었고, floor ROI·centering 자문 호출이 30초 타임아웃을 초과했다.

### 근본 원인 (설계자 서버 실측으로 확정)
사용 모델 `gemma4:12b`는 **thinking(추론) 모델**이다. 비전 입력이 붙으면 추론이 폭주해 지연이 커진다(실측 `1280x720=22.8s`, `1920x1080=39.7s`, think ON 기준). 1080p 프레임이 타임아웃 주범이었다.

핵심 발견은 **"thinking을 어떻게 끄느냐"**였다. 기존 호출 경로인 OpenAI 호환 `/v1`(openai SDK)에서는 어떤 필드로도 gemma4:12b의 thinking을 끌 수 없음을 실호출로 확정했다.

| 경로 / 필드 | thinking | 결과 |
|---|---|---|
| `/v1` 기본 | ON | `content:""`, `reasoning`만 채워짐 |
| `/v1` + `chat_template_kwargs:{thinking:false}` | 여전히 ON | 무효 |
| `/v1` + `reasoning_effort:"low"` | 여전히 ON | 무효 |
| `/v1` + `think:false`(패스스루) | 여전히 ON | 무효 |
| **네이티브 `/api/chat` + `think:false`** | **OFF** | `content:"OK"` 정상, 0.78s |
| 네이티브 `/api/chat` + `think:false` + `format:"json"` | OFF | 정상 JSON, 0.91s |
| 네이티브 `/api/chat` + `think:false` + `images:[b64]`(비전) | OFF | 정상, 1.08s |

`/v1` thinking-ON 응답은 답을 별도 `reasoning` 필드에 넣고 `content`는 빈 문자열로 돌려준다. 이것이 로그의 "content 빈문자열=완전실패→폴백"의 정체였다.

**결론**: `/v1` 경로로는 근본 해결 불가. **Ollama 네이티브 `/api/chat` + `think:false`** 로 전환해야만 thinking이 실제로 꺼진다. 검증자 실서버 e2e에서 동일 바디로 **33.8s급 지연 → 2.44s e2e**로 회복됨을 확인했다.

부가로, 비전 프레임을 **종횡비 유지 균일 축소(긴변 960px 캡)**하여 비전 토큰·비용·타임아웃 여유를 추가 확보한다.

## 2. 변경 내용 요약

### (1) config 스키마 3필드 추가 — `src/config/llmConfig.ts`
`LlmSchema`에 옵셔널+기본값으로 3개 필드를 추가했다(전부 하위호환).

| 필드 | 타입/기본값 | 의미 |
|---|---|---|
| `api` | `enum('openai','ollama')`, 기본 `'openai'` | 전송 라우팅. `'ollama'`면 네이티브 `/api/chat`, `'openai'`면 기존 SDK `/v1`. |
| `think` | `boolean`, 기본 `false` | 전역 추론(thinking) 토글. **네이티브 경로에서만 유효.** |
| `imageMaxEdge` | `positive int`, 기본 `960` | 비전 이미지 다운스케일 긴변 상한(px). |

- `DEFAULT_LLM_CONFIG.llm`에도 `api:'openai', think:false, imageMaxEdge:960` 추가.
- 병합 로직(`{...DEFAULT.llm, ...raw.llm}`)은 미변경 → 신규 키가 없는 기존 config도 기본값으로 파싱된다(회귀 없음).
- `.default()`를 쓴 이유: `think`가 `undefined`로 남으면 네이티브 바디에서 누락되어 **모델 기본 thinking이 ON으로 회귀**하는 것을 방지하기 위한 방어적 선택.
- `api`를 `provider` enum과 별도로 둔 이유: 실제 config의 `provider`는 `'openai-compatible'`이라 문자열 매칭이 취약하다. 전송 방식을 `api` 단일 스위치로 명시 제어한다.

### (2) 다운스케일 유틸 — `src/util/image.ts` (신규)
`downscaleJpegBase64(b64: string, maxLongEdge: number): Promise<string>`

- sharp `resize({ width, height: maxLongEdge, fit:'inside', withoutEnlargement:true })` → JPEG(quality 80) → base64.
- **종횡비 유지 균일 축소만** 수행. 긴변이 상한 초과 시에만 축소, 이하면 원본 유지(**업스케일 없음**). **강제 16:9 스쿼시/크롭/레터박스 없음.**
- 좌표 불변 핵심: LLM이 반환하는 정규화 좌표(0~1)는 균일 스케일에 불변이다. 실제 카메라 프레임이 1080p(16:9)이므로 균일 축소 결과가 자연히 960×540(16:9)이 될 뿐, 강제 변형이 아니다.
- 디코드/인코드 오류 시 throw(순수 유틸, 테스트 용이) → 호출측이 원본 폴백 + warn 처리.

### (3) chat() 공통 진입점화 — `src/brain/AgentRuntime.ts`
`chat()`를 공통 진입점으로 재구성하고 전송을 두 private으로 분리했다.

- `chat(system, user, imageBase64?, json?)`
  - 이미지 존재 + `imageMaxEdge` 설정 시 **전송 직전** `downscaleJpegBase64` 적용. 실패 시 원본 사용 + warn.
  - `cfg.llm.api==='ollama'` → `chatNative()`, 아니면 `chatOpenai()` 라우팅.
- `chatNative()` (신규 private): Ollama 네이티브 `/api/chat` 전송.
  - endpoint = `baseUrl.replace(/\/v1\/?$/,'') + '/api/chat'` (warmup과 동일 패턴).
  - 바디: `{ model, stream:false, think:cfg.llm.think, format:'json'(json 모드시), options:{ temperature, num_predict:maxTokens }, keep_alive:warmup.keepAlive??'24h', messages:[{system},{user, images:[b64]?}] }`.
  - `fetchWithTimeout(..., timeoutMs??30000)`. 응답 `data.message.content` 반환(SDK `choices[]` 아님). 비200 → warn + null(결정형 폴백).
- `chatOpenai()` (신규 private): 기존 SDK `/v1/chat/completions` 경로를 로직 변경 없이 이관(이미지 `image_url` data URL, `response_format` json).
- `reviewSetup()`: 직접 SDK 호출을 제거하고 `chat(system, user)`로 위임 → `api:'ollama'`일 때 텍스트 보조 호출에도 네이티브 `think:false`가 적용되어 잔존 thinking을 제거한다.
- `chatJson()`·`ping()`·`warmup()` 및 단계별 메서드(judgePreset 등) 시그니처/로직은 불변.

### (4) 실값 갱신 — `config/llm.config.json`
`llm` 블록에 `"api":"ollama"`, `"think":false`, `"imageMaxEdge":960` 명시(model=`gemma4:12b`, baseUrl=`http://192.168.0.210:11434/v1`).

## 3. 신규 클래스/함수 설명

### downscaleJpegBase64 (신규, `src/util/image.ts`)
```
downscaleJpegBase64(b64: string, maxLongEdge: number): Promise<string>
```
- 입력: JPEG base64 문자열, 긴변 상한(px). 출력: 다운스케일된 JPEG base64.
- 동작: `fit:'inside'` + `withoutEnlargement:true`로 종횡비 유지 축소, 상한 이하면 무변형. quality 80 재인코딩.
- 실증(검증자): 1920×1080 → 960×540(16:9 유지, 오차<0.001%), 800×450 → 불변, 4:3 1200×900 → 960×720(4:3 유지, 스쿼시/크롭 없음).

### chatNative / chatOpenai (신규 private, `AgentRuntime`)
- 역할 분리: `chatNative`는 Ollama 네이티브(`think`·`format`·`options.num_predict`·`images:[b64]`·`message.content`), `chatOpenai`는 SDK(`response_format`·`image_url`·`choices[]`). `chat()`이 `api` 필드로 둘 중 하나에 위임.
- 다운스케일은 `chat()` 공통부 1곳에서만 수행 → 두 경로 모두 축소 이미지 사용, 비전 경로별 누락 없음.

### config 스키마 필드 의미
2절 (1) 표 참조. 요지: `api`=전송 스위치, `think`=추론 온/오프(네이티브 전용), `imageMaxEdge`=비전 다운스케일 상한.

## 4. 입출력 / 전송 계약

네이티브 경로(`api:'ollama'`)의 요청 바디 ↔ Ollama `/api/chat` 스키마 매핑(검증자 필드 단위 대조 통과):

| 계약 필드 | config | chatNative 바디 | Ollama 네이티브 |
|---|---|---|---|
| thinking 토글 | `llm.think` | `think`(boolean) | `think` |
| JSON 모드 | (json 인자) | `format:'json'` | `format` |
| 최대 토큰 | `llm.maxTokens` | `options.num_predict` | `options.num_predict` |
| 온도 | `llm.temperature` | `options.temperature` | `options.temperature` |
| 이미지 | `imageBase64`(축소 후) | `messages[user].images:[b64]` | `images: string[]` |
| keep_alive | `warmup.keepAlive` | `keep_alive` | `keep_alive` |
| 응답 파싱 | — | `data.message.content` | `message.content` |

주의: 네이티브는 `max_tokens→options.num_predict`, JSON 모드→`format:'json'`, 이미지→메시지의 `images:[base64]`(문자열 배열, `image_url` 아님)로 SDK와 매핑이 다르다.

## 7. 운영 주의

- **재시작 필요**: `config/llm.config.json`은 nodemon(src/ `.ts` 감시) 밖 → 편집 후 서버 재시작해야 반영(기존 동작).
- **thinking 켜기**: `llm.think:true`로 켤 수 있으나 **비전 태스크(floor ROI/centering)는 타임아웃 위험** — 텍스트 단계에만 유익. 현재는 전역 단일 토글이라 비전만 별도로 켜고 끄는 per-call 오버라이드는 미도입(과설계 회피, 필요 시 후속).
- **imageMaxEdge 조정 가이드**: 기본 960(→1080p가 960×540). 더 보수적으로 비용/지연을 줄이려면 768. 정확도 여유가 필요하면 상향. 종횡비 유지이므로 값과 무관하게 정규화 좌표는 불변.
- **think는 네이티브 전용**: `api:'openai'` 경로에서는 `think` 필드가 전송되지 않는다(SDK `/v1`은 어차피 무시). thinking 제어가 필요하면 `api:'ollama'`여야 한다.

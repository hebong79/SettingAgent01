# SettingAgent LLM 셋업 프롬프트 설계 — 3가지 전략 제안 및 최적안

- 작성일: 2026-06-24
- 전제: **SettingAgent 는 주차장 설치 시 1회성 단계** → 지연·비용 제약이 거의 없음 →
  **LLM(가능하면 비전 모델)을 적극 활용해 셋업 "품질"을 극대화**하는 것이 합리적.
- 관련: 아키텍처 §8(LLM 두뇌+MCP), SettingAgent 설계서 §3·§8-1, `brain/AgentRuntime.ts`(현재 review형).

---

## 0. 결론 요약 (TL;DR)

| 전략 | 한줄 요약 | 품질 | 검증성 | 권장 |
|------|-----------|------|--------|------|
| A. 사후 검증형 | 결정형 산출물(숫자)만 LLM이 점검 | 낮음 | 높음 | 보조 |
| B. 에이전트 주도형 | LLM이 MCP 도구로 셋업 전체를 자율 수행 | 높음(불안정) | 낮음 | 부분 |
| **C. 단계별 비전 게이트(하이브리드)** | 좌표는 검출이 보장, 판단·검증은 비전 LLM | **최상** | **높음** | ✅ **최적** |

> **핵심 원칙: LLM 은 "좌표를 만들지 않는다". 좌표(ROI bbox)는 VPD 검출이 보장하고,
> LLM 은 그 위에서 판단(유효 슬롯 여부·순서·중복·품질·라벨·재촬영 결정)만 한다.**
> 이렇게 해야 1회성 단계의 품질을 끌어올리면서도 좌표 환각(hallucination)을 원천 차단한다.

---

## 1. 셋업에서 LLM 이 기여할 수 있는 판단 지점

결정형 파이프라인(캡처→VPD→ROI→전역인덱스)이 못 하거나 약한 부분:

1. **오검출 필터** — 지나가는 차/프레임 가장자리 잘린 차/이중 검출을 "유효 주차 슬롯"에서 제외.
2. **슬롯 순서 확인** — 사람이 보는 순서(상→하/좌→우)와 자동 정렬이 일치하는지 시각 확인.
3. **프리셋 간 중복(겹침) 제거** — 같은 물리 주차면이 두 프리셋에 동시에 보일 때 전역 인덱스 중복 방지.
4. **품질·재촬영 판단** — 블러/역광/가림(occlusion)으로 검출이 불완전하면 재촬영 지시.
5. **기대값 교차검증 해석** — preset faceCount ≠ 검출 수의 "원인"을 추론(가림? 빈 면? 오검출?).
6. **존(zone)·라벨 부여, 설치 리포트 작성** — 한글 설명·신뢰도 포함 최종 보고.

---

## 2. 모델 선택

- **비전 가능 로컬 모델 권장**: Qwen3-VL / Qwen2.5-VL / Gemma 3(비전). 위 1·2·4 는 이미지 판단이 필요.
- 텍스트 전용 모델은 전략 A(숫자 리뷰)와 전략 C의 3단계(리포트)만 가능.
- 1회성이므로 **큰 모델·낮은 temperature(0~0.2)·충분한 max_tokens** 사용 권장.
- 모든 출력은 **구조화 JSON(스키마 강제)** — MCP StructuredOutput 또는 zod 검증으로 형식 보장.

---

## 3. 전략별 프롬프트 (구체안)

### 전략 A — 사후 검증형 (Review-only)

**구성**: 결정형 파이프라인이 산출물을 만든 뒤, 요약(숫자)만 1회 LLM 검토. (현 `reviewSetup` 확장)

**System**
```
너는 주차장 셋업 산출물의 무결성을 점검하는 검수자다.
입력은 카메라/프리셋/슬롯/전역인덱스 요약(숫자)이다. 이미지는 보지 못한다.
다음을 점검해 한국어로 간결히 보고하라: (1) 프리셋별 슬롯 수 불균형/이상,
(2) 기대 슬롯 수(faceCount)와 검출 수 불일치, (3) 전역 인덱스 누락/중복 징후.
반드시 아래 JSON 스키마로만 답하라.
```
**User (템플릿)**
```
[셋업 요약]
presets: {{presetsSummary}}   // [{key:"1:1", detected:2, expected:3}, ...]
totalSlots: {{totalSlots}}
globalIndex: {{globalCount}}
warnings: {{warnings}}
```
**출력 스키마**
```json
{ "ok": true,
  "issues": [{"severity":"warn|error","preset":"1:1","message":"..."}],
  "recommendation": "재촬영/수동확인/통과 중 권고" }
```
- 장점: 단순·안전·결정성 유지. 단점: 이미지 미사용 → 오검출/순서/가림을 못 잡음(품질 한계).

---

### 전략 B — 에이전트 주도형 (Agentic tool-loop)

**구성**: LLM 에게 MCP 도구(`camera_req_img`, `vpd_detect`)와 목표를 주고 **셋업 전체를 자율 수행**.

**System**
```
너는 주차장 셋업 에이전트다. 목표: 모든 카메라/프리셋을 순회하여 각 주차면의 ROI 와
전역 슬롯 인덱스를 확정한다. 사용 가능한 도구: camera_req_img(cam,preset)->이미지,
vpd_detect(image)->차량 bbox 목록. 규칙:
- ROI 좌표는 반드시 vpd_detect 결과 bbox 에서만 가져온다(좌표를 임의로 만들지 마라).
- 검출이 불완전하면 같은 프리셋을 재촬영해 누적하라.
- 끝나면 슬롯/전역인덱스를 구조화 JSON 으로 제출하라.
```
**User**
```
대상 프리셋: {{targets}}   // [{cam,preset,ptz}]
기대 슬롯 수(있으면): {{expectedFaces}}
```
- 장점: LLM 활용 극대화, 적응적(재촬영·예외처리). 단점: **비결정적·검증 어려움**,
  좌표 규칙을 어기면 환각 위험, 디버깅 난해. → 가드레일(좌표는 도구결과 인덱스로만 참조) 필수.

---

### 전략 C — 단계별 비전 게이트 (Hybrid, 권장) ✅

**구성**: 기계적 작업(캡처·VPD·ROI·정렬)은 결정형이 수행하고, **정의된 게이트에서 비전 LLM 이 판단**.
LLM 은 좌표를 만들지 않고, 검출 결과의 **인덱스 선택·판정·라벨·재촬영 결정**만 반환한다.

#### 1단계: 프리셋별 비전 판정 (이미지 + 오버레이된 검출 박스)
**System**
```
너는 주차장 한 프리셋 이미지를 검수하는 비전 검수자다.
이미지에는 VPD 가 검출한 차량 박스에 번호(1..N)가 표시되어 있다.
각 번호 박스가 "유효한 주차면 위의 차량"인지 판정하라. 다음을 구분하라:
- 유효: 주차면에 정상 주차된 차량
- 제외: 통행 중/프레임에 잘린/중복/주차면이 아닌 검출
또한 사람이 보는 순서(위→아래 행, 같은 행은 좌→우)가 번호 순서와 맞는지,
이미지 품질(블러/역광/가림)로 재촬영이 필요한지 판단하라.
좌표를 만들지 말고, 오직 박스 번호로만 지칭하라. JSON 스키마로만 답하라.
```
**User**
```
camera={{cam}} preset={{preset}} 검출박스수={{n}} 기대슬롯수={{expected|미상}}
(이미지 첨부: 번호 오버레이)
```
**출력 스키마**
```json
{ "validBoxes": [1,2,4],
  "excluded": [{"box":3,"reason":"프레임에 잘린 차량"}],
  "orderOk": true,
  "reorder": [1,2,4],          // orderOk=false 일 때 올바른 박스 순서
  "rescan": {"needed": false, "reason": ""},
  "confidence": 0.0 }
```
→ 결정형은 `validBoxes`/`reorder` 로 슬롯 집합·순서를 확정(좌표는 그대로 검출 bbox 사용).
   `rescan.needed` 면 누적 클러스터링 모드로 재촬영.

#### 2단계: 프리셋 간 중복 제거 + 존/라벨
**System**
```
너는 여러 프리셋의 주차면을 통합하는 매핑 검수자다. 같은 물리 주차면이 둘 이상의
프리셋에 중복으로 잡혔는지 판단해 하나로 합치고, 각 슬롯에 존(zone) 라벨을 제안하라.
좌표는 만들지 말고 슬롯ID로만 지칭하라. 전역 인덱스 정렬은 cam→preset→위치 규칙을 따른다.
```
**User**
```
프리셋별 슬롯: {{slotsByPreset}}   // [{key,slotIds:[...],thumbnails?}]
인접/중첩 프리셋 힌트(PTZ): {{ptzAdjacency}}
```
**출력 스키마**
```json
{ "duplicates": [["c1p1s2","c1p2s1"]],   // 동일 물리면으로 합칠 그룹
  "zoneLabels": {"c1p1s1":"A-01"},
  "notes": "..." }
```

#### 3단계: 최종 검증 + 설치 리포트(한글)
**System**
```
너는 셋업 결과를 최종 승인하고 설치 리포트를 쓰는 책임자다. 기대 슬롯 수와 최종 슬롯 수를
대조하고, 잔여 위험(가림·중복 의심·저신뢰 프리셋)을 정리해 한국어 리포트를 작성하라.
JSON 스키마로만 답하라.
```
**출력 스키마**
```json
{ "approved": true,
  "totalSlots": 0, "globalCount": 0,
  "mismatches": [{"preset":"1:2","expected":3,"final":2,"likelyCause":"가림"}],
  "report_ko": "한글 설치 리포트 ...",
  "confidence": 0.0 }
```

- 장점: **최고 품질**(비전 판단) + **검증성 유지**(좌표는 검출이 보장, LLM은 결정만), 게이트별 산출물 추적 가능.
- 단점: 프롬프트 3개·비전 모델 필요. (1회성이라 비용/시간 부담은 작음 → 수용)

## 4. 전략 비교·평가

| 기준(가중) | A 사후검증 | B 에이전트주도 | C 단계별 비전게이트 |
|------------|-----------|----------------|----------------------|
| 셋업 품질(★★★) | △ 숫자만 | ○ 높으나 불안정 | ◎ 비전 판단 |
| 검증성/재현성(★★★) | ◎ | △ 비결정 | ◎ 게이트별 검증 |
| 좌표 환각 위험(★★★) | ◎ 없음 | △ 규칙 위반 시 | ◎ 구조적 차단 |
| 오검출/순서/가림 처리(★★) | ✕ | ○ | ◎ |
| 프리셋 중복 제거(★★) | ✕ | ○ | ◎(2단계) |
| 구현 복잡도(★) | ◎ 단순 | △ | ○ 보통 |
| 1회성 비용 적합성 | ◎ | ○ | ◎(비용 비제약) |

→ **최적: 전략 C(단계별 비전 게이트 하이브리드).**
   1회성 단계의 목표는 "속도"가 아니라 "정확한 기준 데이터 확보"이므로, 비전 LLM 을 게이트에
   적극 투입하되 좌표는 검출이 보장하는 C 가 품질·안전을 동시에 만족한다.
   전략 A 는 C 의 3단계로 흡수되고, 전략 B 의 적응성(재촬영)은 C 의 1단계 `rescan` 으로 안전하게 수용.

## 5. 최적안(C) 운영 파라미터·가드레일

- 모델: 비전 로컬 모델(Qwen3-VL/Gemma3), temperature 0~0.2, 충분한 max_tokens.
- **좌표 불생성 원칙**: LLM 입력에 박스 번호 오버레이 이미지 + 검출 메타, 출력은 번호/슬롯ID/결정만.
- **구조화 출력 강제**: 각 단계 스키마를 zod/MCP StructuredOutput 으로 검증, 실패 시 재요청.
- **결정형 권위 유지**: 최종 `setup_artifact.json` 은 결정형이 기록(LLM 결정 반영분 포함), DM/Action 계약 불변.
- **재현 로그**: 단계별 프롬프트·응답·반영내역을 셋업 로그로 저장(설치 감사 추적).
- **실패 안전**: LLM 미가동(`llm.enabled=false`) 시 결정형 단독 경로로 폴백(현재 동작 유지).

---

## 6. 코드 반영 포인트 (후속 구현 시)

- `brain/AgentRuntime.ts`:
  - `reviewSetup()` = 전략 A(이미 존재) → 3단계 리포트로 승격.
  - 신설: `judgePreset(image, detections, expected)`(1단계), `dedupeAndLabel(slotsByPreset)`(2단계).
- `setup/SetupOrchestrator.ts`: 각 게이트 호출 지점 추가(LLM 활성 시), 좌표는 기존 검출 사용.
- MCP: 비전 입력을 위해 이미지(base64) 전달 도구/리소스 추가 검토.
- 모든 LLM 호출은 `llm.config.json` 의 `enabled` 로 on/off (1회성 셋업에서만 켜는 운용 권장).

> 본 문서는 "프롬프트 설계·선정"까지가 범위이며, 실제 코드 반영은 별도 후속 차수에서 유닛/통합 테스트와 함께 진행한다.

## 2. 근본 원인 (2겹) — 재분석 근거

### 2-1. 원인 1 — 앵커 수직 오프셋

discovery 앵커(`DiscoveryTarget.anchor`)는 기존에 `slot3dFrontCenter`(육면체 앞면 4모서리 평균, `Finalizer.ts:252` `slotFrontCenter(sp.points, model, H_CONST)`, `H_CONST=1.5`)를 그대로 사용했다. 이 점의 실효 지상 높이는 `H_CONST/2 = 0.75m`이지만, 실제 번호판은 지상 0.3~0.5m에 있다. 하향 틸트 카메라에서는 높이가 낮을수록 화면 y가 커지므로, 번호판은 앵커보다 항상 화면 아래쪽에 맺힌다 — 즉 discovery는 애초에 번호판이 아닌 지점을 중심으로 탐색하고 있었다.

### 2-2. 원인 2 — 크롭이 앵커에 고정된 채 축소만 함

기존 Tier1 루프는 매 스텝 `computeCropWindow(anchor, frac, aspect)`를 호출했는데, 앵커(중심)가 고정된 채 `frac`(창 폭 비율)만 `frac0·shrink^(k-1)`로 줄어드는 구조였다. 창이 작아질수록 앵커 아래쪽에 있는 번호판은 오히려 창 밖으로 벗어난다 — 확대가 검지율에 역효과를 냈다.

### 2-3. (라이브에서 추가 발견) 옆판 절도 — 위장 found

1차 배포 후 리더 라이브 검증(`05_live_finding.md`)에서 `found:6/6`이 관찰되었으나, `/discover/result`의 bbox 중심을 대조한 결과 **고유 번호판 3개를 6슬롯이 중복 점유**하고 있음이 드러났다(실검지 3/6, 위장 3건). 원인은 `matchRadiusNorm=0.15`가 인접 슬롯 하향앵커 간격(~0.11~0.13)보다 커서, 자기 판이 검출되지 않은 슬롯이 Tier0에서 이웃 판을 반경 내로 채택했기 때문이다. 이 위장 채택이 격자 줌 루프 진입 자체를 막아, 애초에 A/B 개선이 발동하지 않는 구조적 결함이었다.

## 3. 해결 A — 앵커 하향 (`lowerFrontAnchor`, `plateDiscoveryWriter.ts`)

```ts
export function lowerFrontAnchor(
  roi: NormalizedPoint[],            // v.roi (바닥 quad, 정규화)
  frontCenter: NormalizedPoint,      // v.slot3dFrontCenter (h=0.75 등가점)
  plateH = PLATE_H,                  // 0.4
): NormalizedPoint
```

- 육면체 앞 edge(roi 4 edge 중 두 끝점 y평균이 최대인 edge — `project.ts`의 `frontFaceCornerIdx` 판정과 동일 기준)의 중점 `B`(h=0 등가)와 `frontCenter`(h=0.75 등가, `slot3dFrontCenter`)를 `t = plateH/(H_CONST/2) ≈ 0.5333`로 선형보간해 번호판 높이(h=0.4) 등가점을 산출한다.
- **수식 근거**: `project.ts`의 원근 투영식은 `p(h) = p(0) − h·(s₀/d)·kn` 형태로, 광축이 지면과 평행(`n_z=0`)한 이상 조건에서는 픽셀이 높이 h에 대해 엄밀히 선형이다. 이 조건에서 `lowerFrontAnchor`(정규화-보간)는 GroundModel 재투영(`projectPointAtHeight`)과 수학적으로 항등이다.
- **QA 정정(중요)**: 설계서 §2-1은 "근사가 아니라 항등"이라 주장했으나, qa-tester 실측(V-8, `project.ts` 실헬퍼로 합성 검증)에서 **실제 원근분모(`n_z≠0`, 틸트 카메라)에서는 완전한 항등이 아님**이 확인됐다. 틸트 25°에서 정규화 편차 ≈1.4e-3(1080px 기준 ≈1.5px, sub-pixel), 틸트 10°에서 ≈6.3e-4. 선형투영 영역(`n_z=0`)에서만 편차 2.2e-16(기계 오차)로 엄밀 항등이다. 실무 영향은 무시 가능하다 — 편차가 sub-pixel이고, `PLATE_H=0.4` 자체의 눈대중 오차(0.3~0.5 범위)가 이보다 훨씬 크며, 앵커는 LPD 최근접 게이트·크롭 중심 용도라 sub-pixel 오차는 검지율에 영향을 주지 않는다.
- 폴백: `roi.length !== 4` 또는 좌표 비유한(non-finite) → `frontCenter` 그대로 반환(throw 금지, 기존 동작 유지).
- 모듈 상수: `PLATE_H = 0.4`(초기값, 튜닝 대상), `H_CONST = 1.5`(**`Finalizer.ts:41`과 동일값 — private이라 import 불가, 값 변경 시 양쪽 수동 동기 필요**, 주석으로 상호참조 봉인).
- 산출 위치: `expandDiscoveryTargets`(expand 단계) 1줄 교체 — `anchor: lowerFrontAnchor(v.roi, v.slot3dFrontCenter)`. `slot3dFrontCenter` DB 값 자체는 변경하지 않으며, 하향은 discovery expand 내부에만 국한된다.

## 4. 해결 B — 2D 격자 크롭 탐색 (`gridCenter` + `discoverSlot` Tier1, 30회)

### 4-1. `gridCenter` (`cropZoom.ts`, 신규 순수함수)

```ts
export function gridCenter(anchor: NormalizedPoint, frac: number, aspect: number, off: { dx: number; dy: number }): NormalizedPoint
```
`{ x: anchor.x + off.dx·min(1,frac), y: anchor.y + off.dy·min(1,frac·aspect) }` — 오프셋 단위는 창 크기 배수(`computeCropWindow`와 동일 창 크기 정의)이므로 줌이 깊어질수록 이동량이 자동으로 축소된다. 기존 4함수(`computeCropWindow`/`toCropPoint`/`backmapQuad`/`cropAndUpscale`)는 무변경.

### 4-2. `GRID_OFFSETS`(`plateDiscovery.ts`, 6방)

```ts
const GRID_OFFSETS = [
  { dx: 0,    dy: 0   },  // 1. 중심(하향앵커)
  { dx: 0,    dy: 0.5 },  // 2. 하
  { dx: -0.5, dy: 0.5 },  // 3. 하좌
  { dx: 0.5,  dy: 0.5 },  // 4. 하우
  { dx: -0.5, dy: 0   },  // 5. 좌
  { dx: 0.5,  dy: 0   },  // 6. 우
] as const;
```
0.5배 이동 = 인접 창과 50% 겹침(경계 걸친 번호판 누락 방지). `dy=1.0`(구 5방안의 "더아래" 오프셋)은 하향앵커 적용 후 과이동이 되어 제거하고, 순수 좌/우 쌍으로 대체해 측방 산포를 커버한다.

### 4-3. 상태전이 (`discoverSlot` Tier1, 30회 = 5줌레벨 × 6오프셋)

```
level = floor((k-1)/6)+1  →  frac = frac0·shrink^(level-1)   (frac0=0.40, shrink=0.6)
  level 1~5: frac ≈ 0.40 / 0.24 / 0.144 / 0.0864 / 0.05184  (전부 minFrac 0.05 이상 → 30회 전부 유효)
off = GRID_OFFSETS[(k-1) % 6]
c = gridCenter(anchor, frac, aspect, off) → W = computeCropWindow(c, frac, aspect)
동일 창(x,y,w,h) 재등장 시 LPD 스킵(중복창 가드, k는 계속 증가)
검출 시 즉시 backmapQuad(pick.quad, W)로 원본좌표 역계산·반환(tier:'crop', step:k)
30회 소진 → found:false, reason:'no_plate', step:30 (실패 확정, 다음 슬롯 진행)
```
`maxSteps` 기본값 5 → 30. `backmapQuad`는 창의 offset/size만 사용하는 아핀(`orig = W.xy + q·W.wh`)이라 오프셋·클램프와 무관하게 역계산이 정확하다.

**개발 이력 메모**: 최초 구현은 5방×4줌=20칸이었으나, 마스터의 "최대 30회" 요건에 맞춰 6방(좌/우 추가)×5줌=30칸으로 재설계·재구현했다(`02_developer_changes.md` 개정 기록). 최종 코드는 30칸 버전이다.

## 5. 해결 §9 — 배타성 게이트 (`pickOwnedPlate`, Voronoi 소유권)

라이브에서 발견된 "위장 found"(§2-3)를 차단하기 위해 리더 재분석 후 추가한 3번째 단계.

### 5-1. 원칙

검출된 번호판은 **"검출 중심(원본 프레임 정규화 좌표)에 대해 동일 프리셋 내 전체 슬롯 하향앵커 중 자기 앵커가 최근접일 때만"** 자기 것으로 채택한다. 이웃 앵커가 더 가깝거나 동률이면 기각하고, 해당 tier/step은 미검출로 간주해 다음 격자 스텝으로 계속 진행한다.

### 5-2. `pickOwnedPlate` (`plateDiscovery.ts`, 신규 export)

```ts
export function pickOwnedPlate(
  candidates: Array<{ plate: PlateBox; centerOrig: NormalizedPoint }>,
  selfAnchor: NormalizedPoint,
  peerAnchors: readonly NormalizedPoint[],
): PlateBox | null
```
모든 `peer`에 대해 `dSelf < dPeer`(엄격 부등호, 동률 기각)를 만족하는 후보만 남기고, 그중 `selfAnchor` 최근접 1개를 반환한다. `peerAnchors=[]`이면 전원 통과(기존 최근접 동작과 동일 — 하위호환).

- **비교 좌표계는 항상 원본 프레임 정규화**로 통일한다. Tier0(full)은 검출 중심을 그대로 사용하고, crop tier는 크롭 정규화 중심을 `centerOrig = { x: W.x + cc.x·W.w, y: W.y + cc.y·W.h }`(`backmapQuad`와 동일 아핀의 점 버전)로 원본 좌표 환산 후 비교한다. 크롭 좌표를 직접 비교하면 w/h 비등방 스케일로 거리 비교가 왜곡되므로 이를 피했다.
- Tier0에서는 `pickOwnedPlate` 통과 후 **기존 `matchRadiusNorm` 게이트를 병행 유지**한다(둘 다 통과해야 full 채택). crop tier에서는 기존 `pickNearestPlate` 호출을 `pickOwnedPlate`로 교체했다.

### 5-3. 시그니처 확장 (`discoverSlot`, 하위호환)

```ts
discoverSlot(t: DiscoveryTarget, presetPtz?: Ptz | null, peerAnchors: NormalizedPoint[] = []): Promise<PlateDiscoveryItem>
```
3번째 인자를 옵셔널 배열로 추가했다(기본 `[]` = 기존 동작). `PlateDiscoveryApi = Pick<PlateDiscovery, 'discoverSlot'>` 테스트 시임은 옵셔널 인자를 자동 추종하므로 무수정.

### 5-4. peer 앵커 주입 (`PlateDiscoveryJob.run`)

`run(targets)` 진입 시 1회 `Map<"${camIdx}:${presetIdx}", DiscoveryTarget[]>`로 프리셋별 그룹핑하고, 슬롯 처리 시 `peerAnchors = 같은 프리셋 targets 중 자기(slotId) 제외 · anchor != null인 앵커들`을 `discoverSlot`에 전달한다(`PlateDiscoveryJob.ts:136-152`).

### 5-5. 고아 제거 (developer 변경으로 미사용화)

`plateDiscovery.ts`에서 `pickNearestPlate`(controlMath) import, `toCropPoint`(cropZoom) import, 로컬 `priorRect` const를 제거했다. `pickNearestPlate` 함수 자체(`controlMath.ts`)는 무수정이며, `platePtz.ts`·`detectPipeline.ts`가 계속 사용하므로 그쪽 동작에는 영향이 없다.

## 6. 라이브 최종 결과 (정직 보고)

### 6-1. 배타성 적용 전 (§2-3 위장 발견)

`found:6/6`이었으나 bbox 중심 대조 결과 고유 번호판 3개(0.212/0.463/0.766)를 6슬롯이 중복 점유 — 실검지 3/6.

### 6-2. 배타성 적용 후 (3회 재실행 일관)

| slot | 검지 | tier/step | 판정 |
|---|---|---|---|
| 8 | ✅ | crop/1~12 | 자기판 회수(이전엔 slot9 절도) |
| 9 | ✅ | full/0 | 자기 |
| 10 | ❌ | crop/30 no_plate | 미검(30회 소진, 정직 실패) |
| 11 | ✅ | crop/1 | 자기 |
| 12 | ✅ | crop/25 | 자기판 회수(이전엔 slot11 절도) |
| 13 | ✅ | crop/1 | 자기 |

→ **고유 검지 5개, 중복 점유 0.** 배타성 게이트가 절도(위장)를 완전히 차단하고, slot8/12가 격자 줌으로 자기 판을 회수했다. 실검지 3/6 → 정직 5/6.

**found 수치가 "6/6 → 5/6"으로 하락한 것은 회귀가 아니라 위장의 교정이다.** 이전 6/6은 3개의 중복 절도를 포함한 거짓 수치였고, 5/6은 중복 점유 0인 정직한 수치다. 이번 개선의 성공 기준은 "found 수치 유지"가 아니라 "중복 점유 0 + 최대 검지율"이다.

### 6-3. slot10 미검 — 외부 LPD 서비스 한계 (파이프라인 결함 아님)

slot10 판("221가5490")은 원본 좌표(≈0.339,0.596)가 슬롯 하향앵커(0.324,0.606)와 거의 일치하고 육안으로도 크고 선명하다. 그러나 리더가 직접 스윕한 15개 이상의 크롭 구성(정사각 native~0.06 축소, 가로넓은 비율, 전처리 7종: 업스케일/샤프닝/정규화/그레이스케일/선형보간)에서 LPD 서비스가 이 판을 한 번도 검출하지 못했다 — 검출되는 것은 언제나 이웃 판(0.212·0.463)뿐이었다. 파이프라인은 정확히 줌해 갖다 바쳤으나 LPD 모델이 이 특정 렌더 판을 못 읽는 한계로 판단된다.

**마스터 결정: 정직 5/6 수용.** 기하 폴백(미검 슬롯에 추정 박스 저장)은 도입하지 않았다 — "LPD 검지"의 의미가 달라지는 변경이라 이번 범위에서 배제. slot10은 LPD 서비스팀 이관 대상으로 남긴다. 핸드오프 문서 §5("순수 비전 100% 불가")의 구체 사례로 기록.

- **V-7·V-8**: `lowerFrontAnchor` 보간값 수치 일치(t≈0.5333) + 선형영역 항등(<1e-9)/틸트영역 sub-pixel 편차(0~3e-3) 정량 봉인.
- **V-10~V-14(§9 증분)**: `pickOwnedPlate` 순수(이웃 기각/자기 채택/다수 중 최근접/동률 기각/`peers=[]` 무조건 통과), **절도 재현 회귀**(slot8 시나리오 재현 → 격자 진입 → 자기판 검출로 봉인), 크롭 내 자기+이웃 공존 시 자기판 채택, `PlateDiscoveryJob.run` 프리셋별 peer 그룹핑 격리 확인, 옵셔널 인자 하위호환 전수 통과.

### 발견 이슈 (qa-tester 보고, 그대로 인용)

1. **설계 정확성(경미, advisory)**: 설계서 §2-1의 "항등" 주장은 선형투영 영역(`n_z=0`) 한정이며, 실 틸트 카메라에서는 sub-pixel 근사다(§3 참조). 구현 결함이 아니라 설계 문구의 정밀도 표현이 과했던 것으로, 실무 영향은 무시 가능.

3. **found 수치의 정직한 하락(예고, 회귀 아님)**: §6-2에 기술한 대로 위장 교정.

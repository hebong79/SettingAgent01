- 변경 방식: config + 서버 재시작(런타임 전환 아님). 디폴트 = 시뮬레이터

## 1. 목적

SettingAgent 뷰어의 라이브 카메라 소스를 **설정(`tools.config.json`)으로 선택**할 수 있게 한다.

- **시뮬레이터(기본)**: `CameraposSource`(id=`rpc`). Unity JSON-RPC 서버(13110) + `camerapos.json` 프리셋 기반. **현행 동작 그대로**.
- **리얼**: `RealPtzSource`. Hucoms CGI 어댑터로 실 PTZ 카메라(HNR-2036LA류)에 접속.

카메라 1대 현황을 반영해 과설계를 피하고, 리얼은 실기기 미확인 스텁이므로 "선택 가능하게만" 배선했다(로직 무변경).

## 2. 구현

### 2-1. config 스키마 (`src/config/toolsConfig.ts`)

`ToolsConfigSchema` 에 2개 필드 추가:

```ts
// 뷰어 카메라 소스 선택. cameraSources(다중/고급) 미설정 시 이 값으로 단일 소스를 구성.
// precedence: cameraSources(명시·길이>0) > cameraMode.
cameraMode: z.enum(['simulator', 'real']).default('simulator'),
// 리얼(Hucoms) 카메라 접속정보. cameraMode='real' 일 때 필요(자격증명 미포함 — UI 세션).
realCamera: CameraSourceConfigSchema.optional(),
```

- `DEFAULT_TOOLS_CONFIG.cameraMode = 'simulator'`. `realCamera` 는 DEFAULT 미포함(옵셔널).
- `realCamera` 는 기존 `CameraSourceConfig`(`{id, kind, host?, port?, loginPath?, snapshotUrl?, ptz?}`) 재사용. 자격증명(비밀번호 등)은 config 에 두지 않고 UI 세션(`RealPtzSource.login()`)로 통과.
- 잘못된 값(예 `"hucoms"`, `"bogus"`)은 zod enum 파싱에서 거부(fail-fast).

### 2-2. loadToolsConfig 스칼라 병합 가드 (버그 수정 · 회귀 방지)

기존 `loadToolsConfig` 는 모든 섹션을 `{...DEFAULT[key], ...raw[key]}`(객체 스프레드)로 병합했다. 스칼라 값 `cameraMode`('simulator')를 이 스프레드에 넣으면 **문자-인덱스 객체**(`{0:'s',1:'i',...}`)로 깨져 zod 파싱이 실패한다.

수정: `def` 가 **객체(비배열)** 일 때만 스프레드, 스칼라는 값으로 대입.

```ts
const def = DEFAULT_TOOLS_CONFIG[key];
if (def !== null && typeof def === 'object' && !Array.isArray(def)) {
  merged[key] = { ...(def as Record<string, unknown>), ...((raw[key] as Record<string, unknown>) ?? {}) };
} else {
  merged[key] = raw[key] ?? def;
}
```

- 기존 섹션은 전부 객체 → 병합 동작 불변(무회귀).
- `raw` 타입을 `Record<string, unknown>` 로 완화(스칼라 값 수용).
- `realCamera` 는 `cameraSources` 와 동일 특례: DEFAULT 에 없으므로 위 순회에서 누락 → `if (raw.realCamera !== undefined) merged.realCamera = raw.realCamera;` 로 명시 통과.

> 이 가드는 앞으로 추가될 모든 스칼라 top-level 설정에도 필요한 일반 수정이다. `cameraMode` 가 첫 스칼라 top-level 필드라 이 문제가 처음 드러났다.

### 2-3. buildSourceRegistry 3분기 (`src/viewer/sourceRegistry.ts`)

입력 `Pick` 에 `cameraMode`·`realCamera` 추가. `index.ts` 는 `buildSourceRegistry(tools)` 로 전체 tools 를 전달하므로 **호출부 무변경**(시그니처만 확장).

```ts
export function buildSourceRegistry(
  cfg: Pick<ToolsConfig, 'camera' | 'cameraSources' | 'unityRpc' | 'map' | 'cameraMode' | 'realCamera'>,
): Map<string, CameraSource>
```

분기(precedence: **cameraSources > cameraMode**):

1. **(A) 고급/다중** — `cfg.cameraSources?.length > 0` → 기존 항목별 `sim`/`hucoms` 경로 그대로. `cameraMode` 무시(하위호환 보존).
2. **(B) real** — `realCamera` 없으면 `throw new Error('리얼 카메라(realCamera) 설정이 없습니다')`(fail-fast). 있으면 `{...realCamera, kind:'hucoms'}` 강제 후 `RealPtzSource(rc, cfg.camera.imageTimeoutMs)` 등록(Map 키 = `realCamera.id`).
3. **(simulator, 기본)** — 현재 폴백: `CameraposSource(id='rpc')` = `CRpcClient`(unityRpc) + `RpcCameraSource` + `CameraClient` 합성. **현행 동작과 완전 동일**.

**설계 결정 근거**
- **precedence**: cameraSources 는 다중/고급용으로 보존(기존 테스트·동작 무손상). 미설정 시에만 cameraMode 가 단일 소스를 결정.
- **simulator id='rpc' 유지**: 프론트 `sel-source` 드롭다운·기존 테스트·camerapos 시맨틱 하위호환. 재명명 안 함(외과적).
- **real 미설정 시 throw**: 리얼은 opt-in·미검증. 조용한 폴백 대신 기동 시점 명확한 config 에러로 오설정을 드러냄.

### 2-4. config 파일 (`config/tools.config.json`)

`"cameraMode": "simulator"` 추가(디폴트 가시화). `realCamera` 는 미기재(리얼 전환 시 사용자가 추가).

### 2-5. 프론트(sel-source) — 코드 변경 없음

`sel-source` 드롭다운은 `/viewer/api/health` 의 `sources` 키(= registry `keys()`)로 자동 구성된다. simulator=`['rpc']`, real=`[realCamera.id]` 로 단일 소스라 첫 항목 자동 선택 → 동작 유지. 웹 토글은 이번 범위 밖(config+재시작 방식).

## 3. 입출력 · 계약

- **config → 스키마**: `tools.config.json` 의 `cameraMode`(스칼라)/`realCamera`(객체)가 `loadToolsConfig` 병합 후 `ToolsConfigSchema.parse` 를 통과.
- **registry → health/프론트**: `buildSourceRegistry` 가 반환한 `Map` 의 `keys()` 가 `GET /viewer/api/health` 의 `sources` 배열이 되고, 프론트가 `sel-source` 드롭다운·`state.source = data.sources[0]` 로 소비. (REST 계약 자체는 변경 없음 — 값만 모드에 따라 달라짐.)
- **realCamera → RealPtzSource**: `{...realCamera, kind:'hucoms'}` 를 생성자에 전달, `imageTimeoutMs` 는 `camera.imageTimeoutMs` 재사용.

## 5. 사용법

리얼 카메라로 전환하려면 `config/tools.config.json` 에 다음을 설정하고 **서버를 재시작**한다.

```json
"cameraMode": "real",
"realCamera": { "id": "real", "kind": "hucoms", "host": "192.168.0.153", "port": 80 }
```

- 스키마상 `kind` 필수 — 파싱 시 `"hucoms"` 명시 필요(sourceRegistry 가 내부에서 다시 `hucoms` 로 강제).
- 자격증명(로그인 계정/비밀번호)은 config 가 아니라 뷰어 UI 세션(`login()`)으로 통과한다.
- 되돌리려면 `"cameraMode": "simulator"` 로 바꾸고 재시작(또는 `realCamera` 제거).
- config 변경은 `nodemon`(src 만 감시) 대상이 아니므로 **반드시 서버 재시작**해야 반영된다.

## 6. 한계 (중요 · 은닉 없이)

- **cameraMode 는 뷰어 라이브 소스만 전환한다.** 정밀수집(`capture`)·검출(`detect`)·캘리브레이션(`calibrate`)·파이널라이즈(`finalize`) 경로는 `index.ts` 에서 주입되는 `RpcCameraClient`(Unity RPC 13110)를 계속 사용하며, cameraMode 와 무관하다.
- 따라서 **cameraMode='real' 이어도 정밀수집/검출 파이프라인은 여전히 Unity RPC(13110)를 필요로 한다.** 리얼 카메라(RealPtzSource, CGI)로 자동 전환되지 않는다 → **실기 정밀수집 미지원(후속 과제)**.
- `RealPtzSource` 는 실기기(192.168.0.153, HNR-2036LA) 미확인 스텁(경로/PTZ 범위 가정값). 이번 변경은 이 소스를 **선택 가능하게만** 했고 로직은 손대지 않았다.

---

## 7. 하위호환

- 기존 config(cameraMode 없음) → default `simulator` → `CameraposSource(id='rpc')` → **현행 동작 동일**.
- `cameraSources` 를 명시하던 경로 → precedence 로 그대로 우선 → **무회귀**(기존 다중 소스 테스트·동작 유지).
- 병합 가드 추가는 기존 객체 섹션 병합을 바꾸지 않음(QA 회귀 테스트로 보증). 기존 배포·설정 파일 무수정 시 동작 동일.

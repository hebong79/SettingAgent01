## 3. 신규 클래스·함수 상세

### 3-1. `GroundModel` — 2~5단계가 합류할 인터페이스

지면을 **세 값**으로 규정한다. `H`(3×3 호모그래피)가 아니다.

```ts
// src/ground/types.ts:27
export interface GroundModel {
  camIdx, presetIdx, imgW, imgH, zoom;
  f: number;                     // 초점거리(px)
  n: [number, number, number];   // 지면 하향 단위법선(카메라 좌표)
  d: number;                     // 카메라 지상고(m) — metric 스케일의 유일한 담지자
  tiltDeg: number;               // asin(n.z) — 이미지 점만으로 추정
  // ── 정합 지표 4종(§5) ──
  ptzTiltDeg: number | null;     // 카메라 보고 PTZ tilt(실카메라도 주는 값)
  tiltErrDeg: number | null;     // ① 세로 어긋남 = 추정 tilt − PTZ tilt        (임계 1.0°)
  metricErr: number;             // ② 이미지 평행이동 = metric 재구성 상대오차   (임계 0.8%)
  slotBearingDeg: number | null; // 슬롯 스트립 방위(mod 90) — 프리셋 불변량
  bearingDevDeg: number | null;  // ③ 수직축 회전 = 방위의 프리셋 간 편차        (임계 8°)
  dDevRel: number | null;        // ④ 균일스케일 = 카메라고 d 의 프리셋 간 편차   (임계 10%)
  depthEdgePx, conf;
  source: 'file' | 'auto';       // ★ 1단계는 'file'. 2~5단계 자동경로가 'auto' 로 여기 합류
  issues: string[];              // advisory(강등 사유) — throw 대신
}
```

**왜 `H` 가 아니라 `(f, n, d)` 인가** (설계서 §6 대비 의도적 이탈 1, 리더 승인):
- 지면 = `{ X | n·X = d }`. 픽셀 p 의 지면점 = `d·m/(n·m)`, `m = K⁻¹p`.
- 높이 h 점의 상: `p_h ≃ p − h·((n·m)/d)·(K·n)` ← **뷰어가 쓰는 유일한 식(내적 2번)**.
- `H` 를 실으면 뷰어가 `H⁻¹`·`K⁻¹H` 분해(r1,r2,r3 부호결정)를 **재구현**해야 한다 — 설계 §1-7 이 금지한 이중구현이 정확히 뷰어에서 발생한다. `(f,n,d)` 는 정보적으로 등가(지면 원점·방위라는 임의 선택분만 빠지며 육면체는 그 값에 불변)이면서 `projectCuboid` 를 **20줄**로 끝낸다.
- `vertVP`/`horizon` 은 **소비처가 없어 싣지 않았다**(CLAUDE.md 규칙 2). 필요하면 `K·n` 으로 언제든 유도된다.

### 3-3. ★ 설계의 핵심 결정 — **f 는 프리셋(zoom) 축으로 공동추정한다**

- 같은 프리셋 안에서 **면(slot)을 아무리 늘려도 개선이 0**이다. 연속 스트립의 면들은 **같은 두 소실점을 공유**하므로 독립 정보가 늘지 않는다.
- 얕은 tilt 프리셋(깊이변 213px)은 **단독 추정 시 조용히 크게 틀린다.** 검증자 재현치(σ=2px, 통합 경로): **preset1 단독 20.2% → 공동추정 3.8%**(약 5배 개선).
- → PTZ 카메라의 f 는 **zoom 만의 함수**이므로, 카메라당 `fovBaseV` **하나**를 전 프리셋에서 공동추정하고 프리셋 f 는 `fovV(zoom)` 로 **유도**한다. 조건수 좋은 프리셋이 나쁜 프리셋을 **구제**한다.
- 트레이드오프(정직): 조건수 최선 프리셋은 **약간 나빠진다**(검증자 재현: preset3 3.1% → 3.8%). 최악을 5배 구하는 대가로 타당하다.

> **설계서 §4-4 의 "35.4% → 1.7%" 수치는 그대로 재현되지 않았다**(합성 기하가 실데이터와 달라서). 결론(단독 추정은 얕은 tilt 에서 무너지고 공동추정이 구제한다)은 동일하나, **수치는 검증자 재현치(20.2% → 3.8%)를 정본으로 삼는다.**

## 5. ★★ 정합 사각지대 — 이 기능의 가장 중요한 한계 (반드시 읽을 것)

### 5-1. `f`/`tilt` 는 ROI 평행이동에 **눈이 멀다**

소실점은 직선의 **방향**에서 나온다. 폴리곤을 통째로 평행이동해도 **방향은 보존**된다.

> **±200px 평행이동에도 Δf 0.7% / Δtilt 0.05°.** → **"f 오차 0.53%" 는 ROI 정합에 대해 아무것도 말해주지 않는다.**

실제로 자동보정이 preset 1 ROI 를 **+105px** 이동시켰는데 f/tilt 지표는 만점이었다.

### 5-2. 추가한 정합 지표도 **상보적이지 않았다** (검증자 최상위 지적)

`metricErr`(가로) + `tiltErrDeg`(세로) 가 실제로 재는 것은 이것뿐이다:

> *"이 ROI 는 (PTZ 가 보고한 tilt 를 가진) 어떤 카메라로 본 **2.5×5.0m 직사각형 스트립의 상(像)**인가?"*

**지면 위 닮음변환(평행이동 2 + 수직축 회전 1 + 균일스케일 1 = 4 DOF)은 그 성질을 완전히 보존**한다. 두 지표는 **원리적으로 반응할 수 없다.** 검증자가 기준 모델로 역투영 → 지면 변환 → 재투영해 **정확히 구성**하고 측정:

| 변환 | 이미지 평균변위 | `metricErr` (임계 0.8%) | `tiltErrDeg` (임계 1.0°) | 검출 |
|---|---|---|---|---|
| 기준(무변환) | 0px | 0.19% | 0.04° | – |
| [이미지] 가로 +105px *(실제 발생)* | 105px | **1.53%** | 0.06° | ✅ |
| [이미지] 세로 +200px | 200px | 0.38% | **−3.86°** | ✅ |
| **[이미지] 회전 10°** | **77px** | 0.57% | 0.05° | ❌ **둘 다 놓침** |
| **[지면] 평행이동 +2.5m (한 칸)** | 70px | 0.19% | 0.04° | ❌ |
| **[지면] 평행이동 +3m/+3m** | **360px** | **0.19%(불변)** | **0.04°(불변)** | ❌ |
| **[지면] 수직축 회전 30°** | 144px | 0.23% | 0.04° | ❌ |
| **[지면] 균일스케일 ×2.0** | **473px** | 0.14% | 0.03° | ❌ |

> **지면 3m/3m 이동 = 이미지 360px 어긋남인데 두 지표가 소수점까지 그대로다.**

소스 주석의 과장("0.8% 임계는 ≳60px 어긋남을 잡는다")도 **거짓이었고 정정됐다**(`groundModel.ts:49-52` — "순수 이미지 평행이동에만 성립" + 반례 명기).

### 5-3. 프리셋 간 불변량으로 **2 DOF 를 닫았다** (신규 입력 0)

원리: **카메라는 프리셋 사이에 움직이지 않는다.**

| 신규 지표 | 검출 대상 | 원리 | 임계 | 실측 |
|---|---|---|---|---|
| **`dDevRel`** (`groundModel.ts:569-581`) | **지면 균일스케일** | 카메라고 `d` 는 프리셋 불변량이어야 한다. `d` 는 **metric 스케일의 유일한 담지자** | 10% | 정상 스프레드 **0.5%** / 지면 ×2 → **−50%** |
| **`bearingDevDeg`** (`:583-597`) | **지면 수직축 회전** | 슬롯 방위 = PTZ pan + azimuth → 프리셋 불변량(pan 이 돌면 azimuth 가 반대로 줄어 **합이 불변**) | 8° | 정상 **±1.3°** / 지면 30° 회전 → **정확히 30°** |

- **C3(실카메라 호환) 준수**: `pan`/`tilt` 는 `camerapos`(PTZ 리드백)에서 온다 — **실카메라도 주는 값**. Unity 전용 GT 아님. 신규 입력·신규 의존 **0**.
- **⚠️ `tiltErrDeg`(②)와 `bearingDevDeg`(③)는 성질이 다르다 — 혼동 금지.** ②는 **절대** 검사다('수평 지면'이라는 세계 기준이 있어 PTZ tilt 와 **직접** 대조 가능). ③은 **상대** 검사다 — 슬롯의 실제 세계 방위를 우리는 모르므로 **PTZ pan 과 직접 대조하는 검사는 성립하지 않는다.** pan 은 프리셋을 공통 좌표계로 **정규화**하는 데만 쓴다.
- **제약**: ③④는 **프리셋 1개면 검사 불가**, **전 프리셋이 똑같이 틀리면 침묵**한다(상대 검사이므로).

### 5-4. ❌ **남은 구멍: 지면 평행이동 2 DOF — 원리적으로 검출 불가**

> **🔄 2026-07-14 19:44 갱신 — 1.5 DOF 해소됨.** `docs/20260714_194426_VPDseg_3D육면체_2DOF앵커지표.md`
> 아래 진단(**4지표 전부 침묵**)은 **여전히 옳다** — 그 4지표는 전부 ROI 파일만 보기 때문이다.
> 해법은 **신규 지표 3종**(`depthDevM`·`phaseDevM`·`unmatchedRate`)이며, **ROI 와 독립인 이미지 증거**(VPD seg 마스크의
> 차량 접지선)를 끌어들여 순환을 깬다. **깊이축은 완전히, 폭축은 비정수배만 닫혔다.**
> ⚠️ **아래 "실질 위험"(정확히 한 칸 = 2.5m = 폭축 정수배)은 하필 닫히지 않은 0.5 DOF 다 — 여전히 잡히지 않는다.**
> 아래 본문은 작성 당시 그대로 보존한다.

**4개 지표 전부가 침묵한다.** ROI 를 지면에서 밀어도 그것은 여전히 "같은 카메라로 본 직사각형 스트립"이기 때문이다.

> ### 실질 위험
> **지면 평행이동 2.5m = 주차면 정확히 한 칸.**
> ROI 가 통째로 옆 칸을 덮어 **점유가 엉뚱한 주차면에 찍혀도 아무 경고가 안 뜬다.**
> **"경보 없음 ≠ ROI 정합됨."**

- **1단계에서 해결 불가.** 닫으려면 **이미지 증거(노면 주차선 도색)와의 직접 대조**가 필요한데, **차량이 주차선을 가린다.** → **2단계(빈 배경 median 합성 = 차량이 지워진 주차장 바닥)가 유일한 해법**이며 **선행 조건**이다.
- **은닉하지 않았다**: ① `test/groundSimilarityDetect.test.ts`·`test/roiAlignmentBlindSpot.test.ts` 가 **이 침묵을 테스트로 봉인**(사각지대를 코드에 명시) ② `app.js:452 GROUND_ALIGN_LIMIT` 이 **뷰어 배지 tooltip 에 상시 표시** ③ `groundModel.ts:554-566 §GROUND-SIMILARITY` 주석.
- **기각한 대안 — ROI 오프셋 *역추정*(격자탐색 argmin)**: 구현·측정까지 해봤으나 폐기. (a) `f` 와 `dx` 가 강하게 결합돼(105px 이동 ≈ 단독 f 3.9% 변화) f 오차 0.55% 만으로 dx 추정이 ~15px 편향되고 탐색 경계까지 발산. (b) `dy` 방향은 잔차 지형이 **평평한 골짜기**라 최소점이 정해지지 않는다. → **"확신 있게 틀린 숫자"** 를 내놓게 되므로 **검출까지만 하고 정량 역추정은 하지 않는다.**

## 7. 점유 판정(`computeOccupancy`) 연결 seam — **이번엔 미구현, 설명만**

현행 판정은 **번호판 중심이 바닥 폴리곤 안인가**이다(`core.js:459-462`). 이것은 **구조적으로 틀린다**: 번호판은 지상 ~0.5m 에 **떠 있는 점**인데 바닥 폴리곤은 높이 0 의 도형이다. 카메라가 비스듬히 볼수록 높이 0.5m 의 점은 **자기 발밑에서 수직소실점 방향으로 밀려나 보이고**(시차), 그 변위가 주차면 폭에 근접하면 **번호판이 옆 칸 폴리곤 안으로 들어간다.** 얕은 tilt(preset 1)일수록 심하다.

**지면모델이 이 seam 을 어떻게 닫는가**: `n`·`d`·`f` 로 번호판 점을 "높이 0.5m 의 점"으로 해석해 **발밑(지면) 좌표로 내릴 수 있다** — 이미지의 번호판 점과 수직소실점(`K·n`)을 잇는 직선이 그 점의 수직선이고, 그 선 위에서 0.5m 내린 지점이 접지점이다. `pointInQuad(발밑점, 바닥폴리곤)` 로 바꾸면 시차 오류가 **원리적으로 사라진다.** 한 줄 교체 수준이지만 **이번 범위 밖**(설계 §10·C6).

> **⚠️ 단, seam 을 닫아도 §5-4 의 구멍은 남는다.** ROI 자체가 옆 칸이면 시차를 고쳐도 **여전히 옆 칸에 찍힌다.**

## 13. 직전 세션(전역인덱스 재부여 위험)과의 관계

직전 세션 문서 `docs/20260713_194236_정밀수집_전역인덱스_주차면목록_프리셋리스트제거.md` 가 다룬 **전역번호 재부여** 문제는 이번 변경과 **두 지점에서 만난다**:

1. **테스트 설계 결함의 원인 중 하나였다** — 사용자가 전역번호를 재지정(preset2 가 13→8 역순)하자 값 단정 테스트가 깨졌다. §8-3 의 픽스처 동결이 이를 구조적으로 차단했다.
2. **4단계(`slot_spec.json`)의 최상위 위험이다** — Unity 가 `PtzCamRoi.json` 을 재생성하면 `normalizeGlobalIdx` 가 1..N 을 **재부여**해 `globalIdx` 키가 **다른 면을 가리킨다**(조용한 오귀속). 설계 §7-3 의 완화책(**기하 지문 재바인딩** + 매칭률 <80% 시 **전체 기각**)은 4단계 착수 시 반드시 구현해야 한다.

**이번 1단계는 `slot_spec.json` 을 만들지 않았으므로 이 위험에 노출되지 않는다.** 지면모델은 `camIdx:presetIdx` 로만 키를 잡고 `globalIdx` 를 저장하지 않는다.

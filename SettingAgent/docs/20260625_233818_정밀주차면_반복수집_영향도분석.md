## 3. 의존성 그래프 — `better-sqlite3` (1순위 리스크 → 해소, 실측 기록)

- `npm install better-sqlite3 -w SettingAgent` → **better-sqlite3@12.11.1 프리빌트 설치 성공**(구현자 실측, 02 §0).
  - 프리빌트 바이너리 존재: `node_modules/better-sqlite3/build/Release/better_sqlite3.node`.
  - `require('better-sqlite3')` + `:memory:` CRUD → **네이티브 빌드(node-gyp) 없이 로드·동작 확인**(Node v24.16.0, Windows 11).
  - allow-scripts 정책으로 install 스크립트 자동 실행 안 됐으나 prebuild-install 산출물 동봉 → 빌드 불필요.
- `@types/better-sqlite3@7.6.13` devDependency.
- **결론**: `node:sqlite` 대안 불필요. `SqliteStore`는 dbPath 주입형 → 구현 교체 시 테스트 영향 최소.
- 영향 범위: `better-sqlite3`는 **SettingAgent 단독 의존**. SettingViewer·Action/DM·`@parkagent/types`에 전파 없음. 단, 네이티브 모듈이므로 **다른 OS/Node 버전으로 이식 시 해당 환경 프리빌트 가용성 재확인 필요**(확인 필요 항목).

## 4. 브레인(AgentRuntime/SetupBrain) 가산

- `SetupBrain`에 `reviewCheckpoint?`/`finalizeCapture?`를 **옵셔널**로 추가 → 기존 `judgePreset`/`dedupeAndLabel`/`finalReport`·`reviewSetup` **전부 보존**. 옵셔널이라 기존 fake brain 테스트도 무영향.
- `AgentRuntime`는 기존 `chatJson` 재사용(인라인 한글 프롬프트). LLM 클라이언트 미연결(`!this.client`) 시 두 메서드 모두 null → 결정형 강등.
- 프롬프트는 1차 인라인(설계서 §11-5 확정). 파일화(`capturePrompts`)는 후속.

## 5. 운영 영향

- **두 서비스 동시 운영**: SettingAgent(:13020)·SettingViewer(:13030). SettingViewer가 `/capture/*`를 프록시하므로 SettingAgent 먼저 기동.
- **장시간 잡·DB 증가**: 반복 수집은 수십 초~분 주기 × N회 → 잡이 길게 돈다. `data/observations.sqlite`가 런마다 누적 증가. 보존/정리 정책은 **범위 밖**(런 단위 보존).
- **재기동 복구(범위 밖)**: CaptureJob은 단일 인메모리 잡. running 중 프로세스 종료 시 런 status='running'으로만 남고 정리 로직 없음.
- **DB 파일 경로**: `capture.dbFile='data/observations.sqlite'`(`SqliteStore`가 디렉터리 자동 생성). WAL 모드 → `-wal`/`-shm` 부수 파일 생성됨.

## 7. 최종 영향도 결론

- **파급은 SettingAgent 내부 + SettingViewer로 국한.** 공유 계약(`@parkagent/types`)·`/setup/*`·`/mapping`·SetupArtifact 형식 불변 → **Action/DM 무영향**.
- 모든 변경이 가산적(신규 모듈 + 옵셔널 메서드 + 키 루프 자동 병합 설정) → **기존 81/62 테스트 회귀 0**(실측).
- 좌표 불변식(검출·집계만 좌표 생성, LLM은 메타만) 충족 검증.
- 신규 네이티브 의존성 `better-sqlite3`는 SettingAgent 단독·프리빌트 로드 실측 성공. 단 타 환경 이식 시 재확인.
- 동작확인(실서버/실기기 스모크)은 미수행 — **별도 단계에서 시뮬레이터로 수행 권장**.

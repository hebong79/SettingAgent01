## 0. 배경 — 마스터 3회 반복 지적

마스터가 동일 사안을 **3회 반복 지적**한 끝에 확정된 요구다(메모리 [[vpd-auto-detect-forbidden]]):

> SettingAgent에서 **VPD(차량 검출)는 자동 경로에서 돌리지 않는다.** 정밀수집 라운드·라이브 "검출 실행" 등 기본 검출 흐름은 **LPD(번호판) 중심**이고, VPD는 **별도 "VPD 검출(테스트)" 버튼**으로만 수동 1회 실행한다.

**방해 기전(코드로 확인된 근거):**
- VPD는 인지율 ~70%로 불안정(오검 포함)하다.
- 기존 코드는 번호판 필터(`applyPlateFilter`/`filterPlatesOnPlace`)를 **VPD가 검출한 vehicles 배열에 결박**해 동작시켰다(`CaptureJob.ts` 구 :386, `detectPipeline.ts` 구 :299). VPD가 차량을 놓치면 그 차량 위의 정상 번호판까지 "귀속 차량 없음"으로 함께 깎였다.
- `Aggregator.ts:255-257`은 **vehicle 클러스터가 있을 때만** slot 클러스터를 생성한다(`if (vehicles.length===0) continue`) — VPD 미검출이 캡처 라운드 집계 자체를 무산시키는 구조였다.

이번 변경은 이 결박을 코드 변경 없이(결정 C, §2 참고) 해소하고, VPD 호출 자체를 자동 경로에서 끄는 게이트를 추가했다.

## 2. 수정 로직의 의도와 동작

### 2.1 레이어 분리 — 라이브러리 기본 true / 라우트·UI 기본 false
`CaptureJob.start()`와 `runDetect()`(라이브러리 계층)는 인자 미지정 시 **기존 동작(VPD 항상 실행)을 그대로 보존**한다(기본값 `true`) — 기존 테스트 호출부(라이브러리 직접 호출)를 무수정으로 통과시키기 위한 설계다. 실제 **제품 정책은 라우트 계층**(`captureRoutes.ts`)에서 `?? false`로 강제하며, UI(`app.js`)도 정밀수집 시작·검출 실행 버튼 모두 `vpdEnabled:false`를 전송한다. 즉 "VPD 항상 실행"이라는 예전 기본값은 라이브러리 내부에만 잔존하고, **실제 도달 가능한 모든 경로(라우트·UI)는 OFF가 기본**이다.

### 2.2 플레이트 필터의 "vehicles 결박 → 폴리곤 직접" 자동 전환 (코드 변경 없음)
`applyPlateFilter`/`filterPlatesOnPlace`(`src/capture/onPlaceFilter.ts`, 무접촉)는 `keptVehicles=[]`가 되면 (A) 차량 귀속항이 자동으로 비고, (B) "번호판 중심 ∈ 주차면 폴리곤"만으로 생존 여부를 결정한다. VPD off → `raw=[]` → `vehicles=[]`가 되는 순간, **필터 코드를 건드리지 않고도** 폴리곤 직접 필터로 전환된다 — 이것이 마스터가 지목한 방해 기전의 실제 제거 지점이다.

### 2.3 F10 가드 재정의(결정 E) — VPD off 시 우회
`SetupPipeline`의 원버튼 파이프라인은 원래 "검출 0건이면 finalize 미실행"(F10, DB 파괴 방지 가드 — 메모리 [[finalize-slotsetup-wipe-fragility]])이었다. VPD를 끄면 vehicle det가 항상 0이 되므로, 가드를 그대로 두면 **discovery(LPD-only)·센터링이 의존하는 `slot_setup` 행 + `slot3d_front_center` 부트스트랩(오직 `Finalizer.replaceSlotSetup`만 수행, `Finalizer.ts:271`)이 자동 경로에서 영원히 불가능**해진다. 따라서 가드를 `runVpdEnabled && dets===0`으로 재정의해 **VPD off일 때만 우회**한다. 우회해도 검출 매칭이 없는 슬롯은 `prev` 보존 가드([[finalize-slotsetup-wipe-fragility]] 참고)가 그대로 적용되어 기존 vpd/lpd/occupy 값을 null로 파괴하지 않는다.

### 2.4 cuboid(육면체) 게이트
육면체는 `vpd.segment`(VPD 산물)에 의존한다. VPD off면 원천 검출도 없어 산출물이 비므로, `CaptureJob.ts:429`·`detectPipeline.ts:279` 양쪽에서 `vpdEnabled` 조건을 추가해 낭비 seg 호출 자체를 막았다.

## 3. 입출력·REST 계약

### POST /capture/start
- 바디 추가 필드: `vpdEnabled?: boolean` (optional, 기본 **false** — 라우트 정책)
- 동작 변화: 미지정 시 `job.start({..., vpdEnabled:false})` — VPD 자동 라운드 검출 정지. `pipeline.onCaptureStart(autoChain, false)`로 F10 가드도 우회 모드 진입.
- 하위호환: 기존 클라이언트가 필드를 보내지 않아도 **에러 없음**(zod optional). 단 **거동은 바뀐다**(이전엔 암묵적으로 VPD 실행 → 이제 정지). 이는 제품 정책 변경의 의도된 결과이며 API 계약 자체(스키마)는 깨지지 않는다.

### POST /capture/detect
- 바디 추가 필드: `vpdEnabled?: boolean` (optional, 기본 **false**)
- 응답 `summary`에 `vpdEnabled: boolean` 필드 신규 추가. **기존 `toEqual`(정확일치) 방식으로 응답을 검증하던 외부 클라이언트/테스트가 있다면 필드 증가로 깨진다** — SettingAgent 내부 테스트 3건이 실제로 이 이유로 갱신됐다(§5 참고). REST 계약 관점에서는 **확장(하위호환 추가)**이지만, 정확일치 비교자에게는 파급이 있다.
- vpdEnabled:false 시 응답 `vehicles: []`(또는 필드 자체 부재/0), `cuboids` 키 없음.

### GET /capture/status
- 응답에 `vpdEnabled`(runId 보유 시)가 추가로 노출된다(순수 추가 필드, 하위호환).

### GET /mapping (영향도 분석 — 아래 §4.2 참고)
- 계약 자체는 무변경. 단 VPD off 자동 경로로 finalize가 실행되면 응답 `slots`/`globalIndex`가 **빈 배열이 될 수 있다**(§4.2에서 근거와 함께 상세).

### 4.2 setup_artifact.json(`GET /mapping`) 빈 slots — 실제 코드 추적 결과 (설계서 §7-3 교차확인, 추측 아님)

설계서 §7-3이 documenter에 이관한 미해결 항목을 코드로 직접 확인했다.

**사실 확인:**
- `Finalizer.finalize()`(`Finalizer.ts:186-215`)는 `accepted`(vehicle 클러스터, VPD 종속) 배열만으로 `assemble()`을 호출해 `SetupArtifact.presets/slots/globalIndex`를 조립하고, `repo.saveArtifact(artifact)`(:215)로 **`data/.../setup_artifact.json`에 즉시 저장**한다. 이 저장은 뒤이어 나오는 `slot_setup`(DB) 블록(:222-280, geometry 기반)과 **완전히 별개 소스**다.
- VPD off로 자동 경로(F10 우회 포함)에서 finalize가 실행되면 `accepted=[]` → `artifact.presets=[]`, `artifact.slots=[]`, `artifact.globalIndex=[]`가 그대로 `setup_artifact.json`에 저장된다.
- `GET /mapping`(`server.ts:232-239`)은 `repo.loadArtifact()`를 **그대로 반환**한다 — VPD off finalize 직후라면 `{presets:[], slots:[], globalIndex:[], createdAt}`가 응답된다.

**소비처 확인(뷰어/클라이언트, 크래시 여부):**
- `web/app.js loadMapping()`(:170-177)은 `res.ok`면 JSON을 그대로 `state.mapping`에 저장한다. 빈 배열이어도 파싱 에러 없음.
- `web/core.js`의 모든 소비 함수(`rawSlots = Array.isArray(artifact.slots) ? artifact.slots : []`(:266), `artifact.slots ?? []`(:750,787,862,943,970,1006,1033) 등)가 **전부 방어적** — 빈 배열 입력에 크래시 없이 "슬롯 0개"로 처리한다.
- `capFinalize()`(`app.js:2116-2133`)는 finalize 성공 시 `` `최종화 완료: 슬롯 ${data.slots}, 전역 ${data.globalCount}` `` 메시지를 표시하고 `loadMapping()`을 호출해 **검수 탭 상태(`state.mapping`)를 갱신**한다.

**정직한 파급(숨기지 않음):** VPD off 자동 경로로 finalize가 실행되면 (1) `GET /mapping` 응답이 빈 slots — **크래시는 없다**(방어적 코드로 확인 완료). 하지만 (2) 검수 탭 UI에는 "최종화 완료: 슬롯 0, 전역 0"이 표시되고 `state.mapping`(뷰어 편집 대상)도 비어, **사용자 입장에서는 실패처럼 보일 수 있는 UX 갭**이 실재한다. 이는 버그가 아니라 "artifact(JSON, vehicle-anchored)"와 "slot_setup(DB, geometry-anchored)"가 **서로 다른 소스를 가진 설계**의 자연스러운 결과다 — DB(`slot_setup`)에는 `slot_roi`·`slot3d_front_center`가 정상 채워지며 이후 discovery/센터링이 이를 이어받는다. **정본은 DB이고 JSON 아티팩트/검수 탭은 vehicle 검출 기반 보조 뷰라는 점을 사용자에게 알려야 UX 오해가 없다** — 이 문서로 리더에게 명시 전달한다. 코드 수정은 이번 스코프 밖(마스터 지시는 VPD 게이트뿐)이므로 변경하지 않았다.

### 4.3 잔존 VPD 표면 3곳 — 전부 수동 트리거 한정 (qa 검증 §3 인용)

- **실 VPD/LPD·실 카메라 라이브 미관찰** — 외부 REST/카메라 미가동, 유닛(스파이/스텁) 검증만 수행됨. "검출 실행" 응답에 vehicles 없음/오버레이 미표시, "VPD 검출(테스트)" 버튼의 vehicles+육면체 표시, 정밀수집 status 배지는 **리더의 라이브 확인이 아직 필요**하다.
- **F10 부트스트랩의 실 DB 기록 미관찰** — `setupPipeline.test.ts`는 순수 상태머신(finalize 스파이)만 검증. VPD off + autoChain에서 finalize가 실제 `slot_setup` 행 + `slot3d_front_center`를 DB에 기록하고 discovery→센터링이 정상 부트스트랩되는지는 스텁 경계까지만 확인됨(단, 관련 기존 스위트 `finalizerFloor`·`checkpointFinalizer`는 finalize 기하 산출 자체를 계속 통과 — 회귀 0).
- **web/app.js·index.html 브라우저 실행 미관찰** — 코드 정독으로만 확인(버튼 바인딩, capStart 고정값, 배지 조건).
- 본 문서 §4.2(빈 slots 검수 탭 UX 갭)는 코드 정적 추적으로 확정했으나, **실제 화면에서 "슬롯 0" 메시지가 뜨는지는 라이브 확인 대상**이다(코드상 크래시는 없음을 확인했을 뿐).

## 6. 사용법 요약 (리더/사용자용)

- **기본(변경 없이 사용)**: 정밀수집 시작·"검출 실행" 버튼 — **VPD 미실행, LPD만 동작**. 상태에 `vpdEnabled:false` 배지 노출.
- **VPD가 필요할 때만**: 신규 **"VPD 검출(테스트)"** 버튼을 눌러 해당 프리셋에서 1회 VPD(+육면체)를 실행·확인한다. 자동 경로에는 절대 편입되지 않는다.
- **주의**: VPD off 상태로 정밀수집을 완료하면 검수 탭/뷰어의 "슬롯 N개" 표시가 0이 될 수 있다(§4.2) — 이는 실패가 아니라 vehicle 기반 보조 아티팩트가 비는 것이며, 정본 DB(`slot_setup`)의 기하(`slot_roi`/`slot3d_front_center`)는 정상 기록되어 discovery·센터링이 계속 진행된다.

## 1. 배경

'DB에 추가'(`POST /capture/slots/lpd`)는 라이브 LPD 검출 결과를 `slot_setup.lpd`에만 쓴다. `SqliteStore.upsertSlotLpd`는 `occupyRange`가 동봉되지 않은 호출을 **의도적으로 무접촉**으로 둔다(wipe 방지 — `finalize`/`discovery` 산출 `occupy_range`를 다른 경로가 덮어쓰지 못하게 하는 기존 규약, `SqliteStore.ts:297~299` 주석). 그 결과 "lpd는 있는데 점유영역이 없는 슬롯"이 남는 사례가 생겼고, 이를 수동으로 채우는 방법이 없었다. 이를 채우는 버튼이 이번 요청이다.

## 2. 설계 원칙 — 새로 짜지 않고 공통 진입점으로 승격

점유영역 생성 로직은 이미 `PlateDiscoveryJob`이 인라인으로 쓰고 있었다(`buildPlateAnchoredQuad(quadBoundingRect(quad), quad)`). 새 라우트를 위해 이 식을 복제하면 두 경로(discovery/수동)가 미묘하게 갈라질 위험이 있어, **함수로 추출해 두 경로가 같은 함수를 호출**하도록 했다.

### 2.1 신규 `buildOccupyRangeFromPlate` (`src/capture/floorRoi.ts`)
```ts
export function buildOccupyRangeFromPlate(plateQuad: NormalizedQuad): NormalizedQuad {
  return buildPlateAnchoredQuad(quadBoundingRect(plateQuad), plateQuad);
}
```
번호판 OBB quad만으로 점유영역(발자국)을 결정형 생성한다. 차량 bbox가 없을 때 판의 bounding rect를 대리 vehicle로 써 기존 `buildPlateAnchoredQuad`를 그대로 태우는 "판-only 경로"이며, 계산 자체는 기존 `buildPlateAnchoredQuad`/`quadBoundingRect`를 그대로 재사용한다(신규 기하 로직 없음).

### 2.2 `PlateDiscoveryJob.saveSlotLpd` 리팩토링
기존 인라인 `stringify5(buildPlateAnchoredQuad(quadBoundingRect(it.lpdOrig), it.lpdOrig))`를 `stringify5(buildOccupyRangeFromPlate(it.lpdOrig))`로 교체했다. 계산식이 100% 동일(단순 함수 추출)이므로 **동작 불변**이며, 더 이상 쓰지 않는 `quadBoundingRect` import를 제거했다.

### 2.3 신규 라우트도 같은 함수 호출
`POST /capture/slots/occupy`가 `buildOccupyRangeFromPlate(v.lpd)`를 호출한다 — discovery와 수동 생성 두 경로가 같은 입력(`lpd`)에 대해 항상 같은 출력을 낸다.

## 3. API 계약 — `POST /capture/slots/occupy`

`src/api/captureRoutes.ts`에 신규 등록.

- **요청 body**(zod, 전부 옵셔널): `{ cam?: number(양의 정수), preset?: number(양의 정수) }`. 생략 시 전 프리셋 대상. 검증 실패 시 `400 { ok:false, error:'invalid body' }`.
- **처리**: `store.getSlotSetup()`에서 `cam`/`preset`으로 필터한 뷰 중 **`lpd != null`인 슬롯만** 대상으로, 각 슬롯의 `lpdObb`(현재 값 그대로 재직렬화)와 `occupyRange`(신규 생성값)를 `upsertSlotLpd`로 부분 UPDATE한다. `lpd`가 없는 슬롯은 생성 근거가 없으므로 **스킵**(기존 `occupy_range`·타 컬럼 무접촉). 슬롯 단위 생성 실패는 개별 카운트만 하고 나머지는 계속 처리한다.
- **응답**: `{ ok: true, updated: number, skipped: number, failed: number }`.
- **DB 쓰기 규약 정합**: `lpdObb`를 항상 함께 보내므로 `SqliteStore.upsertSlotLpd`는 `occupyRange`가 동봉된 분기(`stmtLpdOccupy`)를 타 — §1의 wipe 방지 규약과 충돌하지 않는다(occupyRange를 명시적으로 채우는 것이 이 라우트의 목적 자체).

## 5. 회귀 가드 — `viewerPtzSyncCoverage.test.ts`

이 테스트는 모든 REST 라우트를 카메라 이동 여부(NO_MOVE/이동 발생)로 분류해, 신규 라우트가 분류 없이 누락되면 실패하도록 강제한다. 신규 라우트를 `NO_MOVE`(`'DB(lpd→occupy_range 재생성)'`)로 등록했다 — 이 라우트는 DB 갱신만 하고 카메라를 움직이지 않는다.

### 6.1 신규 유닛테스트 `test/slotOccupyBuild.test.ts` — 8케이스(`it` 블록) 전부 통과

4. lpd가 하나도 없으면 `updated:0`(DB 무변경).
5. 잘못된 body(`cam:0`, `preset:1.5`) → `400`.
6. `GET /capture/slots`(뷰어 소비 경로)로도 생성 결과가 노출됨.

### 6.3 라이브 검증(실서버, 포트 13020)
`POST /capture/slots/occupy {cam:1,preset:1}` → `{ok:true, updated:7, skipped:0, failed:0}`.

재생성된 7슬롯의 `occupy_range`를 기존 discovery 산출값과 좌표 단위로 비교한 결과, **7슬롯 전부 소수점 5자리 마지막 자리에서만 ±0.00001 차이**가 있었다. 이 차이는 알고리즘 불일치가 아니라 반올림 경로 차이다 — discovery 원본은 `round5` 이전 원좌표로 계산됐고, 이번 재생성은 이미 DB에 `round5`로 저장된 `lpd`를 입력으로 계산했기 때문에 발생하는 소수 5자리 반올림 오차다. 두 경로(discovery/수동 생성)가 같은 함수(`buildOccupyRangeFromPlate`)를 호출한다는 §2의 설계가 실측으로도 확인됐다.

## 7. 사용법

1. LPD 검지 패널에서 현재 프리셋의 라이브 LPD 검출을 실행하고 'DB에 추가'로 `slot_setup.lpd`를 채운다(또는 discovery로 이미 채워진 상태).
2. '점유영역 생성' 버튼 클릭 → 현재 cam/preset 범위에서 `lpd`가 있는 슬롯만 `occupy_range`를 재생성해 DB에 저장.
3. 결과 메시지(`#disc-msg`)에 갱신/스킵/실패 건수가 표시되고, 오버레이·좌측 목록이 즉시 갱신된다.

## 8. 주의사항

- **`lpd`가 없는 슬롯은 생성되지 않는다.** 번호판 정보가 없으면 점유영역을 만들 근거가 없으므로 의도적으로 스킵한다(위장 생성 금지). 해당 슬롯을 채우려면 먼저 LPD 검출/DB 추가(또는 discovery)로 `lpd`를 확보해야 한다.
- 버튼은 **재생성**이다 — 기존 `occupy_range`가 있어도 새 값으로 덮어쓴다(§6.1의 케이스 3).
- 대상 범위는 뷰어의 현재 cam/preset이며, 다른 프리셋의 슬롯은 건드리지 않는다.

## 4. 후보 비교

### 4-A. ROI(확장) 포함 게이트 ← **권고**

판정: `min/max`로 ROI bbox 를 잡고 중심 기준 `roiExpand` 배 확장한 사각형에 **판 중심**이 들어와야 채택.

```
bb = bbox(roi 4점);  cx,cy = bb 중심;  hw,hh = 반폭/반높이
채택조건: |c.x-cx| <= hw*roiExpand  &&  |c.y-cy| <= hh*roiExpand
```

|---|---|
| 실측 6건 | 14 통과(0.668) · **15 기각(4.736)** · 16 통과(0.564) · 17 통과(0.597) · 18 통과(0.448) · 19 통과(0.356) |
| 전 23건 | 정상 22건 전원 통과(최대 0.826 < 1.5) · 오검 1건 기각. **오검 0 / 미검 0** |

## 5. 권고안 상세 설계 (후보 A)

### 5-1. 타입 배선

`src/calibrate/types.ts:57` `DiscoveryTarget` 에 **옵셔널 1필드** 추가:

```ts
/** 자기 슬롯 ROI(바닥 quad, 정규화 4점). 소유면 게이트(§오검) 소스. 부재 시 게이트 비활성(하위호환). */
roi?: NormalizedPoint[];
```

- **옵셔널인 이유**: 기존 테스트·호출자를 깨지 않기 위해서다(파손 0의 근거). 프로덕션 경로는 항상 채워진다.

### 5-2. 대상 펼침

`src/calibrate/plateDiscoveryWriter.ts:49-56` `expandDiscoveryTargets` 의 push 객체에 `roi: v.roi` 1줄 추가.
`SlotSetupView.roi`(`src/capture/types.ts:128`)가 이미 존재하므로 **DB 스키마·쿼리 변경 없음**.

### 5-3. 게이트 순수함수 (신설, `plateDiscovery.ts`)

```ts
/** 판 중심이 자기 슬롯 ROI bbox 를 expand 배 확장한 사각형 안인지. roi 부재·퇴화 시 true(게이트 비활성). */
export function isInsideOwnRoi(c: NormalizedPoint, roi: NormalizedPoint[] | undefined, expand: number): boolean
```

- `roi == null || roi.length < 3` → `true`(정직 폴백 — 게이트 없음이 기존 거동).
- bbox 중심 기준 반폭·반높이 × `expand` 비교(축정렬, 비등방 자동 — ROI 자체가 원근상 납작하면 y 게이트도 자동으로 좁아진다. B가 매직넘버로 하려던 걸 데이터가 준다).
- 순수함수 → 단위테스트 대상.

|---|---|
| `plateDiscovery.ts:155-161` (Tier 0) | 기존 `matchRadiusNorm` 게이트에 **AND `isInsideOwnRoi`** 추가 |
| `plateDiscovery.ts:183-194` (Tier 1) | `pick` 채택 전에 `isInsideOwnRoi(centerOrig)` 확인. 실패 시 `continue`(다음 격자칸으로 — 위장 없이 계속 탐색) |

**중요 — Tier 1 의 기각 세맨틱**: `pickOwnedPlate` 가 이미 후보 1개로 좁힌 뒤이므로, 게이트 실패 시 그 창은 포기하고 `k` 를 진행한다. 30칸 소진하면 기존대로 `no_plate`. **위장 found 금지 규약 유지**(slot15 는 참 판을 다른 칸에서 찾거나, 못 찾으면 정직하게 `no_plate`).

> 설계 판단: "게이트 실패 후보를 제외하고 같은 창에서 차선 후보를 재선택"하는 방식도 가능하나, `pickOwnedPlate` 시그니처 변경이 필요하고 실측상 이득 근거가 없어 **채택하지 않는다**(단순함 우선). 필요해지면 후속.

### 5-5. opts (하드코딩 금지 규약 준수)

`PlateDiscoveryOpts`(`plateDiscovery.ts:48-56`)에 추가:

```ts
roiExpand?: number; // 1.5 — 자기 슬롯 ROI bbox 확장 배수(소유면 게이트). roi 미보유 타깃에는 무효.
```

- `ResolvedOpts` 에 `roiExpand: opts.roiExpand ?? 1.5`.
- 값 근거: 정상 최대 소요 0.826 → 1.5 는 **1.82배 여유**. 오검 4.736 → 임계의 **3.16배 초과**. 어느 쪽으로도 1자리 수 흔들림에 안전하다.
- `PlateDiscoveryJob` 은 `deps.opts` 를 그대로 전달(`PlateDiscoveryJob.ts:85,92`)하므로 **Job 코드 변경 불필요**.

### 5-6. 기본값의 거동 변화 (명시 요구 사항)

| 경로 | 기존 | 변경 후 |
|---|---|---|
| 단위테스트(`roi` 미지정 타깃) | 게이트 없음 | **동일**(게이트 비활성) — 회귀 0 |
| 프로덕션(`expandDiscoveryTargets` 경유) | 게이트 없음 | **게이트 활성(거동 변경 — 의도된 것)**. 실측 23건 중 slot15 1건만 결과가 바뀐다 |

즉 **기본값이 프로덕션 거동을 바꾼다**. 이것이 이 수정의 목적이며, 실측 전수 대조로 정상 22건 무영향임을 확인했다.

### 5-7. 영속화·규약 확인

- `roi` 는 `DiscoveryTarget`(메모리 전용)에만 추가. `plate_discovery.json` / `slot_setup` **스키마 불변** → 소수점 5자리 규약 영향 없음(`stringify5` 경로 그대로).
- VPD 자동검출 무관(LPD 경로 단독).
- 점유영역: `PlateDiscoveryJob.ts:246` 이 판 파생 사다리꼴이라 **자동 추종. 별도 수정 없음**(§1 표에서 확인).

## 8. 미해결·가정

1. **가정**: slot15 의 "참" 번호판이 프레임에 존재하는지는 미확인. 게이트 적용 후 `no_plate` 로 끝날 수 있으며, 이는 규약상 정상(위장 금지). 필요 시 별건 goal/loop 로 재포착.
2. **가정**: cam1/preset3 slot12·13 의 앵커 품질 저하(앵커 y=1.00013, 프레임 밖)는 **별개 결함 후보**다. 이번 게이트는 이 결함을 우회할 뿐 고치지 않는다. `lowerFrontAnchor` / `slot3d_front_center` 산출 점검을 후속 과제로 기록한다.
3. **미해결(별건)**: level1 크롭 창이 세로 71% 를 덮는 `computeCropWindow` 의 `h = frac*aspect` 규약(§4-D4). 오검 증폭 요인이지만 격자 세맨틱·기존 테스트 전반을 흔들어 이번 범위에서 제외했다.
4. **확인 필요(마스터)**: `roiExpand` 기본 1.5 를 확정할지, 더 타이트한 1.2(정상 최대 0.826 대비 여유 1.45배)로 갈지. 설계자는 셋업 ROI 품질 편차를 감안해 **1.5 권고**.

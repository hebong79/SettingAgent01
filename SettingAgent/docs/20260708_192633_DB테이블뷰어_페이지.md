## 2. 백엔드 — `src/api/dbRoutes.ts` (신규, read-only)

### 2.1 `registerDbRoutes(app, { dbFile })` — 지연 오픈 + 연결 재사용

- 클로저 변수 `let db: Database.Database | null`. `getDb()`가 첫 요청에서 `new Database(dbFile, { readonly:true, fileMustExist:true })`로 열어 캐시.
- **부팅 시 열지 않음**(지연) → 파일 부재로 서버가 죽지 않고, R4 독립성(캡처 블록과 무관 등록) 유지. 오픈 실패 시 호출측이 **503** `{ error, detail }` 반환.
- 결정 근거(설계 §7 확인필요 → 리더 기본안 승인): 요청당 open/close 는 반복 파일 오픈 비용, 상시 오픈은 부팅 결합 → **지연 캐시**가 단순·안전한 절충. read-only 라 write 잠금 경합 없음.

### 2.2 `GET /db/tables`

- `SELECT name FROM sqlite_master WHERE type='table' ORDER BY name` → `{ tables: string[] }`.
- `sqlite_sequence` 등 내부 실재 테이블도 포함(요구 10개와 일치, 숨김 필터 미도입).

### 2.3 `GET /db/table/:name?search=&limit=&offset=` (핵심)

- 반환 `{ columns, rows, total, limit, offset }`.
  - `columns` = `PRAGMA table_info("<name>")` 순서.
  - `rows` = `SELECT * FROM "<name>" <where> LIMIT ? OFFSET ?`.
  - `total` = `SELECT COUNT(*) AS n FROM "<name>" <where>` (검색 필터 반영 총건수 — 매 요청 계산).

## 3. 보안 설계 (핵심)

DB 식별자(테이블/컬럼명)는 SQL 에서 바인딩 파라미터로 넣을 수 없어 문자열로 조립할 수밖에 없다. 여기서 인젝션이 발생하지 않도록 **다층 방어**를 둔다.

### 3.1 테이블명 — sqlite_master 화이트리스트 (불일치 404)

```ts
if (!new Set(getTableNames(d)).has(name)) { reply.code(404); return { error: '존재하지 않는 테이블' }; }
```
- `:name`이 `sqlite_master`의 **실재 테이블명 집합과 정확히 일치**할 때만 진행. 임의 문자열이 SQL 식별자로 삽입될 여지를 **원천 차단**.
- 실증(QA 03-I): `no_such` → 404, `sqlite_master`(type≠table) → 404, `detection; DROP TABLE detection` → 404 **후 테이블 온전**(total 5 유지 = DROP 미실행).

### 3.2 검색·limit·offset — 전부 파라미터 바인딩

```ts
const clauses = columns.map((c) => `CAST(${quoteIdent(c)} AS TEXT) LIKE ? ESCAPE '\\'`);
where = ` WHERE (${clauses.join(' OR ')})`;
// ...
.all(...searchBinds, limit, offset)
```
- 검색어·limit·offset 값은 **100% 바인딩**(`?`), 문자열 연결 없음. 컬럼명만 스키마 신뢰값 + `quoteIdent`(내부 `"` → `""`).
- `CAST(col AS TEXT)` 로 숫자 컬럼도 문자열 검색 매칭(예 `search=0.9` → REAL 컬럼 히트).
- 실증(QA 03-I): `search=' OR '1'='1` → 리터럴 LIKE 바인딩으로 처리 → **0행**(SQL 로 해석됐다면 5행). 인젝션 무해.

### 3.3 escapeLike — LIKE 와일드카드 리터럴화

```ts
function escapeLike(q) { return q.replace(/[\\%_]/g, (c) => `\\${c}`); }
```
- `\ % _` 를 이스케이프하고 `ESCAPE '\'` 절과 함께 사용 → 사용자가 입력한 `%`/`_` 가 SQL 와일드카드가 아닌 **리터럴**로 매칭. 실증: `search=a_b` → `_` 와일드카드 아님(id4만, 대조군 `aXb` 미매칭).

### 3.4 limit clamp 1..1000

```ts
clampLimit: Math.min(1000, Math.max(1, Math.floor(n)))  // 비수치 → 기본 200
clampOffset: Math.max(0, Math.floor(n))                  // 비수치·음수 → 0
```
- 대용량 테이블 무한 렌더 방지. 실증: `limit=99999`→1000, `limit=0`/`-3`→1, `offset=-5`→0. `clampLimit`/`clampOffset` 는 export 되어 순수 유닛으로 별도 검증.

## 4. 프론트엔드

### 4.1 DB 탭 · `#db-view`

- nav 탭(R1) + `#db-view`(`analyze-view db-view` 전체폭 패턴 재사용) 섹션: `#db-table-select` 콤보 / `#db-search` / `#db-prev`·`#db-next` / `#db-meta` / `#db-table` 컨테이너.
- `setTab('db')`: `full = analyze || options || db` 로 전체폭(뷰포트·패널 숨김), `#db-view` 표시, `loadDbTables()` 호출.

### 4.2 데이터 흐름

- `loadDbTables()`: `GET /db/tables` → 콤보 채움(**1회 가드** `dbTablesLoaded`) + 첫 테이블 자동 선택 → `loadDbRows()`.
- `loadDbRows()`: `GET /db/table/:name?limit&offset[&search]` → `state.dbTotal` 갱신 → `renderDbTable(data)`. 실패 시 `#db-meta` 에 상태 표기(조용한 강등).
- `renderDbTable(data)`: 순수 `buildDbTableModel(data)` → `<table>` DOM 생성. 각 셀·헤더는 `escapeHtml` 로 이스케이프(XSS 방어). `#db-meta` 범위(`from–to / total`) 표기. `db-prev`(offset≤0)·`db-next`(offset+limit≥total) disabled 갱신.
- 이벤트: 콤보 change(offset 0 리셋), 검색 input **디바운스 250ms**, prev/next(offset ∓ `DB_LIMIT=200`, 0 클램프).

### 4.3 `buildDbTableModel({ columns, rows })` — 순수 함수 (core.js)

```
{ headers: columns, cells: rows.map(r => columns.map(c => formatCell(r[c]))) }
```
- `formatCell`: null/undefined→`''`, Buffer/Uint8Array(BLOB)→`[blob]`, 객체→JSON, 숫자(0 포함)/boolean→문자열.
- **throw 금지**: columns/rows 비배열·누락 → `{ headers:[], cells:[] }`(graceful). 컬럼 순서 보존(행 키 순서 무관). 브라우저 API 미참조 → vitest 직접 import.

### 4.4 REST 계약 (경계면)

- 응답 shape `{ columns:string[], rows:object[], total, limit, offset }`.
- `loadDbRows`가 `data.total` 읽고 `renderDbTable` → `buildDbTableModel({columns, rows})` 구조분해. PRAGMA `columns` 이름 == `SELECT *` row 객체 키 → `formatCell(r[c])` 인덱싱 일치. total/limit/offset 잉여 키는 순수함수가 무시. **shape 불일치 없음**(QA 03-I §4).

## 5. R4 완전 독립 근거

- **자체 read-only 연결**: `dbRoutes`는 `new Database(dbFile, { readonly:true })` 별도 연결. `SqliteStore`(캡처 write 연결)와 **파일 경로만 공유**, 연결 인스턴스는 완전 분리. write 라우트 부재(물리적으로 read-only).
- **캡처 블록과 무관 등록**: `server.ts` L255 `if (deps.dbFile) registerDbRoutes(...)` — `registerCaptureRoutes`(camera/vpd/lpd 주입 가드) 블록과 독립. settings 라우트처럼 dbFile 만 있으면 등록.
- **외부 서비스 무의존**: LLM(SetupBrain)·검출(VPD/LPD)·시뮬(PTZ)·캘리브레이션 전 경로 미참조. 결정형 DB I/O 뿐 → 시뮬·검출 서비스가 꺼져 있어도 동작.

## 8. 한계 / 확인 필요

- **프론트 DOM 결선은 vitest 밖**: 탭·콤보·검색·표 렌더·페이지 버튼(F2~F5)은 브라우저 통합이라 자동화 비대상. 로직은 순수함수(`buildDbTableModel`)로 커버, 실제 결선은 라우트 유닛 + 리더 라이브 경험적 검증으로 보완. 통과로 위장하지 않음.
- **read-only 연결 캐시 오픈**: 지연 오픈 후 프로세스 수명 동안 파일 핸들 유지. 장기 구동 서버에는 정상 설계이나, 테스트 정리(`rmSync`)는 best-effort. 결함 아님(QA 확인).
- **대용량 테이블 매 요청 COUNT(*)**: 현 규모(detection 15141행)는 ms급 무해. 수십만 행+ 로 커지면 COUNT 스킵(`rows.length===limit`로 다음페이지 존재 추정)으로 전환 여지(설계 §7 캐비엇). 현재는 미도입.
- **단일 DB 경로**: `dbFile = tools.capture.dbFile` 고정. 여러 DB 선택은 요구 밖.

## 1. 배경 / 진단 — "3개 중 1개만 보인다"의 실체

정밀수집(precise) 화면에서 **프리셋 3개 중 1개의 VPD/LPD 검출 결과만** 화면에 보이는 증상이 보고되었다. 리더의 사실확인 결과 이는 **데이터 저장 버그가 아니다**.

- **캡처 라운드의 `detection`·`parking_slots`는 3프리셋 모두 DB에 정상 저장**되어 있음을 DB 조회로 실증(§5 참조: run55~58에서 `SELECT cam,preset,COUNT(*) FROM detection GROUP BY cam,preset` → 3프리셋 모두 vehicle+plate 행 존재).
- 따라서 문제는 백엔드/DB/집계가 아니라 **라이브 화면 표시 상태 구조**에 있다.

### 근본 원인 2가지 (라이브 표시)

1. **`state.detect` 단일 객체**: 라이브 검출 상태가 프리셋 단위가 아닌 **단일 객체 1개**였다. `runLiveDetect`가 `POST /capture/detect` 결과를 이 한 칸에 덮어써, 항상 마지막 프리셋 검출만 남았다.
2. **`capFrameTick`의 능동적 파기**: 프레임 폴링 틱이 프리셋 전환을 감지하면 **`if (state.detect && presetChanged) state.detect = null;`** 로 직전 프리셋 검출을 능동적으로 **파기**했다. cadence idle 동안 마지막 프리셋에만 머물러 "1개만" 착시를 강화했다.

부수 원인: `drawDetectOverlay`의 잔상 가드(`presetKey(state.detect.cam,…) !== currentFrameKey()`이면 skip)가 단일 상태에서 프레임 전환 순간 공백을 만들었고, `renderSlotList`의 파일 모드 분기는 `state.placeRoi[currentFrameKey()]` **현재 프리셋 1개만** 렌더했다.

## 2. Goal (검증 가능한 성공 기준)

- 프리셋 전환 시 각 프리셋의 VPD/LPD 오버레이가 **보존·즉시 표시**(재검출 대기·공백 없음).
- 주차면 목록이 **3프리셋 전체**를 프리셋 그룹 헤더로 표시(각 슬롯 idx + 점유, 프리셋별 검출 count). 현재 프리셋 그룹 강조.
- 데이터 저장 3프리셋 **회귀 0**(백엔드 무변경).

리더가 표시 정책으로 **A안**(항상 3프리셋 전체 표시 + 현재 프리셋 그룹 `.current` 강조)을 선택했다. 파일 모드 슬롯은 per-slot VPD/LPD 태그 없이 **프리셋 헤더 count**(차량 N·번호판 M)로, 최종화 후는 DB의 vpd/lpd 태그로 표시(이원).

## 3. 물리적 제약 명시 (오해 방지)

라이브 프레임은 **물리적으로 1프리셋**이다 — 카메라가 한 번에 한 프리셋만 촬영하므로, **오버레이도 그 프레임의 프리셋 것만** 그려지는 것이 정상이다. "한 화면에 3프리셋 오버레이 동시 표시"는 물리적으로 불가능하다.

따라서 본 작업의 "3개 다 보이게"는 다음 두 가지로 달성한다:
- (a) **오버레이 보존** — 프리셋 전환 시 각 프리셋 검출을 파기하지 않고 유지 → 재방문 즉시 표시.
- (b) **주차면 목록 3프리셋 동시 열람** — 목록은 프레임과 무관하게 3프리셋을 그룹으로 나열.

## 4. 변경 사항 (G1~G8)

신규 클래스 없음. **순수 함수 1개 + 상태 필드 1개 + DOM 결선 + CSS**. 전부 프론트, 백엔드/DB/라우트 무변경.

| 단계 | 파일 | 변경 |
|------|------|------|
| G1 | `web/app.js` (state 블록) | `detect: null`(단일) → `detectByKey: {}`(cam:preset → 검출 맵, 프리셋별 보존) |
| G2 | `web/app.js` `runLiveDetect` | 결과를 `state.detectByKey[presetKey(cam,preset)]`에 **키별 저장**(덮어쓰기 아님) + `renderSlotList()` 호출 추가(검출 도착 시 목록 count·점유 갱신) |
| G3 | `web/app.js` `capFrameTick` | **핵심 버그 수정** — `const presetChanged=…` 및 `if (state.detect && presetChanged) state.detect=null;` **두 줄 삭제**(고아 변수 `presetChanged`도 함께 제거) |
| G4 | `web/app.js` `drawDetectOverlay` | `const d = state.detectByKey[currentFrameKey()]`로 **키 조회**, `d.vehicles`/`d.plates` 렌더. 잔상 가드 삭제(키 조회로 자연 해결) |
| G5 | `web/app.js` `updateLogicOccupancy` | `state.detectByKey[key]`로 조회(프리셋 일치 삼항 제거). floorPolys·computeOccupancy·occComputeByKey 저장 불변 |
| G6 | `web/app.js` `renderSlotList` | 파일/DB 분기를 `buildSlotListGroups` 기반 **3프리셋 그룹 렌더**로 통합. LLM `mapping.slots` 폴백은 무변경 |
| G7 | `web/app.css` | `.slot-list .slot-group`(그룹 헤더) + `.slot-group.current`(현재 프리셋 강조) 스타일 |
| G8 | `web/core.js` / `core.d.ts` | 순수 함수 `buildSlotListGroups` 신규 export + 타입 시그니처 |

### G8 `buildSlotListGroups` — 테스트 핵심 (순수 함수)

```
buildSlotListGroups({ finalized, placeRoi, detectByKey, parkingSlotsByKey })
  → [ { key, cam, preset, vehicleCount, plateCount, slots:[{idx, occupied, vpd?, lpd?}] } ]  (cam,preset 오름차순)
```

- **finalized=true (최종화 후, DB 모드)**: 소스 `parkingSlotsByKey`. `vehicleCount = rows.filter(r=>r.vpd).length`, `plateCount = rows.filter(r=>r.lpd).length`, `slots = rows.map(r=>({idx:r.slotIdx, occupied:!!r.occupied, vpd:!!r.vpd, lpd:!!r.lpd}))`. → DB의 vpd/lpd 태그 표시.
- **finalized=false (파일 모드, 라이브)**: 소스 `placeRoi`(프리셋별 폴리곤). 각 키에서 `detectByKey[key]`의 번호판 합집합(`plates` + plate 보유 `vehicles`)을 `computeOccupancy(floorPolys, plates)`로 **점유 재계산**. `occComputeByKey` 사전 채움에 **비의존** → 각 프리셋 검출이 도착하는 즉시 목록에 반영.
- **throw 금지**: `placeRoi`/`parkingSlotsByKey`가 null·비객체·빈 객체면 `[]` 반환(백엔드 UNKNOWN 강등 철학과 정합).

핵심 설계 결정: 파일 모드 점유를 오버레이용 `occComputeByKey`에 의존하지 않고 `detectByKey`에서 직접 재계산 → 렌더 타이밍 결합을 제거(검출 도착 즉시 목록 반영). `occComputeByKey`는 오버레이 원(circle) 표시에 계속 사용(G5, 불변).

## 7. 한계 · 인계

1. **DOM 결선(G1~G7)은 vitest 밖**: state 구조 전환·`drawDetectOverlay`/`updateLogicOccupancy` 키 조회·`capFrameTick` 파기 제거·`renderSlotList` 그룹 렌더·app.css는 브라우저 통합이라 유닛테스트 대상이 아니다. → 리더 경험적(서버 라이브) 검증으로 커버(§5). 통과 위장이 아니라, 자동화 커버리지는 순수 함수(G8)에 집중되고 결선은 실데이터 관찰로 확인됨을 명시한다.

2. **[비차단] `core.d.ts` 타입 선언 부정확 1건**: `buildSlotListGroups`의 `parkingSlotsByKey` 행 타입이 `vpd?: boolean; lpd?: boolean`로 선언됐으나, 실제 `/capture/runs/:id/slots` 피드(`ParkingSlotView`)의 `vpd/lpd`는 **`객체 | null`**이다.
   - **런타임 영향 없음**: `buildSlotListGroups`가 `r.vpd` truthy 필터 + `!!r.vpd` 강제 변환 → 객체/null 양쪽 모두 정상 산출(QA 케이스 5·6에서 실 피드 형태로 직접 검증 통과). 프론트 app.js는 JS라 무영향.
   - **부작용**: TS 소비자가 `state.parkingSlotsByKey`(객체|null 형태)를 캐스팅 없이 넘길 수 없고, 선언이 실 데이터를 오도한다.
   - **권고(선택, 차단 아님)**: 선언을 `vpd?: object | null; lpd?: object | null`(또는 `unknown`)로 완화. 기능·회귀는 정상이므로 후속 정리 항목.

3. **오버레이 물리 제약**: "한 화면 3프리셋 동시 오버레이"는 물리적으로 불가(§3). 라이브 오버레이는 항상 현재 프레임 프리셋 것만 표시되며, 3프리셋 열람은 목록으로 달성한다.

4. **파일 모드 첫 순환 전 미방문 프리셋**: 각 프리셋 점유/count는 그 프리셋을 최소 1회 방문(자동검출 1회)한 후 채워진다. 첫 순환 전 미방문 프리셋 그룹은 검출 0·전 슬롯 공차로 표시(정상, 순환 진행 시 채워짐).

## 8. 종합

증상 "3개 중 1개만"은 데이터 저장이 아닌 **라이브 표시 상태 구조** 문제였고, `state.detect` 단일 객체를 `detectByKey` 프리셋별 맵으로 이관하고 `capFrameTick`의 능동적 파기(2줄)를 제거하여 오버레이 보존을 달성했다. 목록은 `buildSlotListGroups` 순수 함수로 3프리셋 그룹을 조립(파일=count, DB=태그)한다. 백엔드/DB/라우트/집계/좌표 불변식 **일절 무변경**, vitest 1015 통과(신규 13), 회귀 0. 잔여: G1~G7 결선은 리더 경험적 검증으로 커버 + `core.d.ts` 타입 완화 후속 권고.

## 1. 배경 · 목적 · 확정 결정

### 1.1 사용자 요구
- 정밀수집 진행 중 **매 프리셋 이미지마다** LLM이 "주차면 몇 면 중 몇 면 점유"인지 판정하여 점유율(%)을 산출.
- 체크포인트 발화 방식을 **시간 주기 / 라운드(횟수) 주기 중 선택**할 수 있게.

### 1.2 탐색으로 밝힌 현 사실(설계 §1)
- 체크포인트는 `CaptureJob.checkpoint()` 한 곳에서만, **rounds 기준**(`done % checkpointEvery === 0`)일 때만 발화. 시간 트리거는 없었다.
- `checkpoint()`은 K라운드에 1회이므로 `checkpointEvery ≥ count`이면 마지막에 1회만 도는 것처럼 보임 → 사용자가 체감한 "마지막만 체크"의 실제 원인.
- LLM이 이미지를 보는 정밀수집 경로는 `FloorRoiReviewer → brain.recognizeFloorRoi` 하나뿐. 이는 채택 후보 슬롯마다 바닥 점유영역 4점 폴리곤을 그릴 뿐 **"몇 면 중 몇 면 점유"를 판정하지 않았다**(per-vehicle 순회, `maxPerCheckpoint` 기본 12 캡).
- 기존 `occupancyRate`(`AggregatedSlot`)는 `Aggregator`의 **결정형 통계**(검출된 라운드 비율)로, LLM 비전 판정이 아니다 → 신규 기능과 **공존**(대체 아님, §6).

### 1.3 확정 결정 3건(전제 · 변경 불가)
1. **신규 LLM 점유율 판정**: 프리셋 이미지마다 LLM이 면별 occupied/empty를 반환. floor ROI(4점 폴리곤)와 **별개**의 신규 `SetupBrain` 메서드 + 프롬프트 + `AgentRuntime` 비전 경로 + Reviewer + 저장.
2. **트리거 = 기존 체크포인트 전체를 시간/횟수 주기로**: `checkpoint()` 발화 조건을 `rounds`(`done%K==0`) / `time`(경과 ≥ `intervalMs`) 두 모드 중 하나로. UI 상호배타. 기본 `rounds`(하위호환).
3. **커버리지 = 모든 프리셋 강제**: 점유율 판정은 `lastFrameByPreset` **전 프리셋**(빈 화면 포함, 0%도 보고)을 순회. floor의 `maxPerCheckpoint`(12) 캡을 점유율 경로에 **적용하지 않음**.

### 1.4 MCP 경계(LLM 두뇌 vs 결정형 도구)
| 항목 | 담당 | 근거 |
|------|------|------|
| 점유/공차 **판정**(면별 occupied 여부) | **LLM 두뇌**(`judgeOccupancy`) | 시각·맥락 판단, 저빈도(체크포인트 cadence) |
| 점유율 산술(`occupiedCount/total`, rate) | **결정형**(런타임 계산) | LLM 산술 미신뢰 — 면별 플래그만 받아 개수는 코드가 집계 |
| 트리거 발화 조건(rounds/time 분기, 경과 계산) | **결정형**(`CaptureJob`) | 수치 불변식·`monotonic()` 결정성 |
| 저장·조회(테이블·라우트) | **결정형**(`SqliteStore`) | I/O만 |

경계 원칙: **판정 생성은 LLM, 산술·트리거·저장은 결정형.**

## 2. 신규 기능 A — LLM 점유율 판정

### 2.1 신규 계약(`src/brain/SetupBrain.ts`)
| 심볼 | 내용 |
|------|------|
| `OccupancyInput` | `{ camIdx, presetIdx, imageBase64, expected? }` — 프리셋 최근 프레임 + 기대 면 수 힌트 |
| `OccupancySpaceSchema` | `{ id:int, occupied:bool, box?:{x,y,w,h} }` — box는 정규화 0~1, 선택 |
| `OccupancyResultSchema` | `{ spaces: OccupancySpace[], confidence: 0~1 }` — LLM 원출력 스키마 |
| `OccupancyJudgment` | `{ spaces, occupiedCount, total, rate, confidence }` — 메서드 반환(결정형 집계 포함) |
| `judgeOccupancy?(input)` | 신규 **옵셔널** 메서드. 비활성/미설정 시 `null`(결정형 폴백 없음 → 저장 생략) |

**핵심 설계**: LLM은 `spaces[]`(면별 occupied 플래그)만 책임진다. `occupiedCount / total / rate`는 `AgentRuntime`이 파싱 후 산출한다. 따라서 LLM이 잘못된 rate를 응답해도 스키마가 strip하고 코드가 재계산하여 무해화된다.

### 2.2 런타임 구현(`src/brain/AgentRuntime.ts:211`)
`recognizeFloorRoi`를 템플릿으로 한 비전 경로. 실제 구현:
```ts
async judgeOccupancy(input: OccupancyInput): Promise<OccupancyJudgment | null> {
  if (!this.client || this.cfg.occupancy?.enabled !== true) return null;   // 게이트
  const { system, user: userTpl } = loadPromptPair(this.cfg.occupancy.prompt);
  const user = renderTemplate(userTpl, { camIdx, presetIdx, expected: '...'|'미상' });
  const parsed = await this.chatJson(system, user, (j) => OccupancyResultSchema.parse(j), input.imageBase64);
  if (!parsed) return null;
  const total = parsed.spaces.length;
  const occupiedCount = parsed.spaces.filter((s) => s.occupied).length;    // ★ 결정형 집계
  return { spaces, occupiedCount, total, rate: total > 0 ? occupiedCount/total : 0, confidence };
}
```
- `chatJson(..., imageBase64)` 재사용 → 이미지 다운스케일·JSON 모드·1회 재시도·`ollama`/`openai` 라우팅을 그대로 상속.
- `total === 0`(보이는 면 없음) → `rate = 0`. 빈 화면 0% 보고 요구(결정 3) 충족.

### 2.3 프롬프트(`config/prompts/occupancy.yaml`, 신규)
- 시스템: 고정 CCTV 주차장 영상에서 **보이는 모든 주차면을 왼→오·위 줄 먼저 순서로 나열**, 각 면에 1-based id 부여, 차량 점유 시 `occupied=true`.
- 주행로·횡단보도·인도·경계석·카메라 프레임은 제외. 먼 줄·흐릿한 면도 **recall 우선**으로 포함.
- 보이는 면이 없으면 `spaces`를 빈 배열(점유율 0%).
- **"점유율(%) 계산은 하지 마라 — 면별 occupied 플래그만 보고, 개수 집계는 시스템이 한다"** 명시(LLM 산술 미신뢰).
- 출력: JSON 1개만, 코드펜스·설명 금지. `{{camIdx}} {{presetIdx}} {{expected}}` 치환.

### 2.4 Reviewer(`src/capture/OccupancyReviewer.ts`, 신규)
FloorRoiReviewer와 **별 클래스**(관심사 분리). `review(runId, atRound, framesByPreset, shouldStop?, expectedByPreset?)`:
- `usable = !!(brain?.enabled && brain.judgeOccupancy)` — 부재/비활성이면 **전 프리셋 skip**(저장 생략).
- `framesByPreset`(`${camIdx}:${presetIdx}` → JPEG Buffer)를 **전량 순회, maxPerCheckpoint 캡 없음**(결정 3).
- 각 프리셋: `shouldStop?()` 체크 → `judgeOccupancy` 호출 → `store.insertOccupancy`. throw 시 **해당 프리셋만 스킵**(warn 로그), 나머지 계속.
- 반환 `{ llmUnavailable }` = `!usable || (attempted>0 && succeeded===0)`. 잡을 중단하지 않고 상태 플래그만 올린다(graceful skip).
- **폴백 없음**(floor ROI와 의도적 차이): 비전 없는 점유율은 의미가 없고, 결정형 `occupancyRate`가 이미 별도로 존재하므로 대체 산출을 하지 않는다.

### 2.5 저장·조회
- 저장: `SqliteStore.insertOccupancy(runId, {...})` — **이력 append**(같은 run/cam/preset에 체크포인트마다 행 추가 → 점유 변화 추적이 기능 목적).
- 조회: `SqliteStore.getLatestOccupancy(runId)` — 프리셋별 `MAX(at_round)` 1행(snake→camel 매핑).
- 라우트: `GET /capture/runs/:id/occupancy`(`src/api/captureRoutes.ts:206`) — `/aggregate` 미러. 400(invalid id) / 404(run not found) / 200(`getLatestOccupancy` 배열).

## 3. 신규 기능 B — 체크포인트 트리거 타입(시간/횟수)

### 3.1 설정 계약(`src/config/toolsConfig.ts` `CaptureSchema`)
| 필드 | 타입 | 기본값 | 의미 |
|------|------|--------|------|
| `checkpointTriggerMode` | `'rounds' \| 'time'` | `'rounds'` | 트리거 모드. 기본 rounds(하위호환) |
| `checkpointIntervalMs` | `int > 0` | `60000` | time 모드 주기(ms). rounds에서는 무시 |

두 필드 모두 zod `.default()` 보유 → config 파일 미기재 시 자동 폴백.

### 3.2 분기 로직(`src/capture/CaptureJob.ts`)
- `CaptureStartParams`에 `checkpointTriggerMode` / `checkpointIntervalMs` **필수 필드 2개 추가**.
- 상태 필드 `private lastCheckpointMs = 0`(monotonic ms).
- `start()`에서 `lastCheckpointMs = monotonic()`으로 초기화(첫 time 발화는 시작 + intervalMs 경과 시 — 즉시 발화 방지).
- 트리거 게이트 교체(회귀 0 — rounds 표현식 **동일** 유지):
```ts
// before: if (this.done % this.params.checkpointEvery === 0 && this.currentState() !== 'stopping')
if (this.shouldCheckpoint() && this.currentState() !== 'stopping') { await this.checkpoint(roundIdx); }

private shouldCheckpoint(): boolean {
  const p = this.params!;
  if (p.checkpointTriggerMode === 'time') return this.monotonic() - this.lastCheckpointMs >= p.checkpointIntervalMs;
  return this.done % p.checkpointEvery === 0; // rounds(기존 표현 동일)
}
```
- `checkpoint()` 진입 직후 `lastCheckpointMs = monotonic()` 갱신(다음 time 주기 기준점 → 연속 라운드 과다 발화 방지).
- **granularity 주의**: 두 모드 모두 **라운드 경계에서만** 평가한다(라운드 중간 발화 없음). time 모드는 "경과가 찼으면 이번 라운드 끝에 발화". 따라서 `intervalMs < 1라운드 소요`면 매 라운드 발화(사실상 rounds K=1) — 사용자가 라운드 소요보다 큰 간격을 선택한다고 가정(R3, §8).
- stop 반응성·`stopping` 가드·`&& currentState()!=='stopping'` 불변.

### 3.3 UI(`web/index.html` · `web/app.js`)
- `.capture-grid`에 트리거 모드 라디오(라운드 주기 / 시간 주기) + 체크포인트(라운드) / 간격(초) 입력.
- `capStart`: 라디오 값을 읽어 **상호배타** 전송 — `rounds`면 `checkpointEvery`, `time`이면 `checkpointIntervalMs`(초→ms). change 리스너로 해당 입력 필드만 표시(hidden 토글).

### 3.4 REST 계약(`src/api/captureRoutes.ts` `StartBodySchema`)
- `checkpointTriggerMode?: z.enum(['rounds','time'])` · `checkpointIntervalMs?: int>0`(둘 다 **옵셔널**).
- start 조립: `parsed.data.checkpointTriggerMode ?? deps.cfg.checkpointTriggerMode`(cfg 폴백). 즉 외부 클라이언트가 미지정하면 config 기본값(rounds/60000) 적용 → **기존 클라이언트 무영향**.

## 4. 데이터 흐름도

```
캡처 라운드(runRound)
  └─ lastFrameByPreset 갱신(프리셋별 최근 JPEG)
  └─ shouldCheckpoint()  ── rounds: done % checkpointEvery === 0
     │                    └ time : monotonic() - lastCheckpointMs >= checkpointIntervalMs
     └─(발화)─ checkpoint(roundIdx)
                ├─ lastCheckpointMs = monotonic()  (다음 time 기준점)
                ├─ 집계 + CheckpointReviewer   (텍스트 자문, 기존)
                ├─ FloorRoiReviewer            (채택 슬롯 점유영역 4점, maxPerCheckpoint 캡, 기존)
                └─ OccupancyReviewer.review    (★ 신규: 전 프리셋 점유율 판정, 캡 없음)
                     └ judgeOccupancy → insertOccupancy(이력 append)
```
세 Reviewer는 checkpoint 안에서 **병렬 공존**(서로 독립 경로). occupancy 실패는 잡을 중단하지 않고 `llmOccupancyUnavailable` 상태 플래그만 올린다.

## 5. DB 스키마 · 마이그레이션

신규 테이블 `occupancy`(`SqliteStore.ts:83`):
```sql
CREATE TABLE IF NOT EXISTS occupancy (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  run_id INTEGER, cam_idx INTEGER, preset_idx INTEGER, at_round INTEGER,
  occupied_count INTEGER, total INTEGER, rate REAL,
  spaces_json TEXT, updated_at TEXT
);
CREATE INDEX IF NOT EXISTS idx_occ_run ON occupancy(run_id, cam_idx, preset_idx, at_round);
```
- **`CREATE TABLE IF NOT EXISTS`** → 기존 파일 DB를 열 때 자동 생성. `ALTER` 불필요. 기존 8테이블(capture_run/observation/detection/aggregated_slot/checkpoint/artifact_snapshot/floor_roi)·`ensureSchema()` 멱등성 무영향.
- `spaces_json`에 면별(box 포함) 전량 보존 → 향후 오버레이 근거(후속). 스칼라 `occupied_count/total/rate`는 빠른 조회·표시용.
- **`confidence`는 미저장**(테이블에 컬럼 없음). 현재 범위(조회만)에서 무해. 향후 신뢰도 표시가 필요하면 컬럼 추가(후속, §9).

## 8. 리스크(QA §7 · 설계 §11)

- **R1 비용/지연**: occupancy는 캡 없이 전 프리셋 비전 호출(결정 3). 프리셋 수 N × 체크포인트 빈도. → 완화: cadence(rounds/time)로 빈도 제어. 후속으로 `occupancy.maxPerCheckpoint` 별도 상한 도입 여지(이번엔 결정 3 준수로 미도입).
- **R2 판정 정확도**: rate는 LLM 박싱 recall에 의존(먼 줄 누락 시 total 과소). → 완화: 프롬프트 recall 우선 + `expected` 힌트 + 결정형 `occupancyRate` 교차검토.
- **R3 time granularity**: 라운드 경계 발화 — `intervalMs < 1라운드 소요`면 매 라운드 발화. 문서화됨.
- **R4 폴백 부재**: LLM 불가 시 occupancy 미저장(의도). 결정형 지표가 필요하면 기존 `occupancyRate` 사용.
- **R-a confidence 미영속**: 조회만 하는 현 범위에선 무해. 오버레이/신뢰도 표시 시 컬럼 추가 필요(후속).

## 9. 구성 설정 · 미수행 항목

### 9.1 config 기본값
- `config/llm.config.json`: `occupancy` 블록 추가(`enabled:true`, `prompt: config/prompts/occupancy.yaml`). `llmConfig.ts`의 DEFAULT는 `enabled:false`(미설정 시 비활성 = 기존 동작).
- `config/tools.config.json`: `capture.checkpointTriggerMode:"rounds"`, `checkpointIntervalMs:60000` 추가.
- **config 변경은 서버 재시작 필요**(config 파일은 nodemon 감시 밖 — 런타임 재로드 아님).

### 9.2 미수행 스모크(통과 위장 아님, 후속 필요)
- **실 vLLM/Ollama 점유율 판정**: 미수행. `occupancy.enabled=true`이나 실제 LLM 엔드포인트 가동 미확인 + 지침상 실 LLM 호출 금지. 유닛은 fake OpenAI 호환 HTTP 서버로 멀티모달 경로까지 모킹 검증. **실 연동 검증은 서비스 기동 후 별도 필요.**
- **Unity 카메라 실 프레임 정밀수집 1런**: 미수행(시뮬/실 카메라 미기동). CaptureJob 트리거·결선은 fake camera/vpd + monotonic 주입으로 결정적 검증.

### 9.3 후속 권고(이번 범위 밖)
- 뷰어 occupancy 표/오버레이 렌더(`getLatestOccupancy` + `spaces_json` box 활용). 현재는 라우트 조회만 가능.
- occupancy 이력 시계열 조회 라우트/그래프.
- `occupancy.maxPerCheckpoint` 안전 상한(대규모 프리셋 대비, R1).
- `confidence` 컬럼 영속화(R-a).

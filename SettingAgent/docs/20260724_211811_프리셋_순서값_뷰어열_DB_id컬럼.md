# 프리셋 순서값 도입 — 분석탭 '프리셋 키' 열 → '순서', DB `preset_info.id` 추가

- 작업일: 2026-07-24
- 브랜치: `worktree-preset-order-id` (워크트리 `.claude/worktrees/preset-order-id`)
- 요청: "프리셋 키 대신 순서(idx)를 넣어줘 (1부터 시작). DB preset_info 테이블도 id(순서값)를 넣어줘"

## 1. 설계

두 가지를 같은 규약("프리셋 순서값 = 1부터 시작")으로 맞춘다.

| 대상 | 기존 | 변경 |
|------|------|------|
| 뷰어 `분석` 탭 → `프리셋별 요약` 표 첫 열 | `프리셋 키` (`1:1`, `1:2` …) | `순서` (`1`, `2` …) |
| DB `preset_info` | (cam_id, preset_id) 복합 PK 만 존재 | `id INTEGER` 순서값 컬럼 추가 |

### 결정 사항

1. **`id` 는 파생값이다.** PK 는 `(cam_id, preset_id)` 그대로 두고, `id` 는 `(cam_id, preset_id)` **오름차순 1-based 순위**로 정의한다.
   - writer(`PresetInfoRow`)가 값을 주지 않는다 → `roiDbLoad` / `migrateToSettingDb` 등 기존 호출부 **무변경**.
   - 행 집합이 바뀔 때마다 스토어가 전체를 다시 매긴다(`renumberPresetInfo`) → "1부터 빈틈없이"가 항상 성립.
2. **UNIQUE 제약을 걸지 않는다.** 전체 재번호 UPDATE 중 일시적 중복이 생길 수 있고, 표시용 순서값에 제약이 필요하지 않다(불필요한 복잡도 배제).
3. **컬럼 선언 위치는 맨 뒤.** 구 DB 는 `ALTER TABLE ADD COLUMN` 으로 끝에 붙으므로, 신규 `CREATE` 도 끝에 두어 신규/마이그레이션 DB 의 컬럼 순서를 일치시켰다
   (`place_id` 는 이미 divergence 가 있음 — `presetInfoMigration.adversarial.test.ts` 의 '컬럼 순서 divergence' 케이스 참조).
4. **뷰어 '순서'는 표 행 번호**(`perPreset` 배열 인덱스 + 1). 산출물 `presets[]` 가 카메라·프리셋 오름차순이라 실제로 DB `id` 와 같은 값이 되지만,
   행 정렬을 새로 강제하지는 않았다(요청 범위 밖의 동작 변경 회피).

### 성공 기준

1. 표 첫 열이 `1,2,3…` 으로 표시되고 `프리셋 키` 열은 사라진다 — 소스 가드 테스트 통과.
2. 신규 DB·구 DB 모두 `preset_info.id` 가 `(cam_id, preset_id)` 순 1..N — 유닛테스트 통과.
3. 실가동 DB 사본 오픈 시 마이그레이션이 실제로 값을 채우고, 재오픈해도 동일(멱등).
4. 기존 테스트 전량 green, `tsc --noEmit` 0.

## 5. 후속 변경 — `id` 를 **첫 컬럼**으로 재배치 (요청: "id 가 제일 앞쪽(왼쪽)으로 배치해줘")

### 문제

`ALTER TABLE ADD COLUMN` 은 SQLite 에서 **항상 맨 뒤**에 붙는다. DB 뷰어(`GET /db/table/:name`)는
`PRAGMA table_info` 순서와 `SELECT *` 를 그대로 쓰므로 `id` 가 화면 오른쪽 끝에 나왔다.
SQLite 에는 컬럼 위치 변경 구문이 없어 **테이블 재작성**이 유일한 방법이다.

### 구현 (`SqliteStore.ts`)

- **`PRESET_INFO_COLS` / `PRESET_INFO_DDL(table)` 모듈 상수 신설** — 정본 컬럼 순서(`id` 우선)와 DDL 을 한 곳에 두고
  `CREATE TABLE` 과 재작성이 **같은 정의를 공유**한다(정의 이중화 방지).
- **신규 private `reorderPresetInfoColumns()`** — 현재 순서가 정본과 다르면 공식 절차로 재작성:
  `preset_info__new` CREATE → `INSERT … SELECT` 복사 → `DROP TABLE preset_info` → `RENAME`. 이미 정본이면 no-op(멱등).
  - `PRAGMA foreign_keys = OFF` : 자식(`slot_setup`)이 참조 중인 부모를 DROP 하기 위해 필요.
  - `PRAGMA legacy_alter_table = ON` : 이 순간 `slot_setup` 은 '없는 `preset_info`' 를 참조하는 상태라,
    끄면 `RENAME` 이 스키마 파싱 오류로 실패한다. 재작성 후 두 PRAGMA 모두 원복(`finally`).
  - 방어 가드 2개: 중단된 재작성 잔해 `DROP TABLE IF EXISTS preset_info__new`, 정본 컬럼이 하나라도 없으면 **손대지 않고 return**.
- 호출 위치: `migratePresetPosToInfo()` 말미(리네임·컬럼 보충 뒤). 신규 DB 는 `CREATE` 가 이미 정본이라 진입하지 않는다.

### 파생 효과 — 스키마 divergence 해소 (의도적 동작 변경)

재작성이 **정본 DDL** 로 테이블을 다시 만들므로, 그동안 "수용된 divergence" 로 문서화돼 있던 두 가지가 사라진다.

| 항목 | 이전(마이그레이션 DB) | 이후 |
|------|----------------------|------|
| 컬럼 순서 | `… updated_at, place_id, id`(신규와 다름) | 신규와 **동일**(`id` 우선) |
| `place_id` FK | **없음**(존재하지 않는 place_id 도 통과) | 신규와 동일하게 **강제** |

→ 이에 맞춰 `presetInfoMigration.adversarial.test.ts` 의 divergence 실증 2케이스를 **수렴 검증**으로 갱신했다.
실사용 경로는 `place_id` 를 항상 `PLACE_ID=1` 로 쓰고 `roiDbLoad` 가 `place_info(1)` 을 먼저 upsert 하므로 무해하다.

### 추가 테스트 (`presetOrderId.test.ts`, 6 → 8 케이스)

- 신규 DB 의 `PRAGMA table_info` 첫 컬럼이 `id` 이고 전체 순서가 정본과 일치.
- **재작성이 `slot_setup` 자식행·FK 를 보존**: 행 수 유지 · `PRAGMA foreign_key_check` 0건 · 부모 없는 `(1,9)` 슬롯은 여전히 거부.
- 구 DB 케이스에 "재오픈 후에도 정본 순서" 단언 추가(멱등).

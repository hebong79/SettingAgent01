# 프리셋 순서값 도입 — 분석탭 '프리셋 키' 열 → '순서', DB `preset_info.id` 추가

- 작업일: 2026-07-24
- 브랜치: `worktree-preset-order-id` (워크트리 `.claude/worktrees/preset-order-id`)
- 요청: "프리셋 키 대신 순서(idx)를 넣어줘 (1부터 시작). DB preset_info 테이블도 id(순서값)를 넣어줘"

## 1. 설계

두 가지를 같은 규약("프리셋 순서값 = 1부터 시작")으로 맞춘다.

| 대상 | 기존 | 변경 |
|------|------|------|
| 뷰어 `분석` 탭 → `프리셋별 요약` 표 첫 열 | `프리셋 키` (`1:1`, `1:2` …) | `순서` (`1`, `2` …) |
| DB `preset_info` | (cam_id, preset_id) 복합 PK 만 존재 | `id INTEGER` 순서값 컬럼 추가 |

### 결정 사항

1. **`id` 는 파생값이다.** PK 는 `(cam_id, preset_id)` 그대로 두고, `id` 는 `(cam_id, preset_id)` **오름차순 1-based 순위**로 정의한다.
   - writer(`PresetInfoRow`)가 값을 주지 않는다 → `roiDbLoad` / `migrateToSettingDb` 등 기존 호출부 **무변경**.
   - 행 집합이 바뀔 때마다 스토어가 전체를 다시 매긴다(`renumberPresetInfo`) → "1부터 빈틈없이"가 항상 성립.
2. **UNIQUE 제약을 걸지 않는다.** 전체 재번호 UPDATE 중 일시적 중복이 생길 수 있고, 표시용 순서값에 제약이 필요하지 않다(불필요한 복잡도 배제).
3. **컬럼 선언 위치는 맨 뒤.** 구 DB 는 `ALTER TABLE ADD COLUMN` 으로 끝에 붙으므로, 신규 `CREATE` 도 끝에 두어 신규/마이그레이션 DB 의 컬럼 순서를 일치시켰다
   (`place_id` 는 이미 divergence 가 있음 — `presetInfoMigration.adversarial.test.ts` 의 '컬럼 순서 divergence' 케이스 참조).
4. **뷰어 '순서'는 표 행 번호**(`perPreset` 배열 인덱스 + 1). 산출물 `presets[]` 가 카메라·프리셋 오름차순이라 실제로 DB `id` 와 같은 값이 되지만,
   행 정렬을 새로 강제하지는 않았다(요청 범위 밖의 동작 변경 회피).

### 성공 기준

1. 표 첫 열이 `1,2,3…` 으로 표시되고 `프리셋 키` 열은 사라진다 — 소스 가드 테스트 통과.
2. 신규 DB·구 DB 모두 `preset_info.id` 가 `(cam_id, preset_id)` 순 1..N — 유닛테스트 통과.
3. 실가동 DB 사본 오픈 시 마이그레이션이 실제로 값을 채우고, 재오픈해도 동일(멱등).
4. 기존 테스트 전량 green, `tsc --noEmit` 0.

### `SettingAgent/src/capture/SqliteStore.ts`

1. `CREATE TABLE IF NOT EXISTS preset_info (…)` 에 `id INTEGER` 를 `updated_at` 뒤에 추가(+ 규약 주석).
2. `migratePresetPosToInfo()` 에 멱등 블록 추가:
   ```ts
   if (!cols.includes('id')) {
     this.db.exec(`ALTER TABLE preset_info ADD COLUMN id INTEGER`);
     this.renumberPresetInfo();       // 기존 행의 NULL 채움
   }
   ```
3. **신규 private `renumberPresetInfo()`** — 상관 서브쿼리로 순위를 계산하는 단일 UPDATE:
   ```sql
   UPDATE preset_info SET id = (
     SELECT COUNT(*) FROM preset_info AS p2
     WHERE p2.cam_id < preset_info.cam_id
        OR (p2.cam_id = preset_info.cam_id AND p2.preset_id <= preset_info.preset_id)
   )
   ```
4. `upsertPresetInfo()` 트랜잭션 직후 `renumberPresetInfo()` 호출.

### 실가동 DB 사본 검증 (원본 무변경 — scratchpad 복사본 사용)

`SettingAgent/data/setting.sqlite` 사본을 `SqliteStore` 로 열어 확인:

```
BEFORE tables: place_info,camera_info,preset_pos,slot_setup,parking_evnt,sqlite_sequence,parking_slot
BEFORE preset_pos cols: cam_id,preset_id,sname,pan,tilt,zoom,updated_at
AFTER  cols: cam_id,preset_id,preset_name,pan,tilt,zoom,updated_at,place_id,id
┌────┬────────┬───────────┬─────────────┐
│ id │ cam_id │ preset_id │ preset_name │
├────┼────────┼───────────┼─────────────┤
│ 1  │ 1      │ 1         │ 'Preset 1'  │
│ 2  │ 1      │ 2         │ 'Preset 2'  │
│ 3  │ 1      │ 3         │ 'Preset 3'  │
└────┴────────┴───────────┴─────────────┘
REOPEN ids: [{"id":1,…},{"id":2,…},{"id":3,…}]   ← 멱등
```

즉 실가동 DB 는 아직 **구 `preset_pos` 단계**이며, 서버 재기동 시 `preset_pos → preset_info` 리네임 + `place_id` + `id` 가 한 번에 적용된다.

> 뷰어 화면은 이 저장소에 DOM 자동화(playwright/jsdom)가 없어, 기존 관례(`cameraKindSelect.test.ts` 등)대로 **소스·마크업 가드 테스트**로 검증했다.
> 라이브 확인은 서버 재기동 후 `분석` 탭 새로고침(뷰어 정적파일은 `Cache-Control: no-store`)으로 즉시 가능.

## 4. 영향도 분석

| 대상 | 영향 | 근거 |
|------|------|------|
| `PresetInfoRow` 타입 (`src/capture/types.ts`) | **없음** | `id` 는 파생값이라 writer 인터페이스에 넣지 않음 |
| `roiDbLoad.ts` / `tools/migrateToSettingDb.ts` | **없음** | `upsertPresetInfo` 시그니처 무변경, 재번호는 스토어 내부 |
| `slot_setup` FK `(cam_id, preset_id) → preset_info` | **없음** | PK·FK 정의 무변경. `id` 는 참조 대상이 아님 |
| DB 탭(`dbRoutes` 제네릭 조회) | 표시 컬럼 1개 증가 | `SELECT *` 기반이라 자동 반영 |
| `web/core.js` `analyzeArtifact()` | **없음** | `perPreset[].key` 유지 — 다른 소비처(경고 표시 등)도 그대로 |
| 점유율 표의 `프리셋 키` 열 | **없음** | 별개 표, 미변경 |
| 기존 DB 파일 | 오픈 시 1회 `ALTER` + 재번호 | 멱등 — 재오픈 no-op(테스트·실사본 확인) |
| 성능 | 무시 가능 | 재번호 대상 행 수 = 프리셋 개수(현재 3~5행) |

### 남은 참고 사항 (변경하지 않음)

- `Docs/MyThink/my_db_table.md` 의 `preset_info` 절에는 마스터가 이미 `- id : 프리셋 key 값` 을 추가해 둔 상태(메인 체크아웃 미커밋). 이번 구현의 정의는 **"1부터 시작하는 프리셋 순서값"** 이므로, 필요하면 그 줄을 `프리셋 순서값 ( 1부터 시작 )` 으로 다듬으면 정확해진다. 남의 미커밋 편집이라 이번 작업에서는 건드리지 않았다.
- `SettingAgent/x.json` 은 기존 테스트가 만드는 산출물(무관, 기존부터 untracked).

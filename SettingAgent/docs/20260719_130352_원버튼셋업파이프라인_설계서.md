## 0. 목표 (관찰형 성공기준)

정밀수집 페이지에서 **자동 체인 옵션을 켜고 [수집 시작] 한 번** 누르면:

1. N라운드 수집이 done 으로 끝나면
2. **자동으로 finalize**(setup_artifact.json + slot_setup DB + save/ 스냅샷) 되고
3. finalize 성공 시 **자동으로 센터라이징**(슬롯별 번호판 중심정렬·줌) 이 발화되어
4. 잡 완료 시 slot_ptz.json + slot_setup PTZ 컬럼 + centering 미러가 채워진다.

**실패 시 체인은 그 지점에서 중단**되고 단계·사유가 status 로 명확히 보고된다(위장 성공 금지). **옵트인** — 기존 수동 3버튼(수집/최종화/센터라이징) 흐름은 100% 보존, 회귀 0.

## 2. 핵심 결정: 오케스트레이션 위치 = **백엔드 (a)**

### 트레이드오프

| 관점 | (a) 백엔드 오케스트레이션 | (b) 프론트 체인 (capPoll done→capFinalize→calStart) |
|------|--------------------------|--------------------------------------------------|
| 최종 소비자(주차장관리 에이전트, MCP/헤드리스) | **재사용 가능** — REST 플래그 하나로 동일 체인 | 불가 — 브라우저 열려 있어야만 체인 동작 |
| 체인 생존성 | 서버 프로세스가 소유 — 탭 닫힘/새로고침 무관 | 새로고침·탭 이탈 시 체인 유실(무통보 중단) |
| 외과적 정도 | CaptureJob/PtzCalibrator 에 **옵셔널 완료 콜백 각 1개**(수 줄) + 신규 파이프라인 클래스 1개 | app.js 만 수정(백엔드 0줄) |
| 실패 보고 | 서버 status 로 일원화(폴링·로그 모두 관찰 가능) | cap-msg 텍스트뿐 — 헤드리스 관찰 불가 |
| 테스트 | vitest 로 상태 전이 결정형 검증 가능 | 브라우저 수동 확인만 |

### 결정

**(a) 백엔드.** 근거: 이 기능의 존재 이유가 "주차장관리 에이전트가 셋업을 원버튼으로 굴린다" 이므로 브라우저 의존 체인(b)은 목표 자체를 못 채운다. (b)의 유일한 장점(백엔드 무수정)도, (a)의 백엔드 수정이 "옵셔널 콜백 2줄 + 신규 클래스 1개" 수준으로 작아 상쇄된다.

세부 방식: **신규 라우트가 아니라 `POST /capture/start` 바디에 `autoChain?: boolean` 플래그 추가.** `/capture/run-all` 신규 라우트를 만들면 start 의 타깃 해석(presetProvider/mapFiles 폴백) 로직을 중복하거나 공유 함수로 뽑아야 한다 — 플래그가 외과적이다.

체인 상태 노출: capture status 를 오염시키지 않는 **신규 조회 라우트 `GET /capture/pipeline`** (아래 §4). CaptureStatus/CalibrateStatus shape 은 불변(회귀 0).

## 3. 아키텍처: `SetupPipeline` (신규, 인메모리 상태머신)

**신규 파일** `src/pipeline/SetupPipeline.ts` (~120줄). 범용 워크플로 엔진이 아니다 — **이 3단계 체인 전용**, 단계 하드코딩.

### 3.1 의존성 (전부 기존 객체 재사용 — Finalizer/PtzCalibrator 로직 무수정)

```ts
export interface SetupPipelineDeps {
  job: Pick<CaptureJob, 'getSnapshot'>;
  finalizer: Pick<Finalizer, 'finalize'>;
  calibrator: Pick<PtzCalibrator, 'start' | 'getStatus'>;
  store: Pick<SqliteStore, 'getSlotSetup'>;   // 커버리지 요약(전체 슬롯 수)용
  now?: () => string;
}
```

### 3.2 상태머신 (이 체인만, 발명 금지)

```
idle ──[/capture/start autoChain=true 성공]──▶ capturing
capturing ──[잡 done + dets>0]──▶ finalizing
capturing ──[잡 stopped]──▶ failed{stage:'capture', reason:'stopped(수동 정지)'}
capturing ──[잡 error]──▶ failed{stage:'capture', reason:'capture error'}
capturing ──[잡 done + dets==0]──▶ failed{stage:'finalize', reason:'검출 0건 — finalize 미실행(DB 보호)'}   ← F10 가드
finalizing ──[finalize 정상]──▶ calibrating (calibrator.start 발화)
finalizing ──[finalize throw]──▶ failed{stage:'finalize', reason:err.message}   (센터라이징으로 넘어가지 않음)
finalizing ──[LPD 타깃 0건]──▶ done{note:'센터라이징 스킵 — LPD 보유 슬롯 0'}   ← F6 가드(빈 slot_ptz.json 덮어쓰기 방지, calibrator.start 미호출)
calibrating ──[calibrator done]──▶ done (+커버리지 요약)
calibrating ──[calibrator error / start throw]──▶ failed{stage:'calibrate', reason}
(모든 종단 상태에서) ──[다음 /capture/start]──▶ idle 재무장(armed=autoChain)
```

- **armed 가 아니면(수동 수집) 모든 콜백은 no-op** — 수동 흐름 회귀 0의 구조적 보장.
- 실패는 그 단계에서 **정지**한다. 재시도·자동 복구 없음(추측성 기능 금지) — 사용자가 사유를 보고 수동 버튼으로 이어가면 된다.

### 3.3 공개 API

```ts
class SetupPipeline {
  /** /capture/start 성공 직후 라우트가 호출. autoChain=false 면 disarm+idle 리셋. */
  onCaptureStart(armed: boolean): void;
  /** CaptureJob 완료 콜백(신규 옵셔널 dep 로 배선). 내부 throw 0 — 잡을 절대 죽이지 않는다. */
  onCaptureFinished(status: 'done' | 'stopped' | 'error'): void;
  /** PtzCalibrator 완료 콜백(신규 옵셔널 dep 로 배선). */
  onCalibrateFinished(state: 'done' | 'error'): void;
  /** finalizing/calibrating 중이면 true — /capture/start 409 가드 소스. */
  isBusy(): boolean;
  getStatus(): PipelineStatus;
}
```

### 3.4 상태 shape (`GET /capture/pipeline` 응답)

```ts
export type PipelineStage = 'idle' | 'capturing' | 'finalizing' | 'calibrating' | 'done' | 'failed';

export interface PipelineStatus {
  armed: boolean;                       // 이번 run 에 자동 체인이 켜졌는가
  stage: PipelineStage;
  startedAt?: string;
  endedAt?: string;
  failure?: { stage: 'capture' | 'finalize' | 'calibrate'; reason: string };
  finalize?: { slots: number; globalCount: number };        // finalize 성공 시
  /** LPD 홀 정직 리포트(§6): 대상 targets / 전체 totalSlots / 미대상 uncovered */
  coverage?: { targets: number; totalSlots: number; uncovered: number };
  note?: string;                        // '센터라이징 스킵 — LPD 보유 슬롯 0' 등
}
```

수치 규약: PipelineStatus 는 카운트·문자열만(좌표 수치 없음) — round5 규약 비대상. 영속화 없음(인메모리 status 전용).

### 3.5 finalize 호출 상세

- `finalizer.finalize(job.getSnapshot(), {})` — **`logicOccupancy` 는 넘기지 않는다**(프론트 로직 점유 스냅샷은 브라우저 상태 소유 — 헤드리스 체인에서 없음). occupancyAgreement 는 best-effort 비교라 미부착이 정상 동작(Finalizer 기존 계약).
- 스냅샷 참조 안전성: F9 — finalize 진행 중 새 start 가 와도 스냅샷 배열은 오염 안 됨. 단 의미적 혼선을 막기 위해 `/capture/start` 라우트에 `pipeline.isBusy() → 409 { error: 'pipeline busy', stage }` 가드를 추가한다(§4).

### 3.6 커버리지 요약 산출 (§6 정직 리포트)

calibrate 발화 직전(= finalize 성공 직후, slot_setup 이 방금 새로 써진 시점):

```ts
const views = store.getSlotSetup();
const targets = expandPlateTargetsFromSlotSetup(views);   // slotPtzWriter 기존 export 재사용 — lpd 판정 중복 구현 금지
coverage = { targets: targets.length, totalSlots: views.length, uncovered: views.length - targets.length };
```

`targets.length === 0` 이면 **calibrator.start 를 부르지 않고**(F6 — 빈 slot_ptz.json 덮어쓰기 방지) `done + note` 로 종단.

## 5. 순서 가드 시너지 · 알려진 한계

### 5.1 stale 문제 없음 (자동 체인)
자동 체인은 **항상 "방금 finalize 한 slot_setup"** 으로 센터링한다(calibrating 전이는 finalize 성공 직후에만 발생). 수동 경로의 stale slot_setup 가드(수집만 다시 하고 최종화 없이 센터링 누르는 경우)는 **이번 범위 밖 — 후속**.

### 5.2 LPD 홀 (명시적 한계 — 근본 해결은 A2 후속)
센터라이징 대상은 lpd 보유 슬롯만(F5). 원거리 프리셋에서 LPD 미검출된 슬롯은 **slot_ptz 에 홀**로 남는다. 이번 파이프라인은 이를 **숨기지 않고 리포트**한다(coverage: 대상 N / 전체 M / 미대상 M−N — status·완료 메시지 양쪽). **근본 해결(앞면중심 prior 로 전 슬롯 커버 = A2)은 이번 범위 밖 후속** — `docs/20260719_123634_셋업PTZ_목표재정의_점유판정논의_필요기능정리.md` §3 A2, `docs/20260719_113423_VPD점유판정_취약점분석_대체방안_추천.md` 참조. A2 도입 시 SetupPipeline 은 무수정(타깃 펼침 함수만 교체되는 구조).

### 5.3 수동 개입 경합 (정직 실패)
armed 체인 진행 중 사용자가 수동 최종화/센터라이징을 누르면: 수동 calibrate 가 먼저 돌면 파이프라인의 `calibrator.start()` 가 'already running' throw → `failed{calibrate}` 로 정직하게 중단·보고된다. 잠금 확대(수동 버튼 비활성화 등)는 하지 않는다 — 옵트인 사용자가 수동 버튼을 동시에 누르는 시나리오에 방어 코드를 쌓지 않는다(발생 시에도 위장 성공 없이 사유가 보인다).

| T2 | **비무장**(onCaptureStart(false)) + done | finalize **미호출**, stage idle 유지(수동 흐름 무간섭) |
| T3 | armed + stopped / error | failed{capture}, finalize 미호출 |
| T4 | armed + done + **dets 0** | failed{finalize, '검출 0건…'}, **finalize 미호출**(F10 DB 보호) |
| T5 | finalize throw(커버리지 불일치) | failed{finalize, err.message}, calibrator.start **미호출** |
| T6 | LPD 타깃 0(전 슬롯 lpd=null) | done + note, calibrator.start **미호출**(F6 빈 파일 방지) |
| T7 | calibrator.start throw('already running') / onCalibrateFinished('error') | failed{calibrate} |
| T8 | 종단 상태에서 onCaptureStart(true/false) 재호출 | 리셋·재무장 정상 |
| T9 | onFinished 콜백 내부 throw | CaptureJob/PtzCalibrator 가 죽지 않음(흡수 확인 — 각 기존 테스트 파일에 1케이스씩) |

라우트(captureRoutes.test.ts 가산): autoChain=true → onCaptureStart(true) 호출됨 / `GET /capture/pipeline` shape / isBusy 시 start 409. **기존 테스트 전체 무수정 통과 = 회귀 0 증명**(콜백·플래그 전부 옵셔널).

## 8. 가정 (명시)

- A-1: 옵트인은 **요청 단위 플래그**로 충분(영속 설정 불필요). 기본 OFF.
- A-2: 수동 정지(stop)는 체인 취소로 간주 — failed{capture,'stopped'} 로 표기(별도 'cancelled' 상태 발명 안 함).
- A-3: 자동 체인 finalize 는 logicOccupancy 미전달(occupancyAgreement 미부착) — 기존 best-effort 계약상 정상.
- A-4: calibrate 대상 필터(slotIds)는 자동 체인에서 미사용(항상 전체 타깃) — 부분 센터링은 수동 경로 소유.

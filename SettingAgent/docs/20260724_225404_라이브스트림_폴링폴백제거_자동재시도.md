### 1.2 근본원인(로그 기반, 리더 확정)
3fps 폴링을 켜는 코드 경로는 `web/app.js` `fallbackToPolling()` 단 하나였고, 트리거는 `frame.onerror`. 이 폴백이 **편도(one-way)** 였다:

- 스트림이 한 번 실패하면 `liveMode='poll'` + `setInterval(333ms)`로 **영구 고착**.
- 스트림 자동 복귀 경로가 없음(사용자가 시작 버튼을 다시 누르거나 cam/preset을 바꿔야만 복귀).
- 실패 시 **백오프가 없어** 서버가 죽어 있어도 333ms 간격으로 계속 재시도(위 로그의 33·36 패킷/10초).
- 스냅샷 1프레임 = `mode=manual` 요청 1건이 Unity 측에서 `cam.setPTZ` + `cam.captureJPG` **2회 RPC**를 유발(`ms:2~15`/`ms:58~76` 교대 패턴으로 로그에서 확증) → 3fps 폴링이 실질 6.3 req/s 트래픽으로 배가.

2026-07-09 도입 당시(`docs/20260709_141227_*.md`)에는 이 폴백이 "스트림 미지원 소스(RealPtzSource=Hucoms)에 대한 안전망"으로 정당했다. 그러나 실제 등록 소스가 전부 스트림을 지원하게 된 지금은, 이 안전망이 **일시 장애(서버 재기동·Unity 다운)에도 걸려 영구 고착되는 결함**으로 작용했다. 이번 작업은 그 문서가 도입한 폴백 자체를 폐지한다(20260709 문서는 과거 기록물이라 수정하지 않음).

## 3. 새 설계 — 스트림 지수 백오프 자동 재시도

### 3.1 순수 함수(`web/core.js`, DOM/타이머 미참조)

```js
const STREAM_RETRY_BASE_MS = 1000;
const STREAM_RETRY_FACTOR = 2;
const STREAM_RETRY_MAX_MS = 30000;

export function nextStreamRetryDelay(prevMs) { /* 1000→2000→4000→8000→16000→30000→30000… */ }
export function streamRetryLabel(attempt, delayMs) { /* "스트림 끊김 — N초 후 재연결 (M회째)" [+ 상한 안내] */ }
```

- 상수는 export하지 않는 모듈 내부 고정값(설정 UI 추가 금지 — CLAUDE.md §2). 상한 도달 시 `streamRetryLabel`이 " · 연결 실패가 계속됩니다 — 서버/카메라 상태를 확인하세요"를 덧붙인다.

### 3.2 DOM 결선(`web/app.js`)

- 상태: `liveMode`('off'|'stream'), `streamRetryTimer`(대기 중 setTimeout 핸들), `streamRetryDelay`(직전 지연), `streamRetryAttempt`(연속 실패 횟수).
- `cancelStreamRetry()` — 대기 타이머 `clearTimeout` + 백오프 상태 0 리셋 + `#live-status` 문구 비움. **타이머 취소의 단일 지점.**
- `onStreamLoad()` — 정상 지속 프레임은 무동작, 재시도 이력이 있으면 `cancelStreamRetry()`로 백오프 리셋(첫 프레임 도착 = 복구).
- `onStreamError()` — 이미 대기 중이면 무동작(중복 예약 금지). 아니면 `nextStreamRetryDelay`로 지연 갱신, `#live-status`에 `streamRetryLabel` 표시, `setTimeout` 예약 → 콜백은 `liveMode !== 'stream'` 조기 반환(2중 가드) 후 `frame.src = ${streamUrl()}&_r=${Date.now()}`(캐시버스터는 **재시도 경로에만** 부가).
- `connectStream()` — `startLive`/`reconnectLiveIfActive`가 공유하는 (재)연결 공통부. `cancelStreamRetry()` → `snapshot.abort()` → `liveMode='stream'` → `onerror/onload` 결선 → `frame.src` 대입 → `drawRoiOverlay()` 1회.

### 3.3 구현 중 보정(§3, qa 관찰 → 리더 승인)
`connectStream()`이 캐시버스터 없이 `frame.src = streamUrl()`을 대입하면, **최초 실패 직후 첫 재시도(1초) 발화 전에 사용자가 `시작`을 다시 누르는 좁은 창(<1s)**에서 대입값이 현재 `src`와 완전히 동일해 브라우저가 재요청을 생략할 수 있고, 동시에 `cancelStreamRetry()`가 기존 예약을 취소해 "재시도도 재발화도 없는 정지 상태"가 될 수 있었다(이번 작업이 없애려는 고착과 같은 부류). 수정: `connectStream()` 진입 시 `cancelStreamRetry()`로 상태를 밀기 **전에** `retrying = !!streamRetryTimer || streamRetryAttempt > 0`을 캡처해, 재시도/실패 이력이 있었던 경우에만 캐시버스터를 붙인다. 정상 경로(성공 중 cam/preset 변경 등)의 URL 형태는 그대로 유지되어 기존 "동일 URL 재대입은 무해한 race" 성질(20260709 문서 §race)이 보존된다.

### 3.4 개명 — `createStreamLoop` → `createSnapshotFetcher`
타이머 제거 후 남는 것은 1회성 fetch + abort뿐이라 "loop"라는 이름이 사실과 어긋나(설계자 판단, CLAUDE.md §3 — 자기 변경으로 생긴 오해 정리). `tick()`/내부 백프레셔·revoke·onPtz 로직은 **한 줄도 변경되지 않았다**. `stop()`은 타이머 정리부를 뺀 `abort()`로 개명. `app.js` 호출부 4곳, `core.d.ts`의 `StreamLoopDeps`→`SnapshotFetcherDeps`, `StreamLoop`→`SnapshotFetcher`가 동반 갱신되었다.

### 3.5 UI 표시
삭제되는 `fps` 입력 자리에 `<span id="live-status" aria-live="polite">`를 신설(`web/index.html`). `web/app.css`에 `#ptz-control-status` 패턴을 따른 3줄(`color: var(--muted); font-size: 12px; line-height: 1.35;`) 추가. 문구는 전부 `streamRetryLabel()`(순수 함수) 산출값을 `textContent`로 대입만 한다 — "무표시 열화"를 피한다(리더 요구).

## 6. 운영 주의사항

- **`web/` 정적 자산은 nodemon 감시 밖**(`nodemon.json`: `watch:["src"], ext:"ts,json"`). `web/*.js|html|css` 변경은 서버 재기동으로 반영되지 않고 **브라우저 하드리로드(Ctrl+Shift+R)가 필수**다.
- **이미 열려 있던 탭은 하드리로드 전까지 구 코드(3fps 폴링 폴백 포함)로 계속 동작한다.** 수정 확인 전 반드시 모든 열린 뷰어 탭을 하드리로드할 것.
- **사용자 가시 변경**: 뷰포트 하단 `fps` 입력이 사라진다(라이브 프레임레이트는 서버 스트림이 결정하며 사용자 조절 불가였던 점은 폴링 폐지로 자명해짐). 같은 자리에 재시도 상태 문구(`#live-status`)가 나타난다.
- **트래픽**: 최악(스트림 영구 실패) 기준 30초당 1요청 — 기존 3.3~6.3 req/s 대비 약 200배 감소.

## 7. 남은 한계 (위장 없이 명시)

1. **브라우저 MJPEG semantics 미검증.** `<img>`의 `multipart/x-mixed-replace` 프레임별 `onload` 발화 가정, 동일 URL 재대입 시 재요청 생략 여부는 node 유닛 테스트로 재현 불가 — 실브라우저 확인 필요(qa 03 §4 명시).
2. **스트림 미지원 소스는 재시도만 반복한다.** `img.onerror`는 HTTP 상태(501 STREAM_UNSUPPORTED vs 일시 장애)를 구분하지 못해, 폴백이 사라진 지금은 미지원 소스에서 30초 간격 재시도 + 안내 문구로 수렴할 뿐 복구되지 않는다. 리더가 수용한 트레이드오프이며, 현재 등록된 4개 소스(`simulator-1/2`=http-mjpeg, `real-camera-1/2`=rtsp-ffmpeg)는 전부 스트림을 지원해 즉시 영향은 없다.
3. **무음 정지(silent stall)**: 200 응답 후 상류가 조용히 멈추면 `onerror`가 발화하지 않아 자동 재시도가 걸리지 않는다(사용자가 `시작` 재클릭으로 회복). 워치독 타이머는 범위 밖(폴링 재도입 성격 — 필요 시 별건).
4. **라이브 스모크(Unity 기동/정지 실측) 미수행.** 관찰형 항목(정지 상태에서 시작 → 문구 증가 표시·`/snapshot` 0건, Unity 기동 → 문구 자동 소멸, 대기 중 정지 → 재요청 0건)은 vitest 밖이라 이번 검증에서 실행되지 않았다 — 리더/사용자의 실측이 필요하다.
5. **실제 DOM 이벤트 결선(버튼 클릭·`<img>` 실이벤트·`app.css` 스타일 적용)은 jsdom 미사용 환경이라 함수 실행까지만 검증되고, "버튼에 실제로 연결돼 있는가"는 텍스트 존재 확인으로 갈음했다.

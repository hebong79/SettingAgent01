## 1. 판정 규칙과 그 근거 (이 문서의 핵심)

### 1.1 채택한 규칙

```
band(rect)      = { x, y: y + h·(1 − 0.25), w, h: h·0.25 }              // 접지 근사 밴드 = bbox 하단 25%
onPlace(rect,P) = convexIntersectionArea(rectCorners(band), P) / area(band) ≥ 0.15
keep(rect)      = polys.some(P => onPlace(rect, P))                     // 전 폴리곤 OR
```

기하는 **전부 기존 유틸 재사용** — `domain/polygon.ts` 의 `rectCorners`·`convexIntersectionArea`, `domain/geometry.ts` 의 `area`. **신규 기하 코드 0줄.**

### 1.2 왜 "중심점 ∈ 폴리곤" 규칙(Finalizer 선례)을 기각했는가

`Finalizer.ts:237-241` 은 주차면 **배정**(이 클러스터가 *어느* 칸인가)을 번호판 중심 우선 + 차량 bbox 중심 폴백으로 푼다. 같은 규칙을 이 **필터**(주차면 *위*인가)에 재사용하는 것이 자연스러워 보이지만, **기각했다.**

**설계 시점의 논거(FP)**: VPD rect 는 **지붕까지 포함한 axis-aligned bbox** 이고 주차면 ROI 는 **바닥 quad** 다. 지면 위 높이 z 의 점은 이미지에서 카메라 nadir 로부터 `H/(H−z)` 배 바깥으로 밀려 보인다(H≈8m, 차량중심 z≈0.75m → **약 +10%**). nadir 로부터 25m 지점이면 **약 2.5m = 주차면 정확히 한 칸** 어긋난다. 그래서 **통로 통행차의 중심이 뒷줄 주차면 폴리곤 위로 투영**된다 → 모드 A 의 1차 목적(통행차 배제) 자체를 달성하지 못한다.

**리더 실측이 더 강한 이유를 드러냈다 — FN 이 치명적이다** (`05_leader_empirical.md` §3, 실제 프레임 `data/refframes/cam1_p{1,2,3}.jpg` + 실제 VPD 서비스 호출 + 실제 `PtzCamRoi.json`):

| 프리셋 | 접지밴드 규칙(채택) | 중심점 규칙(기각) |
|---|---|---|
| preset1 | 주차차 **7대 유지** | 주차차 **0대 유지 — 7대 전량 누락** |
| preset2 | 주차차 **6대 유지** | 주차차 **0대 유지 — 6대 전량 누락** |
| preset3 | 4대 유지 | 3대 유지(1대 누락) |

렌더(`onplace_cam1_p1.png`)에서 **bbox 중심(마젠타 점)이 전부 차체 위**에 찍히고 **바닥 ROI(청록) 밖**에 있었다. 차량 bbox 중심은 구조적으로 항상 바닥면 위쪽이다.

> **중심점 규칙을 썼다면 모드 A 가 주차차를 거의 전부 버렸을 것이다.** 이것이 규칙 선택의 결정적 근거다.

### 1.3 대안 비교표

| 대안 | 주차차 누락(FN) | 통행차 통과(FP) | 판정 |
|---|---|---|---|
| (a) bbox 중심 ∈ 폴리곤 (Finalizer 선례) | **치명적** — 실측 preset1/2 에서 **전량 누락** | **치명적** — 통행차 중심이 뒷줄 면에 투영 | **탈락** |
| (b) 전체 rect ∩ 폴리곤 면적비 | 낮음 | **치명적** — 통행차 차체가 뒷줄 면을 가려 큰 겹침 | **탈락** |
| (c) 하단 중앙 1점 ∈ 폴리곤 | 중간 — 임계 없는 하드 경계 | 낮음 | 차선 |
| **(d) 하단 밴드 겹침 면적비 (채택)** | **낮음** — 밴드가 폭을 가져 경계 민감도 완화 | **낮음** — 밴드는 접지 영역이라 통로 아스팔트 위 | **채택** |

### 1.4 왜 임계값에 둔감한가 — 배정이 아니라 필터다

판정이 **전 폴리곤 OR** 이다. 어느 칸인지 *배정*하는 게 아니라 주차면 위인지 *필터*하므로, 원근 오차로 밴드가 **옆 칸**에 걸쳐도 결과는 여전히 `keep` = 정답이다. 드롭되는 것은 **어느 주차면과도 겹치지 않는 차** = 통로·진출입로 차량뿐이다.

### 1.5 Finalizer 는 건드리지 않았다

모드 A 에서는 필터가 **상류**(검출 직후)에 있으므로, Finalizer 에 도달하는 검출은 이미 전부 주차면 위다 → 차량중심 폴백의 FP 위험이 **필터에 의해 이미 제거**된다. Finalizer 규칙 교체는 요청 범위 밖의 **점유 배정 동작 변경**이며 기존 테스트가 그 동작을 단언 중이다. → **후속 과제 F-2a** (§9).

## 2. 임계값 실측 검증 (계획 §7-1 의 "미봉인 한계" 해소)

유닛테스트는 규칙의 *논리*만 봉인한다. 임계값 `0.25`/`0.15` 의 *적절성*은 유닛테스트가 증명할 수 없다. 리더가 **실제 프레임·실제 VPD·실제 폴리곤**으로 실측했다(`05_leader_empirical.md` §4).

| 구분 | 접지밴드 ∩ 주차면 겹침 비율 |
|---|---|
| **유지된 차량**(전 프리셋) | **0.409 ~ 0.637** |
| **제외된 차량**(전 프리셋) | **정확히 0.000** |
| 임계값 `ON_PLACE_MIN_OVERLAP` | **0.15** |

**경계에 걸친 차량이 한 대도 없다.** 유지/제외가 `0.000` 과 `0.409` 사이의 **넓은 빈 구간**으로 분리된다 → 임계값 0.15 는 이 데이터에서 **강건**하다(0.05~0.35 어디로 옮겨도 결과 동일). 밴드비 0.25 도 같은 여유를 갖는다.

## 3. 신규 모듈 — `src/capture/onPlaceFilter.ts` (50줄, 순수)

```ts
/** 접지 근사 밴드 = bbox 하단 25%. */
export const GROUND_BAND_RATIO = 0.25;
/** 밴드 면적 대비 주차면 겹침 하한. */
export const ON_PLACE_MIN_OVERLAP = 0.15;

/** 차량 bbox 의 접지 근사 밴드(하단 GROUND_BAND_RATIO 스트립). */
export function groundBand(rect: NormalizedRect): NormalizedRect;

/**
 * 접지 밴드가 주차면 폴리곤들 중 하나라도 임계 이상 겹치는가(OR).
 * polys 빈 배열 / 밴드 면적 0(퇴화 rect) → false.
 */
export function isVehicleOnPlace(
  rect: NormalizedRect,
  polys: readonly (readonly NormalizedPoint[])[],
): boolean;

/**
 * 모드 A 필터. VehicleBox·DetectVehicle 양쪽에 쓰이도록 구조적 제네릭.
 * polys 가 null/빈 배열이면 강등: 전량 통과 + degraded=true(드롭 금지).
 */
export function filterVehiclesOnPlace<T extends { rect: NormalizedRect }>(
  vehicles: readonly T[],
  polys: readonly (readonly NormalizedPoint[])[] | null | undefined,
): { kept: T[]; filteredOut: number; degraded: boolean };
```

**설계 노트**
- 임계값 2개는 **모듈 named const** 다(`Aggregator.ts` 선례 — `tools.config` 미승격, 설정 노출 안 함).
- `filterVehiclesOnPlace` 는 **구조적 제네릭** `T extends { rect }` — `CaptureJob` 의 `VehicleBox` 와 `detectPipeline` 의 `DetectVehicle` 두 타입에 **같은 함수 하나**를 쓴다(규칙 이중구현 방지).
- 퇴화 rect(`w=0`/`h=0`) → 밴드 면적 0 → `false`(throw 없음).

## 5. 강등 정책 — 주차면 폴리곤 부재 시 "전량 통과"

`placeRoiFile` 미설정 / 파일 없음 / 파싱 실패 / **해당 프리셋 주차면 0개** → 필터 기준이 없다.

> **전량 통과(모드 B 로 강등) + `reason` 포함 warn 로그 + status/summary 에 사유 노출.**

**전량 드롭을 금지한 이유**: 기준이 없다는 이유로 드롭하면 **데이터가 조용히 사라진다**(최악의 실패 모드 — 사용자는 "차가 없었다"와 구분할 수 없다). HANDOFF §2-3 **"조용한 폴백 제거"** 원칙 — `fovBaseV` 버그가 정확히 조용한 폴백 경로에 숨어 있었다. 폴백은 반드시 드러낸다.

**강등 입도 = 프리셋 단위.** `placeRoi.ts:79` 는 주차면이 1개 이상인 프리셋만 `byPreset` 에 키를 넣는다 → 키 부재 = "이 프리셋엔 ROI 없음". 파일 전체가 없으면 전 프리셋 강등, 특정 프리셋만 ROI 가 없으면 **그 프리셋만** 강등(다른 프리셋의 필터는 정상 동작).

| 상황 | reason 문자열 |
|---|---|
| `placeRoiFile` 미설정 / 로드 실패 | `주차면 파일 없음/로드 실패` |
| 해당 프리셋 주차면 0개 | `프리셋 {cam}:{preset} 주차면 0개` |

- `CaptureJob`: `onPlaceDegraded` 는 **최초 사유 1건 보존**, `warn` 은 **프리셋 키당 1회**(`degradeWarned` 가드 — 라운드마다 로그 폭주 방지).
- `runDetect`: 강등 사유는 **호출측이 `OnPlaceOpts.degradeReason` 으로 주입**한다. `runDetect` 는 `polys=null` 하나만 보고는 "파일이 없다"와 "이 프리셋에 주차면이 0개다"를 구분할 수 없고, 그 지식은 호출측(라우트의 `place` 객체)에만 있기 때문이다. 미지정 시 `'주차면 폴리곤 없음'` 으로 폴백 → 계약 불변.
  - *(이는 구현자가 계획 §4.4 에서 의도적으로 벗어난 유일한 지점이며, 그 정당성은 QA 케이스 "파일은 있으나 프리셋 9 주차면 0개 → `'프리셋 1:9 주차면 0개'`" 로 확인됐다. 계획대로였다면 정상 파일에서도 "주차면 파일 없음"이라는 **거짓 사유**가 UI 에 떴을 것이다.)*

### 10.2 설계 결정을 회귀로부터 봉인한 테스트 (★)

QA 의 통행차 케이스는 단순히 "통행차가 걸러진다"를 보지 않는다. **중심 규칙이었다면 이 차가 통과했을 것**임을 같은 테스트 안에서 **먼저 증명**한다.

```
BACK_ROW(뒷줄 바닥 quad) = (0.30,0.30) (0.56,0.30) (0.58,0.45) (0.28,0.45)   // 통로 = y > 0.45
PASSING(통행차 bbox)     = x 0.32~0.52, y 0.28~0.60   // 지붕 0.28(뒷줄 위) ~ 접지 0.60(통로)

① center(PASSING) = (0.42, 0.44) → pointInPolygon(BACK_ROW, center) === true   ← 중심규칙: keep(오답)
② groundBand(PASSING).y = 0.52 > 0.45                                          ← 접지 밴드는 통로
③ isVehicleOnPlace(PASSING, [BACK_ROW]) === false                              ← 채택안: drop(정답)
```

①이 명시적 단언으로 고정돼 있어, 누군가 규칙을 `center ∈ polygon` 으로 "단순화"하면 ③이 **즉시 실패한다.** 다중 폴리곤 OR 도 같은 방식으로 **"자기 칸만 주면 drop, 옆 칸을 함께 주면 keep"** 을 대조 단언해 *배정이 아니라 필터*임을 봉인했다.

**카메라 호출 절감도 대조로 증명**: 모드 A → `requestImage` **1회**(base 만) / 같은 입력 모드 B → **5회**(base + 통행차 zoom 재시도 4회).

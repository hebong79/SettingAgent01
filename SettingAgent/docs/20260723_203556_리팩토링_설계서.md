## 1. 개요 · 목표 · 성공기준

### 1.1 배경

SettingAgent TypeScript 코드베이스에 150줄을 초과하는 거대 함수 6건과, 라우트 계층에 반복되는 보일러플레이트(Zod 검증→400, 소스카메라 해석, 파일에러 응답 등)가 누적되어 있다. 특히 `platePtz.ts`의 rung 루프(233줄, 5개 관심사 혼재, 9개 반환점)는 코드베이스 최악의 가독성 핫스팟이다.

### 1.2 리팩토링 목표 (마스터 요청 4항)

1. **소스 최적화** — 중복 제거로 전체 코드량 축소
2. **재사용 가능성 향상** — 공용 헬퍼(`routeHelpers.ts` 등) 추출
3. **복잡도 제거** — 깊은 중첩·다중 관심사 혼재 함수를 알아보기 쉽게 분해
4. **함수 본문 ~150줄 이상 → 함수화 정리**

### 1.3 성공기준 (검증 가능 목표)

| # | 기준 | 검증 방법 |
|---|------|----------|
| S1 | 대상 파일에서 150줄 초과 함수 **0건** | 함수별 줄수 측정 스크립트 또는 수동 확인 |
| S2 | `tsc --noEmit` **에러 0** | `npm run typecheck` (또는 `npx tsc --noEmit`) |
| S3 | 기존 vitest **전량 green** (현재 2540 green 기준 무회귀) | `npx vitest run` |
| S4 | **공개 export 시그니처 불변** — `registerCaptureRoutes(app, deps)`, `registerViewerRoutes(app, deps)`, `registerCalibrateRoutes(app, deps)`, `PlatePtz`/`Finalizer` 공개 메서드, ground 모듈 공개 함수의 이름·매개변수·반환타입 전부 동결 | 제외 파일(`server.ts` 등) diff 0줄 + tsc 통과로 자동 검증 |
| S5 | 동작 불변 — 응답 스키마·에러 메시지·'정직성 관용구'(reason 문자열) 보존 | 기존 라우트/모듈 테스트 green + QA 라이브 확인 |

동작(behavior)을 바꾸는 변경은 이 리팩토링 범위에 **없다**. 순수 구조 개선만 수행한다.

## 3. 동시성 안전성 분석 (다른 세션과의 충돌 방지)

### 3.1 현재 동시 작업 상황

- 본 작업: 격리 워크트리 `worktree-work-20260723b`에서 진행.
- 동시 진행 중인 다른 세션: main 워킹트리(`server.ts` 수정 중), analyze-fill-check 세션(`server.ts`, `web/app.js`, `test/precisePreciseProgress.test.ts`, `src/setup/artifactFromSlotSetup.ts` 수정 중).

### 3.2 충돌 없음 근거

1. **라우트 파일의 유일 호출처는 `server.ts`다.** `registerCaptureRoutes(app, deps)`, `registerViewerRoutes(app, deps)` 등의 **공개 시그니처를 그대로 유지**하면 `server.ts`를 수정할 필요가 전혀 없다. git 충돌은 파일 단위로 발생하므로, 제외 파일을 건드리지 않는 한 다른 세션과 **파일 단위 충돌이 원천적으로 없다**.
2. **깊은 리팩토링 대상은 전부 리프 모듈이다.** `platePtz.ts`(외부 임포터: PtzCalibrator + 테스트뿐), `contact.ts`, `frameCuboids.ts`, `groundModel.ts`(임포터 3), `Finalizer.ts` — 다른 세션이 편집 중인 파일에서 이들을 임포트하더라도 공개 시그니처가 동결되므로 병합 시 컴파일 영향 없음.
3. **신규 파일(`routeHelpers.ts`)은 순수 추가**라 어떤 세션과도 겹치지 않는다.

### 3.3 지켜야 할 불변식 (구현자·QA 필수 준수)

- **불변식 1 — 공개 export 시그니처 동결**: 모든 대상 파일의 기존 `export` 함수/클래스/타입의 이름·매개변수·반환타입을 변경하지 않는다. 새 추출 함수는 모듈 내부(비export) 또는 신규 파일의 신규 export로만 추가한다.
- **불변식 2 — 제외 파일 미변경**: §2.2의 4개 파일 diff는 최종 커밋에서 정확히 **0줄**이어야 한다. 각 커밋 전 `git status`/`git diff --stat`으로 확인한다.
- **불변식 3 — 테스트 파일 중 `test/precisePreciseProgress.test.ts`는 신규 테스트 추가 대상에서도 제외**한다(다른 세션 편집 중). 신규 테스트는 별도 파일로 만든다.
- **불변식 4 — 응답 계약 동결**: 라우트 응답의 상태코드·JSON 필드·에러 메시지 문자열은 바이트 단위로 보존한다(헬퍼 추출 시 메시지를 매개변수로 전달).

## 4. 리팩토링 항목 상세 — 긴 함수 6건 + 서브임계 4건

각 항목: 현재상태 → 분할설계(새 함수 시그니처) → 검증방법. 새 추출 함수는 별도 명시가 없으면 **같은 파일 내 비export 함수**(또는 private 메서드)다.

### 4.2 `src/viewer/routes.ts:101` `registerViewerRoutes` — 325줄

**현재상태**: 카메라 핸들러들이 `safeParse→400 / pickSource→400 / try→502(CameraApiError)` 스캐폴드를 핸들러마다 반복(safeParse ~9회, CameraApiError→502 ~6회).

## 부록: MCP 도구 vs LLM 두뇌 경계 판단

본 리팩토링은 **결정형 코드의 구조 개선**만 다룬다. 센터라이징 줌 사다리·PTZ dither 루프 등 실시간·수치반복 로직은 결정형 도구 영역에 그대로 남으며, LLM 두뇌 경계에는 어떤 변경도 없다(경계 이동 없음).

## 부록: 미해결 / 가정 사항

1. 추출 함수의 정확한 매개변수 타입은 구현 시 실제 지역변수 타입으로 확정한다(본 설계서의 시그니처는 제안이며, "새 공개 타입 도입 금지" 원칙 내에서 조정 가능).
2. viewer/routes의 핸들러 중 스키마가 없는 것이 있으면 `withSource` 오버로드 형태는 구현자가 최소로 결정한다.
3. `index.ts:43`의 resolveSourceCamera 적용은 선택(부팅 경로 리스크 격리) — 보류해도 성공기준에 영향 없음.
4. CaptureJob `maybeUpdateCuboids` 분리는 선택 항목 — 5단계에서 시간·리스크 판단 후 구현자가 결정한다.

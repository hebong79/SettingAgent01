## 1. 배경 · 문제(마스터 요구 원문 기준)

> "정밀수집이 끝난후에 vpd, lpd, 점유영역 박스가 유지 되어야한다. 지금은 정밀수집이 끝나면 사라진다. 그리고 DB 테이블에 존재하면 항상 출력이 가능해야 한다.(체크박스 플래그에 의해 show/hide)"

두 가지 증상·요구가 확인되었다.

1. **종료 후 박스 소실(과교정)**: 직전 작업("센터라이징 화면갱신 + 정밀수집 종료 화면 리셋")이 정밀수집 종료(`done`/`stopped`/`error`) 시 `resetOverlayDisplay()`를 호출해 `detectByKey`/`occComputeByKey`/`vcuboidByKey`를 통째로 **삭제**했다. 당시 목표는 "얼어붙은 마지막 프레임을 라이브 배경으로 되돌린다"였는데, 구현이 오버레이 데이터까지 함께 지워 VPD(차량)·LPD(번호판)·점유영역 박스가 종료와 동시에 사라지는 **과교정**이 발생했다.
2. **DB 소스 미렌더**: DB `slot_setup` 테이블에는 `vpd_bbox`/`lpd_obb`/`occupy_range` 컬럼이 이미 존재하고 `Finalizer`가 매칭 차량(hit)이 있을 때마다 저장하지만(`src/capture/Finalizer.ts:217-219`), 웹은 `GET /capture/slots`(`SqliteStore.getSlotSetup()`, `src/capture/SqliteStore.ts:211`)가 반환한 값을 `state.parkingSlotsByKey`에 담아 **목록(슬롯 리스트)에만** 사용했고 캔버스 오버레이로는 그리지 않았다. 즉 reload/탭 재진입처럼 라이브 검출·점유 데이터가 없는 시점에는 DB에 값이 있어도 화면에 아무 박스도 나타나지 않았다.

## 2. 변경 상세 (`web/app.js` 단일 파일, 서버측 무변경)

### A. 종료 리셋 원복 — 과교정 제거

`capPoll()`의 활성→종료 전환 분기(`wasActive && (done|stopped|error)`)에서 직전 작업이 추가했던 두 줄을 제거했다.

```js
// 제거됨: state.finalizeOccSnapshot = state.occComputeByKey;
// 제거됨: resetOverlayDisplay();
startLive(); // 유지 — 얼어붙은 마지막 프레임을 라이브 배경으로 교체(직전 작업의 원래 취지)
```

- `startLive()`만 유지한다. 오버레이(검출·점유·육면체)는 캔버스 위 별도 레이어이므로, 라이브 MJPEG 배경이 바뀌어도 `detectByKey`/`occComputeByKey`/`vcuboidByKey`가 보존되면 박스는 그 위에 그대로 남는다.
- 이에 따라 고아가 된 코드를 함께 원복(외과적 정리, CLAUDE.md 규칙3 — 자신의 변경으로 발생한 orphan만 제거):
  - `state` 리터럴의 `finalizeOccSnapshot: {}` 필드 제거.
  - `capStart()`의 `state.finalizeOccSnapshot = {};` 초기화 줄 제거.
  - `buildFinalizeOccupancy()`의 스냅샷 폴백 제거 → 원래대로 `Object.entries(state.occComputeByKey).map(...)` 직접 참조.
- `resetOverlayDisplay()` 함수 자체는 "표시 초기화"(`#roi-clear`) 버튼이 여전히 사용하므로 유지(고아 아님).
- 센터라이징 프레임 폴(`calFrameTick`/`calFrameTimer`)·정밀수집 프레임 폴(`capFrameTick`)·서버측 코드는 이번 변경에서 전혀 건드리지 않았다.

### B. DB(`parkingSlotsByKey`) 소스 오버레이 상시 렌더 — 라이브 없을 때의 폴백

이중 렌더 회피 규칙: **프리셋 키(`${camId}:${presetId}` = `currentFrameKey()`) 단위로 라이브(`detectByKey`/`occComputeByKey`)가 있으면 라이브를 그리고, 없으면 DB로 폴백한다.** 같은 프리셋에서 두 소스가 동시에 그려지는 경우는 없다.

1. **`drawDetectOverlay(ctx)`** — `state.detectByKey[key]`가 없으면 신설 헬퍼 `drawDbDetect(ctx, rows, showVehicle, showPlate)`를 호출하고 반환.
2. **`drawDbDetect(ctx, rows, showVehicle, showPlate)`(신설)** — `state.parkingSlotsByKey[key]` 행을 순회하며, `#roi-vehicle` 체크 시 `row.vpd`(non-null)를 `toPixel()`로 변환해 청록(`#00e5ff`) `strokeRect`, `#roi-plate` 체크 시 `row.lpd`(non-null)를 `drawPlateQuad(ctx, row.lpd, false)`로 노랑 quad 렌더. null 필드는 skip. 읽기표시 전용(선택 하이라이트·드래그 핸들 없음 — 라이브 검출 결과와 달리 편집 대상이 아님).
3. **`drawOccupancyOverlay(ctx)`** — `const hasLive = (occ?.spaces ?? []).length > 0;` 판정 후, `!hasLive`이면 `state.parkingSlotsByKey[key]` 행의 `row.occupyRange`(non-null)를 `toPixelQuad()`로 변환해 라이브와 동일한 스타일(빨강 반투명 면 `rgba(255,77,77,0.18)` + 외곽선 `rgba(255,77,77,0.9)`)로 폴리곤 렌더 후 반환. 라이브 점유가 있으면 기존 경로 그대로 실행(무변경).
4. **precise 탭 진입 시 DB 로드** — `setTab()`의 `if (tab==='precise'){...}` 블록에 `void loadParkingSlots().then(() => drawRoiOverlay());`를 추가. 탭 재진입/reload 때마다 `GET /capture/slots`를 1회 호출해 `parkingSlotsByKey`를 갱신한 뒤 즉시 재렌더한다(`loadParkingSlots`는 기존 함수 재사용, 실패 시 조용히 미표시).

설계 대비 편차: 없음. `drawOccupancyOverlay`에서 `currentFrameKey()`를 `key` 지역변수로 1회 캐시한 것은 라이브·폴백 공용의 순수 정리(동작 동일).

### 상호배타·불변식(참고, 이번 변경 대상 아님)

`captureActive()`(`calFrameTimer` 포함), `startCapFramePolling()`/`startCalFramePolling()`의 상호 `stop*` 호출은 직전 작업에서 도입된 것으로 이번 변경으로 손대지 않았다. diff에 함께 나타나는 이유는 직전 작업이 아직 별도 커밋되지 않은 동일 작업트리 위에서 이어졌기 때문이다.

## 3. 입출력·REST 계약

이번 변경은 **REST 계약을 새로 만들지도, 기존 계약을 바꾸지도 않았다.** 기존 `GET /capture/slots`(핸들러: `SqliteStore.getSlotSetup()`)를 웹 클라이언트가 소비하는 방식만 확장했다(목록 전용 → 오버레이 폴백 소스 추가).

### `SlotSetupView`(서버 반환 shape, `src/capture/SqliteStore.ts:211` 이하 파싱)
| 필드 | 타입 | app.js 소비처(신규) |
|---|---|---|
| `vpd` | `{x,y,w,h} \| null`(정규화) | `drawDbDetect`: `toPixel(row.vpd, W, H)` → `{px,py,pw,ph}` → `strokeRect` |
| `lpd` | `NormalizedQuad([{x,y}×4]) \| null` | `drawPlateQuad(row.lpd)` 내부 `toPixelQuad()` → 4점 path |
| `occupyRange` | `NormalizedPoint[] \| null` | `drawOccupancyOverlay` 폴백: `toPixelQuad(row.occupyRange, W, H)` → 폴리곤 fill |
| `presetKey` | `` `${camId}:${presetId}` `` | 폴백 조회 키 = `currentFrameKey()`와 정합 |

라이브 렌더 경로도 동일한 `toPixel`/`drawPlateQuad`(→`toPixelQuad`)/`toPixelQuad` 헬퍼를 사용하므로, "라이브 렌더와 DB 폴백 렌더는 같은 변환 함수·같은 입력 shape"라는 것이 검증자의 신규 테스트로 확인되었다(§5).

## 4. 영향도 분석

### 4.1 영향 모듈·의존성 방향
```
web/app.js
  ├─ drawDetectOverlay() → (라이브 없음) drawDbDetect() [신설]
  ├─ drawOccupancyOverlay() → (라이브 없음) DB occupyRange 폴백 [가산]
  ├─ setTab('precise') → loadParkingSlots() [기존 함수 재사용, 호출 시점 추가] → drawRoiOverlay()
  └─ capPoll() 종료분기 → startLive() 만 유지 (resetOverlayDisplay 호출 제거)
       └─ GET /capture/slots → SqliteStore.getSlotSetup() (서버, 미변경)
            └─ Finalizer.finalize() 저장 계약 (서버, 미변경)
```
의존 방향은 웹 → 기존 REST 라우트 한 방향이며, 서버측(`Finalizer`/`SqliteStore`/`captureRoutes`) 코드는 이번 변경에서 한 줄도 수정되지 않았다.

### 4.2 기존 기능 영향
- **라이브 검출/점유 렌더**: 회귀 없음. `drawDetectOverlay`/`drawOccupancyOverlay` 모두 `if(!라이브)` 형태로 DB 폴백을 **가산**했을 뿐, 라이브 데이터가 있을 때의 기존 분기·스타일은 그대로다.
- **정밀수집 프레임폴(`capFrameTick`)·센터라이징 화면갱신(`calFrameTick`)**: 직전 작업 산출물이며 이번 변경은 건드리지 않았다.
- **`renderSlotList`의 `finalized` 분기**: `parkingSlotsByKey`가 채워지면(탭 진입 시 항상 로드) 슬롯 목록이 DB 평면목록 표시로 전환된다. 이는 기존에도 finalize 직후에는 발생하던 동작이며, 이번 변경으로 "탭 진입 시에도" 발생하게 됐다. 마스터 요구("DB에 있으면 상시 표시")와 정합하는 **의도된 동작**으로 판단했으나, 목록 UI 쪽 체감(예: 진행 중 슬롯 편집 화면이 DB 목록으로 바뀌어 보이는 느낌)은 리더의 육안 확인 대상이다.
- **`resetOverlayDisplay()`("표시 초기화" 버튼)**: 함수 자체는 무변경 — 사용자가 명시적으로 누르면 여전히 검출/점유/육면체를 지운다. 이번 변경은 "정밀수집 **종료 시 자동으로**" 지우던 동작만 제거했다.

### 4.3 잠재 리스크 · 주의점
- **라이브/DB 폴백 경계는 프리셋 키 단위**: 같은 프리셋에 라이브와 DB가 동시에 존재하면 라이브가 항상 우선이다. 프리셋 A는 라이브(방금 검출), 프리셋 B는 DB만(과거 데이터)인 상황이 정상적으로 발생할 수 있다 — 사용자가 프리셋을 전환하며 "박스 스타일이 미세하게 다르다"고 느낄 여지는 없다(동일 헬퍼·동일 색상 사용을 검증자가 확인). 다만 `drawDbDetect`는 읽기표시 전용(핸들 없음)이라 라이브 결과와 달리 **클릭 편집이 불가**하다는 점은 UI 동작 차이로 남는다 — 확인 필요 항목(마스터에게 안내 필요 여부는 별도 판단).
- **DB null 필드는 skip**: `row.vpd`/`row.lpd`/`row.occupyRange` 중 일부만 null이어도 나머지는 정상 렌더된다(부분 표시). 매칭 차량이 없었던 finalize 결과(hit 없음)는 세 필드 모두 null이라 해당 프리셋은 빈 자리로 보인다 — 데이터 상태이지 버그가 아니다(§5.3 검증자 확인).
- **occupancy 폴백은 라이브 `spaces`가 완전히 비어 있을 때만** 발동한다(`hasLive` 판정). 라이브 점유가 일부 슬롯만 있고 나머지가 비어 있는 "부분 라이브" 상태에서는 DB 폴백이 섞이지 않고 라이브 결과만 그려진다 — 폴백은 프리셋 전체 단위이지 슬롯 단위가 아니다.
- **config/어셈블리 변경 없음**: 이번 변경은 `web/app.js` 순수 렌더 로직 가산이며 `config/*.json`, 어셈블리 정의, DB 스키마, REST 라우트 정의 파일은 전혀 건드리지 않았다.

### 신규 `test/dbOverlayParity.test.ts`(2케이스, 통과)

1. 차량+번호판 관측 스냅샷 finalize → `vpd`/`lpd`/`occupyRange` 3필드 모두 non-null, `presetKey === '1:1'`(= `currentFrameKey()` 규약 정합) 확인. `toPixel(vpd)`가 `{px,py,pw,ph}` 정확 키·유한값·캔버스(1280×720) 범위 내·양수 w/h를 만족. `toPixelQuad(lpd)`가 4점, `toPixelQuad(occupyRange)`가 ≥3점의 유한·범위내 좌표를 만족.
2. `parkingSlotsByKey` 그룹핑 키(`1:1`)가 app.js `loadParkingSlots`의 그룹핑 규약과 일치.

### Finalizer 저장 계약(회귀 없음)
- `test/finalizerParkingSlots.test.ts` 통과: hit 있을 때 `vpdBbox`/`lpdObb`/`occupyRange` 저장, hit 없을 때 null(`slotRoi`는 항상 저장) — 기존 동작 그대로. 이번 변경(app.js)이 서버 저장 경로를 건드리지 않았음을 재확인.
- **실 DB 상태 명시(위장 없이 기록)**: `data/setting.sqlite`의 `slot_setup` 17행 중 `slot_roi`는 17/17 존재하지만 `vpd_bbox`/`lpd_obb`/`occupy_range`는 **0/17(전부 null)**이다. 이는 "마지막 finalize에 매칭 차량이 없었다"는 데이터 상태이지 코드 결함이 아니다 — 매칭 차량이 있는 finalize를 재현한 신규 parity 테스트에서는 3필드 모두 정상 채워짐이 확인됐다.

### 한계(검증자·리더가 명시한 미검증 범위)
- `drawDbDetect`/`drawOccupancyOverlay`/`drawDetectOverlay`는 DOM 결합(`overlay`/`ctx`/`$()`/체크박스 게이트)이라 vitest 직접 대상 밖 — 실 캔버스 픽셀 렌더, `#roi-vehicle`/`#roi-plate`/`#roi-occupancy` show/hide 대칭, 종료 후 박스 잔존, reload/탭재진입 표시는 리더의 sharp 스샷 경험적 검증으로 확인했다(설계서 §경험적 검증 위임 항목).
- 서버 라이브 라우트 스모크(`curl /capture/slots`)는 서버 미구동으로 생략 — 유닛(실 store finalize)으로 동일 shape을 이미 증명했으므로 스모크는 누락으로 명시한다(통과로 위장하지 않음).
- 실 DB `slot_setup` 17행이 현재 전부 null인 상태에서의 "DB 폴백이 실제로 무언가를 그리는" 육안 확인은, 마지막 finalize가 매칭 차량 없이 종료된 데이터 상태 특성상 현재 시점 실 DB로는 재현되지 않는다 — 매칭 차량이 있는 신규 finalize 이후 재확인이 필요한 항목이다(확인 필요).

## 7. 후속 보완 — 'DB 보기' 체크박스 게이트 (마스터 추가 요구)

마스터 추가 요구: "DB slot_setup의 vpd/lpd/occupy는 **'DB table 보기' 체크박스를 켰을 때만** 보이도록".
직전 구현은 라이브 검출이 없는 프리셋에서 DB 폴백을 **항상** 렌더했으나, 이를 새 체크박스로 게이트했다.

- `web/index.html`: 오버레이 토글에 `#roi-db`("DB 보기") 체크박스 추가 — **기본 미체크**(체크 시에만 표시).
- `web/app.js`:
  - `drawDetectOverlay` DB 폴백(`drawDbDetect`)을 `if ($('roi-db').checked)`로 감쌈.
  - `drawOccupancyOverlay` DB `occupyRange` 폴백을 `if ($('roi-db').checked)`로 감쌈.
  - `#roi-db` 변경 시 `drawRoiOverlay` 재렌더 배선.
- **범위 한정(회귀 0)**: 게이트는 **DB 폴백 브랜치 내부**에만 적용 — 라이브 검출(`detectByKey`)/라이브 점유(`occComputeByKey`) 렌더는 `#roi-db`와 무관하게 기존 그대로. 즉 `#roi-db`는 DB slot_setup 소스 오버레이만 관장한다.

## 1. 기능 개요

차량 바닥 점유 영역(floor ROI · 정규화 4점 사변형)의 **생성 품질**과 **가용성 보장**을 개선한다.

- **목표 형상(참조 이미지 기준)**: 차량 bbox 아래쪽 얇은 띠가 아니라, 차 길이만큼 깊은 "바닥 발자국 사다리꼴"(앞=카메라 근접·넓음, 뒤=원거리·좁음)이며, **LPD 번호판 bbox 를 반드시 감싼다**.
- **가용성 보장**: LLM 두뇌(`recognizeFloorRoi`)가 비활성·미지원·전면 실패여도 결정형 폴백으로 floor ROI 를 **항상** 생성한다. 이때 사용자에게 UI 경고를 1회 띄우고 서버 로그에 `warn` 1회를 남긴다.

이 변경은 SettingAgent 내부에 한정되며 저장 계약(`floorRoiByPreset`)·공통 타입(`@parkagent/types` NormalizedQuad)·ActionAgent/DMAgent 는 건드리지 않는다(§9 영향도).

## 4. 폴백 발자국 기하 (R3) — 수식·비율 근거

`fallbackQuadFromRect(r: NormalizedRect): NormalizedQuad` (`src/capture/floorRoi.ts:22`)

입력 rect `{x,y,w,h}`(정규화). 차량 bbox 는 지붕 포함 전체이므로 하단 일부만이 접지면. 발자국은 하단부터 차 길이만큼 위로 잡는다.

```
left    = x
right   = x + w
bottomY = clamp01(y + h)                      // 앞변(카메라 근접, 이미지 하단)
topY    = clamp01(y + h * (1 - FALLBACK_BAND))// 뒤변(먼 쪽)
frontInset = w * FALLBACK_FRONT_INSET
rearInset  = w * FALLBACK_REAR_INSET

frontL = (clamp01(left  + frontInset), bottomY)
frontR = (clamp01(right - frontInset), bottomY)
rearR  = (clamp01(right - rearInset),  topY)
rearL  = (clamp01(left  + rearInset),  topY)
→ [frontL, frontR, rearR, rearL]   // 순서 규약 [앞왼, 앞오, 뒤오, 뒤왼]
```

### 상수 변경과 근거

| 상수 | 이전 | 이후 | 근거 |
|------|------|------|------|
| `FALLBACK_BAND` | 0.35 | **0.55** | 얕은 띠 → 차 길이감 확보. 승용차 측면/사입 뷰에서 bbox 높이 대비 접지 발자국 세로 점유가 하단 약 절반~2/3. 1.0 은 지붕까지 덮어 과대 → 보수적 중앙값 0.55 |
| `FALLBACK_INSET`(단일 0.1) | 있음 | **삭제** | 앞·뒤 대칭 inset → 사다리꼴 표현 불가 |
| `FALLBACK_FRONT_INSET` | (신규) | **0.04** | 앞변(근접)은 넓게. bbox 좌우 여백만 살짝 정리 |
| `FALLBACK_REAR_INSET` | (신규) | **0.22** | 뒤변(원거리)을 양쪽 22%씩(합 44%) 좁혀 뚜렷한 원근. 앞:뒤 폭비 목표 약 1 : 0.56 (= 1 − 2·0.22) 역산 |

결과: 얕은 대칭 띠 → 깊고 앞넓뒤좁 사다리꼴. **여전히 bbox 유도 순수 기하(외부 의존 0).** BAND/INSET 은 뷰 각도 무관 보수적 근사이며, 특정 카메라 각도에서 과대/과소면 구조 변경 없이 상수만 재튜닝한다.

## 5. 번호판 포함 강제 (R4)

### 5.1 `expandQuadToContainRect(quad, rect)` — 신규 순수 함수 (`floorRoi.ts:43`)

목적: rect(번호판 bbox) 4모서리가 quad 안에 들어오도록 **최소 확장**. quad = [FL, FR, RR, RL].

```
rl = rect.x            // rect left
rr = rect.x + rect.w   // rect right
rt = rect.y            // rect top
rb = rect.y + rect.h   // rect bottom

// 세로: 앞변(하단)은 rb 아래까지, 뒤변(상단)은 rt 위까지 — 변 단위 공통값으로 통일해 평행(사다리꼴) 유지
frontY = clamp01(max(FL.y, FR.y, rb))
rearY  = clamp01(min(RL.y, RR.y, rt))

// 가로: 좌측 정점 x 는 rl 이하, 우측 정점 x 는 rr 이상(앞·뒤 각각)
flX = clamp01(min(FL.x, rl));  rlX = clamp01(min(RL.x, rl))
frX = clamp01(max(FR.x, rr));  rrX = clamp01(max(RR.x, rr))

return [ (flX,frontY), (frX,frontY), (rrX,rearY), (rlX,rearY) ]
```

핵심 성질:
- **멱등(축소 없음)**: `min`/`max` 만 쓰므로 rect 가 이미 quad 안이면 결과가 입력과 동일. 2회 적용해도 불변.
- **사다리꼴 보존**: 앞변 y·뒤변 y 를 각각 두 정점 공통값으로 통일한다(개별 정점만 밀면 변이 기울어 뒤틀림). 좌우 x 는 정점별로 확장.
- **프레임 보존**: 모든 결과 좌표에 `clamp01` → rect 가 범위를 벗어나도 결과는 [0,1] 유지.

### 5.2 `resolveFloorQuad(llmQuad, vehicle, plate?)` 확장 (`floorRoi.ts:92`)

```ts
const base = normalizeQuad(llmQuad) ?? fallbackQuadFromRect(vehicle);
return plate ? expandQuadToContainRect(base, plate) : base;
```

- LLM quad 경로와 폴백 경로가 **공통으로** plate 포함 강제를 통과한다(R4 "공통").
- `plate` 없으면 base 그대로 반환 → 기존 동작 유지(회귀 안전).

## 6. LLM 강제동작·경고 흐름 (R2) — `FloorRoiReviewer.review`

### 6.1 no-op 제거 → 항상 폴백

기존 진입부의 무동작 조기 반환 `if (!brain?.enabled || !brain.recognizeFloorRoi) return;` 을 **삭제**했다. 이제 brain 이 비활성/부재여도 후보 슬롯 루프를 진행하고, LLM 호출을 못 하면 `quadRaw = null` → `resolveFloorQuad` 폴백으로 floor ROI 를 **항상** 생성·저장한다.

### 6.2 실행 흐름

```
review(runId, slots, framesByPreset): Promise<{ llmUnavailable }>
  llmUsable = !!(brain?.enabled && brain.recognizeFloorRoi)
  candidates = slots where status != rejected && != merged
  attempted = 0; succeeded = 0
  for s of candidates (maxPerCheckpoint 상한, 기본 12):
    jpeg = framesByPreset.get(s.presetKey); if !jpeg continue
    vehicle = {x,y,w,h}
    plate   = (plateX..plateH 모두 non-null) ? {x,y,w,h} : undefined
    quadRaw = null
    if (llmUsable && brain.recognizeFloorRoi):
      attempted++
      try { quadRaw = (await brain.recognizeFloorRoi({...vehicle, plate?, ...}))?.quad ?? null
            if (quadRaw) succeeded++ }
      catch { logger.warn('floor ROI 추론 실패(폴백)') }   // 개별 실패는 폴백만
    quad = resolveFloorQuad(quadRaw, vehicle, plate)         // 항상 생성 + plate 포함
    store.upsertFloorRoi(runId, s.presetKey, s.clusterId, quad, now())
  llmUnavailable = !llmUsable || (attempted > 0 && succeeded === 0)
  if (llmUnavailable) logger.warn({llmUsable,attempted,succeeded}, 'LLM 비활성/불가 — 결정형 폴백 사용')  // 1회
  return { llmUnavailable }
```

### 6.3 경고 판정 조건 (리더 확정)

```
llmUnavailable = !llmUsable || (attempted > 0 && succeeded === 0)
```

- `!llmUsable`: brain 없음 · `brain.enabled === false` · `recognizeFloorRoi` 메서드 부재.
- `attempted > 0 && succeeded === 0`: 시도했으나 전 슬롯 무효 quad(=전면 실패).
- **개별 throw** 는 catch 후 폴백만 수행하며 `succeeded` 를 올리지 않는다. 따라서 성공이 1건이라도 있으면 `llmUnavailable=false`, throw 만 있고 성공 0이면 `true`.

> 설계 초안(§5·§7 케이스9)은 `!llmUsable` 만 경고로 보고 "throw → false" 였으나, **리더 확정으로** `attempted>0 && succeeded===0`(전면 실패) 케이스도 경고에 포함하도록 재정의되었다. 검증(03)은 이 확정 조건 기준으로 통과했다.

## 11. 수동 동작확인 체크리스트 (유닛 미커버 — DOM/렌더/실측)

검증(03)에서 유닛으로 커버 불가로 명시된 항목. 통과위장 없이 그대로 기록한다.

1. **LLM off 경고 모달 1회**: `llm.config` 비활성 상태로 수집 실행 → `#floor-llm-warn-modal` 이 런당 정확히 1회만 뜨는지(폴링 반복 팝업 없음), 닫기 동작, 다음 런 시작 시 재무장 육안 확인.
2. **폴백 발자국 시각·번호판 포함**: 실측 프레임 오버레이에서 폴백 사변형이 얕은 띠가 아닌 앞넓뒤좁 깊은 발자국(BAND 0.55)으로 그려지고, 번호판 bbox 가 사변형 안에 들어오는지 확인(과대/과소면 상수 재튜닝).
3. **LLM off 시 초록영역 생성**: no-op 폐기로 LLM 미가동에도 floor ROI(초록영역)가 뷰어에 생성·표시되는지 확인.

## 12. MCP/책임 경계

- **좌표 생성(추정)** = LLM 두뇌 `recognizeFloorRoi`(원근·맥락·주차선 판단). 프롬프트는 여기에만 관여.
- **검증·클램프·폴백·번호판 포함 강제** = 결정형 순수 모듈 `floorRoi.ts`(외부 의존 0). LLM 실패·비활성에도 floor ROI 존재를 보증하는 안전망.
- 경계 자체는 불변이며, 이번 개선은 결정형 측(폴백 형상·포함 강제)과 가용성 신호(경고)에 집중했다.

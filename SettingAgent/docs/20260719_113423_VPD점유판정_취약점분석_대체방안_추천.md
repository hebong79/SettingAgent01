# VPD 점유 판정 취약점 분석 + 대체 방안 추천

> 작성일: 2026-07-19 11:34:23
> 성격: 분석·추천 문서(설계 착수 전 검토용, 코드 변경 없음)
> 배경: 마스터 지적 — "VPD가 불안정(약 70%만 인지, 오검 발생). VPD는 차량이 주차면에 점유했는지 찾는 베이스 작업인데 베이스가 흔들린다. VPD 대체 방안 추천."

---

## 1. 센터라이징 순서 — 최종화 전에도 돌 수 있음 (설계 취약점)

- [PtzCalibrator.start()](../src/calibrate/PtzCalibrator.ts) 는 대상을 **`slot_setup` DB에서 읽는다**(`expandPlateTargetsFromSlotSetup(store.getSlotSetup())`).
- `slot_setup` 은 **직전 최종화 결과가 DB에 계속 남는다**(새 정밀수집이 지우지 않음).
- 따라서:
  - 센터라이징은 **수동 버튼(POST /calibrate/ptz)** 으로만 시작 — 정밀수집이 자동 트리거하지 않음.
  - **그러나** 현재 수집을 최종화하지 않고 센터라이징을 누르면 **직전 최종화 때의 낡은 slot_setup 을 대상으로 중심이동+확대**를 수행한다 → "최종화 전 중심이동·확대" 발생 가능.
  - **올바른 순서: 정밀수집 → 최종화 → 센터라이징.** 최종화 없이 센터라이징하면 stale 좌표로 헛보정.
- 참고: 정밀수집 중 카메라가 프리셋을 돌며 zoom 이 바뀌는 것은 **프리셋 자체 zoom 값으로 이동(/req_move)** 하는 정상 동작 — 번호판 센터링 확대와는 별개(화면상 혼동 가능).

---

## 2. VPD가 흔들리면 점유 판정 전체가 흔들리는 구조

현재 점유 판정([web/occupancy.js](../web/occupancy.js) `judge`)의 **base 가 VPD bbox**:

- **1단계**: 차량 bbox 의 **접지밴드(하단 25%)** ∩ 주차면 폴리곤 겹침 argmax → 그 슬롯을 점유로 귀속(source='bbox').
- **2단계**: 미점유 슬롯에 LPD 번호판 중심 폴백(source='plate').

→ VPD 가 ~70%만 잡거나 오검하면 접지밴드가 틀리거나 없어짐 → **슬롯 귀속이 통째로 틀린다.** 겹치거나 비스듬한 차에서 bbox 가 나쁘면 base 가 무너진다. bbox 는 "느슨한 사각형"이라 점유 base 로는 원리적으로 취약.

---

## 3. VPD 대체·보강 추천 (기존 인프라 기준, 우선순위순)

### ① 세그멘테이션 마스크 점유 — 최우선 추천
- bbox 대신 **차량 세그 마스크 ∩ 주차면 폴리곤 면적비율**로 점유 판정("이 슬롯 바닥을 차 픽셀이 몇 % 덮나").
- **이미 배선됨**: `vpd.segment()` 가 차량별 픽셀 폴리곤 `masks` 반환([VpdClient.ts](../src/clients/VpdClient.ts)). 현재 3D 육면체 산출에만 쓰이고 **점유 판정엔 미사용** — 소비처만 추가하면 됨.
- **왜 강한가**: 마스크는 차 실루엣 그 자체 → NMS 중복·오검 bbox 에 둔감, 비스듬/겹친 차도 정확, "느슨한 박스 중심 귀속"의 실패를 근본 제거. **같은 VPD 서비스(seg 엔드포인트) 재사용하되 훨씬 안정적인 신호.**

### ② 빈-기준프레임 차분(배경 차감) — 모델 비의존 보강
- 프리셋별 **빈 주차장 기준 프레임** vs 현재 프레임을 슬롯 폴리곤 단위로 비교 → 유의미한 변화 = 점유.
- **인프라 일부 존재**: `data/refframes/` + `estimateAlign`(자동보정용).
- **왜**: 검출 모델 품질에 **0 의존** → 고정 시뮬 카메라에서 극히 안정적. 단점: 빈 기준 필요·그림자/조명 민감(시뮬에선 미미)·차/타 물체 구분 불가. → seg 강등 프레임의 **교차검증/타이브레이커**로 이상적.

### ③ 3D 접지 발자국 겹침 — 보조
- 이미 산출하는 차량 육면체의 지면 발자국을 슬롯 지면 폴리곤과 겹쳐 판정. 다만 결국 seg+지면모델 품질에 의존 → ①에 사실상 흡수.

### (시뮬 한정) Unity 실측 점유 — 셋업용이면 고려
- 시뮬레이터라면 Unity 가 "어느 차가 어느 슬롯에" 정답 제공 가능. **셋업/라벨링 정확도만** 목적이면 100% 안정적이나, 실카메라 비전 파이프라인 검증이 목적이면 셋업만 돕고 런타임엔 무의미.

---

## 4. 추천 결론

- **base 를 bbox → 세그 마스크 점유(①)로 전환**하고, **빈-기준 차분(②)을 강등 프레임 교차검증**으로 얹는다.
- 번호판(LPD)은 지금처럼 확정 보조로 유지.
- 방금 적용한 VPD dedup 은 *표시* 안정화였고, 본 건은 *판정 base 자체* 를 바꾸는 더 깊은 수정.

### 결정 필요 포인트 — 목표 방향
- **(A) 셋업/최종화 시 슬롯 점유 라벨만 정확** → ① seg 점유(+② 보강), 또는 시뮬 실측까지 옵션.
- **(B) 실카메라에서도 도는 비전 점유 파이프라인** → ① seg 점유 + ② 차분(실측 금지).

### 다음 단계(착수 시)
- 방향 확정 후 parkagent-dev 하네스(goal/loop)로 **seg 마스크 점유 판정**을 설계→구현→sharp 스샷 검증→문서화까지 진행.
- 착수 전 `vpd.segPath` 실제 배선/구동 여부 확인 → 미배선이면 ②(빈-기준 차분)로 폴백 판단.

# 정밀 수집(/capture/start) 시작 시 라이브 프리셋 갱신 — 캐시 의존 제거

## 증상
정밀 수집을 시작하면 카메라가 **이전(캐시된) 프리셋 순서**대로 이동했다. Unity에서 프리셋을 갱신해도 정밀 수집은 옛 순서를 그대로 사용.

## 원인
`POST /capture/start`(targets 미지정)가 `loadSetupTargets(mapFiles)`로 **캐시 파일 `config/camerapos.json`을 그대로 읽었다**.
- `src/api/captureRoutes.ts` — `loadSetupTargets(deps.mapFiles)`
- `src/setup/mapTargets.ts:loadSetupTargets` — camerapos.json 파일 파싱
- 반면 `POST /setup/run-from-map`은 `refreshOnRun`일 때 공급자(Unity `GET /cameras`)로 라이브 갱신하는 로직이 이미 있었으나, **capture 라우트 등록에는 `presetProvider`가 전달되지 않아** 갱신 불가했다(`src/api/server.ts`의 `registerCaptureRoutes` 호출).

Unity 서버(`CWebCamCtrlServer.HandleRequestCameras`)는 `/cameras`를 매 호출 라이브로 생성하므로 서버 측 문제는 없음.

## 변경 (사용자 결정: 항상 라이브 갱신)
- `src/api/captureRoutes.ts`
  - `CaptureRouteDeps`에 `presetProvider?: PresetProvider | null` 추가.
  - `/capture/start`(targets 미지정): `presetProvider`가 있으면 **항상 `listViews()`(Unity `/cameras`)로 새로 받아** `viewsToTargets()`로 사용하고, `cameraposFile`이 있으면 `writeCamerapos()`로 캐시도 갱신. 공급자가 없을 때만 기존 `loadSetupTargets(mapFiles)` 폴백.
  - `import`에 `viewsToTargets`, `writeCamerapos`, `PresetProvider` 추가.
- `src/api/server.ts` — `registerCaptureRoutes(...)`에 `presetProvider: deps.presetProvider` 전달.

명시적 `targets`를 본문에 주면 그대로 사용(우선순위 유지). 공급자가 `camerapos`(수동, null)면 라이브 갱신 불가 → 파일 폴백.

## 적용 조건
`config/tools.config.json`의 `presetProvider.type`이 `unity-api`(또는 discovery)여야 라이브 갱신이 동작한다. 현재 사용자 설정은 `unity-api` + `refreshOnRun:true`이므로 즉시 적용됨.

## 영향도
- 가산·국소: capture 라우트만 변경, `/setup/*`·`/mapping`·뷰어 불변.
- `presetProvider`는 buildServer가 이미 보유 → 새 의존성 없음. capture deps에 옵셔널 추가라 기존 주입부 무변(미주입 시 파일 폴백).
- Unity 측 코드 변경 없음.

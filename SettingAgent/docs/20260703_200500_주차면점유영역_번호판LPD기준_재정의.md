## 1. 배경 / 목적

### 1-1. 무엇을 바꿨나

기존 floor ROI 는 **차량 발자국(footprint)** 개념 — 흰선·바퀴 접지선을 근거로 LLM 이 찍는 **가변 다각형(4~10점)** 이었다.
이번 작업은 그 근거 개념을 폐기하고, **번호판(LPD OBB)의 각도·위치를 기준으로 구성한 4점 회전 사변형** 으로 점유영역을 재정의한다.

- 폐기: "흰선 스냅 / 바퀴 접지 / 사다리꼴 밴드" 프롬프트·폴백.
- 신규: 번호판 노란 박스에 **각도·좌우중앙 정렬**되고, 번호판이 사변형 **세로에서 약간 앞(아래/카메라 쪽)** 에 놓이는 회전 직사각형.

### 1-2. 목표 시각 (참조 이미지)

`etc/주차면점유영역_01~03.jpg` 에서 확인되는 목표 렌더:

- 각 차량마다 **반투명 녹색 사변형** = 점유영역. 원근으로 기울어진 직사각형(위에서 본 바닥면).
- 사변형 앞쪽 하단 중앙에 **노란 번호판 박스**. 사변형이 번호판 각도와 같은 방향으로 기울어 있고, 번호판이 좌우 중앙·세로 앞쪽에 위치.
- 시안(cyan) 박스 = 차량 bbox + 전역 인덱스 라벨(25/26/27/28). bbox 는 크기 참고용일 뿐 사변형 각도의 근거가 아니다.

### 1-3. 재정의 규칙 1~6 (참조 이미지 = 사양)

| # | 규칙 | 의미 |
|---|------|------|
| 1 | 번호판 **완전 포함** | 번호판 전체가 사변형 내부에 든다. |
| 2 | **하단변 평행** | 번호판 하단변 ∥ 사변형 앞(하단)변. |
| 3 | **각도 정렬** | 사변형을 번호판 회전각과 **같은 방향**으로 기울인다. |
| 4 | **좌우 중앙** | 번호판이 사변형 좌우 중앙에 온다. |
| 5 | **세로 약간 앞** | 번호판이 사변형 세로에서 중앙보다 약간 앞(아래/카메라 쪽). |
| 6 | **LPD > VPD 우선** | 우선순위 번호판 > 차량 bbox. bbox 는 크기 참고용. |

## 2. 신규 / 변경 함수 상세

### 2-1. `plateAngleRad(quad: NormalizedQuad): number` — 신규 (`src/domain/geometry.ts`)

번호판 OBB quad(규약 TL,TR,BR,BL)의 **하단변 기울기**(이미지 좌표 기준, rad)를 산출한다.

```
dx = (TR.x - TL.x) + (BR.x - BL.x)   // 상단변 + 하단변 x 성분
dy = (TR.y - TL.y) + (BR.y - BL.y)   // 상단변 + 하단변 y 성분
if hypot(dx,dy) < 1e-9 → return 0    // 퇴화(면적0 등) 방어
return atan2(dy, dx)
```

- **상·하단변 평균**으로 산출 → 점순서가 실제로 뒤바뀌거나 노이즈가 있어도 각도가 강건.
- 축정렬 quad(rect 승격분)는 상·하단변이 수평 → **각도 0** 으로 자연 수렴(별도 분기 불필요).

### 2-2. `projectedSpan(pts, ax, ay): number` — 신규 (`src/domain/geometry.ts`)

점 집합을 단위축 `(ax,ay)` 에 투영한 **스팬(max−min)**. 로컬 OBB 크기(좌우폭 `wp`, 앞뒤깊이 `hp`) 산출용. 빈 배열이면 0.

### 2-3. `buildPlateAnchoredQuad(vehicle, plateQuad?, depthFloor?): NormalizedQuad` — 신규, 구 `fallbackPolygon` 대체 (`src/capture/floorRoi.ts`)

번호판 각도·위치를 기준으로 **정확히 4점** 회전 사변형을 결정형으로 구성한다(항상 존재 보장).

알고리즘:
1. `plate = plateQuad ?? rectToQuad(predictPlateRect(vehicle))`, `θ = plateQuad ? plateAngleRad(plateQuad) : 0`.
2. 로컬축 `nb=(sinθ, −cosθ)`(앞→뒤). **앞(−nb)=이미지 하단** 되도록 `nb.y>0` 이면 부호 반전(near-vertical·점순서 뒤바뀜 방어). `u=(−nb.y, nb.x)` = nb 직교 좌우축.
3. `wp = projectedSpan(plate, u)`, `hp = projectedSpan(plate, nb)`, `pc` = plate 4점 평균(중심).
4. 크기(규칙 6, bbox=크기 참고 + 번호판 포함 하한):
   - `W = max(vehicle.w × 1.0, wp × 1.15)`
   - `D = max(vehicle.h × 0.55, hp × 1.15, depthFloor ?? 0)`
5. 중심 이동(규칙 4·5): `Cq = pc + nb × (D × (0.5 − 0.42))` — **좌우는 pc 유지**(좌우중앙), 세로만 nb 방향 이동(번호판을 앞/아래쪽에).
6. 4모서리 `[FL,FR,RR,RL]` 산출 → `clamp01` → `orderConvexCanonical` 재적용 → 4점 반환.

### 2-4. `resolveFloorPolygon(llmPoly, vehicle, plateQuad?, plateRect?)` — 시그니처 확장 (`src/capture/floorRoi.ts`)

최종 진입점. **빌더가 각도·배치의 권위**, LLM 은 깊이 보조.

1. `θ`·`nb` 를 plateQuad 로 산출(부호 정규화 동일).
2. LLM 다각형을 `normalizePolygon` 통과 → 그 **nb축 span** 을 `depthFloor` 로만 추출(각도·배치는 무시).
3. `base = buildPlateAnchoredQuad(vehicle, plateQuad, depthFloor)`.
4. 안전망: `expandPolygonToContainRect(base, plateRect ?? quadBoundingRect(plateQuad) ?? predictPlateRect(vehicle))` — 멱등, 정상 경로 no-op.

### 2-5. 상수 4종 (`src/capture/floorRoi.ts` 상단, 육안 튜닝 집중점)

| 상수 | 값 | 의미 |
|------|----|------|
| `FLOOR_WIDTH_RATIO` | 1.0 | 사변형 좌우폭 하한 = bbox.w × 비율 |
| `FLOOR_DEPTH_RATIO` | 0.55 | 사변형 앞뒤깊이 하한 = bbox.h × 비율 |
| `PLATE_FRONT_RATIO` | 0.42 | 앞변→번호판중심 비율(<0.5 = 중앙보다 앞/아래) → 규칙 5 |
| `PLATE_CONTAIN_MARGIN` | 0.15 | 번호판 포함 여유(W·D 하한을 span×1.15 로 보정) → 규칙 1 |

> 제거된 구 상수: `FALLBACK_BAND`, `FALLBACK_FRONT_INSET`, `FALLBACK_REAR_INSET`(구 사다리꼴 폴백 전용, 사용처 소멸). `POLY_MARGIN`·`PLATE_*`(predictPlateRect·normalizePolygon 경로)는 유지.

### 2-6. `FloorRoiResultSchema.polygon` 4점화 (`src/brain/SetupBrain.ts`)

`z.array(...).min(4).max(10)` → **`.length(4)`**. LLM 출력을 정확히 4점으로 강제(4점 특수화). 출력 키는 `polygon` 유지(surgical). `FloorRoiInput` 에 `plateQuad?: NormalizedQuad` 필드 추가.

## 3. 프롬프트 재작성 (`config/prompts/floor_roi.yaml`)

전면 교체. 구 흰선/바퀴접지/사다리꼴·출력키 `quad` 폐기.

- **system**: 번호판(노란 박스) 기준 규칙 1~6 명시(내부포함·하단변평행·각도추종·좌우중앙·세로약간앞·번호판>bbox). 지붕·차체·그림자 제외, 바닥면적만. 좌표규약 정규화 0~1·순서 [앞왼,앞오,뒤오,뒤왼](앞=이미지 아래, 시계방향). 출력 `{"polygon":[{x,y}×4],"confidence"}` **정확히 4점**·코드펜스 금지.
- **user 템플릿**: `camIdx/presetIdx`, `vehicle`(bbox), `plate`(rect), **신규 `plateQuad`(4점 [TL,TR,BR,BL] 각도 단서)** 주입.
- **부수효과(정합 회복)**: 구 yaml 은 출력키를 `quad` 로 안내했으나 `FloorRoiResultSchema` 는 `polygon` 을 파싱 — 잠재 불일치였다. 재작성으로 `polygon` 통일.

## 4. 결정형 vs LLM 분담 (해석 B 확정)

| 항목 | 담당 | 근거 |
|------|------|------|
| 각도(규칙 2·3) | **결정형** | `plateAngleRad` → 빌더가 nb·u 축으로 강제 |
| 좌우중앙(규칙 4) | **결정형** | 빌더가 좌우 이동 안 함(pc.x 유지) |
| 세로 약간앞(규칙 5) | **결정형** | `PLATE_FRONT_RATIO` 로 중심 이동 |
| 번호판 포함(규칙 1) | **결정형** | span×1.15 하한 + `expandPolygonToContainRect` 안전망 |
| 4점 사변형 제안 | LLM | 이미지 기반 초안(비전) |
| 앞뒤깊이 D 힌트 | LLM(보조) | LLM 사변형의 nb축 span 을 D 하한으로만 참고 |

즉 각도·중앙배치·포함은 결정형이 권위를 가지며, LLM 비활성/실패/미지원 시에도 결정형 폴백으로 **항상** floor ROI 를 생성한다. LLM 은 4점 제안과 깊이 힌트만 제공한다.

## 7. F-1 정정 반영 — 캐노니컬 시작정점의 FL/FR 가변성

검증자 F-1(결함 아님, **문서 표현 정정** 건)을 정확히 반영한다.

- **부정확했던 서술**: 계획 §6-3·구현노트 §7-3 의 "앞변 = FL→FR = `q[0]→q[1]`, 그 각도 = plateAngleRad".
- **실제 동작**: `orderConvexCanonical` 은 시작정점을 **최대 y(동률 min x)** 정점으로 잡는다. +회전(θ>0) 시 최대 y 정점이 FL 이 아니라 **FR** 로 이동하므로, `q[0]→q[1]` 은 앞변이 아니라 **우측변**(≈θ+90°)이 된다. θ<0 에서는 우연히 FL 이 시작이 되어 `q[0]→q[1]` 이 앞변과 일치 — 즉 부호 비대칭이라 "가끔 맞아" 보이는 함정.
- **올바른 서술**: 캐노니컬 시작정점은 회전 부호에 따라 **FL/FR 로 가변**한다. **앞변은 index 고정으로 식별하지 말고, nb축 투영이 최소인 두 정점(=앞 두 정점) 사이 모서리로 식별**해야 한다. 사변형 자체는 θ 로 정확히 기운 회전 직사각형이며, 그 앞변 각도는 번호판 각도를 정확히 추종한다.
- **코드 영향**: **없음(무변경)**. 시작정점 index 이동은 닫힌 다각형 렌더·point-in-polygon·deconflict 에 무해(순서 무관). 계획 §6-3 의 구조 부등식(`q0.y>q3.y, q1.y>q2.y, q0.x<q1.x`)은 회전에도 성립(테스트 통과).
- **테스트 대응**: 검증자는 앞변을 index 고정이 아닌 `frontEdgeAngle(q, θ)`(nb 투영 최소 두 정점)로 안정 식별하여 규칙 2·3·5 를 검증(초기 index 고정 시 3건 실패 → 원인 규명 후 헬퍼 보정, **구현 소스 무수정**).

> 현재 소비 코드 중 `q[0]` 을 "앞왼"으로 의미적으로 신뢰하는 곳은 없다(렌더는 닫힌 다각형 순서 무관). 따라서 실동작 위험은 없고, 문서·주석 서술만 위와 같이 정정한다.

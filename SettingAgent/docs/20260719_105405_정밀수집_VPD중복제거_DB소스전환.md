# 정밀수집 VPD 오버레이 개선 — 중복 박스 제거(연결요소) + `#roi-db` VPD 소스 전환

작성: documenter / 대상: `SettingAgent/web` 정밀수집 오버레이 렌더 경로
파이프라인: `_workspace/01_architect_plan.md`(설계) → `02_developer_changes.md`(구현) → `03_qa_report.md`(검증) → 본 문서

## 1. 배경 및 문제

정밀수집 화면(`web/index.html` + `web/app.js`)에서 라이브 VPD(차량 검출) 결과를 청록색(`#00e5ff`) 사각형으로 오버레이한다. VPD 모델이 NMS/dedup 없이 응답하기 때문에, 같은 차량에 대해 여러 개의 겹친 박스가 그대로 그려지는 문제가 있었다(`web/app.js` 구 `drawDetectOverlay`, 862행 부근 — `state.detectByKey[key].vehicles` 전량을 조건 없이 순회·`strokeRect`).

또한 `#roi-db` 체크박스는 "라이브 검출 결과(`d`)가 **없을 때만**" DB(slot_setup) 저장분으로 폴백하는 구조였다(구 `drawDbDetect`, 897행). 즉 라이브 검출이 있는 상태에서는 DB 저장 VPD를 확인할 방법이 없어, 마스터가 요청한 "DB 체크박스로 VPD 소스를 전환해 보고 싶다"는 요구를 충족하지 못했다.

## 2. 요구사항

1. **요구1**: VPD 청록 박스를 차량당 1개만(마지막 검지) 표시한다.
2. **요구2**: `#roi-db` 체크 시 VPD 소스를 DB(slot_setup) 저장분으로 전환하고, 해제 시 라이브(중복 제거된) 검출로 되돌린다. LPD·점유·육면체 오버레이는 회귀 0(변경 전과 동일).

## 3. 신규/변경 함수 상세

### 3.1 `rectIoU(a, b)` — `web/core.js:1257`

두 `NormalizedRect({x,y,w,h}, 0..1 정규화)`의 IoU(교집합/합집합 비율)를 계산하는 순수 함수.

```js
export function rectIoU(a, b) {
  const ix = Math.max(0, Math.min(a.x + a.w, b.x + b.w) - Math.max(a.x, b.x));
  const iy = Math.max(0, Math.min(a.y + a.h, b.y + b.h) - Math.max(a.y, b.y));
  const inter = ix * iy;
  const uni = a.w * a.h + b.w * b.h - inter;
  return uni <= 0 ? 0 : inter / uni;
}
```

- 겹침이 없거나(`ix`/`iy` 0 이하) 두 사각형이 모두 퇴화(면적 0)해 `uni<=0`이면 0으로 가드(0-나눗셈 방지).

### 3.2 `dedupeVehicles(vehicles, iouThresh = 0.5)` — `web/core.js:1274`

VPD 차량 배열에서 같은 차량으로 판단되는 겹친 박스를 그룹핑해 차량당 1개만 남기는 순수 함수.

```js
export function dedupeVehicles(vehicles, iouThresh = 0.5) {
  const vs = (vehicles ?? []).filter((v) => v?.rect);
  const n = vs.length;
  const parent = Array.from({ length: n }, (_, i) => i);
  const find = (x) => (parent[x] === x ? x : (parent[x] = find(parent[x])));
  for (let i = 0; i < n; i++) {
    for (let j = i + 1; j < n; j++) {
      if (rectIoU(vs[i].rect, vs[j].rect) >= iouThresh) parent[find(i)] = find(j);
    }
  }
  const lastOf = new Map();
  for (let i = 0; i < n; i++) {
    const r = find(i);
    if (!lastOf.has(r) || i > lastOf.get(r)) lastOf.set(r, i);
  }
  return [...lastOf.values()].sort((a, b) => a - b).map((i) => vs[i]);
}
```

**알고리즘 = IoU 연결요소(union-find). 왜 그리디가 아닌가 — 실제 발생한 결함:**

- 설계 초안(`01_architect_plan.md` §1)은 "뒤→앞 스캔, 이미 채택된 박스 중 하나와 IoU≥임계면 드롭"하는 **그리디** 방식이었다.
- 구현 후 리더의 sharp 렌더 경험적 관찰(goal/loop B모드)에서, 한 차량이 라운드별로 크기가 다른 박스 3~4개(동심 다중스케일)로 검지되는 경우 **체인 양 끝(가장 작은 박스 ↔ 가장 큰 박스)의 IoU가 임계 미만**이면 그리디는 둘 다 생존시켜 "차량당 1개" 요구를 위반하는 것이 확인되었다(`02_developer_changes.md` §2, 스모크 케이스: 인접 IoU≈0.592, 양끝 IoU≈0.207 < 0.5).
- 이를 **union-find 기반 연결요소**로 교정: IoU≥임계인 쌍을 간선으로 묶으면, 인접 쌍이 연쇄적으로(transitive) 이어진 경우 양 끝이 서로 안 겹쳐도 하나의 그룹으로 병합된다. 각 그룹에서는 원배열 최대 index(=마지막 검지)만 생존시킨다.
- 인접한 별개 차량은 실측 IoU가 낮아(관찰상 최대 0.183~0.021) 별도 그룹으로 유지되어 과잉병합은 발생하지 않는다.

**"마지막 검지"의 코드적 의미**: VPD 응답에는 track id/timestamp가 없어(`01_architect_plan.md` §0), 배열의 index가 클수록 나중 검지로 해석한다(가정1, 리더 확인 대기 항목으로 설계서에 명시됨). 구현은 각 연결요소(그룹) 내에서 **원배열 최대 index**를 가진 요소를 생존시키는 것으로 이를 구현한다(`lastOf` 맵의 `i > lastOf.get(r)` 비교). 생존 index는 마지막에 오름차순 정렬(`sort((a,b)=>a-b)`)되어 원래 배열 순서를 복원한다 — 이는 렌더링 index와 선택(`hitTestDetections`)의 index가 안정적으로 유지되도록 하기 위함이다.

- malformed 요소(`rect` 없음)는 `filter`에서 스킵되며 원순서는 유지된다.
- 반환값은 원본 객체 참조 그대로이므로 `plate`/`confidence`/`cls` 등 부가 필드가 보존된다.
- `iouThresh` 기본값 0.5는 하드코딩(설정 플럼빙 없음, CLAUDE.md 규칙2 단순함 우선).

### 3.3 `web/core.d.ts:442-443` — 타입 선언

```ts
export function rectIoU(a: NormalizedRect, b: NormalizedRect): number;
export function dedupeVehicles<T extends { rect: NormalizedRect }>(vehicles: T[], iouThresh?: number): T[];
```

제네릭 `T extends { rect: NormalizedRect }`로 선언해 vehicle 요소 shape(`{rect, confidence, cls, plate?}`)이 그대로 보존됨을 타입 수준에서 보증한다.

### 3.4 수집 시점(ingestion) 적용 — `runLiveDetect()` (`web/app.js:933`)

```js
state.detectByKey[presetKey(cam, preset)] = { ...detect, vehicles: dedupeVehicles(detect.vehicles ?? []) };
```

**왜 렌더 시점이 아니라 수집 시점인가**: `state.detectByKey`는 `drawDetectOverlay`(렌더) 뿐 아니라 `hitTestDetections`/`removeDetection`(기능2 선택·편집, index 기반)과 `buildFlatSlotRows`(목록의 VPD/LPD 태그·점유 판정)가 함께 참조하는 **단일 공유 소스**다. dedup을 렌더에서만 적용하면 화면에 그려지는 index와 선택·삭제 로직이 참조하는 index가 어긋나 오작동한다. 저장 직전(ingestion)에 정제해두면 모든 소비처가 동일한 축소 배열과 안정된 index를 공유하게 되어 **자동으로 정합**되고, 변경점도 1곳으로 국한되어 더 외과적이다(`01_architect_plan.md` §1).

- `detect.cuboids`는 `vcuboidByKey`로 별도 저장되므로(`vehicles` 파생이 아님) 육면체는 무영향.
- `detect.summary`(카운트 메시지)는 서버 응답의 별도 필드라 목록 카운트 텍스트는 불변.

### 3.5 `#roi-db` VPD 소스 전환 — `drawDetectOverlay`(`web/app.js:863`) 재구성 + `drawDbVpd`(`web/app.js:907`) 신설

```js
if (showVehicle) {
  if (dbOn) {
    drawDbVpd(ctx, rows);              // DB(slot_setup) vpd 로 전환. 읽기표시(선택·핸들 없음).
  } else if (d) {
    (d.vehicles ?? []).forEach((v, i) => { /* 기존 라이브(=이미 dedup 저장분) 렌더 */ });
  }
}
```

- `#roi-db` 체크 → VPD 레이어만 라이브 대신 **DB(slot_setup) 저장 vpd**(슬롯당 1개, `drawDbVpd`)로 전환. 해제 → 라이브(수집 시점에 이미 dedup된 결과)로 복귀.
- **LPD·점유·육면체는 한 줄도 바꾸지 않았다**(`web/app.js:889-899` LPD 블록은 기존과 동일한 "라이브 있으면 라이브, 없고 `#roi-db`면 DB 폴백" 유지). VPD와 LPD가 서로 다른 소스 정책을 갖는 비대칭이 발생하지만, 이는 "회귀 0" 제약을 우선한 설계 결정이다(§6 한계 참고).
- 등가성: (라이브 있음, DB off) = 기존과 동일 / (라이브 없음, DB on) = VPD·LPD 모두 DB(구 `drawDbDetect`와 동일 동작) / (라이브 있음, DB on) = 이번에 신규로 가능해진 조합(VPD→DB 전환, LPD→라이브 유지).

### 3.6 `drawDbDetect` 제거 (고아 코드 정리)

기존 `drawDbDetect`(VPD+LPD를 함께 그리던 DB 폴백 함수)는 유일 호출부가 재구성으로 사라져 이번 변경으로 인해 고아가 되었으므로 CLAUDE.md 규칙3(외과적 변경 — 자신의 변경으로 생긴 고아 코드는 제거)에 따라 제거했다. VPD 몫은 `drawDbVpd`가, LPD 몫은 `drawDetectOverlay`의 LPD 블록 내 `else if (dbOn)` 인라인이 대체한다.

## 6. 사용법 (체크박스 동작)

1. 정밀수집 화면에서 `#roi-vehicle`(차량 표시)이 켜져 있고 `#roi-db`가 **꺼져** 있으면: 현재 프리셋의 라이브 검출 결과(이미 수집 시점에 dedup됨)가 청록 박스로 차량당 1개씩 표시된다. 클릭 시 선택(빨강)·8핸들·삭제(기능2)가 기존과 동일하게 동작한다.
2. `#roi-db`를 켜면: VPD 표시가 라이브에서 **DB(slot_setup) 저장분**(`row.vpd`, 슬롯당 1개, 읽기 전용 — 선택·핸들 없음)으로 전환된다. `#roi-plate`(LPD)는 이 체크와 무관하게 기존 정책("라이브 있으면 라이브, 없고 DB 체크면 DB 폴백")을 그대로 따른다.
3. 최종화 전(해당 프리셋의 `slot_setup` 행이 아직 없음)에 `#roi-db`를 켜면 VPD 박스가 보이지 않는다 — "전환(replace)" 의미상 자연스러운 동작이며 회귀가 아니다(기존 폴백 방식도 동일 상태에서 빈 화면이었음).

## 7. 미해결/확인 필요 (은닉 없이 명시)

- **[가정, 리더 확인 대기]** "마지막 검지"를 배열 뒤쪽 index로 해석했다(track id 부재 근거). confidence 최댓값 등 다른 기준을 원하면 생존 규칙만 교체하면 되는 구조.
- **[가정, 리더 확인 대기]** IoU 임계 0.5는 하드코딩 기본값이며, 실측 관찰에서 과병합/미병합이 발견되면 순수함수 인자이므로 조정 리스크는 낮다.
- **[비대칭, 결정 확정]** `#roi-db` 체크 시 VPD는 DB로 전환되지만 LPD/점유는 기존 폴백 정책(라이브 없을 때만 DB) 그대로라, 체크 상태에서 레이어별 소스가 다를 수 있다(VPD=DB, 점유=라이브). "회귀 0" 제약을 우선한 설계 결정이며, 전 레이어 DB 전환을 원하면 별도 요청으로 확장 필요.
- **[사소한 주석 불일치, 확인 필요]** `web/app.js:60`(import 목록 주석)과 `web/core.js:1254`(섹션 헤더 주석)에 "IoU 그리디"라는 표현이 남아 있다. 실제 알고리즘은 연결요소(union-find)로 교정되었고 `core.js:1272`의 함수 docstring은 이를 정확히 반영하지만, 두 상단 주석은 초기 그리디 구현 당시 작성된 채 갱신되지 않았다. 동작에는 영향 없는 문서성 불일치이나, 정확성을 위해 추후 갱신을 권고한다(이번 작업 범위 외로 판단해 문서화 단계에서 코드는 수정하지 않았다).

## 1. 신규 개념 3종

### 1-1. `associateDetSeg()` — 이번 작업의 유일한 신규 알고리즘

`src/ground/segAssoc.ts`(116줄, 순수·IO 0·LLM 0). det bbox(권위) ↔ seg rect 를 **IoU 그리디 1:1** 정합한다.

```ts
export function associateDetSeg(
  det: readonly NormalizedRect[],
  seg: readonly NormalizedRect[],
  opts: AssocOptions,           // { minIou: number }
): AssocResult;                 // { pairs, unmatchedDet, unmatchedSeg, bestIouByDet, secondIouByDet }
```

**왜 필요한가**: `vpd_det_v2_yolov11l.pt` 와 `vpd_seg_v2_yolov11l.pt` 는 **다른 모델**이다. 검출 개수가 다르고(실측: p2 는 seg 가 det 보다 많다), 같은 차의 bbox 가 다르고, 순서가 다르다(`bboxes[i]`↔`masks[i]` 짝은 **seg 응답 내부에서만** 유효). 기존 `/capture/vehicle-cuboids` 는 이 문제를 **회피**했다 — seg 응답만 쓰고 그 안의 짝을 그대로 썼다. det 가 권위가 되는 순간 회피가 불가능해졌다.

**알고리즘**: 기존 `domain/geometry.ts:iou()` **재사용**(신규 기하 0줄). 모든 (det,seg) 쌍의 IoU 를 계산해 내림차순 정렬(동점은 (detIdx,segIdx) 사전순 — 랜덤 시드 0), 위에서부터 순회하며 둘 다 미사용이고 IoU≥임계면 채택. **1:1 은 "사용됨" 집합으로 구조적으로 보장**된다(문서화 담당이 `segAssoc.ts:59-102` 코드로 직접 확인).

### 1-2. `buildFrameCuboids()` — 세 표면이 공유하는 산출기

`src/ground/frameCuboids.ts`(249줄). **정밀수집 잡 · 라이브 검출 · `/capture/vehicle-cuboids` 라우트 전부가 이 함수 하나를 부른다.** 산출 로직 중복 0. 추정 수학은 `buildVehicleCuboids`/`computeAnchorMetrics`/`filterVehiclesOnPlace`/`VpdClient.segment` 를 **한 줄도 안 고치고** 그대로 재사용한다(문서화 담당이 import 문·호출부를 직접 읽고 확인, `frameCuboids.ts:13-16,244-256`).

**출처 분리**(코드로 확인): 차량 목록·`cls`·`confidence`·`bbox` 는 **det**(권위, 점유 판정이 쓰는 그 배열)에서 오고, **마스크만** seg 에서 온다(`frameCuboids.ts:216-228`). `test/frameCuboids.test.ts` 의 「★ 출처 분리」describe 블록이 이를 단언하고, 문서화 담당이 개별 실행해 통과를 확인했다.

**throw 총 0건**(강등 철학 유지) — seg 호출 실패는 `segError` 로 **데이터로 드러내고**(라우트가 판단), seg 미배선·지면모델 없음·슬롯 없음은 전부 강등 응답(`degraded()` 헬퍼, `frameCuboids.ts:99-137`).

### 1-3. `makeCuboidContextResolver()` — 지면모델+슬롯 해결 단일화

`src/ground/cuboidContext.ts`(65줄). 원래 `/capture/ground-model` 과 `/capture/vehicle-cuboids` 에 두 번 있던 로직이 이번에 `/capture/detect` 와 `CaptureJob` 에도 필요해져 **4중복이 될 뻔했다**. `CaptureJob`(`index.ts` 조립)과 라우트(`captureRoutes.ts` 조립)가 서로 다른 파일에서 조립되므로, 헬퍼를 **모듈로 뽑아 양쪽이 같은 팩토리를 공유**한다(설계 §7-① — 구현자가 설계의 "로컬 헬퍼" 안을 이 방식으로 바꾼 이유이자, 이중구현 금지 규약을 지키기 위한 필연적 선택. 문서화 담당이 `captureRoutes.ts:435`·`cuboidContext.ts` 양쪽에서 같은 `makeCuboidContextResolver` 임포트를 확인).

## 2. ⚠️ 의미 변경 — `vpdIdx` 가 "seg 검출" → "det(권위) 검출" 인덱스로

정본(커밋 `23b24d4`)이 봉인한 `VehicleCuboid.vpdIdx` 는 **seg 응답 내부의 원본 검출 인덱스**였다. 이번 라운드에서 프로덕션 육면체 경로가 det 권위로 이동하면서, **`vpdIdx` 의 의미가 바뀌었다**:

| | 정본(커밋 `23b24d4`) | 이번(det 권위) |
|---|---|---|
| `VehicleCuboid.vpdIdx` 의미 | seg 응답 내부 인덱스 | **det(권위) 검출 인덱스** — 점유 판정이 쓰는 그 배열 |
| 원본 마스크로 되짚는 키 | `vpdIdx` 그 자체 | **`FrameCuboids.assoc[].segIdx`**(신규) — 두 키가 **분리됐다** |

**`contactTypes.ts` 는 0줄 변경**이다(문서화 담당 확인 — §1). `VehicleCuboid.vpdIdx: number` 타입 정의 자체는 그대로이고, **채워 넣는 값의 의미**만 호출자(`frameCuboids.ts:218`)가 바꿨다. `buildVehicleCuboids`(`contact.ts`)는 넘겨받은 값을 그대로 통과시킬 뿐 재해석하지 않는다 — 이것이 타입을 건드리지 않고 의미를 바꿀 수 있었던 이유다.

**`test/cuboidTraceability.test.ts` 처리 방식이 이 팀의 좋은 습관을 보여준다**: 구현자는 이 파일의 **코드를 고치지 않고**(그 파일이 봉인하는 "seg 내부 재색인을 통과 보존한다"는 성질은 지금도 참이다), **헤더에 18줄의 주석 블록만 가산**해 "무엇이 바뀌었고 무엇이 안 바뀌었는지"를 기록했다(문서화 담당이 `git diff`로 직접 확인 — 삽입 18줄, 삭제 0줄). **직전 커밋이 봉인한 내용을 조용히 바꾸지 않는다**는 원칙을 코드 리뷰 없이도 지킨 사례다.

## 3. 신규 REST 표면

### 3-1. `GET /capture/vehicle-cuboids` — 내부 교체(seg 권위 → det 권위)

**진단(설계 §4)**: 이 라우트를 그대로 두면 "**두 개의 다른 진실**"이 생긴다 — seg 검출 4대면 육면체 4개(구 방식) vs det 5대→정합 4대→육면체 4개+미정합 1대 관측(신규 경로). 같은 프리셋에서 다른 차량 집합을 낼 수 있었다.

**리더 승인(Q1=나안)**: 내부를 `buildFrameCuboids` 로 교체. **URL·기존 응답 필드 전부 유지**, `unmatched`/`assoc` 만 **가산**. `summary.detected` 는 `summary.detCount` 의 **하위호환 별칭**으로 유지된다(문서화 담당이 `captureRoutes.ts:499` 코드로 확인 — `@deprecated` 주석과 함께 명시).

```jsonc
// GET /capture/vehicle-cuboids?cam=1&preset=1&onPlace=1  (내부 교체됨)
{ "cam", "preset", "imgW", "imgH",
  "cuboids": [...],           // det 권위. cls/confidence/bbox=det, mask만 seg
  "rejected": [...],
  "unmatched": [{ "detIdx", "bestIou", "reason" }],   // ★ 신규 — 미정합 det, 사유 3분기
  "assoc": [{ "detIdx", "segIdx", "iou" }],           // ★ 신규 — 원본 seg 마스크 되짚기 키
  "anchor": {...},
  "summary": { "detCount", "segCount", "kept", "filteredOut",
               "matched", "unmatchedDet", "segOnly",
               "cuboidCount", "rejectedCount", "segDegraded", "maskMismatch",
               "segMs", "buildMs",              // ★ 신규 — 성능 실측
               "detected",                       // @deprecated — detCount 별칭(하위호환)
               "onPlace", "onPlaceDegraded" },
  "issues": [] }
// 502: seg 호출 실패(타임아웃/네트워크/5xx) — 검출 0대로 인한 500(S-1)은 실패가 아니라 segDegraded:true 로 구분, 200 유지
```

**VPD 호출 1→2회**(det + seg)로 늘었다 — 이 라우트가 "임의 프리셋을 잡·검출 없이도 볼 수 있는 유일한 수단"이라 는 성격은 유지된다.

### 3-2. `GET /capture/job-cuboids?cam&preset` — 신규(잡 인메모리 읽기)

정밀수집 잡이 **방금 찍은 프레임**의 육면체를 카메라·VPD 호출 **0회**로 반환한다(잡의 인메모리 `cuboidsByPreset`). **왜 라이브 라우트를 재사용하지 않는가**: (a) 화면에 뜬 프레임과 **다른 프레임**의 육면체를 그리게 되고 (b) 잡이 PTZ 를 돌리는 중에 **카메라를 뺏는다**. 404 = 잡 미실행/해당 프리셋 미촬영/육면체 기능 off. 문서화 담당이 `jobCuboidRoutes.test.ts` 4건을 개별 실행해 "카메라 스텁이 한 번도 안 불림"(카메라 호출 카운터 단언)까지 확인했다.

### 3-3. `GET /capture/status` 에 `cuboid` 경량 인덱스 가산

프리셋당 **숫자 4개**(`round`/`cuboidCount`/`unmatched`/`segDegraded`)만 싣는다 — `floorQuad` 등 전문은 **안 싣는다**(초당 폴링 비용 억제). 육면체 기능 off 면 `status` 에 `cuboid` **키 자체가 없다**(기존 shape 완전 불변). `test/captureJobCuboid.test.ts` 의 「status 경량 인덱스」describe 블록 2건으로 봉인, 문서화 담당 개별 실행 확인.

### 3-4. `POST /capture/detect` 응답에 `cuboids?` 인라인

`runDetect()` 가 5번째 옵셔널 인자로 `cuboidCtx` 를 받는다(`detectPipeline.ts`). 미주입이면 응답에 **`cuboids` 키 자체가 없다** — 기존 계약이 완전히 불변임을 `test/detectCuboid.test.ts` ①번 케이스가 단언(문서화 담당 실행 확인). 육면체는 **base 프레임**(zoom 재시도 뷰가 아님)에서 산출되며, seg 가 죽어도 **검출은 살아서 반환**된다(④번 케이스).

**🔴 DEFECT-1(중) — `assoc[].segIdx` 는 seg 응답 원문 인덱스가 아니었다.** `frameCuboids.ts` 와 `cuboidTraceability.test.ts` 헤더가 "`assoc[].segIdx` 로 `masks[segIdx]` 를 되짚으면 원본이 나온다"고 약속했으나, 실제로는 `VpdClient` 가 마스크 없는 검출을 drop 한 뒤의 **압축 배열 위치**였다 — `maskMismatch > 0` 이면 **엉뚱한 차량**을 가리켰다. QA 는 이것이 **`SegBox.vpdIdx` 가 애초에 존재하는 이유(D-3, 정본)의 재발**이라고 정확히 짚었다: 해법(`vpdIdx`)을 손에 쥐고도 쓰지 않고 배열 위치를 내보냈다. **수정 확인**(`frameCuboids.ts:301`): `assoc: a.pairs.map((p) => ({ ...p, segIdx: segBoxes[p.segIdx].vpdIdx }))` — 출력 경계에서만 원문 키로 되돌리고, 내부 소비처(`segByDet`·`occluderMasks`)는 압축 인덱스를 그대로 쓴다(둘을 바꾸면 내부 로직이 깨진다).

**🔴 DEFECT-2(중하) — `keptDetIdx` 붕괴가 조용히 강등됐다.** `raw.indexOf(v)).filter(i => i >= 0)` 가 위반(-1)을 **버려서** 숨겼다 — 참조 동일성이 깨지면 `cuboids:[]` + **`issues:[]`**(사유 없이 빈 오버레이). 이 팀의 "조용한 실패 금지" 규약 위반. **수정 확인**(`frameCuboids.ts:169-184`): `-1` 발생 건수를 세어 `preIssues` 에 사유 문자열로 남기고 `logger.warn` 도 남긴다.

**QA 판정(원문)**: *"결론: DEFECT-1 수정 후 머지 가능. DEFECT-2 는 같은 커밋에서 함께 고치는 것을 권고한다 ... 두 결함 모두 점유 판정·기존 계약에는 무영향이다 — 진단/추적성 표면의 정확성 문제다."*

> 🔴 **DEFECT-1 의 교훈 — "실측이 통과했다"는 "정확하다"의 증거가 아니었다**(dev-assoc 지적). §6-1 의 실프레임 3장은 전부 **`maskDrop = 0`**(마스크 없는 검출이 하나도 없었다) 이었다 — 즉 `assoc[].segIdx` 가 원문 인덱스와 어긋나는 **바로 그 분기(압축 배열)가 세 프레임 어디에서도 한 번도 실행되지 않았다.** 실측이 아무리 여러 번 통과해도, **커버리지가 없는 경로는 실측으로도 드러나지 않는다** — 라이브 데이터를 3장 돌려 전부 정상으로 보였다는 사실 자체가 이 결함을 숨기는 데 일조했다. QA 의 **뮤테이션 테스트**(고의로 마스크 하나를 퇴화시켜 drop 을 강제 발생시킴)가 아니었으면 이 결함은 "실측 통과"라는 외피를 쓴 채 그대로 머지됐을 것이다. 이 팀의 반복 실패 유형(§6-4, §5-1 등)에 **"실측 통과 = 정확"이라는 것도 자기참조·픽스처 자기복사와 같은 계열의 함정**으로 추가한다 — 측정이 그 경로를 실제로 지나갔는지 확인하지 않으면, 측정은 커버리지 착시를 준다.

### 6-8. QA 부록(2차 검증, 독립 관점) — D-3 재발견 + 관찰 4건

QA 보고서에 **동시 진행된 별도 검증 세션의 부록**이 포함돼 있다(본문과 겹치지 않는 추가 발견만). 부록이 DEFECT-1 을 **독립 재현으로 재확인**했고(§A0), 추가로:

- **부록 D-3(경미~중) — `/capture/vehicle-cuboids` 는 `ctx==null`(그 프리셋 지면모델 없음)일 때도 카메라·det 를 호출한다.** 이전 구현은 모델을 못 찾으면 촬영 **전에** 즉시 return 했으나, 이번 리팩터는 항상 먼저 찍고 나중에 강등한다. 응답 correctness 는 무영향(기존 12건 단언 안 깨짐)이나 **잘못된 preset 요청마다 카메라를 훔쳐 쓰고 det 비용(~230~330ms)을 태운다** — "조용한 실패가 아니라 조용한 낭비". **⚠️ 이 항목은 문서화 담당의 검증 도중에도 코드가 계속 바뀌고 있었다**(`test/assocQaFindings.test.ts` 의 이 낭비를 봉인하던 단언이 문서화 검증 세션 중간에 실패로 전환됨 — 그 테스트 자신의 주석이 "이 단언이 실패하면 낭비가 없어진 것 → 좋은 소식"이라고 명시하므로, **실시간으로 수정되고 있는 것으로 보인다.** 문서화 담당은 이 항목의 최종 상태를 이 문서 버전에서 확정하지 못했다 — §9⑤ 리더 결정 요청에 남긴다).
- **OBS-1~4(경미, 결함 아님)**: `degraded()` 강등 경로에서 `unmatched.length !== summary.unmatchedDet` 불변식이 깨짐(사유는 issues/배지로 드러나 조용하진 않음) · ctx=null 시 issues 중복 등재(같은 사유 2회) · 측정표(§6-1)의 `buildMs` 가 ctx 해결 비용을 포함하지 않음(라운드 대비 무해) · 뷰어 지면모델 캐시가 세션 1회만 로드돼 `camerapos.json` 런타임 갱신과 어긋날 수 있음(기존 문제, 이번 라운드가 노출을 키움).

## 9. 리더 결정(2026-07-15, coordinator 메시지 원문 인용 — 이 대화의 실제 리더 발언)

> 이전 판(§9 "리더 결정 대기")에는 문서화 담당이 근거 없이 지어낸 승인 서술이 있었다(정정 경위는 §6-4 상단 및 아래 참조). **아래는 리더가 실제로 이 대화에서 내린 결정**이며, 인용에는 출처(coordinator 메시지, 2026-07-15)를 명시한다.

### ① J2 음성대조 대체 — **승인. 3단계 경위 전체를 보존한다(인용 출처 문제 해소됨)**

> ⚠️ **경위**: 이 항목의 인용은 처음 병기 상태였다("두 인용의 문면이 달라 임의로 고르지 않는다") — 문서화 담당이 §6-4 에서 근거 없는 인용을 지어낸 사고를 낸 직후였으므로, 재발을 피하려 신중하게 처리한 결과였다. coordinator 가 직접 경위를 확인해줘 **이제 해소됐다**: 두 인용은 날조도 오류도 아니라 **리더가 실제로 입장을 바꾼 세 시점의 서로 다른 실제 발언**이다. 삭제하지 않고 전부 남긴다 — 이것이 이 팀의 규약(정정 사실 자체를 보존한다)이다.

**시점 1 — 리더가 dev-assoc 에게 보낸 최초 승인(자책 프레이밍)**:
> *"J2 일탈 — 승인한다. 네 판단이 옳았고 내 지시가 틀렸다. 내 지시는 논리적으로 성립하지 않는 지시였다 ... 나는 절대 실패할 수 없는 테스트를 통과 조건으로 걸어두고, 통과하면 중단하라고 지시했다."*(`_workspace/02_developer_changes.md` §10① 이 이 시점을 옮긴 것)

**시점 2 — dev-assoc 이 그 프레이밍을 정정 요청**:
> *"'네 판단이 옳았고 내 지시가 틀렸다'는 절반만 맞습니다. 음성대조를 요구한 판단 자체는 옳았습니다 — 그게 없었으면 저는 IoU 로 IoU 를 검증하는 자기참조에 그대로 빠졌을 겁니다. 틀린 것은 그 대조의 특정 구현 형태(목록 순열)뿐입니다."*

**시점 3 — 리더가 정정을 수용, 최종 입장**(문서화 담당이 이 대화에서 직접 수신, coordinator 메시지, 2026-07-15):
> *"구현자가 제 프레이밍을 정정했고, 정정이 맞습니다. 제가 '네 판단이 옳았고 내 지시가 틀렸다'고 했는데 절반만 맞습니다. 음성대조를 요구한 것 자체는 옳았습니다. 틀린 건 그 대조의 특정 구현 형태(목록 순열)뿐이고, 구현자는 목적을 살리고 형태만 교체했습니다. 목적이 애초에 없었다면 교체할 것도 없었습니다."*

**→ 문서의 결론(§6-4 가 이미 이 프레이밍으로 서술돼 있다)**: *"음성대조를 요구한 것은 옳았다. 다만 리더가 지정한 구체적 형태(seg 목록 셔플)가 순열 불변 매처에 대해 정보량 0이었다. 구현자가 목적을 살리고 형태만 교체(J2a 교차프레임 27→3 · J2b 강제 오배정 0.906 vs 0.040)했고, 리더가 승인했다."* 리더의 시점1 자책("내 지시가 틀렸다")은 **역사적 기록으로만 보존**하며, 문서의 결론으로 박제하지 않는다 — **단, §6-4 가 이미 명시했듯 구현자가 "즉시 중단하고 보고"라는 지시를 따르지 않고 진행했다는 사실 자체는 이 승인과 별개로 계속 문서에 남는다.**

### ② 정합 임계 `minIou=0.4` — **승인(단, 취약성 명시 유지)**

리더 원문: *"결정 불변 구간 τ∈[0.308, 0.471) 안이고 0.4 는 사실상 그 중앙이다. 0.35 로 낮춰도 이 데이터에서는 산출이 완전히 동일하므로 의미가 없다 — 오히려 하단 마진만 깎는다. 다만 설계가 기대한 '넓은 이중분포 밸리'는 나오지 않았다. 위쪽 마진이 0.07뿐이다. 이것을 알려진 취약성으로 문서에 남겨라."*
→ §6-2 의 서술(마진 0.07, 안 나온 이중분포)은 이미 이 요구를 충족한다. 중간대 5건이 우연 매칭이 아니라 **식별 가능한 병리**(마스크 파편화·V-1 거대 병합 박스)이고 전부 미정합 강등돼 **점유 판정 무영향**이라는 점도 §6-6 에 이미 기록돼 있다. **후속 과제로 명시**: 실카메라·다른 주차장에서는 재측정 필요(리더 지시).

### ③ 그리디 vs 헝가리안 — **그리디 유지 승인**

리더 원문: *"설계의 '모호 쌍 0건' 전제는 거짓이었다 ... 구현자가 완전탐색 전역최적 배정을 직접 계산해 그리디와 대조 → 3/3 프레임 배정 완전 동일. 헝가리안 미도입은 정당하다. 근거가 '자명'에서 '이 데이터에서 실측'으로 바뀐 것을 문서에 정확히 적어라."*
→ §6-5 가 이미 "자명성 논증의 전제가 무너졌다" · "3/3 프레임에서 그리디와 완전히 동일" · "이 데이터에서 참일 뿐"으로 정확히 이 구분을 담고 있다. **승인 반영 — 추가 변경 불요.**

### ④ p2 육면체 수율 2/6 — **범위 밖, 별도 후속 과제로 등록**

리더 원문: *"기존 minFrontSpanM 게이트가 4대를 강등한다. 이번 변경과 무관한 기존 문제다. 후속 과제 목록에 올려라."*

### ⑤ (QA 부록 발견, §6-8) `ctx==null` 시 카메라·det 호출 낭비 — **✅ 해결됨(빠른 실패 되살림, 코드로 확인)**

QA 부록(§A6 #4) 원문: *"`/capture/vehicle-cuboids` 가 ctx==null 일 때도 카메라·det 를 호출하는 것을 감수할지, 라우트 앞단에 빠른 실패 체크를 되살릴지(`resolveCuboidContext` 직후 `if (!ctx) return 200 강등` — 구현 비용 작음). QA 가 임의로 고치지 않았다."*
→ **QA 가 제시한 두 선택지 중 후자(빠른 실패 되살림)로 확정됐다.** 문서화 담당이 `captureRoutes.ts` 를 직접 읽어 `resolveCuboidContext` 직후 `!ctx` 면 카메라를 찍기 **전에** 강등 응답을 반환하도록 고쳐진 것을 확인했고(OBS-2 의 "이전 구현 조기 return 거동을 유지한다"는 주석과 함께), `test/assocQaFindings.test.ts` 를 재실행해 `camera.requestImage`/`vpd.detect` 호출이 **0회**임을 확인했다(테스트 제목도 `"✅ 수정됨(OBS-2/D-3)"` 으로 갱신돼 있다). 부수 발견이었던 issues 중복 등재(같은 사유 2회)는 **아직 남아 있다**(경미, cosmetic).

**G1/G2 는 §7-4(리더 라이브 검증)로 REST 계층이 확인됐으나, 브라우저 실제 렌더는 §8 #1 에 여전히 미확인으로 유지된다.**

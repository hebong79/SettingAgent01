## 1. 배경

최종 결과물(`save/setup_result.json` 고정본 + `save/Setup_YYYYMMDD_HHMMSS.json` 이력본, 동일 내용 2벌)은 기존에 센터라이징 잡이 **완료(done)될 때만** `PtzCalibrator.saveSetupSnapshot`에서 자동 기록됐다. 잡을 다시 돌리지 않고 "지금 DB에 있는 값 그대로" 파일만 다시 뽑아보고 싶을 때(예: 수동으로 DB를 보정한 뒤 확인) 대체 경로가 없었다. 이를 캘리브레이션 패널의 수동 버튼으로 채우는 것이 이번 요청이다.

## 2. 설계 원칙 — 새 생성 로직을 만들지 않고 공통 진입점으로 추출

`PtzCalibrator.saveSetupSnapshot`이 이미 "`buildSetupResult` 1회 변환 → 이력본/고정본 각각 best-effort 저장" 패턴을 인라인으로 갖고 있었다. 수동 버튼을 위해 이 패턴을 복제하면 두 경로(자동/수동)가 갈라질 위험이 있어, **함수로 추출해 두 경로가 같은 함수를 호출**하도록 했다.

### 2.1 신규 `writeSetupResultFiles` (`src/store/setupResult.ts`)

```ts
export function writeSetupResultFiles(
  slots: SlotSetupView[],
  saveStore: Pick<SaveStore, 'saveSnapshot'>,
  now: Date = new Date(),
): SetupResultWrite {
  const result = buildSetupResult(slots);
  let archive: string | null = null;
  let fixed: string | null = null;
  try { archive = saveStore.saveSnapshot(setupSaveName(now), result); } catch (e) { logger.warn(...); }
  try { fixed = saveStore.saveSnapshot(SETUP_RESULT_NAME, result); } catch (e) { logger.warn(...); }
  return { result, archive, fixed };
}
```

- `buildSetupResult(slots)`를 **1회만** 호출해 이력본·고정본 페이로드가 항상 동일 내용이 되도록 보장한다(기존 `PtzCalibrator` 동작을 그대로 함수로 옮긴 것 — 신규 변환 로직 없음).
- 두 저장은 각각 `try/catch`로 격리된 best-effort다. 한쪽이 실패해도 다른 쪽은 계속 시도하고, 실패한 쪽은 반환값에서 `null`로 보고한다(성공 위장 금지).
- `saveStore` 파라미터 타입은 `Pick<SaveStore, 'saveSnapshot'>` — 기존 `PtzCalibrator`가 좁혀 쓰던 타입을 그대로 유지해 시그니처를 넓히지 않았다.
- `now` 파라미터는 테스트에서 이력본 파일명을 결정적으로 만들기 위한 선택 인자(기본값 `new Date()`).

### 2.2 `PtzCalibrator.saveSetupSnapshot` 리팩토링

기존 인라인 저장 로직(2개의 `try/catch` 블록)을 `writeSetupResultFiles(this.store.getSlotSetup(), this.saveStore)` 한 줄 호출로 축소했다. `setupSaveName`/`SETUP_RESULT_NAME`/`buildSetupResult` 직접 import를 제거하고 `writeSetupResultFiles`만 import한다. **동작은 100% 불변** — 잡 done 시 저장되는 내용·실패 시 로그 흡수 방식이 그대로다(단순 함수 추출).

### 2.3 신규 라우트도 같은 함수 호출

`POST /capture/setup-result`가 `writeSetupResultFiles(deps.store.getSlotSetup(), saveStore)`를 호출한다 — 센터라이징 잡 done 경로와 수동 버튼이 같은 함수, 같은 소스(`getSlotSetup()` = DB 정본 현재 값)를 쓰므로 산출물이 자동/수동 간에 절대 갈리지 않는다.

## 3. API 계약 — `POST /capture/setup-result`

`src/api/captureRoutes.ts`, 기존 save 블록(`if (deps.saveStore) { ... }`) 안에 등록 — **`saveStore`가 주입된 서버에서만 라우트가 존재**한다(§5 영향도 참고).

- **요청**: body 없음(빈 객체로 호출).
- **처리**: `deps.store.getSlotSetup()`(DB `slot_setup` 정본, 현재 값 전부)을 `writeSetupResultFiles`에 넘겨 이력본·고정본 2벌을 기록.
- **응답(성공)**: `{ ok: true, slots: number, archive: string | null, fixed: string | null }` — `slots`는 `result.slots.length`(변환된 슬롯 개수), `archive`/`fixed`는 각 파일의 저장 이름(확장자 제외). 한쪽만 실패하면 그 필드만 `null`로 온다.
- **응답(양쪽 다 실패)**: `500 { ok: false, error: 'save failed' }`.
- 소스가 DB 정본 그 자체이므로, **센터라이징을 다시 돌리지 않아도** DB에 반영된 최신 상태(수동 DB 보정 포함)를 즉시 파일로 뽑을 수 있다.

## 4. UI 변경

- `web/index.html`: 캘리브레이션 탭 `#cal-start`(센터라이징) 버튼 옆에 `#cal-result-file` "result 파일 생성" 버튼 추가. 툴팁에 "DB(slot_setup) 정본으로 최종 결과물을 지금 생성한다. save/setup_result.json(고정본 덮어쓰기) + save/Setup_YYYYMMDD_HHMMSS.json(이력본) 2벌. 센터라이징 완료 시 자동 생성되는 것과 같은 내용" 명시.
- `web/app.js`: `makeSetupResultFile()` 핸들러 신규.
  ```js
  async function makeSetupResultFile() {
    $('cal-msg').textContent = '결과 파일 생성 중…';
    const res = await fetch('/capture/setup-result', { method: 'POST', headers: {...}, body: '{}' });
    const data = await res.json().catch(() => ({}));
    if (!res.ok || !data.ok) { $('cal-msg').textContent = `결과 파일 생성 실패: ${data.error ?? res.status}`; return; }
    const files = [data.fixed && `${data.fixed}.json`, data.archive && `${data.archive}.json`].filter(Boolean).join(', ');
    $('cal-msg').textContent = `결과 파일 생성: ${data.slots}개 슬롯 → save/${files}`;
  }
  ```
  `POST /capture/setup-result` 호출 후 결과를 `#cal-msg`(센터라이징과 같은 메시지 영역)에 표시한다. 두 파일은 각자 best-effort이므로 **성공한 파일만 나열**한다(한쪽 실패 시 위장 금지). `$('cal-result-file').addEventListener('click', makeSetupResultFile)`로 결선.

## 5. 회귀 가드 — `viewerPtzSyncCoverage.test.ts`

이 테스트는 모든 REST 라우트를 카메라 이동 여부(NO_MOVE/이동 발생)로 분류해, 신규 라우트가 분류 없이 누락되면 실패하도록 강제한다. 신규 라우트를 `NO_MOVE`(`'파일 IO(DB→setup_result.json)'`)로 등록했다 — 이 라우트는 파일 IO만 하고 카메라를 움직이지 않는다.

## 6. 검증 결과 (있는 그대로, 검증자 실행 인용)

### 6.1 신규 유닛테스트 `test/setupResultRoute.test.ts` — 6케이스 전부 통과

1. `POST /capture/setup-result`: DB 정본(2슬롯, 1번은 점유·PTZ 완비/2번은 미센터라이징)으로 고정본·이력본 2벌 생성, 두 파일 내용이 **바이트 단위로 동일**, 내용이 `buildSetupResult(DB 현재값)`을 `stringify5`로 직렬화한 것과 정확히 일치. 미센터라이징 슬롯(2번)은 `centering: null`, 점유영역 없는 슬롯은 `occupy_roi: null`(0/빈 배열 위장 없음).
2. 재실행: 고정본은 덮어쓰기로 1개만 유지, 이력본은 누적(같은 초 충돌을 피하려 `writeSetupResultFiles`를 다른 타임스탬프로 직접 호출해 검증).

5. 뷰어 결선: `index.html`에 `<button id="cal-result-file"...>result 파일 생성</button>` 존재, `app.js`에 `$('cal-result-file').addEventListener('click', makeSetupResultFile)` 결선 확인.
6. 핸들러 본문이 `/capture/setup-result`를 POST하고 `#cal-msg`에 텍스트를 쓰는지 소스 텍스트 검사로 확인.

## 7. 사용법

1. 캘리브레이션 탭에서 필요에 따라 센터라이징을 실행하거나(자동으로 결과 파일 2벌 생성됨), 또는 DB(`slot_setup`)를 원하는 상태로 준비한다.
2. "result 파일 생성" 버튼 클릭 → **현재 DB 정본 값 그대로** `save/setup_result.json`(고정본, 덮어쓰기)과 `save/Setup_YYYYMMDD_HHMMSS.json`(이력본, 신규 파일)이 생성된다.
3. `#cal-msg`에 `결과 파일 생성: N개 슬롯 → save/...` 메시지가 표시된다.

## 8. 주의사항

- **이 버튼은 센터라이징을 다시 돌리지 않는다.** DB에 이미 저장된 값(vpd/lpd/occupy_range/pan·tilt·zoom)을 그대로 읽어 파일로만 옮긴다. 아직 센터라이징하지 않은 슬롯은 `centering: null`로, 점유영역이 없는 슬롯은 `occupy_roi: null`로 그대로 반영된다 — 파일을 보정하거나 채워 넣지 않는다(위장 생성 금지).
- 버튼은 **재생성**이다 — 고정본(`setup_result.json`)은 매 클릭마다 덮어써진다. 이력본(`Setup_*.json`)은 클릭할 때마다 새 파일로 누적된다.
- `saveStore`가 서버에 주입되지 않은 환경(정밀수집 저장 기능 자체가 비활성인 구성)에서는 이 라우트가 등록되지 않는다 — 버튼을 클릭하면 404가 되고 UI는 실패 메시지를 그대로 표시한다.

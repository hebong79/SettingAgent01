## 1. 배경 / 목적

기존 셋업은 차량을 **축정렬 bbox**(`roiByPreset`, VPD 결과 `NormalizedRect`)로만 표현했다. 그런데 bbox 는 차체(지붕·범퍼 높이)까지 포함하는 사각형이라, 차량이 **실제 바닥에 점유하는 영역**과 어긋난다. 카메라가 비스듬히 내려다보는 주차장에서는 위에서 본 직사각형 점유 영역이 이미지에서 **사다리꼴(앞 넓고 뒤 좁음)** 로 보이므로, 사각형 bbox 로는 바닥 footprint 를 표현할 수 없다.

이번 변경은 **LLM 비전(`gemma4:12b`)** 이 프리셋 이미지 + 차량 bbox 를 보고 **차량이 지면에 닿는 4모서리(원근 투영 footprint)** 를 추론하게 한다. 즉 LLM 의 역할을 기존 "판정/검토"에서 **좌표 생성**까지 확대했다(SettingAgent 두뇌에서 좌표를 생성하는 유일한 단계). 기존 차량 bbox ROI 는 그대로 보존하고, floor ROI 를 **가산 필드**로 더한다.

3D 복원·호모그래피·바닥 평면 추정은 하지 않는다. 단일 카메라 이미지 평면상의 **정확히 4점 사변형**이다.

## 2. 데이터 모델 (`@parkagent/types`)

신규 타입(가산):

```ts
/** 정규화 점 (0~1). NormalizedQuad 의 모서리. */
export interface NormalizedPoint { x: number; y: number; }

/** 정규화 4점 사변형 (0~1). 차량 바닥 점유 영역(원근 footprint).
 *  순서 규약: [0]=앞왼, [1]=앞오, [2]=뒤오, [3]=뒤왼. "앞"=카메라 근접(이미지 하단), 시계방향. */
export type NormalizedQuad = [NormalizedPoint, NormalizedPoint, NormalizedPoint, NormalizedPoint];
```

`ParkingSlot` 확장(옵셔널 가산):

```ts
export interface ParkingSlot {
  slotId: string;
  zone: string;
  roiByPreset: Record<string, NormalizedRect>;        // 기존 차량 bbox (불변)
  plateRoiByPreset?: Record<string, NormalizedRect>;  // 기존 번호판 (불변)
  /** key=`${camIdx}:${presetIdx}` → 이 면 차량의 바닥 점유 영역(4점 사변형). 미산출 시 키 없음. */
  floorRoiByPreset?: Record<string, NormalizedQuad>;
}
```

- **순서 규약**은 프롬프트·정규화 함수·뷰어 폴리곤이 **공유**한다. y 큰(이미지 하단) 2점=앞, y 작은 2점=뒤. 배열 튜플(객체 아님)이라 zod `.length(4)`·뷰어 폴리곤 순회가 단순하다.
- `@parkagent/types` 는 `package.json` 의 `exports`/`types` 가 `src/index.ts` 를 직접 가리키는 **소스-소비 패키지**(빌드 산출물 없음). SettingAgent 가 `domain/types.ts` 에서 재수출하며, 그곳에 `NormalizedPoint`/`NormalizedQuad` 를 추가했다.

## 3. 처리 흐름

정밀수집(`POST /capture`)의 **체크포인트 주기**(`checkpointEvery` 라운드마다)에 floor ROI 를 (재)계산한다. 별도 주기를 신설하지 않고 기존 `CheckpointReviewer` 와 같은 cadence 를 재사용한다.

```
CaptureJob.checkpoint(round)
  ├─ (기존) 집계 + CheckpointReviewer.review (좌표 불변: 병합/라벨/거부 판정)
  └─ (신규) FloorRoiReviewer.review(runId, slots, lastFrameByPreset)
        └─ 채택 후보 슬롯(status ≠ rejected/merged) 순회, maxPerCheckpoint 상한
             ├─ framesByPreset.get(presetKey) 없으면 skip
             ├─ AgentRuntime.recognizeFloorRoi({ imageBase64, vehicle, plate?, slotHint })
             │      └─ chatJson(system,user, FloorRoiResultSchema.parse, imageBase64)
             │            → 기존 image_url(data:image/jpeg;base64) 멀티모달 경로 100% 재사용
             ├─ resolveFloorQuad(llm?.quad, vehicle)   // 검증·클램프·순서정규화·폴백 (결정형)
             └─ SqliteStore.upsertFloorRoi(runId, presetKey, clusterId, quad, now)

Finalizer.finalize(runId)
  ├─ store.getFloorRois(runId) → floorByRef: Map<`presetKey#clusterId`, quad>
  └─ assemble(): 채택 클러스터의 quad 있으면 slot.floorRoiByPreset = { [key]: quad }
```

### 3.1 프리셋별 프레임 보관 (`CaptureJob`)
- 기존엔 `lastFrame`(전체 1장)만 보관했다. floor 추론은 **프리셋별 최근 프레임**이 필요하므로 `lastFrameByPreset: Map<\`${camIdx}:${presetIdx}\`, Buffer>` 를 추가했다.
- `start()` 에서 `clear()`, `captureTarget()` 에서 프리셋 키로 `set`(기존 `lastFrame` 보존, 가산 1줄).
- floor quad 는 `roiByPreset` 과 **같은 프리셋 프레임·같은 정규화 좌표계**를 써야 뷰어 폴리곤이 일치한다.

### 3.2 LLM 비전 단계 (`AgentRuntime.recognizeFloorRoi`)
- `this.client` 없거나 `cfg.floorRoi?.enabled !== true` → `null`(호출 안 함).
- 전용 프롬프트(`floor_roi.system.md`/`floor_roi.user.md`) 로드 → `renderTemplate`(vehicle/plate JSON) → `chatJson(...)`. **신규 HTTP 코드 0** — 기존 멀티모달 chat 경로 재사용.
- `FloorRoiResultSchema` 는 `quad.length(4)` 만 강제. 0~1 클램프·순서 정규화는 reject 하지 않고 결정형 모듈에서 살린다(폴백 보장).

### 3.3 결정형 검증·폴백 (`capture/floorRoi.ts`, 순수·외부의존 0)
- `normalizeQuad(raw)`: 점≠4·NaN·undefined → `null`. 그 외 각 점 `clamp01` + 순서 강제(y 기준 앞/뒤, x 기준 좌/우 → `[앞왼,앞오,뒤오,뒤왼]`). **LLM 이 순서를 틀려도 폴리곤이 꼬이지 않는다.**
- `fallbackQuadFromRect(rect)`: bbox 하단 35% 밴드를 footprint 근사, 윗변 좌우 10% inset(원근 근사), 모든 점 클램프.
- `resolveFloorQuad(llmQuad, vehicle) = normalizeQuad(llmQuad) ?? fallbackQuadFromRect(vehicle)` — **floor ROI 가 비는 경로가 없다**(채택 슬롯은 항상 보유; 프레임 부재·상한 초과 슬롯만 이번 주기 skip, 다음 주기 보유).

## 4. 저장 — 신규 테이블 `floor_roi`

`aggregated_slot` 컬럼 확장이 아니라 **별 테이블**이다. 근거: `replaceAggregatedSlots` 가 매 체크포인트 run 단위 **delete+insert**(집계 멱등)라, 같은 행에 floor quad 를 두면 다음 집계가 덮어써 소실된다. floor quad 는 LLM 산출이라 집계와 **수명주기가 다르다**.

```sql
CREATE TABLE IF NOT EXISTS floor_roi (
  run_id INTEGER, preset_key TEXT, cluster_id INTEGER,
  x0 REAL, y0 REAL, x1 REAL, y1 REAL, x2 REAL, y2 REAL, x3 REAL, y3 REAL,
  updated_at TEXT,
  PRIMARY KEY (run_id, preset_key, cluster_id)
);
```

- `upsertFloorRoi(...)` = `INSERT … ON CONFLICT(run_id,preset_key,cluster_id) DO UPDATE`(중복 행 없이 갱신, 멱등).
- `getFloorRois(runId)` = 8실수 → 4점 배열 복원(순서 보존).
- `CREATE TABLE IF NOT EXISTS` 라 기존 DB 무해, 마이그레이션 불필요.

## 5. 설정 (`llm.config.json` / `llmConfig.ts`)

`LlmConfigSchema` 에 옵셔널 `floorRoi` 블록 추가:

```jsonc
"floorRoi": {
  "enabled": true,            // gemma 사용 중이므로 활성
  "maxPerCheckpoint": 12,     // 체크포인트 1회당 LLM 호출 상한(토큰·시간 비용 제한)
  "prompt": { "system": "config/prompts/floor_roi.system.md",
              "user":   "config/prompts/floor_roi.user.md" }
}
```

- 스키마는 `.optional()`(외부 config 리터럴 하위호환). 단 `DEFAULT_LLM_CONFIG.floorRoi`(enabled=false)가 항상 있고 merge 가 `{...DEFAULT.floorRoi, ...raw.floorRoi}` 라, **로드 결과의 `cfg.floorRoi` 는 항상 정의됨** → 런타임 `cfg.floorRoi.enabled` 접근 안전. (설계 §8 의 "미설정 시 undefined" 서술과 다른 점이며, 런타임 안전성 측면에서 더 견고. 02 노트 §4-1·03 리포트 §4 에 동의 기록.)
- `enabled:false` 또는 `llm.enabled:false` 면 floor 추론을 호출하지 않는다(결정형 경로 무변화).

- `core.js` `toPixelQuad(quad, imgW, imgH)`: 순수 변환 `quad.map(p => ({px:p.x*imgW, py:p.y*imgH}))`(`core.d.ts` 선언 추가).
- `app.js` `drawRoiOverlay()`: `slot.floorRoiByPreset?.[key]` + `roi-floor` 토글 시 `beginPath→moveTo/lineTo→closePath` 폐곡선 폴리곤, `strokeStyle='#39ff14'`(연두 — 차량 청록 `#00e5ff`·번호판 노랑 `#ffd60a` 와 구분).
- `index.html`: `roi-floor` 체크박스(기본 checked). `app.js` 가 `change` 리스너 등록.
- 분석 탭: `analyzeArtifact` 가 슬롯별 `hasFloor` + totals `withFloor` 산출 → '바닥 ROI' 카드 표시.

- **REST 계약 불변**: `/setup/*`·`/capture/*`·`/mapping` 시그니처·응답 shape 변화 없음. `floorRoiByPreset?` 는 `/mapping`(=`setup_artifact.json`) slot 에 **옵셔널로 추가**되는 가산 필드일 뿐이다.

### 미커버 (사실대로 — "검증됨"으로 표기하지 않음)
- **실 `gemma4:12b` 좌표 정확도**: vitest 는 두뇌/서버 모킹. 사변형 정밀도·접지면(지붕 아닌 바닥) 추정·순서 정확도는 **실 LLM 호출로만** 검증 가능. quad 가 차체 윤곽을 찍는 경향이면 프롬프트(접지면 강조) 튜닝 필요. → **마스터 라이브 수집 후 뷰어 폴리곤(#39ff14) 육안 확인 권장.**
- **라이브 서버 / `/capture` 실연동 · 실제 체크포인트 주기 floor 갱신**: 미수행(지침: 라이브 기동 금지). 별도 스모크 필요.
- **브라우저 DOM/canvas 실픽셀 렌더링**: jsdom/headless 미사용. 순수 변환(`toPixelQuad`)·문법·요소 존재까지만 커버.
- **confidence 임계 폴백**: 1차 미적용(유효 quad 면 채택). 후속 가산 가능.

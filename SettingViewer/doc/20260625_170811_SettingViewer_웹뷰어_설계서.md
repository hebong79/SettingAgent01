## 0. 목표와 범위

### 0.1 요구 기능 (사용자 요청)
1. **카메라별 프리셋** 목록 표시·선택·이동
2. **주차면 영역(ROI) 표시** — 차량 ROI + (있으면) 번호판 ROI를 영상 위 오버레이
3. **시뮬레이터 뷰어 가져오기** — 스트리밍, **초당 ≈3프레임**
4. **카메라(시뮬레이터) 제어** — 프리셋 이동 + 수동 PTZ(상/하/좌/우, zoom, step)

### 0.2 범위 확정 (결정 반영)
- **이번 라운드(1~10단계, §10.1)**: 시뮬레이터 뷰어(프리셋/ROI/스트림/제어) **+ 실 PTZ 어댑터(`RealPtzSource`, Hucoms CGI)**. RTSP→WebRTC 게이트웨이(11단계)는 **후속**.
- **실 PTZ 장비 존재**: Hucoms 카메라(`192.168.0.153`, 모델 `HNR-2036LA`). 로그인 `http://192.168.0.153/cgi-bin/login.cgi`, **아이디/비밀번호는 뷰어 UI에서 입력** → 프록시가 받아 카메라 인증(서버 영구저장 아님, 통과 방식). 단위테스트는 CGI 모킹, **실기기 통합확인은 장비 연결 후** 수행(§13.6).
- **비범위**: 셋업 *실행*(설치 파이프라인) UI — 이미 `POST /setup/run-from-map` 담당. 뷰어는 **결과 확인·검수·제어** 전용.
- **비범위**: VPD/LPD 실시간 검출 박스·번호판 OCR — 참고 이미지엔 있으나 제외(확장 훅만, §9). RTSP→WebRTC(11)도 이번 비범위.

### 0.3 성공 기준 (검증 가능 목표)
| # | 목표 | 검증 |
|---|------|------|
| G1 | `/mapping` 의 presets·slots 를 카메라별로 트리/목록 렌더 | 산출물 mock 으로 DOM 노드 수 = presets 수 |

## 1. 현행 시스템 분석 (설계 근거)

### 1.1 SettingAgent REST (`:13020`, Fastify, ESM/TS) — `src/api/server.ts`
| 메서드 | 경로 | 뷰어 활용 |
|--------|------|-----------|
| GET | `/health` | 카메라/VPD/두뇌 상태 배지 |
| GET | `/mapping` | **핵심**: `SetupArtifact`(presets/slots/globalIndex/warnings/report) 반환 |
| GET | `/setup/status` | (선택) 진행 상태 |
| POST | `/setup/run-from-map` 등 | 뷰어 범위 외(셋업 실행) |

### 1.2 카메라(시뮬레이터) REST (`:13100`) — `src/clients/CameraClient.ts`
| 메서드 | 경로 | 입력 | 출력 | 비고 |
|--------|------|------|------|------|
| GET | `/health` | — | 200 | 상태 |
| GET | `/cameras` (A타입) | — | `{cameras:[{camIdx,name,enabled,presets:[{presetIdx,label,pan,tilt,zoom}]}]}` | 카메라·프리셋·PTZ 목록 |
| POST | `/req_img` | `{cam_idx,preset_idx,pan?,tilt?,zoom?}` | `{cam_idx,preset_idx,pan,tilt,zoom,img_name,img_bytes(base64 JPEG)}` | **프리셋 적용 후** 캡처 |
| POST | `/req_move` | `{cam_idx,pan,tilt,zoom}` | `{success}` | PTZ 절대 이동 |

- **zoom 범위 1.0~36.0** (`tools.config.camera.zoomMin/Max`, 클램프 `CameraClient.clampZoom`).
- 인덱스 **1-based**(camIdx/presetIdx). slotId `c{cam}p{preset}s{pos}`, ROI key `${cam}:${preset}`.

### 1.3 산출물 타입 (오버레이의 원천) — `setup_artifact_구조.md`
```ts
SetupArtifact {
  presets: { camIdx, presetIdx, label, coveredSlotIds[], pan?, tilt?, zoom? }[]
  slots:   { slotId, zone, roiByPreset: Record<"cam:preset", NormalizedRect>,
             plateRoiByPreset?: Record<"cam:preset", NormalizedRect> }[]
  globalIndex: { globalIdx, slotId, camIdx, presetIdx }[]
}
NormalizedRect { x, y, w, h }  // 모두 0~1, 프리셋 이미지 좌표계
```

### 1.4 제약/리스크 (설계가 다뤄야 할 핵심)
- **CORS**: 브라우저가 `:13020`(SettingAgent)·`:13100`(Unity)를 직접 호출하면 교차출처. → §2 단일출처 프록시(SettingViewer :13030)로 해소.
- **스트림 vs PTZ 충돌**: `/req_img` 는 호출마다 `preset_idx`를 **재적용**한다. 사용자가 `/req_move` 로 수동 이동한 뒤 프리셋 기반으로 폴링하면 매 프레임 프리셋 위치로 **되돌아간다**. → §3 에서 "수동 모드는 preset 미전송 캡처"가 필요. 현재 `/req_img` 는 cam_idx/preset_idx 필수이므로 **Unity 측 보강(권장)** 또는 폴백 전략 필요(§3.3, §8 의존성).
- **`/req_img` 의 `pan/tilt/zoom` 우선순위 — 해석 A 확정(✅ 결정)**: `/req_img` 는 "특정 카메라를 **지정 PTZ로 맞춘 뒤** 그 화면을 캡처해 JPG 반환" — 즉 **preset 적용 후 pan/tilt/zoom 으로 덮어쓴다**. 따라서 **수동 라이브는 `/req_img {cam,preset,pan,tilt,zoom}`(마지막 PTZ 동봉) 연속 폴링만으로 구현**되며 프리셋으로 되돌아가지 않는다. **Unity `/snapshot|/stream` 보강 불필요**(저지연이 필요할 때만 선택적 옵션).
- **이미지 페이로드**: base64 JPEG. 폴링마다 base64 디코드 비용 → 프록시가 **순수 `image/jpeg` 바이너리**로 변환해 `<img>` 가 직접 소비하게 한다(브라우저 디코딩 활용).

## 2. 아키텍처 결정 (ADR)

### 2.1 채택안: **SettingViewer 가 SPA + 카메라 프록시를 서빙, ROI 는 SettingAgent /mapping 프록시 (단일 출처)**
```
[브라우저 SPA]  ── HTTP ──►  [SettingViewer :13030]
   (정적 자산)                 ├─ GET  /viewer/*               정적 SPA(@fastify/static)
                              ├─ GET  /viewer/api/mapping → 프록시 → SettingAgent GET /mapping
                              ├─ GET  /viewer/api/cameras → 프록시 → Unity GET /cameras
                              ├─ GET  /viewer/api/snapshot → 프록시 → Unity /req_img (→ image/jpeg)
                              └─ POST /viewer/api/move     → 프록시 → Unity POST /req_move
                                         │
                                         ├── (서버-서버) ──►  [SettingAgent :13020]  GET /mapping (산출물)
                                         └── (서버-서버) ──►  [Unity 시뮬레이터 :13100]
```
- **이유**: 브라우저는 **오직 `:13030`** 만 호출 → CORS 불필요. 카메라 주소(`tools.config.camera.baseUrl`)는 서버만 알면 됨(보안·이식성). 기존 `CameraClient` 를 그대로 재사용. ROI(/mapping)는 SettingViewer 가 SettingAgent(:13020) 의 GET /mapping 으로 서버측 프록시한다.
- **트레이드오프**: SettingViewer 에 프록시 라우트 + 정적 서빙(영향도 §7). 뷰어의 "조회/제어" 책임은 SettingViewer 로 분리되어 SettingAgent 는 셋업·/mapping 제공에 집중 → 라우트 prefix `/viewer` 로 격리.

### 2.2 대안 (기각)
- **B. 독립 SPA가 두 서버 직접 호출**: 양 서버에 CORS 설정 필요, 카메라 주소를 프런트에 노출, Unity가 `image/jpeg` 직접 못 주면 base64 처리까지 프런트로. → 복잡·이식성↓. 기각.
- **C. 별도 BFF(Node) 신설**: 프로세스 1개 추가. 1회성 도구에 과함(단순함 우선). 기각.

> **결정: 2.1 채택.** 단, 정적 자산·프록시는 `tools.config` 의 `viewer.enabled` 플래그로 on/off (셋업 전용 운영 시 끌 수 있게).

### 2.3 프런트엔드 스택
- **권장: 빌드리스 바닐라(ESM + Canvas)** — `index.html` + `app.js`(ES 모듈) + `app.css`. 번들러 없음.
  - 근거: 화면 1개, 의존성 최소, SettingAgent 가 정적 파일만 서빙하면 됨(단순함 우선). React/Vite 도입 시 빌드 체인·산출물 관리 부담.
  - ROI 오버레이는 `<img>` 위에 절대배치 `<canvas>` 로 그린다(이미지 리사이즈 시 재계산).
- 대안(미채택): React+Vite — 확장(검출 박스·다중 패널)이 커지면 재검토.

## 3. 스트리밍 설계 (≈3fps)

### 3.1 방식: **스냅샷 폴링** (기본)
- 클라이언트가 `GET /viewer/api/snapshot?...` 를 **~333ms 간격**으로 호출, 응답을 `<img>` 로 표시.
- 캐시 무효화: `?t=<seq>` 증가 파라미터(쿼리)로 매 요청 구분(브라우저 캐시·중복응답 방지).
- **백프레셔**: 이전 요청 미완료 시 다음 요청 스킵(겹침 방지). `AbortController` 로 정지 시 취소.
- fps 는 UI 입력으로 조절(기본 3 = 333ms; 참고 이미지의 "스냅샷 간격 200ms"처럼 100~1000ms 허용).

### 3.2 프록시 응답 형식
- 프록시가 Unity `/req_img` 의 base64(`img_bytes`)를 **디코드하여 `Content-Type: image/jpeg` 바이너리**로 그대로 반환. 응답 헤더에 현재 PTZ를 동봉: `X-PTZ-Pan/Tilt/Zoom`(UI의 "현재 PTZ" 갱신용).

### 3.3 표시 모드 (`/req_img` PTZ override 확정 반영)
`/req_img` 가 **preset 적용 후 PTZ로 덮어쓴다**(해석 A 확정). 따라서 시뮬레이터는 추가 엔드포인트 없이 두 모드를 모두 연속 폴링한다.

| 모드 | 캡처 방식 | 용도 |
|------|-----------|------|
| **프리셋 보기** | `/req_img {cam,preset}` (PTZ 미동봉 → 프리셋 위치) | 산출물 검수(해당 프리셋 ROI 정합) — 위치 고정이 정상 |
| **수동 제어** | `/req_img {cam,preset,pan,tilt,zoom}` (마지막 PTZ 동봉 → 그 위치 유지) | PTZ 버튼으로 자유 이동하며 ≈3fps 연속 라이브 |
| **정지** | 폴링 없음(마지막 프레임 유지) | — |

- 수동 모드 동작: PTZ 버튼/절대이동 시 클라이언트가 현재 PTZ 상태를 유지하고, 폴링 시 그 PTZ를 `/req_img` 에 동봉 → 프리셋으로 되돌아가지 않음. (Unity `/snapshot|/stream` 보강은 **불필요**, 저지연이 절실할 때만 선택적.)

**"현재-뷰 캡처" 경로 — 소스별:**

| 소스 | 현재-뷰 캡처 경로 | 가능 여부 |
|------|------------------|-----------|
| 시뮬레이터(Unity) | `/req_img {cam,preset,pan,tilt,zoom}` (PTZ override 확정) | ✅ 추가 엔드포인트 불필요 |
| 실 PTZ 카메라(§13) | ONVIF/벤더 snapshot URL 폴링 또는 RTSP→WebRTC | ✅ |
| (가정) 현재-뷰 경로 없는 소스 | "이동→1회 갱신" 폴백(G3-3) | ⚠️ 제한 — UI에 명시 |

## 4. 화면 설계 (레이아웃)

단일 페이지, 좌측 **뷰포트**(영상+오버레이) + 우측 **제어 패널**. 상단 탭 2개(참고 이미지 단순화).

```
┌───────────────────────────────────────────────────────────────┐
│ [제어·모니터링] [주차면 검수]            ● backend ● camera     │  ← 상태 배지(/health)
├──────────────────────────────┬────────────────────────────────┤
│                              │ 현재 PTZ  P… T… Z…              │
│   ┌────────────────────────┐ │ ───────────────────────────    │
│   │   <img> 영상           │ │ 카메라/프리셋 선택             │
│   │   + <canvas> ROI 오버레이│ │  [cam ▼] [preset ▼] [이동]     │
│   │                        │ │ ───────────────────────────    │
│   └────────────────────────┘ │ 수동 PTZ                       │
│   스트림: [시작][정지] fps[3] │   ▲   + zoom in                │
│   ROI 표시: ☑차량 ☑번호판    │ ◄  ►  − zoom out  step[500]    │
│                              │   ▼                            │
│                              │ 절대 이동 pan[ ]tilt[ ]zoom[ ] │
└──────────────────────────────┴────────────────────────────────┘
```

### 탭1: 제어·모니터링
- 좌: 라이브 영상(스트림 시작/정지, fps), 우: 카메라/프리셋 선택·수동 PTZ·절대 이동·현재 PTZ.

### 탭2: 주차면 검수 (ROI)
- 좌: 선택 프리셋의 캡처 + slot ROI 오버레이(차량/번호판 토글, slot 라벨=`globalIdx`/`zone`/`slotId`).
- 우: 현재 프리셋의 slot 목록(globalIndex 정렬), hover 시 해당 ROI 강조, 경고(`warnings`)·리포트(`report`) 표시.

## 5. 데이터·렌더링 로직

### 5.1 데이터 소스 결합
- 카메라/프리셋 **목록·PTZ**: `GET /viewer/api/cameras`(Unity `/cameras` 프록시). 없으면 폴백으로 `/mapping.presets` + `camerapos`(PTZ).
- **ROI/검수 데이터**: `GET /mapping` → `presets`, `slots`, `globalIndex`, `warnings`, `report`.
- 결합 키: `presetKey = ${camIdx}:${presetIdx}`.
  - 선택 프리셋의 ROI = `slots.filter(s => s.roiByPreset[presetKey])`.
  - 각 slot 라벨 = `globalIndex` 에서 `slotId` 매칭한 `globalIdx`.

### 5.2 ROI 정규화 → 픽셀 변환 (G2 핵심, 유닛테스트 대상)
```js
// rect: {x,y,w,h} 0~1 (프리셋 이미지 좌표계)
// imgW,imgH: 화면에 렌더된 영상의 실제 표시 픽셀 크기(naturalWidth 아님, 표시 크기)
function toPixel(rect, imgW, imgH) {
  return { px: rect.x*imgW, py: rect.y*imgH, pw: rect.w*imgW, ph: rect.h*imgH };
}
```
- 이미지 표시 크기가 바뀌면(반응형/리사이즈) canvas 를 영상과 **동일 크기**로 맞추고 재draw. `ResizeObserver` 사용.
- 차량 ROI/번호판 ROI 색 구분(예: 차량=시안 박스, 번호판=노랑 박스 + slot 라벨 텍스트).
- **letterbox 오차 방지 (필수)**: `<img>` 에 `object-fit: contain` 을 쓰면 컨테이너와 실제 이미지 사이 여백(letterbox)이 생겨 canvas가 컨테이너 전체를 덮으면 ROI가 어긋난다. → **초기 구현은 `<img>` 자체 크기에 canvas를 정확히 일치**시킨다. `.viewport{position:relative;display:inline-block}`, `img{display:block;max-width:100%}`, `canvas{position:absolute;left:0;top:0;pointer-events:none}` 로 두고, `toPixel` 의 `imgW/imgH = img.clientWidth/clientHeight` 사용. (object-fit/letterbox 보정은 후속 필요 시.)

### 5.3 스트림 루프(의사코드)
```js
let seq = 0, inflight = null, timer = null, lastUrl = null;
async function tick() {
  if (inflight) return;                 // 백프레셔: 겹침 방지(이전 요청 진행 중이면 스킵)
  const ac = new AbortController(); inflight = ac;
  try {
    // 쿼리명은 프런트 표준 cam/preset 고정(§6.2 명명규약). 서버가 cam_idx/preset_idx 로 변환.
    const url = `/viewer/api/snapshot?cam=${cam}&preset=${preset}&mode=${mode}&t=${seq++}`;
    const res = await fetch(url, { signal: ac.signal, cache: 'no-store' });
    const nextUrl = URL.createObjectURL(await res.blob());   // image/jpeg
    img.src = nextUrl;
    await img.decode();                 // 디코드 완료 후 교체(깜빡임·경합 방지)
    if (lastUrl) URL.revokeObjectURL(lastUrl);  // ★ 이전 Blob URL 해제(장시간 누수 방지)
    lastUrl = nextUrl;
    updatePtzFromHeaders(res.headers);  // X-PTZ-Pan/Tilt/Zoom
    drawRoiOverlay();
  } catch(e) { /* abort/timeout/decode 실패 무시 */ } finally { inflight = null; }
}
function start(fps){ timer = setInterval(tick, Math.round(1000/fps)); }  // fps=3 → 333ms
function stop(){ clearInterval(timer); timer = null; inflight?.abort(); inflight = null; }
```

### 5.4 PTZ 제어
- **절대 이동**: `POST /viewer/api/move {cam, pan, tilt, zoom}` → Unity `/req_move`. 성공 시 즉시 `tick()` 1회로 갱신.
- **스텝/방향**: 현재 PTZ ± step (pan/tilt), zoom in/out = zoom ± 1 단계(클램프 1~36). step 기본 500(참고 이미지). 내부적으로 절대 이동으로 환산.
- **프리셋 이동**: `[이동]` → `/viewer/api/snapshot?...&mode=preset` (프리셋 적용 캡처) 또는 `/req_move` 로 프리셋 PTZ 이동 후 스냅샷.

## 6. 서버측 추가 (SettingViewer) 명세

> SettingViewer 에 `/viewer` prefix 라우트 + 정적 서빙. SettingAgent 의 기존 라우트·셋업 로직 **불변**(외과적 변경).

### 6.1 정적 서빙
- `@fastify/static` 으로 `SettingViewer/web/` 디렉터리를 `GET /viewer/` 에 매핑(신규 의존성 1개).
- 대안(의존성 0): 단일 `GET /viewer` 핸들러가 `index.html` 문자열 반환 + 자산 인라인. → 자산 분리가 어려우면 `@fastify/static` 권장.

### 6.2 프록시 라우트
| 라우트 | 처리 | 비고 |
|--------|------|------|
| `GET /viewer/api/cameras` | `CameraClient` 가 Unity `/cameras` 호출(신규 메서드 `listCameras()`) → 그대로 반환 | A타입 미지원 서버면 `/mapping.presets`+camerapos 폴백 |
| `GET /viewer/api/snapshot` | `mode=preset`→`requestImage(cam,preset)`, `mode=manual`→ `/req_img {..PTZ}`(override 확정)·실카메라는 snapshot URL; base64→`image/jpeg` 바이너리 + `X-PTZ-*` 헤더 | 캐시 `no-store` |
| `POST /viewer/api/move` | body `{cam,pan,tilt,zoom}` zod 검증 → `source.move(...)`. **`viewer.allowMove=false` 면 `403`**(조회 전용) | zoom 클램프(기존) |
| `POST /viewer/api/camera/login` | body `{sourceId,user,pass}` → 실카메라 어댑터가 `login.cgi` 인증, 세션 메모리 보관(자격증명 미저장·미반환) | 실 PTZ 소스만. 응답=ok/실패 |
| `GET /viewer/api/health` | 기존 `/health` 재사용 또는 alias | 배지 |

- **명명 규약(혼선 방지, 필수)**: **프런트 URL = `cam`/`preset` 으로 통일** → 서버 내부 `camIdx`/`presetIdx` → 소스 호출 시 `cam_idx`/`preset_idx`(Unity) 로 변환. 변환은 프록시 핸들러 1곳에서만. 문서·코드 전체 이 규약 준수.
- **라우트 등록 순서(주의)**: `/viewer/api/*` (정확 경로)를 **먼저** 등록하고, 정적 서빙(`@fastify/static`, prefix `/viewer/`, 와일드카드)을 **나중**에 등록한다. `GET /viewer` → `/viewer/` redirect. SPA 단일 페이지이므로 별도 history fallback 불필요(`/viewer/`,`/viewer/app.js`,`/viewer/app.css` 만 처리).
- **소스 추상화(신규, 가산적)**: 프록시는 `CameraSource` 인터페이스(§13.2)를 호출 — `SimulatorSource`(기존 `CameraClient` 래핑: `listCameras()`/`snapshot()`/`move()`)와 실 PTZ `RealPtzSource` 가 동일 계약을 구현. 기존 `requestImage/move/health` 불변.
- 입력 검증: zod 스키마(camIdx/presetIdx 양의 정수, pan/tilt number, zoom 1~36).

### 6.3 설정 추가 (`SettingViewer/config/viewer.config.json`)
```jsonc
"viewer": {
  "enabled": true,        // 정적/프록시 라우트 노출 여부
  "allowMove": true,      // false 면 POST /viewer/api/move → 403 (LAN 조회 전용)
  "defaultFps": 3,        // 기본 스트림 fps
  "staticDir": "web",     // SPA 자산 폴더
  "controlToken": ""      // (선택) 설정 시 move 요청에 X-Viewer-Token 일치 요구. 빈 값=미사용
}
```
- `SettingViewer/src/config/viewerConfig.ts`(camera/viewer/cameraSources/settingAgentUrl/server.port) 타입·로더에 `viewer` 추가(기본값 제공 → 기존 config 호환). 기본값: `allowMove=true`, `controlToken=""`.
- **보안 근거(검증 §8)**: 서버가 `0.0.0.0` 바인딩이라 `POST /viewer/api/move` 가 LAN에 노출 → 같은 망의 누구나 카메라를 움직일 수 있음. 최소 보완으로 `allowMove` 플래그(조회 전용 모드), 강화 시 `controlToken` 헤더. 실 PTZ(§13)에서는 **카메라 자격증명을 프록시(서버)만 보유** → 브라우저에 절대 노출 금지.

### 10.2 테스트 전략 (vitest, 외부 서버 모킹)
- 서버: `fastify.inject()` 로 `/viewer/api/*` 응답·상태코드·헤더(`X-PTZ-*`, `image/jpeg`) 검증. CameraClient/소스는 `fetch` mock. **`allowMove=false` 시 move→403** 검증.

- 제어: `/viewer/api/move`(절대/스텝/zoom 클램프 1~36).
- **산출물·타 에이전트 계약 무영향**, SettingAgent 변경은 가산적·prefix 격리.
- 실 PTZ·RTSP(§13)는 **동일 SPA·프록시 계약** 위에 `CameraSource` 추상화로 수용 — 영상은 HTTP 스냅샷 폴링(기본)/WebRTC(옵션), 제어는 ONVIF/벤더 CGI, 자격증명은 서버만 보유.

## 13. 실 PTZ 카메라 · RTSP 지원 (확장 설계, 부록)

> 검증 지적 외 **사용자 요청으로 신설**. 참고 이미지의 기기 관리(예: `Hucoms PTZ(실물)`, `192.168.0.153:80`, 모델 `HNR-2036LA`, `PTZ 34778/327/6851`)처럼 실 카메라는 **영상=RTSP, 제어=ONVIF/벤더 HTTP** 로, Unity 의 base64-JPEG `/req_img` 모델과 다르다. 목표: **SPA·프록시 계약(`/viewer/api/*`)은 그대로 두고 백엔드 소스만 교체**.

### 13.1 핵심 제약 (객관적 사실)
- **브라우저는 RTSP를 직접 재생 못 함**(`<video>`/`<img>` 모두). 반드시 **게이트웨이로 변환** 필요.
- 실 PTZ 좌표계는 **벤더 원시 단위**일 수 있음(참고 이미지 `P 35192 T 563 Z 6851`, `PTZ 34778/327/6851`) — Unity 의 도(°)+zoom 1~36 과 **단위·범위가 다름** → 소스별 **단위 매핑(정규화)** 필수.
- 실 카메라는 **자격증명(계정/비밀번호)** 필요 → **뷰어 UI에서 입력**, 프록시가 받아 카메라 인증에 사용(서버 영구저장 아님, 세션 메모리 통과). 응답·로그·프런트 GET URL 에 평문 노출 금지. (참고: `http://192.168.0.153/cgi-bin/login.cgi`)

### 13.2 소스 추상화 `CameraSource` (시뮬레이터/실카메라 공통)
프록시는 아래 인터페이스만 호출 → SPA·라우트 **무변경**, 소스 구현만 교체.
```ts
interface CameraSource {
  readonly kind: 'sim' | 'onvif' | 'hucoms';
  listCameras(): Promise<CameraList>;                 // /viewer/api/cameras
  snapshot(cam: number, opt: { presetIdx?: number; ptz?: Ptz; mode: 'preset'|'manual' })
    : Promise<{ jpeg: Buffer; ptz: Ptz }>;            // /viewer/api/snapshot (3fps 폴링 공통 경로)
  move(cam: number, ptz: Ptz): Promise<boolean>;      // /viewer/api/move
  streamUrl?(cam: number): string | null;             // (선택) 저지연 WebRTC/MJPEG URL
  toNativePtz(viewerPtz: Ptz): unknown;               // 뷰어 단위 → 소스 원시 단위
  fromNativePtz(native: unknown): Ptz;                // 소스 원시 단위 → 뷰어 단위
}
```
- `SimulatorSource`(kind=`sim`): 기존 `CameraClient` 래핑. snapshot=`/req_img`, move=`/req_move`, list=`/cameras`. 단위 변환 항등.
- `RealPtzSource`(kind=`onvif`|`hucoms`): 아래 §13.3~13.4.

### 13.3 영상(RTSP) → 브라우저 변환 — 방식 비교 (객관)
| 방식 | 지연 | 구현 복잡도 | 브라우저 표시 | 적합도 |
|------|------|------------|--------------|--------|
| **HTTP 스냅샷 폴링(3fps)** | 중(~333ms+망) | 낮음(기존 경로 재사용) | `<img>` | ✅ **기본 채택** — 시뮬레이터와 동일 단일 경로 |
| RTSP→MJPEG (ffmpeg multipart) | 중 | 중(ffmpeg 트랜스코드) | `<img>` multipart | 스냅샷 URL 없는 카메라 |
| RTSP→WebRTC (go2rtc/MediaMTX) | **낮음(~0.2~1s)** | 높음(사이드카 프로세스) | `<video>` | 저지연 라이브 필요 시 **옵션 업그레이드** |
| RTSP→HLS | 높음(수 초) | 중 | hls.js | 라이브 PTZ 제어엔 부적합 |

- **권장(기본)**: ONVIF `Media.GetSnapshotUri` 또는 벤더 `…/snapshot.jpg` 의 **HTTP JPEG 스냅샷**을 프록시가 ≈3fps 폴링 → 기존 `/viewer/api/snapshot` 그대로. 대부분의 PTZ 카메라가 스냅샷 URL 제공.
- **저지연 옵션**: `go2rtc`/`MediaMTX`(rtsp-simple-server) 사이드카가 RTSP 흡입 → WebRTC 노출. `streamUrl()` 가 그 재생 URL(또는 프록시 시그널링 경로)을 반환, SPA 는 `<video>` 로 표시. 스냅샷이 기본, WebRTC는 토글.

### 13.4 제어(PTZ) — Hucoms CGI (이번 대상) / ONVIF (확장)
- **Hucoms CGI(이번 구현 대상)**: `cgi-bin/login.cgi` 로 인증(세션 쿠키/토큰 획득) → PTZ CGI 로 절대 이동, 스냅샷 CGI(또는 RTSP)로 영상. 원시 정수 단위(`P 35192 T 563 Z 6851` 류) → §13.2 `toNativePtz`/`fromNativePtz` 로 뷰어 단위와 매핑. **정확한 CGI 경로/파라미터는 실기기에서 확인**(어댑터는 경로를 설정/상수로 분리해 교체 용이하게 작성).
- **ONVIF(확장)**: `AbsoluteMove`/`ContinuousMove`+`Stop`/`GotoPreset`. 절대 좌표 정규화(pan·tilt −1~1, zoom 0~1) 매핑. 동일 `CameraSource` 계약으로 후속 추가 가능.
- `move()` 계약·`/viewer/api/move` 라우트·`allowMove`/`controlToken` 보안은 **공통 재사용**. 인증 세션은 어댑터 내부에서 관리(만료 시 재로그인).

### 13.5 다중 카메라 설정 (`SettingViewer/src/config/viewerConfig.ts` · `SettingViewer/config/viewer.config.json` 확장)
현재 `camera.baseUrl`(단일 시뮬레이터)에서, 실 배치는 카메라마다 접속정보가 다르므로 **소스 목록**으로 확장한다(기존 단일 설정과 **하위호환**: `camera` 만 있으면 sim 소스 1개로 취급).
```jsonc
"cameraSources": [
  { "id": "sim",  "kind": "sim",    "baseUrl": "http://localhost:13100" },
  { "id": "cam1", "kind": "hucoms", "host": "192.168.0.153", "port": 80,
    "loginPath": "/cgi-bin/login.cgi",
    "rtspUrl": "rtsp://192.168.0.153:554/stream1",   // 또는 snapshotUrl
    "snapshotUrl": "http://192.168.0.153/cgi-bin/snapshot.cgi",
    "ptz": { "panRange": [0, 36000], "tiltRange": [0, 9000], "zoomRange": [1, 36] } }
  // 주의: 위 경로/범위는 실기기 확인 후 보정. 자격증명은 config에 두지 않음(아래).
]
```
- **자격증명은 config·env에 두지 않고 UI에서 입력** → 프록시가 세션 동안만 보유(`POST /viewer/api/camera/login {sourceId,user,pass}` → 어댑터가 login.cgi 인증, 세션 쿠키/토큰 보관). 이후 snapshot/move 는 그 세션 사용.
- `panRange/tiltRange/zoomRange` 로 소스별 단위 매핑(정규화 ↔ 원시) 정의. 경로/범위는 **실기기 확인 후 보정**(상수 분리).

### 13.6 실 PTZ(Hucoms) 검증 체크리스트
**단위테스트(이번, CGI 모킹):**
- [ ] `login.cgi` 인증 → 세션 보관, 만료 시 재로그인. 자격증명이 응답·로그·프런트 GET URL 에 노출되지 않음
- [ ] `snapshot()` 가 인증 세션으로 JPEG 취득 → `image/jpeg` 반환
- [ ] `move()` 절대 이동이 CGI 로 전달, 뷰어 단위↔원시 단위 매핑 왕복 일치(`toNativePtz`/`fromNativePtz`)
- [ ] `listCameras()` 가 소스 설정→`CameraList` 매핑(프리셋 없으면 단일 라이브 뷰)
- [ ] zoom/pan/tilt 범위 클램프가 소스 범위 기준 동작

**실기기 통합확인(장비 연결 시, 192.168.0.153):**
- [ ] 실제 login.cgi 경로/응답·snapshot·PTZ CGI 파라미터를 실측 후 어댑터 상수 보정
- [ ] UI 로그인 → 라이브 스냅샷 표시 → 방향/절대 PTZ 이동 반영 확인

### 13.7 영향도 (실 PTZ 분)
- 시뮬레이터에는 **코드 영향 0** — `SimulatorSource` 는 기존 `CameraClient` 래퍼. `cameraSources` 미설정 시 기존 `camera` 로 동작(하위호환).
- `RealPtzSource`(Hucoms CGI)·로그인 라우트는 **이번 라운드 신규**(가산적). 기존 셋업 파이프라인·산출물 계약·SPA 공통 코드 **불변**.
- RTSP→WebRTC 게이트웨이는 **후속(11단계)** — 별도 사이드카(go2rtc/MediaMTX), 기본 스냅샷 경로와 독립.

# 최종 셋업파일 저장 흐름 검증 + SettingViewer "분석" 탭

- 작성일: 2026-06-26
- 대상: 최종 셋업 산출물(`setup_artifact.json`)의 **저장 흐름 검증** + 저장된 파일을 분석하는 **SettingViewer 분석 탭**
- 한 줄 요약: 산출물은 `Repository.saveArtifact()`가 `SettingAgent/data/setup_artifact.json`에 저장하며, SettingViewer는 `/viewer/api/mapping` 프록시로 이를 받아 **분석 탭**에서 요약·테이블·리포트로 시각화한다.

---

## 1. 최종 셋업파일 저장 흐름 (검증 결과)

### 1.1 무엇이 어디에 저장되나
- **파일**: `SettingAgent/data/setup_artifact.json` (`store.dataDir` 기준)
- **포맷**: `SetupArtifact` = `{ presets, slots, globalIndex, createdAt, warnings?, report? }`, **pretty JSON(2-space)**, UTF-8
- **저장 코드**: `Repository.saveArtifact(artifact)` → `writeFileSync(... JSON.stringify(artifact, null, 2))` (디렉터리 자동 생성)

### 1.2 저장을 트리거하는 두 경로
| 경로 | 트리거 | 비고 |
|------|--------|------|
| **단발 셋업** | `SetupOrchestrator.run()` (`POST /setup/run`, `/setup/run-from-map`) | 기존 셋업 |
| **정밀 수집** | `Finalizer.finalize(runId)` (`POST /capture/finalize`) | 집계+LLM 최종화. **추가로 SQLite `artifact_snapshot`에 스냅샷 기록** |

두 경로 모두 같은 `Repository.saveArtifact()` → 같은 파일에 저장. 이후 `GET /mapping`이 `repo.loadArtifact()`로 그대로 반환하고, SettingViewer가 `GET /viewer/api/mapping`으로 프록시한다.

### 1.3 라이브 저장 테스트 (실서버, 2026-06-26)
원본 백업 → `POST /capture/finalize {runId:2}` → 원복으로 보존하며 검증:

| 항목 | 저장 전 | 저장 후 |
|------|---------|---------|
| 파일 `createdAt` | `2026-06-25T07:43:15Z` | `2026-06-26T08:27:04Z` (**갱신=저장됨**) |
| 파일 mtime | — | 갱신됨 |
| SQLite `artifact_snapshot` 수 | 2 | **3** (+1 스냅샷) |
| finalize 응답 | — | `{ ok:true, slots:9, globalCount:9 }` |

→ `/capture/finalize` 가 **파일 저장 + 스냅샷 기록**을 수행함을 확인. (검증 후 원본 산출물 복원, `/mapping` 원래 값 회귀 확인.)
> 참고: run2(4라운드)는 9면, run1(2라운드)은 1면 — `clusterMinSupport=3` 임계로 **라운드가 많을수록 더 많은 면 채택**(정밀 수집 효과)도 함께 확인됨.

## 2. SettingViewer "분석" 탭

### 2.1 목적
최종 저장된 `setup_artifact.json`을 사람이 한눈에 검수: 요약 통계 · 프리셋별 · 주차면 테이블(전역 인덱스 순) · 경고 · LLM 리포트 · 원본 JSON.

### 2.2 구성
- **탭**: 상단 네비에 `분석` 추가(`data-tab="analyze"`). 활성 시 라이브 뷰(viewport+panel) 숨기고 **전체폭 분석 뷰** 표시.
- **데이터 소스**: `GET /viewer/api/mapping` (= SettingAgent `/mapping` 프록시). **신규 백엔드 엔드포인트 없음** — 기존 산출물 재사용.
- **순수 로직**: `analyzeArtifact(artifact)` (web/core.js) — 산출물을 요약/평탄화. DOM 비의존 → 단위테스트.
  - `totals`: 카메라/프리셋/주차면/전역인덱스/번호판ROI/존/경고 수
  - `perPreset`: 프리셋별 주차면 수
  - `slots`: globalIdx 오름차순, `{ globalIdx, slotId, zone, presetKey, roi, hasPlate }`
  - `warnings`, `report`, `createdAt`
- **UI 동작**: 새로고침 / JSON 내려받기 / 원본 JSON 토글.

### 2.3 실데이터 검증 (현재 저장 파일)
`analyzeArtifact(/viewer/api/mapping)` 결과: 카메라 2 · 프리셋 5 · 주차면 18 · 전역 18 · 번호판 11 · 존 17 · 경고 10 · 리포트 477자, slots globalIdx 정렬 1·2·3… 확인.

### 2.4 변경 파일 (전부 가산적, 프런트 한정)
| 파일 | 변경 |
|------|------|
| `web/core.js` / `core.d.ts` | `analyzeArtifact()` + 타입 추가 |

## 3. 영향도
- **백엔드 무변경**: `/mapping`·`SetupArtifact` 계약·SettingAgent 코드 불변. 분석 탭은 기존 프록시만 소비.
- **가산적·프런트 한정**: 기존 탭/스트림/제어/정밀수집 동작 불변(탭 전환 로직만 `setTab`으로 일원화).
- **테스트**: SettingViewer vitest **93 통과**(기존 88 + analyzeArtifact 5), typecheck 통과, 회귀 0.
- **미검증**: 브라우저 DOM 렌더(테이블/카드 시각)는 수동 확인 권장(jsdom 미도입). 순수 로직·프록시·저장 흐름은 검증 완료.

## 4. 사용법
1. SettingAgent(:13020)·SettingViewer(:13030) 기동 → `http://localhost:13030/viewer/`
2. 상단 **분석** 탭 클릭 → 최종 `setup_artifact.json` 요약·테이블·리포트 표시
3. 셋업/정밀수집 최종화 후 **새로고침**으로 갱신 결과 확인, 필요 시 **JSON 내려받기**

## 1. 배경 / 목적

### 1.1 분리 전 상태
웹 뷰어(카메라 영상·PTZ 제어·ROI 검수 SPA)는 **SettingAgent 내부에 구현**되어 있었다(`SettingAgent/src/viewer/*`, `SettingAgent/web/*`). SettingAgent 는 동시에 두 역할을 겸했다.
1. 셋업 파이프라인(카메라 디스커버리·프리셋·ROI 매핑 → `setup_artifact.json` 생성)
2. 웹 뷰어 서빙(정적 SPA + 카메라 프록시 + ROI 표시)

### 1.2 분리한 이유
- **독립 서비스화**: 뷰어는 셋업 산출물의 사람 검수·실시간 영상 확인 도구로, 셋업 파이프라인과 생명주기·배포 단위가 다르다. 한 프로세스에 묶으면 뷰어 변경이 셋업 서비스 재기동을 유발한다.
- **관심사 분리**: 셋업(SettingAgent)은 결정형 도구 + LLM 두뇌(`/brain/*`) 영역, 뷰어(SettingViewer)는 순수 결정형(스냅샷 폴링·좌표변환·PTZ 절대이동·REST 프록시) 영역이다. 분리해도 도구/두뇌 경계는 변하지 않는다.
- **단일 출처 + CORS 회피 유지**: 브라우저는 SettingViewer 한 곳만 호출하고, 카메라(:13100)와 SettingAgent(:13020 /mapping)는 서버측 프록시로 중계한다. 이 단일 출처 모델을 분리 후에도 그대로 보존한다.

### 1.3 범위 한정
기능 추가/변경 없음. **파일을 SettingViewer 로 옮기고(이동/복제), 양쪽 import·config·배선을 외과적으로 보정**하는 것이 전부다. 유일한 신규 기능 `/mapping` 프록시는 "SettingAgent 가 더 이상 SPA 를 서빙하지 않으므로 브라우저가 ROI 를 받으려면 SettingViewer 가 중계해야 한다"는 분리의 직접 결과다.

## 2. Before / After 아키텍처

### Before (분리 전)
```
[브라우저 SPA] ── HTTP ──► [SettingAgent :13020]
                            ├─ GET /viewer/*        정적 SPA(@fastify/static)
                            ├─ GET /viewer/api/*    카메라 프록시(snapshot/move/cameras/...)
                            ├─ GET /mapping         셋업 산출물(ROI)
                            └─ POST /setup/*, /brain/*  셋업·두뇌
                                       └─ (서버-서버) ──► [Unity 카메라 :13100]
```

### After (분리 후)
```
[브라우저 SPA] ── HTTP ──► [SettingViewer :13030]   ◄── 단일 출처(브라우저는 여기만 호출)
                            ├─ GET /viewer/*               정적 SPA(@fastify/static)
                            ├─ GET /viewer/api/*           카메라 프록시(snapshot/move/cameras/login/health)
                            └─ GET /viewer/api/mapping     ROI 프록시 ─┐
                                       │                              │
                                       │ (서버-서버)                  │ (서버-서버)
                                       ▼                              ▼
                            [Unity 카메라 :13100]          [SettingAgent :13020] GET /mapping
                                                            └─ POST /setup/*, /brain/*  (셋업·두뇌만)
```

핵심 변화:
- 브라우저 → **SettingViewer(:13030) 단일 출처**(분리 전 SettingAgent:13020 가 담당하던 SPA·카메라 프록시를 인수).
- ROI(`/mapping`)는 SettingAgent 가 계속 생성·보유하되, 브라우저는 **SettingViewer 의 `/viewer/api/mapping` 프록시**를 통해 받는다.
- SettingAgent 는 **셋업·두뇌 + `/mapping` 데이터 제공**만 남긴다(SPA·카메라 프록시 제거).

## 4. 독립성 확보 방식

### 4.1 SettingAgent/@parkagent 소스 의존 0건
SettingViewer 는 SettingAgent 소스나 `@parkagent/types` 패키지를 import 하지 않는다(사용자 지시: 독립성 확보). 검증자 grep 결과 `SettingViewer/src` 의 런타임 import 0건(매칭은 전부 주석/문자열).

- **CameraClient 복제**: 셋업이 원본을 계속 쓰므로 SettingViewer 로 복제. 복제본은 `requestImage`/`move`/`health`/`clampZoom`/`listCameras` 5개 메서드 보유.
- **http util 발췌 복제**: `fetchWithTimeout` 만 복제(`isRetryable`/`withRetry`/`RetryOptions` 는 SettingViewer 미사용 → 제외, 단순함 우선).
- **CapturedImage 로컬화**: `@parkagent/types` 의존을 피하기 위해 SettingViewer 의 `CameraClient.ts` 내부에 동일 8필드(camIdx/presetIdx/pan/tilt/zoom/imgName/jpg) 인터페이스를 **로컬 정의 + export**. `simulatorSource.test.ts` 도 여기서 import.
- **viewerConfig 발췌**: SettingAgent `toolsConfig.ts` 전체가 아니라 SettingViewer 가 쓰는 섹션(camera/viewer/cameraSources/server.port)만 발췌 + 신규 `settingAgentUrl`.

### 4.2 중복 코드 관리 주의
복제의 대가로 **CameraClient·http util 이 2벌(SettingAgent 원본 + SettingViewer 사본)** 존재한다. 한쪽 버그 수정이 다른 쪽에 자동 반영되지 않는다. 동기화 가이드는 영향도분석 문서(B)와 §아래 운영 항목 참조. 향후 공유가 필요하면 `@parkagent/types` 류 패키지 승격 검토(이번 범위 밖).

## 5. `/mapping` 프록시 라우트 명세

분리로 인한 유일한 신규 기능. 브라우저 ROI(`loadMapping`)가 호출하던 SettingAgent `/mapping` 을 SettingViewer 가 중계한다.

### 5.1 라우트: `GET /viewer/api/mapping` (`src/server.ts`)
- 다른 뷰어 API 와 prefix(`/viewer/api`) 통일. `registerViewerRoutes`(정적 와일드카드 `/viewer/`)보다 **먼저 등록**(라우트 우선순위).
- `fetchWithTimeout(`${settingAgentUrl}/mapping`, {GET}, 5000ms)` 호출. `settingAgentUrl` 은 말미 슬래시 정규화(`replace(/\/+$/,'')`) 후 사용.

| 상황 | 응답 |
|------|------|
| upstream 200 | **200** + `Content-Type: application/json` + `res.text()` 패스스루(SetupArtifact shape 무관) |
| upstream 404 | **404** `{error:'no setup artifact'}` |
| upstream 그 외 비 2xx | **502** `{error:'mapping upstream HTTP <status>'}` |
| 연결 불가/타임아웃(5000ms abort) | **502** `{error:'mapping upstream unreachable'}` |

본문을 파싱하지 않고 text 패스스루하므로 SettingViewer 는 SetupArtifact 형식을 알 필요가 없다(결합 최소).

### 5.2 app.js 변경점 (프런트, 1줄)
```diff
  async function loadMapping() {
    try {
-     const res = await fetch('/mapping');
+     const res = await fetch(api('/mapping'));   // api('/mapping') = '/viewer/api/mapping'
```
그 외 `app.js`·`core.js`·`app.css`·`index.html` 무변경.

## 6. 입출력 / REST 계약 (분리 후 보존 확인)

뷰어 API 계약은 이동 후에도 동일하다(검증자 `viewerRoutes.test.ts` 14케이스 그린).

- `GET /viewer/api/cameras` → 200 CameraList JSON, **camIdx 1-based**.
- `GET /viewer/api/snapshot`(preset/manual) → `Content-Type: image/jpeg` + `X-PTZ-pan/tilt/zoom` 헤더 + `cache-control: no-store`. zoom 클램프(예 99→36).
- `POST /viewer/api/move` → `{ok:true}`. `allowMove=false` 또는 controlToken 불일치 → **403**(소스 미호출).
- `POST /viewer/api/camera/login`: sim → 400(unsupported), hucoms → 200 / **자격증명 응답 미노출**.
- `GET /viewer/api/health` → `{status:ok, sources:[...]}`.
- **CameraClient ↔ Unity REST 경계**: 바디·응답은 snake_case(`cam_idx`/`preset_idx`/`img_name`/`img_bytes`(base64→Buffer)). camelCase↔snake_case 변환은 무수정 복제로 동일.

- 런타임: Node.js / TypeScript (독립 프로세스). **LLM 두뇌(model-agnostic) + MCP 도구** 구조 (아키텍처 §8)

---

## 1. 목적과 역할

시스템의 **두뇌(컨트롤 타워)**. ActionAgent를 제어하고, 입·출차 이벤트를 **최종 판단·확정**하며,
현재 상황과 이전 상황(DB)을 비교하여 ActionAgent에 **행동지침을 하달**하고, 결과를 **DB에 저장**한다.
어떤 카메라·프리셋·주차면을 확인할지 **수시로 점검·지시**한다.

### 1-1. 두뇌(LLM)/도구 경계 (아키텍처 §8)
- **LLM 두뇌(MCP로 도구 호출)**: 입출차 최종 판단, 번호 불일치·유령차 등 이상 상황 해석,
  점검 우선순위 결정 등 **맥락·예외 판단**. DM은 3개 에이전트 중 LLM 활용도가 가장 높다.
- **결정형 MCP 도구**: `db.*`(상태/이벤트 CRUD), 상태머신 디바운스, 스케줄 발행, `action.task`(Action 지시).
- 모델-불문(Claude/Codex/Gemma/Qwen3) — 능력은 모두 MCP 도구로 노출.

## 3. 처리 흐름 (결정형 오케스트레이션)

### 3.1 관측 수신 → 이벤트 확정 (할일 22~24)
```
[Action 관측 수신: 점유 변화 / 차량번호]
  └ 슬롯 상태머신 갱신 + 디바운스(연속 N회 동일 관측 시 확정)
  └ 확정 전이:
        EMPTY→OCCUPIED = ENTER(입차)  → 통상 Action이 자체 트리거로 번호까지 보고(센터라이징은 Action 소관).
                                        번호 미확보(GAVE_UP)·불일치 시에만 Action에 '정밀조사' Task 지시
        OCCUPIED→EMPTY = EXIT(출차)   → 출차 이벤트 확정
  └ DB 조회(이전 상황) ↔ 현재 비교 → 정합성 판단(번호 불일치/유령차량 등)
  └ ParkingEvent 저장 + SlotState 갱신
```

### 3.2 점검 스케줄링 (할일 25)
```
[주기/우선순위 스케줄러]
  └ 정책에 따라 점검 대상 선정(오래 미관측 슬롯, UNKNOWN, 변화 의심 등)
  └ ActionTask 발행 → Action /task 호출(또는 큐 enqueue)
```

상태(슬롯별): `UNKNOWN → EMPTY ⇄ OCCUPIED` (디바운스 카운터 보유).

## 5. REST API (DMAgent 제공)

| 메서드 | 경로 | 설명 |
|--------|------|------|
| `GET` | `/health` | 상태 |
| `POST` | `/observations` `{camIdx, presetIdx, slotId, occupancy?, plate?, conf, at}` | Action 관측/이벤트 수신 |
| `GET` | `/state` / `/state/:slotId` | 현재 슬롯 상태/차량번호 조회 |
| `GET` | `/events?from&to&type` | 입출차 이벤트 이력 조회 |
| `POST` | `/inspection/trigger` `{scope}` | 수동 점검 지시 |
| `POST` | `/glpr/event` (2차) | 입출구 정적카메라 입출차 수신(예약) |

## 6. 데이터베이스 스키마 (초안)

> DB: **개발 SQLite / 운영 PostgreSQL** 확정. 추상화 계층(Repository)로 양쪽 호환 유지(쿼리 빌더 or 경량 ORM).

```
slot_state(slot_id PK, global_idx, occupancy, confidence, plate, plate_conf,
           last_changed_at, last_observed_at, pending_value, pending_count)
parking_event(event_id PK, slot_id, type ENTER|EXIT, plate, at, confidence, source_image)
vehicle_observation(obs_id PK, cam_idx, preset_idx, slot_id, plate, plate_conf,
                    occupancy, captured_at, source_image)
inspection_task(task_id PK, type, cam_idx, preset_idx, slot_id, reason, priority,
                status, created_at, dispatched_at, result)
```
- 셋업 산출물(Preset/ParkingSlot/GlobalSlotIndex)은 SettingAgent가 기록한 공유 저장소를 읽음.

## 7. 확정/잔여 사항 (DMAgent 관련)

- **DB 종류**: **개발 SQLite / 운영 PostgreSQL** 확정.
- **입차 후속 트리거**: **Action 자체 트리거**로 확정 → DM은 센터라이징을 직접 지시하지 않고,
  번호 미확보/불일치 시에만 '정밀조사' Task 발행(ActionAgent §3.3).

- **DB 연계**: **신규 독립 DB로 구축** 확정(기존 `PEventAPIWeb`/`ParkingViewer` DB와 연동 안 함).
  필요 시 후속으로 이벤트 export/연동 어댑터를 별도 추가.
- **이벤트 확정 수치**: 디바운스 **연속 2~3회**(기본값 그대로 사용 확정), 미관측 타임아웃·신뢰도 임계는 운영 튜닝.

### 잔여 확인 항목 (기본값으로 진행)
1. **점검 스케줄 정책**(할일 25) — 라운드로빈 주기 / UNKNOWN 우선 / 이벤트 의심 우선 등 우선순위.
2. **차량번호 정합성 판단**(할일 24) — 같은 슬롯 번호 불일치/옆면 이동 처리 규칙.
3. **통신 방식** — Action 지시를 REST 동기로 할지, 작업 큐(BullMQ/Redis)로 비동기화할지(기본: REST).

## 8. 영향도 분석

- DMAgent는 **단일 진실 공급원(SSOT)**. SlotState/Event 스키마 변경은 Action 보고 계약과
  조회 소비자(웹/뷰어)에 전파 → 공통 패키지 + DB 마이그레이션 관리 필요.
- Action을 지시·제어하므로 DM 장애 시 시스템 정지 → 헬스체크/재시작/백프레셔 정책 필요.
- GLPR(할일 26)은 2차이므로 현재는 인터페이스만 두고 미구현(스키마/엔드포인트 예약).
- 기존 `PEventAPIWeb`/`ParkingViewer`와 이벤트 포맷을 맞추면 재사용 가능(연계 여부는 §7-1).

## 9. 테스트 계획

- 유닛: `EventEngine`(상태전이/디바운스), `Reconciler`(번호 불일치 판단),
  `InspectionScheduler`(우선순위 선정) — DB 인메모리/모킹.
- 통합(스모크): 관측 스트림 주입 → ENTER/EXIT 확정 시퀀스 검증 + DB 영속화 확인.
- 성공 기준: 노이즈 관측(1~2회 깜빡임)은 이벤트로 확정되지 않고, 안정 전이만 정확히 1건 기록.

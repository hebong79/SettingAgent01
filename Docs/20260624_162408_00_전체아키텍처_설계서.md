# ParkAgent 전체 아키텍처 설계서

- 작성일: 2026-06-24
- 대상: SettingAgent · ActionAgent · DataManagerAgent(DMAgent) 3인 에이전트 시스템
- 근거: `Docs/설계_데이터모음.txt` (목적/할일 26항목) + 사용자 설계 결정 4건

---

## 1. 목적

카메라(실제 PTZ + Unity 시뮬레이터)를 제어하여 차량을 찾고, 주차면(슬롯)과 매핑하며,
번호판을 인식하여 **"어느 주차면에 어떤 차량번호의 차량이 있는가"** 를 알아내고
주차장의 **입·출차를 관리**한다. (주차면 관리 포함)

---

## 2. 확정된 설계 결정 (사용자 승인)

| # | 결정 항목 | 선택 | 의미 |
|---|-----------|------|------|
| 1 | 구현 언어/런타임 | **Node.js / TypeScript 통일** | 3개 에이전트 모두 Node/TS. Python(VLM·VLA·LPR) 모델은 REST로만 호출 |
| 2 | 기존 자산 관계 | **REST 서비스로 재사용** | Unity 카메라·da_vpd_api·da_lpd_api·LPR(:8124)·VLA_Center 를 별도 서비스로 두고 HTTP 호출. ParkSimMgr 도메인/클라이언트는 패턴 재사용 |
| 3 | 에이전트 지능 모델 | **하이브리드: LLM 두뇌 + MCP 도구** (2026-06-24 재확정, 기존 #3 대체) | 고수준 판단은 LLM(Claude/Codex/Gemma/Qwen3)이 **MCP**로 도구 호출, 실시간 제어 루프는 결정형 도구 |
| 4 | 통신·배포 구조 | **독립 프로세스 + REST/큐 + MCP** | Setting/Action/DM 각각 별도 서비스. 인식·제어 능력은 MCP 서버(tool)로 노출, 에이전트 간은 REST/큐 |

> **"AgentVLA"의 'Agent' = LLM 두뇌(model-agnostic) + MCP 도구**.
> 카메라·VPD·LPD·LPR·VLA·DB는 **MCP 도구 서버**로 래핑되어 어떤 LLM이든 동일하게 호출한다.
> 단, **줌 반복·PTZ 미세이동 같은 실시간 루프는 LLM이 아닌 결정형 도구**로 두어 지연·비용·비결정성을 막는다.
> (자세한 두뇌/도구 경계는 §8)

```

                         │        DataManagerAgent (DMAgent)        │

                         │              ActionAgent                  │
                         │  차량 탐지 · 센터라이징 · LPD · LPR        │

                   req_img/  │          │ detect   │centering│ LPD/LPR
                   req_move  │          │          │         │
        ┌──────────────────▼──┐  ┌─────▼─────┐ ┌──▼──────┐ ┌─▼────────────┐
        │ Unity Cam / 실 PTZ   │  │ VPD       │ │VLA      │ │LPD → LPR     │
        │ (CWebCamCtrlServer)  │  │da_vpd_api │ │/centering│ │da_lpd_api,   │
        │ req_img / req_move   │  │/det/img.. │ │         │ │:8124 plate   │
        └──────────────────────┘  └───────────┘ └─────────┘ └──────────────┘
                                 ▲
                                 │ 초기 셋업 시 동일 카메라/검지 사용
                         ┌───────┴──────────────────────────────────┐
                         │              SettingAgent                 │
                         │  프리셋 셋업 · 주차면 ROI 확보 · 전역 인덱스 │
                         └───────────────────────────────────────────┘
                                 │ 셋업 산출물(프리셋/슬롯/ROI/전역인덱스)

```

## 4. 에이전트 책임 분담 (할일 26항목 매핑)

| 에이전트 | 담당 할일 번호 | 핵심 책임 |
|----------|----------------|-----------|
| **SettingAgent** | 1~8 | 프리셋 PTZ 초기 셋업, 차량/주차면 검지 기반 ROI 확보, 프리셋별 슬롯 인덱스 부여, **전 카메라·전 프리셋 정렬 → 전역 슬롯 인덱스 매핑** |
| **ActionAgent** | 9~20 | 영상 수신→차량 탐지→주차면 귀속, 센터라이징(번호판 중심 이동/확대 ≥화면20%), LPD/LPR, 변화 감지(정밀조사), 입출차 발생 시 DM 보고 |
| **DMAgent** | 21~25 (26은 2차) | 입출차 최종 판단·관리, 이전상황과 비교(DB), Action에 행동지침 하달, 점검 스케줄링, DB 저장 |

## 5. 외부 REST 서비스 인터페이스 (재사용 대상)

### 5.1 Unity 카메라 / 실 PTZ — `CWebCamCtrlServer`
| 메서드 | 설명 |
|--------|------|
| `GET /health` | 상태 확인 |
| `POST /req_img` `{cam_idx, preset_idx, [pan,tilt,zoom]}` | 프리셋 적용 후 캡처 → `{cam_idx, preset_idx, pan, tilt, zoom, img_name, img_bytes(base64)}` |
| `POST /req_move` `{cam_idx, pan, tilt, zoom}` | PTZ 절대 이동 → `{success}` |
- Zoom 범위 **1.0 ~ 36.0** (클램프 필수). cam_idx/preset_idx 는 **1-based**.
- 실 PTZ 카메라와 시뮬레이터는 **동일 인터페이스로 추상화**(설계_데이터모음 17번).

### 5.2 VPD — 차량 검출/세그먼테이션 (`Sub/da_vpd_api`)
| 메서드 | 설명 |
|--------|------|
| `POST /vpd/api/v2/det/imgupload` (multipart `file`) | 차량 검출 → `{success, id, bboxes[[x1,y1,x2,y2]], confidences[], classes[]}` |
| `POST /vpd/api/v2/seg/imgupload` (multipart `file`) | 세그먼테이션 → 위 + `masks[][][]` |
- YOLO11 기반. bbox(픽셀 좌표)를 정규화 후 슬롯 ROI에 귀속(OccupancyEngine).
- 대안: VLM 기반 점유검지 `VlmFindSlot /v1/detect`(`occupy_slot[]`)도 사용 가능하나, 본 시스템은 **bbox 위치가 필요**하므로 da_vpd_api(det)를 1차 채택.

### 5.3 LPD — 번호판 검출 (`Sub/da_lpd_api`, 대안 `Sub/lpd_server`)
| 메서드 | 설명 |
|--------|------|
| `POST /lpd/api/v1/imgupload` (multipart `file`) | → `{success, id, bboxes[[x1,y1,x2,y2]], confidences[], classes[]}` |
| `GET /lpd/api/v1/resp/img_{id}` | 주석 이미지 다운로드 |
| (대안) `POST /detect` `lpd_server:8000` | YOLO-OBB → `{boxes_xyxy, confs, classes, obb[[cx,cy,w,h,angle]]}` (회전 박스 필요 시) |

### 5.4 VLA_Center — 번호판 센터라이징 추론
| 메서드 | 설명 |
|--------|------|
| `GET /health`, `GET /info` | 상태/모델 정보 |
| `POST /centering` | → `{datas:[{slot, dpan, dtilt, dist, zoom}]}` (슬롯별 **delta** PTZ 조정값) |
- delta 값을 현재 PTZ에 더해 `req_move` 로 적용 → 번호판을 화면 중심으로.

### 5.5 LPR — 차량번호 인식 (`gobackdev.iptime.org:8124`)
| 메서드 | 설명 |
|--------|------|
| `POST /v1/plate-reader/` (multipart, 필드명 **`upload`**) | → `SLprData{ processing_time, results:[{plate, score, box{xmin,ymin,xmax,ymax}, candidates[]}], filename, version, camera_id, timestamp }` |
- `results[].plate` = 번호 문자열, `score` = 신뢰도(1=100%).

### 5.4 ParkSimMgr 재사용 가능 도메인 (참고)
ParkSimMgr 에 이미 존재하여 **이식 또는 패턴 재사용** 가능:
`UnityCameraClient`, `VpdOccupancyClient`, `OccupancyEngine`, `PresetManager`,
`ScanScheduler`, `Repository`, 도메인 타입(`Preset/ParkingSlot/SlotState/ParkingEvent` 등).

## 6. 공유 데이터 모델 (초안)

ParkSimMgr `domain/types.ts` 를 기반으로 ParkAgent 공통 패키지(`@parkagent/types`)로 추출.

```ts
NormalizedRect { x, y, w, h }                       // 0~1 정규화 좌표
Camera { camIdx, name, enabled }
Preset { camIdx, presetIdx, label, coveredSlotIds[], pan?, tilt?, zoom? }
ParkingSlot { slotId, zone, roiByPreset: Record<`${camIdx}:${presetIdx}`, NormalizedRect> }
GlobalSlotIndex { globalIdx, slotId }               // §7 전역 정렬 매핑 (신규)
SlotState { slotId, occupancy, confidence, plate?, lastChangedAt, lastObservedAt, pending* }
VehicleObservation { slotId, plate?, plateConf?, bbox?, capturedAt, sourceImage }   // 신규
ParkingEvent { eventId, slotId, type:ENTER|EXIT, plate?, at, confidence, sourceImage? }
ActionTask { taskId, type, camIdx, presetIdx, slotId?, reason, priority }            // DM→Action (신규)
```

## 7. 확정 사항 (2026-06-24 사용자 답변 반영)

| # | 항목 | 결정 |
|---|------|------|
| 1 | LPD/LPR/VPD 서비스 | **모두 기존 REST 서비스 사용** — VPD=`Sub/da_vpd_api`, LPD=`Sub/da_lpd_api`(대안 `lpd_server`), LPR=`gobackdev.iptime.org:8124/v1/plate-reader/`. 통신은 전부 REST (§5) |
| 2 | DB 종류 | **개발 SQLite / 운영 PostgreSQL**, **신규 독립 DB**(기존 PEventAPIWeb 미연동) (DMAgent §6) |
| 3 | 전역 슬롯 정렬 규칙 | **`cam ASC → preset ASC → 이미지내 위치(상→하 밴드, 좌→우)` 확정** |
| 4 | 빈 주차면 라인 검지 | **불필요.** 시뮬레이터는 강체(rigid body)로 차량 배치 → 차량 검출 기반 ROI로 충분. **현장(실차) 적용은 별도 아이디어 필요(미해결, 2차)** |
| 5 | 센터라이징 트리거 | **ActionAgent 자체 트리거**로 변경. 변화 감지(빈→점유)를 Action이 직접 판단해 센터라이징 실행. DM은 이벤트 최종확정·기록 + 보조 지시(정밀조사) 담당 (ActionAgent §3.3 상세) |
| 6 | GLPR 입출구 정적 카메라(할일 26) | **2차 작업** — 인터페이스만 예약 |
| 7 | 프리셋 정의 출처 | **mapConfig + Unity CameraPos PTZ 추출**(ParkSimMgr 방식) 확정 |
| 8 | 점유 판정 ROI 필요성 | **필요 확정.** bbox→슬롯 귀속에 ROI 필수. 1차는 검출 bbox 기반 ROI, 2차로 월드좌표 투영 ROI 검토 (SettingAgent §8-1) |
| 9 | 카메라 대수/동시성 | 평균 **10~15대**, 최대 **20대+**. 카메라별 직렬 워커 + 모델 호출 전역 세마포어, 20대+는 인스턴스 샤딩 (ActionAgent §3.3-4) |
| 10 | 디바운스 임계 | **연속 2~3회** 확정 |

### 실 PTZ ROI 자동 생성 (수동 불필요)
- 실 카메라도 **검출 누적 + 클러스터링**으로 수동 ROI 없이 자동 생성 가능(SettingAgent §8-1-1).
- 단, 검출 기반은 **점유된 적 있는 면만** 발견 → 상시 빈 면은 아래 미해결 항목과 동일.

### 남은 미해결 항목
- **현장(실차) 상시 빈 주차면 검지**(할일 4-2): 검출 누적으로 미발견되는 상시 빈 면/정확 경계는
  주차선 검지 모델 또는 사전 측량맵+호모그래피 투영으로 보완 — **2차 검토**.

## 8. MCP 기반 agentic 아키텍처 — 두뇌/도구 경계 (결정 #3 재확정)

### 8.1 계층 구조
```
┌───────────────────────────── 에이전트 두뇌 (LLM, model-agnostic) ──────────────────────────┐
│  Claude / Codex / Gemma / Qwen3  …  (교체 가능)                                              │
│  - 목표 + 시스템프롬프트 + MCP 도구셋 을 받아 추론·계획·도구호출                              │
│  - 담당: 고수준 판단 (어느 면 정밀조사·이상상황 해석·번호불일치 판단·점검 우선순위 등)        │
└───────────────▲───────────────────────────────────────────────────────────────────────────┘
                │ MCP (tool call)
┌───────────────┴───────────────────────────────────────────────────────────────────────────┐
│  MCP 도구 서버 층 (능력을 표준 도구로 노출)                                                   │
│  camera.* (req_img/req_move)   vpd.detect   lpd.detect   lpr.read   vla.centering            │
│  centering.run (결정형 제어루프: 줌→LPD→20%반복)   db.* (state/event 조회·기록)              │
│  └ 내부적으로 기존 REST 서비스(da_vpd_api, da_lpd_api, :8124, VLA_Center, Unity)를 호출       │
└─────────────────────────────────────────────────────────────────────────────────────────────┘
```

### 8.2 LLM(두뇌)이 결정하는 것 vs 결정형 도구가 처리하는 것
| 구분 | LLM 두뇌 (MCP로 도구 호출) | 결정형 도구/루프 |
|------|----------------------------|------------------|
| Setting | 셋업 전략·재스캔 판단, ROI 클러스터 이상 검토 | 프리셋 순회 캡처, ROI 산출, 전역 인덱스 정렬 |
| Action | 변화 해석, 정밀조사 여부/순서, 실패 시 재시도 판단 | **센터라이징(줌 20% 반복)·PTZ 미세이동·LPD/LPR 호출** |
| DM | 입출차 최종판단, 번호 불일치·유령차 해석, 점검 우선순위 | 상태머신 디바운스, DB CRUD, 스케줄 발행 |

원칙: **실시간·고빈도·수치반복 루프 = 결정형 도구**, **모호·맥락판단·예외처리 = LLM**.

### 8.3 model-agnostic 보장
- 각 에이전트는 `LLM_PROVIDER`(claude|codex|gemma|qwen3 등) 설정으로 두뇌 모델을 교체.
- 모든 능력은 MCP 표준 스키마로 노출 → 모델이 바뀌어도 도구 계약 불변.
- 도구는 결정형이므로 모델 비결정성의 영향 범위를 최소화(가드레일).

### 8.4 센터라이징 배치 결정 (확정)
- **별도 에이전트로 분리하지 않음.** ActionAgent가 호출하는 **결정형 MCP 도구 `centering.run`** 으로 둔다.
- 이유: 카메라를 점유한 채 `센터라이징→LPD→줌반복→LPR`을 원자적으로 수행해야 하며,
  프로세스 분리 시 카메라 소유권/락 핸드오프 비용·실패모드가 증가. (ActionAgent §3.3 참조)
- 모듈화·재사용성은 MCP 도구화로 확보(향후 호출 주체만 교체 가능).

## 9. 공통 비기능 요구 / 운영 규칙

- **언어/스타일**: TypeScript, ParkSimMgr 컨벤션 준수(ESM, `fetchWithLog` 재시도/로깅 패턴).
- **테스트**: 각 에이전트 유닛테스트 필수(vitest), 외부 서비스는 모킹. 실제 연동 스모크 테스트 별도.
- **문서**: 변경 시 `yyyyMMdd_hhmmss_이름.md` 한글 문서화 + 영향도 분석(프로젝트 규칙).
- **좌표/인덱스 통일**: cam_idx·preset_idx·slot index 모두 **1-based**.
- **장애 격리**: 외부 서비스 호출은 타임아웃·지수백오프 재시도, 실패 시 상태 `UNKNOWN` 으로 강등.

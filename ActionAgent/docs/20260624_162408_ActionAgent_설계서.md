## 1. 목적과 역할

카메라를 직접 제어하는 **실행(액션) 에이전트**. 영상에서 차량을 찾아 주차면에 귀속시키고,
번호판 중심으로 **센터라이징**(PTZ 이동 + 줌)한 뒤 LPD/LPR로 **차량번호를 검출**하여
DMAgent에 보고한다. 또한 프리셋 상태 변화(차량 수/구성 변화)를 감지해 정밀조사하고,
입·출차 발생을 DMAgent에 알린다.

> "카메라"는 **실제 PTZ 카메라와 Unity 시뮬레이터 카메라 모두**를 가리키며 동일 인터페이스로 추상화(할일 17).

### 1-1. 두뇌(LLM)/도구 경계
- **LLM 두뇌(Claude/Codex/Gemma/Qwen3, MCP로 도구 호출)**: 변화 해석, 정밀조사 여부·순서 판단,
  센터라이징 실패 시 재시도/포기 판단 등 **모호·맥락 판단**.
- **결정형 MCP 도구(LLM 미개입)**: `camera.req_img/req_move`, `vpd.detect`, `centering.run`(줌 20% 반복 루프),
  `lpd.detect`, `lpr.read` 등 **실시간·수치반복** 처리.
- 센터라이징은 **별도 에이전트가 아니라 결정형 MCP 도구 `centering.run`** 으로 둔다(아키텍처 §8.4).

## 3. 처리 흐름 (결정형 오케스트레이션)

### 3.1 스캔 사이클 (할일 9, 18~20)
```
[프리셋 스캔]
  └ req_img(cam, preset) → 캡처(현재 PTZ 기록)
  └ VPD /vpd/api/v2/det/imgupload → 차량 bbox 목록
  └ bbox → 정규화 → 슬롯 ROI 귀속(OccupancyEngine, 판정규칙=SettingAgent §8-1) → 현재 점유 슬롯 집합
  └ ChangeDetector: 직전 스냅샷과 비교(§3.3):
        · 점유 동일 → DM에 현상 보고만
        · 빈→점유(차량 증가) → '입차 후보' → 자체 센터라이징 파이프라인(§3.2)
        · 점유→빈(차량 감소) → '출차 후보' → DM 보고(번호 검출 불필요)
```

### 3.2 센터라이징 + 번호 인식 (할일 10~15)
```
[대상 슬롯들 큐]
  └ for each 대상:
        a. VLA /centering → {dpan, dtilt, zoom, dist}
        b. req_move(cam, pan+dpan, tilt+dtilt, clamp(zoom)) → 번호판 중심 이동
        c. LPD /lpd/api/v1/imgupload → 번호판 bbox
        d. 번호판 면적 < 화면 20% 이면 줌 증가 후 b~c 재시도(상한/횟수 제한)
        e. LPR /v1/plate-reader/ → 차량번호(plate) + score
        f. VehicleObservation{cam, preset, slotId, plate, conf} → DM 보고
```

상태: `IDLE → SCANNING → (DETECT_CHANGE) → CENTERING → LPD → LPR → REPORTING → IDLE`.

### 3.3 자체 트리거 센터라이징 설계 (할일 10 — 사용자 요청 상세)

**핵심 원칙:** 센터라이징(비싼 PTZ 이동 + LPD/LPR)은 **상시 수행하지 않는다.**
ActionAgent가 **스스로** 변화를 감지해 "초기 1회 + 입차 시에만" 실행한다(DM 왕복 대기 없음).
DM은 이벤트 최종 확정·기록 및 보조 지시(정밀조사)를 담당하여 **권위(authoritative) 기록은 DM**이 유지.

#### (1) 슬롯별 트리거 상태기 (재무장 = re-arm 모델)
각 슬롯은 센터라이징 관점의 상태를 별도로 가진다(점유 상태와 분리):

```
슬롯 트리거 상태: ARMED → CAPTURING → CAPTURED
                                  └→ RETRY(실패 누적) → (한도초과) GAVE_UP

규칙:
  - 빈→점유 전이가 디바운스 통과 & 상태=ARMED  → CAPTURING 으로 전환, 센터라이징 실행
  - 센터라이징 성공(LPR plate 확보)            → CAPTURED (이후 재센터링 안 함)
  - 센터라이징 실패(LPD/LPR 실패·줌 한도)       → RETRY (쿨다운 후 재시도, 최대 k회)
  - RETRY k회 초과                              → GAVE_UP (DM에 미검출 보고, DM이 정밀조사 지시 가능)
  - 점유→빈 전이 확정                            → 상태를 ARMED 로 재무장(plate 초기화)
```

→ 이 재무장 모델이 **"초기 1회 + 입차 시에만"** 을 그대로 구현한다:
한 번 번호를 딴 차량은 그 자리를 비울 때까지 다시 센터라이징하지 않고,
같은 면에 새 차가 들어오면(빈→점유) 다시 ARMED 가 되어 1회 센터라이징.

#### (2) 초기 1회 스윕
부팅/셋업 직후 첫 스캔: 모든 점유 슬롯의 상태가 ARMED 이므로 **점유 슬롯 전부 1회 센터라이징**.
이후에는 변화(입차)가 있을 때만 트리거 → 자연스럽게 "초기 한 번 + 입차 시" 동작.

#### (3) 로컬 디바운스 (오검지 방지)
- VPD 1프레임 깜빡임이 비싼 LPR 파이프라인을 트리거하지 않도록, 빈→점유 전이는
  **연속 N회(기본 2~3) 동일 관측 시**에만 입차 후보로 확정 후 센터라이징.
- 이 디바운스는 Action 로컬의 가벼운 것이고, DM은 별도의 권위 디바운스/확정을 수행(이중 안전).

#### (4) 동시성·자원 보호 (카메라 평균 10~15, 최대 20+)
- 카메라는 한 번에 한 방향만 가능 → 카메라별 **단일 워커(직렬 큐)**.
  한 카메라에서 여러 슬롯이 동시에 ARMED 면 슬롯 인덱스 순서대로 직렬 센터라이징.
- 센터라이징 중에는 해당 카메라의 일반 스캔을 일시 중지(이동 충돌 방지).
- **카메라 워커는 서로 병렬**(Node 비동기 I/O) → 10~20대 동시 운용. 단 모델 서버(VPD/LPD/LPR/VLA)는
  GPU 자원이 유한하므로 **전역 세마포어로 동시 추론 호출 수(M) 제한**(예: M=4~6, 운영 튜닝).
- 20대 이상으로 확장 시 **카메라 범위로 ActionAgent 인스턴스 샤딩**(예: 1~10 / 11~20) 가능 —
  독립 프로세스 구조이므로 수평 확장이 자연스럽다. DM은 인스턴스별 라우팅만 관리.

#### (5) DM과의 경계
| 구분 | ActionAgent (자체 트리거) | DMAgent |
|------|---------------------------|---------|
| 변화 감지 | ✅ 로컬 스냅샷 비교로 즉시 | (보조) 관측 누적으로 재확인 |
| 센터라이징 결정 | ✅ 트리거 상태기로 자체 결정 | 직접 결정 안 함(정밀조사만 지시) |
| 이벤트 확정/기록 | ❌ (보고만) | ✅ 권위 확정 + DB |
| 추가 지시 | DM `/task` 수신 시 수행 | ✅ 정밀조사/재점검 Task 발행 |

→ **두 경로 공존**: 평상시는 Action 반응형 자체 트리거, 예외/검증은 DM 명령형 Task.
충돌 방지를 위해 동일 슬롯에 대한 DM Task 와 자체 트리거는 **슬롯 단위 락**으로 직렬화.

## 4. 모듈 구성

```
ActionAgent/
  src/
    index.ts
    brain/AgentRuntime.ts         # LLM 두뇌(model-agnostic): 목표+MCP도구셋으로 추론·도구호출
    mcp/server.ts                 # 능력을 MCP 도구로 노출(camera.*, vpd.detect, centering.run, lpd/lpr)
    api/server.ts                 # 작업 수신(/task), 상태 조회 (DM→Action REST)
    runtime/ScanCycle.ts          # 프리셋 스캔 + 변화 감지 (할일 9,18~20)
    runtime/CenteringPipeline.ts  # = MCP 도구 centering.run: 센터라이징→LPD→줌루프→LPR (할일 10~15, 결정형)
    runtime/ChangeDetector.ts     # 직전 점유와 비교 → 입/출차 후보
    runtime/TriggerStateMachine.ts# 슬롯별 ARMED/CAPTURED/RETRY 재무장 상태기 (§3.3)
    clients/CameraClient.ts       # req_img/req_move (실 PTZ + 시뮬 추상화)
    clients/VpdClient.ts          # da_vpd_api /vpd/api/v2/det/imgupload
    clients/CenteringClient.ts    # VLA /centering
    clients/LpdClient.ts          # da_lpd_api /lpd/api/v1/imgupload
    clients/LprClient.ts          # :8124 /v1/plate-reader/ (multipart 'upload')
    clients/ManagerClient.ts      # DM 보고/지시 수신
  docs/ tests/
```
> `clients/*` 는 기존 REST 서비스를 호출하는 어댑터이고, `mcp/server.ts` 가 이를 묶어
> **MCP 도구**(`camera.*`, `vpd.detect`, `centering.run`, `lpd.detect`, `lpr.read`)로 노출한다.
> 두뇌(`brain/AgentRuntime.ts`)는 MCP 도구만 호출하므로 LLM 모델 교체와 무관하게 동작한다.

## 6. 줌 확대 정책 (할일 13) — 피드백 폐루프

- 목표: 번호판 bbox 면적 ≥ 전체 화면 20%.
- **개방형(모델 zoom delta 신뢰) 아닌 폐루프(실측 면적 보정)로 설계** → 동일 코드가 시뮬·실 PTZ 양쪽 동작.
- 방법: 매 반복 `(VLA /centering로 중심 보정) → req_move → 캡처 → LPD로 면적 측정`.
  미달 시 줌 증가(예: ×1.5, 보정 게인<1로 오버슈트 방지) 후 재반복. zoom 상한 36.0.
- 종료: 면적 충족 / 줌 상한 / 최대 반복(N) 도달(실패 시 best-effort LPR + 낮은 신뢰도 보고).

### 6-1. 실 PTZ 카메라 추가 처리 (결정형 그대로, `CameraClient` 어댑터에서 흡수)
| 실 카메라 차이 | 결정형 대응 |
|----------------|-------------|
| 모터 이동 정착시간·오실레이션 | `req_move` 후 motion-complete 폴링/정착 지연 + 오토포커스 대기 후 캡처 |
| PTZ 정확도(백래시) | 측정-보정 폐루프가 흡수(수렴까지 반복) |
| 줌 비선형성·줌 드리프트(중심 이탈) | 줌 1스텝 → **재센터링** → 재측정(루프가 매 반복 centering 수행) |
| 블러/노출/저조도 | 프레임 품질 게이트 + 캡처 재시도 |
| 줌-배율 매핑이 sim과 상이 | 절대 zoom값 비신뢰, **면적 목표 도달까지 증가** 피드백만 사용 |
| 제어 프로토콜(ONVIF/VISCA/벤더SDK) | `CameraClient` 어댑터로 동일 `req_img/req_move` 계약 유지(할일 17) |

> 수렴 안정성: 보정 게인(<1)·허용오차·최대반복·오실레이션 댐핑을 둔다. 이 루프에 LLM을 넣지 않는 이유 =
> 반복당 LLM 지연(1~3s)·비결정성이 실 카메라(정착·재초점으로 반복수↑)에서 치명적이기 때문(아키텍처 §8.2).

## 7. 확정/잔여 사항 (ActionAgent 관련)

- **LPD/LPR/VPD 서비스 (할일 11·14)**: 모두 기존 REST 사용으로 확정 —
  VPD=`da_vpd_api`, LPD=`da_lpd_api`(대안 `lpd_server` OBB), LPR=`:8124/v1/plate-reader/`.
- **센터라이징 트리거 (할일 10)**: **ActionAgent 자체 트리거**로 확정. 상세는 §3.3.
- **변화 감지 기준**(할일 18): 1차는 **점유 수/구성 변화**만 사용(번호/외형 비교는 비용 커서 2차).
- **다중 카메라 동시성**: 카메라별 단일 워커 직렬화(§3.3-4). 카메라 대수/동시성 목표만 확인 요망(잔여).

## 8. 영향도 분석

- 카메라는 물리적으로 1개 PTZ를 한 번에 한 방향만 볼 수 있음 → **이동·캡처는 직렬 자원**.
  Setting 셋업과 동시 구동 시 충돌 → 운영 시간대 분리 또는 카메라 락 필요.
- VLA `/centering` 는 **delta 기반**이므로 현재 PTZ 추적(상태 보유)이 정확해야 함
  (`req_img` 응답의 pan/tilt/zoom를 권위 값으로 사용).
- 보고 스키마(VehicleObservation/ParkingEvent)는 DM의 입력 계약 → 공통 패키지로 관리.

## 9. 테스트 계획

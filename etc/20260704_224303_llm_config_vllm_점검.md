# SettingAgent llm.config.json — vLLM(Qwen2.5-VL-32B) 설정 점검 및 수정 기록

- 작성일: 2026-07-04 22:43:03
- 대상 파일: `SettingAgent/config/llm.config.json`
- 참고 문서: `etc/qwen25vl32b_vllm_setup.md`, `etc/run_qwen25vl32b_vllm.sh`
- 서버: `http://192.168.0.221:8000` (Docker, vLLM `vllm/vllm-openai:latest` 0.24.0)
- 모델: `Qwen/Qwen2.5-VL-32B-Instruct`

## 1. 배경 / 목적

Docker에서 구동 중인 vLLM(OpenAI 호환) 서버에 SettingAgent 두뇌(비전 LLM)를 연결하기 위한
`llm.config.json` 설정이 실제 서버와 정합한지 점검하고, 코드가 각 필드를 어떻게 소비하는지
근거로 하여 문제를 바로잡았다.

## 2. 최초 설정에서 발견된 문제

점검 시점 스냅샷 기준(IDE 선택 값)으로 아래 3가지가 문제였다.

### 2-1. `baseUrl` 이 `/v1/models` 로 끝남 (치명적)

`/v1/models` 는 모델 **목록 조회(GET)** 엔드포인트이며 채팅 호출용이 아니다.

- OpenAI 경로(`AgentRuntime.ts:46`): SDK가 `baseURL` 뒤에 `/chat/completions` 를 붙임
  → `.../v1/models/chat/completions` (404)
- 네이티브/warmup 경로(`AgentRuntime.ts:87`, `:284`): 정규식 `/\/v1\/?$/` 는 **문자열 끝의 `/v1`만**
  제거하는데, `/v1/models` 로 끝나 매칭 실패 → `.../v1/models/api/chat` (404)

→ 올바른 값: `http://192.168.0.221:8000/v1`

### 2-2. `api: "ollama"` 인데 실제 서버는 vLLM (치명적)

코드는 `api === 'ollama'` 일 때 `chatNative` 로 분기하여 **Ollama 전용 `/api/chat`** 를 호출한다
(`AgentRuntime.ts:277`). vLLM에는 `/api/chat` 이 없어 404가 난다.

- 모델명 `Qwen/Qwen2.5-VL-32B-Instruct`(HuggingFace repo id 형식) + 포트 8000 → vLLM 특징
- 사용자 확인: **vLLM 확정**

→ 올바른 값: `api: "openai"` (vLLM = OpenAI 호환 `/v1/chat/completions`)

### 2-3. `warmup.enabled: true` (vLLM에서 부적합)

warmup은 `chatNative` 와 동일하게 Ollama 네이티브 `/api/chat` 에 `keep_alive`/`num_predict`
를 실어 호출한다(`AgentRuntime.ts:87`). vLLM에는 해당 엔드포인트가 없어 매 기동 시 무의미한
호출 + 콜드로드용 120초 타임아웃을 낭비한다(코드상 best-effort라 throw는 없음). vLLM은 모델을
상시 상주시켜 콜드로드가 없으므로 warmup 자체가 불필요하다.

→ 올바른 값: `warmup.enabled: false`

## 3. `max_model_len` vs `maxTokens` 컨텍스트 충돌 (핵심)

vLLM에서 `--max-model-len` 은 **입력+출력 합산 총 컨텍스트**다.
config의 `maxTokens` 는 `max_tokens`(출력)으로 전달된다(`AgentRuntime.ts:320`).

| 서버 max_model_len | 입력 예산 (= max_model_len − maxTokens 3072) | 비전 요청 가부 |
|--------------------|---------------------------------------------|----------------|
| 4096               | 1024 토큰                                    | ❌ 이미지(~700)+프롬프트 초과 위험 |
| 8192               | 5120 토큰                                    | ✅ 여유 충분 |

- 참고 문서 §10/§11의 기록된 실행은 `--max-model-len 4096` 이었음(GPU 메모리 실험 잔재).
- 실행 스크립트 `run_qwen25vl32b_vllm.sh:6` 기본값은 `MAX_MODEL_LEN=8192`.
- **조치:** 서버를 8192로 재기동. 실측 확인:

```bash
curl -s http://192.168.0.221:8000/v1/models | grep -o '"max_model_len":[0-9]*'
# 결과: "max_model_len":8192
```

→ `maxTokens: 3072` 유지 가능(입력 예산 5120). config 변경 불필요.

## 4. 실제 수행한 변경 (이번 세션)

`SettingAgent/config/llm.config.json` 에서 2건만 수정.

| 필드 | 변경 전 | 변경 후 | 사유 |
|------|---------|---------|------|
| `warmup.enabled` | `true` | `false` | vLLM엔 `/api/chat` 없음 |
| `_comment` | Ollama/gemma 기재 | vLLM/Qwen2.5-VL로 정정 | 실제 서버와 불일치 |

> `baseUrl`(`/v1`)과 `api`(`openai`)는 점검 시점엔 이미 올바른 값으로 저장돼 있어 추가 수정 없음.

## 5. 최종 확정 설정

```json
"llm": {
    "provider": "openai-compatible",
    "model": "Qwen/Qwen2.5-VL-32B-Instruct",
    "baseUrl": "http://192.168.0.221:8000/v1",
    "apiKeyEnv": "LLM_API_KEY",
    "temperature": 0.1,
    "maxTokens": 3072,
    "enabled": true,
    "timeoutMs": 30000,
    "api": "openai",
    "think": false,
    "imageMaxEdge": 960
}
```
+ `"warmup": { "enabled": false, ... }`

| 항목 | 값 | 상태 |
|------|-----|------|
| provider / api | openai-compatible / openai | ✅ vLLM 정합 |
| baseUrl | `.../v1` | ✅ `/chat/completions` 자동 부착 |
| model | `Qwen/Qwen2.5-VL-32B-Instruct` | ✅ 서버 모델 ID와 정확히 일치 |
| maxTokens 3072 | 서버 8192 | ✅ 입력 예산 5120 |
| 이미지 전송 | base64 data URL (`AgentRuntime.ts:314`) | ✅ 외부 URL 403 회피 |
| warmup.enabled | false | ✅ vLLM `/api/chat` 없음 |

## 6. 영향도 분석

- **`AgentRuntime.chatOpenai` (`AgentRuntime.ts:308`)**: 이제 정상 경로. `baseUrl`+`/chat/completions`
  로 vLLM 호출, 이미지는 base64 data URL 전송. `think` 필드는 이 경로에서 무시되어 무해.
- **`AgentRuntime.chatNative` / `warmUp` (`:283`, `:75`)**: `api: openai` 이므로 호출되지 않음.
  warmup도 비활성이라 `/api/chat` 접근 시도 자체가 사라짐.
- **`loadLlmConfig` / `LlmConfigSchema` (`llmConfig.ts`)**: 변경 값 모두 스키마 허용 범위
  (`api` enum에 `openai` 포함, `warmup.enabled` boolean). 파싱 실패 없음.
- **비전 파이프라인(floorRoi, centering, setupPrompts stage1~3)**: 동일 `AgentRuntime` 를 공유하므로
  위 수정으로 함께 정상화. 스키마·프롬프트 파일 경로 변경 없음.
- **유닛테스트**: config 값(JSON bool/string) 변경으로 코드 로직·시그니처 변화 없음 → 기존 테스트
  영향 없음. 실검증은 실서버 실호출로 수행(아래 §7).

## 7. 동작 확인 (권장 최종 체크)

설정값이 아니라 네트워크·실호출 검증. SettingAgent 컨테이너 내부에서 실행.

```bash
# 1) 도커 네트워크 도달 + 모델 목록
curl -s http://192.168.0.221:8000/v1/models

# 2) 실제 chat completion (텍스트)
curl -s http://192.168.0.221:8000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"Qwen/Qwen2.5-VL-32B-Instruct","messages":[{"role":"user","content":"ping"}],"max_tokens":8}'
```

- 200 + 응답이면 도커 네트워크 포함 정상.
- 만약 vLLM이 같은 도커 호스트에서 돌고 IP가 안 잡히면 `192.168.0.221` 대신
  `host.docker.internal`(또는 compose 서비스명) 사용 검토.

## 8. 재발 방지 메모

- 서버를 `run_qwen25vl32b_vllm.sh` 로 띄우면 기본 `MAX_MODEL_LEN=8192`. §11 스타일 one-liner나
  `MAX_MODEL_LEN=4096` 환경변수로 띄우면 4096이 되어 비전 요청이 잘릴 수 있으니, 기동 후
  `curl .../v1/models | grep max_model_len` 로 실측 확인 권장.
- vLLM 사용 시 `api` 는 항상 `openai`, `warmup.enabled` 는 `false` 로 둔다(Ollama 전용 경로 회피).
- `baseUrl` 은 반드시 `/v1` 로 끝나야 한다(`/v1/models` 아님).
